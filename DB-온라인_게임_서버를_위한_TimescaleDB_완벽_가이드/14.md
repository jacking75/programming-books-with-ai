# Chapter 14: 멀티 리전 및 확장성

게임이 성공하면 플레이어 수가 급증하고, 여러 국가로 서비스가 확장된다. 한국에서 시작한 게임이 북미, 유럽, 동남아로 퍼지면서 전 세계 플레이어들이 동시에 접속한다. 이때 단일 데이터베이스 서버로는 모든 트래픽을 감당할 수 없고, 지역별 네트워크 지연(Latency)도 심각한 문제가 된다. 서울에 있는 데이터베이스에 미국 플레이어가 접속하면 수백 밀리초의 지연이 발생하여 게임 경험이 나빠진다.

이 장에서는 TimescaleDB를 글로벌 규모로 확장하는 방법을 배운다. 복제(Replication)를 통해 읽기 성능을 향상시키고, 멀티 리전 아키텍처로 전 세계 플레이어에게 빠른 서비스를 제공하며, Docker를 활용하여 클러스터를 구성하는 실전 기법을 익힌다.

## 14.1 복제(Replication) 설정

복제는 데이터베이스의 데이터를 다른 서버에 실시간으로 복사하는 기술이다. PostgreSQL(TimescaleDB의 기반)은 강력한 복제 기능을 제공한다.

**복제의 종류와 용도**

PostgreSQL은 두 가지 주요 복제 방식을 지원한다.

**스트리밍 복제(Streaming Replication)**는 Primary 서버의 WAL(Write-Ahead Log)을 Standby 서버로 실시간 전송하여 거의 동일한 데이터를 유지한다. 읽기 부하를 분산하거나 장애 대비용(High Availability)으로 사용한다. 설정이 비교적 간단하고 성능이 우수하다.

**논리 복제(Logical Replication)**는 특정 테이블이나 데이터만 선택적으로 복제할 수 있다. 다른 버전의 PostgreSQL 간 복제도 가능하며, 양방향 복제나 복잡한 토폴로지 구성에 적합하다. 설정은 복잡하지만 유연성이 높다.

게임 서버 모니터링에서는 주로 스트리밍 복제를 사용한다. 모든 메트릭과 로그를 복제하여 읽기 부하를 분산하는 것이 목적이기 때문이다.

**스트리밍 복제 설정하기**

Primary 서버(마스터)와 Standby 서버(슬레이브)를 구성한다. Windows 환경에서는 Docker를 사용하면 편리하다.

먼저 Primary 서버의 `postgresql.conf` 설정을 변경한다.

```ini
# postgresql.conf (Primary 서버)

# WAL 설정
wal_level = replica
max_wal_senders = 10
wal_keep_size = 1GB
synchronous_commit = off  # 비동기 복제 (성능 우선)

# 복제 슬롯 사용 (권장)
max_replication_slots = 10

# 아카이브 설정 (선택사항)
archive_mode = on
archive_command = 'copy "%p" "C:\\WAL\\archive\\%f"'
```

`pg_hba.conf` 파일에서 Standby 서버의 접속을 허용한다.

```
# pg_hba.conf (Primary 서버)

# TYPE  DATABASE        USER            ADDRESS                 METHOD
host    replication     replicator      192.168.1.0/24          md5
```

Primary 서버에서 복제 전용 사용자를 생성한다.

```sql
-- Primary 서버에서 실행
CREATE USER replicator WITH REPLICATION ENCRYPTED PASSWORD 'replica_password_123';
```

이제 Standby 서버를 구성한다. Primary 서버의 데이터를 베이스 백업으로 가져온다.

```powershell
# Windows PowerShell에서 실행 (Standby 서버)

# PostgreSQL bin 디렉토리로 이동
cd "C:\Program Files\PostgreSQL\15\bin"

# 베이스 백업 생성
.\pg_basebackup.exe `
    --host=primary-server-ip `
    --port=5432 `
    --username=replicator `
    --pgdata=C:\PostgreSQL\Standby\data `
    --wal-method=stream `
    --write-recovery-conf `
    --progress `
    --verbose
```

Standby 서버의 `postgresql.conf`를 설정한다.

```ini
# postgresql.conf (Standby 서버)

# Standby 서버는 읽기 전용
hot_standby = on

# 복제 지연 모니터링
hot_standby_feedback = on
```

`standby.signal` 파일을 생성하여 Standby 모드를 활성화한다.

```powershell
# Standby 데이터 디렉토리에 파일 생성
New-Item -Path "C:\PostgreSQL\Standby\data\standby.signal" -ItemType File
```

Standby 서버의 `postgresql.auto.conf` 또는 `recovery.conf` (구버전)에 Primary 연결 정보를 설정한다.

```ini
# postgresql.auto.conf (Standby 서버)

primary_conninfo = 'host=primary-server-ip port=5432 user=replicator password=replica_password_123'
primary_slot_name = 'standby_slot_1'
```

Primary 서버에서 복제 슬롯을 생성한다.

```sql
-- Primary 서버에서 실행
SELECT * FROM pg_create_physical_replication_slot('standby_slot_1');
```

이제 Standby 서버를 시작하면 자동으로 Primary 서버에 연결하여 복제가 시작된다.

**복제 상태 확인**

Primary 서버에서 복제 상태를 확인한다.

```sql
-- Primary 서버에서 실행
SELECT 
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    sync_state,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) as lag_bytes
FROM pg_stat_replication;
```

결과 예시:

```
 client_addr  |   state   |  sent_lsn  | write_lsn  | flush_lsn  | replay_lsn | sync_state | lag_bytes 
--------------+-----------+------------+------------+------------+------------+------------+-----------
 192.168.1.20 | streaming | 0/3000108  | 0/3000108  | 0/3000108  | 0/3000108  | async      |         0
```

`state`가 `streaming`이면 정상적으로 복제 중이고, `lag_bytes`가 0에 가까우면 거의 실시간으로 동기화되고 있다는 의미다.

Standby 서버에서도 확인할 수 있다.

```sql
-- Standby 서버에서 실행
SELECT pg_is_in_recovery(); -- true면 Standby 모드

SELECT 
    pg_last_wal_receive_lsn() as receive,
    pg_last_wal_replay_lsn() as replay,
    pg_last_xact_replay_timestamp() as replay_timestamp,
    now() - pg_last_xact_replay_timestamp() as replication_lag;
```

`replication_lag`이 수 초 이내면 거의 실시간 복제가 이루어지고 있다.

**C#에서 복제 상태 모니터링**

```csharp
using Npgsql;
using System;
using System.Threading.Tasks;

public class ReplicationMonitor
{
    private readonly string _primaryConnectionString;
    private readonly ILogger<ReplicationMonitor> _logger;

    public ReplicationMonitor(string primaryConnectionString, ILogger<ReplicationMonitor> logger)
    {
        _primaryConnectionString = primaryConnectionString;
        _logger = logger;
    }

    public async Task<ReplicationStatus> CheckReplicationStatusAsync()
    {
        using var connection = new NpgsqlConnection(_primaryConnectionString);
        await connection.OpenAsync();

        var query = @"
            SELECT 
                client_addr::text,
                state,
                pg_wal_lsn_diff(sent_lsn, replay_lsn) as lag_bytes,
                EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp()))::int as lag_seconds
            FROM pg_stat_replication
            WHERE application_name = 'walreceiver'";

        using var command = new NpgsqlCommand(query, connection);
        using var reader = await command.ExecuteReaderAsync();

        if (await reader.ReadAsync())
        {
            var status = new ReplicationStatus
            {
                ClientAddress = reader.GetString(0),
                State = reader.GetString(1),
                LagBytes = reader.GetInt64(2),
                LagSeconds = reader.IsDBNull(3) ? 0 : reader.GetInt32(3),
                IsHealthy = reader.GetString(1) == "streaming" && reader.GetInt64(2) < 1024 * 1024 // 1MB 이내
            };

            if (!status.IsHealthy)
            {
                _logger.LogWarning(
                    "복제 지연 발생: {LagBytes} bytes, {LagSeconds}초",
                    status.LagBytes, status.LagSeconds);
            }

            return status;
        }

        _logger.LogError("복제 상태를 확인할 수 없습니다.");
        return null;
    }
}

public class ReplicationStatus
{
    public string ClientAddress { get; set; }
    public string State { get; set; }
    public long LagBytes { get; set; }
    public int LagSeconds { get; set; }
    public bool IsHealthy { get; set; }
}
```

## 14.2 읽기 부하 분산

복제를 설정했으면 이제 읽기 쿼리를 Standby 서버로 분산하여 Primary 서버의 부하를 줄인다.

**읽기/쓰기 분리 아키텍처**

```
                    ┌─────────────────────┐
                    │   Application       │
                    │   (C# Game Server)  │
                    └──────────┬──────────┘
                               │
                ┌──────────────┼──────────────┐
                │              │              │
         [쓰기 쿼리]      [읽기 쿼리]   [읽기 쿼리]
                │              │              │
                ▼              ▼              ▼
        ┌───────────┐  ┌───────────┐  ┌───────────┐
        │  Primary  │  │ Standby 1 │  │ Standby 2 │
        │  (Master) │  │ (Replica) │  │ (Replica) │
        └─────┬─────┘  └───────────┘  └───────────┘
              │               ▲              ▲
              └───복제────────┴──────────────┘
```

쓰기는 Primary로, 읽기는 Standby로 자동 라우팅하는 C# 라이브러리를 만든다.

**읽기/쓰기 분리 QueryFactory 구현**

```csharp
using SqlKata.Execution;
using Npgsql;
using System;
using System.Collections.Generic;
using System.Linq;

public class LoadBalancedQueryFactory
{
    private readonly QueryFactory _writeFactory;
    private readonly List<QueryFactory> _readFactories;
    private readonly Random _random = new Random();
    private int _currentIndex = 0;

    public LoadBalancedQueryFactory(DatabaseConnectionConfig config)
    {
        // Primary 연결 (쓰기용)
        var writeConnection = new NpgsqlConnection(config.PrimaryConnection);
        var writeCompiler = new SqlKata.Compilers.PostgresCompiler();
        _writeFactory = new QueryFactory(writeConnection, writeCompiler);

        // Replica 연결들 (읽기용)
        _readFactories = config.ReadReplicaConnections
            .Select(connStr =>
            {
                var conn = new NpgsqlConnection(connStr);
                var compiler = new SqlKata.Compilers.PostgresCompiler();
                return new QueryFactory(conn, compiler);
            })
            .ToList();

        // Replica가 없으면 Primary를 읽기에도 사용
        if (_readFactories.Count == 0)
        {
            _readFactories.Add(_writeFactory);
        }
    }

    // 쓰기용 QueryFactory
    public QueryFactory Write => _writeFactory;

    // 읽기용 QueryFactory (로드 밸런싱)
    public QueryFactory Read => GetReadFactory();

    private QueryFactory GetReadFactory()
    {
        if (_readFactories.Count == 1)
            return _readFactories[0];

        // Round-robin 방식
        var factory = _readFactories[_currentIndex];
        _currentIndex = (_currentIndex + 1) % _readFactories.Count;
        return factory;

        // 또는 Random 방식
        // return _readFactories[_random.Next(_readFactories.Count)];
    }
}
```

**사용 예시**

```csharp
public class GameMetricsRepository
{
    private readonly LoadBalancedQueryFactory _db;
    private readonly ILogger<GameMetricsRepository> _logger;

    public GameMetricsRepository(LoadBalancedQueryFactory db, ILogger<GameMetricsRepository> logger)
    {
        _db = db;
        _logger = logger;
    }

    // 쓰기: Primary로
    public async Task SaveServerMetricAsync(ServerMetric metric)
    {
        await _db.Write.Query("server_metrics").InsertAsync(new
        {
            server_id = metric.ServerId,
            cpu_usage = metric.CpuUsage,
            memory_usage_gb = metric.MemoryUsageGb,
            timestamp = metric.Timestamp
        });

        _logger.LogDebug("메트릭 저장 완료: {ServerId}", metric.ServerId);
    }

    // 읽기: Standby로 (부하 분산)
    public async Task<List<ServerMetric>> GetRecentMetricsAsync(string serverId, TimeSpan duration)
    {
        var from = DateTime.UtcNow.Subtract(duration);

        var results = await _db.Read.Query("server_metrics")
            .Where("server_id", serverId)
            .Where("timestamp", ">=", from)
            .OrderByDesc("timestamp")
            .GetAsync<ServerMetric>();

        _logger.LogDebug("메트릭 조회: {ServerId}, {Count}건", serverId, results.Count());

        return results.ToList();
    }

    // 통계 쿼리도 Standby로
    public async Task<ServerStatistics> GetServerStatisticsAsync(string serverId, DateTime from, DateTime to)
    {
        var result = await _db.Read.Query("server_metrics")
            .Where("server_id", serverId)
            .WhereBetween("timestamp", from, to)
            .SelectRaw(@"
                AVG(cpu_usage) as avg_cpu,
                MAX(cpu_usage) as max_cpu,
                AVG(memory_usage_gb) as avg_memory,
                MAX(memory_usage_gb) as max_memory,
                COUNT(*) as sample_count
            ")
            .FirstAsync<ServerStatistics>();

        return result;
    }
}

public class ServerStatistics
{
    public double AvgCpu { get; set; }
    public double MaxCpu { get; set; }
    public double AvgMemory { get; set; }
    public double MaxMemory { get; set; }
    public long SampleCount { get; set; }
}
```

**헬스체크와 자동 페일오버**

Replica가 다운되면 자동으로 다른 Replica로 전환한다.

```csharp
public class HealthAwareLoadBalancedQueryFactory
{
    private readonly QueryFactory _writeFactory;
    private readonly List<ReplicaInfo> _replicas;
    private readonly ILogger<HealthAwareLoadBalancedQueryFactory> _logger;
    private readonly Timer _healthCheckTimer;

    public HealthAwareLoadBalancedQueryFactory(
        DatabaseConnectionConfig config,
        ILogger<HealthAwareLoadBalancedQueryFactory> logger)
    {
        _logger = logger;

        // Primary 설정
        var writeConnection = new NpgsqlConnection(config.PrimaryConnection);
        _writeFactory = new QueryFactory(writeConnection, new SqlKata.Compilers.PostgresCompiler());

        // Replica 설정
        _replicas = config.ReadReplicaConnections
            .Select(connStr => new ReplicaInfo
            {
                ConnectionString = connStr,
                Factory = new QueryFactory(
                    new NpgsqlConnection(connStr),
                    new SqlKata.Compilers.PostgresCompiler()),
                IsHealthy = true
            })
            .ToList();

        // 주기적 헬스체크 (30초마다)
        _healthCheckTimer = new Timer(async _ => await CheckHealthAsync(), null, TimeSpan.Zero, TimeSpan.FromSeconds(30));
    }

    public QueryFactory Write => _writeFactory;

    public QueryFactory Read
    {
        get
        {
            var healthyReplicas = _replicas.Where(r => r.IsHealthy).ToList();

            if (healthyReplicas.Count == 0)
            {
                _logger.LogWarning("사용 가능한 Replica가 없습니다. Primary 사용.");
                return _writeFactory;
            }

            // Round-robin
            var replica = healthyReplicas[Random.Shared.Next(healthyReplicas.Count)];
            return replica.Factory;
        }
    }

    private async Task CheckHealthAsync()
    {
        foreach (var replica in _replicas)
        {
            try
            {
                using var connection = new NpgsqlConnection(replica.ConnectionString);
                await connection.OpenAsync();
                
                using var command = new NpgsqlCommand("SELECT 1", connection);
                await command.ExecuteScalarAsync();

                if (!replica.IsHealthy)
                {
                    _logger.LogInformation("Replica 복구됨: {ConnectionString}", MaskConnectionString(replica.ConnectionString));
                    replica.IsHealthy = true;
                }
            }
            catch (Exception ex)
            {
                if (replica.IsHealthy)
                {
                    _logger.LogError(ex, "Replica 다운: {ConnectionString}", MaskConnectionString(replica.ConnectionString));
                    replica.IsHealthy = false;
                }
            }
        }
    }

    private string MaskConnectionString(string connStr)
    {
        // 비밀번호 마스킹
        return connStr.Split(';')
            .Select(part => part.Contains("Password") ? "Password=***" : part)
            .Aggregate((a, b) => $"{a};{b}");
    }

    private class ReplicaInfo
    {
        public string ConnectionString { get; set; }
        public QueryFactory Factory { get; set; }
        public bool IsHealthy { get; set; }
    }
}
```

## 14.3 멀티 리전 아키텍처

게임이 글로벌로 확장되면 지역별로 데이터베이스를 배치하여 네트워크 지연을 최소화한다.

**지역별 배포 전략**

세 가지 주요 전략이 있다.

**단일 글로벌 데이터베이스**는 한 곳에 데이터베이스를 두고 전 세계에서 접속한다. 구현이 가장 단순하지만 지역 간 지연이 크다. 플레이어 수가 적거나 실시간성이 중요하지 않은 경우 적합하다.

**지역별 읽기 복제본**은 각 지역에 Standby 서버를 배치하여 읽기는 빠르게, 쓰기는 중앙 서버로 전송한다. 대부분의 쿼리가 읽기인 경우 효과적이다. 게임 모니터링과 로그 분석에 적합하다.

**지역별 독립 데이터베이스**는 각 지역이 독립적인 데이터베이스를 운영한다. 완전한 데이터 주권과 최소 지연을 보장하지만, 글로벌 통합 분석이 어렵다. 법적 규제가 엄격한 경우 필요하다.

게임 모니터링 시스템은 대부분 **지역별 읽기 복제본** 전략을 사용한다.

**멀티 리전 아키텍처 예시**

```
┌─────────────────────────────────────────────────────────────────┐
│                     Global Architecture                          │
└─────────────────────────────────────────────────────────────────┘

        [Seoul Region]              [US-West Region]          [EU Region]
              │                            │                       │
    ┌─────────▼─────────┐        ┌────────▼────────┐    ┌────────▼────────┐
    │  Game Servers     │        │  Game Servers   │    │  Game Servers   │
    │  (C# Services)    │        │  (C# Services)  │    │  (C# Services)  │
    └─────────┬─────────┘        └────────┬────────┘    └────────┬────────┘
              │                            │                       │
    ┌─────────▼─────────┐        ┌────────▼────────┐    ┌────────▼────────┐
    │  Local Standby    │        │  Local Standby  │    │  Local Standby  │
    │  (Read Replica)   │        │  (Read Replica) │    │  (Read Replica) │
    └─────────┬─────────┘        └────────┬────────┘    └────────┬────────┘
              │                            │                       │
              └────────────────┬───────────┴───────────────────────┘

                     ┌─────────▼──────────┐
```

**지역 인식 연결 관리**

클라이언트의 지역에 따라 자동으로 가까운 Replica에 연결한다.

```csharp
public enum Region
{
    Seoul,
    USWest,
    USEast,
    Europe,
    Japan
}

public class RegionAwareDatabaseConfig
{
    public Region CurrentRegion { get; set; }
    public string PrimaryConnection { get; set; }
    public Dictionary<Region, string> RegionalReplicas { get; set; } = new();
}

public class RegionAwareQueryFactory
{
    private readonly QueryFactory _writeFactory;
    private readonly Dictionary<Region, QueryFactory> _regionalReadFactories;
    private readonly Region _currentRegion;
    private readonly ILogger<RegionAwareQueryFactory> _logger;

    public RegionAwareQueryFactory(
        RegionAwareDatabaseConfig config,
        ILogger<RegionAwareQueryFactory> logger)
    {
        _currentRegion = config.CurrentRegion;
        _logger = logger;

        // Primary (쓰기)
        _writeFactory = new QueryFactory(
            new NpgsqlConnection(config.PrimaryConnection),
            new SqlKata.Compilers.PostgresCompiler());

        // 지역별 Replica (읽기)
        _regionalReadFactories = config.RegionalReplicas
            .ToDictionary(
                kvp => kvp.Key,
                kvp => new QueryFactory(
                    new NpgsqlConnection(kvp.Value),
                    new SqlKata.Compilers.PostgresCompiler()));

        _logger.LogInformation("현재 리전: {Region}", _currentRegion);
    }

    public QueryFactory Write => _writeFactory;

    public QueryFactory Read
    {
        get
        {
            // 현재 리전의 Replica 사용
            if (_regionalReadFactories.TryGetValue(_currentRegion, out var factory))
            {
                return factory;
            }

            // Fallback: Primary 사용
            _logger.LogWarning("리전 {Region}의 Replica가 없습니다. Primary 사용.", _currentRegion);
            return _writeFactory;
        }
    }

    // 특정 리전의 Replica 지정
    public QueryFactory GetReadFactoryForRegion(Region region)
    {
        if (_regionalReadFactories.TryGetValue(region, out var factory))
        {
            return factory;
        }

        _logger.LogWarning("리전 {Region}의 Replica가 없습니다.", region);
        return _writeFactory;
    }
}
```

```csharp
    public async Task<Dictionary<Region, int>> GetConcurrentUsersByRegionAsync()
    {
        var results = new Dictionary<Region, int>();

        foreach (Region region in Enum.GetValues(typeof(Region)))
        {
            try
            {
                var factory = _db.GetReadFactoryForRegion(region);
                var count = await factory.Query("server_metrics")
                    .Where("timestamp", ">", DateTime.UtcNow.AddMinutes(-5))
                    .SelectRaw("SUM(active_connections) as total")
                    .FirstOrDefaultAsync<dynamic>();

                results[region] = count?.total ?? 0;
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "리전 {Region} 데이터 조회 실패", region);
                results[region] = 0;
            }
        }

        return results;
    }

    public async Task<int> GetGlobalConcurrentUsersAsync()
    {
        var byRegion = await GetConcurrentUsersByRegionAsync();
        return byRegion.Values.Sum();
```

## 14.4 Timescale Cloud 활용

Timescale Cloud는 TimescaleDB의 관리형 서비스로, 복제, 백업, 모니터링을 자동화한다. 직접 인프라를 관리하지 않아도 되므로 개발에 집중할 수 있다.

**Timescale Cloud의 장점**

자동 백업과 복구는 매일 자동 백업이 수행되며, 특정 시점으로 복구(PITR)가 가능하다. 복제 설정은 클릭 몇 번으로 읽기 복제본을 추가할 수 있다. 자동 스케일링은 디스크 공간이 부족하면 자동으로 확장된다. 멀티 리전 지원은 전 세계 주요 리전에서 서비스를 제공한다. 모니터링 대시보드는 성능 지표를 실시간으로 확인할 수 있다.

**Timescale Cloud 시작하기**

## 14.6 모니터링과 장애 대응

클러스터 운영 시 모니터링과 장애 대응 전략이 필수다.

**핵심 모니터링 지표**

복제 지연(Replication Lag)은 Standby가 Primary보다 얼마나 뒤처져 있는지를 나타낸다. 수 초 이내가 정상이며, 수 분 이상이면 문제다. 연결 수(Connection Count)는 `max_connections`에 가까워지면 새 연결이 거부된다. 디스크 사용률은 80% 이상이면 경고, 90% 이상이면 위험이다. 쿼리 성능은 느린 쿼리(1초 이상)의 빈도를 추적한다. CPU/메모리 사용률은 지속적으로 90% 이상이면 스케일업이 필요하다.

**모니터링 쿼리 모음**

```sql
-- 복제 상태 확인
SELECT 
    application_name,
    client_addr,
    state,
    sync_state,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) / 1024 / 1024 as lag_mb,
    replay_lag
FROM pg_stat_replication;

-- 데이터베이스 크기
SELECT 
    pg_database.datname,
    pg_size_pretty(pg_database_size(pg_database.datname)) AS size
FROM pg_database
ORDER BY pg_database_size(pg_database.datname) DESC;

-- 테이블별 크기
SELECT 
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size,
    pg_size_pretty(pg_relation_size(schemaname||'.'||tablename)) AS table_size,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename) - pg_relation_size(schemaname||'.'||tablename)) AS index_size
FROM pg_tables
WHERE schemaname NOT IN ('pg_catalog', 'information_schema', '_timescaledb_internal')
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC
LIMIT 20;

-- 활성 연결 수
SELECT 
    count(*) as total_connections,
    count(*) FILTER (WHERE state = 'active') as active_connections,
    count(*) FILTER (WHERE state = 'idle') as idle_connections,
    count(*) FILTER (WHERE state = 'idle in transaction') as idle_in_transaction
FROM pg_stat_activity
WHERE datname = current_database();

-- 느린 쿼리 (현재 실행 중)
SELECT 
    pid,
    now() - query_start as duration,
    state,
    query
FROM pg_stat_activity
WHERE state != 'idle'
    AND query NOT LIKE '%pg_stat_activity%'
ORDER BY duration DESC
LIMIT 10;

-- Hypertable 통계
SELECT 
    hypertable_name,
    num_chunks,
    pg_size_pretty(total_bytes) as total_size,
    pg_size_pretty(total_bytes - index_bytes) as table_size,
    pg_size_pretty(index_bytes) as index_size
FROM timescaledb_information.hypertables
ORDER BY total_bytes DESC;
```

**C# 모니터링 서비스**

```csharp
using System;
using System.Collections.Generic;
using System.Linq;
using System.Threading.Tasks;
using Npgsql;

public class ClusterMonitoringService
{
    private readonly string _primaryConnectionString;
    private readonly ILogger<ClusterMonitoringService> _logger;

    public ClusterMonitoringService(string primaryConnectionString, ILogger<ClusterMonitoringService> logger)
    {
        _primaryConnectionString = primaryConnectionString;
        _logger = logger;
    }

    public async Task<ClusterHealthStatus> GetHealthStatusAsync()
    {
        var status = new ClusterHealthStatus
        {
            CheckTime = DateTime.UtcNow
        };

        using var connection = new NpgsqlConnection(_primaryConnectionString);
        await connection.OpenAsync();

        // 복제 상태
        status.ReplicationStatus = await GetReplicationStatusAsync(connection);

        // 연결 수
        status.ConnectionStats = await GetConnectionStatsAsync(connection);

        // 데이터베이스 크기
        status.DatabaseSizeGB = await GetDatabaseSizeAsync(connection);

        // 느린 쿼리
        status.SlowQueries = await GetSlowQueriesAsync(connection);

        // 전체 상태 판단
        status.IsHealthy = DetermineOverallHealth(status);

        if (!status.IsHealthy)
        {
            _logger.LogWarning("클러스터 상태 이상: {Status}", status);
        }

        return status;
    }

    private async Task<List<ReplicationInfo>> GetReplicationStatusAsync(NpgsqlConnection connection)
    {
        var replicas = new List<ReplicationInfo>();

        var query = @"
            SELECT 
                application_name,
                client_addr::text,
                state,
                sync_state,
                pg_wal_lsn_diff(sent_lsn, replay_lsn) as lag_bytes,
                EXTRACT(EPOCH FROM replay_lag)::int as lag_seconds
            FROM pg_stat_replication";

        using var command = new NpgsqlCommand(query, connection);
        using var reader = await command.ExecuteReaderAsync();

        while (await reader.ReadAsync())
        {
            replicas.Add(new ReplicationInfo
            {
                ApplicationName = reader.GetString(0),
                ClientAddress = reader.GetString(1),
                State = reader.GetString(2),
                SyncState = reader.GetString(3),
                LagBytes = reader.GetInt64(4),
                LagSeconds = reader.IsDBNull(5) ? 0 : reader.GetInt32(5)
            });
        }

        return replicas;
    }

    private async Task<ConnectionStats> GetConnectionStatsAsync(NpgsqlConnection connection)
    {
        var query = @"
            SELECT 
                count(*) as total,
                count(*) FILTER (WHERE state = 'active') as active,
                count(*) FILTER (WHERE state = 'idle') as idle
            FROM pg_stat_activity
            WHERE datname = current_database()";

        using var command = new NpgsqlCommand(query, connection);
        using var reader = await command.ExecuteReaderAsync();

        if (await reader.ReadAsync())
        {
            return new ConnectionStats
            {
                Total = reader.GetInt32(0),
                Active = reader.GetInt32(1),
                Idle = reader.GetInt32(2)
            };
        }

        return new ConnectionStats();
    }

    private async Task<double> GetDatabaseSizeAsync(NpgsqlConnection connection)
    {
        var query = "SELECT pg_database_size(current_database()) / 1024.0 / 1024.0 / 1024.0";
        using var command = new NpgsqlCommand(query, connection);
        var result = await command.ExecuteScalarAsync();
        return Convert.ToDouble(result);
    }

    private async Task<List<SlowQueryInfo>> GetSlowQueriesAsync(NpgsqlConnection connection)
    {
        var queries = new List<SlowQueryInfo>();

        var query = @"
            SELECT 
                pid,
                EXTRACT(EPOCH FROM (now() - query_start))::int as duration_seconds,
                state,
                LEFT(query, 200) as query_preview
            FROM pg_stat_activity
            WHERE state != 'idle'
                AND query NOT LIKE '%pg_stat_activity%'
                AND query_start < now() - interval '10 seconds'
            ORDER BY query_start
            LIMIT 10";

        using var command = new NpgsqlCommand(query, connection);
        using var reader = await command.ExecuteReaderAsync();

        while (await reader.ReadAsync())
        {
            queries.Add(new SlowQueryInfo
            {
                ProcessId = reader.GetInt32(0),
                DurationSeconds = reader.GetInt32(1),
                State = reader.GetString(2),
                QueryPreview = reader.GetString(3)
            });
        }

        return queries;
    }

    private bool DetermineOverallHealth(ClusterHealthStatus status)
    {
        // 복제 지연이 10초 이상이면 경고
        if (status.ReplicationStatus.Any(r => r.LagSeconds > 10))
            return false;

        // 연결 수가 90% 이상이면 경고 (max_connections = 100 가정)
        if (status.ConnectionStats.Total > 90)
            return false;

        // 디스크 사용률 체크는 별도 구현 필요

        return true;
    }
}
```

**자동 장애 대응 - Standby Promotion**

Primary 서버가 다운되면 Standby를 Primary로 승격시킨다.

```csharp
public class FailoverManager
{
    private readonly ILogger<FailoverManager> _logger;
    private readonly string _standbyConnectionString;

    public FailoverManager(string standbyConnectionString, ILogger<FailoverManager> logger)
    {
        _standbyConnectionString = standbyConnectionString;
        _logger = logger;
    }

    // Primary 다운 감지 시 Standby를 Primary로 승격
    public async Task<bool> PromoteStandbyToPrimaryAsync()
    {
        _logger.LogCritical("Standby를 Primary로 승격 시작");

        try
        {
            using var connection = new NpgsqlConnection(_standbyConnectionString);
            await connection.OpenAsync();

            // Recovery 모드 종료 (Standby -> Primary)
            using var command = new NpgsqlCommand("SELECT pg_promote()", connection);
            await command.ExecuteNonQueryAsync();

            _logger.LogInformation("Standby가 Primary로 승격됨");

            // 몇 초 대기 후 확인
            await Task.Delay(5000);

            // Primary 모드 확인
            using var checkCommand = new NpgsqlCommand("SELECT pg_is_in_recovery()", connection);
            var isInRecovery = (bool)await checkCommand.ExecuteScalarAsync();

            if (isInRecovery)
            {
                _logger.LogError("승격 실패: 여전히 Recovery 모드");
                return false;
            }

            _logger.LogInformation("Failover 완료");
            return true;
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Failover 실패");
            return false;
        }
    }
}
```

**장애 시나리오별 대응**

Primary 서버 다운 시 즉시 Standby를 Primary로 승격하고, 애플리케이션 연결 문자열을 변경한다. DNS나 로드 밸런서를 사용하면 자동 전환이 가능하다.

Standby 서버 다운 시 읽기 트래픽을 다른 Standby나 Primary로 전환한다. 다운된 Standby를 재구성하여 복제를 재시작한다.

네트워크 분할(Split-brain) 시 Primary가 여러 개 생기는 것을 방지하기 위해 Quorum 기반 동기 복제를 사용한다. 또는 모니터링 시스템에서 수동 개입을 통해 하나만 Primary로 유지한다.

디스크 풀 시 오래된 데이터를 자동 삭제하거나, 디스크를 확장한다. 사전에 알림을 설정하여 80% 시점에 대응한다.
