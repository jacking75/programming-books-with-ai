# Chapter 19: 보안과 권한 관리

게임 서버의 데이터베이스에는 플레이어의 행동 패턴, 결제 정보, 게임 로그 등 민감한 정보가 저장된다. 만약 보안 설정이 제대로 되어 있지 않다면 외부 공격자가 데이터를 탈취하거나 조작할 수 있고, 내부 직원의 실수나 악의적인 행동으로 인해 데이터가 유출될 수도 있다. 실제로 게임 업계에서는 보안 사고로 인해 수백만 명의 플레이어 정보가 유출되고 회사의 신뢰가 무너지는 사례가 종종 발생한다.

TimescaleDB는 PostgreSQL을 기반으로 하기 때문에 PostgreSQL의 강력한 보안 기능을 모두 활용할 수 있다. 이 장에서는 프로덕션 환경에서 TimescaleDB를 안전하게 운영하기 위한 보안 설정, 사용자 권한 관리, 암호화, 감사 로그를 실무 중심으로 배운다. 단순히 이론만 설명하는 것이 아니라 C#과 SQLKata를 사용하는 게임 서버 환경에 직접 적용할 수 있는 구체적인 방법을 다룬다.

## 19.1 PostgreSQL 보안 기초

TimescaleDB의 보안은 PostgreSQL 보안을 기반으로 한다. 먼저 PostgreSQL의 보안 구조를 이해하고, 기본 설정에서 어떤 부분이 위험한지 파악해야 한다.

**기본 설정의 위험성**

PostgreSQL을 처음 설치하면 보안상 취약한 설정이 많다. 특히 Docker로 간단히 실행한 경우 다음과 같은 문제가 있다:

- 기본 사용자(postgres)가 모든 권한을 가짐
- 암호가 간단하거나 없는 경우도 있음
- 외부에서 접근 가능하도록 포트가 열려 있음
- SSL/TLS 암호화가 비활성화되어 있음
- 감사 로그가 기록되지 않음

개발 환경에서는 편의성을 위해 이런 설정을 사용할 수 있지만, 프로덕션 환경에서는 반드시 강화해야 한다.

**pg_hba.conf 파일 이해하기**

PostgreSQL의 접근 제어는 `pg_hba.conf` 파일에서 관리된다. HBA는 Host-Based Authentication의 약자로, 어떤 호스트에서 어떤 사용자가 어떤 데이터베이스에 접근할 수 있는지를 정의한다.

파일의 기본 구조는 다음과 같다:

```
# TYPE  DATABASE        USER            ADDRESS                 METHOD
local   all             all                                     trust
host    all             all             127.0.0.1/32            trust
host    all             all             ::1/128                 trust
host    all             all             0.0.0.0/0               md5
```

각 필드의 의미는 다음과 같다:

- **TYPE**: 연결 방식 (local은 유닉스 소켓, host는 TCP/IP)
- **DATABASE**: 접근 가능한 데이터베이스 (all은 모든 DB)
- **USER**: 접근 가능한 사용자 (all은 모든 사용자)
- **ADDRESS**: 허용할 IP 주소 또는 범위
- **METHOD**: 인증 방식 (trust는 암호 없음, md5는 암호 필요, scram-sha-256은 더 강력한 암호화)

프로덕션 환경을 위한 안전한 설정 예시는 다음과 같다:

```
# TYPE  DATABASE        USER            ADDRESS                 METHOD

# 로컬 연결은 peer 인증 (OS 사용자와 DB 사용자가 일치해야 함)
local   all             postgres                                peer

# 애플리케이션 서버만 접근 허용 (강력한 암호화 사용)
host    game_monitoring game_api        10.0.1.0/24             scram-sha-256

# 분석팀은 읽기 전용 계정으로 접근
host    game_monitoring game_analyst    10.0.2.0/24             scram-sha-256

# 백업 서버 접근
host    all             backup_user     10.0.3.10/32            scram-sha-256

# 나머지는 모두 차단
host    all             all             0.0.0.0/0               reject
```

이 설정은 다음과 같은 보안 원칙을 따른다:

- 최소 권한의 원칙: 필요한 사용자에게만 필요한 권한만 부여한다
- 네트워크 격리: 특정 IP 대역에서만 접근을 허용한다
- 강력한 인증: trust나 md5 대신 scram-sha-256을 사용한다

**postgresql.conf 보안 설정**

데이터베이스 서버의 전반적인 동작은 `postgresql.conf` 파일에서 설정한다. 보안과 관련된 주요 설정은 다음과 같다:

```ini
# 연결 설정
listen_addresses = '10.0.1.5'  # 특정 IP에서만 리스닝 (0.0.0.0은 위험)
port = 5432
max_connections = 100

# SSL/TLS 설정
ssl = on
ssl_cert_file = '/etc/ssl/certs/server.crt'
ssl_key_file = '/etc/ssl/private/server.key'
ssl_ca_file = '/etc/ssl/certs/ca.crt'
ssl_min_protocol_version = 'TLSv1.2'

# 로깅 설정
logging_collector = on
log_directory = 'log'
log_filename = 'postgresql-%Y-%m-%d_%H%M%S.log'
log_connections = on
log_disconnections = on
log_duration = on
log_line_prefix = '%t [%p]: user=%u,db=%d,app=%a,client=%h '
log_statement = 'ddl'  # DDL 문만 로깅 (모든 쿼리는 'all')

# 암호 정책
password_encryption = scram-sha-256

# 타임아웃 설정
statement_timeout = 300000  # 5분 (느린 쿼리 자동 취소)
idle_in_transaction_session_timeout = 60000  # 1분
```

이 설정을 Docker 환경에서 적용하려면 설정 파일을 마운트한다:

```bash
docker run -d --name timescaledb \
  -p 5432:5432 \
  -v /path/to/postgresql.conf:/etc/postgresql/postgresql.conf \
  -v /path/to/pg_hba.conf:/etc/postgresql/pg_hba.conf \
  -e POSTGRES_PASSWORD=strong_password_here \
  timescale/timescaledb:latest-pg15 \
  -c config_file=/etc/postgresql/postgresql.conf
```

**기본 보안 점검**

프로덕션 배포 전에 다음 사항을 반드시 확인한다:

```sql
-- 1. 기본 postgres 계정의 암호가 설정되어 있는가?
SELECT usename, valuntil 
FROM pg_user 
WHERE usename = 'postgres';

-- 2. 암호 없이 접속 가능한 사용자가 있는가?
SELECT usename 
FROM pg_shadow 
WHERE passwd IS NULL;

-- 3. 슈퍼유저 권한을 가진 계정이 몇 개인가? (최소화해야 함)
SELECT usename, usesuper 
FROM pg_user 
WHERE usesuper = true;

-- 4. 공개 스키마에 모든 사용자가 쓰기 권한을 가지는가?
SELECT nspname, nspacl 
FROM pg_namespace 
WHERE nspname = 'public';
```

## 19.2 사용자 및 역할 관리

게임 서버 환경에서는 다양한 역할을 가진 사용자들이 데이터베이스에 접근한다. 게임 서버 애플리케이션, 분석팀, 운영팀, 백업 시스템 등이 각각 다른 권한을 필요로 한다. PostgreSQL의 역할(Role) 시스템을 활용하여 세밀하게 권한을 관리할 수 있다.

**역할 기반 권한 설계**

게임 서버를 위한 전형적인 역할 구조는 다음과 같다:

```
┌─────────────────────────────────────────────────┐
│                   postgres                      │
│              (슈퍼유저, 긴급 상황만)              │
└─────────────────────────────────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
┌───────▼──────┐ ┌────▼─────┐ ┌──────▼──────┐
│ game_api     │ │ analyst  │ │ backup_user │
│ (읽기/쓰기)   │ │ (읽기만) │ │ (백업만)    │
└──────────────┘ └──────────┘ └─────────────┘
```

**기본 역할 생성**

먼저 세 가지 주요 역할을 생성한다. 여기서 중요한 것은 각 역할이 필요한 최소한의 권한만 가져야 한다는 점이다.

```sql
-- 1. 애플리케이션 역할 (게임 서버가 사용)
CREATE ROLE game_api_role;

-- 필요한 권한만 부여
GRANT CONNECT ON DATABASE game_monitoring TO game_api_role;
GRANT USAGE ON SCHEMA public TO game_api_role;
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO game_api_role;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO game_api_role;

-- 향후 생성될 테이블에도 자동으로 권한 부여
ALTER DEFAULT PRIVILEGES IN SCHEMA public 
  GRANT SELECT, INSERT, UPDATE ON TABLES TO game_api_role;

-- 2. 분석가 역할 (읽기 전용)
CREATE ROLE analyst_role;

GRANT CONNECT ON DATABASE game_monitoring TO analyst_role;
GRANT USAGE ON SCHEMA public TO analyst_role;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO analyst_role;

ALTER DEFAULT PRIVILEGES IN SCHEMA public 
  GRANT SELECT ON TABLES TO analyst_role;

-- 3. 백업 역할 (모든 데이터 읽기)
CREATE ROLE backup_role;

GRANT CONNECT ON DATABASE game_monitoring TO backup_role;
GRANT USAGE ON SCHEMA public TO backup_role;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO backup_role;
-- pg_dump를 위한 추가 권한
GRANT SELECT ON ALL SEQUENCES IN SCHEMA public TO backup_role;
```

**개별 사용자 계정 생성**

역할을 정의했으면 실제 사용자 계정을 만들고 역할을 할당한다. 사용자 계정은 로그인이 가능하며 암호를 가진다.

```sql
-- 게임 서버 애플리케이션 계정
CREATE USER game_api WITH PASSWORD 'very_strong_password_here_128bit';
GRANT game_api_role TO game_api;

-- 만료 기한 설정 (선택사항)
ALTER USER game_api VALID UNTIL '2025-12-31';

-- 동시 연결 수 제한
ALTER USER game_api CONNECTION LIMIT 50;

-- 분석팀 계정들
CREATE USER analyst_john WITH PASSWORD 'another_strong_password';
GRANT analyst_role TO analyst_john;
ALTER USER analyst_john CONNECTION LIMIT 5;

CREATE USER analyst_sarah WITH PASSWORD 'yet_another_password';
GRANT analyst_role TO analyst_sarah;
ALTER USER analyst_sarah CONNECTION LIMIT 5;

-- 백업 시스템 계정
CREATE USER backup_service WITH PASSWORD 'backup_password';
GRANT backup_role TO backup_service;
ALTER USER backup_service CONNECTION LIMIT 2;
```

**테이블 수준 권한 관리**

특정 테이블에 대해서만 선택적으로 권한을 부여할 수도 있다. 예를 들어 결제 정보가 포함된 테이블은 일반 분석가에게 접근을 제한할 수 있다.

```sql
-- 민감한 결제 테이블 생성
CREATE TABLE payment_logs (
    timestamp TIMESTAMPTZ NOT NULL,
    player_id VARCHAR(50),
    amount DECIMAL(10,2),
    currency VARCHAR(3),
    payment_method VARCHAR(50),
    transaction_id VARCHAR(100)
);

SELECT create_hypertable('payment_logs', 'timestamp');

-- 기본적으로 analyst_role은 접근 불가
REVOKE SELECT ON payment_logs FROM analyst_role;

-- 특정 분석가만 접근 가능하도록 별도 역할 생성
CREATE ROLE payment_analyst_role;
GRANT SELECT ON payment_logs TO payment_analyst_role;

-- 시니어 분석가에게만 부여
GRANT payment_analyst_role TO analyst_sarah;
```

**행 수준 보안 (Row-Level Security)**

PostgreSQL의 행 수준 보안 기능을 사용하면 같은 테이블 내에서도 사용자별로 볼 수 있는 행을 제한할 수 있다. 예를 들어 각 지역 관리자는 자신의 지역 데이터만 볼 수 있도록 설정할 수 있다.

```sql
-- 서버 메트릭 테이블에 RLS 활성화
ALTER TABLE server_metrics ENABLE ROW LEVEL SECURITY;

-- 정책 생성: 각 사용자는 자신이 담당하는 서버의 데이터만 볼 수 있음
CREATE POLICY server_access_policy ON server_metrics
    FOR SELECT
    TO analyst_role
    USING (server_id = current_setting('app.current_server_id', true));

-- 사용 예: 연결 시 서버 ID 설정
-- SET app.current_server_id = 'game-server-01';
-- 이후 SELECT 쿼리는 자동으로 해당 서버 데이터만 반환
```

C# 애플리케이션에서 RLS를 활용하는 방법은 다음과 같다:

```csharp
public class SecureQueryService
{
    private readonly NpgsqlConnection _connection;
    private readonly QueryFactory _db;

    public SecureQueryService(string connectionString)
    {
        _connection = new NpgsqlConnection(connectionString);
        _connection.Open();

        var compiler = new PostgresCompiler();
        _db = new QueryFactory(_connection, compiler);
    }

    public async Task<List<ServerMetric>> GetServerMetrics(string userId, string serverId)
    {
        // RLS 컨텍스트 설정
        await _connection.ExecuteAsync(
            $"SET app.current_server_id = '{serverId}'");

        // 이제 쿼리는 자동으로 필터링됨
        var query = new Query("server_metrics")
            .Select("*")
            .WhereRaw("timestamp > NOW() - INTERVAL '1 hour'")
            .OrderByDesc("timestamp")
            .Limit(100);

        var result = await _db.GetAsync<ServerMetric>(query);
        return result.ToList();
    }
}
```

**권한 확인 및 감사**

현재 권한 설정을 확인하는 유틸리티 쿼리를 작성한다:

```sql
-- 모든 사용자와 역할 목록
SELECT 
    r.rolname AS role_name,
    r.rolsuper AS is_superuser,
    r.rolcanlogin AS can_login,
    r.rolconnlimit AS connection_limit,
    r.rolvaliduntil AS valid_until,
    ARRAY(
        SELECT b.rolname 
        FROM pg_catalog.pg_auth_members m 
        JOIN pg_catalog.pg_roles b ON (m.roleid = b.oid) 
        WHERE m.member = r.oid
    ) AS member_of
FROM pg_catalog.pg_roles r
ORDER BY r.rolname;

-- 특정 테이블에 대한 권한 확인
SELECT 
    grantee,
    privilege_type,
    is_grantable
FROM information_schema.table_privileges
WHERE table_name = 'server_metrics'
ORDER BY grantee, privilege_type;

-- 특정 사용자가 가진 모든 권한 확인
SELECT 
    schemaname,
    tablename,
    usename,
    HAS_TABLE_PRIVILEGE(usename, schemaname||'.'||tablename, 'SELECT') AS can_select,
    HAS_TABLE_PRIVILEGE(usename, schemaname||'.'||tablename, 'INSERT') AS can_insert,
    HAS_TABLE_PRIVILEGE(usename, schemaname||'.'||tablename, 'UPDATE') AS can_update,
    HAS_TABLE_PRIVILEGE(usename, schemaname||'.'||tablename, 'DELETE') AS can_delete
FROM pg_tables, pg_user
WHERE schemaname = 'public'
  AND usename = 'analyst_john'
ORDER BY tablename;
```

## 19.3 연결 암호화 (SSL/TLS)

데이터베이스와 애플리케이션 서버 간의 통신은 반드시 암호화해야 한다. 암호화하지 않으면 네트워크를 통해 전송되는 SQL 쿼리, 결과 데이터, 심지어 사용자 암호까지 평문으로 노출된다. 게임 서버가 클라우드 환경에서 동작하거나 여러 서버 간 통신이 발생하는 경우 특히 중요하다.

**SSL 인증서 생성**

먼저 자체 서명된(self-signed) SSL 인증서를 만든다. 프로덕션 환경에서는 Let's Encrypt 같은 인증 기관에서 발급받은 인증서를 사용하는 것이 좋지만, 내부 네트워크에서는 자체 서명 인증서도 충분하다.

Windows 11에서 OpenSSL을 사용하여 인증서를 생성한다. OpenSSL이 없다면 Git for Windows를 설치하면 함께 설치된다.

```bash
# 1. CA(Certificate Authority) 키와 인증서 생성
openssl req -new -x509 -days 365 -nodes -text \
  -out ca.crt \
  -keyout ca.key \
  -subj "/CN=Game Server CA"

# 2. 서버 키 생성
openssl genrsa -out server.key 2048
chmod 600 server.key

# 3. 서버 인증서 요청 생성
openssl req -new -key server.key -out server.csr \
  -subj "/CN=timescale.gameserver.local"

# 4. CA로 서버 인증서 서명
openssl x509 -req -in server.csr -text -days 365 \
  -CA ca.crt -CAkey ca.key -CAcreateserial \
  -out server.crt

# 5. 클라이언트 인증서 생성 (양방향 인증용, 선택사항)
openssl genrsa -out client.key 2048
chmod 600 client.key

openssl req -new -key client.key -out client.csr \
  -subj "/CN=game-api-client"

openssl x509 -req -in client.csr -text -days 365 \
  -CA ca.crt -CAkey ca.key -CAcreateserial \
  -out client.crt
```

**TimescaleDB에 SSL 설정**

생성한 인증서를 Docker 컨테이너에 마운트하고 SSL을 활성화한다.

```bash
# 인증서 파일이 있는 디렉토리 구조
# ./certs/
#   ├── ca.crt
#   ├── server.crt
#   └── server.key

docker run -d --name timescaledb-secure \
  -p 5432:5432 \
  -v $(pwd)/certs:/etc/ssl/certs/postgres \
  -e POSTGRES_PASSWORD=strong_password \
  timescale/timescaledb:latest-pg15 \
  -c ssl=on \
  -c ssl_cert_file=/etc/ssl/certs/postgres/server.crt \
  -c ssl_key_file=/etc/ssl/certs/postgres/server.key \
  -c ssl_ca_file=/etc/ssl/certs/postgres/ca.crt \
  -c ssl_min_protocol_version=TLSv1.2
```

SSL이 제대로 활성화되었는지 확인한다:

```sql
-- SSL 상태 확인
SHOW ssl;

-- 현재 연결의 SSL 정보
SELECT 
    datname,
    usename,
    client_addr,
    ssl,
    version AS ssl_version,
    cipher AS ssl_cipher
FROM pg_stat_ssl
JOIN pg_stat_activity ON pg_stat_ssl.pid = pg_stat_activity.pid
WHERE ssl = true;
```

**C#에서 SSL 연결 사용하기**

**SSL 모드 비교**

Npgsql의 SslMode 옵션은 보안 수준에 따라 선택할 수 있다:

- **Disable**: SSL 사용 안 함 (절대 프로덕션에서 사용 금지)
- **Allow**: 서버가 지원하면 SSL 사용, 아니면 평문 (비권장)
- **Prefer**: 가능하면 SSL, 실패 시 평문 (비권장)
- **Require**: SSL 필수, 인증서 검증 안 함 (중간 수준)
- **VerifyCA**: SSL 필수, CA 인증서 검증 (권장)
- **VerifyFull**: SSL 필수, CA와 호스트명 검증 (가장 안전, 프로덕션 권장)

## 19.4 컬럼 수준 암호화

SSL/TLS는 전송 중인 데이터를 보호하지만, 데이터베이스에 저장된 데이터 자체는 암호화하지 않는다. 만약 디스크가 물리적으로 탈취되거나 백업 파일이 유출되면 데이터가 그대로 노출된다. 특히 플레이어의 개인정보나 결제 정보 같은 민감한 데이터는 데이터베이스 내에서도 암호화해야 한다.

**pgcrypto 확장 설치**

PostgreSQL의 pgcrypto 확장은 다양한 암호화 함수를 제공한다.

```sql
-- pgcrypto 확장 설치
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- 사용 가능한 암호화 함수 확인
\df+ encrypt
\df+ decrypt
\df+ digest
```

**대칭키 암호화**

대칭키 암호화는 같은 키로 암호화하고 복호화한다. 빠르지만 키 관리가 중요하다.

```sql
-- 테스트 테이블 생성
CREATE TABLE player_sensitive_data (
    player_id VARCHAR(50) PRIMARY KEY,
    email_encrypted BYTEA,  -- 암호화된 이메일
    phone_encrypted BYTEA,  -- 암호화된 전화번호
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 데이터 삽입 (암호화)
INSERT INTO player_sensitive_data (player_id, email_encrypted, phone_encrypted)
VALUES (
    'player123',
    pgp_sym_encrypt('player123@email.com', 'my_secret_key_change_this'),
    pgp_sym_encrypt('+82-10-1234-5678', 'my_secret_key_change_this')
);

-- 데이터 조회 (복호화)
SELECT 
    player_id,
    pgp_sym_decrypt(email_encrypted, 'my_secret_key_change_this') AS email,
    pgp_sym_decrypt(phone_encrypted, 'my_secret_key_change_this') AS phone
FROM player_sensitive_data
WHERE player_id = 'player123';
```

**C#에서 암호화된 데이터 처리**

C# 애플리케이션에서 암호화 키를 안전하게 관리하고 사용하는 방법이다. 절대로 소스 코드에 키를 하드코딩하면 안 되며, 환경 변수나 Azure Key Vault 같은 비밀 관리 서비스를 사용해야 한다.

```csharp
public class PlayerSensitiveDataService
{
    private readonly QueryFactory _db;
    private readonly string _encryptionKey;

    public PlayerSensitiveDataService(QueryFactory db, IConfiguration config)
    {
        _db = db;
        // 환경 변수나 Key Vault에서 키 로드
        _encryptionKey = config["Database:EncryptionKey"] 
            ?? throw new Exception("Encryption key not configured");
    }

    // 민감한 데이터 저장
    public async Task SavePlayerSensitiveData(string playerId, string email, string phone)
    {
        // SQLKata는 암호화 함수를 직접 지원하지 않으므로 원시 쿼리 사용
        var sql = @"
            INSERT INTO player_sensitive_data (player_id, email_encrypted, phone_encrypted)
            VALUES (@playerId, 
                    pgp_sym_encrypt(@email, @key),
                    pgp_sym_encrypt(@phone, @key))
            ON CONFLICT (player_id) DO UPDATE
            SET email_encrypted = pgp_sym_encrypt(@email, @key),
                phone_encrypted = pgp_sym_encrypt(@phone, @key)";

        await _db.StatementAsync(sql, new
        {
            playerId,
            email,
            phone,
            key = _encryptionKey
        });

        Console.WriteLine($"Saved encrypted data for player {playerId}");
    }

    // 민감한 데이터 조회
    public async Task<PlayerSensitiveInfo> GetPlayerSensitiveData(string playerId)
    {
        var sql = @"
            SELECT 
                player_id,
                pgp_sym_decrypt(email_encrypted, @key) AS email,
                pgp_sym_decrypt(phone_encrypted, @key) AS phone
            FROM player_sensitive_data
            WHERE player_id = @playerId";

        var result = await _db.FirstOrDefaultAsync<PlayerSensitiveInfo>(sql, new
        {
            playerId,
            key = _encryptionKey
        });

        if (result == null)
        {
            throw new KeyNotFoundException($"Player {playerId} not found");
        }

        return result;
    }

    // 암호화된 데이터 검색 (이메일로 플레이어 찾기)
    // 주의: 암호화된 컬럼은 인덱스를 사용할 수 없어 성능이 떨어짐
    public async Task<string> FindPlayerByEmail(string email)
    {
        var sql = @"
            SELECT player_id
            FROM player_sensitive_data
            WHERE pgp_sym_decrypt(email_encrypted, @key) = @email";

        var result = await _db.FirstOrDefaultAsync<string>(sql, new
        {
            email,
            key = _encryptionKey
        });

        return result;
    }
}

{
    public string PlayerId { get; set; }
    public string Email { get; set; }
    public string Phone { get; set; }
}
```

**해시 함수를 활용한 검색 가능 암호화**

암호화된 컬럼은 인덱스를 사용할 수 없어 검색 성능이 떨어진다. 이를 해결하기 위해 암호화된 값과 함께 해시값도 저장하여 검색에 활용할 수 있다.

```sql
-- 해시 컬럼 추가
ALTER TABLE player_sensitive_data
ADD COLUMN email_hash VARCHAR(64);

-- 해시값 생성 (SHA-256)
UPDATE player_sensitive_data
SET email_hash = encode(digest(
    pgp_sym_decrypt(email_encrypted, 'my_secret_key_change_this'), 
    'sha256'
), 'hex');

-- 해시 컬럼에 인덱스 생성
CREATE INDEX idx_player_email_hash ON player_sensitive_data(email_hash);

-- 이메일로 빠르게 검색 (해시 비교)
SELECT player_id
FROM player_sensitive_data
WHERE email_hash = encode(digest('player123@email.com', 'sha256'), 'hex');
```

C#에서 구현하면 다음과 같다:

```csharp
public class SearchableEncryptionService
{
    private readonly QueryFactory _db;
    private readonly string _encryptionKey;

    // 해시 생성 헬퍼 메서드
    private string ComputeSHA256Hash(string input)
    {
        using var sha256 = System.Security.Cryptography.SHA256.Create();
        var bytes = Encoding.UTF8.GetBytes(input);
        var hashBytes = sha256.ComputeHash(bytes);
        return BitConverter.ToString(hashBytes).Replace("-", "").ToLower();
    }

    // 검색 가능한 암호화 데이터 저장
    public async Task SaveWithSearchableEncryption(string playerId, string email, string phone)
    {
        var emailHash = ComputeSHA256Hash(email);

        var sql = @"
            INSERT INTO player_sensitive_data 
                (player_id, email_encrypted, phone_encrypted, email_hash)
            VALUES (
                @playerId,
                pgp_sym_encrypt(@email, @key),
                pgp_sym_encrypt(@phone, @key),
                @emailHash
            )";

        await _db.StatementAsync(sql, new
        {
            playerId,
            email,
            phone,
            emailHash,
            key = _encryptionKey
        });
    }

    // 해시를 이용한 빠른 검색
    public async Task<string> FindPlayerByEmailFast(string email)
    {
        var emailHash = ComputeSHA256Hash(email);

        var query = new Query("player_sensitive_data")
            .Select("player_id")
            .Where("email_hash", emailHash);

        var result = await _db.FirstOrDefaultAsync<string>(query);
        return result;
    }
}
```

**암호화 키 로테이션**

보안을 위해 주기적으로 암호화 키를 변경해야 한다. 키 로테이션 프로세스는 다음과 같다:

```csharp
public class EncryptionKeyRotationService
{
    private readonly QueryFactory _db;

    public async Task RotateEncryptionKey(string oldKey, string newKey)
    {
        Console.WriteLine("Starting encryption key rotation...");

        // 트랜잭션으로 안전하게 처리
        var connection = _db.Connection as NpgsqlConnection;
        using var transaction = await connection.BeginTransactionAsync();

        try
        {
            // 1. 모든 암호화된 데이터 조회
            var sql = @"
                SELECT player_id, email_encrypted, phone_encrypted
                FROM player_sensitive_data
                FOR UPDATE";  -- 행 잠금

            var rows = await _db.GetAsync<dynamic>(sql);

            // 2. 각 행을 복호화 후 새 키로 재암호화
            foreach (var row in rows)
            {
                var updateSql = @"
                    UPDATE player_sensitive_data
                    SET email_encrypted = pgp_sym_encrypt(
                            pgp_sym_decrypt(email_encrypted, @oldKey), @newKey),
                        phone_encrypted = pgp_sym_encrypt(
                            pgp_sym_decrypt(phone_encrypted, @oldKey), @newKey)
                    WHERE player_id = @playerId";

                await _db.StatementAsync(updateSql, new
                {
                    playerId = row.player_id,
                    oldKey,
                    newKey
                });
            }

            await transaction.CommitAsync();
            Console.WriteLine($"Successfully rotated {rows.Count()} records");
        }
        catch (Exception ex)
        {
            await transaction.RollbackAsync();
            Console.WriteLine($"Key rotation failed: {ex.Message}");
            throw;
        }
    }
}
```

## 19.5 감사 로그 설정

보안 사고가 발생했을 때 누가 언제 어떤 데이터에 접근했는지 추적할 수 있어야 한다. 또한 규제 준수를 위해 데이터 접근 로그를 일정 기간 보관해야 하는 경우도 많다. PostgreSQL의 로깅 기능과 pgAudit 확장을 활용하여 포괄적인 감사 로그 시스템을 구축할 수 있다.

**pgAudit 확장 설치**

pgAudit는 PostgreSQL의 감사 로깅을 강화하는 확장이다. Docker 이미지에 따라서는 이미 포함되어 있을 수 있다.

```sql
-- pgAudit 확장 설치
CREATE EXTENSION IF NOT EXISTS pgaudit;

-- 감사 로그 설정 확인
SHOW pgaudit.log;
```

**감사 로그 설정**

postgresql.conf에 pgAudit 설정을 추가한다:

```ini
# pgAudit 로드
shared_preload_libraries = 'timescaledb,pgaudit'

# 감사할 작업 유형
pgaudit.log = 'read,write,ddl,role'

# 감사 로그 레벨
pgaudit.log_level = 'log'

# 세션별 감사 (모든 SQL 문)
pgaudit.log_client = on

# 카탈로그 테이블 제외 (시스템 테이블 조회는 로깅 안 함)
pgaudit.log_catalog = off

# 파라미터 값 포함
pgaudit.log_parameter = on
```

설정 후 PostgreSQL을 재시작하면 감사 로그가 기록되기 시작한다.

**감사 로그 테이블 생성**

로그 파일만으로는 분석이 어려우므로 별도의 감사 로그 테이블을 만들어 구조화된 데이터로 저장한다.

```sql
-- 감사 로그 테이블
CREATE TABLE audit_log (
    log_id BIGSERIAL,
    log_time TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    user_name VARCHAR(100),
    database_name VARCHAR(100),
    client_addr INET,
    application_name VARCHAR(200),
    operation VARCHAR(20),  -- SELECT, INSERT, UPDATE, DELETE
    table_name VARCHAR(200),
    query_text TEXT,
    parameters TEXT,
    row_count INTEGER,
    duration_ms INTEGER,
    success BOOLEAN,
    error_message TEXT
);

-- Hypertable로 변환
SELECT create_hypertable('audit_log', 'log_time');

-- 인덱스 생성
CREATE INDEX idx_audit_user ON audit_log(user_name, log_time DESC);
CREATE INDEX idx_audit_table ON audit_log(table_name, log_time DESC);
CREATE INDEX idx_audit_operation ON audit_log(operation, log_time DESC);

-- 압축 및 보관 정책
ALTER TABLE audit_log SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'user_name, operation'

SELECT add_compression_policy('audit_log', INTERVAL '7 days');
SELECT add_retention_policy('audit_log', INTERVAL '1 year');
```

**C#에서 감사 로그 기록**

애플리케이션 레벨에서 감사 로그를 기록하는 서비스를 만든다:

```csharp
public class AuditService
{
    private readonly QueryFactory _db;
    private readonly IHttpContextAccessor _httpContext;

    public AuditService(QueryFactory db, IHttpContextAccessor httpContext)
    {
        _db = db;
        _httpContext = httpContext;
    }

    // 감사 로그 기록
    public async Task LogAudit(
        string operation,
        string tableName,
        string queryText,
        object parameters = null,
        int? rowCount = null,
        long? durationMs = null,
        bool success = true,
        string errorMessage = null)
    {
        var httpContext = _httpContext.HttpContext;
        var userName = httpContext?.User?.Identity?.Name ?? "anonymous";
        var clientIp = httpContext?.Connection?.RemoteIpAddress?.ToString() ?? "unknown";

        await _db.Query("audit_log").InsertAsync(new
        {
            user_name = userName,
            database_name = "game_monitoring",
            client_addr = clientIp,
            application_name = "GameServer",
            operation,
            table_name = tableName,
            query_text = queryText,
            parameters = parameters != null ? JsonSerializer.Serialize(parameters) : null,
            row_count = rowCount,
            duration_ms = durationMs,
            success,
            error_message = errorMessage
        });
    }

    // 읽기 작업 감사
    public async Task<T> AuditedQuery<T>(
        Func<Task<T>> queryFunc,
        string tableName,
        string description)
    {
        var stopwatch = Stopwatch.StartNew();
        try
        {
            var result = await queryFunc();
            stopwatch.Stop();

            await LogAudit(
                operation: "SELECT",
                tableName: tableName,
                queryText: description,
                durationMs: stopwatch.ElapsedMilliseconds,
                success: true
            );

            return result;
        }
        catch (Exception ex)
        {
            stopwatch.Stop();

            await LogAudit(
                operation: "SELECT",
                tableName: tableName,
                queryText: description,
                durationMs: stopwatch.ElapsedMilliseconds,
                success: false,
                errorMessage: ex.Message
            );

            throw;
        }
    }
}
```

**감사 로그 분석 쿼리**

감사 로그를 분석하여 보안 이상 징후를 탐지할 수 있다:

```sql
-- 1. 가장 많이 조회되는 테이블 (지난 24시간)
SELECT 
    table_name,
    COUNT(*) AS access_count,
    COUNT(DISTINCT user_name) AS unique_users
FROM audit_log
WHERE log_time > NOW() - INTERVAL '24 hours'
  AND operation = 'SELECT'
GROUP BY table_name
ORDER BY access_count DESC
LIMIT 10;

-- 2. 비정상적으로 많은 쿼리를 실행한 사용자
SELECT 
    user_name,
    COUNT(*) AS query_count,
    COUNT(*) FILTER (WHERE success = false) AS failed_count,
    AVG(duration_ms) AS avg_duration_ms
FROM audit_log
WHERE log_time > NOW() - INTERVAL '1 hour'
GROUP BY user_name
HAVING COUNT(*) > 1000  -- 1시간에 1000회 이상
ORDER BY query_count DESC;

-- 3. 실패한 쿼리 패턴 분석
SELECT 
    user_name,
    operation,
    table_name,
    error_message,
    COUNT(*) AS occurrence
FROM audit_log
WHERE success = false
  AND log_time > NOW() - INTERVAL '7 days'
GROUP BY user_name, operation, table_name, error_message
ORDER BY occurrence DESC
LIMIT 20;

-- 4. 민감한 테이블 접근 로그
SELECT 
    log_time,
    user_name,
    client_addr,
    operation,
    query_text
FROM audit_log
WHERE table_name IN ('payment_logs', 'player_sensitive_data')
  AND log_time > NOW() - INTERVAL '30 days'
ORDER BY log_time DESC;

-- 5. 야간 시간대 이상 접근 탐지 (한국 시간 기준)
SELECT 
    log_time AT TIME ZONE 'Asia/Seoul' AS kst_time,
    user_name,
    client_addr,
    operation,
    table_name,
    query_text
FROM audit_log
WHERE EXTRACT(HOUR FROM log_time AT TIME ZONE 'Asia/Seoul') BETWEEN 0 AND 5
  AND user_name NOT IN ('backup_service', 'monitoring_system')
  AND log_time > NOW() - INTERVAL '7 days'
ORDER BY log_time DESC;
```

## 19.6 보안 체크리스트

프로덕션 환경에 TimescaleDB를 배포하기 전에 다음 체크리스트를 확인한다:

**네트워크 보안**

```
□ 방화벽에서 5432 포트를 필요한 IP만 허용했는가?
□ pg_hba.conf에서 0.0.0.0/0 (모든 IP 허용)을 제거했는가?
□ 클라우드 환경이라면 Security Group이 제대로 설정되었는가?
□ VPN이나 Private Network를 통해서만 접근 가능한가?
```

**인증 및 권한**

```
□ postgres 계정의 강력한 암호를 설정했는가?
□ 암호 없이 접속 가능한 사용자가 있는가? (pg_shadow 확인)
□ 애플리케이션은 전용 계정을 사용하는가? (postgres 계정 사용 금지)
□ 각 사용자는 필요한 최소 권한만 가지는가?
□ 슈퍼유저 권한을 가진 계정이 몇 개인가? (최소화)
□ 사용자 암호 만료 정책이 설정되어 있는가?
□ scram-sha-256 인증을 사용하는가? (md5나 trust는 위험)
```

**암호화**

```
□ SSL/TLS가 활성화되어 있는가?
□ 자체 서명 인증서를 사용한다면 CA 인증서가 클라이언트에 배포되었는가?
□ 프로덕션에서는 VerifyFull SSL 모드를 사용하는가?
□ 민감한 데이터(개인정보, 결제정보)가 암호화되어 저장되는가?
□ 암호화 키가 소스 코드에 하드코딩되어 있지 않은가?
□ 암호화 키가 별도의 Key Management Service에 저장되어 있는가?
```

**감사 및 모니터링**

```
□ pgAudit 또는 로깅이 활성화되어 있는가?
□ DDL 문(CREATE, DROP, ALTER)이 로깅되는가?
□ 감사 로그가 정기적으로 검토되는가?
□ 비정상적인 접근 패턴을 자동으로 탐지하는가?
□ 로그인 실패가 로깅되는가?
□ 감사 로그가 일정 기간 보관되는가? (규제 준수)
```

**백업 및 복구**

```
□ 백업이 암호화되어 저장되는가?
□ 백업 파일에 대한 접근 권한이 제한되어 있는가?
□ 복구 테스트를 정기적으로 수행하는가?
□ 백업 데이터가 안전한 오프사이트에 저장되는가?
```

**운영 정책**

```
□ 데이터베이스 접근 정책 문서가 작성되어 있는가?
□ 직원 퇴사 시 계정을 즉시 비활성화하는 프로세스가 있는가?
□ 프로덕션 DB에 직접 접근하는 인원이 제한되어 있는가?
□ 변경 작업은 Change Management 프로세스를 따르는가?
□ 보안 사고 대응 계획이 수립되어 있는가?
```

**보안 점검 자동화 스크립트**

정기적으로 실행할 수 있는 보안 점검 스크립트를 C#으로 작성한다:

```csharp
public class SecurityCheckService
{
    private readonly QueryFactory _db;

    public async Task<SecurityCheckReport> RunSecurityCheck()
    {
        var report = new SecurityCheckReport();

        // 1. 암호 없는 사용자 확인
        var noPasswordUsers = await _db.Query("pg_shadow")
            .SelectRaw("usename")
            .WhereNull("passwd")
            .GetAsync<string>();

        report.UsersWithoutPassword = noPasswordUsers.ToList();

        // 2. 슈퍼유저 확인
        var superusers = await _db.Query("pg_user")
            .Select("usename")
            .Where("usesuper", true)
            .GetAsync<string>();

        report.SuperUsers = superusers.ToList();

        // 3. SSL 연결 비율
        var sslStats = await _db.StatementAsync(@"
            SELECT 
                COUNT(*) FILTER (WHERE ssl = true) AS ssl_connections,
                COUNT(*) AS total_connections,
                (COUNT(*) FILTER (WHERE ssl = true) * 100.0 / COUNT(*)) AS ssl_percentage
            FROM pg_stat_ssl
            JOIN pg_stat_activity USING (pid)
            WHERE state = 'active'");

        // 4. 최근 로그인 실패 확인 (pgAudit 로그 필요)
        var recentFailures = await _db.Query("audit_log")
            .Where("success", false)
            .Where("operation", "LOGIN")
            .Where("log_time", ">", DateTime.UtcNow.AddHours(-24))
            .SelectRaw("COUNT(*) AS failure_count")
            .FirstAsync<int>();

        report.LoginFailuresLast24Hours = recentFailures;

        // 5. 공개 스키마 권한 확인
        var publicSchemaPrivileges = await _db.StatementAsync(@"
            SELECT 
                grantee,
                privilege_type
            FROM information_schema.schema_privileges
            WHERE schema_name = 'public'
            ORDER BY grantee, privilege_type");

        // 결과 출력
        Console.WriteLine("=== Security Check Report ===");
        Console.WriteLine($"Users without password: {report.UsersWithoutPassword.Count}");
        if (report.UsersWithoutPassword.Any())
        {
            Console.WriteLine($"  - {string.Join(", ", report.UsersWithoutPassword)}");
        }

        Console.WriteLine($"Superusers: {report.SuperUsers.Count}");
        Console.WriteLine($"  - {string.Join(", ", report.SuperUsers)}");

        Console.WriteLine($"Login failures (24h): {report.LoginFailuresLast24Hours}");

        return report;
    }
}

public class SecurityCheckReport
{
    public List<string> UsersWithoutPassword { get; set; } = new();
    public List<string> SuperUsers { get; set; } = new();
    public int LoginFailuresLast24Hours { get; set; }
    public double SslConnectionPercentage { get; set; }
}
```

**침투 테스트 시나리오**

보안 설정이 제대로 되어 있는지 확인하기 위한 테스트 시나리오다:

```csharp
public class SecurityPenetrationTest
{
    // 1. 약한 암호로 로그인 시도
    public async Task TestWeakPasswordAttempt()
    {
        var weakPasswords = new[] { "password", "123456", "admin", "postgres" };

        foreach (var password in weakPasswords)
        {
            try
            {
                var connectionString = new NpgsqlConnectionStringBuilder
                {
                    Host = "production-db.server",
                    Username = "postgres",
                    Password = password,
                    Database = "game_monitoring"
                }.ToString();

                using var connection = new NpgsqlConnection(connectionString);
                await connection.OpenAsync();

                Console.WriteLine($"⚠️  SECURITY ISSUE: Weak password '{password}' worked!");
            }
            catch (NpgsqlException)
            {
                Console.WriteLine($"✓ Weak password '{password}' rejected");
            }
        }
    }

    // 2. SQL Injection 시도
    public async Task TestSqlInjection(QueryFactory db)
    {
        var maliciousInputs = new[]
        {
            "' OR '1'='1",
            "'; DROP TABLE players--",
            "' UNION SELECT * FROM pg_user--"
        };

        foreach (var input in maliciousInputs)
        {
            try
            {
                // SQLKata는 파라미터화된 쿼리를 사용하므로 안전해야 함
                var query = new Query("players")
                    .Where("player_id", input);

                var result = await db.GetAsync<dynamic>(query);

                Console.WriteLine($"✓ SQL injection attempt blocked: {input}");
            }
            catch (Exception ex)
            {
                Console.WriteLine($"⚠️  Unexpected error: {ex.Message}");
            }
        }
    }

    // 3. 무단 테이블 접근 시도
    public async Task TestUnauthorizedTableAccess(QueryFactory db)
    {
        try
        {
            var sensitiveQuery = new Query("payment_logs")
                .Select("*")
                .Limit(1);

            var result = await db.GetAsync<dynamic>(sensitiveQuery);

            Console.WriteLine("⚠️  SECURITY ISSUE: Unauthorized access to payment_logs!");
        }
        catch (NpgsqlException ex) when (ex.Message.Contains("permission denied"))
        {
            Console.WriteLine("✓ Unauthorized table access blocked");
        }
    }
}
```

## 정리

이 장에서는 프로덕션 환경에서 TimescaleDB를 안전하게 운영하기 위한 포괄적인 보안 설정을 다루었다. PostgreSQL의 강력한 보안 기능을 기반으로 네트워크 격리, 사용자 권한 관리, SSL/TLS 암호화, 컬럼 수준 암호화, 감사 로그까지 다층 방어(Defense in Depth) 전략을 구축했다.

보안은 한 번 설정하고 끝나는 것이 아니라 지속적으로 모니터링하고 개선해야 하는 프로세스다. 정기적으로 보안 점검을 수행하고, 감사 로그를 분석하며, 새로운 위협에 대응해야 한다. 또한 팀 전체가 보안의 중요성을 인식하고 보안 정책을 준수하도록 교육하는 것이 중요하다.

게임 서비스의 성공은 플레이어의 신뢰에 달려 있다. 플레이어 데이터를 안전하게 보호하는 것은 기술적인 문제를 넘어서 비즈니스의 생존이 걸린 문제다. 이 장에서 배운 보안 원칙과 실무 기법을 적용하여 안전하고 신뢰할 수 있는 게임 서버 인프라를 구축하기 바란다.
