# 11장. 이미지 최적화

Docker 이미지를 처음 만들면 크기가 수백 MB에서 GB까지 커질 수 있다. 이미지가 크면 빌드 시간이 오래 걸리고, 배포할 때도 시간이 오래 걸린다. 또한 디스크 공간도 많이 차지하고 보안 취약점이 포함될 가능성도 높아진다. 이 장에서는 Docker 이미지를 최적화하여 크기를 줄이고 빌드 속도를 높이며 보안을 강화하는 방법을 실습한다.

## 11.1 멀티 스테이지 빌드

### 문제점 파악하기

먼저 최적화하지 않은 Dockerfile로 이미지를 빌드해본다.

```dockerfile
# Dockerfile.before (최적화 전)
FROM mcr.microsoft.com/dotnet/sdk:8.0
WORKDIR /app

# 소스 코드 복사
COPY . .

# 빌드
RUN dotnet restore
RUN dotnet build -c Release
RUN dotnet publish -c Release -o /app/publish

# 실행
EXPOSE 5000
WORKDIR /app/publish
ENTRYPOINT ["dotnet", "GameServer.dll"]
```

이미지를 빌드하고 크기를 확인한다.

```bash
# 빌드
docker build -t game-server:before -f Dockerfile.before .

# 이미지 크기 확인
docker images game-server:before
```

결과는 대략 다음과 같다.

```
REPOSITORY     TAG      SIZE
game-server    before   1.2GB
```

SDK 이미지는 컴파일 도구와 개발 도구가 모두 포함되어 있어 크기가 크다. 하지만 실제 실행에는 런타임만 있으면 된다.

### 멀티 스테이지 빌드 적용

멀티 스테이지 빌드는 빌드 단계와 실행 단계를 분리하여 최종 이미지에는 실행에 필요한 파일만 포함시킨다.

```dockerfile
# Dockerfile.multistage
# 첫 번째 스테이지: 빌드
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

# 프로젝트 파일만 먼저 복사하고 restore (캐싱 활용)
COPY ["GameServer.csproj", "./"]
RUN dotnet restore

# 나머지 소스 코드 복사
COPY . .

# 빌드 및 퍼블리시
RUN dotnet build -c Release -o /app/build
RUN dotnet publish -c Release -o /app/publish

# 두 번째 스테이지: 실행
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS final
WORKDIR /app

# 빌드 스테이지에서 퍼블리시된 결과물만 복사
COPY --from=build /app/publish .

EXPOSE 5000
ENTRYPOINT ["dotnet", "GameServer.dll"]
```

멀티 스테이지 빌드의 구조를 도식화하면 다음과 같다.

```
┌─────────────────────────────────────┐
│   Stage 1: build (SDK 이미지)        │
│                                     │
│   FROM dotnet/sdk:8.0               │
│   ├─ 소스 코드 복사                  │
│   ├─ dotnet restore                 │
│   ├─ dotnet build                   │
│   └─ dotnet publish                 │
│       └─> /app/publish/             │
└─────────────────┬───────────────────┘
                  │ COPY --from=build
                  ▼
┌─────────────────────────────────────┐
│   Stage 2: final (런타임 이미지)      │
│                                     │
│   FROM dotnet/aspnet:8.0            │
│   ├─ 빌드된 파일만 복사               │
│   └─ 실행 준비                       │
│                                     │
│   최종 이미지 크기: ~210MB           │
└─────────────────────────────────────┘
```

빌드하고 크기를 비교한다.

```bash
# 멀티 스테이지 빌드
docker build -t game-server:multistage -f Dockerfile.multistage .

# 크기 비교
docker images | grep game-server
```

결과:

```
REPOSITORY     TAG          SIZE
game-server    multistage   210MB
game-server    before       1.2GB
```

크기가 약 6배 줄어들었다.

### 소켓 서버 멀티 스테이지 빌드

소켓 게임 서버도 동일한 방식으로 최적화한다.

```dockerfile
# SocketServer/Dockerfile
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

COPY ["SocketServer.csproj", "./"]
RUN dotnet restore

COPY . .
RUN dotnet build -c Release -o /app/build
RUN dotnet publish -c Release -o /app/publish

FROM mcr.microsoft.com/dotnet/runtime:8.0 AS final
WORKDIR /app

# 헬스체크용 nc 설치
RUN apt-get update && \
    apt-get install -y netcat-openbsd && \
    rm -rf /var/lib/apt/lists/*

COPY --from=build /app/publish .

EXPOSE 9000
HEALTHCHECK --interval=30s --timeout=3s \
  CMD nc -z localhost 9000 || exit 1

ENTRYPOINT ["dotnet", "SocketServer.dll"]
```

### 실습: 빌드 단계별 확인

Docker는 각 스테이지를 별도로 빌드할 수 있다. 빌드 과정을 단계별로 확인해본다.

```bash
# build 스테이지까지만 빌드
docker build --target build -t game-server:build-stage .

# 빌드 스테이지 이미지 크기 확인
docker images game-server:build-stage

# final 스테이지까지 빌드 (기본)
docker build -t game-server:final .

# 최종 이미지 크기 확인
docker images game-server:final
```

빌드 과정을 자세히 보려면 다음과 같이 한다.

```bash
# 빌드 과정 상세 출력
docker build --progress=plain -t game-server:multistage .
```

## 11.2 이미지 크기 줄이기

### Alpine Linux 기반 이미지 사용

Alpine Linux는 매우 가벼운 Linux 배포판이다. .NET도 Alpine 버전을 제공한다.

```dockerfile
# Dockerfile.alpine
FROM mcr.microsoft.com/dotnet/sdk:8.0-alpine AS build
WORKDIR /src

COPY ["GameServer.csproj", "./"]
RUN dotnet restore

COPY . .
RUN dotnet build -c Release -o /app/build
RUN dotnet publish -c Release -o /app/publish

FROM mcr.microsoft.com/dotnet/aspnet:8.0-alpine AS final
WORKDIR /app

# Alpine에서는 curl 대신 wget 사용
RUN apk add --no-cache curl

COPY --from=build /app/publish .

EXPOSE 5000
HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:5000/health || exit 1

ENTRYPOINT ["dotnet", "GameServer.dll"]
```

빌드하고 비교한다.

```bash
# Alpine 기반 빌드
docker build -t game-server:alpine -f Dockerfile.alpine .

# 크기 비교
docker images | grep game-server
```

결과:

```
REPOSITORY     TAG          SIZE
game-server    alpine       105MB
game-server    multistage   210MB
game-server    before       1.2GB
```

Alpine을 사용하면 크기가 절반으로 줄어든다.

### 불필요한 파일 제거

빌드 과정에서 생성되는 중간 파일들을 제거한다.

```dockerfile
# Dockerfile.optimized
FROM mcr.microsoft.com/dotnet/sdk:8.0-alpine AS build
WORKDIR /src

COPY ["GameServer.csproj", "./"]
RUN dotnet restore

COPY . .
# 불필요한 파일 제외하고 빌드
RUN dotnet publish -c Release -o /app/publish \
    /p:PublishTrimmed=false \
    /p:PublishSingleFile=false

FROM mcr.microsoft.com/dotnet/aspnet:8.0-alpine AS final
WORKDIR /app

# 필요한 도구만 설치
RUN apk add --no-cache curl && \
    rm -rf /var/cache/apk/*

COPY --from=build /app/publish .

# 불필요한 파일 삭제
RUN rm -rf /app/*.pdb

EXPOSE 5000
HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:5000/health || exit 1

ENTRYPOINT ["dotnet", "GameServer.dll"]
```

### .dockerignore 파일 활용

빌드 컨텍스트에 불필요한 파일이 포함되지 않도록 `.dockerignore` 파일을 만든다.

```
# .dockerignore
# Git 관련
.git
.gitignore
.gitattributes

# IDE 관련
.vs
.vscode
*.user
*.suo

# 빌드 출력
bin/
obj/
publish/
out/

# 기타
*.md
*.log
Dockerfile*
docker-compose*
.dockerignore

# 테스트 관련
**/tests/
**/*.Tests/

# 문서
docs/
*.txt
LICENSE
```

.dockerignore를 사용하면 빌드 컨텍스트 크기가 줄어들어 빌드 속도가 빨라진다.

```bash
# 빌드 컨텍스트 크기 확인
docker build --progress=plain -t game-server:test . 2>&1 | grep "transferring context"
```

### 실습: 크기 비교 분석

여러 최적화 기법을 적용한 이미지들의 크기를 비교한다.

```bash
# 모든 버전 빌드
docker build -t game-server:v1-basic -f Dockerfile.before .
docker build -t game-server:v2-multistage -f Dockerfile.multistage .
docker build -t game-server:v3-alpine -f Dockerfile.alpine .
docker build -t game-server:v4-optimized -f Dockerfile.optimized .

# 크기 비교
docker images | grep game-server | sort -k7 -h

# 각 이미지의 레이어 구조 확인
docker history game-server:v4-optimized --no-trunc
```

이미지 크기 분석 도구를 사용할 수도 있다.

```bash
# dive 도구 설치 (WSL Ubuntu에서)
wget https://github.com/wagoodman/dive/releases/download/v0.11.0/dive_0.11.0_linux_amd64.deb
sudo apt install ./dive_0.11.0_linux_amd64.deb

# 이미지 분석
dive game-server:v4-optimized
```

## 11.3 레이어 캐싱 활용

### Docker 레이어 구조 이해

Docker는 Dockerfile의 각 명령어마다 레이어를 생성한다. 변경되지 않은 레이어는 캐시를 재사용한다.

```
┌─────────────────────────────────┐
│  ENTRYPOINT ["dotnet", ...]     │ ← Layer 5
├─────────────────────────────────┤
│  COPY --from=build /app/publish │ ← Layer 4
├─────────────────────────────────┤
│  RUN apk add curl               │ ← Layer 3
├─────────────────────────────────┤
│  WORKDIR /app                   │ ← Layer 2
├─────────────────────────────────┤
│  FROM aspnet:8.0-alpine         │ ← Layer 1 (Base)
└─────────────────────────────────┘
```

레이어가 변경되면 그 이후의 모든 레이어가 다시 빌드된다.

### 최적화된 레이어 순서

자주 변경되지 않는 것을 먼저 배치하고, 자주 변경되는 것을 나중에 배치한다.

나쁜 예:

```dockerfile
# 비효율적인 순서
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

# 소스 코드 전체를 먼저 복사 (자주 변경됨)
COPY . .

# 나중에 restore (소스 변경 시마다 다시 실행)
RUN dotnet restore
RUN dotnet build -c Release -o /app/build
```

좋은 예:

```dockerfile
# 효율적인 순서
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

# 프로젝트 파일만 먼저 복사 (덜 자주 변경됨)
COPY ["GameServer.csproj", "./"]

# restore 먼저 (의존성은 자주 안 바뀜)
RUN dotnet restore

# 소스 코드는 나중에 복사 (자주 변경됨)
COPY . .

# 빌드
RUN dotnet build -c Release -o /app/build
```

### 실습: 캐싱 효과 확인

첫 번째 빌드와 두 번째 빌드의 시간 차이를 비교한다.

```bash
# 첫 번째 빌드 (캐시 없음)
docker build --no-cache -t game-server:cache-test .

# 소스 코드를 약간 수정 (예: Program.cs에 주석 추가)
echo "// Modified" >> Program.cs

# 두 번째 빌드 (캐시 활용)
docker build -t game-server:cache-test .
```

두 번째 빌드에서 다음과 같은 메시지를 볼 수 있다.

```
Step 3/10 : COPY ["GameServer.csproj", "./"]
 ---> Using cache
Step 4/10 : RUN dotnet restore
 ---> Using cache
```

캐시를 활용하면 빌드 시간이 크게 단축된다.

### 레이어 수 줄이기

RUN 명령어를 체이닝하여 레이어 수를 줄인다.

비효율적인 방법:

```dockerfile
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y wget
RUN apt-get clean
RUN rm -rf /var/lib/apt/lists/*
```

효율적인 방법:

```dockerfile
RUN apt-get update && \
    apt-get install -y curl wget && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

Alpine에서는:

```dockerfile
RUN apk add --no-cache curl wget && \
    rm -rf /var/cache/apk/*
```

### 복잡한 프로젝트의 최적화된 Dockerfile

여러 프로젝트가 있는 솔루션의 경우 다음과 같이 최적화한다.

```dockerfile
# Dockerfile.solution
FROM mcr.microsoft.com/dotnet/sdk:8.0-alpine AS build
WORKDIR /src

# 솔루션 파일과 모든 프로젝트 파일 먼저 복사
COPY ["GameServer.sln", "./"]
COPY ["GameServer.Api/GameServer.Api.csproj", "GameServer.Api/"]
COPY ["GameServer.Core/GameServer.Core.csproj", "GameServer.Core/"]
COPY ["GameServer.Data/GameServer.Data.csproj", "GameServer.Data/"]

# restore (프로젝트 구조가 바뀌지 않으면 캐시 활용)
RUN dotnet restore "GameServer.sln"

# 소스 코드 복사
COPY . .

# 빌드
WORKDIR "/src/GameServer.Api"
RUN dotnet build -c Release -o /app/build

# 퍼블리시
RUN dotnet publish -c Release -o /app/publish

FROM mcr.microsoft.com/dotnet/aspnet:8.0-alpine AS final
WORKDIR /app
RUN apk add --no-cache curl
COPY --from=build /app/publish .

EXPOSE 5000
ENTRYPOINT ["dotnet", "GameServer.Api.dll"]
```

## 11.4 보안 베스트 프랙티스

### 루트 사용자 피하기

컨테이너를 루트 권한으로 실행하는 것은 보안상 위험하다. 일반 사용자를 만들어 사용한다.

```dockerfile
# Dockerfile.secure
FROM mcr.microsoft.com/dotnet/sdk:8.0-alpine AS build
WORKDIR /src

COPY ["GameServer.csproj", "./"]
RUN dotnet restore

COPY . .
RUN dotnet publish -c Release -o /app/publish

FROM mcr.microsoft.com/dotnet/aspnet:8.0-alpine AS final
WORKDIR /app

# 일반 사용자 생성
RUN addgroup -g 1000 gameserver && \
    adduser -D -u 1000 -G gameserver gameserver

# 필요한 패키지 설치
RUN apk add --no-cache curl

COPY --from=build /app/publish .

# 파일 소유권 변경
RUN chown -R gameserver:gameserver /app

# 일반 사용자로 전환
USER gameserver

EXPOSE 5000
HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:5000/health || exit 1

ENTRYPOINT ["dotnet", "GameServer.dll"]
```

사용자가 제대로 설정되었는지 확인한다.

```bash
# 컨테이너 빌드 및 실행
docker build -t game-server:secure -f Dockerfile.secure .
docker run -d --name game-test game-server:secure

# 컨테이너 내부에서 사용자 확인
docker exec game-test whoami
# 출력: gameserver

docker exec game-test id
# 출력: uid=1000(gameserver) gid=1000(gameserver)
```

### 읽기 전용 파일시스템

컨테이너의 파일시스템을 읽기 전용으로 마운트하여 보안을 강화한다.

```bash
# 읽기 전용으로 실행
docker run -d \
  --name game-readonly \
  --read-only \
  --tmpfs /tmp \
  -p 5000:5000 \
  game-server:secure
```

Docker Compose에서는:

```yaml
services:
  game-server:
    image: game-server:secure
    read_only: true
    tmpfs:
      - /tmp
    ports:
      - "5000:5000"
    security_opt:
      - no-new-privileges:true
```

### 민감한 정보 제거

빌드 과정에서 민감한 정보가 이미지에 포함되지 않도록 주의한다.

```dockerfile
# 나쁜 예 - 시크릿이 레이어에 남음
FROM alpine
RUN echo "password123" > /tmp/secret
RUN cat /tmp/secret | some-command
RUN rm /tmp/secret  # 레이어에는 여전히 남아있음
```

```dockerfile
# 좋은 예 - 하나의 레이어에서 처리
FROM alpine
RUN echo "password123" > /tmp/secret && \
    cat /tmp/secret | some-command && \
    rm /tmp/secret
```

더 좋은 방법은 빌드 시 시크릿을 주입하는 것이다.

```dockerfile
# BuildKit을 사용한 시크릿 마운트
FROM alpine
RUN --mount=type=secret,id=api_key \
    API_KEY=$(cat /run/secrets/api_key) && \
    echo "Using API key" && \
    # API 키를 사용하는 작업
    true
```

빌드 시:

```bash
# BuildKit 활성화
export DOCKER_BUILDKIT=1

# 시크릿 파일을 사용하여 빌드
docker build --secret id=api_key,src=./api_key.txt -t game-server:secret .
```

### 최신 베이스 이미지 사용

보안 패치가 적용된 최신 이미지를 사용한다.

```dockerfile
# 특정 버전 태그 사용 (latest 지양)
FROM mcr.microsoft.com/dotnet/aspnet:8.0-alpine

# 주기적으로 업데이트
RUN apk upgrade --no-cache
```

### 실습: 보안 강화된 완전한 Dockerfile

모든 보안 베스트 프랙티스를 적용한 최종 Dockerfile을 작성한다.

```dockerfile
# Dockerfile.production
# 빌드 스테이지
FROM mcr.microsoft.com/dotnet/sdk:8.0-alpine AS build

# 보안: 빌드에 필요한 최소한의 권한만 사용
WORKDIR /src

# 의존성 파일 먼저 복사 (캐싱 최적화)
COPY ["GameServer.csproj", "./"]
RUN dotnet restore

# 소스 코드 복사
COPY . .

# 빌드 및 퍼블리시
RUN dotnet publish -c Release -o /app/publish \
    --no-restore \
    /p:PublishTrimmed=false

# 런타임 스테이지
FROM mcr.microsoft.com/dotnet/aspnet:8.0-alpine AS final

# 보안: 패키지 업데이트
RUN apk upgrade --no-cache && \
    apk add --no-cache curl

# 보안: 일반 사용자 생성
RUN addgroup -g 1000 gameserver && \
    adduser -D -u 1000 -G gameserver gameserver

WORKDIR /app

# 빌드 결과물 복사
COPY --from=build /app/publish .

# 보안: 파일 소유권 변경
RUN chown -R gameserver:gameserver /app

# 보안: 일반 사용자로 전환
USER gameserver

# 포트 노출
EXPOSE 5000

# 헬스체크
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:5000/health || exit 1

# 메타데이터
LABEL maintainer="your-email@example.com" \
      version="1.0" \
      description="Game Server - Production Ready"

# 실행
ENTRYPOINT ["dotnet", "GameServer.dll"]
```

### Docker Compose 프로덕션 설정

최적화된 이미지를 사용하는 프로덕션용 Docker Compose 파일을 작성한다.

```yaml
# docker-compose.prod.yml
services:
  game-api:
    build:
      context: ./GameServer
      dockerfile: Dockerfile.production
    image: game-server:production
    container_name: game-api-prod
    restart: unless-stopped
    read_only: true
    tmpfs:
      - /tmp
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE
    ports:
      - "5000:5000"
    environment:
      - ASPNETCORE_ENVIRONMENT=Production
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:5000/health"]
      interval: 30s
      timeout: 3s
      retries: 3
      start_period: 10s
    networks:
      - game-network
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.5'
          memory: 256M
          
  socket-server:
    build:
      context: ./SocketServer
      dockerfile: Dockerfile.production
    image: socket-server:production
    container_name: socket-prod
    restart: unless-stopped
    read_only: true
    tmpfs:
      - /tmp
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE
    ports:
      - "9000:9000"
    environment:
      - ASPNETCORE_ENVIRONMENT=Production
    networks:
      - game-network
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
          
  redis:
    image: redis:7-alpine
    container_name: redis-prod
    restart: unless-stopped
    read_only: true
    tmpfs:
      - /tmp
    security_opt:
      - no-new-privileges:true
    volumes:
      - redis-data:/data
    networks:
      - game-network
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5

networks:
  game-network:
    driver: bridge

volumes:
  redis-data:
    driver: local
```

프로덕션 환경으로 실행한다.

```bash
# 프로덕션 스택 실행
docker compose -f docker-compose.prod.yml up -d

# 상태 확인
docker compose -f docker-compose.prod.yml ps

# 로그 확인
docker compose -f docker-compose.prod.yml logs -f

# 리소스 사용량 확인
docker stats
```

### 정리

이번 장에서는 Docker 이미지를 최적화하는 다양한 방법을 배웠다.

주요 내용을 정리하면 다음과 같다.

**멀티 스테이지 빌드**
- 빌드 단계와 실행 단계를 분리하여 최종 이미지 크기를 줄인다
- SDK 이미지로 빌드하고 런타임 이미지로 실행하여 약 6배 크기를 감소시킨다
- `--target` 옵션으로 특정 스테이지까지만 빌드할 수 있다

**이미지 크기 줄이기**
- Alpine Linux 기반 이미지를 사용하여 크기를 절반으로 줄인다
- `.dockerignore`로 불필요한 파일을 빌드 컨텍스트에서 제외한다
- 빌드 과정에서 생성된 중간 파일들을 제거한다
- RUN 명령어를 체이닝하여 레이어 수를 줄인다

**레이어 캐싱 활용**
- 자주 변경되지 않는 명령어를 앞에 배치한다
- 프로젝트 파일과 소스 코드를 분리하여 복사한다
- 의존성 설치를 소스 코드 복사보다 먼저 실행한다
- 캐시를 활용하면 빌드 시간이 크게 단축된다

**보안 베스트 프랙티스**
- 일반 사용자를 만들어 루트 권한을 피한다
- 읽기 전용 파일시스템을 사용한다
- 최신 베이스 이미지를 사용하고 정기적으로 업데이트한다
- 민감한 정보가 레이어에 남지 않도록 주의한다
- 이미지 스캔 도구로 취약점을 점검한다

최종적으로 프로덕션 환경에 적합한 이미지는 크기가 작고, 빌드가 빠르며, 보안이 강화되어 있어야 한다. 이번 장에서 배운 기법들을 조합하면 초기 1.2GB 크기의 이미지를 약 100MB까지 줄일 수 있다.
