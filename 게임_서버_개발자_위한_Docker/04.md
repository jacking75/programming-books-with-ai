# 4장. ASP.NET Web API 게임 서버 Docker화

이 장에서는 ASP.NET Core Web API로 간단한 게임 서버를 만들고, 이를 Docker 컨테이너로 패키징하여 실행하는 전 과정을 실습한다. 게임 서버를 Docker화하면 개발 환경과 운영 환경의 차이를 없앤 채로 일관된 배포가 가능해진다.

## 4.1 간단한 REST API 게임 서버 만들기

먼저 플레이어 정보를 관리하는 간단한 게임 서버를 만든다. 이 서버는 플레이어 등록, 조회, 점수 업데이트 등의 기능을 제공한다.

### 프로젝트 생성

WSL 터미널을 열고 다음 명령어로 새 Web API 프로젝트를 생성한다.

```bash
mkdir GameServerDocker
cd GameServerDocker
dotnet new webapi -n GameServer
cd GameServer
```

### 게임 서버 코드 작성

**Models/Player.cs** 파일을 생성하여 플레이어 모델을 정의한다.

```csharp
namespace GameServer.Models
{
    public class Player
    {
        public int Id { get; set; }
        public string Name { get; set; } = string.Empty;
        public int Score { get; set; }
        public DateTime CreatedAt { get; set; }
    }
}
```

**Services/PlayerService.cs** 파일을 생성하여 플레이어 데이터를 관리하는 서비스를 만든다.

```csharp
using GameServer.Models;

namespace GameServer.Services
{
    public class PlayerService
    {
        private readonly List<Player> _players = new();
        private int _nextId = 1;

        public Player CreatePlayer(string name)
        {
            var player = new Player
            {
                Id = _nextId++,
                Name = name,
                Score = 0,
                CreatedAt = DateTime.UtcNow
            };
            _players.Add(player);
            return player;
        }

        public Player? GetPlayer(int id)
        {
            return _players.FirstOrDefault(p => p.Id == id);
        }

        public List<Player> GetAllPlayers()
        {
            return _players;
        }

        public bool UpdateScore(int id, int score)
        {
            var player = GetPlayer(id);
            if (player == null) return false;
            
            player.Score = score;
            return true;
        }
    }
}
```

**Controllers/PlayersController.cs** 파일을 생성하여 API 엔드포인트를 정의한다.

```csharp
using GameServer.Models;
using GameServer.Services;
using Microsoft.AspNetCore.Mvc;

namespace GameServer.Controllers
{
    [ApiController]
    [Route("api/[controller]")]
    public class PlayersController : ControllerBase
    {
        private readonly PlayerService _playerService;

        public PlayersController(PlayerService playerService)
        {
            _playerService = playerService;
        }

        [HttpPost]
        public ActionResult<Player> CreatePlayer([FromBody] string name)
        {
            var player = _playerService.CreatePlayer(name);
            return CreatedAtAction(nameof(GetPlayer), new { id = player.Id }, player);
        }

        [HttpGet("{id}")]
        public ActionResult<Player> GetPlayer(int id)
        {
            var player = _playerService.GetPlayer(id);
            if (player == null)
                return NotFound();
            
            return player;
        }

        [HttpGet]
        public ActionResult<List<Player>> GetAllPlayers()
        {
            return _playerService.GetAllPlayers();
        }

        [HttpPut("{id}/score")]
        public IActionResult UpdateScore(int id, [FromBody] int score)
        {
            if (!_playerService.UpdateScore(id, score))
                return NotFound();
            
            return NoContent();
        }
    }
}
```

**Program.cs** 파일을 수정하여 서비스를 등록한다.

```csharp
using GameServer.Services;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddSingleton<PlayerService>();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.MapControllers();

app.Run();
```

### 로컬 테스트

먼저 로컬에서 서버가 정상 동작하는지 확인한다.

```bash
dotnet run
```

브라우저에서 `http://localhost:5000/swagger`를 열어 Swagger UI를 확인한다. 또는 다른 터미널에서 curl로 테스트한다.

```bash

curl -X POST http://localhost:5000/api/players \
  -H "Content-Type: application/json" \
  -d '"TestPlayer"'

# 플레이어 조회

# 점수 업데이트
curl -X PUT http://localhost:5000/api/players/1/score \
  -H "Content-Type: application/json" \
  -d '100'
```

## 4.2 Dockerfile 작성 실습

Docker 컨테이너로 게임 서버를 실행하려면 Dockerfile이 필요하다. Dockerfile은 이미지를 만들기 위한 레시피와 같다.

### 기본 Dockerfile 구조 이해

Dockerfile의 구조를 다이어그램으로 표현하면 다음과 같다.

```
┌─────────────────────────────────────┐
│   FROM: 베이스 이미지 선택          │
│   (.NET SDK 또는 Runtime)           │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   WORKDIR: 작업 디렉토리 설정       │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   COPY: 프로젝트 파일 복사          │
│   (csproj, 소스 코드)               │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   RUN: 빌드 및 의존성 설치          │
│   (dotnet restore, dotnet build)    │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   EXPOSE: 포트 노출 선언            │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   ENTRYPOINT: 컨테이너 실행 명령    │
│   (dotnet run)                      │
└─────────────────────────────────────┘
```

### 단일 스테이지 Dockerfile 작성

GameServer 디렉토리에 **Dockerfile** 파일을 생성한다.

```dockerfile
# .NET SDK 이미지를 베이스로 사용
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build

# 작업 디렉토리 설정
WORKDIR /app

# csproj 파일 복사 및 의존성 복원
COPY *.csproj ./
RUN dotnet restore

# 나머지 소스 코드 복사
COPY . ./

# 릴리스 모드로 빌드
RUN dotnet publish -c Release -o out

# Runtime 이미지로 전환
FROM mcr.microsoft.com/dotnet/aspnet:8.0

WORKDIR /app

# 빌드 결과물 복사
COPY --from=build /app/out .

# 서버가 사용할 포트 노출
EXPOSE 8080

# 환경 변수 설정
ENV ASPNETCORE_URLS=http://+:8080

# 애플리케이션 실행
ENTRYPOINT ["dotnet", "GameServer.dll"]
```

### Dockerfile 명령어 설명

각 명령어의 역할을 정리하면 다음과 같다.

**FROM**: 베이스 이미지를 지정한다. `mcr.microsoft.com/dotnet/sdk:8.0`은 .NET 8.0 SDK가 포함된 이미지다. `AS build`는 이 스테이지에 이름을 부여한다.

**WORKDIR**: 컨테이너 내부의 작업 디렉토리를 설정한다. 이후 모든 명령어는 이 디렉토리를 기준으로 실행된다.

**COPY**: 호스트의 파일을 컨테이너로 복사한다. `*.csproj`는 프로젝트 파일만 먼저 복사하여 레이어 캐싱을 효율적으로 활용한다.

**RUN**: 이미지 빌드 시 실행할 명령어다. `dotnet restore`는 NuGet 패키지를 다운로드하고, `dotnet publish`는 배포 가능한 형태로 빌드한다.

**EXPOSE**: 컨테이너가 리슨할 포트를 문서화한다. 실제 포트 바인딩은 컨테이너 실행 시 `-p` 옵션으로 설정한다.

**ENV**: 환경 변수를 설정한다. ASP.NET Core는 `ASPNETCORE_URLS`로 리슨 주소를 지정한다.

**ENTRYPOINT**: 컨테이너가 시작될 때 실행할 명령어다.

### .dockerignore 파일 작성

불필요한 파일이 이미지에 포함되지 않도록 **.dockerignore** 파일을 생성한다.

```
bin/
obj/
*.user
.vs/
.vscode/
*.log
```

## 4.3 .NET 게임 서버 이미지 빌드

Dockerfile을 작성했으니 이제 실제로 Docker 이미지를 빌드한다.

### 이미지 빌드 명령어

GameServer 디렉토리에서 다음 명령어를 실행한다.

```bash
docker build -t game-server:v1 .
```

명령어 옵션을 살펴보면 다음과 같다.

- `-t game-server:v1`: 이미지에 태그를 지정한다. `game-server`는 이미지 이름이고 `v1`은 버전이다.
- `.`: 현재 디렉토리를 빌드 컨텍스트로 사용한다.

빌드가 진행되면서 다음과 같은 단계들이 실행된다.

```
[+] Building 45.2s (15/15) FINISHED
 => [internal] load build definition from Dockerfile
 => [internal] load .dockerignore
 => [internal] load metadata for mcr.microsoft.com/dotnet/aspnet:8.0
 => [internal] load metadata for mcr.microsoft.com/dotnet/sdk:8.0
 => [build 1/6] FROM mcr.microsoft.com/dotnet/sdk:8.0
 => [build 2/6] WORKDIR /app
 => [build 3/6] COPY *.csproj ./
 => [build 4/6] RUN dotnet restore
 => [build 5/6] COPY . ./
 => [build 6/6] RUN dotnet publish -c Release -o out
 => [stage-1 2/3] WORKDIR /app
 => [stage-1 3/3] COPY --from=build /app/out .
 => exporting to image
```

### 빌드된 이미지 확인

빌드가 완료되면 이미지 목록을 확인한다.

```bash
docker images
```

출력 결과는 다음과 유사하다.

```
REPOSITORY     TAG       IMAGE ID       CREATED          SIZE
game-server    v1        a1b2c3d4e5f6   2 minutes ago    220MB
```

### 이미지 레이어 구조 이해

Docker 이미지는 레이어로 구성된다. 각 Dockerfile 명령어가 하나의 레이어를 생성한다.

```
┌─────────────────────────────────────┐
│  ENTRYPOINT ["dotnet", ...]         │ ← 최상위 레이어
├─────────────────────────────────────┤
│  ENV ASPNETCORE_URLS=...            │
├─────────────────────────────────────┤
│  COPY --from=build /app/out .       │
├─────────────────────────────────────┤
│  WORKDIR /app                       │
├─────────────────────────────────────┤
│  ASP.NET Runtime Base Image         │
└─────────────────────────────────────┘
```

레이어는 캐시되므로 코드만 변경하면 상위 레이어만 다시 빌드된다.

### 이미지 상세 정보 확인

이미지의 상세 정보를 확인하려면 다음 명령어를 사용한다.

```bash
docker inspect game-server:v1
```

이 명령어는 이미지의 모든 메타데이터를 JSON 형식으로 출력한다.

## 4.4 컨테이너로 게임 서버 실행하기

이미지를 빌드했으니 이제 컨테이너로 실행한다.

### 기본 실행

가장 간단한 형태로 컨테이너를 실행한다.

```bash
docker run -d -p 8080:8080 --name my-game-server game-server:v1
```

옵션 설명은 다음과 같다.

- `-d`: 백그라운드(detached) 모드로 실행한다.
- `-p 8080:8080`: 호스트의 8080 포트를 컨테이너의 8080 포트에 매핑한다.
- `--name my-game-server`: 컨테이너에 이름을 부여한다.

### 포트 매핑 구조

포트 매핑의 동작 방식을 다이어그램으로 표현하면 다음과 같다.

```
┌──────────────────┐         ┌─────────────────────┐
│   클라이언트     │         │   Docker Host       │
│                  │         │   (WSL/Windows)     │
└────────┬─────────┘         └──────────┬──────────┘
         │                              │
         │  http://localhost:8080       │
         └──────────────────────────────┤
                                        │ Port 8080
                              ┌─────────▼──────────┐
                              │   Container        │
                              │   game-server:v1   │
                              │                    │
                              │   Port 8080        │
                              │   (ASP.NET Core)   │
                              └────────────────────┘
```

### 컨테이너 상태 확인

실행 중인 컨테이너를 확인한다.

```bash
docker ps
```

출력 결과 예시는 다음과 같다.

```
CONTAINER ID   IMAGE             COMMAND                  CREATED          STATUS          PORTS                    NAMES
abc123def456   game-server:v1    "dotnet GameServer.dll"  30 seconds ago   Up 29 seconds   0.0.0.0:8080->8080/tcp   my-game-server
```

### API 테스트

컨테이너가 정상 동작하는지 확인한다.

```bash
# 플레이어 생성
curl -X POST http://localhost:8080/api/players \
  -H "Content-Type: application/json" \
  -d '"DockerPlayer"'

# 응답 예시
# {"id":1,"name":"DockerPlayer","score":0,"createdAt":"2025-11-22T10:30:00Z"}

# 모든 플레이어 조회
curl http://localhost:8080/api/players

# 특정 플레이어 조회
curl http://localhost:8080/api/players/1

# 점수 업데이트
curl -X PUT http://localhost:8080/api/players/1/score \
  -H "Content-Type: application/json" \
  -d '500'
```

Windows 브라우저에서도 `http://localhost:8080/api/players`로 접속하여 확인할 수 있다.

### 로그 확인

컨테이너의 로그를 실시간으로 확인한다.

```bash
docker logs -f my-game-server
```

`-f` 옵션은 로그를 계속 추적한다. 중지하려면 `Ctrl+C`를 누른다.

### 환경 변수를 사용한 실행

ASP.NET Core의 환경 설정을 변경하여 실행할 수 있다.

```bash
docker run -d -p 8081:8080 \
  --name game-server-dev \
  -e ASPNETCORE_ENVIRONMENT=Development \
  game-server:v1
```

### 컨테이너 내부 접속

디버깅이 필요하면 컨테이너 내부에 접속한다.

```bash
docker exec -it my-game-server /bin/bash
```

컨테이너 내부에서 프로세스를 확인할 수 있다.

```bash
ps aux
ls -la
exit
```

### 컨테이너 정지 및 삭제

테스트가 끝나면 컨테이너를 정리한다.

```bash
# 컨테이너 정지
docker stop my-game-server

# 컨테이너 삭제
docker rm my-game-server

# 또는 한 번에 정지 및 삭제
docker rm -f my-game-server
```

### 이미지 재빌드 및 실행 패턴

코드를 수정한 후 다시 배포하는 일반적인 패턴은 다음과 같다.

```bash
# 1. 코드 수정 후 이미지 재빌드
docker build -t game-server:v2 .

# 2. 기존 컨테이너 정지 및 삭제
docker rm -f my-game-server

# 3. 새 이미지로 컨테이너 실행
docker run -d -p 8080:8080 --name my-game-server game-server:v2
```

### 실습 체크리스트

이 장에서 배운 내용을 정리하면 다음과 같다.

□ ASP.NET Core Web API 게임 서버를 생성했다  
□ Dockerfile을 작성하여 이미지 빌드 방법을 이해했다  
□ docker build 명령어로 이미지를 생성했다  
□ docker run으로 컨테이너를 실행하고 포트 매핑을 설정했다  
□ curl을 사용하여 컨테이너화된 API를 테스트했다  
□ docker logs로 애플리케이션 로그를 확인했다  
□ docker exec로 컨테이너 내부에 접속했다

다음 장에서는 소켓 기반 게임 서버를 Docker화하는 방법을 배운다. REST API와 달리 TCP 소켓은 지속적인 연결을 유지하므로 네트워크 설정에 주의가 필요하다.
