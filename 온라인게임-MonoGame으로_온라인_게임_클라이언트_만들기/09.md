# 9장: 성능 최적화와 디버깅
온라인 게임에서 성능은 게임의 즐거움을 좌우하는 매우 중요한 요소다. 지금까지 우리는 기본적인 네트워크 통신과 게임 구현에 집중했다면, 이제는 게임이 얼마나 효율적으로 동작하는지 살펴볼 차례다. 이번 장에서는 네트워크 트래픽을 줄이는 방법, 프레임 동기화를 통해 모든 플레이어가 같은 속도로 게임을 즐길 수 있도록 하는 방법, 그리고 문제를 찾아내기 위한 디버깅 도구를 활용하는 방법을 배울 것이다.

## 9.1 네트워크 트래픽 최소화

### 왜 트래픽 최소화가 필요한가
온라인 게임에서 플레이어의 위치 업데이트는 매 프레임마다 발생한다. 60FPS로 게임이 실행된다면 1초에 60번의 위치 업데이트 패킷을 보내야 한다는 뜻이다. 만약 게임에 100명의 플레이어가 있다면 1초에 6000개의 패킷이 네트워크를 통해 이동하게 되는데, 이는 서버의 부하를 크게 증가시킨다.

네트워크 대역폭은 제한된 자원이고, 패킷을 보낼수록 서버의 처리량도 증가하며, 인터넷 연결 비용도 올라간다. 따라서 우리는 필요한 정보만 보내거나, 자주 보내지 않아도 되는 정보는 덜 자주 보내는 방식으로 트래픽을 줄여야 한다.

### 델타 압축(Delta Compression)
델타 압축은 이전 상태와 현재 상태의 차이만 전송하는 기법이다. 예를 들어, 플레이어의 위치가 (100, 200)에서 (101, 200)으로 변했다면, 좌표 전체를 보내지 말고 X 좌표만 1 증가했다는 정보만 보내는 것이다.

다음 코드를 살펴보자. 이 코드는 플레이어의 이전 위치를 기억했다가, 위치가 변했을 때만 변경된 좌표를 포함한 패킷을 생성한다.

```csharp
public class DeltaCompressionExample
{
    // 플레이어의 이전 위치를 저장하는 클래스
    public class PlayerState
    {
        public int PlayerId { get; set; }
        public float X { get; set; }
        public float Y { get; set; }
        public long LastUpdateTime { get; set; }
    }

    // 변경된 좌표만을 나타내는 패킷 구조
    public class DeltaPositionPacket
    {
        public int PlayerId { get; set; }
        public bool HasX { get; set; }
        public float X { get; set; }
        public bool HasY { get; set; }
        public float Y { get; set; }
    }

    // 델타 압축을 통해 패킷을 생성하는 메서드
    public static byte[] CreateDeltaPositionPacket(
        PlayerState currentState, 
        PlayerState previousState,
        float threshold = 0.1f)
    {
        using (var ms = new MemoryStream())
        using (var bw = new BinaryWriter(ms))
        {
            // 패킷 타입 (Position Update = 2)
            bw.Write((byte)2);
            
            // 플레이어 ID
            bw.Write(currentState.PlayerId);

            // X 좌표 변경 여부 확인
            bool hasXChanged = Math.Abs(currentState.X - previousState.X) > threshold;
            bw.Write(hasXChanged);
            if (hasXChanged)
            {
                bw.Write(currentState.X);
            }

            // Y 좌표 변경 여부 확인
            bool hasYChanged = Math.Abs(currentState.Y - previousState.Y) > threshold;
            bw.Write(hasYChanged);
            if (hasYChanged)
            {
                bw.Write(currentState.Y);
            }

            return ms.ToArray();
        }
    }

    // 패킷을 읽고 상태를 업데이트하는 메서드
    public static void ParseDeltaPositionPacket(
        byte[] data, 
        Dictionary<int, PlayerState> playerStates)
    {
        using (var ms = new MemoryStream(data))
        using (var br = new BinaryReader(ms))
        {
            byte packetType = br.ReadByte();
            int playerId = br.ReadInt32();

            if (!playerStates.ContainsKey(playerId))
            {
                playerStates[playerId] = new PlayerState { PlayerId = playerId };
            }

            var playerState = playerStates[playerId];

            // X 좌표 업데이트
            bool hasX = br.ReadBoolean();
            if (hasX)
            {
                playerState.X = br.ReadSingle();
            }

            // Y 좌표 업데이트
            bool hasY = br.ReadBoolean();
            if (hasY)
            {
                playerState.Y = br.ReadSingle();
            }

            playerState.LastUpdateTime = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        }
    }
}
```

이 코드의 목적은 플레이어의 위치가 실제로 변했을 때만 해당 좌표를 전송하는 것이다. threshold 변수를 사용해서 0.1 이상의 변화만 의미 있는 변화로 간주한다. 이렇게 하면 미세한 부동소수점 오차로 인한 불필요한 패킷 전송을 방지할 수 있다.

패킷 구조를 보면, 각 좌표 앞에 `hasX`와 `hasY` 같은 불린 값이 있다. 이를 통해 수신 측에서 어떤 좌표가 변했는지 알 수 있다. 예를 들어, Y 좌표만 변했다면 X 좌표 값 4바이트를 절약할 수 있다.

### 메시지 버칭(Message Batching)
메시지 버칭은 여러 개의 작은 패킷을 하나의 큰 패킷으로 묶어서 보내는 기법이다. 네트워크 통신에서는 패킷의 크기보다 패킷의 개수가 더 큰 오버헤드를 발생시킨다. 따라서 100개의 작은 패킷을 따로 보내는 것보다 1개의 큰 패킷으로 묶어서 보내는 것이 훨씬 효율적이다.

다음은 메시지 버칭을 구현한 예제다.

```csharp
public class MessageBatchingExample
{
    // 전송 대기 중인 메시지를 저장하는 큐
    private Queue<byte[]> pendingMessages = new Queue<byte[]>();
    private float batchInterval = 0.016f; // 약 60FPS (1/60 ≈ 0.0167초)
    private float timeSinceLastBatch = 0f;

    // 메시지를 큐에 추가
    public void QueueMessage(byte[] message)
    {
        pendingMessages.Enqueue(message);
    }

    // 주기적으로 호출되어 메시지를 배치로 전송
    public byte[] GetBatchedMessage(float deltaTime)
    {
        timeSinceLastBatch += deltaTime;

        // 충분한 시간이 지났거나 메시지가 많이 쌓였으면 배치 생성
        if (timeSinceLastBatch >= batchInterval || pendingMessages.Count > 20)
        {
            return CreateBatch();
        }

        return null; // 아직 배치를 보낼 시간이 아님
    }

    // 대기 중인 모든 메시지를 하나의 배치로 만듦
    private byte[] CreateBatch()
    {
        using (var ms = new MemoryStream())
        using (var bw = new BinaryWriter(ms))
        {
            // 배치 패킷 헤더
            bw.Write((byte)255); // 배치 패킷 타입
            bw.Write((ushort)pendingMessages.Count); // 포함된 메시지 개수

            // 각 메시지를 배치에 추가
            while (pendingMessages.Count > 0)
            {
                byte[] message = pendingMessages.Dequeue();
                bw.Write((ushort)message.Length); // 메시지 길이
                bw.Write(message); // 메시지 데이터
            }

            timeSinceLastBatch = 0f;
            return ms.ToArray();
        }
    }

    // 배치 패킷을 파싱하는 메서드
    public static List<byte[]> ParseBatch(byte[] batchData)
    {
        var messages = new List<byte[]>();

        using (var ms = new MemoryStream(batchData))
        using (var br = new BinaryReader(ms))
        {
            byte packetType = br.ReadByte();
            if (packetType != 255)
            {
                return messages; // 배치 패킷이 아님
            }

            ushort messageCount = br.ReadUInt16();

            for (int i = 0; i < messageCount; i++)
            {
                ushort messageLength = br.ReadUInt16();
                byte[] message = br.ReadBytes((int)messageLength);
                messages.Add(message);
            }
        }

        return messages;
    }
}
```

메시지 버칭의 동작 원리를 이해해보자. 게임이 실행되는 동안 플레이어 위치 업데이트, 채팅 메시지, 공격 이벤트 등 다양한 메시지가 발생한다. 이 메시지들을 즉시 보내지 말고 `pendingMessages` 큐에 넣어둔다. 그 다음 `GetBatchedMessage` 메서드가 일정한 시간 간격(약 60FPS)으로 호출될 때 대기 중인 모든 메시지를 하나의 배치로 만들어 전송한다.

이 방식의 장점은 네트워크 오버헤드를 크게 줄일 수 있다는 것이다. 예를 들어, 10개의 메시지가 있다면 10번의 패킷을 보내는 대신 1번만 보내면 된다. 다만, 약간의 지연 시간이 발생한다는 단점이 있다. 위의 코드에서 `batchInterval`이 약 0.016초이므로, 최악의 경우 메시지가 16ms 지연될 수 있다. 대부분의 게임에서 이 정도의 지연은 플레이어가 인지하기 어렵다.

### 상태 압축
게임의 상태 정보를 보낼 때도 트래픽을 줄일 수 있다. 예를 들어, 플레이어의 방향이 8개 방향 중 하나라면, 플레이어가 바라보는 각도(float, 4바이트)를 보내는 대신 0부터 7 사이의 숫자(1바이트)로 방향을 표현할 수 있다.

```csharp
public class StateCompressionExample
{
    // 플레이어 상태를 압축하여 저장
    public class CompressedPlayerState
    {
        public int PlayerId { get; set; }
        public short X { get; set; }  // float 대신 short 사용 (2바이트)
        public short Y { get; set; }  // float 대신 short 사용 (2바이트)
        public byte Direction { get; set; } // 8개 방향을 1바이트로 표현
        public byte State { get; set; } // 플레이어 상태 (대기, 이동, 공격 등)
    }

    // float 좌표를 short로 변환 (범위 제한이 필요함)
    public static short CompressCoordinate(float value, int gridSize)
    {
        // gridSize는 게임 월드의 최대 좌표값
        // -gridSize/2부터 gridSize/2까지를 -32768부터 32767까지의 short로 매핑
        int compressed = (int)((value + gridSize / 2f) / (gridSize / 256f));
        return (short)Math.Clamp(compressed, short.MinValue, short.MaxValue);
    }

    public static float DecompressCoordinate(short value, int gridSize)
    {
        return (value * (gridSize / 256f)) - (gridSize / 2f);
    }

    // 라디안 각도를 1바이트로 변환
    public static byte CompressDirection(float radians)
    {
        // 0 ~ 2π를 0 ~ 255로 매핑
        return (byte)((radians / MathF.PI / 2f) * 255f);
    }

    public static float DecompressDirection(byte compressed)
    {
        return (compressed / 255f) * MathF.PI * 2f;
    }
}
```

상태 압축을 사용하면 플레이어 정보를 전송할 때 필요한 바이트 수를 크게 줄일 수 있다. 압축 전에는 플레이어 ID(4바이트) + X 좌표(4바이트) + Y 좌표(4바이트) + 방향(4바이트) + 상태(1바이트) = 17바이트가 필요하지만, 압축 후에는 플레이어 ID(4바이트) + X 좌표(2바이트) + Y 좌표(2바이트) + 방향(1바이트) + 상태(1바이트) = 10바이트로 줄어든다. 약 41% 정도의 트래픽 절감 효과를 얻을 수 있다.

다만 압축을 사용할 때는 트레이드오프를 고려해야 한다. 좌표를 short로 변환하면 정밀도가 떨어지고, 방향을 1바이트로 표현하면 8개 방향(45도씩)만 표현할 수 있다. 게임의 특성에 따라 어느 정도의 정밀도 손실이 허용되는지 판단해야 한다.

## 9.2 프레임 동기화

### 프레임 드롭의 문제점
온라인 게임에서 프레임 드롭은 단순히 화면이 끊기는 것 이상의 문제를 야기한다. 플레이어의 컴퓨터에서 프레임이 드러나면 게임의 시간 진행 속도가 달라지게 되고, 이는 네트워크를 통해 동기화된 다른 플레이어들과의 시간 차이를 만든다.

예를 들어, 플레이어 A가 공격했을 때 정확히 게임 시간 1000ms에 공격이 발생했어야 하는데, 플레이어 A의 컴퓨터에서 프레임이 드러나서 실제로는 1050ms에 처리된다면, 플레이어 B는 이 공격을 맞아야 하는지 말아야 하는지 판단할 수 없게 된다.

### 고정 타임스텝(Fixed Timestep)
MonoGame에서는 고정 타임스텝을 설정하여 모든 플레이어가 같은 속도로 게임을 진행하도록 강제할 수 있다.

```csharp
public class GameTimingExample : Game
{
    private GraphicsDeviceManager graphics;

    public GameTimingExample()
    {
        graphics = new GraphicsDeviceManager(this);
        
        // 고정 타임스텝 활성화
        this.IsFixedTimeStep = true;
        
        // 목표 프레임 레이트를 60FPS로 설정
        this.TargetElapsedTime = TimeSpan.FromSeconds(1d / 60d);
    }

    protected override void Update(GameTime gameTime)
    {
        // Update 메서드는 정확히 60번/초 호출된다
        // gameTime.ElapsedGameTime은 항상 약 1/60초다
        double deltaTime = gameTime.ElapsedGameTime.TotalSeconds;

        // 게임 로직 업데이트
        UpdatePlayerPosition(deltaTime);
        UpdateNetworkPackets();

        base.Update(gameTime);
    }

    private void UpdatePlayerPosition(double deltaTime)
    {
        // 고정된 deltaTime으로 인해 모든 클라이언트에서 같은 속도로 이동
        float moveDistance = 100f * (float)deltaTime; // 초당 100 픽셀 이동
        // ... 위치 업데이트 코드
    }

    private void UpdateNetworkPackets()
    {
        // 고정 타임스텝 덕분에 네트워크 업데이트도 규칙적으로 발생
        // 이는 입력 지연을 예측 가능하게 만든다
    }

    protected override void Draw(GameTime gameTime)
    {
        GraphicsDevice.Clear(Color.Black);

        // Draw 메서드는 모니터 주사율에 맞춰 호출될 수 있으므로
        // Update와 Draw 호출 빈도가 다를 수 있다
        // 따라서 Draw에서 게임 로직을 처리하면 안 된다

        base.Draw(gameTime);
    }
}
```

고정 타임스텝은 게임 로직을 업데이트하는 빈도를 고정한다. 위의 코드에서 `IsFixedTimeStep`을 true로 설정하고 `TargetElapsedTime`을 1/60초로 설정하면, Update 메서드는 항상 정확히 초당 60번 호출된다. 이렇게 하면 플레이어의 컴퓨터 성능이 다르더라도 모든 플레이어가 같은 속도로 게임을 진행할 수 있다.

한편, Draw 메서드는 모니터의 주사율에 맞춰 호출되므로, 그래픽카드가 좋은 컴퓨터에서는 Draw가 더 자주 호출될 수 있다. 이것이 바로 Update와 Draw를 분리하는 이유다. 게임 로직은 항상 Update에서만 처리해야 한다.

### 네트워크 타이밍 동기화
더 정교한 동기화를 위해서는 서버의 시간을 기준으로 모든 클라이언트를 동기화해야 한다.

```csharp
public class NetworkTimingSynchronization
{
    // 클라이언트의 로컬 시간과 서버 시간의 차이를 저장
    private long serverTimeOffset = 0;
    private long lastSyncTime = 0;
    private Stopwatch localStopwatch = new Stopwatch();

    public NetworkTimingSynchronization()
    {
        localStopwatch.Start();
    }

    // 서버로부터 현재 서버 시간을 받음
    public void ReceiveServerTime(long serverTime)
    {
        long clientTime = localStopwatch.ElapsedMilliseconds;
        
        // 서버 시간과 클라이언트 시간의 차이를 계산
        // 이 차이를 사용하여 클라이언트의 로컬 시간을 보정
        serverTimeOffset = serverTime - clientTime;
        lastSyncTime = DateTime.UtcNow.Ticks;
    }

    // 현재의 동기화된 시간을 반환
    public long GetSynchronizedTime()
    {
        return localStopwatch.ElapsedMilliseconds + serverTimeOffset;
    }

    // 주기적으로 호출하여 서버와 시간을 다시 동기화
    // 네트워크 지연이나 시계 드리프트로 인한 오차를 수정
    public bool ShouldResync()
    {
        long timeSinceLastSync = (DateTime.UtcNow.Ticks - lastSyncTime) / 10000; // 밀리초로 변환
        return timeSinceLastSync > 5000; // 5초마다 재동기화
    }
}
```

이 클래스는 클라이언트의 로컬 시간과 서버의 시간 차이를 추적한다. 게임이 시작될 때 서버로부터 현재 서버 시간을 받은 후, 그 차이를 `serverTimeOffset`에 저장한다. 그 이후로는 `GetSynchronizedTime` 메서드를 호출하여 항상 서버 기준의 시간을 얻을 수 있다. 이렇게 하면 비록 클라이언트의 로컬 시간이 조금씩 차이나더라도, 모든 플레이어가 같은 게임 시간에 기반해서 게임을 진행할 수 있다.

### 입력 지연 보상
고정 타임스텝을 사용하더라도, 플레이어가 입력을 한 후 그 입력이 네트워크를 통해 전달되고 서버에서 처리되기까지의 지연이 존재한다. 이를 입력 지연(input lag)이라고 부른다.

입력 지연을 보상하는 한 가지 방법은 클라이언트 측에서 먼저 입력을 처리하고, 나중에 서버의 응답을 받아서 조정하는 클라이언트 사이드 예측(client-side prediction)이다.

```csharp
public class ClientSidePredictionExample
{
    public class PlayerInputState
    {
        public float X { get; set; }
        public float Y { get; set; }
        public int FrameNumber { get; set; }
    }

    private Queue<PlayerInputState> inputHistory = new Queue<PlayerInputState>();
    private int currentFrame = 0;
    private float playerX = 0;
    private float playerY = 0;
    private const int MAX_HISTORY = 120; // 2초치 입력 저장

    // 플레이어 입력을 처리하고 위치를 즉시 업데이트 (예측)
    public void ProcessInput(float moveX, float moveY)
    {
        // 클라이언트에서 즉시 위치 업데이트
        playerX += moveX;
        playerY += moveY;

        // 입력 기록 저장
        var inputState = new PlayerInputState
        {
            X = playerX,
            Y = playerY,
            FrameNumber = currentFrame
        };

        inputHistory.Enqueue(inputState);
        if (inputHistory.Count > MAX_HISTORY)
        {
            inputHistory.Dequeue();
        }

        currentFrame++;
    }

    // 서버로부터 보정된 위치를 받음
    public void ReceiveServerCorrection(float correctX, float correctY, int frameNumber)
    {
        // 서버가 확인한 프레임 이후의 입력만 다시 적용
        var statesToReplay = new List<PlayerInputState>();

        foreach (var state in inputHistory)
        {
            if (state.FrameNumber > frameNumber)
            {
                statesToReplay.Add(state);
            }
        }

        // 위치를 서버의 보정값으로 재설정
        playerX = correctX;
        playerY = correctY;

        // 보정 이후의 입력들을 다시 재생(replay)
        foreach (var state in statesToReplay)
        {
            playerX = state.X;
            playerY = state.Y;
        }
    }

    public (float, float) GetPlayerPosition()
    {
        return (playerX, playerY);
    }
}
```

이 방식은 다음과 같이 동작한다. 플레이어가 화살표 키를 누르면, 클라이언트는 먼저 자신의 위치를 즉시 업데이트하여 플레이어가 입력에 즉시 반응하는 것처럼 보이게 한다. 동시에 이 입력을 서버에 전송한다. 서버가 입력을 받아서 처리한 후 보정된 위치를 다시 보내면, 클라이언트는 그 위치를 기준으로 다시 정렬한다. 만약 클라이언트의 예측이 서버의 판단과 차이가 나면, 그 차이를 보정하고 그 이후의 입력들을 다시 적용한다.

이 방법으로 게임은 입력이 지연되지 않는 것처럼 느껴지면서도, 최종적으로는 서버의 판정이 적용되도록 할 수 있다.
  
</br>  
  

## 9.3 디버깅 도구 활용

### 네트워크 패킷 분석
온라인 게임의 문제를 해결하기 위해서는 네트워크를 통해 어떤 데이터가 오고 가는지 확인해야 한다. 다음은 간단한 패킷 분석 도구다.  

```csharp
public class NetworkPacketAnalyzer
{
    public class PacketRecord
    {
        public DateTime Timestamp { get; set; }
        public string Direction { get; set; } // "Send" 또는 "Receive"
        public byte PacketType { get; set; }
        public int PacketSize { get; set; }
        public string PayloadPreview { get; set; }
    }

    private List<PacketRecord> packetLog = new List<PacketRecord>();
    private Dictionary<byte, string> packetTypeNames = new Dictionary<byte, string>
    {
        { 1, "Login" },
        { 2, "PositionUpdate" },
        { 3, "ChatMessage" },
        { 255, "Batch" }
    };

    // 전송한 패킷 기록
    public void RecordSentPacket(byte[] data)
    {
        if (data.Length == 0)
            return;

        byte packetType = data[0];
        var record = new PacketRecord
        {
            Timestamp = DateTime.Now,
            Direction = "Send",
            PacketType = packetType,
            PacketSize = data.Length,
            PayloadPreview = GeneratePreview(data)
        };

        packetLog.Add(record);
    }

    // 수신한 패킷 기록
    public void RecordReceivedPacket(byte[] data)
    {
        if (data.Length == 0)
            return;

        byte packetType = data[0];
        var record = new PacketRecord
        {
            Timestamp = DateTime.Now,
            Direction = "Receive",
            PacketType = packetType,
            PacketSize = data.Length,
            PayloadPreview = GeneratePreview(data)
        };

        packetLog.Add(record);
    }

    // 패킷의 내용을 16진수로 미리보기
    private string GeneratePreview(byte[] data, int maxLength = 16)
    {
        int length = Math.Min(data.Length, maxLength);
        var hexParts = new List<string>();
        for (int i = 0; i < length; i++)
        {
            hexParts.Add(data[i].ToString("X2"));
        }

        string preview = string.Join(" ", hexParts);
        if (data.Length > maxLength)
        {
            preview += " ...";
        }

        return preview;
    }

    // 통계 정보 출력
    public void PrintStatistics()
    {
        Console.WriteLine("\n===== 네트워크 패킷 통계 =====");
        Console.WriteLine($"총 패킷 수: {packetLog.Count}");

        int sentCount = packetLog.Count(p => p.Direction == "Send");
        int receivedCount = packetLog.Count(p => p.Direction == "Receive");
        long sentBytes = packetLog.Where(p => p.Direction == "Send").Sum(p => p.PacketSize);
        long receivedBytes = packetLog.Where(p => p.Direction == "Receive").Sum(p => p.PacketSize);

        Console.WriteLine($"전송: {sentCount}개 ({sentBytes} 바이트)");
        Console.WriteLine($"수신: {receivedCount}개 ({receivedBytes} 바이트)");

        // 패킷 타입별 통계
        Console.WriteLine("\n패킷 타입별 통계:");
        var groupedByType = packetLog.GroupBy(p => p.PacketType);
        foreach (var group in groupedByType)
        {
            string typeName = packetTypeNames.ContainsKey(group.Key) 
                ? packetTypeNames[group.Key] 
                : $"Unknown({group.Key})";
            
            int count = group.Count();
            long bytes = group.Sum(p => (long)p.PacketSize);
            double avgSize = bytes / (double)count;

            Console.WriteLine($"  {typeName}: {count}개, {bytes} 바이트, 평균 {avgSize:F1} 바이트/개");
        }
    }

    // 최근 패킷들을 표 형식으로 출력
    public void PrintRecentPackets(int count = 20)
    {
        Console.WriteLine("\n===== 최근 패킷 기록 =====");
        Console.WriteLine($"{"시간",-12} {"방향",-8} {"패킷타입",-15} {"크기(B)",-8} {"내용"}");
        Console.WriteLine(new string('-', 80));

        var recent = packetLog.TakeLast(count);
        foreach (var record in recent)
        {
            string typeName = packetTypeNames.ContainsKey(record.PacketType)
                ? packetTypeNames[record.PacketType]
                : $"Unknown({record.PacketType})";

            Console.WriteLine(
                $"{record.Timestamp:HH:mm:ss.fff} {record.Direction,-8} {typeName,-15} {record.PacketSize,-8} {record.PayloadPreview}"
            );
        }
    }
}
```

이 분석 도구를 사용하면 게임 실행 중에 어떤 패킷들이 오고 가는지 추적할 수 있다. 문제가 발생했을 때, 최근 패킷 기록을 확인하여 어느 시점에서 뭔가 잘못되었는지 알 수 있다. 예를 들어, 플레이어의 위치가 이상하게 업데이트된다면, PositionUpdate 패킷들을 살펴보고 그 내용을 분석할 수 있다.

### 성능 모니터링
게임의 성능을 모니터링하는 것도 중요하다. 특히 네트워크 관련 성능은 게임의 전체 경험에 큰 영향을 미친다.  
  
```csharp
public class PerformanceMonitor
{
    public class PerformanceMetrics
    {
        public float FPS { get; set; }
        public float AverageFrameTime { get; set; }
        public float NetworkLatency { get; set; } // 밀리초
        public int PacketsPerSecond { get; set; }
        public float BytesPerSecond { get; set; }
        public int PendingMessages { get; set; }
    }

    private Stopwatch frameTimer = new Stopwatch();
    private Queue<float> frameTimings = new Queue<float>();
    private const int FRAME_SAMPLE_SIZE = 60;
    
    private Stopwatch networkTimer = new Stopwatch();
    private int packetsThisSecond = 0;
    private float bytesThisSecond = 0f;
    private float currentNetworkLatency = 0f;
    private int pendingMessages = 0;

    public PerformanceMonitor()
    {
        frameTimer.Start();
        networkTimer.Start();
    }

    // 매 프레임마다 호출
    public void RecordFrameTiming(float deltaTime)
    {
        frameTimings.Enqueue(deltaTime);
        if (frameTimings.Count > FRAME_SAMPLE_SIZE)
        {
            frameTimings.Dequeue();
        }
    }

    // 패킷 전송/수신 기록
    public void RecordNetworkPacket(int size)
    {
        packetsThisSecond++;
        bytesThisSecond += size;
    }

    // 네트워크 지연 기록 (핑)
    public void RecordNetworkLatency(float latencyMs)
    {
        // 간단한 이동 평균으로 지연 계산
        currentNetworkLatency = currentNetworkLatency * 0.9f + latencyMs * 0.1f;
    }

    public void SetPendingMessages(int count)
    {
        pendingMessages = count;
    }

    // 현재 성능 지표 반환
    public PerformanceMetrics GetMetrics()
    {
        float avgFrameTime = frameTimings.Count > 0 
            ? frameTimings.Average() 
            : 0;

        float fps = avgFrameTime > 0 ? 1f / avgFrameTime : 0;

        return new PerformanceMetrics
        {
            FPS = fps,
            AverageFrameTime = avgFrameTime * 1000f, // 밀리초로 변환
            NetworkLatency = currentNetworkLatency,
            PacketsPerSecond = packetsThisSecond,
            BytesPerSecond = bytesThisSecond,
            PendingMessages = pendingMessages
        };
    }

    // 성능 지표를 화면에 표시 (디버그용)
    public void DrawDebugInfo(SpriteBatch spriteBatch, SpriteFont font)
    {
        var metrics = GetMetrics();

        var debugText = new StringBuilder();
        debugText.AppendLine($"FPS: {metrics.FPS:F1}");
        debugText.AppendLine($"Frame Time: {metrics.AverageFrameTime:F2}ms");
        debugText.AppendLine($"Network Latency: {metrics.NetworkLatency:F1}ms");
        debugText.AppendLine($"Packets/sec: {metrics.PacketsPerSecond}");
        debugText.AppendLine($"Bytes/sec: {metrics.BytesPerSecond:F0}");
        debugText.AppendLine($"Pending Messages: {metrics.PendingMessages}");

        spriteBatch.DrawString(font, debugText.ToString(), new Vector2(10, 10), Color.White);
    }

    // 1초마다 호출하여 초 단위 통계 초기화
    public void ResetSecondlyStats()
    {
        packetsThisSecond = 0;
        bytesThisSecond = 0f;
    }
}
```

이 성능 모니터는 게임이 실시간으로 어떤 성능을 내고 있는지 추적한다. 특히 다음 정보들이 중요하다:

- **FPS**: 게임이 초당 몇 개의 프레임을 렌더링하고 있는가
- **Frame Time**: 평균적으로 한 프레임을 처리하는 데 걸리는 시간
- **Network Latency**: 클라이언트와 서버 사이의 왕복 지연 시간
- **Packets/sec**: 초당 전송되는 패킷 개수 (트래픽 최소화가 효과적인지 확인)
- **Pending Messages**: 처리 대기 중인 메시지 수 (처리 속도가 네트워크 속도를 따라잡지 못했을 때 증가)

### 로그 기록 시스템
네트워크 문제를 분석하기 위해서는 상세한 로그가 필요하다.

```csharp
public class GameLogger
{
    public enum LogLevel
    {
        Debug = 0,
        Info = 1,
        Warning = 2,
        Error = 3
    }

    private List<string> logBuffer = new List<string>();
    private LogLevel minLogLevel = LogLevel.Debug;
    private string logFilePath = "game_log.txt";
    private const int MAX_LOG_ENTRIES = 10000;

    public void SetLogLevel(LogLevel level)
    {
        minLogLevel = level;
    }

    public void Log(LogLevel level, string category, string message)
    {
        if (level < minLogLevel)
            return;

        string timestamp = DateTime.Now.ToString("HH:mm:ss.fff");
        string logEntry = $"[{timestamp}] [{level}] [{category}] {message}";

        logBuffer.Add(logEntry);

        // 버퍼 크기 제한
        if (logBuffer.Count > MAX_LOG_ENTRIES)
        {
            logBuffer.RemoveRange(0, logBuffer.Count / 2);
        }

        // 디버그 빌드에서는 콘솔에 출력
        #if DEBUG
        Console.WriteLine(logEntry);
        #endif
    }

    public void LogNetworkEvent(string eventType, string details)
    {
        Log(LogLevel.Info, "Network", $"{eventType}: {details}");
    }

    public void LogError(string category, string message)
    {
        Log(LogLevel.Error, category, message);
    }

    public void LogWarning(string category, string message)
    {
        Log(LogLevel.Warning, category, message);
    }

    // 파일에 로그 저장
    public void SaveToFile()
    {
        try
        {
            File.WriteAllLines(logFilePath, logBuffer);
            Console.WriteLine($"로그가 {logFilePath}에 저장되었습니다.");
        }
        catch (Exception ex)
        {
            Console.WriteLine($"로그 저장 실패: {ex.Message}");
        }
    }

    // 최근 로그 출력
    public void PrintRecentLogs(int count = 50)
    {
        Console.WriteLine("\n===== 최근 로그 =====");
        var recent = logBuffer.TakeLast(count);
        foreach (var log in recent)
        {
            Console.WriteLine(log);
        }
    }
}
```

이 로그 시스템을 사용하면 게임 실행 중에 발생하는 모든 중요 이벤트를 기록할 수 있다. 나중에 문제가 발생했을 때 로그 파일을 열어서 어떤 순서로 이벤트들이 발생했는지 확인할 수 있다.

## 종합 디버깅 예제
지금까지 배운 모든 도구들을 함께 사용하는 통합 예제를 살펴보자.  

```csharp
public class IntegratedDebugSystem
{
    private NetworkPacketAnalyzer packetAnalyzer;
    private PerformanceMonitor performanceMonitor;
    private GameLogger logger;
    private Stopwatch secondTimer = new Stopwatch();

    public IntegratedDebugSystem()
    {
        packetAnalyzer = new NetworkPacketAnalyzer();
        performanceMonitor = new PerformanceMonitor();
        logger = new GameLogger();
        logger.SetLogLevel(GameLogger.LogLevel.Info);
        secondTimer.Start();
    }

    // 게임의 Update 메서드에서 호출
    public void Update(float deltaTime, int pendingMessages)
    {
        performanceMonitor.RecordFrameTiming(deltaTime);
        performanceMonitor.SetPendingMessages(pendingMessages);

        // 1초마다 통계 초기화
        if (secondTimer.ElapsedMilliseconds >= 1000)
        {
            performanceMonitor.ResetSecondlyStats();
            secondTimer.Restart();

            // 1초마다 성능 정보 로깅
            var metrics = performanceMonitor.GetMetrics();
            if (metrics.FPS < 55) // FPS가 55 이하면 경고
            {
                logger.LogWarning("Performance", 
                    $"낮은 FPS: {metrics.FPS:F1} (Frame Time: {metrics.AverageFrameTime:F2}ms)");
            }

            if (metrics.NetworkLatency > 200) // 지연이 200ms 이상이면 경고
            {
                logger.LogWarning("Network", 
                    $"높은 지연: {metrics.NetworkLatency:F1}ms");
            }
        }
    }

    // 네트워크 패킷 전송/수신 시 호출
    public void RecordPacket(byte[] data, bool isSent)
    {
        if (isSent)
        {
            packetAnalyzer.RecordSentPacket(data);
        }
        else
        {
            packetAnalyzer.RecordReceivedPacket(data);
        }

        performanceMonitor.RecordNetworkPacket(data.Length);
    }

    // 핑 측정 결과 기록
    public void RecordPing(float latencyMs)
    {
        performanceMonitor.RecordNetworkLatency(latencyMs);
        logger.LogNetworkEvent("Ping", $"{latencyMs:F1}ms");
    }

    // 게임 종료 시 호출
    public void Shutdown()
    {
        logger.PrintRecentLogs(100);
        logger.SaveToFile();
        packetAnalyzer.PrintStatistics();
        packetAnalyzer.PrintRecentPackets(20);
    }
}
```

이 통합 시스템을 사용하면 게임 실행 중에 모든 성능 지표와 네트워크 활동을 추적할 수 있다. 문제가 발생하면 로그를 확인하여 그 원인을 파악할 수 있다.

## 학습 확인 과제
이번 장에서 배운 성능 최적화와 디버깅에 대해 이해했는지 확인하기 위해 다음 과제를 수행해보길 바란다.

### 과제 1: 델타 압축 구현
지금까지 만들어온 온라인 게임에 델타 압축을 적용하자. 플레이어의 위치 패킷을 다음과 같이 수정해야 한다:

1. 플레이어의 이전 위치를 기억하는 코드 추가
2. 위치 변화 감지 (threshold 값 결정)
3. 변경된 좌표만을 포함한 패킷 생성
4. 클라이언트에서 부분적인 위치 정보를 올바르게 파싱

구현 후 다음을 확인하자:
- 위치가 변하지 않았을 때 패킷 크기가 감소했는가?
- 작은 움직임이 무시되는가?
- 수신 측에서 위치 업데이트가 정상적으로 이루어지는가?

### 과제 2: 성능 모니터링 도구 추가
게임에 성능 모니터링 기능을 추가하자. 화면에 다음 정보를 실시간으로 표시해야 한다:

1. 현재 FPS
2. 평균 프레임 시간 (ms)
3. 네트워크 패킷 전송량 (packets/sec)
4. 대기 중인 메시지 개수

요구사항:
- UI 요소나 DirectX를 사용하지 말고, MonoGame의 SpriteBatch와 SpriteFont를 사용해서 텍스트로 표시
- 성능이 나쁠 때 경고 메시지 표시
- ESC 키를 누르면 디버그 정보 표시/숨김 토글

### 과제 3: 로그 시스템 구현
게임의 모든 중요한 사건을 로그하는 시스템을 구현하자:

1. 플레이어 접속/접속 해제 기록
2. 네트워크 오류 발생 시 기록
3. 비정상적인 패킷 수신 시 기록
4. 게임 종료 시 로그를 파일로 저장

요구사항:
- 로그 레벨(Debug, Info, Warning, Error)을 구분
- 타임스탬프 포함
- 각 로그에 카테고리(Network, Game, Error 등) 포함

### 과제 4: 패킷 분석 도구
네트워크 패킷을 분석하는 도구를 구현하고, 게임 종료 시 다음 통계를 출력하자:

1. 총 전송/수신 패킷 개수와 바이트 수
2. 패킷 타입별 통계
3. 평균 패킷 크기
4. 최근 20개 패킷의 상세 정보 (시간, 방향, 타입, 크기)

구현 후, 게임을 5분간 플레이한 후 생성된 통계를 분석해보자:
- 어떤 패킷 타입이 가장 많이 전송되었는가?
- 평균 패킷 크기는 얼마나 되는가?
- 초당 트래픽은 얼마나 되는가?

### 과제 5: 고정 타임스텝 검증
현재 게임이 고정 타임스텝으로 실행되고 있는지 검증하자:

1. `Update` 메서드가 호출될 때마다 `gameTime.ElapsedGameTime`을 기록
2. 1000개 샘플을 수집한 후 평균과 표준편차 계산
3. 표준편차가 충분히 작은지 확인 (0.001초 이하)

요구사항:
- 측정한 값을 파일로 저장
- 그래프나 표로 시각화

이 과제를 통해 게임이 정확히 고정된 시간 간격으로 업데이트되는지 확인할 수 있다. 만약 편차가 크면 타이밍 관련 버그의 원인이 될 수 있으므로 찾아서 수정해야 한다.
