# 7장. Synchronization Barriers

## 7.1 단계별 동기화의 필요성
현대 게임 서버는 복잡한 멀티스레드 환경에서 동작하며, 때로는 여러 스레드가 특정 시점에서 서로를 기다려야 하는 상황이 발생한다. 예를 들어, 게임의 한 틱(tick)을 처리할 때 모든 워커 스레드가 현재 단계의 작업을 완료한 후에야 다음 단계로 진행해야 하는 경우가 있다.

전통적으로 이런 문제는 세마포어나 이벤트 객체를 조합해서 해결했지만, Windows Vista(2006년)부터 도입된 **Synchronization Barrier**는 이러한 패턴을 효율적이고 간단하게 구현할 수 있게 해준다.
  

### 게임 서버에서의 활용 시나리오

```
게임 루프의 단계별 처리:
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│   입력 처리      │ -> │   로직 업데이트   │ -> │   네트워크 전송   │
│  (모든 스레드)   │    │  (모든 스레드)    │    │  (모든 스레드)    │
└─────────────────┘    └─────────────────┘    └─────────────────┘
       ↓                       ↓                       ↓
   [Barrier Wait]          [Barrier Wait]          [Barrier Wait]
```
  

## 7.2 게임 서버의 틱 시스템 구현

### 기본 API 소개
Synchronization Barrier의 핵심 API들을 살펴보겠다:

```cpp
#include <windows.h>
#include <synchapi.h>

// Barrier 초기화
BOOL InitializeSynchronizationBarrier(
    LPSYNCHRONIZATION_BARRIER lpBarrier,    // Barrier 구조체
    LONG lTotalThreads,                     // 참여할 스레드 수
    LONG lSpinCount                         // 스핀 카운트 (선택적)
);

// Barrier에서 대기
BOOL EnterSynchronizationBarrier(
    LPSYNCHRONIZATION_BARRIER lpBarrier,    // Barrier 구조체
    DWORD dwFlags                           // 플래그 (보통 0)
);

// Barrier 해제
BOOL DeleteSynchronizationBarrier(
    LPSYNCHRONIZATION_BARRIER lpBarrier
);
```
  
#### InitializeSynchronizationBarrier

```c
BOOL InitializeSynchronizationBarrier(
    LPSYNCHRONIZATION_BARRIER lpBarrier,
    LONG lTotalThreads,
    LONG lSpinCount
);
```

* `lpBarrier`: 장벽 객체를 담을 `SYNCHRONIZATION_BARRIER` 구조체 포인터다. 애플리케이션이 멤버를 직접 건드리지 않는 불투명(opaque) 구조체다.
* `lTotalThreads`: 이 장벽에 “참여하는” 스레드의 수다. 정확히 이 수만큼 `EnterSynchronizationBarrier`를 호출해야 장벽이 열리고, 그때 모든 스레드가 다음 단계로 진행한다.
* `lSpinCount`: 다른 스레드를 기다릴 때 먼저 “스핀(바쁜 대기)”을 몇 번 할지 정한다.

  * `-1` 이면 기본값으로 2000번 스핀한다.
  * 스핀 횟수를 초과하면 스레드는 블록(수면) 상태로 전환한다. 단, `EnterSynchronizationBarrier`를 `SYNCHRONIZATION_BARRIER_FLAGS_SPIN_ONLY`로 호출한 경우에는 초과하더라도 계속 스핀한다.

성공 시 `TRUE`, 실패 시 `FALSE`를 반환한다.

#### EnterSynchronizationBarrier

```c
BOOL EnterSynchronizationBarrier(
    LPSYNCHRONIZATION_BARRIER lpBarrier,
    DWORD dwFlags
);
```

모든 참여 스레드가 이 함수에 “들어올” 때까지 각 스레드를 대기시킨다. 마지막 스레드가 들어오면 장벽이 열리고, 모든 스레드가 반환되어 다음 페이즈로 넘어간다.

반환값에는 특징이 있다. **매 페이즈마다 정확히 한 스레드만 `TRUE`를 받고, 나머지 스레드는 `FALSE`를 받는다.** 이를 이용해 “페이즈 종료 시 단 한 번만 실행되어야 하는 정리/집계 작업”을 ‘리더’ 스레드에게 맡길 수 있다.

`dwFlags`로 기본 동작을 조절할 수 있다.

* `SYNCHRONIZATION_BARRIER_FLAGS_BLOCK_ONLY`: 즉시 블록 모드로 들어가 마지막 스레드를 기다린다.
* `SYNCHRONIZATION_BARRIER_FLAGS_SPIN_ONLY`: 블록으로 전환하지 않고 끝까지 스핀만 한다(스핀 카운트를 넘어도 계속 스핀한다).
* `SYNCHRONIZATION_BARRIER_FLAGS_NO_DELETE`: 이 장벽을 절대 삭제하지 않을 것이라면 모든 스레드가 이 플래그를 지정해 삭제-안전성 오버헤드를 건너뛰어 성능을 약간 더 낼 수 있다. 단, **모든 스레드가 이 플래그를 지정해야 하며**, 그렇게 해놓고 장벽을 삭제하면 교착이나 접근 위반을 유발할 수 있다.

일반적으로는 장벽이 스스로 적절히 스핀과 블록을 섞어 관리하도록 두고, 위 플래그는 실제 성능 측정으로 이점이 입증될 때만 쓰는 것이 권장된다.

#### DeleteSynchronizationBarrier

```c
BOOL DeleteSynchronizationBarrier(
    LPSYNCHRONIZATION_BARRIER lpBarrier
);
```

장벽 사용이 끝나면 자원을 정리한다. **`EnterSynchronizationBarrier`가 반환된 직후 호출해도 안전**하게 설계되어 있다. 함수 내부에서 모든 스레드가 해당 페이즈를 마쳤음을 보장한 뒤 해제하기 때문이다.

#### 동작 원리(요점)
* 초기화 시 지정한 `lTotalThreads` 만큼 스레드가 특정 지점에 모이면, 모두를 동시에 통과시키는 “문”을 여는 개념이다. 이때 페이즈가 1 증가하며, 다음 페이즈에서도 동일한 참여자 수를 기준으로 반복해서 사용할 수 있다.
* 내부적으로는 페이즈 카운터와 남은 인원 수 등을 관리해 마지막 도착 스레드가 다른 스레드를 깨우고 다음 라운드를 준비한다는 구현적 특징이 알려져 있다.

#### 사용 예시(C)

```c
#include <windows.h>
#include <stdio.h>

#define THREADS 4

SYNCHRONIZATION_BARRIER gBarrier;

DWORD WINAPI Worker(LPVOID lpParam)
{
    int id = (int)(SIZE_T)lpParam;

    // Phase 1: 준비 작업
    // ... 작업 ...
    BOOL isLeader = EnterSynchronizationBarrier(&gBarrier, 0);
    if (isLeader) {
        // 이 페이즈에서 단 한 번만 해야 하는 정리/집계
        // ... 집계 ...
    }

    // Phase 2: 다음 단계 작업
    // ... 작업 ...
    EnterSynchronizationBarrier(&gBarrier, 0);

    return 0;
}

int main(void)
{
    HANDLE th[THREADS];

    // -1: 기본 2000회 스핀 후 블록
    if (!InitializeSynchronizationBarrier(&gBarrier, THREADS, -1)) {
        return 1;
    }

    for (int i = 0; i < THREADS; ++i) {
        th[i] = CreateThread(NULL, 0, Worker, (LPVOID)(SIZE_T)i, 0, NULL);
    }

    WaitForMultipleObjects(THREADS, th, TRUE, INFINITE);

    // EnterSynchronizationBarrier가 반환된 뒤 곧바로 삭제해도 안전함
    DeleteSynchronizationBarrier(&gBarrier);

    for (int i = 0; i < THREADS; ++i) CloseHandle(th[i]);
    return 0;
}
```

* 페이즈마다 한 스레드만 `TRUE`를 받는 점을 이용해 리더 전용 코드를 배치했다.
* 스핀 전략은 기본(-1)으로 두고, 특별한 근거가 있을 때에만 플래그로 강제한다는 원칙을 따랐다.

#### 실무 팁
1. **참여자 수 불변 원칙**을 지킨다. `lTotalThreads`로 정한 참여 스레드 수가 페이즈 동안 변하면 안 된다. 누락되면 장벽에서 영원히 대기하게 된다.
2. **생명주기 관리**를 철저히 한다. 장벽 메모리는 모든 참여 스레드가 해당 페이즈를 빠져나온 뒤에만 재초기화·해제해야 한다. 문서가 `EnterSynchronizationBarrier` 직후의 삭제를 안전하게 보장하므로, 그 타이밍을 지키면 된다.
3. **성능 조정**은 계측 기반으로 한다. `SPIN_ONLY`는 짧은 대기에서 문맥 전환 오버헤드를 줄이는 대신 CPU를 계속 붙잡고, `BLOCK_ONLY`는 반대로 CPU는 아끼지만 깨우기·스케줄링 오버헤드가 든다. 실제 워크로드를 측정해 이득이 있을 때만 강제한다.
4. **영구 장벽**이라면 `NO_DELETE`를 고려하되, 모든 스레드가 동일 플래그를 사용하고 실제로 삭제하지 않겠다는 전제가 필요하다. 일부만 지정하거나 나중에 삭제하면 위험하다.
  
#### 다른 동기화 도구와의 차이
* 크리티컬 섹션이나 뮤텍스는 “단일 소유권(뮤추얼 익스클루전)”을 보장하지만, **장벽은 다수 스레드가 특정 경계에서 ‘동시 출발’을 맞추는 데** 특화되어 있다. 예를 들어 단계적 파이프라인, 시뮬레이션의 타임스텝, 데이터 분할 후 집계 같은 패턴에서 유용하다.


### 실제 구현 예제

```cpp
#include <windows.h>
#include <synchapi.h>
#include <iostream>
#include <thread>
#include <vector>
#include <chrono>
#include <atomic>

class GameTickSystem {
private:
    SYNCHRONIZATION_BARRIER inputBarrier_;
    SYNCHRONIZATION_BARRIER logicBarrier_;
    SYNCHRONIZATION_BARRIER networkBarrier_;
    
    std::atomic<bool> running_{true};
    std::atomic<int> currentTick_{0};
    
    static constexpr int WORKER_THREAD_COUNT = 4;
    
public:
    GameTickSystem() {
        // 각 단계별로 배리어 초기화 (4개 워커 스레드 + 1개 메인 스레드)
        if (!InitializeSynchronizationBarrier(&inputBarrier_, WORKER_THREAD_COUNT, 0)) {
            throw std::runtime_error("Failed to initialize input barrier");
        }
        
        if (!InitializeSynchronizationBarrier(&logicBarrier_, WORKER_THREAD_COUNT, 0)) {
            throw std::runtime_error("Failed to initialize logic barrier");
        }
        
        if (!InitializeSynchronizationBarrier(&networkBarrier_, WORKER_THREAD_COUNT, 0)) {
            throw std::runtime_error("Failed to initialize network barrier");
        }
    }
    
    ~GameTickSystem() {
        DeleteSynchronizationBarrier(&inputBarrier_);
        DeleteSynchronizationBarrier(&logicBarrier_);
        DeleteSynchronizationBarrier(&networkBarrier_);
    }
    
    // 워커 스레드가 실행할 메인 루프
    void WorkerThreadMain(int threadId) {
        while (running_.load()) {
            int tick = currentTick_.load();
            
            // 1단계: 입력 처리
            ProcessInput(threadId, tick);
            EnterSynchronizationBarrier(&inputBarrier_, 0);
            
            // 2단계: 게임 로직 업데이트
            ProcessGameLogic(threadId, tick);
            EnterSynchronizationBarrier(&logicBarrier_, 0);
            
            // 3단계: 네트워크 전송
            ProcessNetworking(threadId, tick);
            EnterSynchronizationBarrier(&networkBarrier_, 0);
            
            // 모든 단계 완료 후 잠시 대기
            std::this_thread::sleep_for(std::chrono::milliseconds(16)); // ~60FPS
        }
    }
    
private:
    void ProcessInput(int threadId, int tick) {
        std::cout << "[Tick " << tick << "] Thread " << threadId 
                  << ": Processing input...\n";
        
        // 시뮬레이션: 입력 처리 작업
        std::this_thread::sleep_for(std::chrono::milliseconds(2));
    }
    
    void ProcessGameLogic(int threadId, int tick) {
        std::cout << "[Tick " << tick << "] Thread " << threadId 
                  << ": Processing game logic...\n";
        
        // 시뮬레이션: 게임 로직 처리
        std::this_thread::sleep_for(std::chrono::milliseconds(5));
    }
    
    void ProcessNetworking(int threadId, int tick) {
        std::cout << "[Tick " << tick << "] Thread " << threadId 
                  << ": Processing networking...\n";
        
        // 시뮬레이션: 네트워크 처리
        std::this_thread::sleep_for(std::chrono::milliseconds(3));
    }
    
public:
    void Run() {
        std::vector<std::thread> workers;
        
        // 워커 스레드들 시작
        for (int i = 0; i < WORKER_THREAD_COUNT; ++i) {
            workers.emplace_back(&GameTickSystem::WorkerThreadMain, this, i);
        }
        
        // 메인 스레드에서 틱 카운터 증가
        for (int tick = 0; tick < 10; ++tick) {
            currentTick_.store(tick);
            std::cout << "\n=== TICK " << tick << " START ===\n";
            
            std::this_thread::sleep_for(std::chrono::milliseconds(50));
        }
        
        // 종료 신호
        running_.store(false);
        
        // 워커 스레드들 종료 대기
        for (auto& worker : workers) {
            worker.join();
        }
    }
};
```  
   
이 코드는 **여러 스레드를 사용하여 게임 루프의 각 단계를 병렬로 처리**하되, 모든 스레드가 **각 단계를 동시에 완료하도록 동기화**하는 '게임 틱 시스템'을 구현한 것이다.

주요 목적은 **데이터 일관성**을 유지하는 것이다. 예를 들어, 모든 스레드가 '입력 처리'를 끝내기 전에 일부 스레드가 '게임 로직' 계산을 시작하면, 해당 틱(tick)의 데이터가 꼬일 수 있다. 이 코드는 이런 문제를 방지한다.

#### 코드의 목적
게임 루프는 보통 **"입력 → 로직 → 렌더링/네트워크"** 순서로 진행된다. 이 코드는 이 작업을 4개의 '워커 스레드(Worker Thread)'가 나누어 처리하도록 설계되었다.

하지만 여러 스레드가 동시에 작업을 처리하면, 어떤 스레드는 '입력'을 끝냈는데 다른 스레드는 아직 '입력'을 처리 중인 상태가 발생한다.

이 코드의 목적은 `SYNCHRONIZATION_BARRIER` (동기화 배리어)라는 장치를 사용해, **모든 스레드가 '입력' 단계를 마칠 때까지 대기**하게 만들고, 모두 마치면 **동시에 '로직' 단계로 넘어가도록** 강제하는 것이다. '로직' → '네트워크' 단계로 넘어갈 때도 마찬가지이다.


#### 동작 방식
코드는 **1개의 '메인 스레드'**와 **4개의 '워커 스레드'**로 나뉘어 동작한다.

#### 1. 동기화 배리어 (SYNCHRONIZATION_BARRIER)

* `GameTickSystem` 생성자에서 `inputBarrier_`, `logicBarrier_`, `networkBarrier_` 3개의 배리어가 생성된다.
* `InitializeSynchronizationBarrier(&inputBarrier_, WORKER_THREAD_COUNT, 0)`는 "이 배리어는 4개의 스레드가 도달할 때까지 기다린다"는 의미이다.
* 배리어는 일종의 "만남의 장소" 이다. 4명이 다 모이기 전까지는 먼저 온 3명은 문 앞에서 기다려야 하며, 4명째가 도착하면 문이 열리고 모두 동시에 다음 장소로 출발할 수 있다.

#### 2. 메인 스레드 (Run() 함수)
`Run` 함수를 실행하는 메인 스레드는 작업자가 아니라 **'감독관'** 역할을 한다.

1.  `WORKER_THREAD_COUNT` (4개)의 워커 스레드를 생성하여 `WorkerThreadMain` 함수를 실행시킨다.
2.  자신은 10번의 틱(tick) 동안 루프를 돈다.
3.  매 틱마다 `currentTick_.store(tick)`을 통해 **"지금은 Tick 0이야", "지금은 Tick 1이야"** 하고 워커 스레드들에게 현재 진행할 틱 번호를 알려준다.
4.  10번의 틱이 끝나면 `running_.store(false)`로 워커 스레드들에게 **"이제 일 그만해"**라고 신호를 보낸다.
5.  모든 워커 스레드가 종료될 때까지 `worker.join()`으로 기다린다.

#### 3. 워커 스레드 (WorkerThreadMain() 함수)
4개의 워커 스레드는 `running_`이 `true`인 동안 무한 루프를 돈다.

**Tick 0을 예로 들면, 4개의 스레드는 다음과 같이 동작한다.**

1.  **입력 단계:**
    * 4개 스레드 모두 `ProcessInput(threadId, 0)`을 실행한다. (각자 2ms 소요)
    * 일을 마친 스레드는 `EnterSynchronizationBarrier(&inputBarrier_, 0)`에서 대기한다.
    * 먼저 끝난 3개의 스레드는 여기서 멈춰있고, 마지막 4번째 스레드가 이 지점에 도달하면 배리어가 열린다.
    * **[결과]** 4개 스레드 모두 입력 처리가 완료됨이 보장된다.

2.  **로직 단계:**
    * 4개 스레드 모두 `ProcessGameLogic(threadId, 0)`을 실행한다. (각자 5ms 소요)
    * 일을 마친 스레드는 `EnterSynchronizationBarrier(&logicBarrier_, 0)`에서 대기한다.
    * 마찬가지로 4번째 스레드가 도달하면 배리어가 열린다.
    * **[결과]** 4개 스레드 모두 Tick 0의 입력을 기반으로 로직 처리가 완료됨이 보장된다.

3.  **네트워크 단계:**
    * 4개 스레드 모두 `ProcessNetworking(threadId, 0)`을 실행한다. (각자 3ms 소요)
    * 일을 마친 스레드는 `EnterSynchronizationBarrier(&networkBarrier_, 0)`에서 대기한다.
    * 4번째 스레드가 도달하면 배리어가 열린다.
    * **[결과]** 4개 스레드 모두 Tick 0의 네트워크 처리가 완료된다.

4.  **반복:**
    * 모든 단계를 마친 스레드들은 16ms 대기 후, `while` 루프의 처음으로 돌아간다.
    * 이때쯤 메인 스레드가 `currentTick_`를 1로 변경했을 것이므로, 워커 스레드들은 `tick = 1`을 읽어 위 과정을 다시 반복한다.

이러한 구조는 **"틱 기반 병렬 처리"**에서 데이터가 섞이는 것을 방지하는 고전적이고 확실한 동기화 기법이다.


## 7.3 실전 예제: 멀티스레드 게임 로직 처리
실제 게임 서버에서 자주 발생하는 시나리오를 바탕으로 더 복잡한 예제를 구현해보겠다.

### 파티클 시스템과 물리 시뮬레이션

```cpp
#include <windows.h>
#include <vector>
#include <random>
#include <memory>

class Particle {
public:
    float x, y, z;
    float vx, vy, vz;
    float life;
    bool active;
    
    Particle() : x(0), y(0), z(0), vx(0), vy(0), vz(0), life(1.0f), active(false) {}
};

class ParticleSystem {
private:
    static constexpr int MAX_PARTICLES = 10000;
    static constexpr int THREAD_COUNT = 4;
    
    std::vector<Particle> particles_;
    SYNCHRONIZATION_BARRIER updateBarrier_;
    SYNCHRONIZATION_BARRIER collisionBarrier_;
    SYNCHRONIZATION_BARRIER renderBarrier_;
    
    std::atomic<bool> running_{true};
    
public:
    ParticleSystem() : particles_(MAX_PARTICLES) {
        InitializeSynchronizationBarrier(&updateBarrier_, THREAD_COUNT, 100);
        InitializeSynchronizationBarrier(&collisionBarrier_, THREAD_COUNT, 100);
        InitializeSynchronizationBarrier(&renderBarrier_, THREAD_COUNT, 100);
        
        InitializeParticles();
    }
    
    ~ParticleSystem() {
        DeleteSynchronizationBarrier(&updateBarrier_);
        DeleteSynchronizationBarrier(&collisionBarrier_);
        DeleteSynchronizationBarrier(&renderBarrier_);
    }
    
    void WorkerThread(int threadId) {
        int startIdx = (MAX_PARTICLES / THREAD_COUNT) * threadId;
        int endIdx = (threadId == THREAD_COUNT - 1) ? 
                     MAX_PARTICLES : startIdx + (MAX_PARTICLES / THREAD_COUNT);
        
        while (running_.load()) {
            // 1단계: 위치 업데이트
            UpdateParticlePositions(startIdx, endIdx);
            EnterSynchronizationBarrier(&updateBarrier_, 0);
            
            // 2단계: 충돌 검사 (모든 파티클 위치 업데이트 완료 후)
            CheckCollisions(startIdx, endIdx);
            EnterSynchronizationBarrier(&collisionBarrier_, 0);
            
            // 3단계: 렌더링 데이터 준비 (모든 충돌 처리 완료 후)
            PrepareRenderData(startIdx, endIdx);
            EnterSynchronizationBarrier(&renderBarrier_, 0);
            
            std::this_thread::sleep_for(std::chrono::milliseconds(16)); // 60FPS
        }
    }
    
private:
    void InitializeParticles() {
        std::random_device rd;
        std::mt19937 gen(rd());
        std::uniform_real_distribution<float> pos(-100.0f, 100.0f);
        std::uniform_real_distribution<float> vel(-10.0f, 10.0f);
        
        for (auto& particle : particles_) {
            particle.x = pos(gen);
            particle.y = pos(gen);
            particle.z = pos(gen);
            particle.vx = vel(gen);
            particle.vy = vel(gen);
            particle.vz = vel(gen);
            particle.life = 1.0f;
            particle.active = true;
        }
    }
    
    void UpdateParticlePositions(int start, int end) {
        const float deltaTime = 0.016f; // 60FPS
        
        for (int i = start; i < end; ++i) {
            if (!particles_[i].active) continue;
            
            // 위치 업데이트
            particles_[i].x += particles_[i].vx * deltaTime;
            particles_[i].y += particles_[i].vy * deltaTime;
            particles_[i].z += particles_[i].vz * deltaTime;
            
            // 중력 적용
            particles_[i].vy -= 9.8f * deltaTime;
            
            // 라이프 감소
            particles_[i].life -= deltaTime;
            if (particles_[i].life <= 0.0f) {
                particles_[i].active = false;
            }
        }
    }
    
    void CheckCollisions(int start, int end) {
        // 간단한 경계 충돌 검사
        for (int i = start; i < end; ++i) {
            if (!particles_[i].active) continue;
            
            // 바닥 충돌
            if (particles_[i].y < -50.0f) {
                particles_[i].y = -50.0f;
                particles_[i].vy = -particles_[i].vy * 0.8f; // 탄성 계수
            }
            
            // 벽 충돌
            if (particles_[i].x < -100.0f || particles_[i].x > 100.0f) {
                particles_[i].vx = -particles_[i].vx * 0.9f;
                particles_[i].x = std::clamp(particles_[i].x, -100.0f, 100.0f);
            }
        }
    }
    
    void PrepareRenderData(int start, int end) {
        // 렌더링을 위한 데이터 정렬, 컬링 등
        // 실제로는 GPU 버퍼 업데이트 등의 작업
        for (int i = start; i < end; ++i) {
            if (!particles_[i].active) continue;
            
            // 뷰 프러스텀 컬링 시뮬레이션
            float distance = sqrt(particles_[i].x * particles_[i].x + 
                                particles_[i].y * particles_[i].y + 
                                particles_[i].z * particles_[i].z);
            
            if (distance > 200.0f) {
                // 너무 멀리 있는 파티클은 비활성화
                particles_[i].active = false;
            }
        }
    }
    
public:
    void Run() {
        std::vector<std::thread> workers;
        
        for (int i = 0; i < THREAD_COUNT; ++i) {
            workers.emplace_back(&ParticleSystem::WorkerThread, this, i);
        }
        
        // 시뮬레이션 실행
        std::this_thread::sleep_for(std::chrono::seconds(5));
        running_.store(false);
        
        for (auto& worker : workers) {
            worker.join();
        }
    }
};
```

### 스핀 카운트 최적화
Synchronization Barrier는 스핀 카운트 매개변수를 통해 성능을 조정할 수 있다:

```cpp
class OptimizedBarrierSystem {
private:
    SYNCHRONIZATION_BARRIER lowLatencyBarrier_;
    SYNCHRONIZATION_BARRIER highThroughputBarrier_;
    
public:
    OptimizedBarrierSystem(int threadCount) {
        // 낮은 지연시간용 (더 많은 스핀)
        InitializeSynchronizationBarrier(&lowLatencyBarrier_, threadCount, 2000);
        
        // 높은 처리량용 (적은 스핀, 빠른 컨텍스트 스위치)
        InitializeSynchronizationBarrier(&highThroughputBarrier_, threadCount, 100);
    }
    
    // 실시간 처리가 중요한 작업용
    void FastPath() {
        EnterSynchronizationBarrier(&lowLatencyBarrier_, 0);
    }
    
    // 배치 처리용
    void BatchPath() {
        EnterSynchronizationBarrier(&highThroughputBarrier_, 0);
    }
};
```

### 동적 스레드 수 조정
게임 서버의 부하에 따라 동적으로 스레드 수를 조정하는 시스템:

```cpp
class AdaptiveBarrierSystem {
private:
    std::unique_ptr<SYNCHRONIZATION_BARRIER> currentBarrier_;
    std::atomic<int> activeThreadCount_{0};
    std::mutex reconfigMutex_;
    
public:
    void ReconfigureBarrier(int newThreadCount) {
        std::lock_guard<std::mutex> lock(reconfigMutex_);
        
        // 기존 배리어 해제
        if (currentBarrier_) {
            DeleteSynchronizationBarrier(currentBarrier_.get());
        }
        
        // 새 배리어 생성
        currentBarrier_ = std::make_unique<SYNCHRONIZATION_BARRIER>();
        if (!InitializeSynchronizationBarrier(currentBarrier_.get(), newThreadCount, 500)) {
            throw std::runtime_error("Failed to reconfigure barrier");
        }
        
        activeThreadCount_.store(newThreadCount);
    }
    
    bool WaitAtBarrier() {
        if (currentBarrier_) {
            return EnterSynchronizationBarrier(currentBarrier_.get(), 0);
        }
        return false;
    }
};
```

### 성능 측정과 분석

```cpp
class BarrierPerformanceAnalyzer {
private:
    struct PerformanceMetrics {
        std::chrono::high_resolution_clock::time_point startTime;
        std::chrono::high_resolution_clock::time_point endTime;
        int threadId;
        int phase;
    };
    
    std::vector<PerformanceMetrics> metrics_;
    std::mutex metricsMutex_;
    
public:
    void RecordPhaseStart(int threadId, int phase) {
        std::lock_guard<std::mutex> lock(metricsMutex_);
        metrics_.push_back({
            std::chrono::high_resolution_clock::now(),
            {},
            threadId,
            phase
        });
    }
    
    void RecordPhaseEnd(int threadId, int phase) {
        auto endTime = std::chrono::high_resolution_clock::now();
        
        std::lock_guard<std::mutex> lock(metricsMutex_);
        for (auto& metric : metrics_) {
            if (metric.threadId == threadId && metric.phase == phase && 
                metric.endTime == std::chrono::high_resolution_clock::time_point{}) {
                metric.endTime = endTime;
                break;
            }
        }
    }
    
    void PrintAnalysis() {
        std::lock_guard<std::mutex> lock(metricsMutex_);
        
        std::cout << "\n=== 성능 분석 결과 ===\n";
        
        std::map<int, std::vector<double>> phaseTimings;
        
        for (const auto& metric : metrics_) {
            if (metric.endTime != std::chrono::high_resolution_clock::time_point{}) {
                auto duration = std::chrono::duration_cast<std::chrono::microseconds>(
                    metric.endTime - metric.startTime).count();
                phaseTimings[metric.phase].push_back(duration / 1000.0); // ms로 변환
            }
        }
        
        for (const auto& [phase, timings] : phaseTimings) {
            double sum = std::accumulate(timings.begin(), timings.end(), 0.0);
            double avg = sum / timings.size();
            
            auto minMax = std::minmax_element(timings.begin(), timings.end());
            
            std::cout << "Phase " << phase << ": "
                      << "평균=" << std::fixed << std::setprecision(2) << avg << "ms, "
                      << "최소=" << *minMax.first << "ms, "
                      << "최대=" << *minMax.second << "ms\n";
        }
    }
};
```

### 메모리 접근 패턴 최적화

```cpp
// 캐시 친화적인 데이터 구조
struct alignas(64) ThreadLocalData {  // 캐시 라인 크기에 맞춤
    int threadId;
    int processedItems;
    char padding[56];  // false sharing 방지
};

class CacheOptimizedBarrierSystem {
private:
    std::vector<ThreadLocalData> threadData_;
    SYNCHRONIZATION_BARRIER barrier_;
    
public:
    CacheOptimizedBarrierSystem(int threadCount) 
        : threadData_(threadCount) {
        InitializeSynchronizationBarrier(&barrier_, threadCount, 1000);
        
        for (int i = 0; i < threadCount; ++i) {
            threadData_[i].threadId = i;
            threadData_[i].processedItems = 0;
        }
    }
    
    void WorkerProcess(int threadId) {
        // 각 스레드는 자신만의 캐시 라인에서 작업
        auto& localData = threadData_[threadId];
        
        while (true) {
            // 작업 수행
            ProcessWork(localData);
            
            // 배리어에서 동기화
            EnterSynchronizationBarrier(&barrier_, 0);
            
            // 통계 업데이트 (배리어 이후에 안전하게)
            UpdateGlobalStatistics(localData);
        }
    }
    
private:
    void ProcessWork(ThreadLocalData& data) {
        // CPU 집약적 작업 시뮬레이션
        for (int i = 0; i < 1000; ++i) {
            data.processedItems++;
        }
    }
    
    void UpdateGlobalStatistics(const ThreadLocalData& data) {
        // 전역 통계 업데이트
        static std::atomic<long long> totalProcessed{0};
        totalProcessed.fetch_add(data.processedItems);
    }
};
```
  


## 마무리
Synchronization Barrier는 게임 서버의 단계별 처리에서 매우 유용한 도구이다. 특히 다음과 같은 상황에서 효과적이다:

1. **프레임 기반 시뮬레이션**: 모든 객체의 상태를 동시에 업데이트해야 하는 경우
2. **파이프라인 처리**: 여러 단계로 나뉜 작업을 순서대로 처리해야 하는 경우  
3. **배치 작업**: 모든 작업 단위가 완료된 후 다음 배치로 진행해야 하는 경우

다음 장에서는 더욱 세밀한 동기화 제어가 가능한 WaitOnAddress API와 Lock-Free 프로그래밍 기법에 대해 살펴보겠다.
