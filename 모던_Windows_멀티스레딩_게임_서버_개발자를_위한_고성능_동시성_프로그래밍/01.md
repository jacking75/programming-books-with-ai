## 1.1 왜 모던 Windows API인가?
2006년 Windows Vista의 등장과 함께 Microsoft는 멀티스레딩 분야에 혁신적인 변화를 가져왔다. 기존의 동기화 프리미티브들이 가지고 있던 성능상의 한계를 극복하고, 멀티코어 시대에 맞는 새로운 동기화 메커니즘들을 도입했다.
  
<pre>  
Windows 멀티스레딩 API 진화
    
    1993-2005 : Win32 초기 API
              : Critical Section
              : Mutex, Semaphore
              : Events
    
    2006-2009 : Vista 혁신
              : SRW Locks
              : Condition Variables
              : One-Time Initialization
    
    2009-2012 : 7/Server 2008 R2
              : Thread Pool API 2.0
              : Synchronization Barriers
    
    2012-현재  : 8/Server 2012+
              : WaitOnAddress
              : UMS (Limited)
              : Performance 최적화
</pre>

게임 서버 개발에서 이러한 모던 API들이 중요한 이유는 다음과 같다:

**성능의 혁신적 개선**: 기존 CRITICAL_SECTION 대비 SRW Lock은 약 2-3배의 성능 향상을 보여준다. 특히 읽기 중심의 워크로드에서는 더욱 극적인 차이를 보인다.

**메모리 효율성**: 새로운 동기화 프리미티브들은 더 적은 메모리를 사용한다. SRW Lock은 단 하나의 포인터 크기만 필요하며, 이는 게임 서버의 메모리 사용량을 크게 줄여준다.

**확장성**: 멀티코어 환경에서의 확장성이 대폭 개선되었다. 코어 수가 증가해도 성능 저하가 최소화된다.
  

## 1.2 레거시 동기화 기법의 한계
기존의 Win32 동기화 API들이 가지고 있던 근본적인 문제점들을 살펴보겠다:

```cpp
// 레거시 방식: CRITICAL_SECTION의 문제점
class LegacyPlayerManager {
private:
    CRITICAL_SECTION cs;
    std::unordered_map<int, Player> players;
    
public:
    LegacyPlayerManager() {
        InitializeCriticalSection(&cs);
    }
    
    // 문제 1: 읽기 작업도 배타적 잠금
    Player* GetPlayer(int playerId) {
        EnterCriticalSection(&cs);  // 읽기만 해도 exclusive lock!
        auto it = players.find(playerId);
        Player* result = (it != players.end()) ? &it->second : nullptr;
        LeaveCriticalSection(&cs);
        return result;
    }
    
    // 문제 2: 생산자-소비자 패턴에서 비효율적 대기
    void WaitForNewPlayer() {
        while (true) {
            EnterCriticalSection(&cs);
            if (!players.empty()) {
                LeaveCriticalSection(&cs);
                break;
            }
            LeaveCriticalSection(&cs);
            Sleep(1);  // 폴링 방식의 비효율적 대기
        }
    }
};
```

```
📊 레거시 API의 성능 문제점

CRITICAL_SECTION 구조:
┌─────────────────────────────────────┐
│ 🔒 Exclusive Lock Only              │
│                                     │
│ Reader 1 ──┐                       │
│            ├─── 🚫 Serialized      │
│ Reader 2 ──┤                       │
│            │                       │
│ Reader 3 ──┘                       │
│                                     │
│ 결과: 읽기 작업도 순차 처리         │
└─────────────────────────────────────┘

메모리 오버헤드:
Critical Section: 24-40 bytes (플랫폼별)
Event Objects: 커널 오브젝트 생성 비용
Mutex: 커널 전환 비용
```

**주요 한계점들:**

1. **읽기-쓰기 구분 없음**: CRITICAL_SECTION은 읽기와 쓰기를 구분하지 않아 불필요한 직렬화가 발생한다.

2. **커널 전환 비용**: Event, Mutex 등은 커널 오브젝트로 구현되어 시스템 콜 오버헤드가 크다.

3. **스핀락 부재**: 짧은 대기 시간에 대한 최적화가 부족하다.

4. **생산자-소비자 패턴의 비효율성**: 조건 변수가 없어 폴링 방식의 대기가 필요하다.
  

## 1.3 게임 서버의 동시성 요구사항
온라인 게임 서버는 다음과 같은 특수한 동시성 요구사항을 가진다:

```
🎮 게임 서버 동시성 특성

높은 동시 접속자 수:
┌─────────────────────────────────────────┐
│     🏠 Game World                       │
│                                         │
│  👤 Player 1    👤 Player 2            │
│      │             │                   │
│      ▼             ▼                   │
│  🔄 Thread 1    🔄 Thread 2            │
│      │             │                   │
│      └─────┬───────┘                   │
│            ▼                           │
│        📊 Shared State                 │
│     (Player Data, Game Logic)          │
└─────────────────────────────────────────┘

실시간 처리 요구사항:
- 16-33ms 틱 레이트 (60-30 FPS)
- 마이크로초 단위 지연시간 민감성
- 예측 가능한 성능
```

**게임 서버의 핵심 동시성 패턴:**

```cpp
// 전형적인 게임 서버 워크로드 패턴
class GameServerWorkload {
public:
    // 패턴 1: 읽기 중심 (90% 읽기, 10% 쓰기)
    void ProcessGameTick() {
        // 대부분 읽기 작업
        for (auto& player : players) {
            ReadPlayerState(player.id);     // 읽기
            CalculateGameLogic(player);     // 읽기
            UpdatePosition(player);         // 쓰기 (가끔)
        }
    }
    
    // 패턴 2: 이벤트 기반 처리
    void HandlePlayerActions() {
        while (true) {
            auto action = actionQueue.WaitForAction();  // 조건 대기
            ProcessAction(action);
        }
    }
    
    // 패턴 3: 배치 작업
    void BatchDatabaseUpdates() {
        barrier.Wait();  // 모든 스레드 동기화
        FlushToDatabase();
    }
};
```

**성능 요구사항:**

- **지연시간**: 1ms 이하의 응답시간
- **처리량**: 초당 수만 건의 동시 요청 처리
- **일관성**: 게임 상태의 강한 일관성 보장
- **확장성**: 코어 수 증가에 따른 선형적 성능 향상

**컴파일러 최적화 설정:**

```cpp
// 성능 최적화를 위한 컴파일러 힌트
#pragma once

// 분기 예측 힌트 (C++23)
#define LIKELY   [[likely]]
#define UNLIKELY [[unlikely]]

// 인라인 힌트
#define FORCE_INLINE __forceinline
#define NO_INLINE __declspec(noinline)

// 메모리 정렬
#define CACHE_ALIGNED alignas(std::hardware_destructive_interference_size)

// 예제 사용법
class CACHE_ALIGNED HighPerformanceCounter {
    std::atomic<uint64_t> counter{0};
    
public:
    FORCE_INLINE uint64_t increment() noexcept {
        return counter.fetch_add(1, std::memory_order_relaxed);
    }
    
    FORCE_INLINE uint64_t get() const noexcept LIKELY {
        return counter.load(std::memory_order_acquire);
    }
};
```
  
`alignas(std::hardware_destructive_interference_size)`를 클래스에 붙이면, 이 타입의 **정렬 단위(alignment)**를 CPU의 캐시 라인 간섭을 피하기 위한 보수적 크기(보통 64B, 일부 아키텍처 128B 등)로 올리게 된다. 그 결과는 대략 다음과 같다.

* 이 타입의 **각 객체 시작 주소가 해당 크기 경계에 맞춰진다**.
* `sizeof(T)`는 `alignof(T)`의 배수가 되어, 보통 **객체 한 개가 최소 한 캐시 라인 크기 이상을 차지**하게 된다.
* 따라서 `std::vector<HighPerformanceCounter>` 같은 연속 저장소에 여러 개를 둘 때 **서로 다른 객체가 같은 캐시 라인을 공유할 가능성이 크게 줄어** 다중 스레드에서의 **false sharing**(한 라인을 여러 코어가 번갈아 쓰느라 불필요한 캐시 무효화가 발생하는 현상)을 완화한다.

주의점도 있다.

* 이것은 **정렬을 보장할 뿐, “주변 메모리가 비어 있다”는 것을 보장하지는 않는다**. 다만 `sizeof(T) ≥ alignof(T)`이므로 연속 컨테이너에서는 사실상 한 객체가 한 라인을 차지하는 효과가 난다.
* 이 타입을 **다른 큰 구조체의 멤버**로 넣는 경우, 해당 멤버 자체는 라인 경계에 놓이지만 **다른 멤버와의 간격까지 자동으로 라인 단위로 벌려주지는 않는다**. 멤버 간 false sharing을 확실히 피하려면 각 멤버에 동일 정렬을 주거나 패딩을 명시적으로 넣어야 한다.
* **메모리 사용량이 크게 증가**한다. 카운터만 수천 개 만드는 경우 메모리·캐시 압박이 커질 수 있다.
* `std::hardware_destructive_interference_size`는 **구현 정의 상수**로, 실제 하드웨어/표준 라이브러리에 따라 값이 다를 수 있다.

요약하면, 이 정렬 지시는 `HighPerformanceCounter` 객체들 사이의 false sharing 가능성을 낮춰 **멀티스레드 증가/조회 경합 시 성능 저하를 줄이는 데 도움**이 된다. 반면 메모리 소비가 늘고, 복합 구조에서 완전한 분리는 별도 패딩/정렬 설계가 필요하다.
