# Day 3: 함수와 에러 처리 기초
오늘은 Rust에서 함수를 정의하고 사용하는 방법을 배우고, 더 나아가 프로그램에서 발생할 수 있는 에러를 우아하게 처리하는 방법을 익힌다. Day 2에서 만든 키-값 설정 파서에 에러 처리를 추가하면서 실전 감각을 키워본다.

## 함수 정의와 반환값
Rust에서 함수는 `fn` 키워드로 정의한다. 함수는 프로그램을 논리적인 단위로 나누어 재사용성과 가독성을 높이는 핵심 도구다. 먼저 간단한 예제부터 시작해본다.

```rust
fn greet(name: &str) {
    println!("안녕하세요, {}님!", name);
}

fn add(a: i32, b: i32) -> i32 {
    a + b  // 세미콜론이 없으면 이 값이 반환된다
}

fn main() {
    greet("철수");
    let sum = add(10, 20);
    println!("합계: {}", sum);
}
```

함수 선언에서 주목할 점은 매개변수의 타입을 반드시 명시해야 한다는 것이다. Rust는 함수 시그니처에서 타입 추론을 하지 않는다. 이는 코드를 읽는 사람이 함수의 계약을 명확히 이해할 수 있게 해준다.

반환값이 있는 함수는 `->` 다음에 반환 타입을 명시한다. Rust의 특징 중 하나는 함수의 마지막 표현식이 자동으로 반환값이 된다는 점이다. 세미콜론을 붙이지 않으면 그 값이 반환되고, 세미콜론을 붙이면 문장(statement)이 되어 아무것도 반환하지 않는다.

```rust
fn get_config_value(key: &str) -> String {
    // 실제로는 파일이나 환경변수에서 읽어온다
    if key == "port" {
        return "8080".to_string();  // 명시적 return도 가능하다
    }
    
    "default".to_string()  // 마지막 표현식이 반환된다
}
```
  

## Option<T>로 값이 없을 수 있는 상황 다루기
실제 프로그램에서는 값이 항상 존재하는 것이 아니다. 설정 파일에서 특정 키를 찾을 때, 그 키가 없을 수도 있다. 많은 언어에서 이런 상황을 `null`이나 `nil`로 표현하지만, Rust는 `Option<T>` 타입을 사용한다.

`Option`은 열거형(enum)으로 정의되어 있으며, 두 가지 상태를 가진다.

```rust
enum Option<T> {
    Some(T),  // 값이 있는 경우
    None,     // 값이 없는 경우
}
```

이제 Day 2에서 만든 설정 파서를 개선해본다. 키가 존재하지 않을 수도 있으므로 `Option`을 반환하도록 만든다.

```rust
fn parse_config_line(line: &str) -> Option<(String, String)> {
    let parts: Vec<&str> = line.split('=').collect();
    
    if parts.len() != 2 {
        return None;  // 형식이 잘못된 경우 None 반환
    }
    
    let key = parts[0].trim().to_string();
    let value = parts[1].trim().to_string();
    
    if key.is_empty() {
        return None;  // 키가 비어있으면 None 반환
    }
    
    Some((key, value))  // 성공하면 Some으로 감싸서 반환
}

fn main() {
    let valid_line = "port=8080";
    let invalid_line = "invalid";
    
    match parse_config_line(valid_line) {
        Some((key, value)) => println!("{}={}", key, value),
        None => println!("파싱 실패"),
    }
    
    // if let으로 더 간결하게 처리할 수도 있다
    if let Some((key, value)) = parse_config_line(valid_line) {
        println!("설정: {} → {}", key, value);
    }
}
```

`match` 표현식은 `Option`의 모든 경우를 처리하도록 강제한다. 컴파일러가 `None` 케이스를 처리하지 않으면 에러를 발생시키기 때문에, null 포인터 에러를 컴파일 타임에 방지할 수 있다.

```
     ┌─────────────┐
     │ parse_line  │
     └──────┬──────┘
            │
            ├─── 형식 올바름? ───┐
            │                   │
           No                  Yes
            │                   │
            v                   v
        ┌──────┐           ┌────────┐
        │ None │           │ Some() │
        └──────┘           └────────┘
```

`Option`은 메서드 체이닝으로도 우아하게 다룰 수 있다.

```rust
fn get_port_number(config_line: &str) -> Option<u16> {
    parse_config_line(config_line)
        .filter(|(key, _)| key == "port")
        .and_then(|(_, value)| value.parse::<u16>().ok())
}

fn main() {
    let port = get_port_number("port=8080").unwrap_or(3000);
    println!("포트: {}", port);  // 8080
    
    let port = get_port_number("invalid").unwrap_or(3000);
    println!("포트: {}", port);  // 3000 (기본값)
}
```
  

## Result<T, E>로 에러 처리하기
`Option`은 값의 유무만 표현할 수 있지만, 실제 프로그램에서는 왜 실패했는지도 알아야 한다. 파일을 읽다가 권한 에러가 발생했는지, 파일이 없어서 실패했는지 구분해야 적절한 대응을 할 수 있다. 이럴 때 `Result<T, E>` 타입을 사용한다.

```rust
enum Result<T, E> {
    Ok(T),   // 성공한 경우 값을 담는다
    Err(E),  // 실패한 경우 에러를 담는다
}
```

문자열을 숫자로 파싱하는 예제를 보자.

```rust
fn parse_port(s: &str) -> Result<u16, String> {
    let num = s.parse::<u16>();
    
    match num {
        Ok(n) => {
            if n > 0 && n < 65536 {
                Ok(n)
            } else {
                Err(format!("포트 범위를 벗어남: {}", n))
            }
        }
        Err(_) => Err(format!("'{}' 는 유효한 숫자가 아니다", s)),
    }
}

fn main() {
    match parse_port("8080") {
        Ok(port) => println!("포트: {}", port),
        Err(e) => println!("에러: {}", e),
    }
    
    match parse_port("99999") {
        Ok(port) => println!("포트: {}", port),
        Err(e) => println!("에러: {}", e),
    }
}
```

`Result`도 `Option`처럼 다양한 메서드를 제공한다. `unwrap()`은 `Ok` 값을 꺼내지만 `Err`이면 패닉을 발생시킨다. 프로토타입 단계에서는 편리하지만 프로덕션 코드에서는 피해야 한다.

```rust
fn main() {
    // unwrap: Ok면 값을 꺼내고, Err면 패닉
    let port = parse_port("8080").unwrap();
    println!("포트: {}", port);
    
    // expect: unwrap과 비슷하지만 에러 메시지를 지정할 수 있다
    let port = parse_port("8080").expect("포트 파싱 실패");
    
    // unwrap_or: Err면 기본값을 사용한다
    let port = parse_port("invalid").unwrap_or(3000);
    println!("포트: {}", port);  // 3000
    
    // unwrap_or_else: Err면 클로저를 실행해 값을 계산한다
    let port = parse_port("invalid").unwrap_or_else(|e| {
        println!("경고: {}", e);
        3000
    });
}
```
  

## 설정 파서에 에러 처리 추가
이제 Day 2에서 만든 설정 파서를 완전하게 개선해본다. 다양한 에러 상황을 구체적으로 표현하고, 사용자에게 명확한 피드백을 제공한다.

```rust
#[derive(Debug)]
enum ConfigError {
    InvalidFormat(String),
    EmptyKey(String),
    InvalidPort(String),
}

fn parse_config_line(line: &str) -> Result<(String, String), ConfigError> {
    let parts: Vec<&str> = line.split('=').collect();
    
    if parts.len() != 2 {
        return Err(ConfigError::InvalidFormat(line.to_string()));
    }
    
    let key = parts[0].trim();
    let value = parts[1].trim();
    
    if key.is_empty() {
        return Err(ConfigError::EmptyKey(line.to_string()));
    }
    
    Ok((key.to_string(), value.to_string()))
}

fn parse_config(content: &str) -> Result<Vec<(String, String)>, ConfigError> {
    let mut config = Vec::new();
    
    for line in content.lines() {
        let line = line.trim();
        
        // 빈 줄이나 주석은 건너뛴다
        if line.is_empty() || line.starts_with('#') {
            continue;
        }
        
        match parse_config_line(line) {
            Ok((key, value)) => {
                // port는 숫자로 파싱할 수 있는지 검증한다
                if key == "port" {
                    if value.parse::<u16>().is_err() {
                        return Err(ConfigError::InvalidPort(value));
                    }
                }
                config.push((key, value));
            }
            Err(e) => return Err(e),
        }
    }
    
    Ok(config)
}

fn main() {
    let config_text = r#"
# 서버 설정
host=localhost
port=8080

# 데이터베이스
db_name=mydb
"#;
    
    match parse_config(config_text) {
        Ok(config) => {
            println!("설정 파싱 성공:");
            for (key, value) in config {
                println!("  {} = {}", key, value);
            }
        }
        Err(e) => {
            println!("설정 파싱 실패:");
            match e {
                ConfigError::InvalidFormat(line) => {
                    println!("  잘못된 형식: '{}'", line);
                    println!("  올바른 형식: key=value");
                }
                ConfigError::EmptyKey(line) => {
                    println!("  빈 키: '{}'", line);
                }
                ConfigError::InvalidPort(port) => {
                    println!("  유효하지 않은 포트: '{}'", port);
                    println!("  포트는 1-65535 사이의 숫자여야 한다");
                }
            }
        }
    }
}
```

위 코드에서 `ConfigError`는 커스텀 에러 타입이다. 열거형으로 정의하여 발생 가능한 에러 종류를 명시적으로 표현한다. `#[derive(Debug)]`는 디버깅 출력을 자동으로 생성해준다.

```
     입력 문자열
          │
          v
    ┌──────────────┐
    │ parse_config │
    └──────┬───────┘
           │
      각 줄 순회
           │
    ┌──────┴──────┐
    │  빈 줄 or   │
    │  주석?      │
    └──────┬──────┘
           │
       ┌───┴───┐
      Yes     No
       │       │
    건너뜀     v
         ┌──────────────────┐
         │ parse_config_line│
         └────────┬──────────┘
                  │
         ┌────────┴─────────┐
         │                  │
        Ok                 Err
         │                  │
         v                  v
    ┌─────────┐       ┌──────────┐
    │ 유효성  │       │ 에러 반환│
    │ 검증    │       └──────────┘
    └────┬────┘
         │
    ┌────┴────┐
    │         │
   Ok        Err
    │         │
  저장    에러 반환
```
  

## 에러를 모으지 말고 즉시 반환하기
위 코드에서 주목할 패턴은 에러를 발견하면 즉시 반환한다는 점이다. 많은 초보자가 에러를 모아두었다가 나중에 처리하려고 하지만, Rust에서는 `Result`를 즉시 반환하는 것이 관례다.

```rust
// 좋은 패턴: 에러 발견 시 즉시 반환
fn process_line(line: &str) -> Result<String, ConfigError> {
    let parsed = parse_config_line(line)?;  // ? 연산자 (나중에 배운다)
    // 추가 처리...
    Ok(parsed.1)
}

// 피해야 할 패턴: 에러를 담아두기
fn process_lines_bad(lines: Vec<&str>) -> Vec<Result<String, ConfigError>> {
    let mut results = Vec::new();
    for line in lines {
        results.push(parse_config_line(line).map(|(_, v)| v));
    }
    results  // 호출자가 모든 Result를 다시 확인해야 한다
}
```
  

## 실전 예제: 포트 번호 추출 함수
배운 내용을 종합해서 설정에서 포트 번호를 안전하게 추출하는 함수를 만들어본다.

```rust
fn get_port_from_config(config_text: &str) -> Result<u16, String> {
    let config = parse_config(config_text)
        .map_err(|e| format!("설정 파싱 실패: {:?}", e))?;
    
    for (key, value) in config {
        if key == "port" {
            return value.parse::<u16>()
                .map_err(|_| format!("포트 파싱 실패: {}", value));
        }
    }
    
    Err("설정에 port가 없다".to_string())
}

fn main() {
    let config = "host=localhost\nport=8080";
    
    match get_port_from_config(config) {
        Ok(port) => println!("서버를 포트 {}에서 시작한다", port),
        Err(e) => println!("에러: {}", e),
    }
}
```

여기서 `map_err`는 에러 타입을 변환하는 메서드다. `ConfigError`를 `String`으로 변환하여 함수의 반환 타입을 통일한다. 실무에서는 모든 에러를 하나의 타입으로 통일하는 것이 관리하기 편하다.
  

## Option과 Result, 무엇을 선택할까
두 타입은 비슷해 보이지만 사용 시나리오가 다르다.

**Option을 사용하는 경우:**
- 값이 없는 것이 정상적인 상황일 때 (예: 선택적 설정값)
- 실패 이유가 중요하지 않을 때
- 컬렉션에서 검색할 때 (예: HashMap의 `get` 메서드)

```rust
fn find_value(key: &str) -> Option<String> {
    // 키가 없는 것은 에러가 아니라 정상적인 상황이다
    if key == "optional_setting" {
        return None;
    }
    Some("value".to_string())
}
```

**Result를 사용하는 경우:**
- 실패 이유를 명확히 알려줘야 할 때
- 에러를 복구하거나 다르게 처리해야 할 때
- I/O 작업이나 파싱처럼 여러 방식으로 실패할 수 있을 때

```rust
fn read_config_file(path: &str) -> Result<String, std::io::Error> {
    // 파일을 읽지 못한 이유(권한, 존재하지 않음 등)를 알아야 한다
    std::fs::read_to_string(path)
}
```
  

## 정리하며
오늘은 Rust 함수의 기본부터 에러 처리의 핵심인 `Option`과 `Result`까지 배웠다. 설정 파서에 견고한 에러 처리를 추가하면서 실전 감각도 익혔다.

핵심 포인트를 정리하면 다음과 같다.

함수는 매개변수와 반환 타입을 명시적으로 선언한다. 마지막 표현식이 세미콜론 없이 작성되면 자동으로 반환값이 된다.

`Option<T>`는 값이 있거나 없을 수 있는 상황을 안전하게 표현한다. `match`나 `if let`으로 모든 경우를 처리하도록 컴파일러가 강제한다.

`Result<T, E>`는 성공과 실패를 모두 표현하며 에러의 구체적인 정보를 전달한다. 커스텀 에러 타입을 만들어 도메인에 특화된 에러를 표현할 수 있다.

에러는 발견 즉시 반환하는 것이 Rust의 관례다. 다음 Day에서 배울 `?` 연산자가 이 패턴을 더욱 간결하게 만들어준다.

내일은 구조체를 사용해서 데이터를 체계적으로 모델링하는 방법을 배운다. 오늘 만든 설정 파서를 구조체로 재구성하면서 코드의 가독성과 유지보수성을 크게 향상시킬 수 있다.
