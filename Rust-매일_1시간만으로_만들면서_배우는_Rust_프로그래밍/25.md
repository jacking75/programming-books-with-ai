# Day 25: 비동기 채널과 스트림

어제는 `select!` 매크로로 여러 작업을 동시에 대기하는 방법을 배웠다. 오늘은 비동기 환경에서 데이터를 효율적으로 전달하는 채널과, 연속된 값들을 처리하는 스트림에 대해 깊이 있게 다룬다. 이 두 가지를 마스터하면 복잡한 메시지 라우팅과 데이터 파이프라인을 우아하게 구현할 수 있다.

## **동기 채널 vs 비동기 채널**

Day 18에서 표준 라이브러리의 `std::sync::mpsc` 채널을 배웠다. 이 채널은 스레드 간 통신에는 완벽하지만, 비동기 환경에서는 문제가 있다. `recv()` 메서드가 블로킹 호출이기 때문에 비동기 태스크 전체를 멈춰버린다.

```
┌─────────────────────────────────────────┐
│       std::sync::mpsc (동기)            │
│                                         │
│  Sender ─────→ Channel ─────→ Receiver │
│                                         │
│  send() 호출    큐에 저장    recv() 블로킹 │
│  (즉시 반환)                (대기)      │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│      tokio::sync::mpsc (비동기)         │
│                                         │
│  Sender ─────→ Channel ─────→ Receiver │
│                                         │
│  send().await   큐에 저장  recv().await │
│  (비동기 대기)            (비동기 대기) │
└─────────────────────────────────────────┘
```

`tokio::sync::mpsc`는 비동기 환경에 특화되어 있다. 메시지를 기다리는 동안 다른 태스크가 실행될 수 있어서 수천 개의 동시 연결을 효율적으로 처리할 수 있다.

## **tokio::sync::mpsc 기본 사용법**

비동기 채널의 기본적인 사용 패턴을 살펴보자. 표준 라이브러리의 채널과 비슷하지만 `.await`이 추가된다.

```rust
use tokio::sync::mpsc;

#[tokio::main]
async fn main() {
    // 버퍼 크기 32인 채널 생성
    let (tx, mut rx) = mpsc::channel::<String>(32);

    // 송신자를 다른 태스크로 이동
    tokio::spawn(async move {
        for i in 0..5 {
            let msg = format!("메시지 {}", i);
            println!("전송: {}", msg);
            
            // 비동기로 전송
            if let Err(e) = tx.send(msg).await {
                eprintln!("전송 실패: {}", e);
                break;
            }
            
            tokio::time::sleep(tokio::time::Duration::from_millis(100)).await;
        }
        println!("송신자 종료");
    });

    // 메인 태스크에서 수신
    while let Some(msg) = rx.recv().await {
        println!("수신: {}", msg);
    }
    
    println!("수신자 종료");
}
```

이 코드를 실행하면 메시지가 순차적으로 전송되고 수신된다. `tx.send().await`은 채널이 가득 찼을 때 공간이 생길 때까지 비동기로 기다린다. `rx.recv().await`은 메시지가 올 때까지 기다리는데, 모든 송신자가 드롭되면 `None`을 반환한다.

여기서 소유권의 흐름을 주목해보자. `tx`는 `move` 클로저로 스폰된 태스크로 이동된다. 메시지 자체도 `send()`를 통해 소유권이 채널로 이동하고, `recv()`에서 다시 수신자에게 이동한다. 이런 명확한 소유권 이전 덕분에 데이터 레이스가 원천적으로 불가능하다.

## **채널 버퍼링과 배압**

채널을 생성할 때 지정하는 버퍼 크기는 중요한 성능 매개변수다. 버퍼가 가득 차면 송신자는 공간이 생길 때까지 기다려야 한다. 이것을 **배압(backpressure)**이라고 부른다.

```rust
use tokio::sync::mpsc;
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    // 작은 버퍼 크기
    let (tx, mut rx) = mpsc::channel::<i32>(2);

    // 빠른 생산자
    tokio::spawn(async move {
        for i in 0..10 {
            println!("[생산자] {} 전송 시도", i);
            tx.send(i).await.unwrap();
            println!("[생산자] {} 전송 완료", i);
        }
    });

    // 느린 소비자
    sleep(Duration::from_secs(1)).await;
    println!("[소비자] 수신 시작");
    
    while let Some(num) = rx.recv().await {
        println!("[소비자] {} 수신", num);
        sleep(Duration::from_millis(500)).await;
    }
}
```

이 코드를 실행하면 생산자가 처음 2개(버퍼 크기)를 빠르게 전송한 후 멈춘다. 소비자가 하나를 가져가면 그제야 다음 것을 전송할 수 있다. 배압이 작동해서 생산자가 소비자의 속도에 맞춰지는 것이다.

실제 서비스에서는 버퍼 크기를 신중하게 설정해야 한다. 너무 작으면 생산자가 자주 대기하고, 너무 크면 메모리를 많이 사용하며 지연 시간이 늘어난다.

```
버퍼 크기 = 2

┌──────────────────────────────────────┐
│  생산자      채널 버퍼      소비자   │
│                                      │
│  [0] ──→    [0][1]                  │
│  [1] ──→    [0][1]  (가득참)        │
│  [2] ──×    [0][1]  (대기...)       │
│             [0][1] ────→ 읽기       │
│  [2] ──→    [1][2]  (공간 생김)    │
└──────────────────────────────────────┘
```

## **브로드캐스트 채널**

`mpsc`는 다중 생산자, 단일 소비자 모델이다. 하지만 채팅 서버처럼 하나의 메시지를 여러 수신자에게 보내야 할 때가 있다. 이럴 때는 `tokio::sync::broadcast`를 사용한다.

```rust
use tokio::sync::broadcast;

#[tokio::main]
async fn main() {
    // 용량 16인 브로드캐스트 채널
    let (tx, mut rx1) = broadcast::channel::<String>(16);
    let mut rx2 = tx.subscribe();
    let mut rx3 = tx.subscribe();

    // 메시지 전송 태스크
    tokio::spawn(async move {
        for i in 0..3 {
            let msg = format!("공지 {}", i);
            println!("전송: {}", msg);
            let _ = tx.send(msg);
            tokio::time::sleep(tokio::time::Duration::from_millis(100)).await;
        }
    });

    // 세 개의 수신자
    let handle1 = tokio::spawn(async move {
        while let Ok(msg) = rx1.recv().await {
            println!("[수신자1] {}", msg);
        }
    });

    let handle2 = tokio::spawn(async move {
        while let Ok(msg) = rx2.recv().await {
            println!("[수신자2] {}", msg);
        }
    });

    let handle3 = tokio::spawn(async move {
        while let Ok(msg) = rx3.recv().await {
            println!("[수신자3] {}", msg);
        }
    });

    let _ = tokio::join!(handle1, handle2, handle3);
}
```

`broadcast` 채널의 특징은 각 수신자가 모든 메시지의 복사본을 받는다는 것이다. `subscribe()`로 새로운 수신자를 생성할 수 있다. 이것은 채팅 서버의 핵심 메커니즘이다.

주의할 점은 수신자가 느리면 메시지를 놓칠 수 있다는 것이다. 버퍼가 가득 차면 가장 오래된 메시지가 덮어씌워진다. `recv()`는 `Err(RecvError::Lagged)`를 반환해서 몇 개의 메시지를 놓쳤는지 알려준다.

## **Stream trait 이해하기**

채널의 수신자는 값들의 연속이다. 이런 개념을 추상화한 것이 `Stream` trait이다. 이것은 비동기 버전의 `Iterator`라고 생각할 수 있다.

```rust
// Iterator (동기)
trait Iterator {
    type Item;
    fn next(&mut self) -> Option<Self::Item>;
}

// Stream (비동기)
trait Stream {
    type Item;
    fn poll_next(
        self: Pin<&mut Self>,
        cx: &mut Context<'_>,
    ) -> Poll<Option<Self::Item>>;
}
```

직접 `poll_next`를 호출하는 대신, `StreamExt` trait의 헬퍼 메서드들을 사용한다. 먼저 `Cargo.toml`에 의존성을 추가하자:

```toml
[dependencies]
tokio = { version = "1", features = ["full"] }
tokio-stream = "0.1"
```

`tokio-stream` 크레이트는 스트림을 다루는 다양한 유틸리티를 제공한다.

```rust
use tokio::sync::mpsc;
use tokio_stream::wrappers::ReceiverStream;
use tokio_stream::StreamExt;

#[tokio::main]
async fn main() {
    let (tx, rx) = mpsc::channel::<i32>(10);

    // 채널 수신자를 스트림으로 변환
    let mut stream = ReceiverStream::new(rx);

    // 데이터 전송
    tokio::spawn(async move {
        for i in 0..10 {
            tx.send(i).await.unwrap();
        }
    });

    // 스트림 메서드 체이닝
    while let Some(value) = stream.next().await {
        println!("값: {}", value);
    }
}
```

`ReceiverStream`은 `mpsc::Receiver`를 `Stream`으로 감싸준다. 이제 `StreamExt`의 강력한 메서드들을 사용할 수 있다.

## **스트림 변환과 조합**

스트림의 진정한 힘은 함수형 프로그래밍 스타일의 변환이다. `map`, `filter`, `fold` 같은 메서드를 체이닝해서 복잡한 데이터 파이프라인을 만들 수 있다.

```rust
use tokio::sync::mpsc;
use tokio_stream::wrappers::ReceiverStream;
use tokio_stream::StreamExt;

#[tokio::main]
async fn main() {
    let (tx, rx) = mpsc::channel::<i32>(10);
    let mut stream = ReceiverStream::new(rx);

    tokio::spawn(async move {
        for i in 0..20 {
            tx.send(i).await.unwrap();
        }
    });

    // 짝수만 필터링하고 제곱한 후 처음 5개만 가져오기
    let mut transformed = stream
        .filter(|x| *x % 2 == 0)
        .map(|x| x * x)
        .take(5);

    while let Some(value) = transformed.next().await {
        println!("변환된 값: {}", value);
    }
}
```

출력 결과:
```
변환된 값: 0
변환된 값: 4
변환된 값: 16
변환된 값: 36
변환된 값: 64
```

이 코드는 0부터 19까지의 숫자 중 짝수만 골라서 제곱한 후 처음 5개를 출력한다. 중요한 점은 이 모든 변환이 지연 평가(lazy evaluation)된다는 것이다. 실제로 값을 소비할 때만 계산이 일어난다.

빌림 관점에서 보면, `filter`와 `map`의 클로저는 각 요소에 대한 참조를 받는다. 소유권을 가져가지 않기 때문에 체이닝이 가능하다. 최종적으로 `next().await`을 호출할 때 값의 소유권이 우리에게 이동한다.

## **실전: 메시지 라우팅 시스템**

이제 배운 내용을 종합해서 실용적인 메시지 라우팅 시스템을 만들어보자. 여러 클라이언트가 메시지를 보내면 주제(topic)에 따라 다른 구독자들에게 전달하는 시스템이다.

```rust
use std::collections::HashMap;
use std::sync::Arc;
use tokio::sync::{mpsc, RwLock};

// 메시지 구조
#[derive(Debug, Clone)]
struct Message {
    topic: String,
    content: String,
    sender: String,
}

// 라우터 구조체
struct MessageRouter {
    // 각 주제별 구독자 목록
    subscribers: Arc<RwLock<HashMap<String, Vec<mpsc::Sender<Message>>>>>,
}

impl MessageRouter {
    fn new() -> Self {
        Self {
            subscribers: Arc::new(RwLock::new(HashMap::new())),
        }
    }

    // 주제 구독
    async fn subscribe(&self, topic: String) -> mpsc::Receiver<Message> {
        let (tx, rx) = mpsc::channel(32);
        
        let mut subs = self.subscribers.write().await;
        subs.entry(topic.clone())
            .or_insert_with(Vec::new)
            .push(tx);
        
        println!("[라우터] '{}' 주제에 새 구독자 추가", topic);
        rx
    }

    // 메시지 발행
    async fn publish(&self, message: Message) {
        let subs = self.subscribers.read().await;
        
        if let Some(subscribers) = subs.get(&message.topic) {
            let mut active_count = 0;
            
            for tx in subscribers {
                if tx.send(message.clone()).await.is_ok() {
                    active_count += 1;
                }
            }
            
            println!(
                "[라우터] '{}' 주제에 메시지 전달: {} → {} 구독자",
                message.topic, message.sender, active_count
            );
        } else {
            println!("[라우터] '{}' 주제에 구독자 없음", message.topic);
        }
    }

    // 주기적으로 죽은 구독자 제거
    async fn cleanup(&self) {
        let mut subs = self.subscribers.write().await;
        
        for (topic, subscribers) in subs.iter_mut() {
            let original_len = subscribers.len();
            subscribers.retain(|tx| !tx.is_closed());
            let removed = original_len - subscribers.len();
            
            if removed > 0 {
                println!("[라우터] '{}' 주제에서 {} 구독자 제거", topic, removed);
            }
        }
    }
}
```

이 라우터의 핵심은 `Arc<RwLock<HashMap>>`이다. 여러 태스크가 동시에 읽을 수 있지만, 쓰기는 배타적으로 이루어진다. `RwLock`는 `Mutex`보다 읽기가 많은 상황에서 효율적이다.

이제 라우터를 사용하는 시뮬레이션을 만들어보자:

```rust
#[tokio::main]
async fn main() {
    let router = Arc::new(MessageRouter::new());

    // 구독자 1: "news" 주제 구독
    let router1 = Arc::clone(&router);
    tokio::spawn(async move {
        let mut rx = router1.subscribe("news".to_string()).await;
        
        while let Some(msg) = rx.recv().await {
            println!("[구독자1] 뉴스 받음: {} - {}", msg.sender, msg.content);
        }
    });

    // 구독자 2: "news"와 "sports" 주제 구독
    let router2 = Arc::clone(&router);
    tokio::spawn(async move {
        let mut rx_news = router2.subscribe("news".to_string()).await;
        let mut rx_sports = router2.subscribe("sports".to_string()).await;
        
        loop {
            tokio::select! {
                Some(msg) = rx_news.recv() => {
                    println!("[구독자2] 뉴스: {}", msg.content);
                }
                Some(msg) = rx_sports.recv() => {
                    println!("[구독자2] 스포츠: {}", msg.content);
                }
                else => break,
            }
        }
    });

    // 구독자 3: "sports" 주제만 구독
    let router3 = Arc::clone(&router);
    tokio::spawn(async move {
        let mut rx = router3.subscribe("sports".to_string()).await;
        
        while let Some(msg) = rx.recv().await {
            println!("[구독자3] 스포츠 받음: {}", msg.content);
        }
    });

    // 잠시 대기 (구독자 등록 시간)
    tokio::time::sleep(tokio::time::Duration::from_millis(100)).await;

    // 발행자 태스크
    let router_pub = Arc::clone(&router);
    tokio::spawn(async move {
        let messages = vec![
            Message {
                topic: "news".to_string(),
                content: "Rust 1.70 출시".to_string(),
                sender: "발행자1".to_string(),
            },
            Message {
                topic: "sports".to_string(),
                content: "월드컵 개막".to_string(),
                sender: "발행자1".to_string(),
            },
            Message {
                topic: "news".to_string(),
                content: "tokio 업데이트".to_string(),
                sender: "발행자2".to_string(),
            },
            Message {
                topic: "weather".to_string(),
                content: "맑음".to_string(),
                sender: "발행자1".to_string(),
            },
        ];

        for msg in messages {
            router_pub.publish(msg).await;
            tokio::time::sleep(tokio::time::Duration::from_millis(200)).await;
        }
    });

    // 정리 태스크
    let router_cleanup = Arc::clone(&router);
    tokio::spawn(async move {
        let mut interval = tokio::time::interval(tokio::time::Duration::from_secs(5));
        
        loop {
            interval.tick().await;
            router_cleanup.cleanup().await;
        }
    });

    // 충분한 시간 대기
    tokio::time::sleep(tokio::time::Duration::from_secs(2)).await;
}
```

이 코드를 실행하면 각 구독자가 관심 있는 주제의 메시지만 받는 것을 볼 수 있다. "weather" 주제는 구독자가 없어서 메시지가 전달되지 않는다.

여기서 빌림과 소유권의 복잡한 상호작용을 관찰할 수 있다. `Arc`를 통해 라우터가 여러 태스크에 공유되고, `RwLock`으로 안전하게 접근된다. 메시지는 `clone()`되어 각 구독자에게 전달되는데, 이것은 성능 트레이드오프다. 큰 메시지라면 `Arc<Message>`로 감싸서 참조만 공유하는 것이 더 효율적이다.

## **스트림 병합과 선택**

때로는 여러 스트림을 하나로 합치거나, 여러 스트림 중 먼저 도착하는 것을 처리해야 한다. `StreamExt`는 이를 위한 메서드들을 제공한다.

```rust
use tokio::sync::mpsc;
use tokio_stream::wrappers::ReceiverStream;
use tokio_stream::StreamExt;

#[tokio::main]
async fn main() {
    let (tx1, rx1) = mpsc::channel::<String>(10);
    let (tx2, rx2) = mpsc::channel::<String>(10);

    // 첫 번째 소스
    tokio::spawn(async move {
        for i in 0..5 {
            tx1.send(format!("소스1-{}", i)).await.unwrap();
            tokio::time::sleep(tokio::time::Duration::from_millis(200)).await;
        }
    });

    // 두 번째 소스
    tokio::spawn(async move {
        for i in 0..5 {
            tx2.send(format!("소스2-{}", i)).await.unwrap();
            tokio::time::sleep(tokio::time::Duration::from_millis(300)).await;
        }
    });

    // 두 스트림 병합
    let stream1 = ReceiverStream::new(rx1);
    let stream2 = ReceiverStream::new(rx2);
    let mut merged = stream1.merge(stream2);

    println!("병합된 스트림에서 수신:");
    while let Some(msg) = merged.next().await {
        println!("  {}", msg);
    }
}
```

`merge()`는 두 스트림을 하나로 합친다. 어느 쪽에서 먼저 값이 오든 순서대로 처리된다. 이것은 여러 소스로부터 이벤트를 받아야 하는 상황에서 유용하다.

```
┌────────────┐     ┌────────────┐
│  스트림 1  │     │  스트림 2  │
│            │     │            │
│  A ────┐   │     │  X ────┐   │
│  B ────┼───┼─────┼─ Y ────┼───┼──→ 병합
│  C ────┘   │     │  Z ────┘   │
└────────────┘     └────────────┘

출력 순서: A, X, B, Y, C, Z (도착 순)
```

## **버퍼링과 배치 처리**

네트워크 애플리케이션에서는 개별 메시지를 하나씩 처리하는 것보다 일정량을 모아서 배치로 처리하는 것이 효율적일 때가 있다. 스트림은 이런 패턴도 지원한다.

```rust
use tokio::sync::mpsc;
use tokio_stream::wrappers::ReceiverStream;
use tokio_stream::StreamExt;

#[tokio::main]
async fn main() {
    let (tx, rx) = mpsc::channel::<i32>(100);

    // 빠른 메시지 생성
    tokio::spawn(async move {
        for i in 0..20 {
            tx.send(i).await.unwrap();
            tokio::time::sleep(tokio::time::Duration::from_millis(50)).await;
        }
    });

    let mut stream = ReceiverStream::new(rx);

    // 3개씩 모아서 처리
    while let Some(chunk) = stream.ready_chunks(3).next().await {
        println!("배치 처리: {:?} ({}개)", chunk, chunk.len());
        // 데이터베이스에 배치 삽입하거나 네트워크로 한 번에 전송
    }
}
```

`ready_chunks(n)`은 최대 n개까지 모아서 벡터로 반환한다. 중요한 점은 n개가 될 때까지 무한정 기다리지 않는다는 것이다. 스트림이 끝나거나 다음 값이 즉시 준비되지 않으면 현재까지 모은 것을 반환한다.

이 패턴은 데이터베이스 배치 삽입, 네트워크 패킷 결합, 로그 버퍼링 등에 매우 유용하다.

## **에러 처리와 복구**

실전에서는 스트림 처리 중 에러가 발생할 수 있다. 네트워크 단절, 파싱 실패, 리소스 부족 등이다. 이런 에러를 우아하게 처리하는 방법을 살펴보자.

```rust
use tokio::sync::mpsc;
use tokio_stream::wrappers::ReceiverStream;
use tokio_stream::StreamExt;

#[derive(Debug)]
enum ProcessingError {
    InvalidData(String),
    Timeout,
}

async fn process_value(value: i32) -> Result<String, ProcessingError> {
    if value < 0 {
        return Err(ProcessingError::InvalidData(format!("음수: {}", value)));
    }
    
    if value > 100 {
        return Err(ProcessingError::InvalidData(format!("너무 큼: {}", value)));
    }
    
    Ok(format!("처리됨: {}", value * 2))
}

#[tokio::main]
async fn main() {
    let (tx, rx) = mpsc::channel::<i32>(10);

    tokio::spawn(async move {
        let values = vec![5, -3, 10, 150, 20, -1, 30];
        for v in values {
            tx.send(v).await.unwrap();
        }
    });

    let mut stream = ReceiverStream::new(rx);
    let mut success_count = 0;
    let mut error_count = 0;

    while let Some(value) = stream.next().await {
        match process_value(value).await {
            Ok(result) => {
                println!("✓ {}", result);
                success_count += 1;
            }
            Err(e) => {
                eprintln!("✗ 에러: {:?}", e);
                error_count += 1;
                // 에러가 발생해도 계속 처리
            }
        }
    }

    println!("\n총 {}개 성공, {}개 실패", success_count, error_count);
}
```

이 패턴은 일부 메시지가 실패하더라도 전체 파이프라인을 멈추지 않는다. 실제 서비스에서는 실패한 메시지를 별도의 채널로 보내서 재처리하거나, 로깅하거나, 알림을 보내는 등의 처리를 추가할 수 있다.

## **실전: 채팅 서버 업그레이드**

이제 모든 개념을 통합해서 Day 24에서 만든 채팅 서버를 스트림 기반으로 업그레이드해보자. 더 깔끔하고 확장 가능한 구조가 될 것이다.

```rust
use std::collections::HashMap;
use std::net::SocketAddr;
use std::sync::Arc;
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use tokio::net::{TcpListener, TcpStream};
use tokio::sync::{broadcast, mpsc, RwLock};
use tokio_stream::wrappers::BroadcastStream;
use tokio_stream::StreamExt;

// 채팅 메시지
#[derive(Debug, Clone)]
struct ChatMessage {
    sender: SocketAddr,
    content: String,
    timestamp: std::time::SystemTime,
}

// 채팅 서버
struct ChatServer {
    broadcast_tx: broadcast::Sender<ChatMessage>,
    clients: Arc<RwLock<HashMap<SocketAddr, mpsc::Sender<String>>>>,
}

impl ChatServer {
    fn new() -> Self {
        let (broadcast_tx, _) = broadcast::channel(100);
        Self {
            broadcast_tx,
            clients: Arc::new(RwLock::new(HashMap::new())),
        }
    }

    async fn register_client(&self, addr: SocketAddr) -> mpsc::Receiver<String> {
        let (tx, rx) = mpsc::channel(32);
        
        let mut clients = self.clients.write().await;
        clients.insert(addr, tx);
        
        println!("[서버] {} 등록 (총 {}명)", addr, clients.len());
        rx
    }

    async fn unregister_client(&self, addr: SocketAddr) {
        let mut clients = self.clients.write().await;
        clients.remove(&addr);
        
        println!("[서버] {} 해제 (총 {}명)", addr, clients.len());
    }

    async fn broadcast(&self, message: ChatMessage) {
        // 브로드캐스트 채널로 전송
        let _ = self.broadcast_tx.send(message.clone());
        
        // 직접 메시지 전송도 시도 (브로드캐스트 대안)
        let clients = self.clients.read().await;
        let formatted = format!(
            "[{}] {}\n",
            message.sender,
            message.content.trim()
        );
        
        for (addr, tx) in clients.iter() {
            // 자기 자신에게는 전송하지 않음
            if *addr != message.sender {
                let _ = tx.send(formatted.clone()).await;
            }
        }
    }

    async fn handle_client(
        self: Arc<Self>,
        socket: TcpStream,
        addr: SocketAddr,
    ) {
        println!("[클라이언트] {} 연결됨", addr);

        // 소켓 분리
        let (reader, mut writer) = socket.into_split();
        let mut reader = BufReader::new(reader);

        // 개인 메시지 채널
        let mut personal_rx = self.register_client(addr).await;

        // 브로드캐스트 구독
        let broadcast_rx = self.broadcast_tx.subscribe();
        let mut broadcast_stream = BroadcastStream::new(broadcast_rx)
            .filter_map(|result| result.ok())
            .filter(move |msg| msg.sender != addr);

        // 환영 메시지
        let welcome = format!("채팅 서버에 오신 것을 환영한다! ({})\n", addr);
        let _ = writer.write_all(welcome.as_bytes()).await;

        // 읽기 루프 태스크
        let self_clone = Arc::clone(&self);
        let read_handle = tokio::spawn(async move {
            let mut line = String::new();
            
            loop {
                line.clear();
                
                match tokio::time::timeout(
                    tokio::time::Duration::from_secs(300),
                    reader.read_line(&mut line),
                )
                .await
                {
                    Ok(Ok(0)) => {
                        println!("[클라이언트] {} 연결 종료", addr);
                        break;
                    }
                    Ok(Ok(_)) => {
                        if line.trim().is_empty() {
                            continue;
                        }

                        let msg = ChatMessage {
                            sender: addr,
                            content: line.clone(),
                            timestamp: std::time::SystemTime::now(),
                        };

                        self_clone.broadcast(msg).await;
                    }
                    Ok(Err(e)) => {
                        eprintln!("[클라이언트] {} 읽기 에러: {}", addr, e);
                        break;
                    }
                    Err(_) => {
                        println!("[클라이언트] {} 타임아웃", addr);
                        break;
                    }
                }
            }
        });

        // 쓰기 루프
        loop {
            tokio::select! {
                // 개인 메시지
                Some(msg) = personal_rx.recv() => {
                    if writer.write_all(msg.as_bytes()).await.is_err() {
                        break;
                    }
                }
                
                // 브로드캐스트 메시지
                Some(msg) = broadcast_stream.next() => {
                    let formatted = format!("[{}] {}\n", msg.sender, msg.content.trim());
                    if writer.write_all(formatted.as_bytes()).await.is_err() {
                        break;
                    }
                }
                
                // 읽기 태스크 종료
                _ = &mut read_handle => {
                    break;
                }
            }
        }

        // 정리
        self.unregister_client(addr).await;
        println!("[클라이언트] {} 핸들러 종료", addr);
    }
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("채팅 서버가 8080 포트에서 시작되었다");

    let server = Arc::new(ChatServer::new());

    loop {
        let (socket, addr) = listener.accept().await?;
        let server = Arc::clone(&server);

        tokio::spawn(async move {
            server.handle_client(socket, addr).await;
        });
    }
}
```

이 구현의 핵심 개선사항들을 살펴보자.

첫째, `BroadcastStream`을 사용해서 브로드캐스트 채널을 스트림으로 변환했다. 이것은 `StreamExt`의 메서드들을 사용할 수 있게 해준다.

둘째, `filter_map`과 `filter`를 체이닝해서 에러를 필터링하고 자기 자신의 메시지를 제외했다. 이것은 이전의 명령형 코드보다 훨씬 선언적이고 읽기 쉽다.

셋째, 읽기와 쓰기를 명확하게 분리했다. 읽기는 별도 태스크에서 타임아웃과 함께 처리되고, 쓰기는 메인 루프에서 여러 소스를 `select!`로 다룬다.

넷째, 클라이언트 등록/해제를 중앙 집중식으로 관리한다. `HashMap`에 각 클라이언트의 송신자를 저장해서 개인 메시지를 보낼 수 있다.

## **성능 고려사항**

채널과 스트림을 사용할 때 성능에 영향을 주는 요소들이 있다.

**버퍼 크기**: 너무 작으면 배압으로 처리량이 떨어지고, 너무 크면 메모리와 지연 시간이 증가한다. 일반적으로 32-128 정도가 적당하다.

**메시지 크기**: 큰 메시지를 여러 수신자에게 보낼 때는 `Arc`로 감싸서 복사 비용을 줄인다.

```rust
// 작은 메시지
broadcast_tx.send("hello".to_string());

// 큰 메시지
let large_data = vec![0u8; 10000];
broadcast_tx.send(Arc::new(large_data));
```

**채널 타입 선택**: 상황에 맞는 채널을 선택한다.
- `mpsc`: 일반적인 생산자-소비자 패턴
- `broadcast`: 모든 수신자에게 전달
- `watch`: 최신 값만 유지
- `oneshot`: 한 번만 전송

힌트: 방 기능은 `HashMap<String, broadcast::Sender>`로 구현할 수 있다. 각 방마다 독립적인 브로드캐스트 채널을 가진다.

```rust
struct RoomManager {
    rooms: Arc<RwLock<HashMap<String, broadcast::Sender<ChatMessage>>>>,
}

impl RoomManager {
    async fn get_or_create_room(&self, room_name: &str) -> broadcast::Sender<ChatMessage> {
        let mut rooms = self.rooms.write().await;
        
        rooms.entry(room_name.to_string())
            .or_insert_with(|| {
                let (tx, _) = broadcast::channel(100);
                println!("[방] '{}' 생성됨", room_name);
                tx
            })
            .clone()
    }
}
```

## **정리**

오늘은 비동기 채널과 스트림을 깊이 있게 다뤘다. `tokio::sync`의 다양한 채널 타입들과 `Stream` trait을 이해하고, 이들을 조합해서 복잡한 메시지 라우팅 시스템을 구축했다.

핵심 개념을 정리하면 다음과 같다. `tokio::sync::mpsc`는 비동기 환경에서 태스크 간 통신의 기본이다. 버퍼 크기로 배압을 제어할 수 있다. `broadcast` 채널은 모든 구독자에게 메시지를 전달하며, 채팅이나 이벤트 시스템에 적합하다. `Stream`은 비동기 이터레이터로, `map`, `filter`, `merge` 같은 조합자를 제공한다. 스트림은 지연 평가되므로 메모리 효율적이다.

소유권 관점에서 보면, 채널은 메시지의 소유권을 명확하게 이전시켜서 데이터 레이스를 방지한다. 여러 수신자에게 보낼 때는 `clone()`하거나 `Arc`로 감싼다. `Arc<RwLock<T>>`는 읽기가 많은 공유 상태에 적합하다.

내일부터는 프로덕션 환경을 위한 실용적인 주제들을 다룬다. 로깅, 설정 관리, 테스트 등 실제 서비스 운영에 필요한 기술들을 배우면서 지금까지 만든 서버를 더욱 견고하게 만들 것이다.
