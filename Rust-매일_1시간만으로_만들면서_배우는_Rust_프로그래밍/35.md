# Day 35: WebSocket으로 실시간 통신

HTTP는 요청-응답 패턴으로 작동한다. 클라이언트가 요청하면 서버가 응답하고 연결이 끊어진다. 하지만 실시간 채팅, 알림, 협업 도구처럼 서버에서 클라이언트로 즉시 데이터를 보내야 하는 경우가 있다. WebSocket은 이를 위한 양방향 통신 프로토콜이다. 한 번 연결하면 양쪽이 원할 때 언제든지 메시지를 보낼 수 있다. 오늘은 Rust에서 WebSocket 서버를 구축하고, 실시간 채팅 시스템을 만들면서 비동기 브로드캐스트 패턴을 익힌다.

## WebSocket의 작동 원리

WebSocket은 HTTP로 시작한다. 클라이언트가 HTTP Upgrade 요청을 보내면, 서버가 동의하고 프로토콜을 WebSocket으로 전환한다. 이후 TCP 연결은 유지되며, 양쪽이 자유롭게 메시지를 주고받는다:

```
Client                           Server
  │                                │
  │  HTTP GET /ws                  │
  │  Upgrade: websocket            │
  ├───────────────────────────────>│
  │                                │
  │  HTTP 101 Switching Protocols  │
  │<───────────────────────────────┤
  │                                │
  │  WebSocket Frame: "Hello"      │
  ├───────────────────────────────>│
  │                                │
  │  WebSocket Frame: "Hi!"        │
  │<───────────────────────────────┤
  │                                │
  │  WebSocket Frame: "How are you?"│
  ├───────────────────────────────>│
  │                                │
```

HTTP와 달리 연결이 계속 열려있다. 이는 소유권과 라이프타임 측면에서 흥미로운 문제를 제기한다. 연결 객체를 어떻게 관리할 것인가? 여러 클라이언트에게 동시에 메시지를 보내려면 어떻게 해야 하는가?

## 기본 WebSocket 서버

먼저 간단한 에코 서버를 만든다. 클라이언트가 보낸 메시지를 그대로 돌려준다:

```rust
use axum::{
    Router,
    routing::get,
    extract::ws::{WebSocket, WebSocketUpgrade, Message},
    response::IntoResponse,
};
use futures_util::{sink::SinkExt, stream::StreamExt};

async fn ws_handler(ws: WebSocketUpgrade) -> impl IntoResponse {
    ws.on_upgrade(handle_socket)
}

async fn handle_socket(mut socket: WebSocket) {
    while let Some(msg) = socket.recv().await {
        match msg {
            Ok(Message::Text(text)) => {
                println!("Received: {}", text);
                
                if socket.send(Message::Text(format!("Echo: {}", text))).await.is_err() {
                    break;
                }
            }
            Ok(Message::Close(_)) => {
                println!("Client disconnected");
                break;
            }
            Err(e) => {
                eprintln!("WebSocket error: {}", e);
                break;
            }
            _ => {}
        }
    }
}

#[tokio::main]
async fn main() {
    let app = Router::new()
        .route("/ws", get(ws_handler));
    
    let listener = tokio::net::TcpListener::bind("127.0.0.1:3000")
        .await
        .unwrap();
    
    println!("WebSocket server running on ws://127.0.0.1:3000/ws");
    
    axum::serve(listener, app).await.unwrap();
}
```

`WebSocketUpgrade`는 HTTP 연결을 WebSocket으로 업그레이드하는 추출자다. `.on_upgrade()`는 핸들러 함수를 등록하는데, 업그레이드가 완료되면 이 함수가 호출된다.

`handle_socket`은 `WebSocket`을 가변으로 받는다. 이는 소유권을 가져온다는 의미다. 함수가 끝나면 소켓이 drop되고 연결이 닫힌다. `.recv().await`는 메시지를 받을 때까지 대기하고, `.send()`는 메시지를 보낸다.

## 채팅 시스템 설계

실시간 채팅을 만들려면 여러 클라이언트를 관리하고, 한 사람이 보낸 메시지를 모두에게 전달해야 한다. 이를 브로드캐스트라고 한다. 구조는 다음과 같다:

```
┌─────────────────────────────────────────────────┐
│              Chat Server State                  │
│                                                  │
│  ┌──────────────────────────────────────────┐  │
│  │  Arc<Mutex<HashMap<Uuid, Sender>>>       │  │
│  │                                           │  │
│  │  Client ID -> Channel Sender             │  │
│  └──────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
         │              │              │
         │              │              │
         ▼              ▼              ▼
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │ Client1 │    │ Client2 │    │ Client3 │
   │ Receiver│    │ Receiver│    │ Receiver│
   └─────────┘    └─────────┘    └─────────┘
```

각 클라이언트는 채널을 통해 메시지를 받는다. 서버는 모든 클라이언트의 sender를 HashMap에 저장한다. 누군가 메시지를 보내면, 서버는 모든 sender를 통해 다른 클라이언트들에게 전달한다.

## 메시지 타입 정의

채팅 메시지를 구조화한다:

```rust
use serde::{Deserialize, Serialize};
use uuid::Uuid;
use chrono::{DateTime, Utc};

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]
enum ChatMessage {
    Join {
        user_id: Uuid,
        username: String,
        timestamp: DateTime<Utc>,
    },
    Leave {
        user_id: Uuid,
        username: String,
        timestamp: DateTime<Utc>,
    },
    Message {
        user_id: Uuid,
        username: String,
        content: String,
        timestamp: DateTime<Utc>,
    },
}

#[derive(Debug, Deserialize)]
struct ClientMessage {
    content: String,
}
```

`#[serde(tag = "type")]`은 태그된 enum 직렬화를 사용한다. JSON에서 `"type": "join"`같은 필드로 variant를 구분한다. 이렇게 하면 클라이언트가 메시지 종류를 쉽게 알 수 있다.

`ChatMessage`는 서버에서 클라이언트로 보내는 메시지고, `ClientMessage`는 클라이언트에서 서버로 받는 메시지다. 분리하면 각각의 형태를 독립적으로 정의할 수 있다.

## 클라이언트 관리 상태

연결된 클라이언트들을 추적하는 공유 상태를 만든다:

```rust
use std::collections::HashMap;
use std::sync::Arc;
use tokio::sync::{Mutex, mpsc};

type Clients = Arc<Mutex<HashMap<Uuid, ClientInfo>>>;

struct ClientInfo {
    username: String,
    sender: mpsc::UnboundedSender<ChatMessage>,
}

#[derive(Clone)]
struct AppState {
    clients: Clients,
}

impl AppState {
    fn new() -> Self {
        Self {
            clients: Arc::new(Mutex::new(HashMap::new())),
        }
    }
}
```

`Clients`는 타입 별칭이다. `Arc<Mutex<HashMap<...>>>`를 반복해서 쓰지 않아도 된다. `Arc`는 여러 태스크가 같은 HashMap을 공유할 수 있게 하고, `Mutex`는 한 번에 하나만 수정할 수 있게 보장한다.

`mpsc::UnboundedSender`는 채널의 송신 끝이다. "Unbounded"는 버퍼 크기 제한이 없다는 의미다. 메시지가 쌓일 수 있지만, send가 블록되지 않는다. 실제 프로덕션에서는 bounded 채널로 백프레셔를 관리하는 것이 좋다.

`ClientInfo`는 사용자명과 채널 sender를 묶는다. 이를 HashMap에 저장하면, 누가 연결되어 있는지, 어떻게 메시지를 보낼지 알 수 있다.

## 브로드캐스트 함수

모든 클라이언트에게 메시지를 보내는 함수다:

```rust
async fn broadcast_message(clients: &Clients, message: ChatMessage, sender_id: Option<Uuid>) {
    let clients_lock = clients.lock().await;
    
    for (client_id, client_info) in clients_lock.iter() {
        // 자기 자신에게는 보내지 않음 (선택적)
        if let Some(sid) = sender_id {
            if *client_id == sid {
                continue;
            }
        }
        
        if let Err(e) = client_info.sender.send(message.clone()) {
            eprintln!("Failed to send to client {}: {}", client_id, e);
        }
    }
}
```

`clients: &Clients`는 `Arc<Mutex<HashMap<...>>>`의 참조다. 함수가 소유권을 가져가지 않고, Arc의 참조 카운트를 증가시키지도 않는다. 호출하는 쪽이 Arc를 계속 소유한다.

`.lock().await`는 뮤텍스를 잠그고 `MutexGuard`를 반환한다. 이 가드가 스코프를 벗어나면 자동으로 잠금이 해제된다. 비동기 환경에서는 `.await` 너머로 가드를 들고 있으면 안 되므로, 이터레이션이 끝나면 가드가 drop된다.

`message.clone()`을 호출하는 것은 비효율적으로 보일 수 있다. 하지만 채널은 소유권을 가져가므로 각 클라이언트에게 독립적인 복사본을 줘야 한다. 메시지가 작다면 큰 문제가 아니다. 최적화가 필요하면 `Arc<ChatMessage>`를 사용할 수 있다.

## WebSocket 핸들러 구현

클라이언트가 연결되면 실행되는 핸들러다:

```rust
use axum::extract::{State, Query};

#[derive(Debug, Deserialize)]
struct ConnectParams {
    username: String,
}

async fn ws_handler(
    ws: WebSocketUpgrade,
    State(state): State<AppState>,
    Query(params): Query<ConnectParams>,
) -> impl IntoResponse {
    ws.on_upgrade(move |socket| handle_client(socket, state, params.username))
}
```

`Query(params)`는 URL 쿼리 파라미터를 추출한다. 클라이언트는 `ws://localhost:3000/ws?username=alice`처럼 연결한다. 실제로는 인증 토큰을 사용하는 것이 더 안전하다.

`move |socket|`는 클로저가 `state`와 `username`의 소유권을 가져간다는 의미다. 업그레이드가 비동기적으로 일어나므로, 클로저가 자신의 환경을 소유해야 한다.

## 클라이언트 핸들링 로직

이제 핵심 로직이다. 클라이언트를 등록하고, 메시지를 받아 브로드캐스트하며, 연결 해제를 처리한다:

```rust
async fn handle_client(socket: WebSocket, state: AppState, username: String) {
    let client_id = Uuid::new_v4();
    
    // 채널 생성
    let (client_sender, mut client_receiver) = mpsc::unbounded_channel::<ChatMessage>();
    
    // 클라이언트 등록
    {
        let mut clients = state.clients.lock().await;
        clients.insert(client_id, ClientInfo {
            username: username.clone(),
            sender: client_sender,
        });
    }
    
    println!("Client {} ({}) connected", username, client_id);
    
    // 입장 메시지 브로드캐스트
    let join_message = ChatMessage::Join {
        user_id: client_id,
        username: username.clone(),
        timestamp: Utc::now(),
    };
    broadcast_message(&state.clients, join_message, None).await;
    
    // WebSocket을 송신/수신으로 분리
    let (mut ws_sender, mut ws_receiver) = socket.split();
    
    // 태스크 1: 채널에서 받은 메시지를 WebSocket으로 전송
    let send_task = tokio::spawn(async move {
        while let Some(message) = client_receiver.recv().await {
            let json = serde_json::to_string(&message).unwrap();
            
            if ws_sender.send(Message::Text(json)).await.is_err() {
                break;
            }
        }
    });
    
    // 태스크 2: WebSocket에서 메시지 수신 및 브로드캐스트
    let client_id_copy = client_id;
    let username_copy = username.clone();
    let state_copy = state.clone();
    
    let recv_task = tokio::spawn(async move {
        while let Some(Ok(msg)) = ws_receiver.next().await {
            if let Message::Text(text) = msg {
                if let Ok(client_msg) = serde_json::from_str::<ClientMessage>(&text) {
                    let chat_message = ChatMessage::Message {
                        user_id: client_id_copy,
                        username: username_copy.clone(),
                        content: client_msg.content,
                        timestamp: Utc::now(),
                    };
                    
                    broadcast_message(&state_copy.clients, chat_message, Some(client_id_copy)).await;
                }
            }
        }
    });
    
    // 둘 중 하나가 끝나면 종료
    tokio::select! {
        _ = send_task => {},
        _ = recv_task => {},
    }
    
    // 클라이언트 제거
    {
        let mut clients = state.clients.lock().await;
        clients.remove(&client_id);
    }
    
    println!("Client {} ({}) disconnected", username, client_id);
    
    // 퇴장 메시지 브로드캐스트
    let leave_message = ChatMessage::Leave {
        user_id: client_id,
        username,
        timestamp: Utc::now(),
    };
    broadcast_message(&state.clients, leave_message, None).await;
}
```

이 함수는 복잡해 보이지만, 단계별로 이해하면 명확하다. 먼저 고유 ID를 생성하고 채널을 만든다. 채널은 브로드캐스트 함수가 이 클라이언트에게 메시지를 보낼 수 있게 한다.

`socket.split()`은 WebSocket을 sender와 receiver로 분리한다. 이는 소유권 문제를 해결하는 패턴이다. 하나의 소켓을 동시에 읽고 쓸 수 없지만, 분리하면 각각을 독립적인 태스크에서 사용할 수 있다.

두 개의 태스크를 생성한다. `send_task`는 채널에서 메시지를 받아 WebSocket으로 보낸다. `recv_task`는 WebSocket에서 메시지를 받아 브로드캐스트한다. 각 태스크는 독립적으로 실행되며, `move` 클로저로 필요한 데이터의 소유권을 가져간다.

`tokio::select!`는 여러 Future 중 하나가 완료되면 나머지를 취소한다. 클라이언트가 연결을 끊거나 에러가 발생하면, 두 태스크 모두 정리된다.

## 소유권과 복사의 미묘함

위 코드에서 `username`을 여러 번 clone하고, `client_id`를 복사하는 것을 볼 수 있다. 이는 Rust의 소유권 규칙 때문이다:

```rust
let username_copy = username.clone();
let client_id_copy = client_id;
```

`username`은 `String`이므로 `Clone`이 필요하다. 두 태스크가 각각 독립적인 복사본을 소유한다. `client_id`는 `Uuid`인데, 이는 `Copy` trait를 구현하므로 할당만으로 복사된다.

`state.clone()`은 `AppState`를 복사하는데, 내부적으로 `Arc`만 clone된다. 실제 HashMap은 복사되지 않고, 참조 카운트만 증가한다. 따라서 비용이 저렴하다.

## 클라이언트 테스트

JavaScript로 간단한 클라이언트를 만들어 테스트할 수 있다:

```html
<!DOCTYPE html>
<html>
<head>
    <title>WebSocket Chat</title>
</head>
<body>
    <h1>Chat Room</h1>
    <div id="messages" style="height: 400px; overflow-y: auto; border: 1px solid #ccc; padding: 10px;"></div>
    <input id="messageInput" type="text" placeholder="Type a message..." style="width: 80%;">
    <button id="sendButton">Send</button>

    <script>
        const username = prompt("Enter your username:");
        const ws = new WebSocket(`ws://localhost:3000/ws?username=${encodeURIComponent(username)}`);
        
        const messagesDiv = document.getElementById('messages');
        const messageInput = document.getElementById('messageInput');
        const sendButton = document.getElementById('sendButton');
        
        ws.onmessage = (event) => {
            const message = JSON.parse(event.data);
            const div = document.createElement('div');
            
            if (message.type === 'join') {
                div.textContent = `${message.username} joined`;
                div.style.color = 'green';
            } else if (message.type === 'leave') {
                div.textContent = `${message.username} left`;
                div.style.color = 'red';
            } else if (message.type === 'message') {
                div.textContent = `${message.username}: ${message.content}`;
            }
            
            messagesDiv.appendChild(div);
            messagesDiv.scrollTop = messagesDiv.scrollHeight;
        };
        
        sendButton.onclick = () => {
            const content = messageInput.value.trim();
            if (content) {
                ws.send(JSON.stringify({ content }));
                messageInput.value = '';
            }
        };
        
        messageInput.onkeypress = (e) => {
            if (e.key === 'Enter') {
                sendButton.click();
            }
        };
    </script>
</body>
</html>
```

이 파일을 브라우저에서 열고, 여러 탭에서 다른 사용자명으로 접속하면 실시간 채팅을 경험할 수 있다.

## 개선: 메시지 히스토리

새로 연결된 클라이언트에게 최근 메시지를 보내주면 더 좋다:

```rust
use std::collections::VecDeque;

const MAX_HISTORY: usize = 100;

struct AppState {
    clients: Clients,
    message_history: Arc<Mutex<VecDeque<ChatMessage>>>,
}

impl AppState {
    fn new() -> Self {
        Self {
            clients: Arc::new(Mutex::new(HashMap::new())),
            message_history: Arc::new(Mutex::new(VecDeque::new())),
        }
    }
}

async fn add_to_history(history: &Arc<Mutex<VecDeque<ChatMessage>>>, message: ChatMessage) {
    let mut hist = history.lock().await;
    hist.push_back(message);
    
    if hist.len() > MAX_HISTORY {
        hist.pop_front();
    }
}
```

`VecDeque`는 양쪽에서 효율적으로 추가/제거할 수 있는 큐다. 새 메시지를 뒤에 추가하고, 오래된 메시지를 앞에서 제거한다.

클라이언트가 연결되면 히스토리를 보낸다:

```rust

    let (client_sender, mut client_receiver) = mpsc::unbounded_channel::<ChatMessage>();
    
    // 히스토리 전송

        for msg in history.iter() {
            let _ = client_sender.send(msg.clone());
        }
    }
    
    // 클라이언트 등록
    {
        let mut clients = state.clients.lock().await;

            username: username.clone(),
            sender: client_sender,
        });
    }
    
    // ... 나머지 로직
}

## 개선: 프라이빗 메시지

특정 사용자에게만 메시지를 보낼 수 있다:

#[derive(Debug, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]

async fn send_private_message(
    clients: &Clients,
    recipient_id: Uuid,
    message: ChatMessage,
) -> Result<(), String> {
    let clients_lock = clients.lock().await;
    
    if let Some(client_info) = clients_lock.get(&recipient_id) {
        client_info.sender.send(message)
            .map_err(|e| format!("Failed to send: {}", e))?;
        Ok(())
    } else {
        Err("Recipient not found".into())
    }
}
```

`get(&recipient_id)`는 HashMap에서 특정 클라이언트를 찾는다. 찾으면 그 클라이언트의 채널로만 메시지를 보낸다.

## 메모리 누수 방지

클라이언트가 비정상적으로 종료되면 어떻게 될까? 채널의 receiver가 drop되면, sender의 `send()`가 에러를 반환한다. 이를 감지하여 정리할 수 있다:

```rust
async fn broadcast_message_with_cleanup(
    clients: &Clients,
    message: ChatMessage,
    sender_id: Option<Uuid>,
) {
    let mut disconnected = Vec::new();
    
    {
        let clients_lock = clients.lock().await;
        
        for (client_id, client_info) in clients_lock.iter() {
            if let Some(sid) = sender_id {
                if *client_id == sid {
                    continue;
                }
            }
            
            if client_info.sender.send(message.clone()).is_err() {
                disconnected.push(*client_id);
            }
        }
    }
    
    // 연결이 끊긴 클라이언트 제거
    if !disconnected.is_empty() {
        let mut clients_lock = clients.lock().await;
        for client_id in disconnected {
            clients_lock.remove(&client_id);
            println!("Cleaned up disconnected client: {}", client_id);
        }
    }
}
```

먼저 잠금을 획득하고 메시지를 보낸 후, 실패한 클라이언트 ID를 수집한다. 그런 다음 다시 잠금을 획득하여 제거한다. 두 번 잠그는 것이 비효율적으로 보이지만, 잠금을 오래 들고 있는 것보다 낫다.

## 성능 고려사항

WebSocket 서버는 많은 동시 연결을 처리해야 한다. 몇 가지 최적화 팁이 있다:

```rust
// 1. Bounded 채널 사용하여 메모리 제한
let (client_sender, mut client_receiver) = mpsc::channel::<ChatMessage>(100);

// 2. 메시지 크기 제한
const MAX_MESSAGE_SIZE: usize = 4096;

if text.len() > MAX_MESSAGE_SIZE {
    eprintln!("Message too large from client {}", client_id);
    continue;
}

// 3. Arc로 메시지 공유 (큰 메시지의 경우)
type SharedMessage = Arc<ChatMessage>;

async fn broadcast_shared(clients: &Clients, message: Arc<ChatMessage>) {
    let clients_lock = clients.lock().await;
    
    for (_id, info) in clients_lock.iter() {
        let _ = info.sender.send(Arc::clone(&message));
    }
}
```

Bounded 채널은 백프레셔를 제공한다. 클라이언트가 메시지를 소비하는 속도보다 빠르게 메시지가 도착하면, `send()`가 블록된다. 이는 메모리를 보호하지만, 브로드캐스트 함수가 느린 클라이언트 때문에 블록될 수 있다.

`Arc<ChatMessage>`를 사용하면 메시지를 복사하지 않고 참조만 공유한다. 메시지가 크거나 많은 클라이언트에게 보낼 때 효율적이다.

## 인증 통합

실제 애플리케이션에서는 WebSocket 연결도 인증해야 한다:

```rust
use axum::extract::ws::CloseFrame;

#[derive(Debug, Deserialize)]
struct ConnectParams {
    token: String,
}

async fn ws_handler(
    ws: WebSocketUpgrade,
    State(state): State<AppState>,
    Query(params): Query<ConnectParams>,
) -> impl IntoResponse {
    // 토큰 검증
    match validate_token(&params.token) {
        Ok(claims) => {
            ws.on_upgrade(move |socket| {
                handle_client(socket, state, claims.username)
            })
        }
        Err(_) => {
            // 인증 실패 - WebSocket 거부
            ws.on_upgrade(|mut socket| async move {
                let _ = socket.close().await;
            })
        }
    }
}
```

`validate_token`은 Day 34에서 만든 JWT 검증 함수다. 토큰이 유효하지 않으면 연결을 즉시 닫는다.

## 오늘 배운 내용 정리

오늘은 Rust에서 WebSocket을 사용한 실시간 통신을 구현했다. axum의 WebSocket 지원으로 프로토콜 세부사항을 추상화하고, 비즈니스 로직에 집중할 수 있었다. 여러 클라이언트를 관리하기 위해 `Arc<Mutex<HashMap<>>>`를 사용하고, 채널로 메시지를 전달하는 패턴을 익혔다.

특히 `socket.split()`으로 송신과 수신을 분리하고, 각각을 독립적인 태스크에서 처리하는 방법을 배웠다. 소유권과 빌림이 비동기 환경에서 어떻게 작동하는지, `move` 클로저가 어떻게 데이터를 태스크 안으로 이동시키는지 이해했다.

브로드캐스트 패턴은 실시간 협업 도구, 게임 서버, 모니터링 대시보드 등 다양한 곳에 적용할 수 있다. 오늘 만든 채팅 서버는 이런 시스템의 기초가 된다.

내일은 trait을 사용한 추상화를 배운다. 공통 동작을 trait으로 정의하고, 다양한 타입에 구현하며, trait 객체와 제네릭의 차이를 이해한다. 플러그인 가능한 핸들러 시스템을 만들면서 Rust의 다형성을 경험할 것이다.
