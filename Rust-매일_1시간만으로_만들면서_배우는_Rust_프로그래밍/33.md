# Day 33: 데이터베이스 연동

이제 우리가 만든 서버에 영속성을 부여할 차례다. 메모리에만 데이터를 저장하면 서버가 재시작될 때마다 모든 정보가 사라진다. 실제 서비스에서는 사용자 정보, 메시지 이력, 설정 등을 데이터베이스에 저장해야 한다. 오늘은 Rust의 비동기 생태계에서 가장 널리 쓰이는 `sqlx` 크레이트를 사용하여 PostgreSQL 데이터베이스와 연동하는 방법을 배운다.

## sqlx가 특별한 이유

Rust에는 여러 데이터베이스 라이브러리가 있지만, sqlx는 몇 가지 독특한 장점을 가지고 있다. 첫째, 완전히 비동기로 작동하여 tokio 런타임과 완벽하게 통합된다. 둘째, 컴파일 타임에 SQL 쿼리를 검증할 수 있다. 셋째, 연결 풀을 내장하고 있어 별도의 라이브러리 없이도 효율적인 연결 관리가 가능하다.

무엇보다 sqlx는 Rust의 소유권 시스템과 자연스럽게 어우러진다. 데이터베이스에서 가져온 데이터를 구조체로 변환할 때, 소유권 이전과 빌림 규칙이 그대로 적용되어 메모리 안전성을 보장받는다.

## 프로젝트 설정

먼저 `Cargo.toml`에 필요한 의존성을 추가한다:

```toml
[dependencies]
tokio = { version = "1", features = ["full"] }
sqlx = { version = "0.7", features = ["runtime-tokio-rustls", "postgres", "chrono"] }
chrono = "0.4"
serde = { version = "1.0", features = ["derive"] }
```

PostgreSQL이 설치되어 있다고 가정하고, 다음과 같이 데이터베이스를 생성한다:

```bash
createdb chatdb
```

환경 변수로 데이터베이스 URL을 설정한다:

```bash
export DATABASE_URL="postgres://username:password@localhost/chatdb"
```

## 메시지 저장 시스템 만들기

우리는 채팅 메시지를 저장하는 간단한 시스템을 만들 것이다. 먼저 메시지를 표현하는 구조체를 정의한다:

```rust
use chrono::{DateTime, Utc};
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize, sqlx::FromRow)]
struct Message {
    id: i64,
    username: String,
    content: String,
    created_at: DateTime<Utc>,
}

#[derive(Debug, Serialize, Deserialize)]
struct NewMessage {
    username: String,
    content: String,
}
```

여기서 `sqlx::FromRow`는 derive 매크로다. 이것은 데이터베이스 row를 자동으로 우리의 구조체로 변환해준다. 내부적으로는 각 컬럼의 값을 구조체 필드로 이동시키는데, 이 과정에서 소유권이 새로 생성된 구조체로 이전된다.

## 데이터베이스 스키마 생성

테이블을 생성하는 마이그레이션 코드를 작성한다:

```rust
use sqlx::{PgPool, postgres::PgPoolOptions};

async fn create_schema(pool: &PgPool) -> Result<(), sqlx::Error> {
    sqlx::query(
        r#"
        CREATE TABLE IF NOT EXISTS messages (
            id BIGSERIAL PRIMARY KEY,
            username VARCHAR(255) NOT NULL,
            content TEXT NOT NULL,
            created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
        )
        "#
    )
    .execute(pool)
    .await?;
    
    Ok(())
}
```

이 함수는 `&PgPool`을 빌린다. 연결 풀은 여러 곳에서 동시에 사용되므로 소유권을 가져가면 안 된다. 참조를 사용하면 함수가 끝난 후에도 호출자가 계속 풀을 사용할 수 있다.

`sqlx::query` 매크로는 SQL 문자열을 받아 실행 가능한 쿼리 객체를 만든다. `.execute(pool)`는 실제로 쿼리를 실행하고, `.await`로 비동기 작업이 완료되기를 기다린다.

## 연결 풀 초기화

연결 풀은 데이터베이스 연결을 미리 여러 개 만들어두고 재사용하는 메커니즘이다. 매번 새 연결을 만드는 것은 비용이 크기 때문에, 풀을 통해 성능을 크게 향상시킬 수 있다:

```rust
async fn init_pool(database_url: &str) -> Result<PgPool, sqlx::Error> {
    let pool = PgPoolOptions::new()
        .max_connections(5)
        .connect(database_url)
        .await?;
    
    create_schema(&pool).await?;
    
    Ok(pool)
}
```

`PgPoolOptions::new()`는 빌더 패턴을 사용한다. `.max_connections(5)`는 최대 5개의 연결을 풀에 유지하도록 설정한다. `.connect()`는 실제로 데이터베이스에 연결하고 `PgPool`을 반환하는데, 이 풀은 `Arc`와 비슷하게 내부적으로 참조 카운팅을 사용한다. 따라서 `clone()`해도 복사본이 아닌 같은 풀을 가리키는 새 핸들이 생성된다.

## 메시지 저장하기

이제 메시지를 데이터베이스에 저장하는 함수를 만든다:

```rust
async fn save_message(
    pool: &PgPool,
    new_msg: NewMessage,
) -> Result<Message, sqlx::Error> {
    let message = sqlx::query_as::<_, Message>(
        r#"
        INSERT INTO messages (username, content)
        VALUES ($1, $2)
        RETURNING id, username, content, created_at
        "#
    )
    .bind(&new_msg.username)
    .bind(&new_msg.content)
    .fetch_one(pool)
    .await?;
    
    Ok(message)
}
```

여기서 주목할 점이 몇 가지 있다. `sqlx::query_as::<_, Message>`는 쿼리 결과를 `Message` 구조체로 변환한다. 타입 추론이 가능한 경우 `_`로 첫 번째 타입 매개변수를 생략할 수 있다.

`.bind()` 메서드는 SQL의 플레이스홀더(`$1`, `$2`)에 값을 바인딩한다. 여기서 `&new_msg.username`을 사용하는데, 이는 소유권을 이전하지 않고 빌리기만 한다. sqlx는 내부적으로 이 문자열을 데이터베이스로 전송할 때 복사하거나 참조를 유지한다.

`RETURNING` 절은 PostgreSQL의 기능으로, 삽입된 row를 즉시 반환받는다. 이렇게 하면 별도의 SELECT 쿼리 없이 자동 생성된 ID와 타임스탬프를 얻을 수 있다.

## 메시지 조회하기

저장된 메시지들을 가져오는 함수도 만든다:

```rust
async fn get_recent_messages(
    pool: &PgPool,
    limit: i64,
) -> Result<Vec<Message>, sqlx::Error> {
    let messages = sqlx::query_as::<_, Message>(
        r#"
        SELECT id, username, content, created_at
        FROM messages
        ORDER BY created_at DESC
        LIMIT $1
        "#
    )
    .bind(limit)
    .fetch_all(pool)
    .await?;
    
    Ok(messages)
}
```

`.fetch_all()`은 쿼리 결과의 모든 row를 `Vec<Message>`로 반환한다. 각 row가 `Message` 구조체로 변환되고, 이 구조체들의 소유권이 반환되는 벡터로 이전된다. 함수를 호출한 쪽에서는 이 벡터를 완전히 소유하게 되므로, 필요에 따라 수정하거나 이동시킬 수 있다.

## 특정 사용자의 메시지 조회

좀 더 복잡한 쿼리를 작성해본다:

```rust
async fn get_messages_by_user(
    pool: &PgPool,
    username: &str,
    limit: i64,
) -> Result<Vec<Message>, sqlx::Error> {
    let messages = sqlx::query_as::<_, Message>(
        r#"
        SELECT id, username, content, created_at
        FROM messages
        WHERE username = $1
        ORDER BY created_at DESC
        LIMIT $2
        "#
    )
    .bind(username)
    .bind(limit)
    .fetch_all(pool)
    .await?;
    
    Ok(messages)
}
```

여기서 `username: &str`을 받는다. `String`이 아닌 `&str`을 사용하는 것은 일반적인 Rust 관용구다. 호출자가 `String`을 소유하고 있든 문자열 리터럴을 전달하든, 참조로 받으면 모든 경우를 처리할 수 있다. 함수 내부에서는 읽기만 하면 되므로 소유권이 필요 없다.

## 웹 서버와 통합하기

이제 실제 HTTP 서버에 데이터베이스 연결을 통합한다. axum을 사용하는 예제다:

```rust
use axum::{
    Router,
    routing::{get, post},
    extract::{State, Json, Query},
    response::IntoResponse,
    http::StatusCode,
};
use std::sync::Arc;

#[derive(Clone)]
struct AppState {
    db_pool: PgPool,
}

async fn create_message(
    State(state): State<Arc<AppState>>,
    Json(new_msg): Json<NewMessage>,
) -> impl IntoResponse {
    match save_message(&state.db_pool, new_msg).await {
        Ok(message) => (StatusCode::CREATED, Json(message)),
        Err(e) => {
            eprintln!("Database error: {}", e);
            (StatusCode::INTERNAL_SERVER_ERROR, Json(Message {
                id: 0,
                username: String::new(),
                content: String::from("Error saving message"),
                created_at: Utc::now(),
            }))
        }
    }
}
```

여기서 중요한 패턴이 등장한다. `AppState`는 데이터베이스 풀을 포함하고, `Arc`로 감싸서 여러 핸들러가 공유할 수 있게 한다. `State(state): State<Arc<AppState>>`는 axum의 추출자(extractor)로, 요청마다 이 상태에 접근할 수 있게 해준다.

`&state.db_pool`을 `save_message`에 전달할 때, `Arc` 내부의 풀을 빌린다. `Arc`는 불변 참조를 안전하게 공유할 수 있게 해주고, `PgPool` 자체가 내부적으로 동기화를 처리하므로 여러 스레드나 태스크에서 동시에 사용해도 안전하다.

아래는 메시지 조회 엔드포인트다:

```rust
#[derive(Deserialize)]
struct ListParams {
    limit: Option<i64>,
    username: Option<String>,
}

async fn list_messages(
    State(state): State<Arc<AppState>>,
    Query(params): Query<ListParams>,
) -> impl IntoResponse {
    let limit = params.limit.unwrap_or(50);
    
    let messages = if let Some(username) = params.username {
        get_messages_by_user(&state.db_pool, &username, limit).await
    } else {
        get_recent_messages(&state.db_pool, limit).await
    };
    
    match messages {
        Ok(msgs) => (StatusCode::OK, Json(msgs)),
        Err(e) => {
            eprintln!("Database error: {}", e);
            (StatusCode::INTERNAL_SERVER_ERROR, Json(Vec::<Message>::new()))
        }
    }
}
```

`params.username`이 `Option<String>`이므로, `if let Some(username)`으로 값이 있는지 확인한다. 값이 있으면 `&username`으로 빌려서 함수에 전달한다. 이렇게 하면 `String`을 함수 안으로 이동시키지 않고도 사용할 수 있다.

## 전체 서버 구성

이제 모든 부분을 조합하여 완전한 서버를 만든다:

```rust
#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let database_url = std::env::var("DATABASE_URL")
        .expect("DATABASE_URL must be set");
    
    let pool = init_pool(&database_url).await?;
    
    let app_state = Arc::new(AppState {
        db_pool: pool,
    });
    
    let app = Router::new()
        .route("/messages", post(create_message))
        .route("/messages", get(list_messages))
        .with_state(app_state);
    
    let listener = tokio::net::TcpListener::bind("127.0.0.1:3000")
        .await?;
    
    println!("Server running on http://127.0.0.1:3000");
    
    axum::serve(listener, app).await?;
    
    Ok(())
}
```

`init_pool()`은 연결 풀을 생성하고, 이를 `AppState`에 담아 `Arc`로 감싼다. `.with_state(app_state)`는 모든 라우트 핸들러가 이 상태를 사용할 수 있게 만든다.

## 연결 풀과 소유권의 관계

연결 풀을 사용할 때 소유권 시스템이 어떻게 작동하는지 이해하는 것이 중요하다. 다음 다이어그램은 연결 풀의 동작을 보여준다:

```
                    Arc<AppState>
                         |
                    db_pool: PgPool
                         |
        +----------------+----------------+
        |                |                |
   Handler 1         Handler 2       Handler 3
        |                |                |
   .acquire()       .acquire()       .acquire()
        |                |                |
   Connection 1    Connection 2    Connection 3
        |                |                |
   Query Result    Query Result    Query Result
        |                |                |
   .release()      .release()      .release()
```

각 핸들러가 쿼리를 실행할 때, 풀에서 연결을 빌려온다. 쿼리가 완료되면 연결이 자동으로 풀로 반환된다. 이 모든 과정에서 Rust의 RAII(Resource Acquisition Is Initialization) 패턴이 적용되어, 연결이 스코프를 벗어나면 자동으로 정리된다.

## 트랜잭션 사용하기

여러 쿼리를 원자적으로 실행해야 할 때는 트랜잭션을 사용한다:

```rust
async fn transfer_message_ownership(
    pool: &PgPool,
    message_id: i64,
    new_owner: String,
) -> Result<(), sqlx::Error> {
    let mut tx = pool.begin().await?;
    
    // 메시지가 존재하는지 확인
    let message = sqlx::query_as::<_, Message>(
        "SELECT id, username, content, created_at FROM messages WHERE id = $1"
    )
    .bind(message_id)
    .fetch_optional(&mut *tx)
    .await?;
    
    if message.is_none() {
        tx.rollback().await?;
        return Err(sqlx::Error::RowNotFound);
    }
    
    // 소유권 이전 기록
    sqlx::query(
        "INSERT INTO ownership_history (message_id, old_owner, new_owner) 
         SELECT $1, username, $2 FROM messages WHERE id = $1"
    )
    .bind(message_id)
    .bind(&new_owner)
    .execute(&mut *tx)
    .await?;
    
    // 메시지 소유자 업데이트
    sqlx::query("UPDATE messages SET username = $1 WHERE id = $2")
        .bind(&new_owner)
        .bind(message_id)
        .execute(&mut *tx)
        .await?;
    
    tx.commit().await?;
    
    Ok(())
}
```

`pool.begin()`은 새 트랜잭션을 시작하고 `Transaction` 객체를 반환한다. 이 객체는 가변 참조(`&mut`)로 사용되어야 한다. `&mut *tx`는 트랜잭션의 가변 참조를 얻는 관용구다.

트랜잭션은 소유권 시스템과 잘 어울린다. `tx`가 스코프를 벗어나면 자동으로 롤백되므로, 중간에 에러가 발생해도 데이터베이스가 일관된 상태를 유지한다. 명시적으로 `.commit()`을 호출해야만 변경사항이 영구적으로 반영된다.

## 컴파일 타임 쿼리 검증

sqlx의 강력한 기능 중 하나는 `query!` 매크로를 사용하여 컴파일 타임에 SQL을 검증하는 것이다:

```rust
async fn get_message_by_id(
    pool: &PgPool,
    id: i64,
) -> Result<Option<Message>, sqlx::Error> {
    let message = sqlx::query_as!(
        Message,
        r#"
        SELECT id, username, content, created_at
        FROM messages
        WHERE id = $1
        "#,
        id
    )
    .fetch_optional(pool)
    .await?;
    
    Ok(message)
}
```

`query_as!` 매크로는 빌드 시점에 실제 데이터베이스에 연결하여 쿼리를 검증한다. 컬럼 이름이 틀렸거나 타입이 맞지 않으면 컴파일 에러가 발생한다. 이를 위해서는 환경 변수 `DATABASE_URL`이 설정되어 있어야 한다.

이 방식의 장점은 런타임 에러를 컴파일 타임으로 앞당긴다는 것이다. 오타나 스키마 불일치를 배포 전에 발견할 수 있다.

## 연결 풀 크기 조정

적절한 연결 풀 크기는 애플리케이션의 특성에 따라 다르다. 다음은 고려해야 할 사항들이다:

```rust
async fn init_pool_advanced(database_url: &str) -> Result<PgPool, sqlx::Error> {
    let pool = PgPoolOptions::new()
        .max_connections(20)           // 최대 연결 수
        .min_connections(5)            // 최소 유지 연결 수
        .acquire_timeout(Duration::from_secs(30))  // 연결 획득 타임아웃
        .idle_timeout(Duration::from_secs(600))    // 유휴 연결 유지 시간
        .max_lifetime(Duration::from_secs(3600))   // 연결 최대 수명
        .connect(database_url)
        .await?;
    
    Ok(pool)
}
```

CPU 코어가 4개라면, 최대 연결 수를 `코어 수 * 2 + 1` 정도로 설정하는 것이 일반적이다. 하지만 데이터베이스 서버의 연결 제한도 고려해야 한다. PostgreSQL의 기본 최대 연결 수는 100개이므로, 여러 애플리케이션 인스턴스가 동시에 연결한다면 각 인스턴스의 풀 크기를 줄여야 한다.

## 에러 처리 개선

데이터베이스 에러를 세밀하게 처리하면 더 나은 사용자 경험을 제공할 수 있다:

```rust
use axum::http::StatusCode;

async fn create_message_improved(
    State(state): State<Arc<AppState>>,
    Json(new_msg): Json<NewMessage>,
) -> Result<(StatusCode, Json<Message>), (StatusCode, String)> {
    if new_msg.username.is_empty() {
        return Err((StatusCode::BAD_REQUEST, "Username cannot be empty".into()));
    }
    
    if new_msg.content.len() > 1000 {
        return Err((StatusCode::BAD_REQUEST, "Message too long".into()));
    }
    
    match save_message(&state.db_pool, new_msg).await {
        Ok(message) => Ok((StatusCode::CREATED, Json(message))),
        Err(sqlx::Error::Database(db_err)) => {
            eprintln!("Database error: {}", db_err);
            Err((StatusCode::INTERNAL_SERVER_ERROR, "Database error".into()))
        }
        Err(sqlx::Error::PoolTimedOut) => {
            eprintln!("Connection pool timeout");
            Err((StatusCode::SERVICE_UNAVAILABLE, "Service busy, try again later".into()))
        }
        Err(e) => {
            eprintln!("Unexpected error: {}", e);
            Err((StatusCode::INTERNAL_SERVER_ERROR, "Internal error".into()))
        }
    }
}
```

`sqlx::Error`는 다양한 에러 종류를 구분할 수 있는 enum이다. 패턴 매칭으로 각 에러에 적절히 대응하면 클라이언트가 문제를 이해하기 쉽다.

## 마이그레이션 관리

프로덕션 환경에서는 스키마 변경을 체계적으로 관리해야 한다. sqlx는 마이그레이션 도구를 제공한다:

```rust
use sqlx::migrate::MigrateDatabase;

async fn run_migrations(pool: &PgPool) -> Result<(), sqlx::Error> {
    sqlx::migrate!("./migrations")
        .run(pool)
        .await?;
    
    Ok(())
}
```

`migrations` 디렉토리에 SQL 파일을 순서대로 배치한다:

```
migrations/
  ├── 001_create_messages_table.sql
  ├── 002_add_username_index.sql
  └── 003_add_ownership_history.sql
```

각 파일은 다음과 같은 형식이다:

```sql
-- 001_create_messages_table.sql
CREATE TABLE messages (
    id BIGSERIAL PRIMARY KEY,
    username VARCHAR(255) NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

마이그레이션은 한 번만 실행되며, sqlx가 내부적으로 어떤 마이그레이션이 이미 적용되었는지 추적한다.

## 성능 고려사항

데이터베이스 쿼리는 종종 병목이 된다. 몇 가지 최적화 기법이 있다:

```rust
async fn get_messages_with_user_info(
    pool: &PgPool,
    limit: i64,
) -> Result<Vec<MessageWithUser>, sqlx::Error> {
    // JOIN을 사용하여 한 번의 쿼리로 관련 데이터를 가져온다
    let messages = sqlx::query_as::<_, MessageWithUser>(
        r#"
        SELECT 
            m.id,
            m.username,
            m.content,
            m.created_at,
            COUNT(r.id) as reply_count
        FROM messages m
        LEFT JOIN replies r ON r.message_id = m.id
        GROUP BY m.id, m.username, m.content, m.created_at
        ORDER BY m.created_at DESC
        LIMIT $1
        "#
    )
    .bind(limit)
    .fetch_all(pool)
    .await?;
    
    Ok(messages)
}
```

N+1 쿼리 문제를 피하기 위해 JOIN을 사용한다. 각 메시지마다 별도로 답글 수를 조회하는 대신, 한 번의 쿼리로 모든 정보를 가져온다.

## 준비된 쿼리 재사용

동일한 쿼리를 반복적으로 실행한다면 준비된 쿼리를 캐싱할 수 있다:

```rust
struct MessageRepository {
    pool: PgPool,
}

impl MessageRepository {
    fn new(pool: PgPool) -> Self {
        Self { pool }
    }
    
    async fn save(&self, new_msg: NewMessage) -> Result<Message, sqlx::Error> {
        // sqlx는 자동으로 쿼리를 준비하고 캐싱한다
        let message = sqlx::query_as::<_, Message>(
            "INSERT INTO messages (username, content) VALUES ($1, $2)
             RETURNING id, username, content, created_at"
        )
        .bind(&new_msg.username)
        .bind(&new_msg.content)
        .fetch_one(&self.pool)
        .await?;
        
        Ok(message)
    }
    
    async fn find_by_id(&self, id: i64) -> Result<Option<Message>, sqlx::Error> {
        let message = sqlx::query_as::<_, Message>(
            "SELECT id, username, content, created_at FROM messages WHERE id = $1"
        )
        .bind(id)
        .fetch_optional(&self.pool)
        .await?;
        
        Ok(message)
    }
}
```

Repository 패턴을 사용하면 데이터베이스 접근 로직을 한 곳에 모을 수 있다. `pool`을 구조체에 저장하고, 각 메서드에서 `&self`로 접근한다. 이렇게 하면 풀을 여러 번 전달하지 않아도 된다.

## 오늘 배운 내용 정리

오늘은 Rust에서 데이터베이스를 다루는 방법을 배웠다. sqlx를 사용하여 비동기적으로 PostgreSQL에 연결하고, 연결 풀을 관리하며, 쿼리를 실행하는 방법을 익혔다. 특히 Rust의 소유권 시스템이 데이터베이스 작업에서 어떻게 적용되는지 이해했다.

연결 풀은 `Arc`처럼 공유 가능하며, 참조로 빌려서 사용한다. 쿼리 결과는 구조체로 변환되어 소유권이 호출자에게 이전된다. 트랜잭션은 RAII 패턴을 활용하여 자동으로 롤백되고, 명시적으로 커밋해야만 변경사항이 반영된다.

내일은 인증과 보안을 다룬다. JWT 토큰을 검증하고, 비밀번호를 안전하게 저장하며, 미들웨어로 보호된 엔드포인트를 만드는 방법을 배울 것이다. 데이터베이스에 사용자 정보를 저장하고, 세션을 관리하는 방법도 살펴본다.
