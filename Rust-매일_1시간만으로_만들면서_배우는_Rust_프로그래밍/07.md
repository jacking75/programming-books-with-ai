# Day 7: 참조와 빌림 - 데이터 공유하기

## 소유권의 딜레마
지난 Day에서 소유권을 배웠다. 값을 함수에 전달하면 소유권이 이동하고, 원래 변수는 사용할 수 없게 된다. 이것이 메모리 안전성을 보장하지만, 실용적인 문제가 생긴다.

```rust
fn main() {
    let log = String::from("에러: 데이터베이스 연결 실패");
    print_log(log);
    // log를 다시 사용하고 싶은데...
    // save_to_file(log); // 컴파일 에러!
}

fn print_log(message: String) {
    println!("[LOG] {}", message);
}
```

로그 메시지를 출력하고 나서 파일에도 저장하고 싶다. 하지만 `print_log`를 호출하는 순간 소유권이 이동했으므로 불가능하다. 소유권을 되돌려받는 방법도 있지만 번거롭다. 이 문제의 해법이 바로 **참조(reference)**와 **빌림(borrowing)**이다.
  

## 불변 참조: 읽기만 하기
참조는 데이터를 소유하지 않고 가리키기만 한다. `&` 기호로 참조를 만든다.

```rust
fn main() {
    let log = String::from("에러: 데이터베이스 연결 실패");
    print_log(&log);  // 참조를 전달
    save_to_file(&log); // 여전히 사용 가능!
    println!("원본: {}", log); // 이것도 가능!
}

fn print_log(message: &String) {
    println!("[LOG] {}", message);
}

fn save_to_file(message: &String) {
    println!("파일 저장: {}", message);
}
```

`&log`는 `log`의 참조를 생성한다. 소유권은 여전히 `main`의 `log`가 가지고 있고, 함수들은 그것을 잠시 빌려서 읽기만 한다. 함수가 끝나도 원본은 그대로 유효하다.

참조의 타입은 `&String`이다. "String의 참조"라고 읽는다. 함수 시그니처만 봐도 소유권을 가져가지 않는다는 것을 알 수 있다.
  

## 참조의 메모리 구조
참조가 메모리에서 어떻게 작동하는지 이해하면 더 명확해진다.

```
스택                                힙
┌──────────────┐
│ log:         │
│   ptr: 0x100 │────────┐
│   len: 30    │        │      ┌─────────────────────┐
│   cap: 30    │        └─────>│ "에러: 데이터베이..."│
└──────────────┘               └─────────────────────┘
       ↑
       │ (가리킴)
       │
┌──────────────┐
│ message:     │
│   0x100      │ (스택의 log 주소)
└──────────────┘
```

`log`는 힙의 문자열 데이터를 가리킨다. `&log`는 스택의 `log` 변수 자체를 가리킨다. 참조는 포인터의 포인터가 아니라, 원본 변수를 가리키는 포인터다. 참조를 따라가면 원본 데이터에 접근할 수 있다.
  

## 빌림 규칙 1: 여러 개의 불변 참조
동시에 여러 개의 불변 참조를 만들 수 있다. 읽기만 한다면 문제가 없기 때문이다.

```rust
fn main() {
    let log = String::from("서버 시작됨");
    
    let ref1 = &log;
    let ref2 = &log;
    let ref3 = &log;
    
    println!("ref1: {}", ref1);
    println!("ref2: {}", ref2);
    println!("ref3: {}", ref3);
    println!("원본: {}", log);
}
```

모두 같은 데이터를 읽고 있다. 누구도 데이터를 변경하지 않으므로 안전하다. 이것은 실제 프로그램에서 매우 유용하다. 같은 설정 객체를 여러 함수가 읽어야 하는 경우를 생각해보자.

```rust
struct Config {
    host: String,
    port: u16,
}

fn start_server(config: &Config) {
    println!("서버 시작: {}:{}", config.host, config.port);
}

fn log_config(config: &Config) {
    println!("설정 로그: {}:{}", config.host, config.port);
}

fn validate_config(config: &Config) -> bool {
    !config.host.is_empty() && config.port > 0
}

fn main() {
    let config = Config {
        host: String::from("127.0.0.1"),
        port: 8080,
    };
    
    if validate_config(&config) {
        log_config(&config);
        start_server(&config);
    }
}
```

`config`의 소유권은 `main`에 그대로 있고, 각 함수는 필요할 때 참조를 받아서 읽는다. 소유권을 이동시켰다면 첫 번째 함수 호출 후에는 사용할 수 없었을 것이다.
  

## 가변 참조: 데이터 변경하기
데이터를 변경하려면 가변 참조가 필요하다. `&mut` 키워드를 사용한다.

```rust
fn main() {
    let mut log = String::from("서버 시작");
    
    add_timestamp(&mut log);
    println!("{}", log); // "2024-03-15 10:30:00 서버 시작"
    
    add_level(&mut log);
    println!("{}", log); // "[INFO] 2024-03-15 10:30:00 서버 시작"
}

fn add_timestamp(message: &mut String) {
    message.insert_str(0, "2024-03-15 10:30:00 ");
}

fn add_level(message: &mut String) {
    message.insert_str(0, "[INFO] ");
}
```

`&mut log`는 가변 참조를 만든다. 함수는 이 참조를 통해 원본 데이터를 변경할 수 있다. 소유권은 여전히 `main`에 있지만, 잠시 동안 변경 권한을 빌려준 것이다.

가변 참조를 만들려면 원본 변수도 `mut`로 선언되어야 한다. 불변 변수의 가변 참조는 만들 수 없다. 이것은 논리적으로 당연하다.
  

## 빌림 규칙 2: 가변 참조는 단 하나만
가변 참조의 핵심 규칙이 있다. **같은 스코프에서 동시에 하나의 가변 참조만 존재할 수 있다.**

```rust
fn main() {
    let mut log = String::from("메시지");
    
    let ref1 = &mut log;
    // let ref2 = &mut log; // 컴파일 에러!
    
    ref1.push_str(" - 수정됨");
    println!("{}", ref1);
}
```

두 개의 가변 참조를 동시에 만들려고 하면 컴파일러가 거부한다. 왜 이런 제약이 있을까?

```rust
// 만약 이것이 가능했다면...
fn main() {
    let mut data = String::from("중요한 데이터");
    
    let ref1 = &mut data;
    let ref2 = &mut data;
    
    ref1.clear(); // 데이터를 지움
    ref2.push_str("새 내용"); // ref2는 이미 무효한 데이터를 가리킴!
}
```

두 개의 가변 참조가 동시에 존재하면, 하나가 데이터를 크게 변경했을 때 다른 참조는 무효해질 수 있다. 이것을 **데이터 레이스(data race)**라고 한다. Rust는 이런 상황을 컴파일 시점에 차단한다.
  

## 빌림 규칙 3: 가변 참조와 불변 참조는 공존할 수 없다
더 엄격한 규칙이 있다. **가변 참조가 있으면 불변 참조를 만들 수 없고, 불변 참조가 있으면 가변 참조를 만들 수 없다.**

```rust
fn main() {
    let mut log = String::from("메시지");
    
    let r1 = &log; // 불변 참조
    let r2 = &log; // 불변 참조 - OK
    // let r3 = &mut log; // 가변 참조 - 에러!
    
    println!("{} {}", r1, r2);
}
```

불변 참조를 가진 사람은 데이터가 변하지 않을 것이라고 믿는다. 그런데 다른 곳에서 가변 참조를 통해 데이터를 바꾼다면? 불변 참조는 예상치 못한 변경을 보게 된다. Rust는 이것도 허용하지 않는다.

하지만 중요한 예외가 있다. **참조의 스코프는 마지막으로 사용된 시점까지**다.

```rust
fn main() {
    let mut log = String::from("메시지");
    
    let r1 = &log;
    let r2 = &log;
    println!("{} {}", r1, r2);
    // r1과 r2는 여기서 마지막으로 사용됨
    
    let r3 = &mut log; // 이제 OK! r1, r2는 더 이상 사용되지 않음
    r3.push_str(" - 수정");
    println!("{}", r3);
}
```

`r1`과 `r2`는 `println!` 이후로 사용되지 않는다. 따라서 스코프가 거기서 끝난다. 그 다음에는 가변 참조를 만들 수 있다. 이것을 **Non-Lexical Lifetimes (NLL)**라고 하며, Rust 2018부터 도입되었다.
  

## 빌림 규칙이 방지하는 문제들
빌림 규칙이 왜 중요한지 구체적인 예를 보자.

**문제 1: 반복 중 수정**

```rust
fn main() {
    let mut logs = vec![
        String::from("로그1"),
        String::from("로그2"),
        String::from("로그3"),
    ];
    
    for log in &logs { // 불변 참조로 반복
        // logs.push(String::from("새 로그")); // 에러!
        println!("{}", log);
    }
}
```

벡터를 반복하면서 동시에 그 벡터를 수정하려고 한다. 만약 가능했다면? 벡터가 재할당되면서 반복자가 무효해질 수 있다. Rust는 이것을 막는다.

**문제 2: 댕글링 참조**

```rust
fn main() {
    let reference;
    {
        let data = String::from("임시 데이터");
        // reference = &data; // 에러!
    } // data가 여기서 해제됨
    // println!("{}", reference); // 무효한 메모리를 참조!
}
```

`data`는 내부 스코프에서만 살아있다. 그것의 참조를 외부로 빼내려고 하면 컴파일러가 거부한다. 참조는 항상 유효한 데이터를 가리켜야 한다.
  

## 실전: 로그 필터링 함수 구현
이제 배운 내용을 실제 프로그램에 적용해보자. 로그 메시지 컬렉션을 필터링하고 변환하는 시스템을 만들 것이다.

```rust
struct LogEntry {
    level: String,
    message: String,
    timestamp: String,
}

impl LogEntry {
    fn new(level: &str, message: &str) -> LogEntry {
        LogEntry {
            level: level.to_string(),
            message: message.to_string(),
            timestamp: String::from("2024-03-15 10:30:00"),
        }
    }
}

struct LogFilter {
    entries: Vec<LogEntry>,
}

impl LogFilter {
    fn new() -> LogFilter {
        LogFilter {
            entries: Vec::new(),
        }
    }
    
    fn add(&mut self, entry: LogEntry) {
        self.entries.push(entry);
    }
    
    // 불변 참조로 읽기만 함
    fn count_by_level(&self, level: &str) -> usize {
        self.entries.iter()
            .filter(|entry| entry.level == level)
            .count()
    }
    
    // 불변 참조를 반환하여 효율적으로 공유
    fn get_entry(&self, index: usize) -> Option<&LogEntry> {
        self.entries.get(index)
    }
    
    // 가변 참조로 모든 메시지에 접두사 추가
    fn add_prefix_to_all(&mut self, prefix: &str) {
        for entry in &mut self.entries {
            entry.message.insert_str(0, prefix);
        }
    }
    
    // 불변 참조를 받아서 조건에 맞는 것만 출력
    fn print_filtered(&self, min_level: &str) {
        for entry in &self.entries {
            if should_print(&entry.level, min_level) {
                println!("[{}] {} - {}", 
                    entry.timestamp, 
                    entry.level, 
                    entry.message);
            }
        }
    }
}

fn should_print(level: &str, min_level: &str) -> bool {
    let levels = ["DEBUG", "INFO", "WARN", "ERROR"];
    let level_idx = levels.iter().position(|&l| l == level).unwrap_or(0);
    let min_idx = levels.iter().position(|&l| l == min_level).unwrap_or(0);
    level_idx >= min_idx
}

fn main() {
    let mut filter = LogFilter::new();
    
    filter.add(LogEntry::new("INFO", "서버 시작됨"));
    filter.add(LogEntry::new("DEBUG", "설정 로드 완료"));
    filter.add(LogEntry::new("ERROR", "데이터베이스 연결 실패"));
    filter.add(LogEntry::new("WARN", "메모리 사용량 높음"));
    
    // 불변 참조로 카운트
    println!("에러 개수: {}", filter.count_by_level("ERROR"));
    
    // 불변 참조로 특정 항목 접근
    if let Some(entry) = filter.get_entry(0) {
        println!("첫 번째 로그: {}", entry.message);
    }
    
    // 가변 참조로 모든 메시지 수정
    filter.add_prefix_to_all("[SERVER] ");
    
    // 불변 참조로 필터링된 출력
    println!("\nWARN 이상만 출력:");
    filter.print_filtered("WARN");
}
```

이 코드에서 참조와 빌림이 어떻게 사용되는지 자세히 살펴보자.

**count_by_level 메서드**는 `&self`를 받는다. 불변 참조로 자기 자신을 빌린다. 데이터를 읽기만 하므로 여러 번 호출할 수 있다. `level` 매개변수도 `&str`로 참조를 받는다. 짧은 문자열이지만 복사하지 않고 참조로 전달한다.

**get_entry 메서드**는 `&LogEntry`를 반환한다. 벡터 안의 항목을 소유권 이동 없이 참조로 제공한다. 호출자는 로그 엔트리를 읽을 수 있지만 소유하지는 않는다.

**add_prefix_to_all 메서드**는 `&mut self`를 받는다. 내부 데이터를 변경해야 하기 때문이다. `for entry in &mut self.entries`는 벡터의 각 항목을 가변 참조로 반복한다. 각 `entry`는 `&mut LogEntry`다.

**print_filtered 메서드**는 `&self`를 받고, `should_print` 함수에 `&entry.level`을 전달한다. 참조의 참조를 만드는 것이다. 모두 읽기만 하므로 문제없다.
  

## 참조의 규칙 시각화
빌림 규칙을 다이어그램으로 정리하면 이렇다.

```
시간 →
───────────────────────────────────────────
let x = ...;           (x가 데이터를 소유)
let r1 = &x;           (불변 참조 1)
let r2 = &x;           (불변 참조 2) ✓ 여러 개 OK
let r3 = &x;           (불변 참조 3)
println!(r1, r2, r3);  (모두 사용)
                       (불변 참조들 스코프 끝)
let m = &mut x;        (가변 참조) ✓ 이제 OK
m.modify();            (사용)
                       (가변 참조 스코프 끝)
let r4 = &x;           (다시 불변 참조) ✓ OK
───────────────────────────────────────────

❌ 불가능한 패턴:
let r = &x;
let m = &mut x;  // 에러: 불변과 가변 동시 존재

let m1 = &mut x;
let m2 = &mut x; // 에러: 가변 참조 2개
```
  

## 참조를 받는 함수 vs 소유권을 받는 함수
언제 참조를 사용하고 언제 소유권을 이동시켜야 할까? 일반적인 가이드라인은 이렇다.

```rust
// 읽기만 한다면: 불변 참조
fn analyze_log(log: &LogEntry) -> bool {
    log.level == "ERROR"
}

// 수정해야 한다면: 가변 참조
fn add_timestamp(log: &mut LogEntry) {
    log.timestamp = String::from("2024-03-15 11:00:00");
}

// 소비해야 한다면: 소유권 이동
fn archive_log(log: LogEntry) {
    // log를 파일에 쓰고 메모리에서 제거
    println!("보관: {}", log.message);
}

// 변환해서 반환한다면: 소유권 이동 + 반환
fn format_log(log: LogEntry) -> String {
    format!("[{}] {}", log.level, log.message)
}
```

대부분의 경우 참조를 사용하는 것이 좋다. 소유권 이동은 정말 필요할 때만 사용한다. 함수 시그니처만 봐도 의도가 명확해진다.
  

## 메서드에서의 self, &self, &mut self
메서드를 정의할 때 `self`를 받는 방법이 세 가지다.

```rust
impl LogEntry {
    // 소유권을 가져감 - 객체를 소비함
    fn into_string(self) -> String {
        format!("{}: {}", self.level, self.message)
    }
    
    // 불변 참조 - 읽기만 함
    fn is_error(&self) -> bool {
        self.level == "ERROR"
    }
    
    // 가변 참조 - 수정함
    fn set_level(&mut self, level: String) {
        self.level = level;
    }
}

fn main() {
    let mut entry = LogEntry::new("INFO", "테스트");
    
    println!("에러인가? {}", entry.is_error()); // &self
    entry.set_level(String::from("ERROR"));     // &mut self
    let text = entry.into_string();             // self (소비됨)
    // entry.is_error(); // 에러! entry는 이미 이동됨
}
```

`into_string`을 호출하면 `entry`가 소비된다. 이후로는 사용할 수 없다. 이름에 `into_`가 붙은 메서드는 보통 소유권을 가져간다는 관례가 있다.
  

## 실전 패턴: 읽기와 쓰기 분리
실제 프로그램에서 자주 사용하는 패턴은 읽기 작업과 쓰기 작업을 명확히 분리하는 것이다.

```rust
struct LogDatabase {
    entries: Vec<LogEntry>,
}

impl LogDatabase {
    fn new() -> LogDatabase {
        LogDatabase {
            entries: Vec::new(),
        }
    }
    
    // 쓰기: 가변 참조
    fn insert(&mut self, entry: LogEntry) {
        self.entries.push(entry);
    }
    
    // 읽기: 불변 참조 반환
    fn find_by_level(&self, level: &str) -> Vec<&LogEntry> {
        self.entries.iter()
            .filter(|entry| entry.level == level)
            .collect()
    }
    
    // 읽기: 불변 참조
    fn total_count(&self) -> usize {
        self.entries.len()
    }
    
    // 쓰기: 가변 참조
    fn clear_old_entries(&mut self, keep_count: usize) {
        if self.entries.len() > keep_count {
            self.entries.drain(0..self.entries.len() - keep_count);
        }
    }
}

fn main() {
    let mut db = LogDatabase::new();
    
    // 쓰기 작업
    db.insert(LogEntry::new("ERROR", "에러 1"));
    db.insert(LogEntry::new("INFO", "정보 1"));
    db.insert(LogEntry::new("ERROR", "에러 2"));
    
    // 읽기 작업들 (연속 호출 가능)
    let total = db.total_count();
    let errors = db.find_by_level("ERROR");
    println!("전체: {}, 에러: {}", total, errors.len());
    
    // 다시 쓰기 작업
    db.clear_old_entries(2);
}
```

읽기 메서드들(`find_by_level`, `total_count`)은 `&self`를 받으므로 연속으로 여러 번 호출할 수 있다. 쓰기 메서드들(`insert`, `clear_old_entries`)은 `&mut self`를 받으므로 한 번에 하나씩만 호출된다.
  

## 참조와 반복자
컬렉션을 반복할 때도 참조가 중요하다.

```rust
fn main() {
    let logs = vec![
        String::from("로그 1"),
        String::from("로그 2"),
        String::from("로그 3"),
    ];
    
    // 소유권 이동 - 벡터가 소비됨
    // for log in logs {
    //     println!("{}", log);
    // }
    // logs를 더 이상 사용할 수 없음
    
    // 불변 참조 - 읽기만
    for log in &logs {
        println!("{}", log);
    }
    println!("여전히 사용 가능: {}", logs.len());
    
    // 가변 참조 - 수정 가능
    let mut logs = logs;
    for log in &mut logs {
        log.push_str(" - 수정됨");
    }
    
    for log in &logs {
        println!("{}", log);
    }
}
```

`for log in &logs`는 벡터의 각 항목을 불변 참조로 반복한다. `log`의 타입은 `&String`이다. `for log in &mut logs`는 가변 참조로 반복하며, `log`의 타입은 `&mut String`이다.
  

## 실습: 로그 통계 분석기
배운 내용을 종합하는 실습이다. 로그를 분석하여 통계를 내는 시스템을 만들어보자.

```rust
struct LogStats {
    total: usize,
    by_level: std::collections::HashMap<String, usize>,
}

impl LogStats {
    fn new() -> LogStats {
        LogStats {
            total: 0,
            by_level: std::collections::HashMap::new(),
        }
    }
    
    // 불변 참조로 로그 컬렉션을 받아서 분석
    fn analyze(&mut self, entries: &[LogEntry]) {
        self.total = entries.len();
        
        for entry in entries {
            let count = self.by_level.entry(entry.level.clone()).or_insert(0);
            *count += 1;
        }
    }
    
    fn print_report(&self) {
        println!("=== 로그 통계 ===");
        println!("전체: {}", self.total);
        println!("\n레벨별:");
        for (level, count) in &self.by_level {
            let percentage = (*count as f64 / self.total as f64) * 100.0;
            println!("  {}: {} ({:.1}%)", level, count, percentage);
        }
    }
}

fn main() {
    let mut db = LogDatabase::new();
    
    db.insert(LogEntry::new("INFO", "서버 시작"));
    db.insert(LogEntry::new("DEBUG", "설정 로드"));
    db.insert(LogEntry::new("ERROR", "연결 실패"));
    db.insert(LogEntry::new("INFO", "재시도 중"));
    db.insert(LogEntry::new("ERROR", "타임아웃"));
    db.insert(LogEntry::new("WARN", "성능 저하"));
    
    let mut stats = LogStats::new();
    stats.analyze(&db.entries); // 불변 참조로 전달
    stats.print_report();
    
    // db는 여전히 사용 가능
    println!("\n데이터베이스 로그 수: {}", db.total_count());
}
```

`analyze` 메서드는 `&[LogEntry]`를 받는다. 이것은 슬라이스의 참조다(다음 Day에서 자세히 다룬다). 벡터를 소유하지 않고 읽기만 한다. 따라서 분석 후에도 원본 데이터베이스를 계속 사용할 수 있다.
  

## 참조의 성능 이점
참조를 사용하면 성능도 향상된다. 큰 데이터 구조를 복사하지 않기 때문이다.

```rust
struct LargeData {
    buffer: Vec<u8>,
}

impl LargeData {
    fn new(size: usize) -> LargeData {
        LargeData {
            buffer: vec![0; size],
        }
    }
}

// 소유권 이동 - 느림
fn process_owned(data: LargeData) {
    println!("처리: {} 바이트", data.buffer.len());
}

// 참조 - 빠름
fn process_ref(data: &LargeData) {
    println!("처리: {} 바이트", data.buffer.len());
}

fn main() {
    let large = LargeData::new(1_000_000); // 1MB
    
    // 참조로 전달 - 포인터 하나만 복사됨 (8바이트)
    process_ref(&large);
    process_ref(&large);
    process_ref(&large);
    
    // 마지막에 소유권 이동
    process_owned(large);
}
```

1MB 데이터를 참조로 전달하면 포인터 하나(8바이트)만 복사된다. 소유권을 이동시키면 벡터의 메타데이터는 복사되지만, 힙 데이터는 그대로 둔다. 하지만 여러 번 전달할 때는 참조가 훨씬 효율적이다.
  

## 흔한 실수와 해결법

**실수 1: 가변 참조와 불변 참조 동시 사용**

```rust
fn main() {
    let mut data = String::from("hello");
    
    let r = &data;
    // data.push_str(" world"); // 에러!
```

**실수 2: 반복 중 컬렉션 수정**

```rust
fn main() {
    let mut logs = vec![String::from("a"), String::from("b")];
    
    // for log in &logs {
    //     logs.push(String::from("c")); // 에러!
    // }
    
    // 해결: 인덱스로 반복하거나, 수정할 항목을 먼저 수집

        .collect();

}
```

## 다음 단계를 위한 준비
오늘 참조와 빌림의 기초를 배웠다. 소유권을 이동하지 않고도 데이터를 공유하는 방법, 그리고 빌림 규칙이 어떻게 메모리 안전성과 동시성 안전성을 보장하는지 이해했다.

핵심 규칙을 다시 정리하면 이렇다. 어느 시점에든 **여러 개의 불변 참조 또는 단 하나의 가변 참조**만 존재할 수 있다. 참조는 항상 유효해야 한다.

다음 Day에서는 슬라이스를 배운다. 슬라이스는 컬렉션의 일부를 참조하는 특별한 참조 타입이다. 문자열 슬라이스(`&str`)와 배열 슬라이스를 사용하여 로그 파서를 만들 것이다. 시간, 레벨, 메시지를 분리하는 실용적인 프로그램을 작성하면서 슬라이스의 강력함을 경험할 것이다.

참조와 빌림은 처음에는 제약처럼 느껴지지만, 곧 자연스러워진다. 컴파일러 메시지를 잘 읽고 따르다 보면, 어느새 안전하고 효율적인 코드를 작성하게 된다. Rust의 빌림 검사기(borrow checker)는 가장 엄격한 선생님이자, 가장 든든한 동료다.
