# Day 17: Mutex로 안전한 변경

어제 우리는 `Arc`를 사용해서 여러 스레드가 데이터를 읽을 수 있게 했다. 하지만 `Arc`만으로는 데이터를 변경할 수 없다. 오늘은 `Mutex`를 배워서 여러 스레드가 안전하게 공유 데이터를 변경하는 방법을 익힌다.

## 문제 상황: 데이터를 변경하고 싶다

서버에서 연결된 클라이언트들의 목록을 유지하고 싶다고 가정하자. 새 클라이언트가 연결되면 목록에 추가하고, 연결이 끊어지면 제거해야 한다.

```rust
let clients = Arc::new(Vec::new());
// 어떻게 여기에 추가할까?
```

`Arc`는 불변 공유만 제공한다. 다음 코드는 컴파일되지 않는다.

```rust
use std::sync::Arc;
use std::thread;

fn main() {
    let data = Arc::new(vec![1, 2, 3]);
    let data_clone = Arc::clone(&data);
    
    thread::spawn(move || {
        data_clone.push(4);  // 컴파일 에러!
    });
}
```

컴파일러는 명확하게 알려준다.

```
error[E0596]: cannot borrow data in an `Arc` as mutable
```

`Arc`를 통해서는 불변 참조만 얻을 수 있다. 여러 스레드가 동시에 같은 데이터를 변경하려고 하면 데이터 레이스가 발생하기 때문이다.

## 상호 배제(Mutual Exclusion)의 필요성

여러 스레드가 같은 데이터에 접근할 때, 한 번에 하나의 스레드만 데이터를 변경할 수 있어야 한다. 이것을 상호 배제라고 한다.

예를 들어, 두 스레드가 동시에 벡터에 요소를 추가하려고 한다고 가정하자.

```
Thread 1:                  Thread 2:
vec.push(1)                vec.push(2)
  ├─ 크기 확인 (3개)         ├─ 크기 확인 (3개)
  ├─ 메모리 확보             ├─ 메모리 확보
  └─ 값 쓰기                └─ 값 쓰기
      ↓                         ↓
결과: 예측 불가능! (충돌, 데이터 손실, 또는 크래시)
```

`Mutex`는 한 번에 하나의 스레드만 데이터에 접근할 수 있게 보장한다.

```
Thread 1:                  Thread 2:
mutex.lock()               (대기중...)
vec.push(1)                (대기중...)
mutex.unlock()             mutex.lock()
                           vec.push(2)
                           mutex.unlock()
```

이것을 시각화하면 다음과 같다.

```
      Mutex (자물쇠)
          │
    ┌─────┼─────┐
    │  [Lock]   │
    │           │
    │  Vector   │
    │           │
    └───────────┘

Thread 1: 🔑 (열쇠를 가짐) → 접근 가능
Thread 2: ⏳ (대기중)      → 접근 불가
Thread 3: ⏳ (대기중)      → 접근 불가
```

## Mutex의 기본 사용법

`Mutex`는 데이터를 감싸서 보호한다. 데이터에 접근하려면 반드시 락(lock)을 획득해야 한다.

```rust
use std::sync::Mutex;

fn main() {
    let counter = Mutex::new(0);
    
    {
        let mut num = counter.lock().unwrap();
        *num += 1;
        println!("카운터: {}", *num);
    } // MutexGuard가 여기서 드롭되면서 락이 해제됨
    
    println!("락 해제 후 접근: {:?}", counter);
}
```

**핵심 개념들:**

**1. Mutex::new()로 생성**

`Mutex::new(value)`는 값을 `Mutex`로 감싼다. 이제 이 값에 직접 접근할 수 없다.

**2. lock()으로 접근**

`lock()`을 호출하면 락을 획득할 때까지 현재 스레드가 블록된다. 락을 획득하면 `MutexGuard`가 반환된다.

**3. MutexGuard를 통한 접근**

`MutexGuard`는 스마트 포인터처럼 작동한다. `Deref`와 `DerefMut`을 구현해서 내부 값에 접근할 수 있게 해준다.

**4. 자동 락 해제**

`MutexGuard`가 스코프를 벗어나면 자동으로 락이 해제된다. 이것은 RAII(Resource Acquisition Is Initialization) 패턴이다.

```rust
{
    let guard = mutex.lock().unwrap();
    // 락 획득됨
    // guard를 사용해서 데이터 접근
} // guard가 드롭되면서 락 자동 해제
```

## Arc<Mutex<T>> 패턴

실제로는 `Arc`와 `Mutex`를 함께 사용한다. `Arc`는 여러 스레드가 같은 `Mutex`를 소유할 수 있게 하고, `Mutex`는 안전한 변경을 가능하게 한다.

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];
    
    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            let mut num = counter.lock().unwrap();
            *num += 1;
        });
        handles.push(handle);
    }
    
    for handle in handles {
        handle.join().unwrap();
    }
    
    println!("결과: {}", *counter.lock().unwrap());
}
```

이 코드는 10개의 스레드가 안전하게 같은 카운터를 증가시킨다. 결과는 항상 10이다.

구조를 시각화하면 다음과 같다.

```
         Arc (공유 소유권)
              │
    ┌─────────┼─────────┐
    │         │         │
Thread 1   Thread 2  Thread 3
    │         │         │
    └──── Mutex ────────┘
         (상호 배제)
              │
           counter
```

각 스레드는 자신의 `Arc` 클론을 가지고 있다. 모든 `Arc`는 같은 `Mutex`를 가리킨다. 각 스레드는 `lock()`을 호출해서 순차적으로 카운터에 접근한다.

## 실전: 공유 카운터 구현

이제 TCP 서버에 공유 카운터를 추가해보자. 요청 수와 연결 수를 추적한다.

```rust
use std::sync::{Arc, Mutex};
use std::net::{TcpListener, TcpStream};
use std::io::{BufReader, BufRead, Write};
use std::thread;

struct ServerStats {
    active_connections: usize,
    total_requests: usize,
}

impl ServerStats {
    fn new() -> Self {
        ServerStats {
            active_connections: 0,
            total_requests: 0,
        }
    }
    
    fn connection_started(&mut self) {
        self.active_connections += 1;
    }
    
    fn connection_ended(&mut self) {
        self.active_connections -= 1;
    }
    
    fn request_received(&mut self) {
        self.total_requests += 1;
    }
    
    fn print(&self) {
        println!("활성 연결: {}", self.active_connections);
        println!("총 요청: {}", self.total_requests);
    }
}

fn handle_client(
    mut stream: TcpStream,
    stats: Arc<Mutex<ServerStats>>,
) -> std::io::Result<()> {
    let peer_addr = stream.peer_addr()?;
    
    // 연결 시작
    {
        let mut stats = stats.lock().unwrap();
        stats.connection_started();
        println!("[{}] 연결됨. 활성: {}", peer_addr, stats.active_connections);
    } // 락 해제
    
    writeln!(stream, "에코 서버에 오신 것을 환영합니다!")?;
    writeln!(stream, "'stats' 입력시 서버 통계를 볼 수 있습니다.")?;
    
    let mut reader = BufReader::new(stream.try_clone()?);
    let mut line = String::new();
    
    loop {
        line.clear();
        match reader.read_line(&mut line) {
            Ok(0) => break,
            Ok(_) => {
                // 요청 카운트 증가
                {
                    let mut stats = stats.lock().unwrap();
                    stats.request_received();
                }
                
                let trimmed = line.trim();
                
                if trimmed == "stats" {
                    let stats = stats.lock().unwrap();
                    writeln!(stream, "=== 서버 통계 ===")?;
                    writeln!(stream, "활성 연결: {}", stats.active_connections)?;
                    writeln!(stream, "총 요청: {}", stats.total_requests)?;
                    writeln!(stream, "================")?;
                } else {
                    write!(stream, "Echo: {}", line)?;
                }
            }
            Err(e) => {
                eprintln!("[{}] 에러: {}", peer_addr, e);
                break;
            }
        }
    }
    
    // 연결 종료
    {
        let mut stats = stats.lock().unwrap();
        stats.connection_ended();
        println!("[{}] 연결 종료. 활성: {}", peer_addr, stats.active_connections);
    }
    
    Ok(())
}

fn main() -> std::io::Result<()> {
    let listener = TcpListener::bind("127.0.0.1:8080")?;
    let stats = Arc::new(Mutex::new(ServerStats::new()));
    
    println!("서버가 8080 포트에서 대기 중...");
    
    // 주기적으로 통계를 출력하는 스레드
    {
        let stats = Arc::clone(&stats);
        thread::spawn(move || {
            loop {
                thread::sleep(std::time::Duration::from_secs(10));
                let stats = stats.lock().unwrap();
                println!("\n=== 주기적 통계 ===");
                stats.print();
                println!("==================\n");
            }
        });
    }
    
    for stream in listener.incoming() {
        let stream = stream?;
        let stats = Arc::clone(&stats);
        
        thread::spawn(move || {
            if let Err(e) = handle_client(stream, stats) {
                eprintln!("클라이언트 처리 중 에러: {}", e);
            }
        });
    }
    
    Ok(())
}
```

## MutexGuard의 의미와 중요성

`lock()`이 반환하는 `MutexGuard`는 단순한 래퍼가 아니다. 중요한 역할을 한다.

**1. RAII 패턴 구현**

`MutexGuard`가 드롭되면 자동으로 락이 해제된다. 이것은 예외나 early return이 있어도 락이 해제됨을 보장한다.

```rust
fn process(mutex: &Mutex<Vec<i32>>) -> Result<(), Error> {
    let mut data = mutex.lock().unwrap();
    
    if some_condition {
        return Err(Error);  // MutexGuard가 자동으로 드롭되고 락 해제
    }
    
    data.push(42);
    Ok(())
} // 정상 종료시도 MutexGuard가 드롭되고 락 해제
```

**2. 스코프를 통한 락 범위 제어**

블록을 사용해서 락을 빨리 해제할 수 있다.

```rust
// 나쁜 예: 락을 너무 오래 잡고 있음
let mut data = mutex.lock().unwrap();
data.push(1);
expensive_computation();  // 락을 잡은 채로 긴 작업!
data.push(2);

// 좋은 예: 필요한 만큼만 락을 잡음
{
    let mut data = mutex.lock().unwrap();
    data.push(1);
} // 락 해제
expensive_computation();  // 다른 스레드가 접근 가능
{
    let mut data = mutex.lock().unwrap();
    data.push(2);
}
```

**3. 타입 시스템을 통한 안전성**

`MutexGuard`를 가지고 있어야만 데이터에 접근할 수 있다. 락 없이 데이터에 접근하는 것은 타입 시스템 수준에서 불가능하다.

```rust
let mutex = Mutex::new(vec![1, 2, 3]);

// 컴파일 에러: Mutex를 직접 역참조할 수 없음
// let first = mutex[0];

// 올바른 방법: 반드시 lock을 통해서만 접근
let guard = mutex.lock().unwrap();
let first = guard[0];
```

## lock()의 Result와 poisoning

`lock()`은 `Result`를 반환한다. `unwrap()`을 계속 사용했지만, 실제로는 에러가 발생할 수 있다.

```rust
pub fn lock(&self) -> LockResult<MutexGuard<T>>
```

`Mutex`는 "poisoned" 상태가 될 수 있다. 락을 잡고 있던 스레드가 패닉하면, 해당 `Mutex`는 poisoned 상태가 된다. 이것은 데이터가 불일치 상태일 수 있음을 의미한다.

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));
    let data_clone = Arc::clone(&data);
    
    let handle = thread::spawn(move || {
        let mut v = data_clone.lock().unwrap();
        v.push(4);
        panic!("문제 발생!");  // 락을 잡은 채로 패닉
    });
    
    let _ = handle.join();
    
    // Mutex가 poisoned 상태가 됨
    match data.lock() {
        Ok(guard) => println!("데이터: {:?}", *guard),
        Err(poisoned) => {
            println!("Mutex가 poisoned 상태입니다!");
            // 그래도 데이터에 접근할 수는 있음
            let guard = poisoned.into_inner();
            println!("데이터: {:?}", *guard);
        }
    }
}
```

대부분의 경우 `unwrap()`으로 충분하다. 하지만 프로덕션 코드에서는 적절히 처리하는 것이 좋다.

```rust
let stats = match stats.lock() {
    Ok(guard) => guard,
    Err(poisoned) => {
        eprintln!("Mutex가 poisoned 상태입니다. 복구 시도...");
        poisoned.into_inner()
    }
};
```

## 데드락의 위험과 방지

`Mutex`를 사용할 때 가장 조심해야 할 것이 데드락이다. 두 스레드가 서로를 기다리면서 무한정 블록되는 상황이다.

```rust
use std::sync::{Arc, Mutex};
use std::thread;
use std::time::Duration;

fn main() {
    let resource1 = Arc::new(Mutex::new(0));
    let resource2 = Arc::new(Mutex::new(0));
    
    let r1 = Arc::clone(&resource1);
    let r2 = Arc::clone(&resource2);
    
    let handle1 = thread::spawn(move || {
        let _lock1 = r1.lock().unwrap();
        println!("Thread 1: resource1 락 획득");
        thread::sleep(Duration::from_millis(100));
        
        println!("Thread 1: resource2를 기다리는 중...");
        let _lock2 = r2.lock().unwrap();  // 데드락!
        println!("Thread 1: resource2 락 획득");
    });
    
    let r1 = Arc::clone(&resource1);
    let r2 = Arc::clone(&resource2);
    
    let handle2 = thread::spawn(move || {
        let _lock2 = r2.lock().unwrap();
        println!("Thread 2: resource2 락 획득");
        thread::sleep(Duration::from_millis(100));
        
        println!("Thread 2: resource1을 기다리는 중...");
        let _lock1 = r1.lock().unwrap();  // 데드락!
        println!("Thread 2: resource1 락 획득");
    });
    
    handle1.join().unwrap();
    handle2.join().unwrap();
}
```

이 코드는 데드락에 빠진다.

```
시간 →
Thread 1: [R1 락] ──── [R2 대기중...] (영원히)
Thread 2: [R2 락] ──── [R1 대기중...] (영원히)
```

**데드락 방지 전략:**

**1. 락 순서 일관성 유지**

항상 같은 순서로 락을 획득한다.

```rust
// 좋은 예: 항상 resource1 먼저, resource2 나중에
let _lock1 = resource1.lock().unwrap();
let _lock2 = resource2.lock().unwrap();
```

**2. 락 범위 최소화**

락을 잡고 있는 시간을 최소화한다.

```rust
// 나쁜 예
let mut data = mutex.lock().unwrap();
expensive_computation();
data.push(42);

// 좋은 예
let value = expensive_computation();
{
    let mut data = mutex.lock().unwrap();
    data.push(value);
}
```

**3. try_lock() 사용**

락을 획득할 수 없으면 블록하지 않고 에러를 반환한다.

```rust
match mutex.try_lock() {
    Ok(guard) => {
        // 락 획득 성공
        println!("데이터: {:?}", *guard);
    }
    Err(TryLockError::WouldBlock) => {
        // 락을 획득할 수 없음
        println!("락이 사용 중입니다. 나중에 다시 시도...");
    }
    Err(TryLockError::Poisoned(e)) => {
        println!("Mutex가 poisoned 상태입니다");
    }
}
```

**4. 중첩 락 피하기**

한 번에 하나의 락만 잡도록 한다.

```rust
// 나쁜 예: 중첩 락
let data1 = mutex1.lock().unwrap();
let data2 = mutex2.lock().unwrap();  // 위험!

// 좋은 예: 필요한 데이터만 가져오기
let value = {
    let data1 = mutex1.lock().unwrap();
    data1.clone()
};
{
    let mut data2 = mutex2.lock().unwrap();
    data2.push(value);
}
```

## 실전 예제: 클라이언트 목록 관리

이제 더 복잡한 예제를 만들어보자. 연결된 클라이언트들의 목록을 유지한다.

```rust
use std::sync::{Arc, Mutex};
use std::net::{TcpListener, TcpStream};
use std::io::{BufReader, BufRead, Write};
use std::thread;
use std::collections::HashMap;

type ClientId = usize;

struct Client {
    id: ClientId,
    address: String,
    request_count: usize,
}

struct ClientManager {
    clients: HashMap<ClientId, Client>,
    next_id: ClientId,
}

impl ClientManager {
    fn new() -> Self {
        ClientManager {
            clients: HashMap::new(),
            next_id: 1,
        }
    }
    
    fn add_client(&mut self, address: String) -> ClientId {
        let id = self.next_id;
        self.next_id += 1;
        
        let client = Client {
            id,
            address,
            request_count: 0,
        };
        
        self.clients.insert(id, client);
        println!("클라이언트 {} 추가됨. 총 {}명", id, self.clients.len());
        id
    }
    
    fn remove_client(&mut self, id: ClientId) {
        if let Some(client) = self.clients.remove(&id) {
            println!("클라이언트 {} 제거됨 (요청 {}회). 남은 클라이언트: {}명",
                     id, client.request_count, self.clients.len());
        }
    }
    
    fn increment_requests(&mut self, id: ClientId) {
        if let Some(client) = self.clients.get_mut(&id) {
            client.request_count += 1;
        }
    }
    
    fn print_all(&self) {
        println!("\n=== 연결된 클라이언트 ===");
        if self.clients.is_empty() {
            println!("(없음)");
        } else {
            for client in self.clients.values() {
                println!("  ID {}: {} - 요청 {}회",
                         client.id, client.address, client.request_count);
            }
        }
        println!("========================\n");
    }
}

fn handle_client(
    mut stream: TcpStream,
    manager: Arc<Mutex<ClientManager>>,
) -> std::io::Result<()> {
    let peer_addr = stream.peer_addr()?.to_string();
    
    // 클라이언트 등록
    let client_id = {
        let mut manager = manager.lock().unwrap();
        manager.add_client(peer_addr.clone())
    }; // 락 해제
    
    writeln!(stream, "환영합니다! 당신의 ID는 {}입니다.", client_id)?;
    writeln!(stream, "'list' - 모든 클라이언트 목록")?;
    writeln!(stream, "'quit' - 연결 종료")?;
    
    let mut reader = BufReader::new(stream.try_clone()?);
    let mut line = String::new();
    
    loop {
        line.clear();
        match reader.read_line(&mut line) {
            Ok(0) => break,
            Ok(_) => {
                // 요청 카운트 증가
                {
                    let mut manager = manager.lock().unwrap();
                    manager.increment_requests(client_id);
                }
                
                let trimmed = line.trim();
                
                match trimmed {
                    "quit" => {
                        writeln!(stream, "안녕히 가세요!")?;
                        break;
                    }
                    "list" => {
                        let manager = manager.lock().unwrap();
                        writeln!(stream, "=== 클라이언트 목록 ===")?;
                        for client in manager.clients.values() {
                            writeln!(stream, "  ID {}: {} - 요청 {}회",
                                     client.id, client.address, client.request_count)?;
                        }
                        writeln!(stream, "=======================")?;
                    }
                    _ => {
                        write!(stream, "Echo: {}", line)?;
                    }
                }
            }
            Err(e) => {
                eprintln!("[Client {}] 에러: {}", client_id, e);
                break;
            }
        }
    }
    
    // 클라이언트 제거
    {
        let mut manager = manager.lock().unwrap();
        manager.remove_client(client_id);
    }
    
    Ok(())
}

fn main() -> std::io::Result<()> {
    let listener = TcpListener::bind("127.0.0.1:8080")?;
    let manager = Arc::new(Mutex::new(ClientManager::new()));
    
    println!("서버가 8080 포트에서 대기 중...");
    
    // 주기적으로 클라이언트 목록 출력
    {
        let manager = Arc::clone(&manager);
        thread::spawn(move || {
            loop {
                thread::sleep(std::time::Duration::from_secs(15));
                let manager = manager.lock().unwrap();
                manager.print_all();
            }
        });
    }
    
    for stream in listener.incoming() {
        let stream = stream?;
        let manager = Arc::clone(&manager);
        
        thread::spawn(move || {
            if let Err(e) = handle_client(stream, manager) {
                eprintln!("클라이언트 처리 중 에러: {}", e);
            }
        });
    }
    
    Ok(())
}
```

이 코드의 핵심 특징들을 살펴보자.

**1. 블록을 사용한 락 범위 제어**

```rust
let client_id = {
    let mut manager = manager.lock().unwrap();
    manager.add_client(peer_addr.clone())
}; // 여기서 락 해제
```

블록을 사용해서 락을 필요한 만큼만 잡는다. 블록이 끝나면 `MutexGuard`가 드롭되고 락이 자동으로 해제된다.

**2. 짧은 크리티컬 섹션**

```rust
// 요청 카운트 증가만 하고 즉시 락 해제
{
    let mut manager = manager.lock().unwrap();
    manager.increment_requests(client_id);
}

// 여기서 다른 작업 수행 (락 없이)
```

락을 잡고 있는 시간을 최소화해서 다른 스레드들이 기다리는 시간을 줄인다.

**3. 읽기 전용 접근도 락 필요**

```rust
let manager = manager.lock().unwrap();
manager.print_all();
```

`Mutex`는 읽기와 쓰기를 구분하지 않는다. 읽기만 해도 락을 획득해야 한다. 만약 여러 스레드가 동시에 읽을 수 있게 하려면 `RwLock`을 사용해야 한다. (나중에 배운다.)

## 빌림 규칙이 Mutex에도 적용되는 방식

`Mutex`를 사용해도 Rust의 빌림 규칙은 여전히 적용된다. `MutexGuard`는 가변 참조처럼 작동한다.

```rust
let mutex = Mutex::new(vec![1, 2, 3]);

let mut guard = mutex.lock().unwrap();

// MutexGuard를 통해 가변 참조를 얻을 수 있음
guard.push(4);

// 하지만 guard가 살아있는 동안 다시 락을 획득할 수 없음
// let guard2 = mutex.lock().unwrap();  // 데드락!
```

`MutexGuard`가 존재하는 동안은 다른 스레드가 락을 획득할 수 없다. 이것은 가변 참조가 존재하는 동안 다른 참조를 만들 수 없는 것과 같은 원리다.

```
단일 스레드에서:
    &mut T        : 한 번에 하나만
    
다중 스레드에서:
    MutexGuard<T> : 한 번에 하나만
```

## 성능 고려사항

`Mutex`는 동기화 오버헤드가 있다. 성능에 민감한 코드에서 고려해야 할 사항들이 있다.

**1. 락 경합(Lock Contention)**

여러 스레드가 동시에 같은 `Mutex`를 획득하려고 하면 성능이 저하된다.

```rust
// 나쁜 예: 모든 스레드가 하나의 카운터를 공유
let counter = Arc::new(Mutex::new(0));

// 좋은 예: 각 스레드가 로컬 카운터를 유지하고 나중에 합침
let counters: Vec<_> = (0..10)
    .map(|_| Arc::new(Mutex::new(0)))
    .collect();
```

**2. 락을 잡고 있는 시간**

락을 오래 잡고 있으면 다른 스레드들이 기다려야 한다.

```rust
// 나쁜 예
let data = mutex.lock().unwrap();
slow_operation(&data);

// 좋은 예
let data_copy = {
    let data = mutex.lock().unwrap();
    data.clone()
};
slow_operation(&data_copy);
```

**3. AtomicUsize vs Mutex<usize>**

단순한 정수 카운터라면 `AtomicUsize`가 더 효율적이다.

```rust
// 더 느림
let counter = Arc::new(Mutex::new(0usize));
*counter.lock().unwrap() += 1;

// 더 빠름
use std::sync::atomic::{AtomicUsize, Ordering};
let counter = Arc::new(AtomicUsize::new(0));
counter.fetch_add(1, Ordering::SeqCst);
```

## 대안: RwLock 맛보기

읽기가 쓰기보다 훨씬 많은 경우, `RwLock`(Reader-Writer Lock)을 사용할 수 있다. 여러 읽기 락은 동시에 가능하지만, 쓰기 락은 배타적이다.

```rust
use std::sync::RwLock;

let data = RwLock::new(vec![1, 2, 3]);

// 여러 스레드가 동시에 읽을 수 있음
let reader1 = data.read().unwrap();
let reader2 = data.read().unwrap();
println!("{:?}", *reader1);
println!("{:?}", *reader2);

drop(reader1);
drop(reader2);

// 쓰기는 배타적
let mut writer = data.write().unwrap();
writer.push(4);
```

```
Mutex:
    읽기: 배타적
    쓰기: 배타적
    
RwLock:
    읽기: 여러 개 동시 가능
    쓰기: 배타적
```

하지만 `RwLock`은 오버헤드가 더 크다. 읽기가 압도적으로 많지 않으면 `Mutex`가 더 빠를 수 있다.

## 정리: Arc와 Mutex의 조합

`Arc<Mutex<T>>`는 Rust에서 공유 상태를 다루는 표준 패턴이다.

```
Arc<Mutex<T>>
│
├─ Arc:   여러 스레드가 소유 (참조 카운팅)
│
└─ Mutex: 한 번에 하나만 접근 (상호 배제)
    │
    └─ T: 실제 데이터
```

각각의 역할이 명확하다.

- `Arc`: "누가 이 데이터를 소유하는가?"
- `Mutex`: "누가 이 데이터를 변경할 수 있는가?"

다른 언어에서는 이런 문제들이 런타임 버그로 나타난다. Rust는 타입 시스템과 빌림 검사기로 컴파일 타임에 잡아낸다.

---

**오늘 배운 핵심 개념:**

- `Mutex`는 상호 배제를 통해 안전한 변경을 가능하게 한다
- `lock()`은 `MutexGuard`를 반환하며, 드롭될 때 자동으로 락이 해제된다
- `Arc<Mutex<T>>`는 여러 스레드가 데이터를 공유하고 변경하는 표준 패턴이다
- 블록을 사용해서 락 범위를 최소화할 수 있다
- 데드락을 피하려면 락 순서를 일관되게 유지하고 락 범위를 최소화해야 한다
- 빌림 규칙은 `MutexGuard`를 통해 스레드 간에도 적용된다

내일은 채널(channel)을 배워서 스레드 간에 메시지를 주고받는 방법을 익힌다. 이것은 공유 상태 대신 메시지 전달을 사용하는 다른 접근법이다!
