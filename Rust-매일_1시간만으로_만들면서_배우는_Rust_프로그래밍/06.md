# Day 6: 소유권의 기초 - 왜 필요한가

## 메모리 관리의 세 가지 방식
프로그래밍 언어는 메모리를 관리하는 방식에 따라 크게 세 가지로 나뉜다. 가비지 컬렉터를 사용하는 언어(Java, Python)는 런타임에 자동으로 메모리를 회수한다. 수동 관리 언어(C, C++)는 프로그래머가 직접 메모리를 할당하고 해제해야 한다. Rust는 제3의 방식을 택했다. 컴파일 시점에 소유권 규칙을 검사하여 메모리 안전성을 보장하되, 런타임 오버헤드는 없다.

이것이 왜 혁신적인가? 메모리 버그는 프로그램 충돌의 가장 흔한 원인이다. 널 포인터 참조, 메모리 누수, 댕글링 포인터, 데이터 레이스 같은 문제들은 발견하기도 어렵고 수정하기는 더 어렵다. Rust는 이런 버그들을 컴파일 시점에 원천적으로 차단한다.

## 스택과 힙: 두 가지 메모리 영역
소유권을 이해하려면 먼저 메모리가 어떻게 구성되는지 알아야 한다. 프로그램이 실행될 때 사용하는 메모리는 크게 스택과 힙으로 나뉜다.

**스택 메모리**는 함수 호출과 지역 변수를 저장하는 곳이다. 접시를 쌓듯이 데이터가 쌓이고(push), 맨 위부터 제거된다(pop). 크기가 컴파일 시점에 정해진 데이터만 스택에 저장할 수 있다. 정수, 부동소수점, 불리언 같은 기본 타입들이 여기에 속한다. 스택은 매우 빠르다. CPU가 항상 스택의 맨 위가 어디인지 알고 있기 때문이다.

**힙 메모리**는 크기가 런타임에 결정되는 데이터를 저장한다. 사용자 입력을 받아서 만든 문자열이나, 크기가 계속 변하는 벡터 같은 것들이다. 힙에 데이터를 저장하려면 운영체제에 공간을 요청해야 한다. 운영체제는 충분한 공간을 찾아서 "사용 중"으로 표시하고, 그 주소(포인터)를 반환한다. 이 과정을 할당(allocation)이라고 한다. 힙은 스택보다 느리다. 공간을 찾아야 하고, 포인터를 따라가서 데이터에 접근해야 하기 때문이다.

```
스택                          힙
┌──────────────┐
│ len: 13      │──────┐       ┌─────────────────┐
│ ptr: 0x1234  │      └──────>│ H e l l o , ...  │
│ capacity: 20 │              └─────────────────┐
└──────────────┘
```

문자열 `"Hello, world!"`를 담은 `String`이 있다고 생각해보자. 스택에는 문자열의 길이(13), 힙 메모리의 주소(0x1234), 그리고 할당된 용량(20)이 저장된다. 실제 문자 데이터는 힙에 있다.

## 소유권의 세 가지 규칙
Rust의 소유권은 세 가지 간단한 규칙으로 정리된다.

1. **모든 값은 소유자(owner)라고 불리는 변수를 가진다.**
2. **한 번에 딱 하나의 소유자만 존재할 수 있다.**
3. **소유자가 스코프 밖으로 벗어나면, 값은 버려진다(dropped).**

이 규칙들이 실제로 무엇을 의미하는지 코드로 살펴보자.

```rust
fn main() {
    let message = String::from("서버 시작됨");
    println!("{}", message);
} // message가 스코프를 벗어남. String이 자동으로 해제됨
```

`message` 변수가 `"서버 시작됨"`이라는 문자열을 담은 `String`의 소유자다. `main` 함수가 끝나면 `message`가 스코프를 벗어나고, Rust는 자동으로 힙에 할당된 문자열 데이터를 해제한다. 프로그래머가 `free()`나 `delete`를 호출할 필요가 없다.
 
  
## 이동(Move) 의미론
소유권의 가장 독특한 특징은 이동 의미론이다. 값을 다른 변수에 할당하면, 소유권이 이동한다.

```rust
fn main() {
    let message1 = String::from("연결 수락됨");
    let message2 = message1; // 소유권이 message2로 이동
    
    // println!("{}", message1); // 컴파일 에러! message1은 더 이상 유효하지 않음
    println!("{}", message2); // 정상 작동
}
```

`message1`에서 `message2`로 할당할 때, 문자열 데이터가 복사되는 것이 아니다. 힙 메모리의 주소(포인터)만 복사되고, 소유권은 `message2`로 이동한다. 이후 `message1`은 사용할 수 없다. 왜 이렇게 설계했을까?

만약 `message1`과 `message2`가 모두 같은 힙 메모리를 가리킨다고 상상해보자. 둘 다 스코프를 벗어날 때 각자 메모리를 해제하려고 시도할 것이다. 같은 메모리를 두 번 해제하는 것은 심각한 버그다(double free). Rust는 이 문제를 소유권 이동으로 해결한다. 오직 하나의 소유자만 메모리를 해제할 책임을 진다.

```
이동 전:
message1 ──> [힙 메모리: "연결 수락됨"]

이동 후:
message1 (무효화됨)
message2 ──> [힙 메모리: "연결 수락됨"]
```
  

## 함수 호출과 소유권
함수에 값을 전달할 때도 소유권이 이동한다.

```rust
fn main() {
    let log = String::from("에러: 연결 실패");
    print_log(log); // log의 소유권이 함수로 이동
    // println!("{}", log); // 컴파일 에러! log는 더 이상 유효하지 않음
}

fn print_log(message: String) {
    println!("[LOG] {}", message);
} // message가 스코프를 벗어나며 String이 해제됨
```

`print_log` 함수를 호출하면서 `log`를 전달했다. 이 순간 소유권이 함수의 `message` 매개변수로 이동한다. 함수가 끝나면 `message`가 스코프를 벗어나고, 문자열이 해제된다. `main`의 `log` 변수는 더 이상 사용할 수 없다.

이것이 불편하게 느껴질 수 있다. 함수를 호출했더니 내 데이터를 다시 쓸 수 없다니! 하지만 이것이 Rust의 철학이다. 소유권을 명확히 하여 메모리 안전성을 보장한다. 다음 Day에서 배울 참조와 빌림이 이 문제의 해법이다.
  

## 반환값과 소유권
함수가 값을 반환하면 소유권도 함께 반환된다.

```rust
fn main() {
    let log = create_log();
    println!("{}", log); // 정상 작동
}

fn create_log() -> String {
    let message = String::from("서버 준비 완료");
    message // 소유권이 호출자에게 이동
}
```

`create_log` 함수 안에서 만든 `message`의 소유권이 반환값을 통해 `main`의 `log`로 이동한다. 이렇게 소유권은 함수 경계를 넘나들며 이동할 수 있다.

## 실전: 로그 메시지 수집기 만들기
이제 배운 내용을 실제 프로그램에 적용해보자. 로그 메시지를 수집하고 관리하는 간단한 시스템을 만들 것이다.

```rust
struct LogCollector {
    messages: Vec<String>,
}

impl LogCollector {
    fn new() -> LogCollector {
        LogCollector {
            messages: Vec::new(),
        }
    }
    
    fn add(&mut self, message: String) {
        self.messages.push(message);
    }
    
    fn print_all(&self) {
        for (index, message) in self.messages.iter().enumerate() {
            println!("[{}] {}", index, message);
        }
    }
    
    fn take_all(self) -> Vec<String> {
        self.messages
    }
}

fn main() {
    let mut collector = LogCollector::new();
    
    let log1 = String::from("서버 시작");
    collector.add(log1); // log1의 소유권이 collector로 이동
    // println!("{}", log1); // 에러! log1은 이미 이동됨
    
    collector.add(String::from("클라이언트 연결: 192.168.1.100"));
    collector.add(String::from("요청 처리 완료"));
    
    collector.print_all();
    
    let all_logs = collector.take_all(); // collector의 소유권이 이동
    // collector.print_all(); // 에러! collector는 이미 이동됨
    
    println!("\n총 {} 개의 로그", all_logs.len());
}
```

이 코드에서 소유권이 어떻게 움직이는지 주의 깊게 살펴보자.

**LogCollector 구조체**는 `Vec<String>`을 소유한다. 각 `String`은 힙에 할당된 문자열 데이터를 소유한다.

**add 메서드**는 `String`을 매개변수로 받는다. 소유권이 전달되므로, 호출 후에는 원래 변수를 사용할 수 없다. 메서드 내부에서 `self.messages.push(message)`를 호출하면, `message`의 소유권이 벡터로 이동한다.

**print_all 메서드**는 `&self`를 받는다. 이것은 참조다(다음 Day에서 자세히 다룬다). 소유권을 가져오지 않고, 읽기만 한다. 따라서 호출 후에도 `collector`를 계속 사용할 수 있다.

**take_all 메서드**는 `self`를 받는다. 소유권을 완전히 가져간다. 이 메서드를 호출하면 `collector` 자체가 소비되어 더 이상 사용할 수 없다. 하지만 내부의 `messages` 벡터는 반환되어 호출자에게 소유권이 이동한다.
  

## 소유권과 메모리 안전성
로그 수집기 예제를 통해 소유권이 실제로 무엇을 방지하는지 이해할 수 있다.

```rust
fn dangerous_code() {
    let collector = LogCollector::new();
    let logs = collector.take_all(); // collector 소비됨
    // collector.print_all(); // 컴파일 에러!
}
```

만약 Rust가 `collector.print_all()`을 허용했다면 어떻게 될까? `take_all`은 내부의 벡터를 꺼내갔고, `collector`는 무효한 메모리를 가리키게 된다. 이것이 바로 use-after-free 버그다. Rust 컴파일러는 이런 코드를 절대 허용하지 않는다.
  

## Copy 타입: 예외 사례
모든 타입이 이동하는 것은 아니다. 정수, 부동소수점, 불리언 같은 단순한 타입들은 `Copy` 트레이트를 구현한다. 이런 타입들은 스택에만 저장되고, 복사가 매우 저렴하다.

```rust
fn main() {
    let x = 5;
    let y = x; // x가 y로 복사됨
    println!("x: {}, y: {}", x, y); // 둘 다 사용 가능
    
    let count = 42;
    print_count(count); // count가 복사됨
    println!("count: {}", count); // 여전히 사용 가능
}

fn print_count(n: i32) {
    println!("count: {}", n);
}
```

`i32` 같은 타입은 4바이트에 불과하다. 포인터를 복사하는 것과 값 자체를 복사하는 것이 똑같은 비용이다. 따라서 Rust는 이런 타입들을 자동으로 복사한다. 반면 `String`은 힙 메모리를 가리키므로, 복사하면 혼란스러운 소유권 문제가 발생한다. 그래서 이동한다.
  

## 명시적 복사: clone()
때로는 실제로 데이터를 복사하고 싶을 때가 있다. 그럴 때는 `clone()` 메서드를 사용한다.

```rust
fn main() {
    let log1 = String::from("중요한 메시지");
    let log2 = log1.clone(); // 힙 데이터까지 깊은 복사
    
    println!("log1: {}", log1); // 정상 작동
    println!("log2: {}", log2); // 정상 작동
}
```

`clone()`은 힙에 새로운 메모리를 할당하고, 문자열 데이터를 전부 복사한다. 이제 `log1`과 `log2`는 각자의 데이터를 소유한다. 하지만 `clone()`은 비용이 든다. 큰 데이터를 복사하면 성능에 영향을 준다. Rust는 복사를 명시적으로 만들어서, 프로그래머가 의식적으로 선택하도록 한다.
  

## 소유권이 제공하는 것
소유권 시스템이 처음에는 제약처럼 느껴질 수 있다. 하지만 이것이 제공하는 보장은 강력하다.

**메모리 누수 방지**: 소유자가 스코프를 벗어나면 자동으로 메모리가 해제된다. 가비지 컬렉터 없이도 메모리 누수가 발생하지 않는다.

**댕글링 포인터 방지**: 한 번에 하나의 소유자만 존재하므로, 이미 해제된 메모리를 참조하는 일이 없다.

**데이터 레이스 방지**: 소유권 규칙은 동시성 프로그래밍에도 적용된다. 여러 스레드가 동시에 데이터를 변경하는 것을 컴파일 시점에 차단한다(이것은 나중에 더 자세히 다룬다).

**제로 비용 추상화**: 이 모든 것이 런타임 비용 없이 컴파일 시점에 검사된다. 실행 파일에는 소유권 검사 코드가 남아있지 않다.
  

## 메모리 레이아웃 시각화
로그 수집기가 메모리에서 어떻게 구성되는지 그림으로 이해해보자.

```
스택                              힙
┌─────────────────┐
│ LogCollector    │
│   messages:     │───┐
│     ptr ────────┼───┼──> ┌─────────────────────┐
│     len: 3      │   │    │ String 0            │
│     capacity: 4 │   │    │   ptr ──> [서버 시작]│
└─────────────────┘   │    ├─────────────────────┤
                      │    │ String 1            │
                      └───>│   ptr ──> [클라...]  │
                           ├─────────────────────┤
                           │ String 2            │
                           │   ptr ──> [요청...]  │
                           └─────────────────────┘
```

`LogCollector`의 `messages` 필드는 `Vec<String>`이다. 벡터 자체는 스택에 포인터, 길이, 용량을 저장한다. 벡터의 데이터는 힙에 있다. 각 `String`도 또다시 힙을 가리킨다. 이런 중첩된 구조에서도 소유권 규칙은 명확하게 작동한다. `LogCollector`가 벡터를 소유하고, 벡터가 각 문자열을 소유한다.
  

## 실습: 메시지 필터 만들기
배운 내용을 확인하기 위한 실습이다. 특정 키워드를 포함한 로그만 수집하는 필터를 만들어보자.

```rust
struct FilteredCollector {
    keyword: String,
    messages: Vec<String>,
}

impl FilteredCollector {
    fn new(keyword: String) -> FilteredCollector {
        FilteredCollector {
            keyword,
            messages: Vec::new(),
        }
    }
    
    fn add(&mut self, message: String) {
        if message.contains(&self.keyword) {
            self.messages.push(message);
        }
        // message의 소유권을 받았지만, 조건에 맞지 않으면 버려진다
        // 스코프를 벗어날 때 자동으로 해제됨
    }
    
    fn count(&self) -> usize {
        self.messages.len()
    }
}

fn main() {
    let mut filter = FilteredCollector::new(String::from("에러"));
    
    filter.add(String::from("정보: 서버 시작"));
    filter.add(String::from("에러: 연결 실패"));
    filter.add(String::from("경고: 느린 응답"));
    filter.add(String::from("에러: 타임아웃"));
    
    println!("에러 로그 개수: {}", filter.count()); // 2
}
```

이 코드에서 주목할 점은 `add` 메서드가 `message`의 소유권을 받는다는 것이다. 조건에 맞으면 벡터에 저장하고, 맞지 않으면 그냥 버린다. 함수가 끝나면서 자동으로 해제된다. 수동으로 메모리를 관리하는 언어였다면 `free()`를 호출해야 했겠지만, Rust는 자동으로 처리한다.
  

## 소유권이 강제하는 설계
소유권 시스템은 단순히 메모리 안전성만 제공하는 것이 아니다. 더 나은 API 설계를 유도한다. 함수 시그니처만 봐도 데이터가 어떻게 사용되는지 명확하다.

```rust
fn consume_log(log: String) { /* log의 소유권을 가져감 */ }
fn read_log(log: &String) { /* log를 읽기만 함 */ }
fn modify_log(log: &mut String) { /* log를 변경함 */ }
```

매개변수 타입만 봐도 함수가 데이터를 소비하는지, 읽는지, 변경하는지 알 수 있다. 이것은 문서보다 강력하다. 타입 시스템이 보장하기 때문이다.
  

## 다음 단계를 위한 준비
오늘 배운 소유권의 기초는 Rust의 가장 독특하고 중요한 특징이다. 처음에는 낯설고 제약적으로 느껴질 수 있다. 하지만 이것이 메모리 안전성과 동시성 안전성을 제공하는 기반이다.

다음 Day에서는 참조와 빌림을 배운다. 소유권을 이전하지 않고도 데이터를 공유하는 방법이다. 불변 참조와 가변 참조의 차이, 그리고 빌림 규칙이 어떻게 데이터 레이스를 방지하는지 알아볼 것이다. 로그 필터링 함수를 구현하면서 실전 감각을 키울 것이다.

소유권은 Rust의 심장이다. 이것을 이해하면 나머지는 자연스럽게 따라온다. 컴파일러가 처음에는 적처럼 느껴질 수 있지만, 곧 가장 유용한 동료가 된다. 런타임에 발생할 수 있는 버그를 컴파일 시점에 잡아주기 때문이다.
