# Day 27: 설정 관리와 환경 변수

## 실전 시나리오: 유연한 서버 설정 시스템

지금까지 우리는 TCP 서버와 비동기 프로그래밍을 배우며 많은 하드코딩된 값을 사용했다. 포트 번호는 8080으로 고정되어 있고, 타임아웃은 코드에 직접 적혀 있으며, 로그 레벨도 변경하려면 코드를 수정해야 했다. 실무에서는 이런 방식이 통하지 않는다. 개발 환경, 스테이징, 프로덕션마다 다른 설정이 필요하고, 배포할 때마다 재컴파일하는 것은 비효율적이다.

오늘은 JSON 설정 파일을 읽고, 환경 변수로 특정 값을 오버라이드하며, 이 모든 것을 타입 안전하게 처리하는 시스템을 만들어본다. 이 과정에서 Rust의 소유권과 빌림이 설정 데이터를 여러 곳에서 안전하게 공유하는 데 어떻게 도움이 되는지 직접 경험하게 될 것이다.

## 프로젝트 준비

먼저 필요한 크레이트들을 `Cargo.toml`에 추가한다.

```toml
[dependencies]
serde = { version = "1.0", features = ["derive"] }
serde_json = "1.0"
tokio = { version = "1", features = ["full"] }
```

`serde`는 Rust 데이터 구조를 직렬화하고 역직렬화하는 표준 라이브러리다. `derive` 피처를 활성화하면 매크로를 통해 자동으로 직렬화 코드를 생성할 수 있다. `serde_json`은 JSON 형식을 다루는 구현체다.

## 설정 구조체 정의하기

설정을 표현할 구조체를 정의하는 것부터 시작한다. 우리 서버에는 네트워크 설정, 타임아웃, 로그 레벨 등이 필요하다.

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ServerConfig {
    pub server: NetworkConfig,
    pub timeout: TimeoutConfig,
    pub logging: LoggingConfig,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct NetworkConfig {
    pub host: String,
    pub port: u16,
    pub max_connections: usize,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TimeoutConfig {
    pub idle_seconds: u64,
    pub shutdown_seconds: u64,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct LoggingConfig {
    pub level: String,
    pub format: String,
}
```

여기서 중요한 점이 몇 가지 있다. `Serialize`와 `Deserialize` trait를 derive하면 이 구조체를 JSON으로 변환하거나 JSON에서 읽어올 수 있다. `Clone`을 derive한 이유는 설정을 여러 스레드나 태스크에서 사용할 때 복사가 필요하기 때문이다. `Debug`는 개발 중 설정을 출력해서 확인하는 데 유용하다.

구조체를 중첩해서 정의한 것은 설정을 논리적으로 그룹화하기 위해서다. `server.host`, `timeout.idle_seconds` 같은 식으로 명확한 계층 구조를 만들 수 있다.

## 기본 설정 구현하기

설정 파일이 없거나 일부 값이 누락된 경우를 대비해 기본값을 제공하는 것이 좋다. Rust에서는 `Default` trait를 구현해서 이를 처리한다.

```rust
impl Default for ServerConfig {
    fn default() -> Self {
        ServerConfig {
            server: NetworkConfig::default(),
            timeout: TimeoutConfig::default(),
            logging: LoggingConfig::default(),
        }
    }
}

impl Default for NetworkConfig {
    fn default() -> Self {
        NetworkConfig {
            host: "127.0.0.1".to_string(),
            port: 8080,
            max_connections: 1000,
        }
    }
}

impl Default for TimeoutConfig {
    fn default() -> Self {
        TimeoutConfig {
            idle_seconds: 300,
            shutdown_seconds: 10,
        }
    }
}

impl Default for LoggingConfig {
    fn default() -> Self {
        LoggingConfig {
            level: "info".to_string(),
            format: "json".to_string(),
        }
    }
}
```

이렇게 하면 `ServerConfig::default()`를 호출해서 모든 기본값이 채워진 설정을 얻을 수 있다. 이 패턴은 Rust 생태계에서 매우 흔하게 사용된다.

## JSON 파일에서 설정 읽기

이제 실제로 파일에서 설정을 읽는 함수를 작성한다. 파일이 없거나 형식이 잘못되었을 때의 에러 처리도 함께 구현한다.

```rust
use std::fs;
use std::path::Path;

impl ServerConfig {
    pub fn from_file<P: AsRef<Path>>(path: P) -> Result<Self, ConfigError> {
        let content = fs::read_to_string(path.as_ref())
            .map_err(|e| ConfigError::FileRead(e.to_string()))?;
        
        let config: ServerConfig = serde_json::from_str(&content)
            .map_err(|e| ConfigError::ParseError(e.to_string()))?;
        
        Ok(config)
    }
}

#[derive(Debug)]
pub enum ConfigError {
    FileRead(String),
    ParseError(String),
    ValidationError(String),
}

impl std::fmt::Display for ConfigError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            ConfigError::FileRead(msg) => write!(f, "설정 파일 읽기 실패: {}", msg),
            ConfigError::ParseError(msg) => write!(f, "설정 파싱 실패: {}", msg),
            ConfigError::ValidationError(msg) => write!(f, "설정 검증 실패: {}", msg),
        }
    }
}

impl std::error::Error for ConfigError {}
```

`from_file` 함수는 제네릭 타입 `P: AsRef<Path>`를 받는다. 이는 `&str`, `String`, `PathBuf` 등 경로로 변환할 수 있는 모든 타입을 받을 수 있다는 뜻이다. 이런 유연성은 실무에서 매우 유용하다.

`fs::read_to_string`은 파일 전체를 `String`으로 읽어온다. 여기서 소유권이 중요하다. 읽어온 문자열은 함수가 소유하며, 다음 줄의 `serde_json::from_str`에 참조(`&content`)로 전달된다. JSON 파싱이 끝나면 `content`는 자동으로 정리되지만, 파싱된 `ServerConfig`는 새로 할당된 데이터이므로 함수에서 반환할 수 있다.

## 환경 변수로 오버라이드하기

프로덕션 환경에서는 설정 파일의 특정 값을 환경 변수로 오버라이드하고 싶을 때가 많다. 예를 들어 포트 번호만 다르게 하거나, 로그 레벨만 변경하고 싶을 수 있다. 쿠버네티스나 도커 환경에서는 환경 변수를 통한 설정이 표준이다.

```rust
use std::env;

impl ServerConfig {
    pub fn from_file_with_env<P: AsRef<Path>>(path: P) -> Result<Self, ConfigError> {
        // 먼저 파일에서 기본 설정을 로드한다
        let mut config = if path.as_ref().exists() {
            Self::from_file(path)?
        } else {
            println!("설정 파일이 없어 기본값을 사용함");
            Self::default()
        };
        
        // 환경 변수로 오버라이드한다
        if let Ok(host) = env::var("SERVER_HOST") {
            println!("환경 변수로 호스트 오버라이드: {}", host);
            config.server.host = host;
        }
        
        if let Ok(port_str) = env::var("SERVER_PORT") {
            let port = port_str.parse::<u16>()
                .map_err(|_| ConfigError::ValidationError(
                    format!("잘못된 포트 번호: {}", port_str)
                ))?;
            println!("환경 변수로 포트 오버라이드: {}", port);
            config.server.port = port;
        }
        
        if let Ok(max_conn_str) = env::var("SERVER_MAX_CONNECTIONS") {
            let max_conn = max_conn_str.parse::<usize>()
                .map_err(|_| ConfigError::ValidationError(
                    format!("잘못된 최대 연결 수: {}", max_conn_str)
                ))?;
            config.server.max_connections = max_conn;
        }
        
        if let Ok(level) = env::var("LOG_LEVEL") {
            println!("환경 변수로 로그 레벨 오버라이드: {}", level);
            config.logging.level = level;
        }
        
        if let Ok(idle_str) = env::var("TIMEOUT_IDLE_SECONDS") {
            let idle = idle_str.parse::<u64>()
                .map_err(|_| ConfigError::ValidationError(
                    format!("잘못된 타임아웃 값: {}", idle_str)
                ))?;
            config.timeout.idle_seconds = idle;
        }
        
        Ok(config)
    }
}
```

이 함수는 먼저 파일에서 설정을 읽고(파일이 없으면 기본값을 사용), 그 다음 환경 변수를 확인해서 존재하는 값들로 오버라이드한다. `env::var`는 환경 변수가 없으면 `Err`를 반환하므로 `if let Ok(...)` 패턴으로 값이 있을 때만 처리한다.

가변 참조(`&mut config`)를 사용해서 설정 값을 변경하고 있다. 이것이 Rust의 소유권 시스템이 빛을 발하는 순간이다. `config`는 이 함수가 소유하고 있으며, 우리는 그것을 변경할 권한이 있다. 함수가 끝나면 수정된 설정이 호출자에게 이동된다.

## 실전 사용 예제

이제 실제로 사용해보자.

```rust
#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    // 설정 로드
    let config = ServerConfig::from_file_with_env("config.json")?;
    
    // 설정 검증
    config.validate()?;
    
    println!("=== 서버 설정 ===");
    println!("주소: {}:{}", config.server.host, config.server.port);
    println!("최대 연결: {}", config.server.max_connections);
    println!("유휴 타임아웃: {}초", config.timeout.idle_seconds);
    println!("로그 레벨: {}", config.logging.level);
    println!("================\n");
    
    // 설정을 Arc로 감싸서 여러 태스크에서 공유
    let config = std::sync::Arc::new(config);
    
    // 서버 시작
    run_server(config).await?;
    
    Ok(())
}
```

환경 변수를 설정해서 실행해본다.

```bash
SERVER_PORT=9999 LOG_LEVEL=trace cargo run
```

출력:

```

환경 변수로 로그 레벨 오버라이드: trace
=== 서버 설정 ===
주소: 0.0.0.0:9999
최대 연결: 5000
유휴 타임아웃: 600초
로그 레벨: trace
================
```

파일의 설정은 그대로 두고 특정 값만 환경 변수로 변경할 수 있다.

## 설정 검증하기

설정을 로드한 후에는 값이 유효한지 확인하는 것이 좋다. 예를 들어 포트 번호가 범위 내에 있는지, 로그 레벨이 올바른 값인지 등을 체크한다.

```rust
impl ServerConfig {
    pub fn validate(&self) -> Result<(), ConfigError> {
        // 포트 범위 확인 (이미 u16이므로 자동으로 0-65535 범위)
        if self.server.port < 1024 {
            println!("경고: 1024 미만의 포트는 root 권한이 필요할 수 있음");
        }
        
        // 최대 연결 수 확인
        if self.server.max_connections == 0 {
            return Err(ConfigError::ValidationError(
                "최대 연결 수는 0보다 커야 함".to_string()
            ));
        }
        
        // 로그 레벨 확인
        let valid_levels = ["trace", "debug", "info", "warn", "error"];
        if !valid_levels.contains(&self.logging.level.as_str()) {
            return Err(ConfigError::ValidationError(
                format!("잘못된 로그 레벨: {}. 가능한 값: {:?}", 
                    self.logging.level, valid_levels)
            ));
        }
        
        // 타임아웃 값 확인
        if self.timeout.idle_seconds == 0 {
            println!("경고: 유휴 타임아웃이 0이면 연결이 즉시 끊길 수 있음");
        }
        
        Ok(())
    }
}
```

이 검증 로직은 실수로 잘못된 설정으로 서버를 시작하는 것을 방지한다. 프로그램이 시작되자마자 설정 문제를 발견하는 것이, 나중에 이상한 동작으로 나타나는 것보다 훨씬 낫다.

## Arc로 설정 공유하기

비동기 서버에서 여러 태스크가 설정에 접근해야 한다면 `Arc`(Atomic Reference Counting)를 사용한다. 설정은 읽기 전용이므로 `Mutex`는 필요 없다.

```rust
use std::sync::Arc;
use tokio::net::TcpListener;

async fn run_server(config: Arc<ServerConfig>) -> Result<(), Box<dyn std::error::Error>> {
    let addr = format!("{}:{}", config.server.host, config.server.port);
    let listener = TcpListener::bind(&addr).await?;
    println!("서버가 {}에서 대기 중", addr);
    
    loop {
        let (socket, peer_addr) = listener.accept().await?;
        println!("새 연결: {}", peer_addr);
        
        // config를 clone해서 태스크로 이동
        // Arc의 clone은 포인터만 복사하므로 저렴하다
        let config_clone = config.clone();
        
        tokio::spawn(async move {
            if let Err(e) = handle_client(socket, config_clone).await {
                eprintln!("클라이언트 처리 에러: {}", e);
            }
        });
    }
}

async fn handle_client(
    mut socket: tokio::net::TcpStream,
    config: Arc<ServerConfig>,
) -> Result<(), Box<dyn std::error::Error>> {
    // config의 값을 읽을 수 있다
    let timeout = std::time::Duration::from_secs(config.timeout.idle_seconds);
    
    println!("타임아웃 {}초로 클라이언트 처리", config.timeout.idle_seconds);
    
    // 실제 처리 로직...
    
    Ok(())
}
```

여기서 소유권의 흐름을 주의 깊게 봐야 한다. `Arc<ServerConfig>`는 `clone()`을 호출할 때마다 참조 카운트만 증가시킨다. 실제 `ServerConfig` 데이터는 한 번만 할당되어 있다. 각 태스크는 `Arc`의 소유권을 가지며, 태스크가 끝나면 자동으로 참조 카운트가 감소한다. 마지막 참조가 사라지면 설정 데이터도 자동으로 해제된다.

```
┌─────────────────────────────────────────────────┐
│                  메모리 구조                      │
├─────────────────────────────────────────────────┤
│                                                 │
│  힙(Heap)                                        │
│  ┌─────────────────────────────────────┐        │
│  │  Arc 내부                            │        │
│  │  ┌──────────────┐                   │        │
│  │  │ ref_count: 4 │  ←─────────┐      │        │
│  │  ├──────────────┤            │      │        │
│  │  │ ServerConfig │            │      │        │
│  │  │   - server   │            │      │        │
│  │  │   - timeout  │            │      │        │
│  │  │   - logging  │            │      │        │
│  │  └──────────────┘            │      │        │
│  └─────────────────────────────────────┘        │
│                                  │               │
│  스택(Stack)                      │               │
│  ┌───────────────────┐           │               │
│  │ main()            │           │               │

│  │                   │                           │

│  │                   │           │               │
```

## 설정 핫 리로드 (선택적 고급 기능)

프로덕션 서버를 재시작하지 않고 설정을 변경하고 싶을 때가 있다. 이를 "핫 리로드"라고 한다. 파일 변경을 감지하고 자동으로 재로드하는 기능을 간단히 구현해본다.

```rust
use tokio::time::{interval, Duration};
use std::sync::Arc;
use tokio::sync::RwLock;

pub struct DynamicConfig {
    config: Arc<RwLock<ServerConfig>>,
    path: String,
}

impl DynamicConfig {
    pub fn new(path: String, config: ServerConfig) -> Self {
        DynamicConfig {
            config: Arc::new(RwLock::new(config)),
            path,
        }
    }
    
    pub fn get(&self) -> Arc<RwLock<ServerConfig>> {
        self.config.clone()
    }
    
    pub async fn start_auto_reload(&self) {
        let config_ref = self.config.clone();
        let path = self.path.clone();
        
        tokio::spawn(async move {
            let mut interval = interval(Duration::from_secs(30));
            
            loop {
                interval.tick().await;
                
                match ServerConfig::from_file_with_env(&path) {
                    Ok(new_config) => {
                        if let Err(e) = new_config.validate() {
                            eprintln!("새 설정 검증 실패: {}", e);
                            continue;
                        }
                        
                        let mut config = config_ref.write().await;
                        *config = new_config;
                        println!("설정이 리로드됨");
                    }
                    Err(e) => {
                        eprintln!("설정 리로드 실패: {}", e);
                    }
                }
            }
        });
    }
}
```

사용 예제:

```rust
#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let config = ServerConfig::from_file_with_env("config.json")?;
    config.validate()?;
    
    let dynamic_config = DynamicConfig::new("config.json".to_string(), config);
    dynamic_config.start_auto_reload().await;
    
    // 설정 읽기
    let config_lock = dynamic_config.get();
    let config = config_lock.read().await;
    println!("현재 포트: {}", config.server.port);
    drop(config); // 명시적으로 락 해제
    
    // 서버 실행...
    
    Ok(())
}
```

여기서는 `RwLock`(읽기-쓰기 락)을 사용한다. 여러 태스크가 동시에 설정을 읽을 수 있지만(`read()`), 쓰기(`write()`)는 한 번에 하나만 가능하다. 이는 읽기가 훨씬 빈번한 경우에 효율적이다.

`read().await`는 읽기 가드를 반환한다. 이 가드가 존재하는 동안 다른 곳에서 쓰기를 할 수 없다. 가드가 드롭되면(스코프를 벗어나면) 락이 자동으로 해제된다. 이것이 Rust의 RAII(Resource Acquisition Is Initialization) 패턴이며, 락을 해제하는 것을 잊어버릴 수 없게 만든다.

## 실전 팁과 주의사항

**설정 파일 위치 결정하기**: 개발 중에는 프로젝트 루트의 `config.json`을 사용하지만, 프로덕션에서는 `/etc/myapp/config.json` 같은 표준 위치를 사용하는 것이 좋다. 환경 변수 `CONFIG_PATH`로 경로를 지정할 수 있게 하면 더욱 유연하다.

**민감한 정보 처리하기**: 데이터베이스 비밀번호 같은 민감한 정보는 설정 파일에 직접 넣지 않는다. 환경 변수로만 받거나, 별도의 시크릿 관리 시스템을 사용한다.

**타입 안전성의 가치**: `String`으로 모든 것을 받는 대신, 가능하면 정확한 타입을 사용한다. 포트는 `u16`, 타임아웃은 `u64`, 최대 연결은 `usize`다. 잘못된 타입의 값이 들어오면 파싱 단계에서 바로 에러가 발생한다.

**Clone vs Arc**: 설정 구조체에 `Clone`을 derive한 이유는 간단한 경우에는 복사해서 사용할 수 있게 하기 위해서다. 하지만 설정이 크거나 여러 곳에서 공유해야 한다면 `Arc`를 사용하는 것이 효율적이다. `Arc`는 포인터만 복사하므로 설정 크기와 무관하게 비용이 일정하다.

**불변 설정의 장점**: 대부분의 경우 설정은 프로그램 시작 시 한 번 로드하고 이후에는 변경하지 않는다. 이렇게 하면 `Arc`만으로 충분하고 `Mutex`나 `RwLock`이 필요 없다. 코드가 단순해지고 성능도 좋아진다.

## 전체 흐름 정리

```
┌──────────────────────────────────────────────────┐
│                설정 로딩 플로우                    │
└──────────────────────────────────────────────────┘

1. config.json 파일 읽기
   ↓
2. JSON → ServerConfig 역직렬화 (serde)
   ↓
3. 환경 변수 확인 및 오버라이드
   ↓
4. 설정 검증 (validate)
   ↓
5. Arc로 감싸기
   ↓
6. 각 태스크에 Arc clone 전달
   ↓
7. 태스크에서 설정 읽기 (불변 참조)
```

## 마무리

오늘은 실무에서 필수적인 설정 관리 시스템을 구축했다. JSON 파일로 기본 설정을 관리하고, 환경 변수로 유연하게 오버라이드하며, 타입 안전하게 검증하는 전체 과정을 다뤘다. 

특히 중요한 것은 Rust의 소유권 시스템이 설정 데이터를 안전하게 공유하는 데 어떻게 도움이 되는지 직접 경험한 것이다. `Arc`를 사용해서 여러 태스크가 동일한 설정을 참조하면서도, 각 태스크는 자신의 소유권을 가지고 있다. 데이터 레이스나 메모리 안전성 문제를 걱정할 필요가 없다.

내일은 이렇게 만든 실전 코드에 테스트를 작성하는 방법을 배운다. 설정 로딩이 제대로 작동하는지, 환경 변수 오버라이드가 의도대로 동작하는지, 검증 로직이 잘못된 값을 잡아내는지 확인하는 테스트를 작성할 것이다.
