# Day 16: Arc로 데이터 공유하기

어제 우리는 스레드를 사용해서 여러 클라이언트를 동시에 처리하는 서버를 만들었다. 하지만 각 스레드는 완전히 독립적이었고, 스레드 간에 어떤 정보도 공유하지 못했다. 오늘은 `Arc`(Atomic Reference Counting)를 사용해서 여러 스레드가 안전하게 데이터를 공유하는 방법을 배운다.

## 문제 상황: 연결된 클라이언트 수를 세고 싶다

실용적인 서버라면 현재 몇 명의 클라이언트가 연결되어 있는지 알아야 한다. 모니터링, 로깅, 또는 리소스 제한을 위해서 필요한 정보다.

단순하게 생각하면 다음과 같이 작성하고 싶을 것이다.

```rust
fn main() -> std::io::Result<()> {
    let listener = TcpListener::bind("127.0.0.1:8080")?;
    let mut client_count = 0;
    
    for stream in listener.incoming() {
        let stream = stream?;
        client_count += 1;
        println!("새 연결! 현재 클라이언트: {}", client_count);
        
        thread::spawn(move || {
            handle_client(stream);
            client_count -= 1;  // 컴파일 에러!
        });
    }
    
    Ok(())
}
```

이 코드는 컴파일되지 않는다. 컴파일러는 다음과 같은 에러를 보여준다.

```
error[E0373]: closure may outlive the current function, but it borrows `client_count`
error[E0499]: cannot borrow `client_count` as mutable more than once at a time
```

왜 이런 문제가 발생할까? 여러 가지 이유가 있다.

**문제 1: 소유권 이동**

`move` 클로저는 캡처한 변수의 소유권을 가져간다. `client_count`를 첫 번째 스레드로 이동시키면 두 번째 스레드는 접근할 수 없다.

**문제 2: 가변 참조의 배타성**

여러 스레드가 동시에 `client_count`를 변경하려면 가변 참조가 필요하다. 하지만 Rust의 빌림 규칙은 한 번에 하나의 가변 참조만 허용한다.

**문제 3: 데이터 레이스**

여러 스레드가 동시에 같은 메모리를 읽고 쓰면 데이터 레이스가 발생한다. 한 스레드가 값을 읽는 동안 다른 스레드가 값을 변경하면 예측할 수 없는 결과가 나온다.

이 모든 문제를 해결하는 것이 바로 `Arc`다.

## 참조 카운팅의 원리

`Arc`는 "Atomic Reference Counting"의 약자다. 여러 소유자가 같은 데이터를 공유할 수 있게 해준다. 작동 원리를 이해해보자.

보통 Rust에서는 하나의 값에 하나의 소유자만 있다. 소유자가 스코프를 벗어나면 값이 드롭된다.

```
Owner ──→ Data
  │
  └─ 스코프 종료
      ↓
   [Data drop]
```

하지만 여러 스레드가 같은 데이터를 사용해야 한다면 어떻게 할까? 모든 스레드가 끝날 때까지 데이터가 살아있어야 한다. `Arc`는 참조 카운터를 사용해서 이 문제를 해결한다.

```
      Arc (count: 3)
         │
    ┌────┼────┐
    │    │    │
Owner1 Owner2 Owner3
    │    │    │
    └────┼────┘
         │
    (count: 0)
         ↓
    [Data drop]
```

`Arc`를 클론할 때마다 참조 카운터가 증가한다. 각 `Arc`가 드롭될 때마다 카운터가 감소한다. 카운터가 0이 되면 그때 데이터가 드롭된다. "Atomic"이라는 수식어가 붙은 이유는 이 카운터 조작이 원자적으로(atomically) 수행되어 스레드 안전하기 때문이다.

## Arc의 기본 사용법

먼저 단순한 예제로 `Arc`의 사용법을 익혀보자.

```rust
use std::sync::Arc;
use std::thread;

fn main() {
    let message = Arc::new(String::from("Hello from Arc!"));
    
    let message_clone = Arc::clone(&message);
    
    let handle = thread::spawn(move || {
        println!("스레드에서: {}", message_clone);
    });
    
    println!("메인에서: {}", message);
    handle.join().unwrap();
}
```

**핵심 포인트들:**

**1. Arc::new()로 생성**

`Arc::new()`는 데이터를 힙에 할당하고 참조 카운터를 1로 초기화한다.

**2. Arc::clone()으로 복제**

`Arc::clone()`은 데이터를 복사하지 않는다. 단지 참조 카운터를 증가시키고 새로운 `Arc` 포인터를 반환할 뿐이다. 비용이 매우 저렴하다.

```rust
let arc1 = Arc::new(vec![1, 2, 3]);  // count: 1
let arc2 = Arc::clone(&arc1);         // count: 2
let arc3 = Arc::clone(&arc1);         // count: 3
drop(arc2);                           // count: 2
drop(arc3);                           // count: 1
drop(arc1);                           // count: 0 → 데이터 드롭
```

**3. move 클로저에 전달**

클론한 `Arc`를 `move` 클로저에 전달하면, 그 `Arc`의 소유권만 이동한다. 원본 `Arc`는 여전히 메인 스레드가 소유하고 있다.

## 실전: 연결된 클라이언트 수 추적하기

이제 서버에 클라이언트 수 추적 기능을 추가해보자. 먼저 읽기 전용으로 시작한다. (다음 Day에서 변경 가능하게 만들 것이다.)

```rust
use std::sync::Arc;
use std::sync::atomic::{AtomicUsize, Ordering};
use std::net::{TcpListener, TcpStream};
use std::io::{BufReader, BufRead, Write};
use std::thread;

fn handle_client(
    mut stream: TcpStream,
    client_count: Arc<AtomicUsize>,
) -> std::io::Result<()> {
    let peer_addr = stream.peer_addr()?;
    
    // 클라이언트 수 증가
    let count = client_count.fetch_add(1, Ordering::SeqCst);
    println!("[{}] 연결됨. 현재 클라이언트: {}", peer_addr, count + 1);
    
    let mut reader = BufReader::new(stream.try_clone()?);
    let mut line = String::new();
    
    loop {
        line.clear();
        match reader.read_line(&mut line) {
            Ok(0) => break,
            Ok(_) => {
                print!("[{}] 받음: {}", peer_addr, line);
                stream.write_all(line.as_bytes())?;
            }
            Err(e) => {
                eprintln!("[{}] 에러: {}", peer_addr, e);
                break;
            }
        }
    }
    
    // 클라이언트 수 감소
    let count = client_count.fetch_sub(1, Ordering::SeqCst);
    println!("[{}] 연결 종료. 현재 클라이언트: {}", peer_addr, count - 1);
    
    Ok(())
}

fn main() -> std::io::Result<()> {
    let listener = TcpListener::bind("127.0.0.1:8080")?;
    let client_count = Arc::new(AtomicUsize::new(0));
    
    println!("서버가 8080 포트에서 대기 중...");
    
    for stream in listener.incoming() {
        let stream = stream?;
        let client_count = Arc::clone(&client_count);
        
        thread::spawn(move || {
            if let Err(e) = handle_client(stream, client_count) {
                eprintln!("클라이언트 처리 중 에러: {}", e);
            }
        });
    }
    
    Ok(())
}
```

여기서 `AtomicUsize`를 사용했는데, 이것은 스레드 안전하게 정수를 읽고 쓸 수 있는 타입이다. `fetch_add`와 `fetch_sub`는 원자적으로 값을 증가하거나 감소시킨다.

## Arc의 내부 구조 이해하기

`Arc`가 메모리에서 어떻게 표현되는지 보면 이해가 더 명확해진다.

```
Stack (Thread 1)         Heap                    Stack (Thread 2)
    │                     │                          │
    │                 ┌───────┐                      │
arc1 ├──────────────→ │Strong:2│ ←───────────────┤ arc2
    │                 │Weak: 0 │                   │
    │                 ├───────┤                    │
    │                 │ Data  │                    │
    │                 │"Hello"│                    │
    │                 └───────┘                    │
```

`Arc`는 실제로 두 개의 포인터를 포함한다.

- 데이터를 가리키는 포인터
- 참조 카운트를 포함하는 메타데이터를 가리키는 포인터

각 `Arc` 인스턴스는 스택에 있는 작은 포인터일 뿐이다. 실제 데이터와 카운터는 힙에 있다.

## 빌림 규칙이 스레드 간에도 적용되는 방식

`Arc`를 사용하더라도 Rust의 빌림 규칙은 여전히 적용된다. 중요한 점은 `Arc<T>`가 불변 참조처럼 작동한다는 것이다.

```rust
use std::sync::Arc;

fn main() {
    let data = Arc::new(vec![1, 2, 3]);
    let data2 = Arc::clone(&data);
    
    // 읽기는 가능
    println!("{:?}", data);
    println!("{:?}", data2);
    
    // 하지만 변경은 불가능
    // data.push(4);  // 컴파일 에러!
}
```

`Arc`는 공유 소유권을 제공하지만, 여러 소유자가 동시에 데이터를 변경하도록 허용하지는 않는다. 이것은 Rust의 기본 빌림 규칙을 따르는 것이다.

- 여러 개의 불변 참조는 허용
- 하나의 가변 참조만 허용
- 불변 참조와 가변 참조는 동시에 존재할 수 없음

`Arc`는 여러 개의 불변 참조를 제공하는 것과 같다. 데이터를 변경하려면 다음 Day에서 배울 `Mutex`가 필요하다.

## Arc vs Rc: 언제 무엇을 사용할까

Rust에는 두 가지 참조 카운팅 타입이 있다.

**Rc<T>: Reference Counting**
- 단일 스레드에서만 사용
- 더 빠름 (원자적 연산 불필요)
- `Send`나 `Sync` trait을 구현하지 않음

**Arc<T>: Atomic Reference Counting**
- 여러 스레드 간에 공유 가능
- 약간 느림 (원자적 연산 필요)
- `Send`와 `Sync` trait을 구현

```
단일 스레드 프로그램:
    Rc<T> ✓ (더 효율적)
    Arc<T> ✓ (작동하지만 불필요)

다중 스레드 프로그램:
    Rc<T> ✗ (컴파일 에러)
    Arc<T> ✓
```

스레드를 사용하지 않는다면 `Rc`를 사용하는 것이 더 효율적이다. 스레드 간에 데이터를 공유해야 한다면 `Arc`를 사용해야 한다.

```rust
use std::rc::Rc;
use std::thread;

fn main() {
    let data = Rc::new(vec![1, 2, 3]);
    
    thread::spawn(move || {  // 컴파일 에러!
        println!("{:?}", data);
    });
}
```

컴파일러는 명확하게 알려준다.

```
error[E0277]: `Rc<Vec<i32>>` cannot be sent between threads safely
```

## 실전 예제: 서버 통계 수집하기

이제 더 복잡한 예제를 만들어보자. 여러 가지 통계를 수집하는 서버다.

```rust
use std::sync::Arc;
use std::sync::atomic::{AtomicUsize, Ordering};
use std::net::{TcpListener, TcpStream};
use std::io::{BufReader, BufRead, Write};
use std::thread;
use std::time::Instant;

// 서버 통계를 담는 구조체
struct ServerStats {
    total_connections: AtomicUsize,
    active_connections: AtomicUsize,
    total_messages: AtomicUsize,
}

impl ServerStats {
    fn new() -> Self {
        ServerStats {
            total_connections: AtomicUsize::new(0),
            active_connections: AtomicUsize::new(0),
            total_messages: AtomicUsize::new(0),
        }
    }
    
    fn connection_started(&self) {
        self.total_connections.fetch_add(1, Ordering::SeqCst);
        self.active_connections.fetch_add(1, Ordering::SeqCst);
    }
    
    fn connection_ended(&self) {
        self.active_connections.fetch_sub(1, Ordering::SeqCst);
    }
    
    fn message_received(&self) {
        self.total_messages.fetch_add(1, Ordering::SeqCst);
    }
    
    fn print_stats(&self) {
        let total = self.total_connections.load(Ordering::SeqCst);
        let active = self.active_connections.load(Ordering::SeqCst);
        let messages = self.total_messages.load(Ordering::SeqCst);
        
        println!("=== 서버 통계 ===");
        println!("총 연결 수: {}", total);
        println!("현재 활성 연결: {}", active);
        println!("총 메시지 수: {}", messages);
        println!("================");
    }
}

fn handle_client(
    mut stream: TcpStream,
    stats: Arc<ServerStats>,
) -> std::io::Result<()> {
    let peer_addr = stream.peer_addr()?;
    stats.connection_started();
    
    println!("[{}] 클라이언트 연결됨", peer_addr);
    writeln!(stream, "에코 서버에 오신 것을 환영합니다!")?;
    writeln!(stream, "'stats'를 입력하면 서버 통계를 볼 수 있습니다.")?;
    writeln!(stream)?;
    
    let mut reader = BufReader::new(stream.try_clone()?);
    let mut line = String::new();
    
    loop {
        line.clear();
        match reader.read_line(&mut line) {
            Ok(0) => break,
            Ok(_) => {
                stats.message_received();
                
                let trimmed = line.trim();
                
                if trimmed == "stats" {
                    let total = stats.total_connections.load(Ordering::SeqCst);
                    let active = stats.active_connections.load(Ordering::SeqCst);
                    let messages = stats.total_messages.load(Ordering::SeqCst);
                    
                    writeln!(stream, "총 연결: {}", total)?;
                    writeln!(stream, "현재 활성: {}", active)?;
                    writeln!(stream, "총 메시지: {}", messages)?;
                } else {
                    write!(stream, "Echo: {}", line)?;
                }
            }
            Err(e) => {
                eprintln!("[{}] 에러: {}", peer_addr, e);
                break;
            }
        }
    }
    
    stats.connection_ended();
    println!("[{}] 클라이언트 연결 종료", peer_addr);
    
    Ok(())
}

fn main() -> std::io::Result<()> {
    let listener = TcpListener::bind("127.0.0.1:8080")?;
    let stats = Arc::new(ServerStats::new());
    
    println!("서버가 8080 포트에서 대기 중...");
    
    // 주기적으로 통계를 출력하는 스레드
    let stats_clone = Arc::clone(&stats);
    thread::spawn(move || {
        loop {
            thread::sleep(std::time::Duration::from_secs(10));
            stats_clone.print_stats();
        }
    });
    
    for stream in listener.incoming() {
        let stream = stream?;
        let stats = Arc::clone(&stats);
        
        thread::spawn(move || {
            if let Err(e) = handle_client(stream, stats) {
                eprintln!("클라이언트 처리 중 에러: {}", e);
            }
        });
    }
    
    Ok(())
}
```

이 코드의 핵심 특징들을 살펴보자.

**1. 구조체로 관련 데이터 그룹화**

여러 개의 `AtomicUsize`를 하나의 구조체로 묶었다. 이렇게 하면 코드가 더 깔끔하고 관리하기 쉽다.

**2. Arc의 중첩 사용**

```
Arc<ServerStats>
    ├─ AtomicUsize (total_connections)
    ├─ AtomicUsize (active_connections)
    └─ AtomicUsize (total_messages)
```

`Arc`는 전체 구조체를 공유하고, 각 `AtomicUsize`는 독립적으로 원자적 연산을 수행한다.

**3. 여러 스레드가 같은 데이터 접근**

워커 스레드들과 통계 출력 스레드가 모두 같은 `ServerStats`를 공유한다. 각 스레드는 자신의 `Arc` 클론을 가지고 있지만, 모두 같은 데이터를 가리킨다.

```
Main Thread:  Arc<ServerStats> ┐
Worker 1:     Arc<ServerStats> ├─→ [Heap: ServerStats]
Worker 2:     Arc<ServerStats> │
Stats Thread: Arc<ServerStats> ┘
```

## AtomicUsize 깊이 이해하기

`AtomicUsize`는 스레드 안전한 정수 타입이다. 일반 `usize`와의 차이를 보자.

```rust
// 일반 usize (스레드 안전하지 않음)
let mut count = 0;
count += 1;  // Read -> Modify -> Write (3단계)

// AtomicUsize (스레드 안전)
let count = AtomicUsize::new(0);
count.fetch_add(1, Ordering::SeqCst);  // 원자적 연산
```

일반 증가 연산은 실제로 세 단계로 이루어진다.

1. 메모리에서 값을 읽는다
2. 값을 1 증가시킨다
3. 결과를 메모리에 쓴다

두 스레드가 동시에 이 작업을 수행하면 문제가 발생한다.

```
시간 →
Thread 1: [읽기: 5] ─── [증가: 6] ─── [쓰기: 6]
Thread 2:     [읽기: 5] ─── [증가: 6] ─── [쓰기: 6]

결과: 6 (예상: 7)
```

`AtomicUsize`는 이 세 단계를 하나의 원자적 연산으로 만든다. CPU 수준의 특수 명령어를 사용해서 중간에 다른 스레드가 끼어들 수 없게 한다.

```
시간 →
Thread 1: [원자적 증가: 5→6] ────────────────
Thread 2:                    [원자적 증가: 6→7]

결과: 7 (정확!)
```

## Ordering 파라미터의 의미

`fetch_add`, `load`, `store` 같은 원자적 연산은 `Ordering` 파라미터를 받는다. 이것은 메모리 순서를 제어한다.

```rust
pub enum Ordering {
    Relaxed,   // 가장 약한 순서 보장
    Acquire,   // 읽기에 사용
    Release,   // 쓰기에 사용
    AcqRel,    // 읽기+쓰기에 사용
    SeqCst,    // 가장 강한 순서 보장 (Sequential Consistency)
}
```

대부분의 경우 `SeqCst`를 사용하면 된다. 가장 직관적이고 안전하다. 더 약한 순서를 사용하면 성능이 약간 향상될 수 있지만, 정확성을 보장하기 어렵다.

```rust
// 일반적인 사용: SeqCst
let count = AtomicUsize::new(0);
count.fetch_add(1, Ordering::SeqCst);
let value = count.load(Ordering::SeqCst);

// 고급 최적화가 필요한 경우에만 다른 순서 사용
count.store(10, Ordering::Relaxed);  // 순서 보장 필요 없음
```

## Arc의 성능 특성

`Arc`를 사용할 때 성능에 대해 알아야 할 사항들이 있다.

**클론 비용**

`Arc::clone()`은 데이터를 복사하지 않지만, 원자적으로 참조 카운터를 증가시켜야 한다. 이것은 일반 포인터 복사보다 약간 비싸다.

```rust
// 빠름: 단순 포인터 복사
let ptr = &data;

// 약간 느림: 원자적 카운터 증가
let arc = Arc::clone(&arc);
```

**Deref 비용**

`Arc`는 `Deref` trait을 구현하므로 자동으로 내부 값을 참조할 수 있다. 이것은 일반 참조와 성능이 같다.

```rust
let data = Arc::new(vec![1, 2, 3]);
println!("{}", data.len());  // Arc를 통한 접근, 추가 비용 없음
```

**Drop 비용**

`Arc`가 드롭될 때 원자적으로 카운터를 감소시켜야 한다. 마지막 `Arc`가 드롭될 때는 내부 데이터도 드롭한다.

```rust
{
    let arc1 = Arc::new(vec![1, 2, 3]);
    let arc2 = Arc::clone(&arc1);
    // arc2 drop: 카운터 감소
} // arc1 drop: 카운터 감소 → 0 → Vec drop
```

## 실전 패턴: Arc를 언제 사용할까

다음은 `Arc`를 사용하는 일반적인 패턴들이다.

**패턴 1: 설정 공유**

서버 설정을 여러 스레드가 읽어야 할 때.

```rust
struct Config {
    max_connections: usize,
    timeout_secs: u64,
    server_name: String,
}

let config = Arc::new(Config {
    max_connections: 1000,
    timeout_secs: 30,
    server_name: "MyServer".to_string(),
});

// 모든 워커가 설정을 읽을 수 있음
for _ in 0..10 {
    let config = Arc::clone(&config);
    thread::spawn(move || {
        println!("서버: {}", config.server_name);
    });
}
```

**패턴 2: 읽기 전용 캐시**

계산 비용이 큰 결과를 여러 스레드가 공유할 때.

```rust
let expensive_data = Arc::new(compute_expensive_data());

for item in work_items {
    let data = Arc::clone(&expensive_data);
    thread::spawn(move || {
        process(item, &data);
    });
}
```

**패턴 3: 스레드 풀에 데이터 전달**

여러 워커 스레드가 같은 데이터에 접근할 때.

```rust
let shared_state = Arc::new(AppState::new());

let workers: Vec<_> = (0..4)
    .map(|id| {
        let state = Arc::clone(&shared_state);
        thread::spawn(move || worker_loop(id, state))
    })
    .collect();
```

## 일반적인 실수와 해결법

**실수 1: Arc를 너무 많이 클론하기**

```rust
// 비효율적
for i in 0..1000 {
    let data = Arc::clone(&data);
    process(i, data);  // 같은 스레드에서 불필요한 클론
}

// 효율적
for i in 0..1000 {
    process(i, &data);  // 참조만 전달
}
```

같은 스레드 내에서는 `Arc`를 클론할 필요가 없다. 참조를 전달하면 된다.

**실수 2: Arc 안에 Arc 중첩하기**

```rust
// 불필요한 중첩
let data = Arc::new(Arc::new(vec![1, 2, 3]));

// 간단하게
let data = Arc::new(vec![1, 2, 3]);
```

`Arc` 안에 `Arc`를 넣을 필요는 거의 없다. 한 번만 감싸면 충분하다.

**실수 3: Arc로 가변성 얻으려고 하기**

```rust
let data = Arc::new(vec![1, 2, 3]);
data.push(4);  // 컴파일 에러!
```

`Arc`는 불변 공유만 제공한다. 변경이 필요하면 `Arc<Mutex<T>>`를 사용해야 한다. (다음 Day에서 배운다.)

## 시각화: Arc의 생명주기

`Arc`가 어떻게 생성되고 소멸되는지 단계별로 보자.

```
1. 생성
   let arc1 = Arc::new(data);
   
   [Heap]
   ┌─────────┐
   │count: 1 │
   │data: .. │
   └─────────┘
      ↑
      │
   [arc1]

2. 클론
   let arc2 = Arc::clone(&arc1);
   
   [Heap]
   ┌─────────┐
   │count: 2 │ ←─┐
   │data: .. │   │
   └─────────┘   │
      ↑          │
      │          │
   [arc1]    [arc2]

3. 하나 드롭
   drop(arc1);
   
   [Heap]
   ┌─────────┐
   │count: 1 │
   │data: .. │
   └─────────┘
      ↑
      │
   [arc2]

4. 마지막 드롭
   drop(arc2);
   
   [Heap]
   (메모리 해제됨)
```

## 다음 단계: Arc만으로는 부족하다

오늘 우리는 `Arc`를 사용해서 여러 스레드 간에 데이터를 공유하는 법을 배웠다. 하지만 `Arc`는 읽기 전용 공유만 제공한다. 실제 서버에서는 공유 데이터를 변경해야 하는 경우가 많다.

예를 들어, 연결된 모든 클라이언트 목록을 유지하고 싶다면 어떻게 할까? 새 클라이언트가 연결되면 목록에 추가하고, 연결이 끊어지면 제거해야 한다.

```rust
let clients = Arc::new(Vec::new());  // 어떻게 변경할까?
```

이 문제를 해결하는 것이 바로 `Mutex`다. 다음 Day에서는 `Arc<Mutex<T>>`를 사용해서 여러 스레드가 안전하게 데이터를 변경하는 방법을 배운다.

---

**오늘 배운 핵심 개념:**

- `Arc`는 여러 스레드가 같은 데이터를 소유할 수 있게 해준다
- 참조 카운팅으로 마지막 소유자가 사라질 때 데이터가 드롭된다
- `Arc::clone()`은 데이터를 복사하지 않고 카운터만 증가시킨다
- `Arc`는 불변 공유만 제공하며, 변경이 필요하면 `Mutex`가 필요하다
- `AtomicUsize` 같은 원자적 타입으로 간단한 카운터를 구현할 수 있다
- 빌림 규칙은 스레드 간에도 적용되어 데이터 레이스를 방지한다

내일은 `Mutex`를 배워서 공유 데이터를 안전하게 변경하는 방법을 익힌다!
