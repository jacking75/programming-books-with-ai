# Day 15: 스레드로 동시 처리 시작
지금까지 우리가 만든 TCP 에코 서버는 한 번에 한 클라이언트만 처리할 수 있었다. 첫 번째 클라이언트가 연결을 끊기 전까지는 두 번째 클라이언트가 아무리 기다려도 응답을 받을 수 없다. 이것은 실용적인 서버라고 할 수 없다. 오늘은 스레드를 사용해서 여러 클라이언트를 동시에 처리할 수 있도록 서버를 개선한다.

## 문제 상황: 순차 처리의 한계

먼저 현재 서버의 문제를 명확히 이해해보자. Day 14에서 만든 코드는 대략 다음과 같은 구조였다.

```rust
use std::net::{TcpListener, TcpStream};
use std::io::{BufReader, BufRead, Write};

fn handle_client(stream: TcpStream) -> std::io::Result<()> {
    let mut reader = BufReader::new(&stream);
    let mut writer = stream;
    let mut line = String::new();
    
    loop {
        line.clear();
        let bytes_read = reader.read_line(&mut line)?;
        if bytes_read == 0 {
            break; // 연결 종료
        }
        writer.write_all(line.as_bytes())?;
    }
    
    Ok(())
}

fn main() -> std::io::Result<()> {
    let listener = TcpListener::bind("127.0.0.1:8080")?;
    println!("서버가 8080 포트에서 대기 중...");
    
    for stream in listener.incoming() {
        let stream = stream?;
        println!("새 클라이언트 연결!");
        handle_client(stream)?;
    }
    
    Ok(())
}
```

이 코드는 `handle_client` 함수가 완전히 끝날 때까지 다음 클라이언트를 처리하지 않는다. 만약 첫 번째 클라이언트가 연결을 유지한 채로 메시지를 천천히 보낸다면, 두 번째 클라이언트는 무한정 기다려야 한다.

```
시간 →
Client 1: [연결] ─────── [메시지] ────── [메시지] ───── [종료]
Client 2:          [대기중...........................] [연결]
Client 3:          [대기중.......................................
```

## 스레드로 해결하기

Rust는 표준 라이브러리에서 운영체제 스레드를 제공한다. 각 클라이언트를 별도의 스레드에서 처리하면 여러 클라이언트를 동시에 처리할 수 있다.

```
시간 →
Thread 1 (Client 1): [연결] ─── [메시지] ─── [메시지] ─── [종료]
Thread 2 (Client 2):     [연결] ───── [메시지] ──── [종료]
Thread 3 (Client 3):          [연결] ────── [메시지] ─────
Main Thread:         [대기] [대기] [대기] [대기] ...
```

## std::thread::spawn의 기본 사용법

스레드를 생성하는 것은 간단하다. `std::thread::spawn` 함수에 클로저를 전달하면 된다.

```rust
use std::thread;
use std::time::Duration;

fn main() {
    thread::spawn(|| {
        for i in 1..5 {
            println!("새 스레드: {}", i);
            thread::sleep(Duration::from_millis(500));
        }
    });
    
    for i in 1..3 {
        println!("메인 스레드: {}", i);
        thread::sleep(Duration::from_millis(300));
    }
}
```

이 코드를 실행하면 예상과 다른 결과를 볼 수 있다. 새 스레드가 채 끝나기도 전에 프로그램이 종료된다. 메인 스레드가 끝나면 프로그램 전체가 종료되고, 다른 스레드들도 강제로 종료되기 때문이다.

스레드가 끝날 때까지 기다리려면 `join`을 호출해야 한다.

```rust
fn main() {
    let handle = thread::spawn(|| {
        for i in 1..5 {
            println!("새 스레드: {}", i);
            thread::sleep(Duration::from_millis(500));
        }
    });
    
    for i in 1..3 {
        println!("메인 스레드: {}", i);
        thread::sleep(Duration::from_millis(300));
    }
    
    handle.join().unwrap();
}
```

`spawn`은 `JoinHandle`을 반환한다. 이것의 `join` 메서드를 호출하면 해당 스레드가 끝날 때까지 현재 스레드가 대기한다.

## move 클로저: 소유권을 스레드로 넘기기

스레드는 독립적으로 실행되므로, 스레드가 사용하는 데이터는 그 스레드가 소유해야 한다. 다음 코드는 컴파일되지 않는다.

```rust
fn main() {
    let message = String::from("안녕하세요");
    
    thread::spawn(|| {
        println!("{}", message); // 컴파일 에러!
    });
}
```

컴파일러는 다음과 같은 에러를 보여준다.

```
error[E0373]: closure may outlive the current function, but it borrows `message`
```

왜 이런 문제가 발생할까? 클로저는 기본적으로 변수를 빌린다. 하지만 새 스레드가 얼마나 오래 실행될지 컴파일러는 알 수 없다. 만약 메인 스레드가 먼저 끝나서 `message`가 드롭되면, 새 스레드는 유효하지 않은 메모리를 참조하게 된다.

해결 방법은 `move` 키워드를 사용하는 것이다. 이것은 클로저에게 캡처한 변수의 소유권을 가져가라고 명시한다.

```rust
fn main() {
    let message = String::from("안녕하세요");
    
    thread::spawn(move || {
        println!("{}", message);
    });
    
    // 여기서는 message를 더 이상 사용할 수 없다
}
```

이제 `message`의 소유권이 새 스레드로 이동했다. 새 스레드가 끝나면 `message`도 자동으로 드롭된다.

## 다중 클라이언트 서버 구현

이제 우리 서버에 스레드를 적용해보자. 각 클라이언트마다 새 스레드를 만들어서 처리한다.

```rust
use std::net::{TcpListener, TcpStream};
use std::io::{BufReader, BufRead, Write};
use std::thread;

fn handle_client(mut stream: TcpStream) -> std::io::Result<()> {
    let peer_addr = stream.peer_addr()?;
    println!("[{}] 클라이언트 연결됨", peer_addr);
    
    let mut reader = BufReader::new(stream.try_clone()?);
    let mut line = String::new();
    
    loop {
        line.clear();
        match reader.read_line(&mut line) {
            Ok(0) => {
                println!("[{}] 클라이언트 연결 종료", peer_addr);
                break;
            }
            Ok(_) => {
                print!("[{}] 받음: {}", peer_addr, line);
                stream.write_all(line.as_bytes())?;
            }
            Err(e) => {
                eprintln!("[{}] 에러: {}", peer_addr, e);
                break;
            }
        }
    }
    
    Ok(())
}

fn main() -> std::io::Result<()> {
    let listener = TcpListener::bind("127.0.0.1:8080")?;
    println!("서버가 8080 포트에서 대기 중...");
    
    for stream in listener.incoming() {
        let stream = stream?;
        
        thread::spawn(move || {
            if let Err(e) = handle_client(stream) {
                eprintln!("클라이언트 처리 중 에러: {}", e);
            }
        });
    }
    
    Ok(())
}
```

핵심 변경사항을 살펴보자.

**1. stream.try_clone()의 사용**

`TcpStream`은 읽기와 쓰기를 모두 지원한다. `BufReader`를 만들 때 스트림의 참조를 빌려주면, 원본 스트림으로 쓰기를 할 수 없게 된다. 이 문제를 해결하기 위해 `try_clone()`으로 같은 소켓을 가리키는 또 다른 핸들을 만든다. 운영체제 수준에서는 같은 소켓이지만, Rust의 소유권 시스템에서는 독립적인 두 개의 값이다.

```
   TcpStream (original)  ──┐
                           ├──→ [OS Socket]
   TcpStream (cloned)   ──┘
```

**2. move 클로저로 소유권 이전**

`thread::spawn(move || { ... })`를 사용해서 `stream`의 소유권을 새 스레드로 이동시킨다. 이제 각 스레드는 자신의 클라이언트 연결을 완전히 소유한다.

**3. peer_addr()로 클라이언트 식별**

여러 클라이언트를 동시에 처리하므로, 로그에서 어떤 클라이언트인지 구분할 수 있도록 `peer_addr()`로 클라이언트의 주소를 가져온다.

## 소유권 흐름 이해하기

스레드를 사용할 때 소유권이 어떻게 움직이는지 명확히 이해하는 것이 중요하다. 다음 다이어그램으로 정리해보자.

```
Main Thread                    Worker Thread 1          Worker Thread 2
    │                                                    
    ├─ listener.incoming()                              
    │                                                    
    ├─ stream1 생성                                      
    │      │                                             
    │      └─────[move]────→ stream1 소유               
    │                            │                       
    ├─ stream2 생성               │                      
    │      │                     ├─ handle_client()     
    │      │                     │                       
    │      └────────────[move]───┼─────→ stream2 소유   
    │                            │           │          
    ├─ stream3 생성               │           │          
    │                            │           ├─ handle_client()
    │                            │           │          
    └─ 계속 대기...               │           │          
                                 ↓           ↓          
                            [종료시 drop]  [종료시 drop]
```

각 `TcpStream`은 정확히 하나의 스레드가 소유한다. 메인 스레드는 연결을 받아서 즉시 워커 스레드로 소유권을 넘기고, 다음 연결을 기다린다. 워커 스레드는 클라이언트 처리가 끝나면 `TcpStream`을 자동으로 드롭하여 연결을 닫는다.

## 스레드 관리의 현실

우리가 만든 서버는 작동하지만, 프로덕션 환경에서는 몇 가지 문제가 있다.

**1. 무제한 스레드 생성**

현재 코드는 클라이언트마다 스레드를 생성한다. 만약 1만 명의 클라이언트가 동시에 접속하면 1만 개의 스레드가 생성된다. 운영체제 스레드는 상당한 메모리를 사용하며(보통 스레드당 수 MB), 컨텍스트 스위칭 비용도 크다.

```
클라이언트 수    스레드 수    대략적인 메모리 사용
    10    →      10     →      ~20 MB
   100    →     100     →     ~200 MB
  1000    →    1000     →      ~2 GB
 10000    →   10000     →     ~20 GB (!!)
```

**2. 조인하지 않는 스레드**

우리는 `spawn`의 반환값인 `JoinHandle`을 무시하고 있다. 이것은 스레드를 "분리된(detached)" 상태로 만든다. 프로그램이 종료될 때 실행 중인 스레드가 있어도 강제로 종료된다. 클라이언트 연결을 우아하게 닫지 못할 수 있다.

**3. 에러 전파의 어려움**

워커 스레드에서 발생한 에러는 메인 스레드에 전달되지 않는다. 각 스레드가 독립적으로 에러를 처리해야 한다.

## Arc와 Mutex가 필요한 이유 맛보기

지금은 각 스레드가 완전히 독립적이다. 하지만 실제 서버에서는 스레드 간에 데이터를 공유해야 하는 경우가 많다.

예를 들어, 현재 연결된 클라이언트 수를 세고 싶다면 어떻게 할까?

```rust
fn main() -> std::io::Result<()> {
    let listener = TcpListener::bind("127.0.0.1:8080")?;
    let mut client_count = 0;  // 이것을 어떻게 공유할까?
    
    for stream in listener.incoming() {
        let stream = stream?;
        client_count += 1;
        println!("현재 클라이언트 수: {}", client_count);
        
        thread::spawn(move || {
            handle_client(stream);
            // 여기서 client_count를 어떻게 감소시킬까?
        });
    }
    
    Ok(())
}
```

단순히 `client_count`를 워커 스레드에서 접근하려고 하면 여러 문제가 발생한다.

**문제 1: 소유권**

`move` 클로저는 값을 이동시킨다. `client_count`를 첫 번째 스레드로 이동시키면, 두 번째 스레드는 접근할 수 없다.

**문제 2: 가변 참조 규칙**

여러 스레드가 동시에 `client_count`를 변경하려면 가변 참조가 필요한데, Rust는 한 번에 하나의 가변 참조만 허용한다.

**문제 3: 데이터 레이스**

여러 스레드가 동시에 같은 메모리를 변경하면 데이터 레이스가 발생한다. 예측할 수 없는 결과가 나온다.

이 문제들을 해결하는 것이 바로 `Arc`(Atomic Reference Counting)와 `Mutex`(Mutual Exclusion)다. `Arc`는 여러 스레드가 같은 데이터를 소유할 수 있게 하고, `Mutex`는 한 번에 하나의 스레드만 데이터를 변경할 수 있게 한다.

```rust
use std::sync::{Arc, Mutex};

let client_count = Arc::new(Mutex::new(0));
```

```
         Arc (참조 카운팅)
              │
    ┌─────────┼─────────┐
    │         │         │
Thread 1   Thread 2  Thread 3
    │         │         │
    └────── Mutex ──────┘
           (상호 배제)
              │
         client_count
```

다음 Day에서 이것을 자세히 배울 것이다. 지금은 여러 스레드가 데이터를 안전하게 공유하려면 특별한 타입이 필요하다는 것만 이해하면 된다.

## 빌림 검사기가 우리를 보호하는 방식

Rust의 빌림 검사기는 컴파일 타임에 데이터 레이스를 방지한다. 다른 언어에서는 런타임에 발견되는 버그들을 Rust는 컴파일 단계에서 잡아낸다.

다음 코드를 시도해보자.

```rust
use std::thread;

fn main() {
    let mut data = vec![1, 2, 3];
    
    thread::spawn(|| {
        data.push(4);  // 컴파일 에러!
    });
    
    data.push(5);  // 컴파일 에러!
}
```

컴파일러는 다음과 같이 지적한다.

```
error[E0373]: closure may outlive the current function, but it borrows `data`
```

여기서 두 가지 문제가 있다. 첫째, 클로저가 `data`를 빌리는데 스레드가 얼마나 오래 살지 모른다. 둘째, `move`를 추가하더라도 두 스레드가 모두 `data`를 변경하려고 한다.

Rust는 이런 상황을 허용하지 않는다. `Arc<Mutex<Vec<i32>>>`와 같은 명시적인 동기화 메커니즘을 사용해야 한다. 이것은 번거로워 보이지만, 런타임에 디버깅하기 어려운 동시성 버그를 원천적으로 차단한다.

## 실전 팁: 스레드 디버깅

스레드를 사용할 때 유용한 디버깅 팁을 몇 가지 소개한다.

**1. 스레드 이름 지정**

`thread::Builder`를 사용하면 스레드에 이름을 붙일 수 있다. 패닉 메시지나 디버거에서 어떤 스레드인지 쉽게 식별할 수 있다.

```rust
use std::thread;

thread::Builder::new()
    .name(format!("client-{}", peer_addr))
    .spawn(move || {
        handle_client(stream)
    })
    .unwrap();
```

**2. 스레드 로컬 변수**

각 스레드마다 별도의 값을 가지는 변수를 만들 수 있다.

```rust
use std::cell::RefCell;

thread_local! {
    static REQUEST_COUNT: RefCell<u32> = RefCell::new(0);
}

fn handle_client(stream: TcpStream) {
    REQUEST_COUNT.with(|count| {
        *count.borrow_mut() += 1;
        println!("이 스레드에서 처리한 요청: {}", count.borrow());
    });
}
```

**3. 패닉 핸들링**

스레드가 패닉하면 해당 스레드만 종료되고 메인 스레드는 계속 실행된다. `JoinHandle`의 `join()`은 `Result`를 반환하므로 패닉을 감지할 수 있다.

```rust
let handle = thread::spawn(|| {
    panic!("문제 발생!");
});

match handle.join() {
    Ok(_) => println!("스레드가 정상 종료됨"),
    Err(_) => println!("스레드가 패닉함"),
}
```

## 전체 코드와 실습

오늘 완성한 다중 클라이언트 서버의 전체 코드를 정리한다.

```rust
use std::net::{TcpListener, TcpStream};
use std::io::{BufReader, BufRead, Write};
use std::thread;

fn handle_client(mut stream: TcpStream) -> std::io::Result<()> {
    let peer_addr = stream.peer_addr()?;
    println!("[{}] 클라이언트 연결됨", peer_addr);
    
    // 읽기용과 쓰기용으로 스트림 복제
    let mut reader = BufReader::new(stream.try_clone()?);
    let mut line = String::new();
    
    loop {
        line.clear();
        match reader.read_line(&mut line) {
            Ok(0) => {
                println!("[{}] 클라이언트 연결 종료", peer_addr);
                break;
            }
            Ok(n) => {
                print!("[{}] 받음 ({} bytes): {}", peer_addr, n, line);
                
                // 에코: 받은 내용을 그대로 돌려보냄
                if let Err(e) = stream.write_all(line.as_bytes()) {
                    eprintln!("[{}] 쓰기 에러: {}", peer_addr, e);
                    break;
                }
            }
            Err(e) => {
                eprintln!("[{}] 읽기 에러: {}", peer_addr, e);
                break;
            }
        }
    }
    
    Ok(())
}

fn main() -> std::io::Result<()> {
    let listener = TcpListener::bind("127.0.0.1:8080")?;
    println!("서버가 8080 포트에서 대기 중...");
    println!("여러 터미널에서 'telnet localhost 8080'으로 접속해보세요");
    
    for stream in listener.incoming() {
        match stream {
            Ok(stream) => {
                // 각 클라이언트를 별도 스레드에서 처리
                thread::spawn(move || {
                    if let Err(e) = handle_client(stream) {
                        eprintln!("클라이언트 처리 중 에러: {}", e);
                    }
                });
            }
            Err(e) => {
                eprintln!("연결 수락 중 에러: {}", e);
            }
        }
    }
    
    Ok(())
}
```

**실습 과제:**

1. 서버를 실행하고 여러 터미널에서 동시에 접속해보자. 각 클라이언트가 독립적으로 작동하는지 확인한다.

2. 한 클라이언트에서 긴 메시지를 천천히 입력하는 동안, 다른 터미널에서 새로 접속해보자. 즉시 응답을 받을 수 있어야 한다.

3. `thread::sleep(Duration::from_secs(5))`를 `handle_client` 함수 시작 부분에 추가해보자. 각 클라이언트가 5초 지연되지만, 다른 클라이언트는 영향을 받지 않는다.

4. 현재 코드를 수정해서 각 클라이언트에게 환영 메시지를 보내보자.

```rust

    let peer_addr = stream.peer_addr()?;
    
    // 환영 메시지
    writeln!(stream, "에코 서버에 오신 것을 환영합니다! [{}]", peer_addr)?;
    
    // 나머지 코드...
}
```

## 다음 단계

오늘은 스레드의 기초를 배우고 다중 클라이언트 서버를 만들었다. 하지만 아직 스레드 간에 데이터를 공유하는 방법을 배우지 않았다. 

Day 16에서는 `Arc`를 사용해서 여러 스레드가 안전하게 데이터를 공유하는 방법을 배운다. 연결된 클라이언트 수를 세거나, 모든 클라이언트에게 브로드캐스트하는 기능을 추가할 것이다.

Day 17에서는 `Mutex`를 사용해서 공유 데이터를 안전하게 변경하는 방법을 배운다. 빌림 규칙이 스레드 환경에서 어떻게 작동하는지 깊이 이해하게 될 것이다.

스레드를 사용하면 동시성 문제가 발생할 수 있지만, Rust의 소유권 시스템은 컴파일 타임에 대부분의 문제를 잡아낸다. 이것이 Rust로 안전한 동시성 코드를 작성할 수 있는 이유다.

---

**오늘 배운 핵심 개념:**

- `std::thread::spawn`으로 새 스레드를 생성한다
- `move` 클로저는 캡처한 변수의 소유권을 스레드로 이동시킨다
- 각 스레드는 독립적으로 실행되며 자신의 스택과 데이터를 가진다
- 스레드 간에 데이터를 공유하려면 `Arc`와 `Mutex` 같은 특별한 타입이 필요하다
- Rust의 소유권 시스템은 데이터 레이스를 컴파일 타임에 방지한다

내일은 연결된 클라이언트 수를 추적하면서 `Arc`의 작동 원리를 실전으로 배운다!
