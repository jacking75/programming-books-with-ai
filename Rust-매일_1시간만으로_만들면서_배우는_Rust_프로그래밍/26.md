# Day 26: 구조화된 로깅

지금까지 만든 채팅 서버는 `println!`과 `eprintln!`으로 상태를 출력했다. 개발 중에는 괜찮지만, 프로덕션 환경에서는 체계적인 로깅이 필수다. 오늘은 Rust 생태계의 표준 로깅 프레임워크를 배우고, 우리 서버에 프로덕션급 로깅을 추가한다.

## **println! 의 한계**

개발하면서 사용한 `println!` 매크로는 간단하지만 실전에서는 여러 문제가 있다.

```rust
println!("클라이언트 연결: {}", addr);
println!("메시지 수신: {}", msg);
eprintln!("에러 발생: {}", e);
```

이런 코드의 문제점들을 살펴보자. 첫째, 중요도를 구분할 수 없다. 정보성 메시지와 심각한 에러가 같은 방식으로 출력된다. 둘째, 로그 레벨을 조절할 수 없다. 디버깅 정보를 프로덕션에서 끄려면 코드를 수정하고 다시 컴파일해야 한다. 셋째, 구조화되지 않았다. 로그를 파싱하고 분석하기 어렵다. 넷째, 타임스탬프나 스레드 정보가 없다. 비동기 환경에서 어디서 발생한 로그인지 추적하기 힘들다.

```
┌─────────────────────────────────────────┐
│         println! 방식                   │
├─────────────────────────────────────────┤
│  클라이언트 연결: 127.0.0.1:5432        │
│  메시지 수신: hello                     │
│  에러 발생: connection reset            │
│                                         │
│  문제: 시간? 중요도? 컨텍스트?         │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│        구조화된 로깅                    │
├─────────────────────────────────────────┤
│ [2024-01-15 14:23:11 INFO]              │
│   server::connection                    │
│   client_connected addr=127.0.0.1:5432  │
│                                         │
│ [2024-01-15 14:23:12 DEBUG]             │
│   server::handler                       │
│   message_received content="hello"      │
│                                         │
│ [2024-01-15 14:23:13 ERROR]             │
│   server::connection                    │
│   connection_error reason="reset"       │
└─────────────────────────────────────────┘
```

## **log 크레이트 소개**

Rust 생태계는 로깅을 위해 표준화된 인터페이스를 제공한다. `log` 크레이트는 로깅 파사드(facade)로, 실제 로그 출력 방식과 독립적으로 로그 메시지를 작성할 수 있게 해준다.

먼저 `Cargo.toml`에 의존성을 추가하자:

```toml
[dependencies]
tokio = { version = "1", features = ["full"] }
log = "0.4"
env_logger = "0.11"
```

`log`는 인터페이스만 제공하고, `env_logger`는 실제 로그를 출력하는 구현체다. 다른 구현체(예: `simple_logger`, `fern`)로 쉽게 교체할 수 있다.

기본적인 사용법을 살펴보자:

```rust
use log::{debug, error, info, trace, warn};

fn main() {
    // 로거 초기화
    env_logger::init();

    trace!("매우 상세한 디버깅 정보");
    debug!("디버깅 정보");
    info!("일반 정보");
    warn!("경고");
    error!("에러");
}
```

로그 레벨은 다섯 단계로 구분된다:

- **TRACE**: 가장 상세한 정보, 모든 실행 흐름 추적
- **DEBUG**: 디버깅에 유용한 정보
- **INFO**: 일반적인 정보성 메시지
- **WARN**: 경고, 문제가 될 수 있는 상황
- **ERROR**: 에러, 즉시 처리가 필요한 문제

## **로그 레벨 제어**

`env_logger`는 환경 변수로 로그 레벨을 제어한다. 코드 수정 없이 실행 시점에 조절할 수 있다.

```bash
# 모든 로그 출력
RUST_LOG=trace cargo run

# INFO 이상만 출력
RUST_LOG=info cargo run

# 특정 모듈만 디버깅
RUST_LOG=my_server::connection=debug cargo run

# 복합 설정
RUST_LOG=info,my_server::handler=debug cargo run
```

코드 내에서도 기본 레벨을 설정할 수 있다:

```rust
use env_logger::Builder;
use log::LevelFilter;

fn main() {
    Builder::new()
        .filter_level(LevelFilter::Info)  // 기본은 INFO
        .filter_module("my_server::connection", LevelFilter::Debug)  // 특정 모듈은 DEBUG
        .init();

    info!("서버 시작");
}
```

이 유연성 덕분에 개발 환경에서는 상세한 로그를, 프로덕션에서는 필요한 것만 볼 수 있다.

## **채팅 서버에 로깅 추가하기**

이제 Day 25에서 만든 채팅 서버에 체계적인 로깅을 추가해보자. `println!`을 모두 적절한 로그 매크로로 교체한다.

```rust
use log::{debug, error, info, warn};
use std::net::SocketAddr;
use std::sync::Arc;
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use tokio::net::{TcpListener, TcpStream};
use tokio::sync::broadcast;

#[derive(Debug, Clone)]
struct ChatMessage {
    sender: SocketAddr,
    content: String,
}

struct ChatServer {
    broadcast_tx: broadcast::Sender<ChatMessage>,
}

impl ChatServer {
    fn new() -> Self {
        let (broadcast_tx, _) = broadcast::channel(100);
        info!("채팅 서버 생성됨 (채널 용량: 100)");
        Self { broadcast_tx }
    }

    async fn handle_client(
        self: Arc<Self>,
        socket: TcpStream,
        addr: SocketAddr,
    ) {
        info!(target: "connection", "클라이언트 연결됨: {}", addr);

        let (reader, mut writer) = socket.into_split();
        let mut reader = BufReader::new(reader);
        let mut line = String::new();
        let mut broadcast_rx = self.broadcast_tx.subscribe();

        // 환영 메시지
        let welcome = format!("채팅 서버에 오신 것을 환영한다!\n");
        if let Err(e) = writer.write_all(welcome.as_bytes()).await {
            error!(target: "connection", "환영 메시지 전송 실패 {}: {}", addr, e);
            return;
        }

        let mut message_count = 0u64;

        loop {
            line.clear();

            tokio::select! {
                result = reader.read_line(&mut line) => {
                    match result {
                        Ok(0) => {
                            info!(target: "connection", 
                                  "클라이언트 정상 종료: {} (메시지: {}개)", 
                                  addr, message_count);
                            break;
                        }
                        Ok(n) => {
                            debug!(target: "handler", 
                                   "메시지 수신: {} ({} 바이트)", addr, n);

                            let trimmed = line.trim();
                            if trimmed.is_empty() {
                                continue;
                            }

                            message_count += 1;

                            let msg = ChatMessage {
                                sender: addr,
                                content: line.clone(),
                            };

                            // 브로드캐스트
                            match self.broadcast_tx.send(msg) {
                                Ok(receiver_count) => {
                                    debug!(target: "broadcast", 
                                           "브로드캐스트 성공: {} → {}명", 
                                           addr, receiver_count);
                                }
                                Err(e) => {
                                    warn!(target: "broadcast", 
                                          "브로드캐스트 실패: {}: {}", addr, e);
                                }
                            }
                        }
                        Err(e) => {
                            error!(target: "connection", 
                                   "읽기 에러: {}: {}", addr, e);
                            break;
                        }
                    }
                }

                result = broadcast_rx.recv() => {
                    match result {
                        Ok(msg) => {
                            if msg.sender != addr {
                                let formatted = format!("[{}] {}", msg.sender, msg.content);
                                
                                if let Err(e) = writer.write_all(formatted.as_bytes()).await {
                                    error!(target: "connection", 
                                           "쓰기 에러: {}: {}", addr, e);
                                    break;
                                }
                            }
                        }
                        Err(e) => {
                            warn!(target: "broadcast", 
                                  "브로드캐스트 수신 에러: {}: {}", addr, e);
                        }
                    }
                }
            }
        }

        info!(target: "connection", 
              "핸들러 종료: {} (총 메시지: {}개)", addr, message_count);
    }
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    // 로거 초기화
    env_logger::Builder::from_default_env()
        .filter_level(log::LevelFilter::Info)
        .init();

    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    info!(target: "server", "채팅 서버 시작: 127.0.0.1:8080");

    let server = Arc::new(ChatServer::new());
    let mut connection_count = 0u64;

    loop {
        match listener.accept().await {
            Ok((socket, addr)) => {
                connection_count += 1;
                info!(target: "server", 
                      "연결 수락: {} (총 {}개)", addr, connection_count);

                let server = Arc::clone(&server);
                tokio::spawn(async move {
                    server.handle_client(socket, addr).await;
                });
            }
            Err(e) => {
                error!(target: "server", "연결 수락 실패: {}", e);
            }
        }
    }
}
```

이 코드의 핵심 개선사항들을 살펴보자.

첫째, `target` 파라미터로 로그의 출처를 명시했다. 이것은 로그를 필터링하고 분석하는 데 매우 유용하다. `RUST_LOG=connection=debug`로 연결 관련 로그만 볼 수 있다.

둘째, 각 로그에 의미 있는 컨텍스트를 추가했다. 클라이언트 주소, 메시지 개수, 바이트 수 등이다. 이런 정보는 문제 추적에 필수적이다.

셋째, 적절한 로그 레벨을 사용했다. 정상 흐름은 `info`, 디버깅 정보는 `debug`, 문제는 `error`나 `warn`으로 구분한다.

넷째, 에러 상황을 명확하게 로깅했다. 단순히 에러를 출력하는 것이 아니라 어떤 작업이 실패했는지도 함께 기록한다.

## **구조화된 데이터 로깅**

때로는 단순한 텍스트보다 구조화된 데이터를 로깅하는 것이 유용하다. 키-값 쌍으로 로그를 작성하면 자동화된 분석이 가능하다.

```rust
use log::info;

fn log_connection_stats(addr: SocketAddr, messages: u64, bytes: u64, duration_secs: u64) {
    info!(
        target: "stats",
        "client_stats addr={} messages={} bytes={} duration_secs={}",
        addr, messages, bytes, duration_secs
    );
}
```

이런 로그는 정규식으로 파싱하기 쉽고, 로그 수집 시스템(예: ELK 스택)에서 효율적으로 처리할 수 있다.

더 나아가, JSON 형식으로 로그를 출력할 수도 있다:

```rust
use serde_json::json;

fn log_as_json(addr: SocketAddr, event: &str, data: serde_json::Value) {
    info!(
        target: "events",
        "{}",
        json!({
            "timestamp": chrono::Utc::now().to_rfc3339(),
            "addr": addr.to_string(),
            "event": event,
            "data": data,
        })
    );
}

// 사용
log_as_json(
    addr,
    "message_received",
    json!({
        "content": msg.content,
        "length": msg.content.len(),
    }),
);
```

## **tracing 크레이트 소개**

`log`는 간단하고 널리 사용되지만, 비동기 애플리케이션에서는 `tracing` 크레이트가 더 강력하다. `tracing`은 구조화된 로깅과 분산 추적(distributed tracing)을 지원한다.

`Cargo.toml`에 추가:

```toml
[dependencies]
tokio = { version = "1", features = ["full"] }
tracing = "0.1"
tracing-subscriber = { version = "0.3", features = ["env-filter"] }
```

기본 사용법:

```rust
use tracing::{debug, error, info, warn, instrument};

#[tokio::main]
async fn main() {
    // 구독자 초기화
    tracing_subscriber::fmt()
        .with_env_filter("info")
        .init();

    info!("서버 시작");
    process_request(42).await;
}

#[instrument]
async fn process_request(request_id: u32) {
    info!("요청 처리 시작");
    
    let result = perform_task(request_id).await;
    
    match result {
        Ok(data) => info!(data = ?data, "처리 완료"),
        Err(e) => error!(error = %e, "처리 실패"),
    }
}

#[instrument]
async fn perform_task(id: u32) -> Result<String, std::io::Error> {
    debug!("작업 수행 중");
    tokio::time::sleep(tokio::time::Duration::from_millis(100)).await;
    Ok(format!("결과-{}", id))
}
```

`#[instrument]` 매크로는 함수 진입/종료를 자동으로 로깅하고, 파라미터 값도 포함시킨다. 출력은 이렇게 보인다:

```
2024-01-15T14:23:11.123Z  INFO server: 서버 시작
2024-01-15T14:23:11.124Z  INFO process_request{request_id=42}: 요청 처리 시작
2024-01-15T14:23:11.125Z DEBUG perform_task{id=42}: 작업 수행 중
2024-01-15T14:23:11.225Z  INFO process_request{request_id=42}: 처리 완료 data="결과-42"
```

각 로그가 어느 함수 호출 컨텍스트에서 발생했는지 명확하다. 이것은 복잡한 비동기 코드를 디버깅할 때 엄청난 도움이 된다.

## **스팬(Span)과 컨텍스트**

`tracing`의 핵심 개념은 스팬(span)이다. 스팬은 작업의 시작부터 끝까지를 나타내며, 중첩될 수 있다.

```rust
use tracing::{info, info_span};

async fn handle_connection(addr: SocketAddr) {
    let span = info_span!("connection", client = %addr);
    let _enter = span.enter();

    info!("연결 수락");
    
    // 모든 로그가 자동으로 client 정보를 포함한다
    process_messages().await;
    
    info!("연결 종료");
}

async fn process_messages() {
    let span = info_span!("messages");
    let _enter = span.enter();
    
    info!("메시지 처리 시작");
    // ...
}
```

스팬 내부의 모든 로그는 자동으로 컨텍스트(여기서는 `client` 필드)를 포함한다. 이것은 복잡한 시스템에서 로그를 추적하는 데 매우 유용하다.

```
┌──────────────────────────────────────┐
│  connection{client=127.0.0.1:5432}   │
│  ├─ 연결 수락                        │
│  ├─ messages                         │
│  │  └─ 메시지 처리 시작              │
│  └─ 연결 종료                        │
└──────────────────────────────────────┘
```

## **실전: tracing으로 채팅 서버 개선**

이제 `tracing`을 사용해서 채팅 서버를 더욱 개선해보자. 각 클라이언트 연결을 스팬으로 감싸서 로그를 추적하기 쉽게 만든다.

```rust
use std::net::SocketAddr;
use std::sync::Arc;
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use tokio::net::{TcpListener, TcpStream};
use tokio::sync::broadcast;
use tracing::{debug, error, info, info_span, instrument, warn};

#[derive(Debug, Clone)]
struct ChatMessage {
    sender: SocketAddr,
    content: String,
}

struct ChatServer {
    broadcast_tx: broadcast::Sender<ChatMessage>,
}

impl ChatServer {
    fn new() -> Self {
        let (broadcast_tx, _) = broadcast::channel(100);
        info!(capacity = 100, "채팅 서버 생성");
        Self { broadcast_tx }
    }

    #[instrument(skip(self, socket), fields(client = %addr))]
    async fn handle_client(
        self: Arc<Self>,
        socket: TcpStream,
        addr: SocketAddr,
    ) {
        info!("클라이언트 연결 수락");

        let (reader, mut writer) = socket.into_split();
        let mut reader = BufReader::new(reader);
        let mut line = String::new();
        let mut broadcast_rx = self.broadcast_tx.subscribe();

        // 환영 메시지
        if let Err(e) = writer.write_all(b"환영한다!\n").await {
            error!(error = %e, "환영 메시지 전송 실패");
            return;
        }

        let mut stats = ConnectionStats::new();

        loop {
            line.clear();

            tokio::select! {
                result = reader.read_line(&mut line) => {
                    match result {
                        Ok(0) => {
                            info!(
                                messages = stats.messages_received,
                                bytes = stats.bytes_received,
                                "클라이언트 정상 종료"
                            );
                            break;
                        }
                        Ok(n) => {
                            stats.record_message(n);
                            
                            debug!(
                                bytes = n,
                                total_messages = stats.messages_received,
                                "메시지 수신"
                            );

                            let trimmed = line.trim();
                            if trimmed.is_empty() {
                                continue;
                            }

                            let msg = ChatMessage {
                                sender: addr,
                                content: line.clone(),
                            };

                            match self.broadcast_tx.send(msg) {
                                Ok(count) => {
                                    debug!(receivers = count, "브로드캐스트 성공");
                                }
                                Err(e) => {
                                    warn!(error = %e, "브로드캐스트 실패");
                                }
                            }
                        }
                        Err(e) => {
                            error!(error = %e, "읽기 에러");
                            break;
                        }
                    }
                }

                result = broadcast_rx.recv() => {
                    match result {
                        Ok(msg) if msg.sender != addr => {
                            let formatted = format!("[{}] {}", msg.sender, msg.content);
                            
                            if let Err(e) = writer.write_all(formatted.as_bytes()).await {
                                error!(error = %e, "쓰기 에러");
                                break;
                            }
                            
                            stats.record_broadcast();
                        }
                        Err(e) => {
                            warn!(error = %e, "브로드캐스트 수신 에러");
                        }
                        _ => {}
                    }
                }
            }
        }

        info!(
            duration_secs = stats.duration().as_secs(),
            messages_received = stats.messages_received,
            messages_sent = stats.messages_sent,
            bytes_received = stats.bytes_received,
            "연결 종료"
        );
    }
}

struct ConnectionStats {
    started: std::time::Instant,
    messages_received: u64,
    messages_sent: u64,
    bytes_received: u64,
}

impl ConnectionStats {
    fn new() -> Self {
        Self {
            started: std::time::Instant::now(),
            messages_received: 0,
            messages_sent: 0,
            bytes_received: 0,
        }
    }

    fn record_message(&mut self, bytes: usize) {
        self.messages_received += 1;
        self.bytes_received += bytes as u64;
    }

    fn record_broadcast(&mut self) {
        self.messages_sent += 1;
    }

    fn duration(&self) -> std::time::Duration {
        self.started.elapsed()
    }
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    // tracing 구독자 초기화
    tracing_subscriber::fmt()
        .with_env_filter(
            tracing_subscriber::EnvFilter::from_default_env()
                .add_directive("chat_server=info".parse()?)
        )
        .with_target(true)
        .with_thread_ids(true)
        .with_line_number(true)
        .init();

    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    info!(addr = "127.0.0.1:8080", "채팅 서버 시작");

    let server = Arc::new(ChatServer::new());

    loop {
        match listener.accept().await {
            Ok((socket, addr)) => {
                info!(client = %addr, "새 연결");
                
                let server = Arc::clone(&server);
                tokio::spawn(async move {
                    server.handle_client(socket, addr).await;
                });
            }
            Err(e) => {
                error!(error = %e, "연결 수락 실패");
            }
        }
    }
}
```

이 코드의 핵심 개선사항들을 살펴보자.

첫째, `#[instrument]` 매크로로 각 클라이언트 핸들러를 추적한다. `client` 필드가 자동으로 모든 로그에 포함되어, 어느 클라이언트의 로그인지 명확하다.

둘째, 구조화된 필드를 사용했다. `messages = count` 같은 방식으로 데이터를 기록하면, 나중에 쿼리하고 집계하기 쉽다.

셋째, `ConnectionStats` 구조체로 각 연결의 통계를 추적한다. 연결이 끊길 때 전체 통계를 로깅해서, 클라이언트 행동 패턴을 분석할 수 있다.

넷째, 구독자 설정을 커스터마이즈했다. `with_target`, `with_thread_ids`, `with_line_number` 같은 옵션으로 더 상세한 정보를 얻을 수 있다.

## **로그와 빌림**

로깅할 때 소유권과 빌림을 주의해야 한다. 로그 매크로는 포맷팅을 위해 값을 빌린다.

```rust
// ✅ 정상: 참조 자동 처리
let msg = String::from("hello");
info!("메시지: {}", msg);
// msg는 여전히 사용 가능

// ✅ 정상: Debug trait
info!("메시지: {:?}", msg);

// ✅ 정상: Display trait을 위한 참조
info!("메시지: {}", &msg);

// tracing의 경우
info!(content = %msg, "메시지 수신");  // % = Display
info!(content = ?msg, "메시지 수신");  // ? = Debug
```

복잡한 타입을 로깅할 때는 적절한 trait이 구현되어 있어야 한다:

```rust
use std::fmt;

#[derive(Debug)]
struct Message {
    content: String,
    priority: u8,
}

// Display 구현으로 커스텀 포맷
impl fmt::Display for Message {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "[P{}] {}", self.priority, self.content)
    }
}

// 사용
let msg = Message {
    content: "urgent".to_string(),
    priority: 1,
};

info!("수신: {}", msg);    // Display 사용: [P1] urgent
debug!("수신: {:?}", msg);  // Debug 사용: Message { content: "urgent", priority: 1 }
```

## **성능 고려사항**

로깅은 성능 오버헤드가 있다. 특히 높은 처리량이 필요한 핫패스(hot path)에서는 주의해야 한다.

```rust
// ❌ 나쁨: 항상 문자열 포맷팅
for i in 0..1000000 {
    debug!("반복: {}", i);  // DEBUG가 꺼져있어도 포맷팅 발생
}

// ✅ 좋음: 레벨 체크
use log::log_enabled;

for i in 0..1000000 {
    if log_enabled!(log::Level::Debug) {
        debug!("반복: {}", i);
    }
}

// 더 나은 방법: 배치 로깅
for i in 0..1000000 {
    // 작업 수행
}
debug!("반복 완료: 1000000개");
```

`tracing`은 이런 문제를 더 잘 처리한다. 비활성화된 스팬과 이벤트는 거의 오버헤드가 없다.

## **프로덕션 로깅 전략**

실제 서비스에서 로깅 전략을 세울 때 고려할 사항들이다.

**로그 레벨 가이드라인:**

- **ERROR**: 즉시 대응이 필요한 문제, 서비스 중단 가능성
- **WARN**: 주의가 필요하지만 서비스는 정상 작동
- **INFO**: 중요한 비즈니스 이벤트, 시작/종료, 연결/해제
- **DEBUG**: 개발/문제 해결용 상세 정보
- **TRACE**: 매우 상세한 실행 흐름

**로그 출력 대상:**

```rust
use tracing_subscriber::layer::SubscriberExt;
use tracing_subscriber::util::SubscriberInitExt;

#[tokio::main]
async fn main() {
    // 콘솔과 파일에 동시 출력
    let file_appender = tracing_appender::rolling::daily("./logs", "chat-server.log");
    let (non_blocking, _guard) = tracing_appender::non_blocking(file_appender);

    tracing_subscriber::registry()
        .with(
            tracing_subscriber::fmt::layer()
                .with_writer(std::io::stdout)
        )
        .with(
            tracing_subscriber::fmt::layer()
                .with_writer(non_blocking)
                .with_ansi(false)
        )
        .with(
            tracing_subscriber::EnvFilter::from_default_env()
        )
        .init();

    info!("서버 시작 - 로그는 콘솔과 파일에 기록됨");
}
```

## **실습 과제**

오늘 배운 내용을 바탕으로 다음 기능을 추가해보자:

1. 메트릭 로깅: 매 분마다 활성 연결 수, 총 메시지 수, 평균 메시지 크기를 로깅하는 태스크를 추가한다.

2. 에러 집계: 같은 종류의 에러가 연속으로 발생하면 첫 번째만 자세히 로깅하고, 나머지는 카운트만 기록한다.

3. 느린 작업 경고: 특정 작업(예: 브로드캐스트)이 100ms 이상 걸리면 WARN 레벨로 로깅한다.

힌트: 느린 작업 감지는 스팬으로 구현할 수 있다:

```rust
use tracing::Instrument;

async fn broadcast_message(msg: ChatMessage) {
    let span = info_span!("broadcast", message_id = ?msg.id);
    
    async {
        let start = std::time::Instant::now();
        // 브로드캐스트 로직
        let elapsed = start.elapsed();
        
        if elapsed > std::time::Duration::from_millis(100) {
            warn!(duration_ms = elapsed.as_millis(), "느린 브로드캐스트");
        }
    }
    .instrument(span)
    .await
}
```

## **정리**

오늘은 프로덕션 환경에 필수적인 구조화된 로깅을 배웠다. `log` 크레이트로 기본적인 로깅 인터페이스를 익히고, `env_logger`로 실제 출력을 구현했다. 그리고 비동기 애플리케이션에 더 적합한 `tracing` 크레이트를 소개했다.

핵심 개념을 정리하면 다음과 같다. 로그 레벨로 중요도를 구분하고, 환경 변수로 런타임에 조절한다. 구조화된 필드를 사용해서 로그를 쿼리하고 분석하기 쉽게 만든다. `tracing`의 스팬과 인스트루먼테이션으로 복잡한 비동기 흐름을 추적한다. 성능을 고려해서 핫패스에서는 로깅을 최소화한다.

빌림 관점에서 보면, 로그 매크로는 값을 빌려서 포맷팅한다. 소유권은 이동되지 않으므로 로깅 후에도 값을 계속 사용할 수 있다. 복잡한 타입은 `Debug`나 `Display` trait을 구현해서 로깅 가능하게 만든다.

내일은 설정 관리를 다룬다. 환경 변수, 설정 파일, 커맨드 라인 인자를 통합해서 유연한 설정 시스템을 만들 것이다. 이것은 로깅과 함께 프로덕션 서비스의 핵심 기반이다.
