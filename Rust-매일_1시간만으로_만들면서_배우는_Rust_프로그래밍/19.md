# Day 19: 브로드캐스트 기능 추가

지금까지 우리는 Arc, Mutex, 그리고 채널을 개별적으로 배웠다. 오늘은 이 모든 것을 종합해서 실전에서 가장 흔히 마주치는 도전 과제를 해결해볼 것이다. 바로 **한 클라이언트의 메시지를 모든 다른 클라이언트에게 전달하는 브로드캐스트 기능**이다. 간단한 채팅 서버를 만들면서 동시성 프로그래밍의 여러 개념이 어떻게 조화롭게 작동하는지 깊이 이해하게 될 것이다.

브로드캐스트 기능을 구현하려면 연결된 모든 클라이언트의 송신 채널을 추적해야 한다. 클라이언트가 들어오고 나갈 때마다 이 목록을 업데이트해야 하므로 공유 가변 상태가 필요하다. 또한 각 클라이언트 핸들러 스레드에서 이 목록에 접근해야 하므로 스레드 간 공유가 필요하다. 여기서 `Arc<Mutex<Vec<Sender>>>`라는 강력한 패턴이 등장한다.

## 브로드캐스트 시스템의 구조

먼저 전체적인 구조를 시각화해보자.

```
                    ┌─────────────────────────┐
                    │   Arc<Mutex<Vec<...>>>  │
                    │   클라이언트 목록       │
                    └───────────┬─────────────┘
                                │
              ┌─────────────────┼─────────────────┐
              │                 │                 │
              ▼                 ▼                 ▼
        ┌──────────┐      ┌──────────┐      ┌──────────┐
        │ Client 1 │      │ Client 2 │      │ Client 3 │
        │  Thread  │      │  Thread  │      │  Thread  │
        └────┬─────┘      └────┬─────┘      └────┬─────┘
             │                 │                 │
             │ "Hello"         │                 │
             └────────────────►│                 │
                               ├────────────────►│
                               │                 │
                  (모든 클라이언트에게 브로드캐스트)
```

각 클라이언트 핸들러 스레드는 자신의 수신 채널(`Receiver`)을 가지고 있다. 동시에 모든 핸들러는 공유 목록에 있는 송신 채널(`Sender`)들을 통해 다른 클라이언트에게 메시지를 보낼 수 있다. 한 클라이언트가 메시지를 보내면, 그 핸들러는 목록을 잠그고 모든 송신자에게 메시지를 전달한다.

## 기본 구조 설계하기

먼저 필요한 타입들을 정의한다. 각 클라이언트는 채널을 통해 메시지를 받을 것이므로 `mpsc::Sender`와 `Receiver` 쌍이 필요하다.

```rust
use std::sync::{Arc, Mutex};
use std::sync::mpsc::{self, Sender, Receiver};
use std::net::{TcpListener, TcpStream};
use std::io::{BufRead, BufReader, Write};
use std::thread;

type ClientList = Arc<Mutex<Vec<Sender<String>>>>;
```

`ClientList`는 모든 클라이언트의 송신 채널을 담는 벡터다. `Arc`로 감싸서 여러 스레드가 공유할 수 있게 하고, `Mutex`로 감싸서 한 번에 하나의 스레드만 수정할 수 있게 한다. 타입 별칭을 만들면 코드가 훨씬 읽기 쉬워진다.

## 클라이언트 등록과 해제

클라이언트가 연결되면 새 채널을 만들고 송신자를 공유 목록에 추가해야 한다. 연결이 끊어지면 목록에서 제거해야 한다. 먼저 클라이언트 핸들러 함수의 기본 골격을 만들어보자.

```rust
fn handle_client(
    mut stream: TcpStream,
    client_id: usize,
    clients: ClientList,
) {
    println!("[클라이언트 {}] 연결됨: {}", client_id, stream.peer_addr().unwrap());

    // 이 클라이언트를 위한 채널 생성
    let (tx, rx) = mpsc::channel::<String>();

    // 클라이언트 목록에 송신자 추가
    {
        let mut client_list = clients.lock().unwrap();
        client_list.push(tx);
        println!("[클라이언트 {}] 등록됨. 총 {}명", client_id, client_list.size());
    } // 락이 여기서 해제됨

    // ... 메시지 처리 로직 ...

    println!("[클라이언트 {}] 연결 종료", client_id);
}
```

중괄호 블록 `{ }` 안에서 락을 획득하고 작업을 수행한 후 블록을 벗어나면 자동으로 락이 해제된다는 점이 중요하다. `MutexGuard`는 `Drop` trait을 구현하므로 스코프를 벗어날 때 자동으로 언락된다. 이것이 바로 **빌림 범위 최소화**의 핵심 패턴이다. 락을 잡고 있는 시간을 최소화하면 다른 스레드가 기다리는 시간이 줄어든다.

하지만 여기에는 문제가 있다. 클라이언트가 연결을 끊을 때 목록에서 해당 송신자를 어떻게 제거할까? `Vec<Sender<String>>`에서 특정 `Sender`를 찾아 제거하는 것은 쉽지 않다. `Sender`는 `PartialEq`를 구현하지 않기 때문이다.

더 나은 방법은 각 클라이언트에 고유 ID를 부여하고, 송신자와 함께 저장하는 것이다.

```rust
type ClientList = Arc<Mutex<Vec<(usize, Sender<String>)>>>;

fn handle_client(
    mut stream: TcpStream,
    client_id: usize,
    clients: ClientList,
) {
    println!("[클라이언트 {}] 연결됨", client_id);

    let (tx, rx) = mpsc::channel::<String>();

    // 등록
    {
        let mut client_list = clients.lock().unwrap();
        client_list.push((client_id, tx));
        println!("현재 접속자 수: {}", client_list.len());
    }

    // ... 메시지 처리 ...

    // 해제
    {
        let mut client_list = clients.lock().unwrap();
        client_list.retain(|(id, _)| *id != client_id);
        println!("[클라이언트 {}] 목록에서 제거됨. 남은 인원: {}", 
                 client_id, client_list.len());
    }
}
```

`retain` 메서드는 조건을 만족하는 요소만 남기고 나머지를 제거한다. 여기서는 현재 클라이언트 ID와 다른 것들만 남긴다. 이렇게 하면 연결이 끊어진 클라이언트의 송신자가 깔끔하게 제거된다.

## 메시지 브로드캐스팅

이제 핵심 기능인 브로드캐스트를 구현해보자. 한 클라이언트로부터 메시지를 받으면 모든 클라이언트에게 전달해야 한다.

```rust
fn broadcast_message(clients: &ClientList, sender_id: usize, message: String) {
    let client_list = clients.lock().unwrap();
    
    for (id, tx) in client_list.iter() {
        // 자기 자신에게는 보내지 않음
        if *id == sender_id {
            continue;
        }

        // 메시지 전송 (실패해도 계속 진행)
        if let Err(e) = tx.send(message.clone()) {
            eprintln!("[브로드캐스트] 클라이언트 {}에게 전송 실패: {}", id, e);
        }
    }
}
```

이 함수는 클라이언트 목록을 잠그고 각 송신자에게 메시지를 보낸다. 자기 자신은 제외한다. 만약 전송이 실패하면(수신자가 이미 드롭된 경우) 에러를 출력하고 계속 진행한다. 한 클라이언트에게 전송 실패했다고 해서 다른 클라이언트들에게도 보내지 않으면 안 되기 때문이다.

여기서 `message.clone()`을 호출한다는 점에 주목하자. `send()`는 소유권을 가져가므로, 여러 클라이언트에게 같은 메시지를 보내려면 각각 복사본이 필요하다. `String`은 `Clone`을 구현하므로 복사할 수 있다. 메시지가 매우 크다면 `Arc<String>`을 사용해서 복사 비용을 줄일 수도 있다.

## 양방향 통신 구조

각 클라이언트 핸들러는 두 가지 일을 동시에 해야 한다. 하나는 TCP 스트림에서 메시지를 읽어 다른 클라이언트들에게 브로드캐스트하는 것이고, 다른 하나는 채널에서 메시지를 받아 자신의 TCP 스트림으로 쓰는 것이다. 이를 위해 스레드를 하나 더 생성한다.

```
┌─────────────────────────────────────┐
│      클라이언트 핸들러 스레드       │
│                                     │
│  ┌──────────┐        ┌───────────┐ │
│  │ 읽기     │        │ 쓰기      │ │
│  │ 스레드   │        │ 스레드    │ │
│  │          │        │           │ │
│  │ TCP 읽기 │        │ 채널 수신 │ │
│  │    ↓     │        │    ↓      │ │
│  │ 브로드   │        │ TCP 쓰기  │ │
│  │ 캐스트   │        │           │ │
│  └──────────┘        └───────────┘ │
└─────────────────────────────────────┘
```

전체 핸들러 함수를 구현해보자.

```rust
fn handle_client(
    stream: TcpStream,
    client_id: usize,
    clients: ClientList,
) {
    println!("[클라이언트 {}] 연결됨: {}", client_id, 
             stream.peer_addr().unwrap());

    let (tx, rx) = mpsc::channel::<String>();

    // 클라이언트 등록
    {
        let mut client_list = clients.lock().unwrap();
        client_list.push((client_id, tx));
        println!("총 접속자: {}", client_list.len());
    }

    // TCP 스트림 복제 (읽기용, 쓰기용)
    let read_stream = stream.try_clone().unwrap();
    let mut write_stream = stream;

    // 쓰기 스레드: 채널에서 메시지를 받아 클라이언트에게 전송
    let write_handle = thread::spawn(move || {
        for message in rx {
            if let Err(e) = writeln!(write_stream, "{}", message) {
                eprintln!("[클라이언트 {}] 쓰기 에러: {}", client_id, e);
                break;
            }
        }
    });

    // 읽기 루프: 클라이언트로부터 메시지를 받아 브로드캐스트
    let reader = BufReader::new(read_stream);
    for line in reader.lines() {
        match line {
            Ok(message) => {
                if message.trim().is_empty() {
                    continue;
                }
                println!("[클라이언트 {}] 메시지: {}", client_id, message);
                
                let formatted = format!("[{}]: {}", client_id, message);
                broadcast_message(&clients, client_id, formatted);
            }
            Err(e) => {
                eprintln!("[클라이언트 {}] 읽기 에러: {}", client_id, e);
                break;
            }
        }
    }

    // 클라이언트 해제
    {
        let mut client_list = clients.lock().unwrap();
        client_list.retain(|(id, _)| *id != client_id);
        println!("[클라이언트 {}] 제거됨. 남은 인원: {}", 
                 client_id, client_list.len());
    }

    // 쓰기 스레드 종료 대기
    write_handle.join().unwrap();
    println!("[클라이언트 {}] 핸들러 종료", client_id);
}
```

이 코드에서 몇 가지 중요한 점을 살펴보자. 먼저 `stream.try_clone()`을 호출해서 TCP 스트림을 복제한다. `TcpStream`은 내부적으로 파일 디스크립터를 참조 카운팅하므로 복제가 가능하다. 하나는 읽기용으로, 하나는 쓰기용으로 사용한다.

쓰기 스레드는 채널 `rx`에서 메시지를 받아 `writeln!` 매크로로 TCP 스트림에 쓴다. `rx`를 이터레이터로 사용하므로, 송신자들이 모두 드롭되면 자동으로 루프가 종료된다.

메인 스레드(읽기 루프)는 `BufReader`를 사용해서 라인 단위로 메시지를 읽는다. 각 메시지를 받으면 형식을 갖춰서(클라이언트 ID 포함) 브로드캐스트한다. 연결이 끊어지거나 에러가 발생하면 루프를 빠져나온다.

루프를 빠져나오면 클라이언트를 목록에서 제거하고, 쓰기 스레드가 종료될 때까지 기다린다. 클라이언트를 목록에서 제거하면 더 이상 새 메시지를 받지 못하고, 채널의 송신자가 드롭되면서 수신자의 이터레이션이 종료된다.

## 서버 메인 루프

이제 메인 함수에서 서버를 시작하고 연결을 받는 루프를 만든다.

```rust
fn main() {
    let listener = TcpListener::bind("127.0.0.1:8080")
        .expect("포트 바인딩 실패");
    println!("채팅 서버 시작: 127.0.0.1:8080");

    let clients: ClientList = Arc::new(Mutex::new(Vec::new()));
    let mut next_id = 1;

    for stream in listener.incoming() {
        match stream {
            Ok(stream) => {
                let client_id = next_id;
                next_id += 1;

                let clients_clone = Arc::clone(&clients);
                thread::spawn(move || {
                    handle_client(stream, client_id, clients_clone);
                });
            }
            Err(e) => {
                eprintln!("연결 에러: {}", e);
            }
        }
    }
}
```

`clients`를 `Arc`로 생성하고, 각 클라이언트 핸들러 스레드를 생성할 때 `Arc::clone()`으로 복제본을 넘긴다. 각 스레드는 같은 `Mutex<Vec<...>>`를 공유하지만, 독립적인 `Arc` 핸들을 가진다. 마지막 `Arc`가 드롭될 때 실제 데이터도 정리된다.

## 빌림 범위 최소화로 데드락 방지

지금까지 작성한 코드에서 락을 잡는 부분을 다시 보자.

```rust
{
    let mut client_list = clients.lock().unwrap();
    client_list.push((client_id, tx));
}
```

중괄호로 스코프를 명시적으로 제한했다. 이것이 왜 중요할까? 만약 락을 더 오래 잡고 있으면 어떻게 될까?

```rust
// 나쁜 예: 락을 너무 오래 잡음
let mut client_list = clients.lock().unwrap();
client_list.push((client_id, tx));

// 다른 작업들...
for line in reader.lines() {
    // 여기서도 여전히 락을 잡고 있음!
}
```

이렇게 하면 이 스레드가 락을 잡고 있는 동안 다른 모든 스레드가 클라이언트 목록에 접근할 수 없다. 특히 `broadcast_message`도 같은 락을 얻어야 하는데, 만약 순환 대기 상황이 발생하면 데드락이 생긴다.

데드락의 고전적인 예를 보자.

```rust
// 데드락 가능성이 있는 코드 (실제로 실행하지 마세요!)
fn dangerous_broadcast(clients: &ClientList, message: String) {
    let client_list = clients.lock().unwrap(); // 락 획득
    
    for (id, tx) in client_list.iter() {
        // 메시지 전송 중에 수신자가 블로킹되어 있고,
        // 그 수신자가 같은 락을 기다리면 데드락!
        tx.send(message.clone()).unwrap();
    }
    // 락이 여기서야 해제됨
}
```

만약 송신자가 `send()`에서 블로킹되는 동기 채널을 사용한다면, 수신자가 처리하기를 기다리는 동안 락을 잡고 있게 된다. 하지만 수신자도 같은 락이 필요하면 영원히 기다리게 된다.

우리 코드에서는 비동기 채널(`mpsc::channel`)을 사용하므로 `send()`가 즉시 반환된다. 하지만 일반적인 원칙은 **락을 잡고 있는 동안 블로킹 작업을 하지 않는 것**이다. 락을 획득하고, 필요한 최소한의 작업만 하고, 즉시 해제한다.

다음은 좋은 패턴이다.

```rust
// 1. 락을 잡고 데이터 복사
let senders = {
    let client_list = clients.lock().unwrap();
    client_list.clone() // 벡터 복사
}; // 락 해제

// 2. 락 없이 작업 수행
for (id, tx) in senders {
    tx.send(message.clone()).unwrap();
}
```

벡터를 복사하는 비용이 들지만, 락을 빨리 해제함으로써 동시성이 향상된다. 트레이드오프를 고려해서 상황에 맞는 선택을 해야 한다.

## 연결 끊김 처리 개선

실제 사용 중에 한 가지 문제를 발견할 수 있다. 클라이언트가 비정상 종료되면 송신자에게 `send()`를 호출할 때 에러가 발생하지만, 목록에서 자동으로 제거되지 않는다. 브로드캐스트 함수를 개선해서 실패한 클라이언트를 표시하고, 정기적으로 정리하는 방법을 생각해볼 수 있다.

더 나은 방법은 실패한 송신자를 즉시 제거하는 것이다. 하지만 이미 락을 잡고 있는 상태에서 이터레이션하면서 제거하는 것은 복잡하다. 대신 실패한 ID들을 모아두고 락을 해제한 후 다시 락을 잡고 제거할 수 있다.

```rust
fn broadcast_message(clients: &ClientList, sender_id: usize, message: String) {
    let mut failed_clients = Vec::new();

    {
        let client_list = clients.lock().unwrap();
        
        for (id, tx) in client_list.iter() {
            if *id == sender_id {
                continue;
            }

            if let Err(_) = tx.send(message.clone()) {
                failed_clients.push(*id);
            }
        }
    } // 락 해제

    // 실패한 클라이언트 제거
    if !failed_clients.is_empty() {
        let mut client_list = clients.lock().unwrap();
        client_list.retain(|(id, _)| !failed_clients.contains(id));
        println!("제거된 클라이언트: {:?}", failed_clients);
    }
}
```

락을 두 번 획득하지만, 각각의 임계 구역(critical section)이 짧아서 전체적인 성능은 더 좋을 수 있다. 특히 실패가 드문 경우에는 대부분 첫 번째 락만 사용한다.

## 개선: 환영 메시지와 시스템 알림

채팅 서버를 더 친근하게 만들기 위해 클라이언트가 입장하거나 퇴장할 때 모두에게 알려보자.

```rust
fn handle_client(
    stream: TcpStream,
    client_id: usize,
    clients: ClientList,
) {
    println!("[클라이언트 {}] 연결됨", client_id);

    let (tx, rx) = mpsc::channel::<String>();

    // 등록
    {
        let mut client_list = clients.lock().unwrap();
        client_list.push((client_id, tx.clone()));
    }

    // 환영 메시지 전송

    // 다른 모두에게 입장 알림
    let join_message = format!("*** 클라이언트 {}님이 입장하셨습니다 ***", client_id);
    broadcast_message(&clients, client_id, join_message);

    // ... (쓰기 스레드와 읽기 루프) ...

    // 퇴장 알림
    let leave_message = format!("*** 클라이언트 {}님이 퇴장하셨습니다 ***", client_id);
    broadcast_message(&clients, client_id, leave_message);

    // 클라이언트 해제
    {
        let mut client_list = clients.lock().unwrap();
        client_list.retain(|(id, _)| *id != client_id);
    }
```

이제 서버가 훨씬 대화형이 되었다. 사용자들은 누가 들어오고 나가는지 알 수 있다.

## Arc와 Mutex의 참조 카운팅 이해하기

`Arc`가 어떻게 작동하는지 더 깊이 이해해보자. 각 `Arc::clone()` 호출은 참조 카운트를 증가시킨다. 클론이 드롭되면 카운트가 감소한다. 카운트가 0이 되면 실제 데이터가 해제된다.

```rust
use std::sync::Arc;

fn main() {
    let data = Arc::new(vec![1, 2, 3]);
    println!("강한 참조 수: {}", Arc::strong_count(&data)); // 1

    {
        let data2 = Arc::clone(&data);
        println!("강한 참조 수: {}", Arc::strong_count(&data)); // 2
        
        let data3 = Arc::clone(&data);
        println!("강한 참조 수: {}", Arc::strong_count(&data)); // 3
    } // data2와 data3 드롭됨

    println!("강한 참조 수: {}", Arc::strong_count(&data)); // 1
} // data 드롭되고 벡터도 해제됨
```

우리 채팅 서버에서 클라이언트 목록 `Arc`는 메인 스레드와 각 클라이언트 핸들러 스레드에 복제본이 있다. 모든 클라이언트가 연결을 끊고 메인 스레드도 종료되면 참조 카운트가 0이 되고 `Mutex<Vec<...>>`가 정리된다.

## 전체 코드 정리

지금까지 만든 것을 하나의 완전한 프로그램으로 정리해보자.

```rust
use std::sync::{Arc, Mutex};
use std::sync::mpsc::{self, Sender, Receiver};
use std::net::{TcpListener, TcpStream};
use std::io::{BufRead, BufReader, Write};
use std::thread;

type ClientList = Arc<Mutex<Vec<(usize, Sender<String>)>>>;

fn broadcast_message(clients: &ClientList, sender_id: usize, message: String) {
    let mut failed_clients = Vec::new();

    {
        let client_list = clients.lock().unwrap();
        
        for (id, tx) in client_list.iter() {
            if *id == sender_id {
                continue;
            }

            if tx.send(message.clone()).is_err() {
                failed_clients.push(*id);
            }
        }
    }

    if !failed_clients.is_empty() {
        let mut client_list = clients.lock().unwrap();
        client_list.retain(|(id, _)| !failed_clients.contains(id));
        eprintln!("끊어진 클라이언트 제거: {:?}", failed_clients);
    }
}

fn handle_client(stream: TcpStream, client_id: usize, clients: ClientList) {
    let addr = stream.peer_addr().unwrap();
    println!("[클라이언트 {}] 연결: {}", client_id, addr);

    let (tx, rx) = mpsc::channel::<String>();

    // 클라이언트 등록
    {
        let mut client_list = clients.lock().unwrap();
        client_list.push((client_id, tx.clone()));
        println!("총 접속자: {}", client_list.len());
    }

    // 환영 메시지
    tx.send(format!("=== 채팅 서버에 오신 것을 환영합니다! ===")).unwrap();
    tx.send(format!("당신은 클라이언트 {}입니다.", client_id)).unwrap();

    // 입장 알림
    broadcast_message(&clients, client_id, 
        format!("*** [{}]님이 입장하셨습니다 ***", client_id));

    // TCP 스트림 분리
    let read_stream = stream.try_clone().unwrap();
    let mut write_stream = stream;

    // 쓰기 스레드
    let writer_id = client_id;
    let write_handle = thread::spawn(move || {
        for message in rx {
            if writeln!(write_stream, "{}", message).is_err() {
                eprintln!("[클라이언트 {}] 쓰기 실패", writer_id);
                break;
            }
        }
    });

    // 읽기 루프
    let reader = BufReader::new(read_stream);
    for line in reader.lines() {
        match line {
            Ok(message) => {
                let message = message.trim();
                if message.is_empty() {
                    continue;
                }

                println!("[{}] {}", client_id, message);
                
                let formatted = format!("[{}]: {}", client_id, message);
                broadcast_message(&clients, client_id, formatted);
            }
            Err(e) => {
                eprintln!("[클라이언트 {}] 읽기 에러: {}", client_id, e);
                break;
            }
        }
    }

    // 퇴장 알림
    broadcast_message(&clients, client_id,
        format!("*** [{}]님이 퇴장하셨습니다 ***", client_id));

    // 클라이언트 제거
    {
        let mut client_list = clients.lock().unwrap();
        client_list.retain(|(id, _)| *id != client_id);
        println!("[클라이언트 {}] 제거. 남은 인원: {}", 
                 client_id, client_list.len());
    }

    write_handle.join().unwrap();
    println!("[클라이언트 {}] 핸들러 종료", client_id);
}

fn main() {
    let listener = TcpListener::bind("127.0.0.1:8080")
        .expect("포트 바인딩 실패");
    
    println!("╔════════════════════════════════════╗");
    println!("║    Rust 채팅 서버 v1.0            ║");
    println!("║    127.0.0.1:8080                 ║");
    println!("╚════════════════════════════════════╝");

    let clients: ClientList = Arc::new(Mutex::new(Vec::new()));
    let mut next_id = 1;

    for stream in listener.incoming() {
        match stream {
            Ok(stream) => {
                let client_id = next_id;
                next_id += 1;

                let clients_clone = Arc::clone(&clients);
                thread::spawn(move || {
                    handle_client(stream, client_id, clients_clone);
                });
            }
            Err(e) => {
                eprintln!("연결 수락 에러: {}", e);
            }
        }
    }
}
```

## 성능과 확장성 고려사항

현재 구현은 교육 목적으로 훌륭하지만, 실제 프로덕션 환경에서는 몇 가지 개선이 필요하다.

첫째, 클라이언트마다 스레드를 생성하는 것은 수천 개의 연결에서는 비효율적이다. 스레드는 스택 메모리(보통 2MB)를 소비하고 컨텍스트 스위칭 비용이 있다. Day 21부터 배울 비동기 프로그래밍(async/await)으로 전환하면 수만 개의 동시 연결도 처리할 수 있다.

둘째, 모든 메시지마다 벡터 전체를 순회하며 복사하는 것은 클라이언트가 많아지면 병목이 된다. 각 클라이언트가 $$O(n)$$의 비용이므로, $$n$$명이 메시지를 보내면 $$O(n^2)$$이 된다. 더 효율적인 자료구조나 pub-sub 패턴을 고려할 수 있다.

셋째, 에러 처리가 단순하다. 실제로는 재연결, 타임아웃, 백프레셔 등을 다뤄야 한다.

하지만 이 모든 것에도 불구하고, 현재 코드는 Rust의 동시성 안전성을 완벽하게 보여준다. 데이터 경쟁, 댕글링 포인터, 이중 해제 같은 문제가 컴파일 타임에 방지된다.

## 실전 예제: 명령어 시스템 추가

채팅 서버를 더 실용적으로 만들어보자. `/list`, `/quit` 같은 명령어를 추가한다.

```rust
// 읽기 루프 부분을 수정
for line in reader.lines() {
    match line {
        Ok(message) => {
            let message = message.trim();
            if message.is_empty() {
                continue;
            }

            // 명령어 처리
            if message.starts_with('/') {
                handle_command(&clients, &tx, client_id, message);

            println!("[{}] {}", client_id, message);
            let formatted = format!("[{}]: {}", client_id, message);
            broadcast_message(&clients, client_id, formatted);
        }
        Err(e) => {

            break;
        }
    }
}
```

명령어 핸들러를 구현한다.

```rust
fn handle_command(
    clients: &ClientList,
    my_tx: &Sender<String>,
    my_id: usize,
    command: &str,
) {
    let parts: Vec<&str> = command.split_whitespace().collect();
    let cmd = parts.get(0).unwrap_or(&"");

    match *cmd {
        "/list" => {
            let client_list = clients.lock().unwrap();
            let count = client_list.len();
            my_tx.send(format!("=== 접속자 목록 ({}) ===", count)).unwrap();
            for (id, _) in client_list.iter() {
                let marker = if *id == my_id { " (나)" } else { "" };
                my_tx.send(format!("  클라이언트 {}{}", id, marker)).unwrap();
            }
        }
        "/quit" => {
            my_tx.send(String::from("안녕히 가세요!")).unwrap();
            // 연결을 끊으려면... 여기서는 메시지만 보내고 클라이언트가 끊도록 함
        }
        "/help" => {
            my_tx.send(String::from("=== 사용 가능한 명령어 ===")).unwrap();
            my_tx.send(String::from("  /list - 접속자 목록")).unwrap();
            my_tx.send(String::from("  /help - 도움말")).unwrap();
            my_tx.send(String::from("  /quit - 종료")).unwrap();
        }
        _ => {
            my_tx.send(format!("알 수 없는 명령어: {}. /help를 입력하세요.", cmd)).unwrap();
        }
    }
}
```

이제 클라이언트가 `/list`를 입력하면 현재 접속자 목록을 볼 수 있다. 이 명령어는 브로드캐스트되지 않고 자기 자신에게만 응답한다.

## 정리

오늘 우리는 Rust의 동시성 도구들을 종합적으로 활용해서 실제 작동하는 채팅 서버를 만들었다. `Arc<Mutex<Vec<(usize, Sender<String>)>>>`라는 복잡해 보이는 타입은 사실 각 부분이 명확한 역할을 한다. `Arc`는 스레드 간 공유를, `Mutex`는 안전한 변경을, `Vec`는 동적 목록을, 그리고 `Sender`는 메시지 전달을 담당한다.

빌림 범위를 최소화하는 패턴을 배웠다. 중괄호로 스코프를 명시하고, 락을 잡고 있는 동안에는 최소한의 작업만 수행한다. 이것이 데드락을 방지하고 동시성을 높이는 핵심이다.

채널을 통한 메시지 전달과 공유 상태 관리를 조합했다. 각 클라이언트는 자신의 채널을 가지지만, 모든 송신자는 공유 목록에 저장된다. 한 클라이언트의 메시지는 목록을 통해 모든 다른 클라이언트에게 전달된다.

소유권 시스템 덕분에 우리는 복잡한 멀티스레드 코드를 작성하면서도 컴파일 타임에 안전성을 보장받았다. 데이터 경쟁이나 댕글링 포인터 같은 고전적인 동시성 버그는 애초에 컴파일되지 않는다.

내일은 우아한 종료(graceful shutdown)를 구현해볼 것이다. Ctrl+C 시그널을 처리하고, 모든 클라이언트에게 알린 후, 스레드들을 정리하는 방법을 배운다. 또한 `Drop` trait을 활용해서 리소스를 자동으로 정리하는 패턴도 살펴볼 것이다. 이것으로 동기 멀티스레드 네트워킹의 기초를 마무리하고, 비동기 프로그래밍으로 넘어갈 준비를 하게 된다.
