# Day 29: 에러 처리 라이브러리

## 실전 시나리오: 에러 처리의 고충

지금까지 우리는 `Result<T, E>`와 커스텀 에러 타입을 사용해서 에러를 처리했다. 하지만 실제 프로젝트에서는 여러 종류의 에러가 발생한다. 파일 읽기 에러, JSON 파싱 에러, 네트워크 에러, 설정 검증 에러 등이 섞여 있다. 이런 다양한 에러를 하나의 타입으로 통합하려면 보일러플레이트 코드가 많아진다.

```rust
// Day 12에서 만든 방식 - 보일러플레이트가 많다
#[derive(Debug)]
pub enum ServerError {
    Io(std::io::Error),
    Json(serde_json::Error),
    Config(ConfigError),
    Network(String),
}

impl From<std::io::Error> for ServerError {
    fn from(err: std::io::Error) -> Self {
        ServerError::Io(err)
    }
}

impl From<serde_json::Error> for ServerError {
    fn from(err: serde_json::Error) -> Self {
        ServerError::Json(err)
    }
}

impl std::fmt::Display for ServerError {
    fn fmt(&self, f: &mut std::fmt::Formatter) -> std::fmt::Result {
        match self {
            ServerError::Io(e) => write!(f, "IO 에러: {}", e),
            ServerError::Json(e) => write!(f, "JSON 에러: {}", e),
            ServerError::Config(e) => write!(f, "설정 에러: {}", e),
            ServerError::Network(e) => write!(f, "네트워크 에러: {}", e),
        }
    }
}

impl std::error::Error for ServerError {}
```

매번 이런 코드를 작성하는 것은 지루하고 실수하기 쉽다. 오늘은 Rust 생태계에서 표준처럼 사용되는 두 라이브러리 `anyhow`와 `thiserror`를 배운다. 이들은 에러 처리를 극적으로 간소화하면서도 강력한 기능을 제공한다.

## anyhow: 애플리케이션 에러 처리의 혁명

`anyhow`는 애플리케이션 코드에서 에러를 쉽게 다루기 위한 라이브러리다. 핵심 아이디어는 간단하다. 모든 에러를 `anyhow::Error`로 통합하고, 필요할 때 컨텍스트를 추가한다.

먼저 `Cargo.toml`에 추가한다.

```toml
[dependencies]
anyhow = "1.0"
serde = { version = "1.0", features = ["derive"] }
serde_json = "1.0"
tokio = { version = "1", features = ["full"] }
```

### 기본 사용법

가장 간단한 예제부터 시작한다.

```rust
use anyhow::{Result, Context};
use std::fs;

fn load_config(path: &str) -> Result<String> {
    let content = fs::read_to_string(path)
        .context("설정 파일을 읽을 수 없음")?;
    
    Ok(content)
}

fn parse_config(content: &str) -> Result<ServerConfig> {
    let config: ServerConfig = serde_json::from_str(content)
        .context("JSON 파싱 실패")?;
    
    config.validate()
        .context("설정 검증 실패")?;
    
    Ok(config)
}

fn main() -> Result<()> {
    let content = load_config("config.json")?;
    let config = parse_config(&content)?;
    
    println!("설정 로드 완료: {:?}", config);
    Ok(())
}
```

여기서 주목할 점이 몇 가지 있다.

**`anyhow::Result<T>`**: `std::result::Result<T, anyhow::Error>`의 별칭이다. 에러 타입을 명시할 필요가 없어진다.

**`.context()` 메서드**: 에러에 추가 정보를 덧붙인다. 이것이 `anyhow`의 핵심 기능이다. 원래 에러는 그대로 보존되면서, 왜 그 에러가 문제인지 설명하는 레이어가 추가된다.

**자동 변환**: `?` 연산자가 자동으로 모든 에러를 `anyhow::Error`로 변환한다. `From` trait를 구현할 필요가 없다.

설정 파일이 없을 때 실행하면 이런 에러 메시지를 볼 수 있다.

```
Error: 설정 파일을 읽을 수 없음

Caused by:
    No such file or directory (os error 2)
```

에러의 전체 체인을 보여준다. "설정 파일을 읽을 수 없음"이라는 컨텍스트와 함께 근본 원인인 "파일이 없음"도 표시된다.

### 에러 체인 시각화

에러가 여러 레이어를 거쳐 전파될 때 `anyhow`는 전체 컨텍스트를 유지한다.

```
┌─────────────────────────────────────────────────┐
│              에러 체인 구조                       │
├─────────────────────────────────────────────────┤
│                                                 │
│  애플리케이션 레벨                                │
│  ┌──────────────────────────────────────────┐  │
│  │ "서버 시작 실패"                          │  │
│  └──────────────┬───────────────────────────┘  │
│                 │                               │
│  설정 레벨       ↓                               │
│  ┌──────────────────────────────────────────┐  │
│  │ "설정 파일을 읽을 수 없음"                │  │
│  └──────────────┬───────────────────────────┘  │
│                 │                               │
│  시스템 레벨     ↓                               │
│  ┌──────────────────────────────────────────┐  │
│  │ std::io::Error                           │  │
│  │ "No such file or directory"              │  │
│  └──────────────────────────────────────────┘  │
│                                                 │
└─────────────────────────────────────────────────┘
```

### with_context: 동적 컨텍스트 생성

때로는 변수 값을 포함한 동적 메시지를 추가하고 싶다. `.with_context()`를 사용하면 클로저로 지연 평가가 가능하다.

```rust
use anyhow::{Result, Context};

fn load_user_config(user_id: u64) -> Result<String> {
    let path = format!("/config/user_{}.json", user_id);
    
    fs::read_to_string(&path)
        .with_context(|| format!("사용자 {}의 설정 파일 로드 실패: {}", user_id, path))?;
    
    Ok(content)
}
```

`.context()`는 즉시 평가되지만, `.with_context()`는 에러가 실제로 발생했을 때만 클로저를 실행한다. 에러가 없으면 문자열 포맷팅 비용이 들지 않는다.

### bail!과 ensure! 매크로

`anyhow`는 편리한 매크로도 제공한다.

```rust
use anyhow::{bail, ensure, Result};

fn validate_port(port: u16) -> Result<()> {
    // 조건을 만족하지 않으면 즉시 에러 반환
    ensure!(port > 0, "포트는 0보다 커야 함");
    ensure!(port < 65536, "포트는 65536보다 작아야 함");
    
    if port < 1024 {
        // 명시적으로 에러를 발생시킴
        bail!("1024 미만의 포트는 관리자 권한이 필요함");
    }
    
    Ok(())
}

fn start_server(config: &ServerConfig) -> Result<()> {
    validate_port(config.server.port)?;
    
    ensure!(
        config.server.max_connections > 0,
        "최대 연결 수는 양수여야 함: {}",
        config.server.max_connections
    );
    
    // 서버 시작 로직...
    Ok(())
}
```

`ensure!`는 `assert!`의 에러 처리 버전이다. 조건이 거짓이면 `Result::Err`를 반환한다. `bail!`은 즉시 에러를 반환하는 편리한 방법이다.

### 실전 예제: 서버 초기화

이제 실제 서버 초기화 코드를 `anyhow`로 개선해본다.

```rust
use anyhow::{Context, Result};
use std::sync::Arc;
use tokio::net::TcpListener;

async fn initialize_server() -> Result<()> {
    // 설정 로드
    let config = load_server_config()
        .context("서버 설정 초기화 실패")?;
    
    // 설정 검증
    validate_config(&config)
        .context("설정 검증 실패")?;
    
    // 로거 초기화
    setup_logging(&config.logging)
        .context("로깅 시스템 초기화 실패")?;
    
    // TCP 리스너 바인딩
    let addr = format!("{}:{}", config.server.host, config.server.port);
    let listener = TcpListener::bind(&addr)
        .await
        .with_context(|| format!("{}에 바인딩 실패", addr))?;
    
    println!("서버가 {}에서 대기 중", addr);
    
    // 공유 상태 생성
    let state = Arc::new(ServerState::new(config));
    
    // 메인 루프 실행
    run_server(listener, state)
        .await
        .context("서버 실행 중 에러 발생")?;
    
    Ok(())
}

fn load_server_config() -> Result<ServerConfig> {
    let config_path = std::env::var("CONFIG_PATH")
        .unwrap_or_else(|_| "config.json".to_string());
    
    let content = std::fs::read_to_string(&config_path)
        .with_context(|| format!("설정 파일 읽기 실패: {}", config_path))?;
    
    let config: ServerConfig = serde_json::from_str(&content)
        .context("JSON 파싱 실패")?;
    
    Ok(config)
}

fn validate_config(config: &ServerConfig) -> Result<()> {
    ensure!(
        config.server.port > 0,
        "포트가 유효하지 않음: {}",
        config.server.port
    );
    
    ensure!(
        config.server.max_connections <= 10000,
        "최대 연결 수가 너무 큼: {} (최대 10000)",
        config.server.max_connections
    );
    
    let valid_log_levels = ["trace", "debug", "info", "warn", "error"];
    ensure!(
        valid_log_levels.contains(&config.logging.level.as_str()),
        "잘못된 로그 레벨: {}. 가능한 값: {:?}",
        config.logging.level,
        valid_log_levels
    );
    
    Ok(())
}

#[tokio::main]
async fn main() -> Result<()> {
    initialize_server().await?;
    Ok(())
}
```

이 코드의 장점은 명확하다. 에러 처리 보일러플레이트가 거의 없으면서도, 각 단계에서 무슨 일이 일어나는지 명확하게 추적할 수 있다. 에러가 발생하면 전체 컨텍스트를 볼 수 있다.

```
Error: 서버 설정 초기화 실패

Caused by:
    0: 설정 파일 읽기 실패: config.json
    1: No such file or directory (os error 2)
```

## thiserror: 라이브러리용 커스텀 에러

`anyhow`는 애플리케이션 코드에 완벽하지만, 라이브러리를 만들 때는 다르다. 라이브러리는 구체적인 에러 타입을 제공해서 사용자가 패턴 매칭으로 에러를 처리할 수 있어야 한다. 이때 `thiserror`를 사용한다.

`Cargo.toml`에 추가한다.

```toml
[dependencies]
thiserror = "1.0"
```

### 기본 사용법

`thiserror`는 derive 매크로로 에러 타입을 간단하게 만든다.

```rust
use thiserror::Error;

#[derive(Error, Debug)]
pub enum ConfigError {
    #[error("파일 읽기 실패: {0}")]
    FileRead(#[from] std::io::Error),
    
    #[error("JSON 파싱 실패: {0}")]
    ParseError(#[from] serde_json::Error),
    
    #[error("설정 검증 실패: {message}")]
    ValidationError { message: String },
    
    #[error("필수 필드 누락: {field}")]
    MissingField { field: String },
    
    #[error("잘못된 값: {field}={value} (이유: {reason})")]
    InvalidValue {
        field: String,
        value: String,
        reason: String,
    },
}
```

이 간단한 코드가 이전에 우리가 수동으로 작성했던 모든 것을 생성한다.

**`#[error("...")]`**: `Display` trait를 자동으로 구현한다. 중괄호 안의 필드 이름이나 튜플 인덱스를 사용할 수 있다.

**`#[from]`**: 다른 에러 타입에서 자동 변환을 위한 `From` trait를 생성한다.

**`Error` trait**: 자동으로 구현된다.

### 실전 예제: 설정 모듈 개선

Day 27에서 만든 설정 모듈을 `thiserror`로 개선한다.

```rust
use thiserror::Error;
use serde::{Deserialize, Serialize};
use std::path::Path;
use std::fs;

#[derive(Error, Debug)]
pub enum ConfigError {
    #[error("설정 파일 읽기 실패: {path}")]
    FileRead {
        path: String,
        #[source]
        source: std::io::Error,
    },
    
    #[error("JSON 파싱 실패")]
    JsonParse(#[from] serde_json::Error),
    
    #[error("환경 변수 파싱 실패: {key}={value}")]
    EnvVarParse {
        key: String,
        value: String,
        #[source]
        source: Box<dyn std::error::Error + Send + Sync>,
    },
    
    #[error("설정 검증 실패: {0}")]
    Validation(String),
    
    #[error("포트 범위 초과: {port} (허용 범위: 1-65535)")]
    InvalidPort { port: u32 },
    
    #[error("잘못된 로그 레벨: {level}")]
    InvalidLogLevel { level: String },
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ServerConfig {
    pub server: NetworkConfig,
    pub timeout: TimeoutConfig,
    pub logging: LoggingConfig,
}

impl ServerConfig {
    pub fn from_file<P: AsRef<Path>>(path: P) -> Result<Self, ConfigError> {
        let path_str = path.as_ref().display().to_string();
        
        let content = fs::read_to_string(path.as_ref())
            .map_err(|source| ConfigError::FileRead {
                path: path_str,
                source,
            })?;
        
        let config: ServerConfig = serde_json::from_str(&content)?;
        
        config.validate()?;
        
        Ok(config)
    }
    
    pub fn from_file_with_env<P: AsRef<Path>>(path: P) -> Result<Self, ConfigError> {
        let mut config = if path.as_ref().exists() {
            Self::from_file(path)?
        } else {
            Self::default()
        };
        
        // 환경 변수로 포트 오버라이드
        if let Ok(port_str) = std::env::var("SERVER_PORT") {
            let port: u16 = port_str.parse()
                .map_err(|e| ConfigError::EnvVarParse {
                    key: "SERVER_PORT".to_string(),
                    value: port_str.clone(),
                    source: Box::new(e),
                })?;
            config.server.port = port;
        }
        
        // 환경 변수로 로그 레벨 오버라이드
        if let Ok(level) = std::env::var("LOG_LEVEL") {
            config.logging.level = level;
        }
        
        config.validate()?;
        
        Ok(config)
    }
    
    pub fn validate(&self) -> Result<(), ConfigError> {
        // 포트 검증
        if self.server.port == 0 {
            return Err(ConfigError::InvalidPort { port: 0 });
        }
        
        // 로그 레벨 검증
        let valid_levels = ["trace", "debug", "info", "warn", "error"];
        if !valid_levels.contains(&self.logging.level.as_str()) {
            return Err(ConfigError::InvalidLogLevel {
                level: self.logging.level.clone(),
            });
        }
        
        // 최대 연결 수 검증
        if self.server.max_connections == 0 {
            return Err(ConfigError::Validation(
                "최대 연결 수는 0보다 커야 함".to_string()
            ));
        }
        
        Ok(())
    }
}
```

이제 에러를 구체적으로 처리할 수 있다.

        }

## anyhow와 thiserror 함께 사용하기

실제 프로젝트에서는 두 라이브러리를 함께 사용한다. 라이브러리 코드는 `thiserror`로 구체적인 에러를 정의하고, 애플리케이션 코드는 `anyhow`로 간편하게 처리한다.

```rust
// src/config.rs (라이브러리 부분)
use thiserror::Error;

#[derive(Error, Debug)]
pub enum ConfigError {
    #[error("파일 읽기 실패: {0}")]
    FileRead(#[from] std::io::Error),
    
    #[error("파싱 실패: {0}")]
    Parse(#[from] serde_json::Error),
    
    #[error("검증 실패: {0}")]
    Validation(String),
}

impl ServerConfig {
    pub fn from_file(path: &str) -> Result<Self, ConfigError> {
        // ConfigError를 반환
        let content = std::fs::read_to_string(path)?;
        let config: ServerConfig = serde_json::from_str(&content)?;
        config.validate()?;
        Ok(config)
    }
}

// src/main.rs (애플리케이션 부분)
use anyhow::{Context, Result};

#[tokio::main]
async fn main() -> Result<()> {
    // ConfigError는 자동으로 anyhow::Error로 변환됨
    let config = ServerConfig::from_file("config.json")
        .context("서버 설정 로드 실패")?;
    
    start_server(config)
        .await
        .context("서버 실행 실패")?;
    
    Ok(())
}

async fn start_server(config: ServerConfig) -> Result<()> {
    let addr = format!("{}:{}", config.server.host, config.server.port);
    
    let listener = TcpListener::bind(&addr)
        .await
        .with_context(|| format!("주소 {}에 바인딩 실패", addr))?;
    
    println!("서버가 {}에서 대기 중", addr);
    
    // 서버 로직...
    
    Ok(())
}
```

이 패턴의 장점은 명확하다. 라이브러리 사용자는 구체적인 에러 타입으로 정밀하게 처리할 수 있고, 애플리케이션 개발자는 간단하게 에러를 전파하고 컨텍스트를 추가할 수 있다.

## 에러 다운캐스팅

때로는 `anyhow::Error` 안에 있는 구체적인 에러 타입을 꺼내야 할 때가 있다. `downcast_ref()`를 사용한다.

```rust
use anyhow::{Result, Context};

fn process_with_retry() -> Result<()> {
    for attempt in 1..=3 {
        match load_config() {
            Ok(config) => {
                println!("설정 로드 성공");
                return Ok(());
            }
            Err(e) => {
                // ConfigError::FileRead인 경우에만 재시도
                if let Some(config_err) = e.downcast_ref::<ConfigError>() {
                    match config_err {
                        ConfigError::FileRead { .. } => {
                            eprintln!("시도 {}/3: 파일 읽기 실패, 재시도 중...", attempt);
                            std::thread::sleep(std::time::Duration::from_secs(1));
                            continue;
                        }
                        _ => return Err(e),
                    }
                }
                return Err(e);
            }
        }
    }
    
    Err(anyhow::anyhow!("3번 시도 후에도 설정 로드 실패"))
}

fn load_config() -> Result<ServerConfig> {
    ServerConfig::from_file("config.json")
        .context("설정 로드 중 에러")
        .map_err(|e| e.into()) // ConfigError를 anyhow::Error로 변환
}
```

## 실전 팁: 언제 무엇을 사용할까

**anyhow를 사용하는 경우**:

- 애플리케이션 코드 (바이너리 프로젝트)
- 빠른 프로토타이핑
- 에러의 구체적인 타입이 중요하지 않을 때
- 다양한 외부 라이브러리 에러를 통합할 때

**thiserror를 사용하는 경우**:

- 라이브러리 코드 (pub으로 공개되는 API)
- 에러를 패턴 매칭으로 처리해야 할 때
- 구체적인 에러 타입이 API의 일부일 때
- 에러별로 다른 복구 전략이 필요할 때

**둘 다 사용하는 경우**:

- 대부분의 실제 프로젝트
- 라이브러리 부분은 thiserror, 애플리케이션 부분은 anyhow

## 프로토콜 파서에 적용하기

이제 Day 13의 프로토콜 파서에 에러 처리를 개선한다.

```rust
use thiserror::Error;

#[derive(Error, Debug)]
pub enum ProtocolError {
    #[error("잘못된 메시지 형식: {0}")]
    InvalidFormat(String),
    
    #[error("메시지가 너무 큼: {size} bytes (최대: {max})")]
    MessageTooLarge { size: usize, max: usize },
    
    #[error("빈 메시지")]
    EmptyMessage,
    
    #[error("지원하지 않는 명령: {0}")]
    UnsupportedCommand(String),
    
    #[error("IO 에러")]
    Io(#[from] std::io::Error),
}

#[derive(Debug, PartialEq)]
pub enum Message {
    Ping,
    Pong,
    Echo(String),
}

impl Message {
    pub fn parse(input: &str) -> Result<Self, ProtocolError> {
        let trimmed = input.trim();
        
        if trimmed.is_empty() {
            return Err(ProtocolError::EmptyMessage);
        }
        
        if trimmed.len() > 1024 {
            return Err(ProtocolError::MessageTooLarge {
                size: trimmed.len(),
                max: 1024,
            });
        }
        
        if trimmed.eq_ignore_ascii_case("PING") {
            Ok(Message::Ping)
        } else if trimmed.eq_ignore_ascii_case("PONG") {
            Ok(Message::Pong)
        } else if let Some(content) = trimmed.strip_prefix("ECHO ") {
            Ok(Message::Echo(content.to_string()))
        } else {
            Err(ProtocolError::UnsupportedCommand(trimmed.to_string()))
        }
    }
    
    pub fn to_bytes(&self) -> Vec<u8> {
        let s = match self {
            Message::Ping => "PING",
            Message::Pong => "PONG",
            Message::Echo(content) => return format!("ECHO {}", content).into_bytes(),
        };
        s.as_bytes().to_vec()
    }
}

// 사용 예제
use anyhow::{Result, Context};

async fn handle_client(mut socket: TcpStream) -> Result<()> {
    let mut buffer = [0u8; 1024];
    
    loop {
        let n = socket.read(&mut buffer)
            .await
            .context("클라이언트에서 읽기 실패")?;
        
        if n == 0 {
            break; // 연결 종료
        }
        
        let input = String::from_utf8_lossy(&buffer[..n]);
        
        match Message::parse(&input) {
            Ok(Message::Ping) => {
                socket.write_all(b"PONG\n")
                    .await
                    .context("PONG 응답 전송 실패")?;
            }
            Ok(Message::Echo(content)) => {
                let response = format!("ECHO {}\n", content);
                socket.write_all(response.as_bytes())
                    .await
                    .context("ECHO 응답 전송 실패")?;
            }
            Ok(Message::Pong) => {
                // 클라이언트가 PONG을 보냄 (무시)
            }
            Err(ProtocolError::UnsupportedCommand(cmd)) => {
                let response = format!("ERROR: 알 수 없는 명령: {}\n", cmd);
                socket.write_all(response.as_bytes())
                    .await
                    .context("에러 응답 전송 실패")?;
            }
            Err(e) => {
                eprintln!("프로토콜 에러: {}", e);
                let response = format!("ERROR: {}\n", e);
                socket.write_all(response.as_bytes())
                    .await
                    .context("에러 응답 전송 실패")?;
            }
        }
    }
    
    Ok(())
}
```

## 테스트에서의 에러 처리

#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_parse_empty_message() {

        assert!(matches!(result, Err(ProtocolError::EmptyMessage)));
    }

    fn test_parse_message_too_large() {

        match result {
            Err(ProtocolError::MessageTooLarge { size, max }) => {

        }
    }

## 에러 메트릭 수집

프로덕션 환경에서는 어떤 에러가 얼마나 자주 발생하는지 추적하는 것이 중요하다.

```rust
use std::sync::atomic::{AtomicU64, Ordering};
use std::sync::Arc;

pub struct ErrorMetrics {
    protocol_errors: AtomicU64,
    network_errors: AtomicU64,
    config_errors: AtomicU64,
}

impl ErrorMetrics {
    pub fn new() -> Self {
        Self {
            protocol_errors: AtomicU64::new(0),
            network_errors: AtomicU64::new(0),
            config_errors: AtomicU64::new(0),
        }
    }
    
    pub fn record_error(&self, error: &anyhow::Error) {
        if error.downcast_ref::<ProtocolError>().is_some() {
            self.protocol_errors.fetch_add(1, Ordering::Relaxed);
        } else if error.downcast_ref::<std::io::Error>().is_some() {
            self.network_errors.fetch_add(1, Ordering::Relaxed);
        } else if error.downcast_ref::<ConfigError>().is_some() {
            self.config_errors.fetch_add(1, Ordering::Relaxed);
        }
    }
    
    pub fn report(&self) {
        println!("에러 통계:");
        println!("  프로토콜 에러: {}", self.protocol_errors.load(Ordering::Relaxed));
        println!("  네트워크 에러: {}", self.network_errors.load(Ordering::Relaxed));
        println!("  설정 에러: {}", self.config_errors.load(Ordering::Relaxed));
    }
}

// 사용 예제
async fn handle_connection(
    socket: TcpStream,
    metrics: Arc<ErrorMetrics>,
) -> Result<()> {
    if let Err(e) = process_connection(socket).await {
        metrics.record_error(&e);
        return Err(e);
    }
    Ok(())
}
```

## 마무리

오늘은 `anyhow`와 `thiserror`를 사용해서 에러 처리를 극적으로 개선했다. `anyhow`는 애플리케이션 코드에서 다양한 에러를 쉽게 통합하고 컨텍스트를 추가하게 해준다. `thiserror`는 라이브러리 코드에서 타입 안전하고 사용하기 쉬운 커스텀 에러를 정의하게 해준다.

이 두 라이브러리는 Rust 생태계에서 사실상 표준이 되었다. 보일러플레이트 코드를 크게 줄이면서도, 에러의 원인을 추적하고 적절하게 처리할 수 있는 강력한 도구를 제공한다. 특히 `anyhow`의 컨텍스트 기능은 복잡한 코드에서 에러가 발생한 정확한 위치와 상황을 파악하는 데 매우 유용하다.

내일은 성능 측정과 최적화를 배운다. 지금까지 작성한 코드가 얼마나 빠른지 측정하고, 불필요한 복사를 제거하며, 참조와 빌림을 활용해서 제로 카피 최적화를 달성하는 방법을 다룬다. 오늘 배운 에러 처리 기법은 성능 측정 과정에서도 계속 사용될 것이다.
