# Day 36: trait으로 추상화하기

오늘은 Rust의 trait 시스템을 활용하여 코드를 추상화하는 방법을 배운다. trait은 다른 언어의 인터페이스와 비슷하지만, Rust의 소유권 시스템과 결합되어 훨씬 강력한 기능을 제공한다. 실전에서는 우리가 만들어온 서버에 플러그인 가능한 메시지 핸들러 시스템을 구축하면서 trait의 활용법을 익힌다.

## **trait의 기본 개념**

trait은 특정 타입이 가져야 하는 기능의 집합을 정의한다. 여러 타입이 같은 trait을 구현하면, 그 타입들을 동일한 방식으로 다룰 수 있다. 이것이 바로 추상화의 핵심이다.

간단한 예제부터 시작해보자. 우리 서버에서 다양한 종류의 메시지를 처리해야 한다고 가정한다. 각 메시지 타입마다 처리 방법이 다르지만, 모든 메시지는 "처리할 수 있다"는 공통점이 있다.

```rust
// 메시지 처리 능력을 나타내는 trait
trait MessageHandler {
    fn handle(&self, message: &str) -> Result<String, String>;
    fn name(&self) -> &str;
}

// 에코 핸들러: 받은 메시지를 그대로 돌려준다
struct EchoHandler;

impl MessageHandler for EchoHandler {
    fn handle(&self, message: &str) -> Result<String, String> {
        Ok(message.to_string())
    }
    
    fn name(&self) -> &str {
        "Echo"
    }
}

// 대문자 변환 핸들러
struct UppercaseHandler;

impl MessageHandler for UppercaseHandler {
    fn handle(&self, message: &str) -> Result<String, String> {
        Ok(message.to_uppercase())
    }
    
    fn name(&self) -> &str {
        "Uppercase"
    }
}

// 역순 핸들러
struct ReverseHandler;

impl MessageHandler for ReverseHandler {
    fn handle(&self, message: &str) -> Result<String, String> {
        Ok(message.chars().rev().collect())
    }
    
    fn name(&self) -> &str {
        "Reverse"
    }
}
```

이제 어떤 핸들러든 `MessageHandler` trait을 구현하기만 하면 동일한 방식으로 사용할 수 있다. trait을 사용하는 방법은 크게 두 가지가 있다.

## **제네릭과 trait bounds: 정적 디스패치**

첫 번째 방법은 제네릭 타입과 trait bounds를 사용하는 것이다. 컴파일 시점에 구체적인 타입이 결정되므로 성능이 뛰어나다.

```rust
// 제네릭을 사용한 메시지 처리 함수
fn process_message<H: MessageHandler>(handler: &H, msg: &str) {
    println!("[{}] Processing: {}", handler.name(), msg);
    
    match handler.handle(msg) {
        Ok(result) => println!("[{}] Result: {}", handler.name(), result),
        Err(e) => println!("[{}] Error: {}", handler.name(), e),
    }
}

fn main() {
    let echo = EchoHandler;
    let upper = UppercaseHandler;
    let reverse = ReverseHandler;
    
    let message = "Hello, Rust!";
    
    process_message(&echo, message);
    process_message(&upper, message);
    process_message(&reverse, message);
}
```

여기서 `<H: MessageHandler>`는 "H는 MessageHandler trait을 구현하는 어떤 타입"이라는 의미다. 컴파일러는 각 호출마다 별도의 함수 버전을 생성한다. 이것을 단형화(monomorphization)라고 하며, 런타임 오버헤드가 없다는 장점이 있다.

trait bounds는 여러 가지 방법으로 표현할 수 있다. 복잡한 경우에는 `where` 절을 사용하면 가독성이 좋아진다.

```rust
// 여러 trait bounds를 가진 함수
fn process_and_log<H>(handler: &H, msg: &str) 
where 
    H: MessageHandler + std::fmt::Debug,
{
    println!("{:?}", handler);
    process_message(handler, msg);
}
```

## **trait 객체와 동적 디스패치**

두 번째 방법은 trait 객체를 사용하는 것이다. 런타임에 어떤 구체적 타입이 사용될지 결정되므로, 서로 다른 타입을 하나의 컬렉션에 담을 수 있다.

```rust
use std::collections::HashMap;

struct HandlerRegistry {
    handlers: HashMap<String, Box<dyn MessageHandler>>,
}

impl HandlerRegistry {
    fn new() -> Self {
        HandlerRegistry {
            handlers: HashMap::new(),
        }
    }
    
    fn register(&mut self, command: String, handler: Box<dyn MessageHandler>) {
        self.handlers.insert(command, handler);
    }
    
    fn handle(&self, command: &str, message: &str) -> Result<String, String> {
        match self.handlers.get(command) {
            Some(handler) => {
                println!("Using handler: {}", handler.name());
                handler.handle(message)
            }
            None => Err(format!("Unknown command: {}", command)),
        }
    }
}

fn main() {
    let mut registry = HandlerRegistry::new();
    
    // 다양한 타입의 핸들러를 하나의 HashMap에 저장
    registry.register("echo".to_string(), Box::new(EchoHandler));
    registry.register("upper".to_string(), Box::new(UppercaseHandler));
    registry.register("reverse".to_string(), Box::new(ReverseHandler));
    
    // 명령어에 따라 적절한 핸들러 사용
    let commands = vec![
        ("echo", "Hello"),
        ("upper", "world"),
        ("reverse", "Rust"),
        ("unknown", "test"),
    ];
    
    for (cmd, msg) in commands {
        match registry.handle(cmd, msg) {
            Ok(result) => println!("'{}' -> {}", msg, result),
            Err(e) => println!("Error: {}", e),
        }
    }
}
```

`Box<dyn MessageHandler>`가 바로 trait 객체다. `dyn` 키워드는 동적 디스패치를 사용한다는 것을 명시한다. trait 객체는 포인터 두 개로 구성되는데, 하나는 실제 데이터를 가리키고 다른 하나는 가상 함수 테이블(vtable)을 가리킨다.

```
┌─────────────────────────────────────┐
│    Box<dyn MessageHandler>          │
├─────────────────────────────────────┤
│ data pointer    -> EchoHandler 인스턴스│
│ vtable pointer  -> MessageHandler vtable│
└─────────────────────────────────────┘
```

trait 객체를 사용할 때는 몇 가지 제약이 있다. trait이 "객체 안전(object safe)"해야 한다는 것인데, 주요 규칙은 다음과 같다.

- 반환 타입이 `Self`인 메서드가 없어야 한다
- 제네릭 타입 매개변수가 없어야 한다

## **실전: 비동기 메시지 핸들러 시스템**

이제 실제 서버에 적용할 수 있는 비동기 핸들러 시스템을 만들어보자. 앞서 만든 채팅 서버를 확장하여 다양한 명령어를 처리할 수 있게 한다.

```rust
use async_trait::async_trait;
use std::sync::Arc;
use tokio::sync::RwLock;
use std::collections::HashMap;

// 비동기 trait - async_trait 크레이트 필요
#[async_trait]
trait AsyncMessageHandler: Send + Sync {
    async fn handle(&self, message: &str) -> Result<String, String>;
    fn name(&self) -> &str;
    fn description(&self) -> &str;
}

// 공유 상태를 가진 카운터 핸들러
struct CounterHandler {
    count: Arc<RwLock<u64>>,
}

impl CounterHandler {
    fn new() -> Self {
        CounterHandler {
            count: Arc::new(RwLock::new(0)),
        }
    }
}

#[async_trait]
impl AsyncMessageHandler for CounterHandler {
    async fn handle(&self, _message: &str) -> Result<String, String> {
        let mut count = self.count.write().await;
        *count += 1;
        Ok(format!("Message count: {}", *count))
    }
    
    fn name(&self) -> &str {
        "counter"
    }
    
    fn description(&self) -> &str {
        "Counts the number of messages received"
    }
}

// 시간을 반환하는 핸들러
struct TimeHandler;

#[async_trait]
impl AsyncMessageHandler for TimeHandler {
    async fn handle(&self, _message: &str) -> Result<String, String> {
        use chrono::Local;
        let now = Local::now();
        Ok(format!("Current time: {}", now.format("%Y-%m-%d %H:%M:%S")))
    }
    
    fn name(&self) -> &str {
        "time"
    }
    
    fn description(&self) -> &str {
        "Returns the current server time"
    }
}

// 외부 API를 호출하는 핸들러
struct WeatherHandler {
    client: reqwest::Client,
}

impl WeatherHandler {
    fn new() -> Self {
        WeatherHandler {
            client: reqwest::Client::new(),
        }
    }
}

#[async_trait]
impl AsyncMessageHandler for WeatherHandler {
    async fn handle(&self, city: &str) -> Result<String, String> {
        // 실제로는 API 키가 필요하지만, 예제로만 보여준다
        tokio::time::sleep(tokio::time::Duration::from_millis(100)).await;
        Ok(format!("Weather in {}: Sunny, 22°C", city))
    }
    
    fn name(&self) -> &str {
        "weather"
    }
    
    fn description(&self) -> &str {
        "Gets weather information for a city"
    }
}
```

이제 핸들러 레지스트리를 만들어서 동적으로 핸들러를 등록하고 사용할 수 있게 한다.

```rust
type HandlerBox = Arc<dyn AsyncMessageHandler>;

struct AsyncHandlerRegistry {
    handlers: RwLock<HashMap<String, HandlerBox>>,
}

impl AsyncHandlerRegistry {
    fn new() -> Self {
        AsyncHandlerRegistry {
            handlers: RwLock::new(HashMap::new()),
        }
    }
    
    async fn register(&self, handler: HandlerBox) {
        let name = handler.name().to_string();
        let mut handlers = self.handlers.write().await;
        handlers.insert(name, handler);
    }
    
    async fn handle(&self, command: &str, message: &str) -> Result<String, String> {
        let handlers = self.handlers.read().await;
        
        match handlers.get(command) {
            Some(handler) => {
                println!("[{}] Executing: {}", handler.name(), message);
                handler.handle(message).await
            }
            None => Err(format!("Unknown command: {}", command)),
        }
    }
    
    async fn list_handlers(&self) -> Vec<(String, String)> {
        let handlers = self.handlers.read().await;
        handlers
            .values()
            .map(|h| (h.name().to_string(), h.description().to_string()))
            .collect()
    }
}
```

여기서 중요한 점은 `RwLock`을 사용한 것이다. 여러 스레드가 동시에 핸들러를 사용할 수 있지만(읽기), 핸들러를 등록할 때만 독점적 접근이 필요하다(쓰기). 이것이 바로 빌림 규칙이 비동기 환경에서 작동하는 방식이다.

## **프로토콜 파싱과 통합**

실제 서버에서 사용하려면 클라이언트의 입력을 파싱하여 적절한 핸들러를 호출해야 한다.

```rust
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use tokio::net::TcpStream;

async fn handle_client(
    stream: TcpStream,
    registry: Arc<AsyncHandlerRegistry>,
) -> Result<(), Box<dyn std::error::Error>> {
    let (reader, mut writer) = stream.into_split();
    let reader = BufReader::new(reader);
    let mut lines = reader.lines();
    
    // 환영 메시지
    writer.write_all(b"Welcome! Type 'help' for commands.\n").await?;
    
    while let Some(line) = lines.next_line().await? {
        let line = line.trim();
        
        if line.is_empty() {
            continue;
        }
        
        // 명령어와 인자를 분리
        let parts: Vec<&str> = line.splitn(2, ' ').collect();
        let command = parts[0];
        let message = parts.get(1).unwrap_or(&"");
        
        let response = match command {
            "help" => {
                let handlers = registry.list_handlers().await;
                let mut help_text = String::from("Available commands:\n");
                for (name, desc) in handlers {
                    help_text.push_str(&format!("  {} - {}\n", name, desc));
                }
                help_text.push_str("  help - Show this help\n");
                help_text.push_str("  quit - Disconnect\n");
                Ok(help_text)
            }
            "quit" => {
                writer.write_all(b"Goodbye!\n").await?;
                break;
            }
            _ => registry.handle(command, message).await,
        };
        
        match response {
            Ok(result) => {
                writer.write_all(result.as_bytes()).await?;
                writer.write_all(b"\n").await?;
            }
            Err(e) => {
                writer.write_all(format!("Error: {}\n", e).as_bytes()).await?;
            }
        }
    }
    
    Ok(())
}
```

이제 메인 함수에서 모든 것을 조립한다.

```rust
use tokio::net::TcpListener;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let registry = Arc::new(AsyncHandlerRegistry::new());
    
    // 핸들러 등록
    registry.register(Arc::new(CounterHandler::new())).await;
    registry.register(Arc::new(TimeHandler)).await;
    registry.register(Arc::new(WeatherHandler::new())).await;
    registry.register(Arc::new(EchoHandler)).await;
    registry.register(Arc::new(UppercaseHandler)).await;
    registry.register(Arc::new(ReverseHandler)).await;
    
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("Server running on 127.0.0.1:8080");
    
    loop {
        let (stream, addr) = listener.accept().await?;
        println!("New connection from: {}", addr);
        
        let registry = Arc::clone(&registry);
        
        tokio::spawn(async move {
            if let Err(e) = handle_client(stream, registry).await {
                eprintln!("Error handling client {}: {}", addr, e);
            }
        });
    }
}
```

## **trait의 고급 기능들**

trait은 기본 구현을 제공할 수도 있다. 이렇게 하면 trait을 구현하는 타입이 필요한 메서드만 오버라이드하면 된다.

```rust
trait Logger {
    fn log(&self, level: &str, message: &str) {
        println!("[{}] {}", level, message);
    }
    
    fn info(&self, message: &str) {
        self.log("INFO", message);
    }
    
    fn error(&self, message: &str) {
        self.log("ERROR", message);
    }
}

// 기본 구현을 사용하는 타입
struct SimpleLogger;
impl Logger for SimpleLogger {}

// 일부만 오버라이드하는 타입
struct FileLogger {
    file: String,
}

impl Logger for FileLogger {
    fn log(&self, level: &str, message: &str) {
        // 파일에 기록하는 로직
        println!("[{}] {} -> {}", level, message, self.file);
    }
    // info와 error는 기본 구현 사용
}
```

trait은 연관 타입(associated types)을 가질 수 있다. 이것은 trait을 구현하는 타입이 특정 타입을 지정하도록 한다.

```rust
trait Parser {
    type Output;
    type Error;
    
    fn parse(&self, input: &str) -> Result<Self::Output, Self::Error>;
}

struct JsonParser;

impl Parser for JsonParser {
    type Output = serde_json::Value;
    type Error = serde_json::Error;
    
    fn parse(&self, input: &str) -> Result<Self::Output, Self::Error> {
        serde_json::from_str(input)
    }
}

struct IntParser;

impl Parser for IntParser {
    type Output = i32;
    type Error = std::num::ParseIntError;
    
    fn parse(&self, input: &str) -> Result<Self::Output, Self::Error> {
        input.trim().parse()
    }
}

// 제네릭 함수에서 사용
fn parse_and_print<P: Parser>(parser: &P, input: &str) 
where
    P::Output: std::fmt::Debug,
    P::Error: std::fmt::Debug,
{
    match parser.parse(input) {
        Ok(output) => println!("Parsed: {:?}", output),
        Err(e) => println!("Error: {:?}", e),
    }
}
```

## **trait과 소유권의 상호작용**

trait을 사용할 때 소유권과 빌림 규칙이 어떻게 작용하는지 이해하는 것이 중요하다. trait 메서드는 `self`, `&self`, `&mut self` 중 어떤 것을 받을지 결정할 수 있다.

```rust
trait Processor {
    // 값을 소비한다
    fn consume(self) -> String;
    
    // 불변 참조로 읽기만 한다
    fn read(&self) -> String;
    
    // 가변 참조로 수정한다
    fn modify(&mut self, value: String);
}

struct TextProcessor {
    buffer: String,
}

impl Processor for TextProcessor {
    fn consume(self) -> String {
        // self의 소유권을 가져와서 buffer를 이동시킨다
        self.buffer
    }
    
    fn read(&self) -> String {
        // buffer를 복제해서 반환한다
        self.buffer.clone()
    }
    
    fn modify(&mut self, value: String) {
        // buffer를 직접 수정한다
        self.buffer.push_str(&value);
    }
}

fn demonstrate_ownership() {
    let mut processor = TextProcessor {
        buffer: String::from("Hello"),
    };
    
    // 불변 참조 - 여러 번 호출 가능
    println!("{}", processor.read());
    println!("{}", processor.read());
    
    // 가변 참조 - 수정 가능
    processor.modify(String::from(", World"));
    println!("{}", processor.read());
    
    // 소유권 이동 - 이후 processor 사용 불가
    let final_string = processor.consume();
    println!("{}", final_string);
    
    // 컴파일 에러: processor는 이미 소비됨
    // println!("{}", processor.read());
}
```

trait 객체를 사용할 때는 `Send`와 `Sync` trait이 중요하다. 이들은 타입이 스레드 간에 안전하게 전송되거나 공유될 수 있는지를 나타낸다.

```rust
// Send: 다른 스레드로 소유권을 이동시킬 수 있다
// Sync: 여러 스레드에서 동시에 참조할 수 있다

// 스레드 안전한 핸들러
fn create_thread_safe_handler() -> Box<dyn AsyncMessageHandler> {
    Box::new(TimeHandler)
}

// 여러 스레드에서 공유 가능한 레지스트리
async fn use_from_multiple_threads() {
    let registry = Arc::new(AsyncHandlerRegistry::new());
    
    let mut handles = vec![];
    
    for i in 0..10 {
        let registry = Arc::clone(&registry);
        let handle = tokio::spawn(async move {
            let result = registry.handle("time", "").await;
            println!("Thread {}: {:?}", i, result);
        });
        handles.push(handle);
    }
    
    for handle in handles {
        handle.await.unwrap();
    }
}
```

## **실전 패턴: 빌더 패턴과 trait**

복잡한 핸들러를 만들 때는 빌더 패턴과 trait을 결합하면 유연성이 크게 향상된다.

```rust
struct HttpHandlerBuilder {
    timeout: Option<u64>,
    max_retries: Option<u32>,
    base_url: Option<String>,
}

impl HttpHandlerBuilder {
    fn new() -> Self {
        HttpHandlerBuilder {
            timeout: None,
            max_retries: None,
            base_url: None,
        }
    }
    
    fn timeout(mut self, seconds: u64) -> Self {
        self.timeout = Some(seconds);
        self
    }
    
    fn max_retries(mut self, retries: u32) -> Self {
        self.max_retries = Some(retries);
        self
    }
    
    fn base_url(mut self, url: String) -> Self {
        self.base_url = Some(url);
        self
    }
    
    fn build(self) -> HttpHandler {
        HttpHandler {
            client: reqwest::Client::new(),
            timeout: self.timeout.unwrap_or(30),
            max_retries: self.max_retries.unwrap_or(3),
            base_url: self.base_url.unwrap_or_else(|| "http://localhost".to_string()),
        }
    }
}

struct HttpHandler {
    client: reqwest::Client,
    timeout: u64,
    max_retries: u32,
    base_url: String,
}

#[async_trait]
impl AsyncMessageHandler for HttpHandler {
    async fn handle(&self, endpoint: &str) -> Result<String, String> {
        let url = format!("{}/{}", self.base_url, endpoint);
        
        for attempt in 0..self.max_retries {
            match tokio::time::timeout(
                tokio::time::Duration::from_secs(self.timeout),
                self.client.get(&url).send()
            ).await {
                Ok(Ok(response)) => {
                    return Ok(format!("Success: {}", response.status()));
                }
                Ok(Err(e)) => {
                    if attempt == self.max_retries - 1 {
                        return Err(format!("Failed after {} retries: {}", self.max_retries, e));
                    }
                }
                Err(_) => {
                    if attempt == self.max_retries - 1 {
                        return Err("Request timeout".to_string());
                    }
                }
            }
        }
        
        Err("Unexpected error".to_string())
    }
    
    fn name(&self) -> &str {
        "http"
    }
    
    fn description(&self) -> &str {
        "Makes HTTP requests to specified endpoints"
    }
}

// 사용 예
async fn setup_http_handler() {
    let handler = HttpHandlerBuilder::new()
        .timeout(10)
        .max_retries(5)
        .base_url("https://api.example.com".to_string())
        .build();
    
    let result = handler.handle("status").await;
    println!("{:?}", result);
}
```

## **정리와 실전 조언**

오늘 배운 내용을 요약하면 다음과 같다.

trait은 Rust에서 추상화를 구현하는 핵심 도구다. 제네릭과 trait bounds를 사용하면 컴파일 타임에 타입이 결정되어 성능이 뛰어나고, trait 객체를 사용하면 런타임에 유연하게 다양한 타입을 다룰 수 있다. 각 방식은 장단점이 있으므로 상황에 맞게 선택해야 한다.

비동기 환경에서 trait을 사용할 때는 `async_trait` 크레이트가 필수적이다. 또한 `Send`와 `Sync` trait을 이해하고 적절히 사용해야 스레드 안전성을 보장할 수 있다.

실전에서는 핸들러 레지스트리 패턴을 자주 사용한다. 이 패턴을 사용하면 런타임에 기능을 추가하거나 제거할 수 있어 플러그인 시스템을 구축하기 좋다. `Arc<RwLock<HashMap>>`을 사용하여 여러 스레드에서 안전하게 핸들러를 공유하고 관리할 수 있다.

빌더 패턴과 trait을 결합하면 복잡한 객체를 유연하게 생성할 수 있다. 이 방식은 가독성과 유지보수성을 크게 향상시킨다.

마지막으로, trait 메서드가 `self`, `&self`, `&mut self` 중 무엇을 받는지 신중하게 결정해야 한다. 이것은 소유권과 빌림 규칙에 직접적인 영향을 미치므로, API의 사용성과 안전성을 좌우하는 중요한 설계 결정이다.

내일은 이터레이터 trait을 깊이 있게 다루면서 데이터 처리 파이프라인을 효율적으로 구축하는 방법을 배운다.
