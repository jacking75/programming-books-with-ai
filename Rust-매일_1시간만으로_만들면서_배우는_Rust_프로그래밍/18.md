# Day 18: 채널로 메시지 전달하기

지금까지 우리는 Arc와 Mutex를 사용해서 여러 스레드가 같은 데이터를 안전하게 공유하는 방법을 배웠다. 하지만 공유 상태를 직접 다루는 것은 때로 복잡하고 데드락 같은 문제를 일으킬 수 있다. Rust는 또 다른 강력한 동시성 패턴을 제공하는데, 바로 **메시지 전달(message passing)**이다. "메모리를 공유해서 통신하지 말고, 통신해서 메모리를 공유하라"는 Go 언어의 유명한 격언처럼, 채널을 사용하면 데이터의 소유권을 스레드 간에 안전하게 이동시킬 수 있다.

오늘은 Rust의 채널(channel)을 사용해서 스레드 간 메시지를 전달하는 방법을 배우고, 실용적인 로깅 시스템을 구현해볼 것이다. 채널을 통해 소유권이 어떻게 이동하는지 직접 경험하면서, 빌림과 소유권의 규칙이 동시성 환경에서 어떻게 우리를 보호하는지 깊이 이해하게 될 것이다.

## 채널이란 무엇인가

채널은 한쪽 끝에서 데이터를 보내고 다른 쪽 끝에서 받는 일종의 파이프다. 실생활에서 우편함이나 메시지 큐를 생각하면 된다. Rust의 표준 라이브러리는 `std::sync::mpsc` 모듈을 통해 채널을 제공하는데, mpsc는 **multiple producer, single consumer**(다중 생산자, 단일 소비자)의 약자다. 여러 스레드가 메시지를 보낼 수 있지만, 받는 쪽은 하나만 존재한다는 의미다.

```
[스레드 1] ----\
                \
[스레드 2] -------> [채널] -------> [수신 스레드]
                /
[스레드 3] ----/
```

채널의 핵심 개념은 데이터의 소유권이 송신자에서 수신자로 **이동**한다는 것이다. 데이터를 채널에 보내면 더 이상 송신자는 그 데이터에 접근할 수 없다. 이것이 바로 Rust의 소유권 시스템이 데이터 경쟁을 컴파일 타임에 방지하는 방식이다.

## 첫 번째 채널 만들기

간단한 예제부터 시작해보자. 새 프로젝트를 만들고 기본적인 채널 사용법을 살펴본다.

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    // 채널 생성: (송신자, 수신자) 튜플 반환
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        let message = String::from("안녕하세요");
        println!("메시지 전송: {}", message);
        tx.send(message).unwrap();
        // 여기서 message를 다시 사용하려고 하면 컴파일 에러!
        // println!("{}", message); // 에러: value borrowed after move
    });

    // 메시지 수신 (블로킹)
    let received = rx.recv().unwrap();
    println!("메시지 수신: {}", received);
}
```

이 코드에서 중요한 점들을 살펴보자. `mpsc::channel()`은 송신자(`tx`, transmitter)와 수신자(`rx`, receiver)의 쌍을 반환한다. 스레드를 생성할 때 `move` 클로저를 사용해서 `tx`의 소유권을 스레드로 이동시킨다. 그리고 `tx.send(message)`를 호출하면 `message`의 소유권이 채널로 이동한다. 이후 원래 스레드에서는 `message`를 사용할 수 없다. 컴파일러가 이를 강제하기 때문에 실수로 같은 데이터를 동시에 접근하는 일이 생길 수 없다.

수신 쪽에서는 `rx.recv()`를 호출해서 메시지를 기다린다. 이 함수는 메시지가 도착할 때까지 블로킹된다. 메시지가 도착하면 `Result<T, RecvError>`를 반환하는데, 송신자가 모두 드롭되면 에러가 발생한다.

## 여러 메시지 주고받기

실제 프로그램에서는 한 번이 아니라 여러 메시지를 주고받는다. 송신자가 여러 메시지를 보내고 수신자가 순서대로 받는 예제를 보자.

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        let messages = vec![
            "첫 번째 메시지",
            "두 번째 메시지",
            "세 번째 메시지",
        ];

        for msg in messages {
            tx.send(msg.to_string()).unwrap();
            thread::sleep(Duration::from_millis(500));
        }
    });

    // 채널이 닫힐 때까지 계속 수신
    for received in rx {
        println!("받음: {}", received);
    }
}
```

여기서 `rx`를 이터레이터로 사용할 수 있다는 점이 흥미롭다. `for received in rx` 구문은 송신자가 모두 드롭될 때까지 계속 메시지를 받는다. 각 메시지는 순서대로 도착하며, 송신자가 sleep하는 동안 수신자는 블로킹된다.

## 다중 생산자 패턴

mpsc 채널의 "mp" 부분, 즉 다중 생산자 기능을 활용해보자. 여러 스레드가 같은 채널로 메시지를 보낼 수 있다. 송신자를 복제하려면 `clone()` 메서드를 사용한다.

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    let (tx, rx) = mpsc::channel();

    // 첫 번째 송신자
    let tx1 = tx.clone();
    thread::spawn(move || {
        for i in 1..=3 {
            tx1.send(format!("스레드 1: 메시지 {}", i)).unwrap();
            thread::sleep(Duration::from_millis(300));
        }
    });

    // 두 번째 송신자
    let tx2 = tx.clone();
    thread::spawn(move || {
        for i in 1..=3 {
            tx2.send(format!("스레드 2: 메시지 {}", i)).unwrap();
            thread::sleep(Duration::from_millis(500));
        }
    });

    // 원래 tx도 사용 가능
    thread::spawn(move || {
        tx.send(String::from("메인 송신자: 완료")).unwrap();
    });

    // 모든 송신자가 드롭될 때까지 수신
    for received in rx {
        println!("{}", received);
    }
}
```

이 예제에서는 세 개의 스레드가 각자 자신의 송신자를 가지고 메시지를 보낸다. 메시지들은 각 스레드의 타이밍에 따라 뒤섞여서 도착한다. 모든 송신자(`tx`, `tx1`, `tx2`)가 드롭되어야 수신자의 이터레이션이 종료된다. 만약 하나라도 살아있으면 `rx`는 계속 기다린다.

## 실전: 로깅 시스템 구축하기

이제 배운 내용을 실용적인 예제에 적용해보자. 멀티스레드 TCP 서버에서 여러 스레드가 동시에 로그를 남기려고 한다면 어떻게 해야 할까? 각 스레드가 파일에 직접 쓰면 동기화 문제가 생긴다. 채널을 사용하면 모든 로그 메시지를 하나의 전담 스레드로 모아서 순차적으로 처리할 수 있다.

먼저 로그 메시지를 표현할 구조체를 정의한다.

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;
use std::fmt;

#[derive(Debug, Clone)]
enum LogLevel {
    Info,
    Warning,
    Error,
}

impl fmt::Display for LogLevel {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        match self {
            LogLevel::Info => write!(f, "INFO"),
            LogLevel::Warning => write!(f, "WARN"),
            LogLevel::Error => write!(f, "ERROR"),
        }
    }
}

struct LogMessage {
    level: LogLevel,
    thread_id: String,
    content: String,
}

impl LogMessage {
    fn new(level: LogLevel, thread_id: String, content: String) -> Self {
        LogMessage {
            level,
            thread_id,
            content,
        }
    }
}
```

`LogMessage` 구조체는 로그 레벨, 스레드 ID, 실제 메시지 내용을 담는다. 이제 로거 스레드를 만들어서 메시지를 받아 처리하도록 한다.

```rust
fn logger_thread(rx: mpsc::Receiver<LogMessage>) {
    for log in rx {
        println!("[{}] [{}] {}", log.level, log.thread_id, log.content);
        // 실제로는 여기서 파일에 쓰거나 외부 시스템으로 전송할 수 있다
    }
    println!("로거 종료: 모든 송신자 드롭됨");
}
```

이 함수는 채널에서 로그 메시지를 받아서 출력한다. 실제 시스템에서는 파일에 쓰거나, 로그 수집 시스템으로 전송하거나, 버퍼링해서 배치로 처리할 수 있다. 중요한 점은 **모든 로그 작업이 단일 스레드에서 순차적으로 일어난다**는 것이다. 따라서 동기화 문제나 데이터 경쟁이 발생하지 않는다.

이제 전체 시스템을 구성해보자.

```rust
fn main() {
    let (tx, rx) = mpsc::channel();

    // 로거 스레드 시작
    let logger_handle = thread::spawn(move || {
        logger_thread(rx);
    });

    // 여러 작업 스레드 시뮬레이션
    let mut handles = vec![];

    for i in 1..=3 {
        let thread_tx = tx.clone();
        let handle = thread::spawn(move || {
            let thread_id = format!("Worker-{}", i);
            
            thread_tx.send(LogMessage::new(
                LogLevel::Info,
                thread_id.clone(),
                String::from("작업 시작"),
            )).unwrap();

            // 작업 시뮬레이션
            thread::sleep(Duration::from_millis(100 * i as u64));

            if i == 2 {
                thread_tx.send(LogMessage::new(
                    LogLevel::Warning,
                    thread_id.clone(),
                    String::from("경고: 재시도 필요"),
                )).unwrap();
            }

            thread_tx.send(LogMessage::new(
                LogLevel::Info,
                thread_id,
                String::from("작업 완료"),
            )).unwrap();
        });
        handles.push(handle);
    }

    // 원래 tx를 드롭해야 로거가 종료될 수 있다
    drop(tx);

    // 모든 작업 스레드 종료 대기
    for handle in handles {
        handle.join().unwrap();
    }

    // 로거 스레드 종료 대기
    logger_handle.join().unwrap();
}
```

이 코드에서 핵심적인 부분을 자세히 살펴보자. 먼저 메인 채널 `(tx, rx)`를 생성하고, `rx`는 로거 스레드로 이동시킨다. 그 다음 여러 작업 스레드를 생성하는데, 각각 `tx.clone()`으로 자신만의 송신자를 받는다. 각 스레드는 작업 중에 로그 메시지를 자유롭게 보낼 수 있다.

특히 중요한 부분은 `drop(tx)` 호출이다. 우리가 작업 스레드에 `tx`를 클론해서 넘겨줬지만, 메인 스레드의 원래 `tx`도 여전히 살아있다. 만약 이것을 명시적으로 드롭하지 않으면, 모든 작업 스레드가 끝나더라도 로거 스레드는 계속 메시지를 기다린다. 메인 스레드의 `tx`가 아직 살아있기 때문이다. `drop(tx)`를 호출함으로써 "더 이상 메시지를 보내지 않겠다"고 명시적으로 알려준다.

## 소유권과 채널의 상호작용

채널을 사용할 때 소유권 규칙이 어떻게 작동하는지 더 자세히 이해해보자. 다음 코드를 보자.

```rust
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        let data = vec![1, 2, 3, 4, 5];
        tx.send(data).unwrap();
        // data는 이제 사용할 수 없다
        // println!("{:?}", data); // 컴파일 에러!
    });

    let received_data = rx.recv().unwrap();
    println!("받은 데이터: {:?}", received_data);
    // 이제 received_data의 소유권은 이 스레드에 있다
}
```

`send()`를 호출하면 데이터의 소유권이 채널로 이동한다. 송신 스레드는 더 이상 그 데이터에 접근할 수 없다. 수신 스레드가 `recv()`로 데이터를 받으면, 소유권이 수신 스레드로 이동한다. 이것이 바로 "통신으로 메모리를 공유하는" 방식이다. 데이터는 한 순간에 한 스레드만 소유한다.

만약 데이터를 복사하고 싶다면 명시적으로 해야 한다.

```rust
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        let data = vec![1, 2, 3, 4, 5];
        tx.send(data.clone()).unwrap(); // 복사본을 보낸다
        println!("원본 데이터 사용 가능: {:?}", data);
    });

    let received_data = rx.recv().unwrap();
    println!("받은 데이터: {:?}", received_data);
}
```

하지만 대부분의 경우 불필요한 복사는 피하는 것이 좋다. 채널의 장점은 소유권 이동을 통해 복사 없이 데이터를 전달할 수 있다는 것이다.

## 채널과 빌림의 상호작용

채널을 통해 참조를 보낼 수 있을까? 이것은 흥미로운 질문이다. 다음 코드를 시도해보자.

```rust
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    let data = vec![1, 2, 3];

    thread::spawn(move || {
        // tx.send(&data).unwrap(); // 컴파일 에러!
    });

    // let received = rx.recv().unwrap();
}
```

이 코드는 컴파일되지 않는다. 왜 그럴까? 참조는 빌린 값인데, 빌림은 라이프타임을 가진다. 스레드는 임의의 시간 동안 실행될 수 있고, 원본 데이터보다 오래 살아있을 수 있다. Rust 컴파일러는 이런 댕글링 참조를 허용하지 않는다.

채널은 기본적으로 `Send` trait을 구현한 타입만 전송할 수 있다. `Send`는 스레드 간에 안전하게 전송할 수 있는 타입을 표시하는 마커 trait이다. 대부분의 소유된(owned) 타입은 `Send`를 구현하지만, 참조나 `Rc<T>` 같은 타입은 그렇지 않다.

만약 참조를 공유하고 싶다면 `Arc`를 사용해야 한다.

```rust
use std::sync::{mpsc, Arc};
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    let data = Arc::new(vec![1, 2, 3, 4, 5]);
    let data_clone = Arc::clone(&data);

    thread::spawn(move || {
        tx.send(data_clone).unwrap(); // Arc는 Send를 구현한다
    });

    let received = rx.recv().unwrap();
    println!("받은 데이터: {:?}", received);
    println!("원본 데이터: {:?}", data);
}
```

`Arc`는 `Send`를 구현하므로 채널을 통해 전송할 수 있다. 수신자는 `Arc`의 복제본을 받아서 데이터에 접근할 수 있고, 원본 스레드도 여전히 자신의 `Arc`를 통해 접근할 수 있다.

## 논블로킹 수신과 타임아웃

지금까지 본 `recv()`는 블로킹 호출이다. 메시지가 도착할 때까지 무한정 기다린다. 때로는 바로 확인만 하거나, 일정 시간만 기다리고 싶을 수 있다.

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        thread::sleep(Duration::from_secs(2));
        tx.send("늦은 메시지").unwrap();
    });

    // try_recv: 즉시 반환
    match rx.try_recv() {
        Ok(msg) => println!("받음: {}", msg),
        Err(_) => println!("아직 메시지 없음"),
    }

    // recv_timeout: 일정 시간 대기
    match rx.recv_timeout(Duration::from_secs(1)) {
        Ok(msg) => println!("받음: {}", msg),
        Err(_) => println!("1초 안에 메시지 안 옴"),
    }

    // 충분히 기다리면 받을 수 있다
    thread::sleep(Duration::from_secs(2));
    match rx.try_recv() {
        Ok(msg) => println!("받음: {}", msg),
        Err(_) => println!("여전히 없음"),
    }
}
```

`try_recv()`는 채널을 즉시 확인하고 메시지가 있으면 반환하고, 없으면 에러를 반환한다. `recv_timeout()`은 지정된 시간 동안만 기다린다. 이런 메서드들은 다른 작업과 함께 주기적으로 채널을 확인해야 할 때 유용하다.

## TCP 서버에 로깅 시스템 통합하기

이전 Day에서 만든 멀티스레드 TCP 서버에 우리의 로깅 시스템을 통합해보자. 전체 구조는 다음과 같다.

```
┌─────────────┐
│   메인      │
│  스레드     │
└──────┬──────┘
       │ 채널 생성
       ├─────────────┐
       │             │
       ▼             ▼
┌─────────────┐  ┌─────────────┐
│   로거      │  │  리스너     │
│  스레드     │  │  루프       │
└─────────────┘  └──────┬──────┘
       ▲                │ 연결마다
       │                │ 스레드 생성
       │                ▼
       │         ┌─────────────┐
       └─────────│  클라이언트 │
                 │  핸들러     │
                 │  스레드들   │
                 └─────────────┘
```

다음은 구현 코드다.

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;
use std::io::{Read, Write};
use std::net::{TcpListener, TcpStream};

// 이전에 정의한 LogLevel과 LogMessage 구조체 사용

fn handle_client(mut stream: TcpStream, client_id: usize, log_tx: mpsc::Sender<LogMessage>) {
    let thread_id = format!("Client-{}", client_id);
    
    log_tx.send(LogMessage::new(
        LogLevel::Info,
        thread_id.clone(),
        format!("연결됨: {}", stream.peer_addr().unwrap()),
    )).unwrap();

    let mut buffer = [0; 512];
    loop {
        match stream.read(&mut buffer) {
            Ok(0) => {
                log_tx.send(LogMessage::new(
                    LogLevel::Info,
                    thread_id.clone(),
                    String::from("연결 종료"),
                )).unwrap();
                break;
            }
            Ok(n) => {
                log_tx.send(LogMessage::new(
                    LogLevel::Info,
                    thread_id.clone(),
                    format!("{}바이트 수신", n),
                )).unwrap();

                if stream.write_all(&buffer[..n]).is_err() {
                    log_tx.send(LogMessage::new(
                        LogLevel::Error,
                        thread_id.clone(),
                        String::from("쓰기 실패"),
                    )).unwrap();
                    break;
                }
            }
            Err(e) => {
                log_tx.send(LogMessage::new(
                    LogLevel::Error,
                    thread_id.clone(),
                    format!("읽기 에러: {}", e),
                )).unwrap();
                break;
            }
        }
    }
}

fn run_server() {
    let (log_tx, log_rx) = mpsc::channel();

    // 로거 스레드
    let logger_handle = thread::spawn(move || {
        logger_thread(log_rx);
    });

    let listener = TcpListener::bind("127.0.0.1:8080").unwrap();
    
    log_tx.send(LogMessage::new(
        LogLevel::Info,
        String::from("Main"),
        String::from("서버 시작: 127.0.0.1:8080"),
    )).unwrap();

    let mut client_id = 0;
    for stream in listener.incoming() {
        match stream {
            Ok(stream) => {
                client_id += 1;
                let thread_tx = log_tx.clone();
                thread::spawn(move || {
                    handle_client(stream, client_id, thread_tx);
                });
            }
            Err(e) => {
                log_tx.send(LogMessage::new(
                    LogLevel::Error,
                    String::from("Main"),
                    format!("연결 에러: {}", e),
                )).unwrap();
            }
        }
    }

    drop(log_tx);
    logger_handle.join().unwrap();
}
```

이 구현에서 각 클라이언트 핸들러 스레드는 자신의 `log_tx` 복제본을 받는다. 클라이언트와의 상호작용 중 발생하는 모든 이벤트를 로거 채널로 전송한다. 로거 스레드는 모든 메시지를 순서대로 받아서 처리한다. 이렇게 하면 여러 스레드의 로그가 뒤섞이지 않고 깔끔하게 기록된다.

## 채널의 용량과 성능

지금까지 사용한 `mpsc::channel()`은 무한 버퍼를 가진 비동기 채널이다. 송신자는 `send()`를 호출하면 즉시 반환되고, 메시지는 큐에 쌓인다. 하지만 이것은 메모리를 무한정 사용할 수 있다는 뜻이다. 만약 수신자가 처리하는 속도보다 송신자가 보내는 속도가 빠르면 큐가 계속 커진다.

Rust는 동기(synchronous) 채널도 제공한다.

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    // 용량이 0인 동기 채널
    let (tx, rx) = mpsc::sync_channel(0);

    thread::spawn(move || {
        println!("메시지 전송 시작");
        tx.send("첫 메시지").unwrap(); // 수신자가 받을 때까지 블로킹
        println!("메시지 전송 완료");
    });

    thread::sleep(Duration::from_secs(2));
    println!("메시지 수신");
    let msg = rx.recv().unwrap();
    println!("받음: {}", msg);
}
```

`sync_channel(0)`은 용량이 0인 채널을 만든다. 송신자가 `send()`를 호출하면 수신자가 `recv()`를 호출할 때까지 블로킹된다. 양쪽이 준비될 때까지 기다리는 랑데부(rendezvous) 방식이다. 용량을 늘리면 그만큼 버퍼링이 가능하다.

```rust
let (tx, rx) = mpsc::sync_channel(5); // 최대 5개 버퍼링
```

송신자는 큐에 5개의 메시지가 쌓일 때까지는 블로킹되지 않지만, 6번째 메시지부터는 수신자가 하나를 가져갈 때까지 기다린다. 이런 백프레셔(backpressure) 메커니즘은 시스템이 과부하되는 것을 방지한다.

## 채널 닫기와 종료 시그널

채널을 사용할 때 언제 작업이 끝났는지 알리는 것이 중요하다. 일반적인 패턴은 특별한 "종료" 메시지를 보내는 것이다.

```rust
use std::sync::mpsc;
use std::thread;

enum WorkerMessage {
    Task(String),
    Shutdown,
}

fn main() {
    let (tx, rx) = mpsc::channel();

    let worker = thread::spawn(move || {
        loop {
            match rx.recv() {
                Ok(WorkerMessage::Task(task)) => {
                    println!("작업 처리: {}", task);
                }
                Ok(WorkerMessage::Shutdown) => {
                    println!("종료 신호 받음");
                    break;
                }
                Err(_) => {
                    println!("채널 닫힘");
                    break;
                }
            }
        }
    });

    tx.send(WorkerMessage::Task(String::from("작업 1"))).unwrap();
    tx.send(WorkerMessage::Task(String::from("작업 2"))).unwrap();
    tx.send(WorkerMessage::Shutdown).unwrap();

    worker.join().unwrap();
}
```

메시지 타입을 열거형으로 정의하고 `Shutdown` 배리언트를 추가했다. 워커 스레드는 이 메시지를 받으면 루프를 종료한다. 이 패턴은 우아한 종료(graceful shutdown)를 구현할 때 유용하다.

## 채널 vs Arc<Mutex<T>>

그렇다면 언제 채널을 사용하고 언제 `Arc<Mutex<T>>`를 사용해야 할까? 일반적인 가이드라인은 다음과 같다.

채널을 사용하는 것이 좋은 경우는 데이터가 한 방향으로 흐를 때다. 생산자가 만들고 소비자가 처리하는 파이프라인 구조에 적합하다. 소유권을 명확히 이전하고 싶을 때, 그리고 작업 큐나 이벤트 시스템을 구현할 때 좋다.

`Arc<Mutex<T>>`를 사용하는 것이 좋은 경우는 여러 스레드가 같은 데이터를 읽고 쓸 때다. 데이터가 특정 위치에 머물러야 할 때, 그리고 복잡한 상태를 공유해야 할 때 적합하다.

많은 경우 두 가지를 조합해서 사용한다. 예를 들어, 공유 상태는 `Arc<Mutex<T>>`로 관리하고, 이벤트 알림은 채널로 전달하는 식이다.

```rust
use std::sync::{Arc, Mutex, mpsc};
use std::thread;

struct SharedState {
    counter: Mutex<u32>,
}

enum Event {
    Increment,
    GetValue,
}

fn main() {
    let state = Arc::new(SharedState {
        counter: Mutex::new(0),
    });

    let (tx, rx) = mpsc::channel();

    // 이벤트 처리 스레드
    let state_clone = Arc::clone(&state);
    let event_handler = thread::spawn(move || {
        for event in rx {
            match event {
                Event::Increment => {
                    let mut count = state_clone.counter.lock().unwrap();
                    *count += 1;
                    println!("카운터 증가: {}", *count);
                }
                Event::GetValue => {
                    let count = state_clone.counter.lock().unwrap();
                    println!("현재 값: {}", *count);
                }
            }
        }
    });

    // 여러 스레드가 이벤트 전송
    let mut handles = vec![];
    for _ in 0..3 {
        let thread_tx = tx.clone();
        let handle = thread::spawn(move || {
            thread_tx.send(Event::Increment).unwrap();
        });
        handles.push(handle);
    }

    drop(tx);
    for handle in handles {
        handle.join().unwrap();
    }

    event_handler.join().unwrap();
}
```

이 패턴에서 공유 상태는 `Arc<Mutex<T>>`로 관리되지만, 상태를 변경하는 명령은 채널을 통해 전달된다. 단일 스레드가 모든 변경을 처리하므로 락 경쟁이 줄어들고 코드가 더 명확해진다.

## 정리

오늘은 Rust의 채널을 통한 메시지 전달 방식을 깊이 있게 배웠다. 채널은 소유권 시스템과 완벽하게 통합되어 있어서 컴파일 타임에 동시성 안전성을 보장한다. 데이터를 채널로 보내면 소유권이 이동하므로 데이터 경쟁이 원천적으로 불가능하다.

`mpsc::channel()`을 사용해서 다중 생산자, 단일 소비자 패턴을 구현했고, 이를 활용해서 실용적인 로깅 시스템을 만들었다. 여러 스레드에서 발생하는 로그를 하나의 전담 스레드로 모아서 처리함으로써 동기화 문제 없이 깔끔한 로그를 남길 수 있었다.

채널을 통해 참조를 보낼 수 없다는 제약은 처음에는 불편해 보일 수 있지만, 실제로는 댕글링 참조를 방지하는 안전장치다. 데이터를 공유하려면 소유권을 이전하거나 `Arc`를 사용해야 한다. 이런 명시적인 선택이 코드의 의도를 분명히 하고 버그를 줄여준다.

논블로킹 수신(`try_recv`)과 타임아웃(`recv_timeout`), 동기 채널(`sync_channel`) 같은 다양한 옵션들도 배웠다. 각각의 상황에 맞는 도구를 선택할 수 있다.

채널과 `Arc<Mutex<T>>`는 상호 배타적이지 않다. 많은 실전 코드에서 두 가지를 적절히 조합해서 사용한다. 채널은 데이터 흐름을 명확히 하고, `Arc<Mutex<T>>`는 필요할 때만 공유 상태를 관리하는데 쓴다.

내일은 이제까지 배운 모든 동시성 기법을 종합해서 브로드캐스트 기능을 가진 채팅 서버를 만들어볼 것이다. 한 클라이언트의 메시지를 모든 클라이언트에게 전달하려면 `Arc<Mutex<Vec<Sender>>>`와 채널을 함께 사용해야 한다. 또한 빌림의 범위를 최소화해서 데드락을 방지하는 기법도 배울 것이다.
