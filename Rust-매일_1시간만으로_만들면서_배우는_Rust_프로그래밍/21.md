# Day 21: async/await 기초

지난 20일 동안 우리는 스레드 기반 동시성으로 여러 클라이언트를 동시에 처리하는 TCP 서버를 만들었다. 각 클라이언트마다 스레드를 생성하는 방식은 직관적이고 이해하기 쉽지만, 클라이언트가 수천 개로 늘어나면 문제가 생긴다. 각 스레드는 메모리를 소비하고(보통 수 MB), 컨텍스트 스위칭 비용도 만만치 않다.

오늘은 이 문제를 해결하는 비동기 프로그래밍을 배운다. Rust의 `async/await`를 사용하면 수천, 수만 개의 동시 연결을 적은 수의 스레드로 처리할 수 있다. 하지만 먼저 왜 비동기가 필요한지, 그리고 어떻게 작동하는지 이해해야 한다.

## 동기 vs 비동기: 커피숍 비유

동기 프로그래밍은 커피숍에서 직원이 한 명의 주문을 완전히 처리할 때까지 다음 손님을 받지 않는 것과 같다. 손님 A의 커피를 만들고, 완성될 때까지 기다리고, 전달한 후에야 손님 B를 받는다.

```
직원 → [손님A 주문받기] → [커피 만들기...] → [전달] → [손님B 주문받기] → ...
        ↑________________기다림________________↑
```

비동기 프로그래밍은 커피를 만드는 동안 다른 손님의 주문을 받는 것과 같다. 손님 A의 주문을 받고 커피 머신을 작동시킨 후, 커피가 나오길 기다리는 동안 손님 B, C, D의 주문을 받는다.

```
직원 → [A주문] → [머신작동] → [B주문] → [C주문] → [A커피완성] → [A전달] → ...
```

네트워크 프로그래밍에서는 대부분의 시간을 데이터를 기다리는 데 소비한다. 클라이언트로부터 메시지가 도착하길 기다리거나, 데이터베이스 쿼리 결과를 기다리거나, 파일 I/O가 완료되길 기다린다. 이 "기다림" 동안 CPU는 거의 놀고 있다. 비동기 프로그래밍은 이 기다림 시간에 다른 작업을 처리한다.

## Rust의 async/await 모델

Rust의 비동기는 다른 언어와 조금 다르다. Rust는 런타임을 언어 자체에 포함하지 않는다. 대신 `async/await` 문법만 제공하고, 실제 실행은 외부 런타임 라이브러리가 담당한다. 가장 인기 있는 런타임은 **tokio**다.

먼저 프로젝트에 tokio를 추가한다:

```bash
cargo new async_basics
cd async_basics
cargo add tokio --features full
```

## 첫 비동기 함수

간단한 비동기 함수부터 시작한다:

```rust
use tokio::time::{sleep, Duration};

async fn make_coffee() -> String {
    println!("커피 만들기 시작...");
    sleep(Duration::from_secs(2)).await;
    println!("커피 완성!");
    String::from("☕ 아메리카노")
}

#[tokio::main]
async fn main() {
    let coffee = make_coffee().await;
    println!("받은 음료: {}", coffee);
}
```

여기서 중요한 키워드들을 살펴본다:

**`async fn`**: 이 함수는 비동기 함수다. 호출하면 즉시 실행되지 않고, `Future`라는 타입을 반환한다. Future는 "나중에 완료될 작업"을 나타낸다.

**`.await`**: 이 지점에서 Future가 완료될 때까지 기다린다. 하지만 스레드를 블록하지 않는다! 런타임은 이 작업이 기다리는 동안 다른 작업을 실행할 수 있다.

**`#[tokio::main]`**: 이 매크로는 비동기 런타임을 설정하고 `main` 함수를 실행한다. 비동기 함수는 런타임 안에서만 실행될 수 있다.

실행하면 다음과 같이 출력된다:

```
커피 만들기 시작...
(2초 대기)
커피 완성!
받은 음료: ☕ 아메리카노
```

## 동시에 여러 작업 실행하기

비동기의 진짜 힘은 여러 작업을 동시에 처리할 때 나타난다. 세 잔의 커피를 만들어본다:

```rust
use tokio::time::{sleep, Duration};

async fn make_coffee(name: &str, seconds: u64) -> String {
    println!("[{}] 커피 만들기 시작", name);
    sleep(Duration::from_secs(seconds)).await;
    println!("[{}] 완성!", name);
    format!("☕ {} 커피", name)
}

#[tokio::main]
async fn main() {
    // 방법 1: 순차적 실행 (비효율적)
    println!("=== 순차적 실행 ===");
    let start = std::time::Instant::now();
    
    let coffee1 = make_coffee("아메리카노", 2).await;
    let coffee2 = make_coffee("라떼", 3).await;
    let coffee3 = make_coffee("카푸치노", 2).await;
    
    println!("모두 완성: {}, {}, {}", coffee1, coffee2, coffee3);
    println!("총 소요 시간: {:?}\n", start.elapsed());
}
```

이 코드는 2 + 3 + 2 = 7초가 걸린다. 각 커피를 차례대로 만들기 때문이다. 이제 동시에 만들어본다:

```rust
#[tokio::main]
async fn main() {
    // 방법 2: 동시 실행 (효율적)
    println!("=== 동시 실행 ===");
    let start = std::time::Instant::now();
    
    // 세 작업을 동시에 시작
    let task1 = make_coffee("아메리카노", 2);
    let task2 = make_coffee("라떼", 3);
    let task3 = make_coffee("카푸치노", 2);
    
    // 모두 완료될 때까지 기다림
    let (coffee1, coffee2, coffee3) = tokio::join!(task1, task2, task3);
    
    println!("모두 완성: {}, {}, {}", coffee1, coffee2, coffee3);
    println!("총 소요 시간: {:?}", start.elapsed());
}
```

출력:

```
=== 동시 실행 ===
[아메리카노] 커피 만들기 시작
[라떼] 커피 만들기 시작
[카푸치노] 커피 만들기 시작
[아메리카노] 완성!
[카푸치노] 완성!
[라떼] 완성!
모두 완성: ☕ 아메리카노 커피, ☕ 라떼 커피, ☕ 카푸치노 커피
총 소요 시간: 3.00s
```

이제 3초만 걸린다! 세 커피를 동시에 만들었기 때문이다. `tokio::join!` 매크로는 여러 Future를 동시에 실행하고, 모두 완료될 때까지 기다린다.

## 비동기 작업의 실행 흐름

비동기 코드가 어떻게 실행되는지 다이어그램으로 이해해본다:

```
시간 →

동기 실행:
스레드1: [Task A ─────] [Task B ─────] [Task C ───]
         0s     2s      3s      6s     7s    9s

비동기 실행:
태스크A: [시작]...(기다림)...[완료]
태스크B: [시작].......(기다림).......[완료]
태스크C: [시작]...(기다림)...[완료]

스레드1: [A시작B시작C시작]─[A완료C완료]─[B완료]
         0s              2s          3s
```

비동기에서는 한 태스크가 I/O를 기다리는 동안, 런타임이 다른 태스크로 전환한다. 이 전환은 매우 가볍고 빠르다. 스레드 컨텍스트 스위칭과 달리 OS의 개입이 필요 없다.

## 실전 예제: 여러 URL 동시에 가져오기

실용적인 예제로 여러 웹 페이지를 동시에 다운로드해본다. 먼저 의존성을 추가한다:

```bash
cargo add reqwest --features json
```

```rust
use reqwest;
use tokio::time::Instant;

async fn fetch_url(url: &str) -> Result<usize, reqwest::Error> {
    println!("[{}] 요청 시작", url);
    let response = reqwest::get(url).await?;
    let body = response.text().await?;
    println!("[{}] 완료: {} bytes", url, body.len());
    Ok(body.len())
}

#[tokio::main]
async fn main() {
    let urls = vec![
        "https://www.rust-lang.org",
        "https://docs.rs",
        "https://crates.io",
    ];
    
    let start = Instant::now();
    
    // 모든 URL을 동시에 요청
    let tasks: Vec<_> = urls.iter()
        .map(|url| fetch_url(url))
        .collect();
    
    // 모든 결과를 기다림
    let results = futures::future::join_all(tasks).await;
    
    println!("\n=== 결과 ===");
    for (url, result) in urls.iter().zip(results.iter()) {
        match result {
            Ok(size) => println!("{}: {} bytes", url, size),
            Err(e) => println!("{}: 실패 - {}", url, e),
        }
    }
    
    println!("\n총 소요 시간: {:?}", start.elapsed());
}
```

이 코드는 세 개의 웹사이트를 동시에 요청한다. 순차적으로 실행했다면 각 요청의 시간을 모두 더한 만큼 걸렸겠지만, 비동기로 실행하면 가장 느린 요청만큼만 걸린다.

## async 함수의 타입: Future

`async fn`이 반환하는 것은 실제로는 `Future` trait을 구현한 타입이다. Future는 다음과 같이 정의된다:

```rust
trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context) -> Poll<Self::Output>;
}

enum Poll<T> {
    Ready(T),      // 작업 완료
    Pending,       // 아직 완료 안 됨
}
```

`async fn`을 작성하면 컴파일러가 자동으로 이 Future를 구현한 상태 머신을 생성한다. `.await`를 만나면 런타임은 `poll()`을 호출해서 작업이 완료됐는지 확인한다. `Pending`이면 다른 작업으로 전환하고, `Ready`면 결과를 반환한다.

우리는 보통 이 세부사항을 신경 쓸 필요 없다. `async/await` 문법이 모든 복잡함을 숨겨준다. 하지만 Future가 `.await` 없이는 실행되지 않는다는 것을 기억해야 한다:

```rust
async fn do_something() {
    println!("실행됨!");
}

#[tokio::main]
async fn main() {
    // 이것은 아무것도 출력하지 않는다!
    do_something();  // Future만 생성되고 실행되지 않음
    
    // .await를 붙여야 실행된다
    do_something().await;  // "실행됨!" 출력
}
```

## 비동기 블록과 클로저

함수 전체가 아니라 일부만 비동기로 만들고 싶을 때는 `async` 블록을 사용한다:

```rust
#[tokio::main]
async fn main() {
    let name = String::from("고객A");
    
    // async 블록은 Future를 반환
    let future = async {
        println!("{} 주문 처리 중...", name);
        tokio::time::sleep(Duration::from_secs(1)).await;
        format!("{}님의 주문 완료", name)
    };
    
    let result = future.await;
    println!("{}", result);
}
```

`async` 블록은 주변 변수를 캡처할 수 있다. 여기서는 `name`을 캡처했다. 소유권 규칙은 똑같이 적용된다:

```rust
#[tokio::main]
async fn main() {
    let name = String::from("고객A");
    
    // name의 소유권이 async 블록으로 이동
    let future = async move {
        println!("{} 주문 처리 중...", name);
        tokio::time::sleep(Duration::from_secs(1)).await;
        name  // 소유권이 여기서 반환됨
    };
    
    // 여기서 name을 사용하려고 하면 컴파일 에러!
    // println!("{}", name);  // 에러: value borrowed after move
    
    let returned_name = future.await;
    println!("{}님 주문 완료", returned_name);
}
```

## tokio::spawn으로 백그라운드 태스크 생성

`tokio::join!`은 모든 작업이 완료될 때까지 기다린다. 하지만 때로는 작업을 백그라운드에서 독립적으로 실행하고 싶을 때가 있다. `tokio::spawn`을 사용한다:

```rust
use tokio::time::{sleep, Duration};

async fn background_task(id: u32) {
    for i in 1..=3 {
        println!("[태스크 {}] 작업 중... {}/3", id, i);
        sleep(Duration::from_secs(1)).await;
    }
    println!("[태스크 {}] 완료!", id);
}

#[tokio::main]
async fn main() {
    println!("메인 시작");
    
    // 두 개의 백그라운드 태스크 생성
    let handle1 = tokio::spawn(background_task(1));
    let handle2 = tokio::spawn(background_task(2));
    
    println!("태스크들 생성 완료, 다른 작업 수행 가능");
    sleep(Duration::from_millis(1500)).await;
    println!("메인에서 중간 작업 완료");
    
    // 태스크들이 완료되길 기다림
    handle1.await.unwrap();
    handle2.await.unwrap();
    
    println!("메인 종료");
}
```

출력:

```
메인 시작
태스크들 생성 완료, 다른 작업 수행 가능
[태스크 1] 작업 중... 1/3
[태스크 2] 작업 중... 1/3
[태스크 1] 작업 중... 2/3
[태스크 2] 작업 중... 2/3
메인에서 중간 작업 완료
[태스크 1] 작업 중... 3/3
[태스크 2] 작업 중... 3/3
[태스크 1] 완료!
[태스크 2] 완료!
메인 종료
```

`tokio::spawn`은 `JoinHandle`을 반환한다. 이것은 스레드의 `JoinHandle`과 비슷하지만, 실제로는 새 스레드를 생성하지 않는다. tokio 런타임의 스레드 풀에서 실행될 뿐이다.

## 비동기와 소유권: 'static 요구사항

`tokio::spawn`에 전달하는 Future는 `'static` 라이프타임을 가져야 한다. 왜냐하면 spawn된 태스크는 언제 끝날지 모르고, 부모 스코프보다 오래 살 수 있기 때문이다:

```rust
#[tokio::main]
async fn main() {
    let message = String::from("안녕");
    
    // 컴파일 에러!
    // tokio::spawn(async {
    //     println!("{}", message);  // message는 main의 소유
    // });
    
    // 해결 방법 1: move로 소유권 이동
    tokio::spawn(async move {
        println!("{}", message);  // 이제 소유권이 이동됨
    });
    
    // 해결 방법 2: Arc로 공유
    let message = std::sync::Arc::new(String::from("안녕"));
    let message_clone = message.clone();
    
    tokio::spawn(async move {
        println!("{}", message_clone);
    });
    
    println!("원본 메시지: {}", message);
}
```

이 패턴은 내일 배울 tokio TCP 서버에서 자주 사용된다.

## 비동기 함수에서 에러 처리

비동기 함수도 `Result`를 반환할 수 있다. `?` 연산자도 똑같이 작동한다:

```rust
use tokio::fs::File;
use tokio::io::{self, AsyncReadExt};

async fn read_file(path: &str) -> io::Result<String> {
    let mut file = File::open(path).await?;
    let mut contents = String::new();
    file.read_to_string(&mut contents).await?;
    Ok(contents)
}

#[tokio::main]
async fn main() {
    match read_file("example.txt").await {
        Ok(contents) => println!("파일 내용:\n{}", contents),
        Err(e) => println!("에러: {}", e),
    }
}
```

여러 비동기 작업의 결과를 모을 때도 에러 처리가 필요하다:

```rust
async fn fetch_all(urls: Vec<&str>) -> Result<Vec<String>, reqwest::Error> {
    let mut results = Vec::new();
    
    for url in urls {
        let response = reqwest::get(url).await?;
        let body = response.text().await?;
        results.push(body);
    }
    
    Ok(results)
}
```

하지만 이 코드는 순차적으로 실행된다. 동시에 실행하면서 에러도 처리하려면 조금 복잡해진다:

```rust
async fn fetch_all_concurrent(urls: Vec<&str>) -> Vec<Result<String, reqwest::Error>> {
    let tasks: Vec<_> = urls.iter()
        .map(|url| async move {
            let response = reqwest::get(*url).await?;
            response.text().await
        })
        .collect();
    
    futures::future::join_all(tasks).await
}
```

## 실전 프로그램: 간단한 비동기 타이머

오늘 배운 내용을 종합해서 여러 타이머를 동시에 실행하는 프로그램을 만든다:

```rust
use tokio::time::{sleep, Duration, Instant};
use std::io::{self, Write};

// 타이머 하나를 실행
async fn run_timer(name: &str, seconds: u64) {
    println!("[{}] {}초 타이머 시작", name, seconds);
    let start = Instant::now();
    
    sleep(Duration::from_secs(seconds)).await;
    
    let elapsed = start.elapsed();
    println!("[{}] 완료! (실제 시간: {:.2}초)", name, elapsed.as_secs_f64());
}

// 카운트다운 타이머
async fn countdown_timer(name: &str, seconds: u64) {
    println!("[{}] {}초 카운트다운 시작", name, seconds);
    
    for remaining in (1..=seconds).rev() {
        print!("[{}] {}... ", name, remaining);
        io::stdout().flush().unwrap();
        sleep(Duration::from_secs(1)).await;
    }
    
    println!("\n[{}] 시간 종료!", name);
}

#[tokio::main]
async fn main() {
    println!("=== 비동기 타이머 프로그램 ===\n");
    
    // 여러 타이머를 동시에 실행
    let timer1 = run_timer("타이머1", 3);
    let timer2 = run_timer("타이머2", 2);
    let countdown = countdown_timer("카운트다운", 5);
    
    // 모두 완료될 때까지 기다림
    tokio::join!(timer1, timer2, countdown);
    
    println!("\n모든 타이머 완료!");
}
```

실행하면:

```

[카운트다운] 5... [카운트다운] 4... [타이머2] 완료! (실제 시간: 2.00초)
[카운트다운] 3... [타이머1] 완료! (실제 시간: 3.00초)
```

## 스레드 vs 비동기: 언제 무엇을 사용할까

지금까지 배운 스레드와 비동기는 각각 장단점이 있다:

**스레드 사용이 좋은 경우:**
- CPU 집약적인 작업 (계산, 이미지 처리 등)
- 블로킹 작업을 피할 수 없는 경우
- 동시 작업의 수가 적은 경우 (수십 개 이하)
- 코드가 더 단순하고 직관적이어야 하는 경우

**비동기 사용이 좋은 경우:**
- I/O 집약적인 작업 (네트워크, 파일, 데이터베이스)
- 동시 작업의 수가 많은 경우 (수천 개 이상)
- 메모리와 CPU 효율이 중요한 경우
- 높은 처리량(throughput)이 필요한 경우

우리의 TCP 서버는 전형적인 I/O 집약적 작업이다. 클라이언트로부터 메시지를 기다리는 시간이 대부분이므로, 비동기가 완벽하게 맞는다.

## 비동기 런타임 이해하기

tokio는 내부적으로 작은 수의 스레드(보통 CPU 코어 수만큼)를 가진 스레드 풀을 생성한다. 이 스레드들이 수천 개의 비동기 태스크를 번갈아 실행한다:

```
런타임 스레드 풀 (4개 스레드)
┌─────────┬─────────┬─────────┬─────────┐
│Thread 1 │Thread 2 │Thread 3 │Thread 4 │
└────┬────┴────┬────┴────┬────┴────┬────┘
     │         │         │         │
     ↓         ↓         ↓         ↓
  [Task A] [Task E] [Task I] [Task M]
  [Task B] [Task F] [Task J] [Task N]
  [Task C] [Task G] [Task K] [Task O]
  [Task D] [Task H] [Task L] [Task P]
```

각 태스크가 `.await`를 만나 대기하면, 런타임은 즉시 다른 태스크로 전환한다. 이 전환은 매우 빠르다(마이크로초 단위). 반면 OS 스레드 컨텍스트 스위칭은 느리다(밀리초 단위).

## 내일 배울 것

오늘은 `async/await`의 기초를 배웠다. 비동기 함수를 작성하고, `.await`로 결과를 기다리고, `tokio::join!`으로 여러 작업을 동시에 실행하고, `tokio::spawn`으로 백그라운드 태스크를 생성했다.

내일은 이 지식을 바탕으로 TCP 에코 서버를 비동기로 재작성한다. `tokio::net::TcpListener`를 사용해서 수천 개의 동시 연결을 적은 메모리로 처리하는 방법을 배운다. 스레드 기반 서버와 비교해서 얼마나 효율적인지 직접 확인할 것이다.

## 핵심 정리

이번 시간에 배운 핵심 개념들을 정리한다:

**`async fn`**: 비동기 함수를 정의한다. 호출하면 `Future`를 반환한다.

**`.await`**: Future가 완료될 때까지 기다린다. 스레드를 블록하지 않고, 런타임이 다른 작업을 실행할 수 있다.

**`tokio::join!`**: 여러 Future를 동시에 실행하고 모두 완료될 때까지 기다린다.

**`tokio::spawn`**: 백그라운드에서 독립적으로 실행될 태스크를 생성한다. `'static` 라이프타임이 필요하다.

**비동기의 장점**: I/O 대기 시간을 활용해서 다른 작업을 처리한다. 적은 스레드로 많은 동시 작업을 처리할 수 있다.

비동기 프로그래밍은 처음엔 어렵게 느껴질 수 있다. 하지만 기본 패턴을 익히면, 스레드보다 훨씬 효율적인 동시성 코드를 작성할 수 있다. 내일부터 실전에서 어떻게 활용하는지 배운다.
