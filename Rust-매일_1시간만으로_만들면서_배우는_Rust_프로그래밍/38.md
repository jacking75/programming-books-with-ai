# Day 38: 스마트 포인터들

오늘은 Rust의 스마트 포인터들을 다룬다. 스마트 포인터는 단순한 포인터를 넘어 추가적인 메타데이터와 기능을 가진 데이터 구조다. Rust의 소유권 시스템은 강력하지만, 때로는 기본 규칙만으로는 해결하기 어려운 문제가 있다. 스마트 포인터는 이런 상황에서 해결책을 제공한다. 실전에서는 서버의 설정 시스템, 플러그인 아키텍처, 그래프 구조를 구현하면서 각 스마트 포인터가 어떤 문제를 해결하는지 배운다.

## **Box<T>: 힙 할당의 기본**

`Box<T>`는 가장 단순한 스마트 포인터다. 데이터를 스택이 아닌 힙에 할당한다. 언뜻 보면 단순해 보이지만, 실전에서는 매우 유용하다.

```rust
fn basic_box_example() {
    // 스택에 할당
    let x = 5;
    
    // 힙에 할당
    let y = Box::new(5);
    
    println!("x = {}, y = {}", x, *y);
}
```

`Box`는 다음과 같은 상황에서 필요하다.

```
┌─────────────────────────────────────┐
│         Stack (고정 크기)            │
├─────────────────────────────────────┤
│  Box pointer (8 bytes)              │
│         ↓                           │
└─────────│───────────────────────────┘
          │
          ↓
┌─────────────────────────────────────┐
│         Heap (동적 크기)             │
├─────────────────────────────────────┤
│  실제 데이터 (크기 무관)              │
└─────────────────────────────────────┘
```

### **재귀 타입 정의하기**

컴파일 타임에 크기를 알 수 없는 타입을 정의할 때 `Box`가 필수다. 재귀 데이터 구조가 대표적인 예다.

```rust
// 컴파일 에러: 무한 크기
// enum List {
//     Cons(i32, List),
//     Nil,
// }

// Box를 사용하여 해결
enum List {
    Cons(i32, Box<List>),
    Nil,
}

impl List {
    fn new() -> Self {
        List::Nil
    }
    
    fn prepend(self, value: i32) -> Self {
        List::Cons(value, Box::new(self))
    }
    
    fn len(&self) -> usize {
        match self {
            List::Cons(_, tail) => 1 + tail.len(),
            List::Nil => 0,
        }
    }
    
    fn print(&self) {
        match self {
            List::Cons(value, tail) => {
                print!("{} -> ", value);
                tail.print();
            }
            List::Nil => println!("Nil"),
        }
    }
}

fn linked_list_example() {
    let list = List::new()
        .prepend(3)
        .prepend(2)
        .prepend(1);
    
    list.print(); // 1 -> 2 -> 3 -> Nil
    println!("Length: {}", list.len());
}
```

### **실전: 플러그인 시스템의 동적 디스패치**

서버에 플러그인을 추가할 수 있는 시스템을 만들 때, trait 객체를 `Box`에 담아 저장한다.

```rust
trait Plugin {
    fn name(&self) -> &str;
    fn execute(&self, input: &str) -> String;
}

struct UppercasePlugin;

impl Plugin for UppercasePlugin {
    fn name(&self) -> &str {
        "uppercase"
    }
    
    fn execute(&self, input: &str) -> String {
        input.to_uppercase()
    }
}

struct ReversePlugin;

impl Plugin for ReversePlugin {
    fn name(&self) -> &str {
        "reverse"
    }
    
    fn execute(&self, input: &str) -> String {
        input.chars().rev().collect()
    }
}

struct PluginManager {
    plugins: Vec<Box<dyn Plugin>>,
}

impl PluginManager {
    fn new() -> Self {
        PluginManager {
            plugins: Vec::new(),
        }
    }
    
    fn register(&mut self, plugin: Box<dyn Plugin>) {
        println!("Registering plugin: {}", plugin.name());
        self.plugins.push(plugin);
    }
    
    fn execute(&self, plugin_name: &str, input: &str) -> Option<String> {
        self.plugins
            .iter()
            .find(|p| p.name() == plugin_name)
            .map(|p| p.execute(input))
    }
    
    fn list_plugins(&self) {
        println!("Available plugins:");
        for plugin in &self.plugins {
            println!("  - {}", plugin.name());
        }
    }
}

fn plugin_system_example() {
    let mut manager = PluginManager::new();
    
    manager.register(Box::new(UppercasePlugin));
    manager.register(Box::new(ReversePlugin));
    
    manager.list_plugins();
    
    if let Some(result) = manager.execute("uppercase", "hello") {
        println!("Result: {}", result);
    }
    
    if let Some(result) = manager.execute("reverse", "rust") {
        println!("Result: {}", result);
    }
}
```

`Box`를 사용하면 서로 다른 크기의 타입들을 하나의 컬렉션에 저장할 수 있다. 각 플러그인은 다른 크기를 가질 수 있지만, `Box`는 항상 포인터 크기(8바이트)이므로 `Vec`에 저장 가능하다.

## **Rc<T>: 단일 스레드 참조 카운팅**

`Rc<T>`는 Reference Counted의 약자다. 하나의 데이터를 여러 소유자가 공유해야 할 때 사용한다. 단, 단일 스레드에서만 안전하다.

```rust
use std::rc::Rc;

fn basic_rc_example() {
    let data = Rc::new(String::from("shared data"));
    
    println!("Reference count: {}", Rc::strong_count(&data)); // 1
    
    {
        let data2 = Rc::clone(&data);
        println!("Reference count: {}", Rc::strong_count(&data)); // 2
        
        let data3 = Rc::clone(&data);
        println!("Reference count: {}", Rc::strong_count(&data)); // 3
    }
    
    println!("Reference count: {}", Rc::strong_count(&data)); // 1
}
```

`Rc::clone()`은 데이터를 복사하지 않고 참조 카운트만 증가시킨다. 이것이 일반적인 `clone()`과의 차이다.

```
┌─────────────────────────────────────┐
│  Rc<String>                         │
│  ┌───────────────┐                  │
│  │ ref_count: 3  │                  │
│  │ data: "..."   │                  │
│  └───────────────┘                  │
│         ↑    ↑    ↑                 │
│         │    │    │                 │
│      data data2 data3               │
└─────────────────────────────────────┘
```

### **실전: 서버 설정 공유하기**

서버의 여러 컴포넌트가 같은 설정을 읽어야 하는 경우가 많다. `Rc`를 사용하면 설정을 효율적으로 공유할 수 있다.

```rust
use std::rc::Rc;
use std::collections::HashMap;

#[derive(Debug, Clone)]
struct ServerConfig {
    host: String,
    port: u16,
    max_connections: usize,
    timeout_seconds: u64,
    settings: HashMap<String, String>,
}

impl ServerConfig {
    fn new(host: String, port: u16) -> Self {
        ServerConfig {
            host,
            port,
            max_connections: 1000,
            timeout_seconds: 30,
            settings: HashMap::new(),
        }
    }
    
    fn set(&mut self, key: String, value: String) {
        self.settings.insert(key, value);
    }
    
    fn get(&self, key: &str) -> Option<&String> {
        self.settings.get(key)
    }
}

struct RequestHandler {
    config: Rc<ServerConfig>,
    handler_id: usize,
}

impl RequestHandler {
    fn new(config: Rc<ServerConfig>, handler_id: usize) -> Self {
        RequestHandler { config, handler_id }
    }
    
    fn handle_request(&self) {
        println!("Handler {} using config:", self.handler_id);
        println!("  Host: {}:{}", self.config.host, self.config.port);
        println!("  Max connections: {}", self.config.max_connections);
    }
}

struct Logger {
    config: Rc<ServerConfig>,
}

impl Logger {
    fn new(config: Rc<ServerConfig>) -> Self {
        Logger { config }
    }
    
    fn log(&self, message: &str) {
        println!("[Logger] {} - {}", self.config.host, message);
    }
}

fn config_sharing_example() {
    let mut config = ServerConfig::new("127.0.0.1".to_string(), 8080);
    config.set("log_level".to_string(), "info".to_string());
    
    let config = Rc::new(config);
    
    println!("Initial reference count: {}", Rc::strong_count(&config));
    
    let handler1 = RequestHandler::new(Rc::clone(&config), 1);
    let handler2 = RequestHandler::new(Rc::clone(&config), 2);
    let logger = Logger::new(Rc::clone(&config));
    
    println!("After creating components: {}", Rc::strong_count(&config)); // 4
    
    handler1.handle_request();
    handler2.handle_request();
    logger.log("Server started");
}
```

이 패턴의 장점은 설정을 복사하지 않고도 여러 컴포넌트가 공유할 수 있다는 것이다. 메모리 효율적이고 일관성도 보장된다.

## **RefCell<T>: 내부 가변성**

`RefCell<T>`은 컴파일 타임이 아닌 런타임에 빌림 규칙을 확인한다. 이것을 내부 가변성(interior mutability)이라고 한다.

```rust
use std::cell::RefCell;

fn basic_refcell_example() {
    let data = RefCell::new(5);
    
    println!("Initial value: {}", data.borrow());
    
    // 가변 빌림
    *data.borrow_mut() = 10;
    
    println!("Modified value: {}", data.borrow());
}
```

`RefCell`의 빌림 규칙은 일반 참조와 동일하다. 차이는 확인 시점이다.

```
일반 참조:
  컴파일 타임에 확인
  ✓ 안전 보장
  ✗ 유연성 제한

RefCell:
  런타임에 확인
  ✓ 유연성 증가
  ✗ 패닉 가능성
```

런타임에 빌림 규칙을 위반하면 프로그램이 패닉한다.

```rust
fn refcell_panic_example() {
    let data = RefCell::new(5);
    
    let r1 = data.borrow();
    // let r2 = data.borrow_mut(); // 패닉! 이미 불변 빌림이 있음
    
    println!("r1: {}", r1);
}
```

### **Rc와 RefCell 조합하기**

`Rc`는 공유 소유권을 제공하지만 불변이다. `RefCell`은 가변성을 제공하지만 단일 소유자만 가능하다. 둘을 조합하면 공유 가능하면서도 변경 가능한 데이터를 만들 수 있다.

```rust
use std::rc::Rc;
use std::cell::RefCell;

#[derive(Debug)]
struct Node {
    value: i32,
    neighbors: RefCell<Vec<Rc<Node>>>,
}

impl Node {
    fn new(value: i32) -> Rc<Self> {
        Rc::new(Node {
            value,
            neighbors: RefCell::new(Vec::new()),
        })
    }
    
    fn add_neighbor(&self, neighbor: Rc<Node>) {
        self.neighbors.borrow_mut().push(neighbor);
    }
    
    fn print_neighbors(&self) {
        print!("Node {} -> ", self.value);
        for neighbor in self.neighbors.borrow().iter() {
            print!("{} ", neighbor.value);
        }
        println!();
    }
}

fn mutable_graph_example() {
    let node1 = Node::new(1);
    let node2 = Node::new(2);
    let node3 = Node::new(3);
    
    // 그래프 간선 추가
    node1.add_neighbor(Rc::clone(&node2));
    node1.add_neighbor(Rc::clone(&node3));
    node2.add_neighbor(Rc::clone(&node3));
    
    node1.print_neighbors(); // Node 1 -> 2 3
    node2.print_neighbors(); // Node 2 -> 3
    node3.print_neighbors(); // Node 3 ->
}
```

이 패턴은 그래프, 트리, 링크드 리스트 같은 복잡한 데이터 구조를 구현할 때 필수적이다.

### **실전: 캐시 시스템 구현**

서버에서 자주 사용하는 패턴은 읽기는 많지만 쓰기는 드문 캐시다. `Rc<RefCell<T>>`로 구현할 수 있다.

```rust
use std::rc::Rc;
use std::cell::RefCell;
use std::collections::HashMap;

type Cache = Rc<RefCell<HashMap<String, String>>>;

struct CacheManager {
    cache: Cache,
}

impl CacheManager {
    fn new() -> Self {
        CacheManager {
            cache: Rc::new(RefCell::new(HashMap::new())),
        }
    }
    
    fn get_cache(&self) -> Cache {
        Rc::clone(&self.cache)
    }
    
    fn set(&self, key: String, value: String) {
        self.cache.borrow_mut().insert(key, value);
    }
    
    fn get(&self, key: &str) -> Option<String> {
        self.cache.borrow().get(key).cloned()
    }
    
    fn clear(&self) {
        self.cache.borrow_mut().clear();
    }
    
    fn size(&self) -> usize {
        self.cache.borrow().len()
    }
}

struct Worker {
    id: usize,
    cache: Cache,
}

impl Worker {
    fn new(id: usize, cache: Cache) -> Self {
        Worker { id, cache }
    }
    
    fn read_cache(&self, key: &str) {
        let cache = self.cache.borrow();
        match cache.get(key) {
            Some(value) => println!("Worker {}: {} = {}", self.id, key, value),
            None => println!("Worker {}: {} not found", self.id, key),
        }
    }
    
    fn write_cache(&self, key: String, value: String) {
        self.cache.borrow_mut().insert(key.clone(), value.clone());
        println!("Worker {}: Set {} = {}", self.id, key, value);
    }
}

fn cache_example() {
    let manager = CacheManager::new();
    
    // 초기 데이터
    manager.set("user:1".to_string(), "Alice".to_string());
    manager.set("user:2".to_string(), "Bob".to_string());
    
    // 여러 워커가 캐시를 공유
    let worker1 = Worker::new(1, manager.get_cache());
    let worker2 = Worker::new(2, manager.get_cache());
    
    worker1.read_cache("user:1");
    worker2.read_cache("user:2");
    
    worker1.write_cache("user:3".to_string(), "Charlie".to_string());
    worker2.read_cache("user:3");
    
    println!("Total cache entries: {}", manager.size());
}
```

주의할 점은 `borrow()`나 `borrow_mut()`의 스코프를 최소화해야 한다는 것이다. 그렇지 않으면 패닉이 발생할 수 있다.

```rust
fn dangerous_pattern() {
    let cache: Rc<RefCell<HashMap<String, String>>> = 
        Rc::new(RefCell::new(HashMap::new()));
    
    let cache_ref = cache.borrow_mut(); // 가변 빌림 시작
    
    // 이 상태에서 다른 빌림 시도하면 패닉
    // let another_ref = cache.borrow(); // 패닉!
    
    drop(cache_ref); // 명시적으로 빌림 종료
    
    let another_ref = cache.borrow(); // 이제 안전
}

fn safe_pattern() {
    let cache: Rc<RefCell<HashMap<String, String>>> = 
        Rc::new(RefCell::new(HashMap::new()));
    
    {
        let mut cache_ref = cache.borrow_mut();
        cache_ref.insert("key".to_string(), "value".to_string());
    } // 스코프 종료로 자동 해제
    
    let cache_ref = cache.borrow(); // 안전
    println!("{:?}", cache_ref);
}
```

## **Weak<T>: 순환 참조 방지**

`Rc`를 사용할 때 주의해야 할 점은 순환 참조다. 두 개의 `Rc`가 서로를 가리키면 참조 카운트가 절대 0이 되지 않아 메모리 누수가 발생한다.

```rust
use std::rc::{Rc, Weak};
use std::cell::RefCell;

#[derive(Debug)]
struct Node {
    value: i32,
    parent: RefCell<Weak<Node>>,
    children: RefCell<Vec<Rc<Node>>>,
}

impl Node {
    fn new(value: i32) -> Rc<Self> {
        Rc::new(Node {
            value,
            parent: RefCell::new(Weak::new()),
            children: RefCell::new(Vec::new()),
        })
    }
    
    fn add_child(parent: &Rc<Node>, child: Rc<Node>) {
        // 자식의 parent를 약한 참조로 설정
        *child.parent.borrow_mut() = Rc::downgrade(parent);
        
        // 부모의 children에 추가
        parent.children.borrow_mut().push(child);
    }
    
    fn print_tree(&self, depth: usize) {
        let indent = "  ".repeat(depth);
        println!("{}Node {}", indent, self.value);
        
        for child in self.children.borrow().iter() {
            child.print_tree(depth + 1);
        }
    }
}

fn tree_example() {
    let root = Node::new(1);
    let child1 = Node::new(2);
    let child2 = Node::new(3);
    
    Node::add_child(&root, child1);
    Node::add_child(&root, child2);
    
    root.print_tree(0);
    
    println!("Root strong count: {}", Rc::strong_count(&root));
    println!("Root weak count: {}", Rc::weak_count(&root));
}
```

`Weak`는 참조 카운트를 증가시키지 않으므로 순환 참조를 방지한다.

```
Parent (Rc)
   ↓ strong reference
Child (Rc)
   ↑ weak reference
Parent

Weak 참조는 카운트에 포함되지 않음
→ Parent의 strong count가 0이 되면 해제됨
→ Child도 자동으로 해제됨
```

### **실전: 이벤트 리스너 시스템**

이벤트 발생자와 리스너 사이의 관계는 순환 참조가 발생하기 쉽다. `Weak`를 사용하면 안전하게 구현할 수 있다.

```rust
use std::rc::{Rc, Weak};
use std::cell::RefCell;

trait EventListener {
    fn on_event(&self, event: &str);
}

struct Logger {
    name: String,
}

impl EventListener for Logger {
    fn on_event(&self, event: &str) {
        println!("[{}] Event received: {}", self.name, event);
    }
}

struct EventEmitter {
    listeners: RefCell<Vec<Weak<dyn EventListener>>>,
}

impl EventEmitter {
    fn new() -> Self {
        EventEmitter {
            listeners: RefCell::new(Vec::new()),
        }
    }
    
    fn subscribe(&self, listener: Weak<dyn EventListener>) {
        self.listeners.borrow_mut().push(listener);
    }
    
    fn emit(&self, event: &str) {
        let mut listeners = self.listeners.borrow_mut();
        
        // 만료된 약한 참조 제거
        listeners.retain(|weak| {
            if let Some(listener) = weak.upgrade() {
                listener.on_event(event);
                true
            } else {
                false
            }
        });
    }
    
    fn listener_count(&self) -> usize {
        self.listeners.borrow().len()
    }
}

fn event_system_example() {
    let emitter = EventEmitter::new();
    
    {
        let logger1: Rc<dyn EventListener> = Rc::new(Logger {
            name: "Logger1".to_string(),
        });
        
        let logger2: Rc<dyn EventListener> = Rc::new(Logger {
            name: "Logger2".to_string(),
        });
        
        emitter.subscribe(Rc::downgrade(&logger1));
        emitter.subscribe(Rc::downgrade(&logger2));
        
        println!("Active listeners: {}", emitter.listener_count());
        emitter.emit("server_started");
    } // logger1, logger2 스코프 종료
    
    println!("\nAfter loggers dropped:");
    println!("Active listeners: {}", emitter.listener_count());
    emitter.emit("server_stopped"); // 리스너 없음
}
```

이 패턴의 핵심은 리스너가 삭제되면 자동으로 구독이 취소된다는 것이다. 수동으로 구독 해제를 관리할 필요가 없다.

## **스마트 포인터 비교와 선택 가이드**

각 스마트 포인터는 서로 다른 문제를 해결한다.

```rust
// Box<T>: 힙 할당이 필요할 때
fn use_box() {
    // 재귀 타입
    let list = Box::new(List::Cons(1, Box::new(List::Nil)));
    
    // 큰 데이터를 스택에서 힙으로
    let large_data = Box::new([0u8; 1_000_000]);
    
    // trait 객체
    let plugin: Box<dyn Plugin> = Box::new(UppercasePlugin);
}

// Rc<T>: 여러 소유자가 필요할 때 (단일 스레드)
fn use_rc() {
    let config = Rc::new(ServerConfig::new("localhost".to_string(), 8080));
    let handler1 = Handler::new(Rc::clone(&config));
    let handler2 = Handler::new(Rc::clone(&config));
}

// RefCell<T>: 불변 참조로 받았지만 내부를 변경해야 할 때
fn use_refcell() {
    let cache = RefCell::new(HashMap::new());
    cache.borrow_mut().insert("key", "value");
}

// Rc<RefCell<T>>: 공유되면서 변경 가능한 데이터
fn use_rc_refcell() {
    let shared_state = Rc::new(RefCell::new(HashMap::new()));
    let worker1 = Worker::new(Rc::clone(&shared_state));
    let worker2 = Worker::new(Rc::clone(&shared_state));
}

// Weak<T>: 순환 참조를 피해야 할 때
fn use_weak() {
    let parent = Rc::new(Node::new(1));
    let child = Rc::new(Node::new(2));
    // child.parent = Rc::downgrade(&parent);
}
```

선택 기준을 정리하면 다음과 같다.

| 상황 | 선택 | 이유 |
|------|------|------|
| 큰 데이터를 힙에 할당 | `Box<T>` | 스택 오버플로우 방지 |
| 재귀 타입 | `Box<T>` | 고정 크기 필요 |
| trait 객체 저장 | `Box<dyn Trait>` | 동적 디스패치 |
| 여러 소유자 (단일 스레드) | `Rc<T>` | 공유 소유권 |
| 불변 값의 내부 변경 | `RefCell<T>` | 내부 가변성 |
| 공유 + 변경 | `Rc<RefCell<T>>` | 조합 |
| 순환 참조 가능성 | `Weak<T>` | 메모리 누수 방지 |

## **실전: 설정 관리 시스템**

실제 서버에서 사용할 수 있는 설정 관리 시스템을 만들어보자. 여러 스마트 포인터를 조합한다.

```rust
use std::rc::{Rc, Weak};
use std::cell::RefCell;
use std::collections::HashMap;

type ConfigValue = String;
type ConfigMap = HashMap<String, ConfigValue>;

struct ConfigNode {
    name: String,
    values: RefCell<ConfigMap>,
    parent: RefCell<Weak<ConfigNode>>,
    children: RefCell<Vec<Rc<ConfigNode>>>,
}

impl ConfigNode {
    fn new(name: String) -> Rc<Self> {
        Rc::new(ConfigNode {
            name,
            values: RefCell::new(HashMap::new()),
            parent: RefCell::new(Weak::new()),
            children: RefCell::new(Vec::new()),
        })
    }
    
    fn add_child(parent: &Rc<ConfigNode>, child: Rc<ConfigNode>) {
        *child.parent.borrow_mut() = Rc::downgrade(parent);
        parent.children.borrow_mut().push(child);
    }
    
    fn set(&self, key: String, value: String) {
        self.values.borrow_mut().insert(key, value);
    }
    
    fn get(&self, key: &str) -> Option<String> {
        // 현재 노드에서 찾기
        if let Some(value) = self.values.borrow().get(key) {
            return Some(value.clone());
        }
        
        // 부모 노드에서 찾기 (상속)
        if let Some(parent) = self.parent.borrow().upgrade() {
            return parent.get(key);
        }
        
        None
    }
    
    fn print_tree(&self, depth: usize) {
        let indent = "  ".repeat(depth);
        println!("{}[{}]", indent, self.name);
        
        for (key, value) in self.values.borrow().iter() {
            println!("{}  {} = {}", indent, key, value);
        }
        
        for child in self.children.borrow().iter() {
            child.print_tree(depth + 1);
        }
    }
}

struct ConfigManager {
    root: Rc<ConfigNode>,
    cache: Rc<RefCell<HashMap<String, Weak<ConfigNode>>>>,
}

impl ConfigManager {
    fn new() -> Self {
        let root = ConfigNode::new("root".to_string());
        
        ConfigManager {
            root,
            cache: Rc::new(RefCell::new(HashMap::new())),
        }
    }
    
    fn create_section(&self, parent: &Rc<ConfigNode>, name: String) -> Rc<ConfigNode> {
        let section = ConfigNode::new(name.clone());
        ConfigNode::add_child(parent, Rc::clone(&section));
        
        // 캐시에 저장
        self.cache.borrow_mut().insert(name, Rc::downgrade(&section));
        
        section
    }
    
    fn get_section(&self, name: &str) -> Option<Rc<ConfigNode>> {
        self.cache.borrow().get(name)?.upgrade()
    }
    
    fn print(&self) {
        println!("=== Configuration Tree ===");
        self.root.print_tree(0);
    }
}

fn config_manager_example() {
    let manager = ConfigManager::new();
    
    // 전역 설정
    manager.root.set("app_name".to_string(), "MyServer".to_string());
    manager.root.set("version".to_string(), "1.0.0".to_string());
    
    // 데이터베이스 설정
    let db_section = manager.create_section(&manager.root, "database".to_string());
    db_section.set("host".to_string(), "localhost".to_string());
    db_section.set("port".to_string(), "5432".to_string());
    
    // 로깅 설정
    let log_section = manager.create_section(&manager.root, "logging".to_string());
    log_section.set("level".to_string(), "info".to_string());
    log_section.set("file".to_string(), "/var/log/app.log".to_string());
    
    // 설정 상속 테스트
    println!("DB app_name: {:?}", db_section.get("app_name")); // 부모에서 상속
    println!("DB host: {:?}", db_section.get("host")); // 자신의 값
    
    // 전체 트리 출력
    manager.print();
    
    // 캐시에서 섹션 가져오기
    if let Some(section) = manager.get_section("database") {
        println!("\nRetrieved from cache: {}", section.name);
    }
}
```

이 시스템은 계층적 설정을 지원하며, 자식 노드가 부모의 설정을 상속받을 수 있다. `Weak`를 사용하여 순환 참조를 방지하고, `Rc<RefCell<T>>`로 공유 가능하면서도 변경 가능한 캐시를 구현했다.

## **성능 고려사항**

스마트 포인터는 편리하지만 비용이 있다.

```rust
use std::time::Instant;

fn benchmark_smart_pointers() {
    const ITERATIONS: usize = 1_000_000;
    
    // 일반 변수
    let start = Instant::now();
    let mut sum = 0;
    for i in 0..ITERATIONS {
        sum += i;
    }
    println!("Direct: {:?}, sum: {}", start.elapsed(), sum);
    
    // Box
    let start = Instant::now();
    let mut sum = Box::new(0);
    for i in 0..ITERATIONS {
        *sum += i;
    }
    println!("Box: {:?}, sum: {}", start.elapsed(), sum);
    
    // Rc
    let start = Instant::now();
    let sum = Rc::new(RefCell::new(0));
    for i in 0..ITERATIONS {
        *sum.borrow_mut() += i;
    }
    println!("Rc<RefCell>: {:?}, sum: {}", start.elapsed(), sum.borrow());
}
```

`Rc<RefCell<T>>`는 참조 카운트 업데이트와 런타임 빌림 검사로 인해 오버헤드가 있다. 성능이 중요한 곳에서는 신중하게 사용해야 한다.

## **정리와 실전 조언**

오늘 배운 내용을 정리하면 다음과 같다.

`Box<T>`는 가장 단순하지만 강력한 스마트 포인터다. 재귀 타입, 큰 데이터의 힙 할당, trait 객체 저장에 사용한다. 오버헤드가 거의 없으므로 필요할 때 주저 없이 사용할 수 있다.

`Rc<T>`는 단일 스레드에서 여러 소유자가 필요할 때 사용한다. 설정, 캐시, 그래프 구조에 적합하다. 참조 카운팅으로 인한 약간의 오버헤드가 있지만, 대부분의 경우 문제가 되지 않는다. 멀티스레드에서는 `Arc<T>`를 대신 사용해야 한다.

`RefCell<T>`는 내부 가변성을 제공한다. 불변 참조로 받았지만 내부를 변경해야 할 때 유용하다. 런타임 빌림 검사로 인해 오버헤드가 있고, 패닉 가능성이 있으므로 빌림 스코프를 최소화해야 한다.

`Rc<RefCell<T>>`는 공유되면서 변경 가능한 데이터를 만든다. 캐시, 설정, 복잡한 데이터 구조에 자주 사용되는 패턴이다. 비용이 있으므로 정말 필요한 경우에만 사용해야 한다.

`Weak<T>`는 순환 참조를 방지한다. 부모-자식 관계, 이벤트 시스템, 옵저버 패턴에서 필수적이다. `upgrade()`가 `None`을 반환할 수 있음을 항상 확인해야 한다.

실전에서는 가장 단순한 것부터 시작한다. 일반 참조로 해결되면 그것을 사용하고, 안 되면 `Box`, 그래도 안 되면 `Rc`, 마지막으로 `Rc<RefCell>`을 고려한다. 각 단계마다 복잡도와 비용이 증가하므로, 꼭 필요한 만큼만 사용하는 것이 좋다.

내일은 `unsafe` 코드와 FFI를 다루면서, Rust의 안전성 보장을 일시적으로 해제해야 하는 상황과 그 방법을 배운다.
