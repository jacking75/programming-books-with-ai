# 매일 1시간만으로 만들면서 배우는 Rust 프로그래밍:   

# Day 34: 인증과 보안

애플리케이션에 사용자 인증을 추가하는 것은 단순히 기능을 추가하는 것 이상의 의미를 갖는다. 보안은 타협할 수 없는 요구사항이며, 잘못 구현하면 사용자 데이터가 위험에 노출된다. 오늘은 Rust에서 안전하게 인증 시스템을 구축하는 방법을 배운다. JWT 토큰을 사용한 인증, 비밀번호의 안전한 저장, 그리고 미들웨어를 통한 보호된 엔드포인트 구현까지 다룬다.

## 보안의 기본 원칙

인증 시스템을 만들기 전에 몇 가지 원칙을 이해해야 한다. 첫째, 비밀번호를 평문으로 저장하면 안 된다. 해시 함수를 사용하여 비가역적으로 변환한다. 둘째, 토큰이나 세션 정보는 안전하게 생성하고 검증해야 한다. 셋째, 민감한 데이터는 메모리에서도 최소한의 시간만 유지하고, 가능한 한 빨리 제거한다.

Rust의 타입 시스템과 소유권 모델은 보안 코드를 작성하는 데 유리하다. 컴파일러가 많은 실수를 미리 잡아주고, 메모리 안전성이 보장되므로 버퍼 오버플로우 같은 취약점을 원천적으로 방지한다.

## 사용자 모델 정의

먼저 사용자를 표현하는 구조체를 만든다:

```rust
use chrono::{DateTime, Utc};
use serde::{Deserialize, Serialize};
use sqlx::FromRow;

#[derive(Debug, Clone, FromRow)]
struct User {
    id: i64,
    username: String,
    email: String,
    password_hash: String,
    created_at: DateTime<Utc>,
}

#[derive(Debug, Deserialize)]
struct RegisterRequest {
    username: String,
    email: String,
    password: String,
}

#[derive(Debug, Deserialize)]
struct LoginRequest {
    username: String,
    password: String,
}

#[derive(Debug, Serialize)]
struct AuthResponse {
    token: String,
    username: String,
}
```

`User` 구조체는 `password_hash`를 포함하지만, 실제 비밀번호는 포함하지 않는다. 비밀번호는 해시화되어 저장되며, 원본은 즉시 폐기된다. `RegisterRequest`와 `LoginRequest`는 클라이언트로부터 받는 데이터를 역직렬화하는 데 사용된다.

## 비밀번호 해싱

bcrypt는 의도적으로 느린 해시 함수다. 이것이 장점이다. 공격자가 무차별 대입 공격을 시도할 때 각 시도마다 시간이 오래 걸리므로 공격이 비실용적이 된다:

```rust
use bcrypt::{hash, verify, DEFAULT_COST};

fn hash_password(password: &str) -> Result<String, bcrypt::BcryptError> {
    hash(password, DEFAULT_COST)
}

fn verify_password(password: &str, hash: &str) -> Result<bool, bcrypt::BcryptError> {
    verify(password, hash)
}
```

`DEFAULT_COST`는 계산 비용을 나타낸다. 값이 클수록 해싱에 시간이 더 걸린다. 기본값은 보통 12인데, 이는 현대 하드웨어에서 약 250밀리초 정도 걸린다. 이 정도면 사용자 경험을 해치지 않으면서도 공격을 어렵게 만든다.

여기서 중요한 점은 `password: &str`로 참조를 받는다는 것이다. 비밀번호 문자열의 소유권을 가져오지 않고, 읽기만 한다. 함수가 끝나면 호출자가 원한다면 비밀번호를 메모리에서 지울 수 있다.

## JWT 토큰 구조

JWT는 세 부분으로 구성된다: 헤더, 페이로드, 서명. 우리는 페이로드에 사용자 정보를 담는다:

```rust
use jsonwebtoken::{encode, decode, Header, Validation, EncodingKey, DecodingKey, Algorithm};

#[derive(Debug, Serialize, Deserialize)]
struct Claims {
    sub: String,      // subject (사용자 ID)
    username: String,
    exp: usize,       // expiration time
}

impl Claims {
    fn new(user_id: i64, username: String, expires_in_hours: i64) -> Self {
        let expiration = Utc::now()
            .checked_add_signed(chrono::Duration::hours(expires_in_hours))
            .expect("valid timestamp")
            .timestamp() as usize;
        
        Self {
            sub: user_id.to_string(),
            username,
            exp: expiration,
        }
    }
}
```

`Claims` 구조체는 토큰에 포함될 정보를 정의한다. `sub`는 주체(subject)로 사용자를 식별하고, `exp`는 만료 시간이다. 만료 시간을 설정하면 도난당한 토큰의 사용 기간을 제한할 수 있다.

`new` 메서드는 `String`을 값으로 받는다. 이는 소유권을 가져온다는 의미다. `Claims` 객체가 문자열을 소유하므로, 토큰이 생성된 후에도 데이터가 유효하게 유지된다.

## 토큰 생성과 검증

비밀 키를 사용하여 토큰을 생성하고 검증한다:

```rust
const JWT_SECRET: &[u8] = b"your-secret-key-change-in-production";

fn create_token(user_id: i64, username: String) -> Result<String, jsonwebtoken::errors::Error> {
    let claims = Claims::new(user_id, username, 24); // 24시간 유효
    
    encode(
        &Header::default(),
        &claims,
        &EncodingKey::from_secret(JWT_SECRET)
    )
}

fn validate_token(token: &str) -> Result<Claims, jsonwebtoken::errors::Error> {
    let token_data = decode::<Claims>(
        token,
        &DecodingKey::from_secret(JWT_SECRET),
        &Validation::new(Algorithm::HS256)
    )?;
    
    Ok(token_data.claims)
}
```

실제 프로덕션에서는 `JWT_SECRET`을 환경 변수나 설정 파일에서 읽어야 한다. 소스 코드에 하드코딩하면 안 된다. 비밀 키가 노출되면 공격자가 유효한 토큰을 위조할 수 있다.

`validate_token`은 토큰이 우리가 발급한 것인지, 만료되지 않았는지 확인한다. `decode` 함수는 서명을 검증하고 만료 시간을 체크한다. 실패하면 에러를 반환한다.

## 사용자 등록 구현

새 사용자를 등록하는 엔드포인트를 만든다:

```rust
use axum::{
    extract::State,
    http::StatusCode,
    Json,
};
use std::sync::Arc;
use sqlx::PgPool;

#[derive(Clone)]
struct AppState {
    db_pool: PgPool,
}

async fn register(
    State(state): State<Arc<AppState>>,
    Json(req): Json<RegisterRequest>,
) -> Result<(StatusCode, Json<AuthResponse>), (StatusCode, String)> {
    // 입력 검증
    if req.username.len() < 3 {
        return Err((StatusCode::BAD_REQUEST, "Username too short".into()));
    }
    
    if req.password.len() < 8 {
        return Err((StatusCode::BAD_REQUEST, "Password must be at least 8 characters".into()));
    }
    
    // 비밀번호 해싱
    let password_hash = hash_password(&req.password)
        .map_err(|e| {
            eprintln!("Hashing error: {}", e);
            (StatusCode::INTERNAL_SERVER_ERROR, "Failed to process password".into())
        })?;
    
    // 사용자 생성
    let user = sqlx::query_as::<_, User>(
        r#"
        INSERT INTO users (username, email, password_hash)
        VALUES ($1, $2, $3)
        RETURNING id, username, email, password_hash, created_at
        "#
    )
    .bind(&req.username)
    .bind(&req.email)
    .bind(&password_hash)
    .fetch_one(&state.db_pool)
    .await
    .map_err(|e| {
        eprintln!("Database error: {}", e);
        if e.to_string().contains("duplicate key") {
            (StatusCode::CONFLICT, "Username or email already exists".into())
        } else {
            (StatusCode::INTERNAL_SERVER_ERROR, "Failed to create user".into())
        }
    })?;
    
    // JWT 토큰 생성
    let token = create_token(user.id, user.username.clone())
        .map_err(|e| {
            eprintln!("Token creation error: {}", e);
            (StatusCode::INTERNAL_SERVER_ERROR, "Failed to create token".into())
        })?;
    
    Ok((
        StatusCode::CREATED,
        Json(AuthResponse {
            token,
            username: user.username,
        })
    ))
}
```

이 함수는 여러 단계로 구성된다. 먼저 입력을 검증하고, 비밀번호를 해싱하며, 데이터베이스에 저장한 후, JWT 토큰을 생성한다. 각 단계에서 에러가 발생하면 적절한 HTTP 상태 코드와 메시지를 반환한다.

`req.password`는 함수가 끝나면 자동으로 메모리에서 제거된다. Rust의 소유권 시스템 덕분에 명시적으로 지우지 않아도 스코프를 벗어나면 `drop`이 호출된다. 하지만 `String`의 기본 `Drop` 구현은 보안을 위해 메모리를 덮어쓰지 않는다. 더 높은 보안이 필요하면 `zeroize` 크레이트를 사용할 수 있다.

## 로그인 구현

기존 사용자가 인증하는 엔드포인트다:

```rust
async fn login(
    State(state): State<Arc<AppState>>,
    Json(req): Json<LoginRequest>,
) -> Result<Json<AuthResponse>, (StatusCode, String)> {
    // 사용자 조회
    let user = sqlx::query_as::<_, User>(
        "SELECT id, username, email, password_hash, created_at 
         FROM users WHERE username = $1"
    )
    .bind(&req.username)
    .fetch_optional(&state.db_pool)
    .await
    .map_err(|e| {
        eprintln!("Database error: {}", e);
        (StatusCode::INTERNAL_SERVER_ERROR, "Database error".into())
    })?;
    
    let user = user.ok_or_else(|| {
        (StatusCode::UNAUTHORIZED, "Invalid credentials".into())
    })?;
    
    // 비밀번호 검증
    let is_valid = verify_password(&req.password, &user.password_hash)
        .map_err(|e| {
            eprintln!("Verification error: {}", e);
            (StatusCode::INTERNAL_SERVER_ERROR, "Authentication error".into())
        })?;
    
    if !is_valid {
        return Err((StatusCode::UNAUTHORIZED, "Invalid credentials".into()));
    }
    
    // JWT 토큰 생성
    let token = create_token(user.id, user.username.clone())
        .map_err(|e| {
            eprintln!("Token creation error: {}", e);
            (StatusCode::INTERNAL_SERVER_ERROR, "Failed to create token".into())
        })?;
    
    Ok(Json(AuthResponse {
        token,
        username: user.username,
    }))
}
```

보안상 중요한 점이 있다. 사용자가 존재하지 않을 때와 비밀번호가 틀렸을 때 같은 에러 메시지를 반환한다. "Invalid credentials"는 모호하게 만들어 공격자가 어떤 사용자명이 존재하는지 알 수 없게 한다.

`fetch_optional`은 `Option<User>`를 반환한다. 사용자가 없으면 `None`이고, 이 경우 즉시 에러를 반환한다. 타이밍 공격을 방어하려면 실제로는 사용자가 없어도 해싱 작업을 수행하는 것이 좋지만, 여기서는 간단히 처리했다.

## 인증 미들웨어 구현

보호된 엔드포인트에 접근하려면 유효한 JWT 토큰이 필요하다. 미들웨어를 사용하여 토큰을 검증한다:

```rust
use axum::{
    extract::{Request, FromRequestParts},
    http::{header, request::Parts, StatusCode},
    middleware::Next,
    response::{IntoResponse, Response},
};

#[derive(Debug, Clone)]
struct AuthUser {
    user_id: i64,
    username: String,
}

#[axum::async_trait]
impl<S> FromRequestParts<S> for AuthUser
where
    S: Send + Sync,
{
    type Rejection = (StatusCode, String);
    
    async fn from_request_parts(parts: &mut Parts, _state: &S) -> Result<Self, Self::Rejection> {
        // Authorization 헤더 추출
        let auth_header = parts
            .headers
            .get(header::AUTHORIZATION)
            .and_then(|value| value.to_str().ok())
            .ok_or_else(|| {
                (StatusCode::UNAUTHORIZED, "Missing authorization header".into())
            })?;
        
        // Bearer 토큰 파싱
        let token = auth_header
            .strip_prefix("Bearer ")
            .ok_or_else(|| {
                (StatusCode::UNAUTHORIZED, "Invalid authorization format".into())
            })?;
        
        // 토큰 검증
        let claims = validate_token(token)
            .map_err(|e| {
                eprintln!("Token validation error: {}", e);
                (StatusCode::UNAUTHORIZED, "Invalid or expired token".into())
            })?;
        
        let user_id = claims.sub.parse::<i64>()
            .map_err(|_| {
                (StatusCode::UNAUTHORIZED, "Invalid token claims".into())
            })?;
        
        Ok(AuthUser {
            user_id,
            username: claims.username,
        })
    }
}
```

`FromRequestParts` trait를 구현하면 `AuthUser`를 핸들러의 매개변수로 사용할 수 있다. axum이 자동으로 이 코드를 호출하여 인증을 처리한다. 검증에 실패하면 즉시 401 Unauthorized를 반환한다.

`parts: &mut Parts`는 HTTP 요청의 메타데이터를 빌린다. 여기서는 헤더만 읽으므로 불변 참조로도 충분하지만, trait 정의상 가변 참조를 받는다.

이제 보호된 엔드포인트를 만들 수 있다:

```rust
async fn protected_route(
    auth_user: AuthUser,
) -> impl IntoResponse {
    Json(serde_json::json!({
        "message": "This is a protected route",
        "user_id": auth_user.user_id,
        "username": auth_user.username,
    }))
}

async fn get_profile(
    auth_user: AuthUser,
    State(state): State<Arc<AppState>>,
) -> Result<Json<User>, (StatusCode, String)> {
    let user = sqlx::query_as::<_, User>(
        "SELECT id, username, email, password_hash, created_at 
         FROM users WHERE id = $1"
    )
    .bind(auth_user.user_id)
    .fetch_one(&state.db_pool)
    .await
    .map_err(|e| {
        eprintln!("Database error: {}", e);
        (StatusCode::INTERNAL_SERVER_ERROR, "Failed to fetch profile".into())
    })?;
    
    Ok(Json(user))
}
```

`auth_user: AuthUser`를 매개변수로 선언하면, axum이 자동으로 인증을 처리한다. 토큰이 유효하지 않으면 핸들러가 호출되지 않는다. 이는 매우 편리한 패턴이다.

## 라우터 구성

모든 엔드포인트를 라우터에 등록한다:

```rust
use axum::{
    Router,
    routing::{get, post},
};

fn create_router(state: Arc<AppState>) -> Router {
    Router::new()
        // 공개 엔드포인트
        .route("/auth/register", post(register))
        .route("/auth/login", post(login))
        // 보호된 엔드포인트
        .route("/protected", get(protected_route))
        .route("/profile", get(get_profile))
        .with_state(state)
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let database_url = std::env::var("DATABASE_URL")
        .expect("DATABASE_URL must be set");
    
    let pool = sqlx::PgPool::connect(&database_url).await?;
    
    // 사용자 테이블 생성
    sqlx::query(
        r#"
        CREATE TABLE IF NOT EXISTS users (
            id BIGSERIAL PRIMARY KEY,
            username VARCHAR(255) UNIQUE NOT NULL,
            email VARCHAR(255) UNIQUE NOT NULL,
            password_hash VARCHAR(255) NOT NULL,
            created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
        )
        "#
    )
    .execute(&pool)
    .await?;
    
    let app_state = Arc::new(AppState {
        db_pool: pool,
    });
    
    let app = create_router(app_state);
    
    let listener = tokio::net::TcpListener::bind("127.0.0.1:3000").await?;
    println!("Server running on http://127.0.0.1:3000");
    
    axum::serve(listener, app).await?;
    
    Ok(())
}
```

## 보안 강화: 비밀 키 관리

JWT 비밀 키를 안전하게 관리하는 것이 중요하다. 환경 변수를 사용하는 개선된 버전이다:

```rust
use std::sync::Arc;

struct JwtKeys {
    encoding_key: EncodingKey,
    decoding_key: DecodingKey,
}

impl JwtKeys {
    fn from_secret(secret: &[u8]) -> Self {
        Self {
            encoding_key: EncodingKey::from_secret(secret),
            decoding_key: DecodingKey::from_secret(secret),
        }
    }
}

#[derive(Clone)]
struct AppState {
    db_pool: PgPool,
    jwt_keys: Arc<JwtKeys>,
}

fn create_token_with_keys(
    user_id: i64,
    username: String,
    keys: &JwtKeys,
) -> Result<String, jsonwebtoken::errors::Error> {
    let claims = Claims::new(user_id, username, 24);
    encode(&Header::default(), &claims, &keys.encoding_key)
}

fn validate_token_with_keys(
    token: &str,
    keys: &JwtKeys,
) -> Result<Claims, jsonwebtoken::errors::Error> {
    let token_data = decode::<Claims>(
        token,
        &keys.decoding_key,
        &Validation::new(Algorithm::HS256)
    )?;
    Ok(token_data.claims)
}
```

`JwtKeys`를 `Arc`로 감싸서 여러 핸들러가 공유한다. 키를 복사하지 않고 참조만 전달하므로 효율적이다. `Arc`는 스레드 안전하므로 비동기 환경에서도 안전하게 사용할 수 있다.

이제 `AppState`에 키를 포함시키고 초기화한다:

```rust
#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let database_url = std::env::var("DATABASE_URL")
        .expect("DATABASE_URL must be set");
    
    let jwt_secret = std::env::var("JWT_SECRET")
        .expect("JWT_SECRET must be set");
    
    let pool = sqlx::PgPool::connect(&database_url).await?;
    
    let app_state = Arc::new(AppState {
        db_pool: pool,
        jwt_keys: Arc::new(JwtKeys::from_secret(jwt_secret.as_bytes())),
    });
    
    let app = create_router(app_state);
    
    let listener = tokio::net::TcpListener::bind("127.0.0.1:3000").await?;
    println!("Server running on http://127.0.0.1:3000");
    
    axum::serve(listener, app).await?;
    
    Ok(())
}
```

## 토큰 갱신 구현

장시간 로그인 상태를 유지하려면 토큰 갱신 메커니즘이 필요하다:

```rust
async fn refresh_token(
    auth_user: AuthUser,
    State(state): State<Arc<AppState>>,
) -> Result<Json<AuthResponse>, (StatusCode, String)> {
    // 사용자가 여전히 존재하는지 확인
    let user = sqlx::query_as::<_, User>(
        "SELECT id, username, email, password_hash, created_at 
         FROM users WHERE id = $1"
    )
    .bind(auth_user.user_id)
    .fetch_optional(&state.db_pool)
    .await
    .map_err(|e| {
        eprintln!("Database error: {}", e);
        (StatusCode::INTERNAL_SERVER_ERROR, "Database error".into())
    })?
    .ok_or_else(|| {
        (StatusCode::UNAUTHORIZED, "User not found".into())
    })?;
    
    // 새 토큰 발급
    let token = create_token_with_keys(user.id, user.username.clone(), &state.jwt_keys)
        .map_err(|e| {
            eprintln!("Token creation error: {}", e);
            (StatusCode::INTERNAL_SERVER_ERROR, "Failed to create token".into())
        })?;
    
    Ok(Json(AuthResponse {
        token,
        username: user.username,
    }))
}
```

클라이언트는 토큰이 만료되기 전에 이 엔드포인트를 호출하여 새 토큰을 받을 수 있다. 현재 토큰이 유효해야만 새 토큰을 발급받을 수 있으므로 안전하다.

## 비밀번호 변경 구현

사용자가 비밀번호를 변경할 수 있는 기능을 추가한다:

```rust
#[derive(Debug, Deserialize)]
struct ChangePasswordRequest {
    old_password: String,
    new_password: String,
}

async fn change_password(
    auth_user: AuthUser,
    State(state): State<Arc<AppState>>,
    Json(req): Json<ChangePasswordRequest>,
) -> Result<StatusCode, (StatusCode, String)> {
    // 새 비밀번호 검증
    if req.new_password.len() < 8 {
        return Err((StatusCode::BAD_REQUEST, "Password must be at least 8 characters".into()));
    }
    
    // 현재 사용자 조회
    let user = sqlx::query_as::<_, User>(
        "SELECT id, username, email, password_hash, created_at 
         FROM users WHERE id = $1"
    )
    .bind(auth_user.user_id)
    .fetch_one(&state.db_pool)
    .await
    .map_err(|e| {
        eprintln!("Database error: {}", e);
        (StatusCode::INTERNAL_SERVER_ERROR, "Database error".into())
    })?;
    
    // 기존 비밀번호 확인
    let is_valid = verify_password(&req.old_password, &user.password_hash)
        .map_err(|e| {
            eprintln!("Verification error: {}", e);
            (StatusCode::INTERNAL_SERVER_ERROR, "Authentication error".into())
        })?;
    
    if !is_valid {
        return Err((StatusCode::UNAUTHORIZED, "Current password is incorrect".into()));
    }
    
    // 새 비밀번호 해싱
    let new_hash = hash_password(&req.new_password)
        .map_err(|e| {
            eprintln!("Hashing error: {}", e);
            (StatusCode::INTERNAL_SERVER_ERROR, "Failed to process password".into())
        })?;
    
    // 비밀번호 업데이트
    sqlx::query("UPDATE users SET password_hash = $1 WHERE id = $2")
        .bind(&new_hash)
        .bind(auth_user.user_id)
        .execute(&state.db_pool)
        .await
        .map_err(|e| {
            eprintln!("Database error: {}", e);
            (StatusCode::INTERNAL_SERVER_ERROR, "Failed to update password".into())
        })?;
    
    Ok(StatusCode::NO_CONTENT)
}
```

여기서 중요한 보안 패턴은 기존 비밀번호를 확인한다는 것이다. 토큰이 있어도 비밀번호를 바꾸려면 현재 비밀번호를 알아야 한다. 이는 세션 하이재킹 공격의 피해를 제한한다.

## 역할 기반 접근 제어

더 세밀한 권한 관리를 위해 역할을 추가할 수 있다:

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
enum Role {

#[derive(Debug, Clone)]
struct AuthUser {
```

## 레이트 리미팅

무차별 대입 공격을 방지하기 위해 요청 제한을 추가할 수 있다:

```rust
use tower_governor::{GovernorLayer, GovernorConfigBuilder};

fn create_router_with_rate_limit(state: Arc<AppState>) -> Router {
    let governor_conf = Box::new(
        GovernorConfigBuilder::default()
            .per_second(2)  // 초당 2개 요청
            .burst_size(5)  // 최대 5개까지 버스트 허용
            .finish()
            .unwrap()
    );
    
    Router::new()
        .route("/auth/login", post(login))
        .layer(GovernorLayer {
            config: Box::leak(governor_conf),
        })
        .route("/auth/register", post(register))
        .route("/protected", get(protected_route))
        .route("/profile", get(get_profile))
        .with_state(state)
}
```

`tower-governor` 크레이트는 IP 주소별로 요청을 제한한다. 로그인 엔드포인트에 레이트 리미팅을 적용하면 공격자가 빠르게 여러 비밀번호를 시도할 수 없다.

## 보안 헤더 추가

HTTP 보안 헤더를 설정하여 다양한 공격을 방어한다:

```rust
use tower_http::set_header::SetResponseHeaderLayer;
use axum::http::header;

fn create_secure_router(state: Arc<AppState>) -> Router {
    Router::new()
        .route("/auth/register", post(register))
        .route("/auth/login", post(login))
        .route("/protected", get(protected_route))
        .route("/profile", get(get_profile))
        .layer(SetResponseHeaderLayer::if_not_present(
            header::X_CONTENT_TYPE_OPTIONS,
            "nosniff".parse().unwrap(),
        ))
        .layer(SetResponseHeaderLayer::if_not_present(
            header::X_FRAME_OPTIONS,
            "DENY".parse().unwrap(),
        ))
        .layer(SetResponseHeaderLayer::if_not_present(
            header::HeaderName::from_static("x-xss-protection"),
            "1; mode=block".parse().unwrap(),
        ))
        .with_state(state)
}
```

이런 헤더들은 브라우저에게 보안 정책을 알려준다. `X-Content-Type-Options: nosniff`는 MIME 타입 스니핑을 방지하고, `X-Frame-Options: DENY`는 클릭재킹을 막으며, `X-XSS-Protection`은 XSS 공격을 차단한다.

## 테스트 작성

인증 시스템은 철저히 테스트해야 한다:

```rust
#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_password_hashing() {
        let password = "my_secure_password";
        let hash = hash_password(password).unwrap();
        
        assert!(verify_password(password, &hash).unwrap());
        assert!(!verify_password("wrong_password", &hash).unwrap());
    }
    
    #[test]
    fn test_token_lifecycle() {
        let token = create_token(1, "testuser".to_string()).unwrap();
        let claims = validate_token(&token).unwrap();
        
        assert_eq!(claims.sub, "1");
        assert_eq!(claims.username, "testuser");
    }
    
    #[test]
    fn test_expired_token() {
        // 과거 시간으로 토큰 생성
        let claims = Claims {
            sub: "1".to_string(),
            username: "testuser".to_string(),
            exp: (Utc::now().timestamp() - 3600) as usize, // 1시간 전
        };
        
        let token = encode(
            &Header::default(),
            &claims,
            &EncodingKey::from_secret(JWT_SECRET)
        ).unwrap();
        
        // 만료된 토큰은 검증 실패해야 함
        assert!(validate_token(&token).is_err());
    }
}
```

단위 테스트로 핵심 로직을 검증한다. 비밀번호 해싱이 올바르게 작동하는지, 토큰이 유효한지, 만료된 토큰이 거부되는지 확인한다.

## 오늘 배운 내용 정리

오늘은 Rust에서 안전한 인증 시스템을 구축하는 방법을 배웠다. bcrypt로 비밀번호를 해싱하고, JWT로 상태 없는 인증을 구현하며, 미들웨어로 보호된 엔드포인트를 만들었다. Rust의 타입 시스템과 소유권 모델이 보안 코드 작성에 어떻게 도움이 되는지 경험했다.

특히 `Arc`를 사용하여 JWT 키를 안전하게 공유하고, `FromRequestParts` trait로 인증 로직을 재사용 가능한 추출자로 만드는 패턴을 익혔다. 비밀번호 같은 민감한 데이터는 소유권 시스템 덕분에 스코프를 벗어나면 자동으로 정리된다.

내일은 WebSocket을 사용한 실시간 통신을 배운다. 양방향 통신 채널을 열고, 클라이언트와 서버 간에 실시간으로 데이터를 주고받으며, 비동기 브로드캐스트 패턴을 구현한다. 인증된 사용자만 WebSocket에 연결할 수 있게 하는 방법도 살펴본다.
