# 2장: 메모리 관리와 소유권
메모리 관리는 C++ 프로그래밍에서 가장 중요하면서도 가장 어려운 부분이다. 올바른 메모리 관리 없이는 메모리 누수, 댕글링 포인터, 이중 해제 등 심각한 버그가 발생한다. 이 장에서는 안전한 메모리 관리를 위한 현대적인 C++ 기법들을 배운다.
  

## 2.1 Raw 포인터의 위험성과 대안
Raw 포인터는 C++에서 가장 강력하면서도 가장 위험한 도구다. 메모리 주소를 직접 다루기 때문에 유연성이 높지만, 그만큼 실수할 여지도 많다.

### 메모리 누수

```cpp
// 안전하지 않은 코드: 메모리 누수
void unsafe_memory_leak() {
    int* ptr = new int(42);
    
    if (some_condition) {
        return;  // ptr이 해제되지 않음!
    }
    
    delete ptr;
}
```

위 코드는 조건문에서 함수가 조기 반환될 때 메모리가 해제되지 않는다. 함수가 여러 번 호출되면 메모리 누수가 누적되어 결국 시스템 메모리가 고갈될 수 있다.

```cpp
// 안전한 코드: RAII 패턴 사용
void safe_no_leak() {
    std::unique_ptr<int> ptr = std::make_unique<int>(42);
    
    if (some_condition) {
        return;  // ptr이 자동으로 해제됨
    }
    
    // 명시적 delete 불필요
}
```

`std::unique_ptr`은 RAII(Resource Acquisition Is Initialization) 패턴을 따른다. 스코프를 벗어날 때 소멸자가 자동으로 호출되어 메모리를 해제하므로, 어떤 경로로 함수를 빠져나가든 메모리 누수가 발생하지 않는다.

### 댕글링 포인터

```cpp
// 안전하지 않은 코드: 댕글링 포인터
int* unsafe_dangling() {
    int* ptr = new int(42);
    delete ptr;
    return ptr;  // 이미 해제된 메모리를 가리킴
}

void use_unsafe_dangling() {
    int* p = unsafe_dangling();
    *p = 10;  // 미정의 동작!
}
```

메모리를 해제한 후에도 포인터는 여전히 같은 주소를 가리킨다. 이 주소의 메모리는 이미 해제되었거나 다른 용도로 재사용되었을 수 있어서, 접근하면 미정의 동작이 발생한다.

```cpp
// 안전한 코드: 값으로 반환
int safe_return_value() {
    int value = 42;
    return value;  // 값을 복사하여 반환
}

// 또는 스마트 포인터로 소유권 이전
std::unique_ptr<int> safe_return_ownership() {
    return std::make_unique<int>(42);
}
```

값으로 반환하거나 스마트 포인터로 소유권을 명시적으로 이전하면 댕글링 포인터 문제를 피할 수 있다.

### 이중 해제

```cpp
// 안전하지 않은 코드: 이중 해제
void unsafe_double_free() {
    int* ptr = new int(42);
    delete ptr;
    delete ptr;  // 이중 해제, 미정의 동작!
}
```

같은 메모리를 두 번 해제하면 힙 메타데이터가 손상되어 프로그램이 크래시하거나 보안 취약점이 발생할 수 있다.

```cpp
// 안전한 코드: 스마트 포인터 사용
void safe_no_double_free() {
    std::unique_ptr<int> ptr = std::make_unique<int>(42);
    // 자동으로 정확히 한 번만 해제됨
}
```

스마트 포인터는 소멸자가 정확히 한 번만 호출되도록 보장한다.

### Raw 포인터의 올바른 사용처
Raw 포인터가 완전히 쓸모없는 것은 아니다. 소유권을 나타내지 않는 관찰 목적으로는 여전히 유용하다.

```cpp
class Node {
    int value;
    std::unique_ptr<Node> next;  // 소유권을 가짐
    Node* prev;  // 소유권 없음, 단순 참조
    
public:
    Node(int v) : value(v), next(nullptr), prev(nullptr) {}
    
    void set_next(std::unique_ptr<Node> n) {
        next = std::move(n);
        if (next) {
            next->prev = this;  // 역방향 참조는 raw 포인터
        }
    }
    
    Node* get_prev() const {
        return prev;  // 관찰 목적의 포인터 반환
    }
};
```

이중 연결 리스트에서 한쪽 방향은 `std::unique_ptr`로 소유권을 표현하고, 반대 방향은 raw 포인터로 단순 참조를 표현한다. 이렇게 하면 순환 참조 문제를 피하면서도 양방향 탐색이 가능하다.

### 배열과 포인터

```cpp
// 안전하지 않은 코드: raw 배열
void unsafe_array() {
    int* arr = new int[10];
    arr[10] = 42;  // 범위 초과!
    delete arr;    // 잘못된 delete, delete[] 사용해야 함
}

// 안전한 코드: std::vector 사용
void safe_array() {
    std::vector<int> arr(10);
    arr.at(10) = 42;  // 예외 발생, 안전하게 처리 가능
}
```

Raw 배열은 크기 정보를 포함하지 않아서 경계 검사가 불가능하다. `std::vector`는 크기 정보를 관리하고 `at()` 메서드로 안전한 접근을 제공한다.

## 2.2 스마트 포인터로 안전한 메모리 관리
스마트 포인터는 포인터와 유사하게 동작하지만, 메모리 관리를 자동화하여 안전성을 제공한다. C++ 표준 라이브러리는 세 가지 주요 스마트 포인터를 제공한다.

### `std::unique_ptr`과 독점적 소유권
`std::unique_ptr`은 리소스에 대한 독점적 소유권을 나타낸다. 복사할 수 없지만 이동할 수 있어서, 소유권 이전이 명확하다.

```cpp
// 기본 사용법
void basic_unique_ptr() {
    std::unique_ptr<int> ptr1 = std::make_unique<int>(42);
    
    // std::unique_ptr<int> ptr2 = ptr1;  // 컴파일 에러! 복사 불가
    std::unique_ptr<int> ptr2 = std::move(ptr1);  // 소유권 이동
    
    if (ptr1) {
        std::cout << *ptr1;  // 실행되지 않음, ptr1은 nullptr
    }
    
    if (ptr2) {
        std::cout << *ptr2 << std::endl;  // 42 출력
    }
}
```

`std::make_unique`는 C++14에서 도입되었으며, `new`를 직접 사용하는 것보다 안전하다. 예외가 발생해도 메모리 누수가 없다.

```cpp
// 예외 안전성
class Resource {
public:
    Resource() {
        std::cout << "Resource 생성" << std::endl;
    }
    ~Resource() {
        std::cout << "Resource 해제" << std::endl;
    }
};

void may_throw() {
    throw std::runtime_error("에러 발생");
}

// 안전하지 않은 코드
void unsafe_exception() {
    try {
        Resource* r1 = new Resource();
        may_throw();  // 예외 발생
        Resource* r2 = new Resource();  // 실행되지 않음
        
        delete r1;  // 실행되지 않음, r1 누수!
        delete r2;
    } catch (...) {
        // r1이 이미 누수됨
    }
}

// 안전한 코드
void safe_exception() {
    try {
        auto r1 = std::make_unique<Resource>();
        may_throw();  // 예외 발생
        auto r2 = std::make_unique<Resource>();  // 실행되지 않음
        
        // 명시적 delete 불필요
    } catch (...) {
        // r1은 자동으로 해제됨
    }
}
```

`std::unique_ptr`은 함수 매개변수와 반환값으로 사용하여 소유권 의미를 명확하게 표현할 수 있다.

```cpp
// 소유권을 받는 함수
void take_ownership(std::unique_ptr<Resource> res) {
    // res를 소유함
    // 함수가 끝나면 자동으로 해제됨
}

// 소유권을 반환하는 함수
std::unique_ptr<Resource> create_resource() {
    return std::make_unique<Resource>();
}

// 소유권 없이 관찰만 하는 함수
void observe_resource(const Resource* res) {
    // res를 읽기만 함, 소유권 없음
}

void ownership_example() {
    auto res = create_resource();  // 소유권 받음
    observe_resource(res.get());   // 관찰만 함
    take_ownership(std::move(res)); // 소유권 이전
    
    // res는 이제 nullptr
}
```

커스텀 삭제자를 사용하면 메모리 외의 리소스도 관리할 수 있다.

```cpp
// 파일 핸들 관리
void file_handle_example() {
    auto file_deleter = [](FILE* fp) {
        if (fp) {
            std::cout << "파일 닫기" << std::endl;
            fclose(fp);
        }
    };
    
    std::unique_ptr<FILE, decltype(file_deleter)> file(
        fopen("test.txt", "w"),
        file_deleter
    );
    
    if (file) {
        fprintf(file.get(), "안전한 파일 쓰기\n");
    }
    // 스코프를 벗어나면 자동으로 파일이 닫힘
}
```

배열을 관리할 때도 `std::unique_ptr`을 사용할 수 있지만, 대부분의 경우 `std::vector`가 더 나은 선택이다.

```cpp
// unique_ptr로 배열 관리 (권장하지 않음)
void unique_ptr_array() {
    std::unique_ptr<int[]> arr(new int[10]);
    arr[0] = 42;
}

// 더 나은 방법
void vector_array() {
    std::vector<int> arr(10);
    arr[0] = 42;
    // 크기 정보, 범위 검사 등 추가 기능 제공
}
```

### `std::shared_ptr`과 공유 소유권
`std::shared_ptr`은 여러 소유자가 같은 리소스를 공유할 수 있게 한다. 참조 카운팅을 사용하여 마지막 소유자가 사라질 때 리소스를 해제한다.

```cpp
// 기본 사용법
void basic_shared_ptr() {
    std::shared_ptr<int> ptr1 = std::make_shared<int>(42);
    std::cout << "참조 카운트: " << ptr1.use_count() << std::endl;  // 1
    
    {
        std::shared_ptr<int> ptr2 = ptr1;  // 복사 가능
        std::cout << "참조 카운트: " << ptr1.use_count() << std::endl;  // 2
        std::cout << "ptr1: " << *ptr1 << ", ptr2: " << *ptr2 << std::endl;
    }  // ptr2가 스코프를 벗어남
    
    std::cout << "참조 카운트: " << ptr1.use_count() << std::endl;  // 1
}  // ptr1이 스코프를 벗어나면 메모리 해제
```

`std::make_shared`는 객체와 제어 블록(참조 카운트 등)을 한 번의 할당으로 생성하여 성능과 예외 안전성을 개선한다.

```cpp
// make_shared의 장점
void make_shared_advantage() {
    // 비효율적: 두 번의 할당
    std::shared_ptr<Resource> ptr1(new Resource());
    
    // 효율적: 한 번의 할당
    std::shared_ptr<Resource> ptr2 = std::make_shared<Resource>();
}
```

`std::shared_ptr`은 여러 객체가 같은 데이터를 참조해야 할 때 유용하다.

```cpp
// 공유 데이터 구조
class Document {
    std::string content;
public:
    explicit Document(std::string c) : content(std::move(c)) {}
    const std::string& get_content() const { return content; }
};

class Editor {
    std::shared_ptr<Document> doc;
public:
    explicit Editor(std::shared_ptr<Document> d) : doc(std::move(d)) {}
    void print() const {
        std::cout << "Editor: " << doc->get_content() << std::endl;
    }
};

class Viewer {
    std::shared_ptr<Document> doc;
public:
    explicit Viewer(std::shared_ptr<Document> d) : doc(std::move(d)) {}
    void print() const {
        std::cout << "Viewer: " << doc->get_content() << std::endl;
    }
};

void shared_document_example() {
    auto doc = std::make_shared<Document>("중요한 문서 내용");
    
    Editor editor(doc);
    Viewer viewer(doc);
    
    editor.print();
    viewer.print();
    
    // doc, editor, viewer 중 어느 것이 먼저 소멸되든
    // 마지막 소유자가 사라질 때 Document가 해제됨
}
```

`std::shared_ptr`은 멀티스레드 환경에서 참조 카운트 업데이트가 원자적으로 수행되어 안전하다. 하지만 객체 자체의 동시 접근은 보호하지 않는다.

```cpp
// 스레드 안전성
void thread_safety_example() {
    auto ptr = std::make_shared<int>(0);
    
    // 안전: 참조 카운트 업데이트는 원자적
    std::thread t1([ptr]() {
        auto local_ptr = ptr;  // 안전한 복사
    });
    
    std::thread t2([ptr]() {
        auto local_ptr = ptr;  // 안전한 복사
    });
    
    // 안전하지 않음: 객체 자체의 동시 수정
    std::thread t3([ptr]() {
        (*ptr)++;  // 데이터 레이스!
    });
    
    std::thread t4([ptr]() {
        (*ptr)++;  // 데이터 레이스!
    });
    
    t1.join();
    t2.join();
    t3.join();
    t4.join();
}
```

데이터 자체를 보호하려면 뮤텍스나 원자 변수를 함께 사용해야 한다.

```cpp
// 올바른 스레드 안전성
void correct_thread_safety() {
    struct SafeCounter {
        std::mutex mtx;
        int value = 0;
    };
    
    auto counter = std::make_shared<SafeCounter>();
    
    auto increment = [counter]() {
        for (int i = 0; i < 1000; ++i) {
            std::lock_guard<std::mutex> lock(counter->mtx);
            counter->value++;
        }
    };
    
    std::thread t1(increment);
    std::thread t2(increment);
    
    t1.join();
    t2.join();
    
    std::cout << "최종 값: " << counter->value << std::endl;  // 2000
}
```

### `std::weak_ptr`로 순환 참조 방지
`std::weak_ptr`은 `std::shared_ptr`이 관리하는 객체를 참조하지만 참조 카운트를 증가시키지 않는다. 순환 참조를 끊는 데 사용된다.

```cpp
// 안전하지 않은 코드: 순환 참조
class BadNode {
    std::shared_ptr<BadNode> next;
    std::shared_ptr<BadNode> prev;  // 순환 참조!
    int value;
    
public:
    explicit BadNode(int v) : value(v) {
        std::cout << "BadNode " << value << " 생성" << std::endl;
    }
    
    ~BadNode() {
        std::cout << "BadNode " << value << " 해제" << std::endl;
    }
    
    void set_next(std::shared_ptr<BadNode> n) {
        next = n;
        n->prev = shared_from_this();  // 순환 참조 생성
    }
};

void circular_reference_problem() {
    auto node1 = std::make_shared<BadNode>(1);
    auto node2 = std::make_shared<BadNode>(2);
    
    // 순환 참조 생성: node1 -> node2 -> node1
    node1->set_next(node2);
    
    // 함수가 끝나도 node1과 node2가 서로를 참조하고 있어서
    // 참조 카운트가 0이 되지 않아 메모리 누수 발생!
}
```

`std::weak_ptr`을 사용하면 순환 참조 문제를 해결할 수 있다.

```cpp
// 안전한 코드: weak_ptr로 순환 참조 방지
class GoodNode : public std::enable_shared_from_this<GoodNode> {
    std::shared_ptr<GoodNode> next;
    std::weak_ptr<GoodNode> prev;  // weak_ptr 사용
    int value;
    
public:
    explicit GoodNode(int v) : value(v) {
        std::cout << "GoodNode " << value << " 생성" << std::endl;
    }
    
    ~GoodNode() {
        std::cout << "GoodNode " << value << " 해제" << std::endl;
    }
    
    void set_next(std::shared_ptr<GoodNode> n) {
        next = n;
        n->prev = shared_from_this();  // weak_ptr에 할당
    }
    
    std::shared_ptr<GoodNode> get_prev() {
        return prev.lock();  // weak_ptr을 shared_ptr로 변환
    }
    
    int get_value() const { return value; }
};

void no_circular_reference() {
    auto node1 = std::make_shared<GoodNode>(1);
    auto node2 = std::make_shared<GoodNode>(2);
    
    node1->set_next(node2);
    
    // prev는 weak_ptr이므로 참조 카운트를 증가시키지 않음
    if (auto prev = node2->get_prev()) {
        std::cout << "이전 노드 값: " << prev->get_value() << std::endl;
    }
    
}  // node1과 node2가 정상적으로 해제됨
```

`std::weak_ptr`은 객체가 여전히 존재하는지 확인한 후 사용해야 한다.

```cpp
// weak_ptr 사용 패턴
void weak_ptr_usage() {
    std::weak_ptr<int> weak;
    
    {
        auto shared = std::make_shared<int>(42);
        weak = shared;
        
        std::cout << "expired: " << weak.expired() << std::endl;  // false
        
        if (auto locked = weak.lock()) {
            std::cout << "값: " << *locked << std::endl;  // 42
        }
    }  // shared가 스코프를 벗어남
    
    std::cout << "expired: " << weak.expired() << std::endl;  // true
    
    if (auto locked = weak.lock()) {
        std::cout << "값: " << *locked << std::endl;  // 실행되지 않음
    } else {
        std::cout << "객체가 이미 해제됨" << std::endl;
    }
}
```

캐시 구현에도 `std::weak_ptr`이 유용하다.

```cpp
// 캐시 구현 예제
class ImageCache {
    std::map<std::string, std::weak_ptr<Image>> cache;
    
public:
    std::shared_ptr<Image> get_image(const std::string& path) {
        // 캐시에서 찾기
        auto it = cache.find(path);
        if (it != cache.end()) {
            if (auto img = it->second.lock()) {
                std::cout << "캐시에서 이미지 반환: " << path << std::endl;
                return img;
            }
        }
        
        // 캐시에 없으면 로드
        std::cout << "이미지 로드: " << path << std::endl;
        auto img = std::make_shared<Image>(path);
        cache[path] = img;
        return img;
    }
    
    void cleanup() {
        // 만료된 엔트리 제거
        for (auto it = cache.begin(); it != cache.end();) {
            if (it->second.expired()) {
                it = cache.erase(it);
            } else {
                ++it;
            }
        }
    }
};
```

## 2.3 C++26의 Trivial Relocatability
C++26은 `[[trivially_relocatable]]` 속성을 도입하여 객체 이동을 최적화하면서도 안전성을 유지한다.

### 객체 이동 최적화와 안전성
전통적으로 C++에서 객체를 이동할 때는 이동 생성자와 소멸자를 호출한다. 하지만 많은 타입들은 메모리를 단순히 복사하는 것만으로도 안전하게 이동할 수 있다.

```cpp
// 전통적인 이동 방식
class TraditionalString {
    char* data;
    size_t length;
    
public:
    TraditionalString(const char* s) {
        length = strlen(s);
        data = new char[length + 1];
        strcpy(data, s);
    }
    
    // 이동 생성자
    TraditionalString(TraditionalString&& other) noexcept
        : data(other.data), length(other.length) {
        other.data = nullptr;
        other.length = 0;
    }
    
    // 소멸자
    ~TraditionalString() {
        delete[] data;
    }
};

void traditional_move() {
    std::vector<TraditionalString> vec;
    vec.reserve(100);
    
    // 벡터가 재할당될 때:
    // 1. 새 메모리 할당
    // 2. 각 요소마다 이동 생성자 호출
    // 3. 각 요소마다 소멸자 호출
    // 4. 이전 메모리 해제
    for (int i = 0; i < 100; ++i) {
        vec.emplace_back("긴 문자열 내용");
    }
}
```

위 코드는 벡터가 재할당될 때마다 모든 요소의 이동 생성자와 소멸자를 호출한다. 하지만 `TraditionalString`의 경우 메모리를 단순히 복사(`memcpy`)하는 것만으로도 충분하다.

```cpp
// C++26 trivially relocatable
class [[trivially_relocatable]] OptimizedString {
    char* data;
    size_t length;
    
public:
    OptimizedString(const char* s) {
        length = strlen(s);
        data = new char[length + 1];
        strcpy(data, s);
    }
    
    // 이동 생성자와 소멸자는 여전히 정의
    OptimizedString(OptimizedString&& other) noexcept
        : data(other.data), length(other.length) {
        other.data = nullptr;
        other.length = 0;
    }
    
    ~OptimizedString() {
        delete[] data;
    }
};

void optimized_move() {
    std::vector<OptimizedString> vec;
    vec.reserve(100);
    
    // 벡터가 재할당될 때:
    // 1. 새 메모리 할당
    // 2. memcpy로 모든 요소를 한 번에 복사
    // 3. 이전 메모리 해제 (소멸자 호출 없음)
    for (int i = 0; i < 100; ++i) {
        vec.emplace_back("긴 문자열 내용");
    }
}
```

### `[[trivially_relocatable]]` 속성 활용
이 속성은 타입이 다음 조건을 만족할 때 사용할 수 있다:

1. 이동 생성 후 즉시 소멸자를 호출하는 것이 `memcpy`와 동등하다
2. 객체의 주소가 변경되어도 객체의 의미가 변하지 않는다

```cpp
// trivially relocatable한 타입들
class [[trivially_relocatable]] SimpleData {
    int value;
    double ratio;
    // 포인터나 참조가 없음
};

class [[trivially_relocatable]] SmartPointerData {
    std::unique_ptr<int> ptr;
    // unique_ptr도 trivially relocatable
};

// trivially relocatable하지 않은 타입
class NotTriviallyRelocatable {
    int* self_ptr;
    
public:
    NotTriviallyRelocatable() {
        self_ptr = reinterpret_cast<int*>(this);  // 자신의 주소를 저장
    }
    
    // 이동하면 self_ptr이 잘못된 주소를 가리킴
};
```

실제 성능 차이를 벤치마크로 확인할 수 있다.

```cpp
// 성능 비교 예제
#include <chrono>
#include <iostream>

template<typename String>
void benchmark_vector_reallocation() {
    auto start = std::chrono::high_resolution_clock::now();
    
    std::vector<String> vec;
    for (int i = 0; i < 1000000; ++i) {
        vec.emplace_back("벤치마크 문자열");
    }
    
    auto end = std::chrono::high_resolution_clock::now();
    auto duration = std::chrono::duration_cast<std::chrono::milliseconds>(
        end - start
    );
    
    std::cout << "소요 시간: " << duration.count() << "ms" << std::endl;
}

void performance_comparison() {
    std::cout << "Traditional: ";
    benchmark_vector_reallocation<TraditionalString>();
    
    std::cout << "Optimized: ";
    benchmark_vector_reallocation<OptimizedString>();
}
```

복잡한 데이터 구조에서도 `[[trivially_relocatable]]`를 활용할 수 있다.

```cpp
// 복잡한 구조체
struct [[trivially_relocatable]] GameObject {
    std::string name;
    std::unique_ptr<Mesh> mesh;
    std::shared_ptr<Texture> texture;
    std::vector<Component> components;
    
    // 모든 멤버가 trivially relocatable하면
    // 구조체 전체도 trivially relocatable
};

// 게임 엔진의 엔티티 관리
class EntityManager {
    std::vector<GameObject> entities;
    
public:
    void add_entity(GameObject obj) {
        entities.push_back(std::move(obj));
        // 벡터 재할당 시 최적화된 이동
    }
    
    void remove_entity(size_t index) {
        if (index < entities.size()) {
            entities.erase(entities.begin() + index);
            // 요소 이동 시 최적화
        }
    }
};
```

주의할 점은 `[[trivially_relocatable]]`를 잘못 사용하면 미정의 동작이 발생할 수 있다는 것이다.

```cpp
// 잘못된 사용 예
class [[trivially_relocatable]] WrongUsage {  // 버그!
    std::thread worker;
    std::atomic<bool> running{true};
    
public:
    WrongUsage() {
        worker = std::thread([this]() {
            while (running) {
                // this 포인터 사용
            }
        });
    }
    
    ~WrongUsage() {
        running = false;
        if (worker.joinable()) {
            worker.join();
        }
    }
    
    // memcpy로 이동하면 worker 스레드의 this 포인터가
    // 잘못된 주소를 가리키게 됨!
};
```

컴파일러는 `[[trivially_relocatable]]`이 안전한지 완전히 검증할 수 없으므로, 개발자가 신중하게 사용해야 한다.

```cpp
// 안전한 사용 가이드라인
class [[trivially_relocatable]] SafeUsage {
    // 1. 자기 참조 포인터를 가지지 않음
    // 2. 스레드나 비동기 작업에서 this를 사용하지 않음
    // 3. 외부 리소스가 객체 주소에 의존하지 않음
    
    std::unique_ptr<Data> data;
    std::string name;
    int value;
    
public:
    SafeUsage() = default;
    // 기본 이동 연산이 안전함
};
```

### 실전 응용: 고성능 컨테이너
`[[trivially_relocatable]]`를 활용하여 고성능 컨테이너를 구현할 수 있다.

```cpp
// 최적화된 동적 배열
template<typename T>
    requires std::is_trivially_relocatable_v<T>
class FastVector {
    T* data;
    size_t size;
    size_t capacity;
    
    void reallocate(size_t new_capacity) {
        T* new_data = static_cast<T*>(
            ::operator new(new_capacity * sizeof(T))
        );
        
        // trivially relocatable이므로 memcpy 사용
        if (data) {
            std::memcpy(new_data, data, size * sizeof(T));
            ::operator delete(data);
        }
        
        data = new_data;
        capacity = new_capacity;
    }
    
public:
    FastVector() : data(nullptr), size(0), capacity(0) {}
    
    ~FastVector() {
        // 각 요소의 소멸자 호출
        for (size_t i = 0; i < size; ++i) {
            data[i].~T();
        }
        ::operator delete(data);
    }
    
    void push_back(T value) {
        if (size == capacity) {
            reallocate(capacity == 0 ? 1 : capacity * 2);
        }
        new (data + size) T(std::move(value));
        ++size;
    }
    
    T& operator[](size_t index) {
        return data[index];
    }
};
```

이 장에서 배운 메모리 관리 기법들을 정리하면:

1. Raw 포인터 대신 스마트 포인터를 사용하여 메모리 안전성을 확보한다
2. `std::unique_ptr`로 독점적 소유권을 명확하게 표현한다
3. `std::shared_ptr`로 공유 소유권을 안전하게 관리한다
4. `std::weak_ptr`로 순환 참조를 방지한다
5. `[[trivially_relocatable]]`로 성능과 안전성을 동시에 달성한다

다음 장에서는 타입 안전성과 컴파일 타임 검증을 통해 런타임 오류를 사전에 방지하는 방법을 배운다.
