# 7장: 스레드 안전성과 데이터 레이스 방지

## 7.1 전통적인 동시성 문제들
멀티스레드 프로그래밍에서 가장 큰 문제는 데이터 레이스(data race)다. 데이터 레이스는 두 개 이상의 스레드가 동시에 같은 메모리 위치에 접근하고, 그 중 하나 이상이 쓰기 작업을 수행할 때 발생한다. 이는 C++에서 미정의 동작(Undefined Behavior)을 야기하며, 프로그램의 동작을 예측할 수 없게 만든다.

### 안전하지 않은 코드: 뮤텍스 없는 공유 데이터 접근

```cpp
#include <thread>
#include <vector>
#include <iostream>

class UnsafeCounter {
    int count = 0;
public:
    void increment() {
        ++count;  // 데이터 레이스 발생!
    }
    
    int get() const { return count; }
};

int main() {
    UnsafeCounter counter;
    std::vector<std::thread> threads;
    
    // 10개의 스레드가 각각 1000번씩 증가
    for (int i = 0; i < 10; ++i) {
        threads.emplace_back([&counter]() {
            for (int j = 0; j < 1000; ++j) {
                counter.increment();
            }
        });
    }
    
    for (auto& t : threads) {
        t.join();
    }
    
    // 결과: 10000이 아닐 가능성이 높다
    std::cout << "Count: " << counter.get() << std::endl;
}
```

위 코드는 여러 스레드가 `count` 변수에 동시에 접근하면서 데이터 레이스가 발생한다. `++count` 연산은 읽기-수정-쓰기의 세 단계로 이루어지며, 이 과정에서 다른 스레드의 개입이 발생하면 값이 손실된다.

### 기본적인 해결 방법: 뮤텍스 사용

```cpp
#include <thread>
#include <vector>
#include <mutex>
#include <iostream>

class MutexCounter {
    int count = 0;
    mutable std::mutex mtx;
public:
    void increment() {
        std::lock_guard<std::mutex> lock(mtx);
        ++count;
    }
    
    int get() const {
        std::lock_guard<std::mutex> lock(mtx);
        return count;
    }
};

int main() {
    MutexCounter counter;
    std::vector<std::thread> threads;
    
    for (int i = 0; i < 10; ++i) {
        threads.emplace_back([&counter]() {
            for (int j = 0; j < 1000; ++j) {
                counter.increment();
            }
        });
    }
    
    for (auto& t : threads) {
        t.join();
    }
    
    // 결과: 항상 10000
    std::cout << "Count: " << counter.get() << std::endl;
}
```

뮤텍스를 사용하면 안전성은 확보되지만, 성능 오버헤드가 크다. 특히 경합(contention)이 심한 경우 성능이 급격히 저하된다. 또한 데드락(deadlock)이나 우선순위 역전(priority inversion) 같은 새로운 문제가 발생할 수 있다.

### 원자적 연산을 이용한 개선

```cpp
#include <thread>
#include <vector>
#include <atomic>
#include <iostream>

class AtomicCounter {
    std::atomic<int> count{0};
public:
    void increment() {
        count.fetch_add(1, std::memory_order_relaxed);
    }
    
    int get() const {
        return count.load(std::memory_order_relaxed);
    }
};

int main() {
    AtomicCounter counter;
    std::vector<std::thread> threads;
    
    for (int i = 0; i < 10; ++i) {
        threads.emplace_back([&counter]() {
            for (int j = 0; j < 1000; ++j) {
                counter.increment();
            }
        });
    }
    
    for (auto& t : threads) {
        t.join();
    }
    
    // 결과: 항상 10000, 뮤텍스보다 빠름
    std::cout << "Count: " << counter.get() << std::endl;
}
```

원자적 연산은 뮤텍스보다 훨씬 가볍지만, 복잡한 자료구조에서는 사용이 어렵다. 특히 포인터를 사용하는 lock-free 자료구조에서는 ABA 문제 같은 미묘한 버그가 발생할 수 있다.
  

## 7.2 C++26 Hazard Pointer 활용
C++26에서는 `std::hazard_pointer`를 도입하여 lock-free 자료구조를 안전하게 구현할 수 있게 되었다. Hazard pointer는 특정 메모리 영역이 현재 사용 중임을 표시하여, 다른 스레드가 해당 메모리를 해제하지 못하도록 보호한다.

### Lock-free 스택의 안전하지 않은 구현

```cpp
#include <atomic>
#include <memory>

template<typename T>
class UnsafeLockFreeStack {
    struct Node {
        T data;
        Node* next;
        Node(const T& d) : data(d), next(nullptr) {}
    };
    
    std::atomic<Node*> head{nullptr};
    
public:
    void push(const T& value) {
        Node* new_node = new Node(value);
        new_node->next = head.load();
        while (!head.compare_exchange_weak(new_node->next, new_node)) {
            // CAS 실패 시 재시도
        }
    }
    
    bool pop(T& result) {
        Node* old_head = head.load();
        while (old_head && 
               !head.compare_exchange_weak(old_head, old_head->next)) {
            // CAS 실패 시 재시도
        }
        
        if (!old_head) return false;
        
        result = old_head->data;
        delete old_head;  // 위험! 다른 스레드가 아직 접근 중일 수 있다
        return true;
    }
};
```

위 코드의 문제점은 `pop()`에서 `old_head`를 삭제할 때 발생한다. `compare_exchange_weak`가 성공한 직후에도 다른 스레드가 여전히 `old_head`를 참조하고 있을 수 있다. 이 경우 use-after-free 버그가 발생한다.

### Hazard Pointer를 이용한 안전한 구현

```cpp
#include <atomic>
#include <memory>
#include <hazard_pointer>  // C++26

template<typename T>
class SafeLockFreeStack {
    struct Node {
        T data;
        std::atomic<Node*> next;
        Node(const T& d) : data(d), next(nullptr) {}
    };
    
    std::atomic<Node*> head{nullptr};
    std::hazard_pointer_domain domain;
    
public:
    void push(const T& value) {
        Node* new_node = new Node(value);
        new_node->next = head.load(std::memory_order_relaxed);
        
        while (!head.compare_exchange_weak(
            new_node->next, new_node,
            std::memory_order_release,
            std::memory_order_relaxed)) {
            // CAS 실패 시 재시도
        }
    }
    
    bool pop(T& result) {
        std::hazard_pointer hp = std::make_hazard_pointer(domain);
        Node* old_head;
        
        do {
            old_head = head.load(std::memory_order_acquire);
            if (!old_head) return false;
            
            // hazard pointer로 old_head 보호
            hp.protect(old_head);
            
            // 보호 후 head가 변경되었는지 확인
            if (head.load(std::memory_order_acquire) != old_head) {
                continue;  // 변경되었으면 재시도
            }
            
        } while (!head.compare_exchange_weak(
            old_head, old_head->next.load(std::memory_order_relaxed),
            std::memory_order_release,
            std::memory_order_relaxed));
        
        result = old_head->data;
        
        // 안전한 회수를 위해 retire
        hp.retire(old_head, [](Node* p) { delete p; });
        return true;
    }
    
    ~SafeLockFreeStack() {
        while (Node* node = head.load()) {
            head.store(node->next);
            delete node;
        }
    }
};
```

Hazard pointer를 사용하면 `old_head`가 아직 다른 스레드에서 사용 중인지 확인할 수 있다. `hp.protect(old_head)`는 해당 포인터를 보호하며, `hp.retire()`는 더 이상 다른 스레드가 참조하지 않을 때까지 삭제를 지연한다.

### ABA 문제 해결
ABA 문제는 포인터 `A`가 `B`로 변경되었다가 다시 `A`로 돌아왔을 때, compare-and-swap이 변경을 감지하지 못하는 문제다. Hazard pointer는 이 문제도 해결한다.

```cpp
#include <atomic>
#include <hazard_pointer>
#include <iostream>

template<typename T>
class ABAResistantStack {
    struct Node {
        T data;
        std::atomic<Node*> next;
        Node(const T& d) : data(d), next(nullptr) {}
    };
    
    std::atomic<Node*> head{nullptr};
    std::hazard_pointer_domain domain;
    
public:
    void push(const T& value) {
        Node* new_node = new Node(value);
        new_node->next = head.load(std::memory_order_relaxed);
        
        while (!head.compare_exchange_weak(
            new_node->next, new_node,
            std::memory_order_release,
            std::memory_order_relaxed)) {
        }
    }
    
    bool pop(T& result) {
        std::hazard_pointer hp = std::make_hazard_pointer(domain);
        Node* old_head;
        
        do {
            old_head = head.load(std::memory_order_acquire);
            if (!old_head) return false;
            
            hp.protect(old_head);
            
            // ABA 문제 방지: 보호 후 재확인
            Node* current_head = head.load(std::memory_order_acquire);
            if (current_head != old_head) {
                // 중간에 head가 변경됨 (ABA 상황 가능)
                continue;
            }
            
        } while (!head.compare_exchange_weak(
            old_head, old_head->next.load(std::memory_order_relaxed),
            std::memory_order_release,
            std::memory_order_relaxed));
        
        result = old_head->data;
        hp.retire(old_head, [](Node* p) { delete p; });
        return true;
    }
};

// 사용 예제
int main() {
    ABAResistantStack<int> stack;
    
    stack.push(1);
    stack.push(2);
    stack.push(3);
    
    int value;
    while (stack.pop(value)) {
        std::cout << "Popped: " << value << std::endl;
    }
}
```

Hazard pointer의 보호 메커니즘은 ABA 문제를 근본적으로 해결한다. 포인터가 보호되는 동안 해당 메모리는 재사용될 수 없기 때문에, 같은 주소에 다른 객체가 할당되는 상황을 방지한다.
  


## 7.3 RCU(Read-Copy-Update) 메커니즘
RCU는 읽기가 많고 쓰기가 적은 상황에서 탁월한 성능을 발휘하는 동기화 메커니즘이다. C++26에서는 `std::rcu`를 통해 표준화된 RCU 인터페이스를 제공한다.

### 안전하지 않은 읽기-쓰기 패턴

```cpp
#include <memory>
#include <mutex>
#include <shared_mutex>
#include <string>
#include <vector>

class UnsafeConfig {
    struct Data {
        std::string server_address;
        int port;
        std::vector<std::string> allowed_ips;
    };
    
    std::shared_ptr<Data> config_data;
    mutable std::shared_mutex mtx;
    
public:
    UnsafeConfig() {
        config_data = std::make_shared<Data>(
            Data{"localhost", 8080, {"127.0.0.1"}}
        );
    }
    
    std::string get_server_address() const {
        std::shared_lock lock(mtx);
        return config_data->server_address;  // 복사 비용 발생
    }
    
    void update_server(const std::string& addr, int p) {
        std::unique_lock lock(mtx);
        config_data->server_address = addr;
        config_data->port = p;
        // 쓰기 동안 모든 읽기가 차단됨
    }
};
```

`std::shared_mutex`를 사용하면 여러 읽기 스레드가 동시에 접근할 수 있지만, 쓰기 스레드는 모든 읽기를 차단해야 한다. 또한 데이터를 복사하는 비용이 발생한다.

### RCU를 이용한 읽기 최적화

```cpp
#include <memory>
#include <string>
#include <vector>
#include <rcu>  // C++26
#include <atomic>

class RCUConfig {
    struct Data {
        std::string server_address;
        int port;
        std::vector<std::string> allowed_ips;
    };
    
    std::atomic<Data*> config_data;
    std::rcu_domain domain;
    
public:
    RCUConfig() {
        config_data.store(new Data{"localhost", 8080, {"127.0.0.1"}},
                         std::memory_order_release);
    }
    
    // 읽기: 락 없이 고성능
    std::string get_server_address() const {
        std::rcu_reader reader(domain);
        Data* data = config_data.load(std::memory_order_acquire);
        return data->server_address;
    }
    
    int get_port() const {
        std::rcu_reader reader(domain);
        Data* data = config_data.load(std::memory_order_acquire);
        return data->port;
    }
    
    // 쓰기: copy-update 패턴
    void update_server(const std::string& addr, int p) {
        // 1. 현재 데이터 복사
        Data* old_data = config_data.load(std::memory_order_acquire);
        Data* new_data = new Data(*old_data);
        
        // 2. 복사본 수정
        new_data->server_address = addr;
        new_data->port = p;
        
        // 3. 포인터 교체
        Data* expected = old_data;
        while (!config_data.compare_exchange_weak(
            expected, new_data,
            std::memory_order_release,
            std::memory_order_acquire)) {
            // CAS 실패 시 재시도
            delete new_data;
            old_data = expected;
            new_data = new Data(*old_data);
            new_data->server_address = addr;
            new_data->port = p;
            expected = old_data;
        }
        
        // 4. 안전한 회수
        std::rcu_retire(domain, old_data, [](Data* p) { delete p; });
    }
    
    ~RCUConfig() {
        std::rcu_barrier(domain);  // 모든 회수 작업 완료 대기
        delete config_data.load();
    }
};
```

RCU의 핵심은 읽기 경로에 어떤 잠금도 사용하지 않는다는 점이다. `std::rcu_reader`는 현재 스레드가 RCU 보호 영역에 있음을 표시하며, 이 영역 내에서 접근하는 데이터는 회수되지 않는다.

### 실전 예제: 동시성 캐시 구현

```cpp
#include <rcu>
#include <atomic>
#include <memory>
#include <unordered_map>
#include <string>
#include <optional>

template<typename Key, typename Value>
class RCUCache {
    using Map = std::unordered_map<Key, Value>;
    
    std::atomic<Map*> cache_map;
    std::rcu_domain domain;
    
public:
    RCUCache() {
        cache_map.store(new Map{}, std::memory_order_release);
    }
    
    // 읽기 전용 작업: 락 없음
    std::optional<Value> get(const Key& key) const {
        std::rcu_reader reader(domain);
        Map* map = cache_map.load(std::memory_order_acquire);
        
        auto it = map->find(key);
        if (it != map->end()) {
            return it->second;
        }
        return std::nullopt;
    }
    
    // 쓰기 작업: copy-update
    void insert(const Key& key, const Value& value) {
        Map* old_map = cache_map.load(std::memory_order_acquire);
        Map* new_map = new Map(*old_map);
        
        (*new_map)[key] = value;
        
        Map* expected = old_map;
        while (!cache_map.compare_exchange_weak(
            expected, new_map,
            std::memory_order_release,
            std::memory_order_acquire)) {
            
            delete new_map;
            old_map = expected;
            new_map = new Map(*old_map);
            (*new_map)[key] = value;
            expected = old_map;
        }
        
        std::rcu_retire(domain, old_map, [](Map* p) { delete p; });
    }
    
    void erase(const Key& key) {
        Map* old_map = cache_map.load(std::memory_order_acquire);
        
        // 키가 존재하지 않으면 아무것도 하지 않음
        if (old_map->find(key) == old_map->end()) {
            return;
        }
        
        Map* new_map = new Map(*old_map);
        new_map->erase(key);
        
        Map* expected = old_map;
        while (!cache_map.compare_exchange_weak(
            expected, new_map,
            std::memory_order_release,
            std::memory_order_acquire)) {
            
            delete new_map;
            old_map = expected;
            
            if (old_map->find(key) == old_map->end()) {
                return;  // 다른 스레드가 이미 삭제함
            }
            
            new_map = new Map(*old_map);
            new_map->erase(key);
            expected = old_map;
        }
        
        std::rcu_retire(domain, old_map, [](Map* p) { delete p; });
    }
    
    size_t size() const {
        std::rcu_reader reader(domain);
        Map* map = cache_map.load(std::memory_order_acquire);
        return map->size();
    }
    
    ~RCUCache() {
        std::rcu_barrier(domain);
        delete cache_map.load();
    }
};

// 사용 예제
#include <thread>
#include <iostream>

int main() {
    RCUCache<std::string, int> cache;
    
    // 쓰기 스레드
    std::thread writer([&cache]() {
        for (int i = 0; i < 1000; ++i) {
            cache.insert("key" + std::to_string(i % 10), i);
        }
    });
    
    // 읽기 스레드들 (고성능, 락 없음)
    std::vector<std::thread> readers;
    for (int i = 0; i < 10; ++i) {
        readers.emplace_back([&cache]() {
            for (int j = 0; j < 10000; ++j) {
                auto value = cache.get("key" + std::to_string(j % 10));
                if (value) {
                    // 값 사용
                }
            }
        });
    }
    
    writer.join();
    for (auto& r : readers) {
        r.join();
    }
    
    std::cout << "Final cache size: " << cache.size() << std::endl;
}
```

### RCU의 장단점 이해
RCU는 읽기가 압도적으로 많은 워크로드에서 뛰어난 성능을 제공한다. 읽기 작업은 메모리 장벽만 필요하고 어떤 원자적 연산도 수행하지 않는다. 하지만 쓰기 작업은 전체 데이터 구조를 복사해야 하므로 비용이 크다.

```cpp
#include <rcu>
#include <atomic>
#include <vector>
#include <algorithm>
#include <chrono>
#include <iostream>

// 성능 비교 예제
class PerformanceComparison {
public:
    // RCU 버전: 읽기 최적화
    static void rcu_read_heavy_workload() {
        using Map = std::vector<int>;
        std::atomic<Map*> data;
        std::rcu_domain domain;
        data.store(new Map(1000, 0), std::memory_order_release);
        
        auto start = std::chrono::high_resolution_clock::now();
        
        std::vector<std::thread> threads;
        // 100개의 읽기 스레드
        for (int i = 0; i < 100; ++i) {
            threads.emplace_back([&data, &domain]() {
                for (int j = 0; j < 100000; ++j) {
                    std::rcu_reader reader(domain);
                    Map* map = data.load(std::memory_order_acquire);
                    int sum = 0;
                    for (int val : *map) {
                        sum += val;
                    }
                }
            });
        }
        
        // 1개의 쓰기 스레드
        threads.emplace_back([&data, &domain]() {
            for (int i = 0; i < 100; ++i) {
                Map* old_map = data.load(std::memory_order_acquire);
                Map* new_map = new Map(*old_map);
                (*new_map)[i % 1000] = i;
                
                Map* expected = old_map;
                while (!data.compare_exchange_weak(
                    expected, new_map,
                    std::memory_order_release,
                    std::memory_order_acquire)) {
                    delete new_map;
                    old_map = expected;
                    new_map = new Map(*old_map);
                    (*new_map)[i % 1000] = i;
                    expected = old_map;
                }
                
                std::rcu_retire(domain, old_map, [](Map* p) { delete p; });
            }
        });
        
        for (auto& t : threads) {
            t.join();
        }
        
        auto end = std::chrono::high_resolution_clock::now();
        auto duration = std::chrono::duration_cast<std::chrono::milliseconds>(
            end - start);
        
        std::cout << "RCU duration: " << duration.count() << "ms" << std::endl;
        
        std::rcu_barrier(domain);
        delete data.load();
    }
};
```

RCU는 읽기 성능이 중요한 설정 데이터, 캐시, 라우팅 테이블 등에 이상적이다. 반면 쓰기가 빈번한 경우에는 복사 비용 때문에 오히려 성능이 저하될 수 있다.

### 메모리 순서와 안전성 보장
RCU를 사용할 때는 메모리 순서를 올바르게 지정하는 것이 중요하다. 포인터를 교체할 때는 `memory_order_release`를, 읽을 때는 `memory_order_acquire`를 사용하여 happens-before 관계를 확립한다.

```cpp
#include <rcu>
#include <atomic>
#include <string>

class SafeRCUString {
    std::atomic<std::string*> str_ptr;
    std::rcu_domain domain;
    
public:
    SafeRCUString(const std::string& initial) {
        str_ptr.store(new std::string(initial), std::memory_order_release);
    }
    
    std::string read() const {
        std::rcu_reader reader(domain);
        // acquire: str_ptr 이전의 모든 쓰기를 볼 수 있음
        std::string* ptr = str_ptr.load(std::memory_order_acquire);
        return *ptr;
    }
    
    void update(const std::string& new_value) {
        std::string* new_str = new std::string(new_value);
        std::string* old_str = str_ptr.load(std::memory_order_relaxed);
        
        // release: 이전의 모든 쓰기가 완료됨을 보장
        while (!str_ptr.compare_exchange_weak(
            old_str, new_str,
            std::memory_order_release,
            std::memory_order_acquire)) {
            // 재시도
        }
        
        std::rcu_retire(domain, old_str, 
                       [](std::string* p) { delete p; });
    }
    
    ~SafeRCUString() {
        std::rcu_barrier(domain);
        delete str_ptr.load();
    }
};
```

올바른 메모리 순서 지정은 다음을 보장한다:

1. 쓰기 스레드가 새 데이터를 완전히 초기화한 후에만 포인터가 교체된다
2. 읽기 스레드는 교체된 포인터를 읽으면 완전히 초기화된 데이터를 본다
3. 회수는 모든 읽기 스레드가 이전 데이터를 더 이상 참조하지 않을 때만 발생한다

이러한 보장을 통해 RCU는 use-after-free 같은 메모리 안전성 문제를 방지하면서도 고성능을 유지한다.
