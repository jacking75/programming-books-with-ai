## 5.1 조인(JOIN)의 이해와 활용

### JOIN의 개념
조인은 두 개 이상의 테이블을 결합하여 데이터를 조회하는 기법이다. 관계형 데이터베이스의 정규화된 테이블 구조에서 필수적이다.

게임에서 유저 정보와 캐릭터 정보가 분리되어 있을 때, 유저의 모든 캐릭터를 조회하려면 JOIN이 필요하다.

**테이블 구조 예시:**

```
users 테이블                    characters 테이블
┌─────────┬──────────┐         ┌──────────────┬─────────┬────────────┐
│ user_id │ username │         │ character_id │ user_id │ char_name  │
├─────────┼──────────┤         ├──────────────┼─────────┼────────────┤
│    1    │  hero1   │         │      1       │    1    │  warrior1  │
│    2    │  hero2   │         │      2       │    1    │  mage1     │
│    3    │  hero3   │         │      3       │    2    │  archer1   │
└─────────┴──────────┘         └──────────────┴─────────┴────────────┘
```

### INNER JOIN, LEFT JOIN, RIGHT JOIN

#### INNER JOIN
두 테이블에서 조건이 일치하는 행만 반환한다. 가장 많이 사용되는 조인이다.

```sql
-- SQL 방식
SELECT u.username, c.char_name, c.level
FROM users u
INNER JOIN characters c ON u.user_id = c.user_id;
```

```csharp
// SQLKata 방식
using (var db = dbContext.CreateQueryFactory())
{
    var result = db.Query("users as u")
                  .Join("characters as c", "u.user_id", "c.user_id")
                  .Select("u.username", "c.char_name", "c.level")
                  .Get();
    
    foreach (var row in result)
    {
        Console.WriteLine($"{row.username}의 캐릭터: {row.char_name} (레벨 {row.level})");
    }
}
```

**INNER JOIN 동작 원리:**

```
users                    characters              결과
┌──────┬──────┐         ┌──────┬────────┐      ┌──────┬────────┐
│  1   │ hero1│    ⟷   │  1   │warrior1│  →  │hero1 │warrior1│
│  2   │ hero2│    ⟷   │  2   │ mage1  │  →  │hero1 │ mage1  │
│  3   │ hero3│    ✗   │  3   │archer1 │  →  │hero2 │archer1 │
└──────┴──────┘         └──────┴────────┘      └──────┴────────┘
                                                (user_id 3은 제외됨)
```

#### LEFT JOIN (LEFT OUTER JOIN)
왼쪽 테이블의 모든 행을 반환하고, 오른쪽 테이블에 일치하는 데이터가 없으면 NULL을 반환한다.

```sql
-- SQL 방식
SELECT u.username, c.char_name
FROM users u
LEFT JOIN characters c ON u.user_id = c.user_id;
```

```csharp
// SQLKata 방식
using (var db = dbContext.CreateQueryFactory())
{
    var result = db.Query("users as u")
                  .LeftJoin("characters as c", "u.user_id", "c.user_id")
                  .Select("u.username", "c.char_name")
                  .Get();
    
    foreach (var row in result)
    {
        string charName = row.char_name ?? "캐릭터 없음";
        Console.WriteLine($"{row.username}: {charName}");
    }
}
```

**LEFT JOIN 동작 원리:**

```
users                    characters              결과
┌──────┬──────┐         ┌──────┬────────┐      ┌──────┬────────┐
│  1   │ hero1│    ⟷   │  1   │warrior1│  →  │hero1 │warrior1│
│  2   │ hero2│    ⟷   │  2   │ mage1  │  →  │hero1 │ mage1  │
│  3   │ hero3│    ✗   │  3   │archer1 │  →  │hero2 │archer1 │
└──────┴──────┘         └──────┴────────┘      │hero3 │  NULL  │
                                                └──────┴────────┘
                                                (모든 유저 포함)
```

**사용 사례: 캐릭터가 없는 유저 찾기**

```csharp
// 캐릭터를 생성하지 않은 유저 조회
using (var db = dbContext.CreateQueryFactory())
{
    var usersWithoutCharacters = db.Query("users as u")
                                  .LeftJoin("characters as c", "u.user_id", "c.user_id")
                                  .WhereNull("c.character_id")
                                  .Select("u.user_id", "u.username")
                                  .Get();
    
    Console.WriteLine("캐릭터를 생성하지 않은 유저:");
    foreach (var user in usersWithoutCharacters)
    {
        Console.WriteLine($"- {user.username}");
    }
}
```

#### RIGHT JOIN (RIGHT OUTER JOIN)
LEFT JOIN의 반대다. 오른쪽 테이블의 모든 행을 반환한다. 실무에서는 거의 사용하지 않는다. LEFT JOIN으로 대체 가능하다.

```csharp
// SQLKata는 RightJoin도 지원한다
var result = db.Query("users as u")
              .RightJoin("characters as c", "u.user_id", "c.user_id")
              .Select("u.username", "c.char_name")
              .Get();
```

### 다중 테이블 조인
게임에서는 유저-캐릭터-인벤토리-아이템처럼 여러 테이블을 함께 조회해야 하는 경우가 많다.

**테이블 관계:**

```
users ──(1:N)── characters ──(1:N)── inventories ──(N:1)── items
```

```csharp
// 유저별 보유 아이템 조회
using (var db = dbContext.CreateQueryFactory())
{
    var userItems = db.Query("users as u")
                     .Join("characters as c", "u.user_id", "c.user_id")
                     .Join("inventories as inv", "c.character_id", "inv.character_id")
                     .Join("items as it", "inv.item_id", "it.item_id")
                     .Where("u.user_id", 1)
                     .Select(
                         "u.username",
                         "c.char_name",
                         "it.item_name",
                         "inv.quantity",
                         "it.rarity"
                     )
                     .Get();
    
    Console.WriteLine("유저의 전체 아이템:");
    foreach (var item in userItems)
    {
        Console.WriteLine($"[{item.char_name}] {item.item_name} x{item.quantity} ({item.rarity})");
    }
}
```

생성되는 SQL:

```sql
SELECT u.username, c.char_name, it.item_name, inv.quantity, it.rarity
FROM users as u
INNER JOIN characters as c ON u.user_id = c.user_id
INNER JOIN inventories as inv ON c.character_id = inv.character_id
INNER JOIN items as it ON inv.item_id = it.item_id
WHERE u.user_id = 1
```

**조인 실행 순서:**

```
1. users (u.user_id = 1)
   ↓
2. JOIN characters (u.user_id = c.user_id)
   ↓
3. JOIN inventories (c.character_id = inv.character_id)
   ↓
4. JOIN items (inv.item_id = it.item_id)
   ↓
5. SELECT 결과 반환
```

```csharp
var optimized = db.Query("users as u")
```

#### 4. 서브쿼리 대신 JOIN 사용

```csharp

var withSubquery = db.Query("users")
                    .WhereIn("user_id", q => q
```

#### 5. STRAIGHT_JOIN으로 조인 순서 강제
MySQL은 자동으로 조인 순서를 최적화하지만, 때로는 수동으로 지정하는 것이 더 빠를 수 있다.

```csharp
// SQLKata에서는 SelectRaw로 처리
var result = db.Query()
              .SelectRaw(@"
                  SELECT STRAIGHT_JOIN u.username, c.char_name
                  FROM users u
                  JOIN characters c ON u.user_id = c.user_id

              ")
              .Get();
```

### 조인 실전 예제: 길드 시스템

```sql
-- 길드 테이블 생성
CREATE TABLE guilds (
    guild_id INT AUTO_INCREMENT PRIMARY KEY,
    guild_name VARCHAR(100) NOT NULL UNIQUE,
    master_user_id BIGINT NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (master_user_id) REFERENCES users(user_id)
) ENGINE=InnoDB;

CREATE TABLE guild_members (
    member_id BIGINT AUTO_INCREMENT PRIMARY KEY,
    guild_id INT NOT NULL,
    user_id BIGINT NOT NULL,
    join_date DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    member_rank ENUM('master', 'officer', 'member') NOT NULL DEFAULT 'member',
    
    FOREIGN KEY (guild_id) REFERENCES guilds(guild_id) ON DELETE CASCADE,
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    UNIQUE KEY uk_user (user_id),
    INDEX idx_guild (guild_id)
) ENGINE=InnoDB;
```

```csharp
public class GuildService
{
    private readonly DatabaseContext _dbContext;
    
    public GuildService(DatabaseContext dbContext)
    {
        _dbContext = dbContext;
    }
    
    /// <summary>
    /// 길드 멤버 목록 조회 (3-way JOIN)
    /// </summary>
    public List<GuildMemberInfo> GetGuildMembers(int guildId)
    {
        using (var db = _dbContext.CreateQueryFactory())
        {
            var members = db.Query("guilds as g")
                           .Join("guild_members as gm", "g.guild_id", "gm.guild_id")
                           .Join("users as u", "gm.user_id", "u.user_id")
                           .Where("g.guild_id", guildId)
                           .Select(
                               "u.user_id",
                               "u.username",
                               "u.level",
                               "gm.member_rank",
                               "gm.join_date"
                           )
                           .OrderByDesc("gm.member_rank")
                           .ThenByDesc("u.level")
                           .Get<GuildMemberInfo>();
            
            return members.ToList();
        }
    }
    
    /// <summary>
    /// 길드 전체 통계 조회 (집계 함수와 JOIN)
    /// </summary>
    public GuildStatistics GetGuildStats(int guildId)
    {
        using (var db = _dbContext.CreateQueryFactory())
        {
            var stats = db.Query("guilds as g")
                         .LeftJoin("guild_members as gm", "g.guild_id", "gm.guild_id")
                         .LeftJoin("users as u", "gm.user_id", "u.user_id")
                         .Where("g.guild_id", guildId)
                         .Select("g.guild_name")
                         .SelectRaw("COUNT(gm.member_id) as member_count")
                         .SelectRaw("AVG(u.level) as avg_level")
                         .SelectRaw("MAX(u.level) as max_level")
                         .SelectRaw("SUM(u.gold) as total_gold")
                         .GroupBy("g.guild_id", "g.guild_name")
                         .First<GuildStatistics>();
            
            return stats;
        }
    }
}

public class GuildMemberInfo
{
    public long user_id { get; set; }
    public string username { get; set; }
    public int level { get; set; }
    public string member_rank { get; set; }
    public DateTime join_date { get; set; }
}

public class GuildStatistics
{
    public string guild_name { get; set; }
    public int member_count { get; set; }
    public decimal avg_level { get; set; }
    public int max_level { get; set; }
    public long total_gold { get; set; }
}
```

## 5.2 인덱스(Index) 설계

### 인덱스의 원리와 B-Tree 구조
인덱스는 책의 색인과 같다. 책 전체를 읽지 않고 색인을 통해 원하는 페이지를 빠르게 찾을 수 있다. 데이터베이스도 마찬가지로 인덱스를 통해 원하는 데이터를 빠르게 찾는다.

**인덱스가 없는 경우 (Full Table Scan):**

```
SELECT * FROM users WHERE username = 'hero1';

┌────────────────────────────────┐
│ 100만 개 행을 순차적으로 검색   │
│ Row 1: admin                    │
│ Row 2: player1                  │
│ Row 3: gamer99                  │
│ ...                             │
│ Row 500,000: hero1 ← 찾음!      │
└────────────────────────────────┘
검색 시간: 수 초
```

**인덱스가 있는 경우 (Index Scan):**

```
SELECT * FROM users WHERE username = 'hero1';

B-Tree 인덱스를 통해 검색
       [m]
      /   \
   [h]     [t]
   / \     / \
 [e][j] [p][z]
  |
[hero1] ← 빠르게 찾음!

검색 시간: 수 밀리초
```

#### B-Tree 구조

MySQL의 InnoDB 엔진은 B-Tree(Balanced Tree) 구조를 사용한다.

```
                    [50, 100]               ← Root Node
                   /    |    \
                 /      |      \
         [10,30,40]  [60,80]  [110,150]    ← Branch Nodes
         /  |  |  \   /  |  \   /   |   \
       [1] [15] [35] [45] [55] [70] [90] [105] [120] [160]  ← Leaf Nodes
        ↓   ↓    ↓    ↓    ↓    ↓    ↓     ↓     ↓     ↓
      Data Data Data Data Data Data Data  Data  Data  Data
```

**B-Tree의 특징:**

- 균형 잡힌 트리 구조로 모든 리프 노드의 깊이가 같다
- 검색, 삽입, 삭제가 모두 O(log n) 시간 복잡도를 가진다
- 범위 검색에 효율적이다

**검색 과정:**

```
WHERE user_id = 70 을 찾는다면:

1. Root [50, 100] 확인 → 70은 50과 100 사이
2. Branch [60, 80] 확인 → 70은 60과 80 사이
3. Leaf [70] 찾음 → 데이터 반환

3번의 디스크 I/O로 검색 완료 (100만 건 중에서도!)
```

### 인덱스 생성 전략

#### 1. 인덱스를 생성해야 하는 경우

```sql
-- ✅ WHERE 절에 자주 사용되는 컬럼
CREATE INDEX idx_username ON users(username);

-- ✅ JOIN 조건에 사용되는 컬럼
CREATE INDEX idx_user_id ON characters(user_id);

-- ✅ ORDER BY에 자주 사용되는 컬럼
CREATE INDEX idx_level ON users(level);

-- ✅ GROUP BY에 사용되는 컬럼
CREATE INDEX idx_item_type ON items(item_type);
```

```csharp
// C#에서 인덱스 생성
using (var db = dbContext.CreateQueryFactory())
{
    db.Statement("CREATE INDEX idx_username ON users(username)");
    db.Statement("CREATE INDEX idx_level_desc ON users(level DESC)");
}
```

#### 2. 인덱스를 생성하지 말아야 하는 경우

```sql
-- ❌ 데이터가 적은 테이블 (수백 건 이하)
-- 인덱스 오버헤드가 더 크다

-- ❌ 카디널리티가 낮은 컬럼 (중복값이 많은 컬럼)
-- 예: is_banned (true/false 두 가지 값만)
CREATE INDEX idx_is_banned ON users(is_banned);  -- 비효율적

-- ❌ 자주 수정되는 컬럼
-- INSERT/UPDATE/DELETE 시 인덱스도 함께 수정되어 느려짐

-- ❌ 텍스트가 긴 컬럼
CREATE INDEX idx_description ON items(description);  -- description이 TEXT 타입이면 비효율적
```

#### 3. 카디널리티(Cardinality)
카디널리티는 컬럼의 고유값 비율이다. 카디널리티가 높을수록 인덱스 효율이 좋다.

```
카디널리티 = (고유값 개수 / 전체 행 수) × 100

✅ 높은 카디널리티 (인덱스 효과 좋음)
- user_id: 1,000,000개 / 1,000,000개 = 100%
- email: 990,000개 / 1,000,000개 = 99%

❌ 낮은 카디널리티 (인덱스 효과 나쁨)
- gender: 3개 / 1,000,000개 = 0.0003%
- is_banned: 2개 / 1,000,000개 = 0.0002%
```

### 복합 인덱스와 커버링 인덱스

#### 복합 인덱스 (Composite Index)
여러 컬럼을 조합한 인덱스다. 컬럼 순서가 매우 중요하다.

```sql
-- 복합 인덱스 생성
CREATE INDEX idx_level_gold ON users(level, gold);
```

**컬럼 순서 규칙:**

```
1. WHERE 절에 = 조건으로 사용되는 컬럼을 앞에
2. 범위 조건(<, >, BETWEEN)은 뒤에
3. ORDER BY에 사용되는 컬럼을 고려
```

```sql
-- ✅ 인덱스 활용 가능
SELECT * FROM users WHERE level = 50 AND gold > 10000;
-- idx_level_gold 인덱스 사용

-- ✅ 인덱스 부분 활용
SELECT * FROM users WHERE level = 50;
-- idx_level_gold 인덱스의 level 부분만 사용

-- ❌ 인덱스 사용 불가
SELECT * FROM users WHERE gold > 10000;
-- level 조건이 없으면 인덱스 사용 불가 (가장 왼쪽 컬럼이 아님)
```

**복합 인덱스 시각화:**

```
CREATE INDEX idx_level_gold ON users(level, gold);

인덱스 구조:
┌─────────────────────────┐
│ level=10, gold=100      │
│ level=10, gold=500      │
│ level=10, gold=1000     │ ← level로 먼저 정렬
│ level=20, gold=200      │
│ level=20, gold=800      │
│ level=30, gold=300      │
└─────────────────────────┘
     ↓ gold로 2차 정렬
```

**실전 예제:**

```csharp
// 복합 인덱스 생성
using (var db = dbContext.CreateQueryFactory())
{
    // 랭킹 조회에 최적화된 인덱스
    db.Statement("CREATE INDEX idx_score_time ON rankings(score DESC, clear_time ASC)");
}

// 인덱스 활용 쿼리
public List<RankingInfo> GetTopRankings(int limit = 100)
{
    using (var db = _dbContext.CreateQueryFactory())
    {
        // idx_score_time 인덱스를 활용하여 빠르게 조회
        var rankings = db.Query("rankings as r")
                        .Join("users as u", "r.user_id", "u.user_id")
                        .Select(
                            "r.ranking_id",
                            "u.username",
                            "r.score",
                            "r.clear_time"
                        )
                        .OrderByDesc("r.score")
                        .ThenBy("r.clear_time")
                        .Limit(limit)
                        .Get<RankingInfo>();
        
        return rankings.ToList();
    }
}
```

#### 커버링 인덱스 (Covering Index)
쿼리가 필요로 하는 모든 컬럼을 인덱스가 포함하고 있으면, 테이블에 접근하지 않고 인덱스만으로 결과를 반환할 수 있다. 이를 커버링 인덱스라고 한다.

```sql
-- 커버링 인덱스 생성
CREATE INDEX idx_username_level ON users(username, level);

-- ✅ 커버링 인덱스 활용 (매우 빠름)
SELECT username, level FROM users WHERE username = 'hero1';
-- 인덱스만 읽고 테이블 접근 안 함

-- ❌ 커버링 인덱스 활용 불가
SELECT username, level, gold FROM users WHERE username = 'hero1';
-- gold는 인덱스에 없으므로 테이블 접근 필요
```

**커버링 인덱스 효과:**

```
일반 인덱스:
1. 인덱스에서 username = 'hero1' 찾기
2. 인덱스에서 Row ID 확인
3. 테이블에서 Row ID로 전체 데이터 읽기 ← 추가 I/O
4. 결과 반환

커버링 인덱스:
1. 인덱스에서 username = 'hero1' 찾기
2. 인덱스에서 바로 username, level 읽기
3. 결과 반환 (테이블 접근 없음) ← 빠름!
```

```csharp
// 커버링 인덱스 활용 예제
public class UserLevelInfo
{
    public string username { get; set; }
    public int level { get; set; }
}

public List<UserLevelInfo> GetHighLevelUsers(int minLevel)
{
    using (var db = _dbContext.CreateQueryFactory())
    {
        // idx_level_username 커버링 인덱스 활용
        db.Statement("CREATE INDEX idx_level_username ON users(level DESC, username)");
        
        // 인덱스만으로 처리 가능 (매우 빠름)
        var users = db.Query("users")
                     .Where("level", ">=", minLevel)
                     .Select("username", "level")  // 인덱스에 포함된 컬럼만 조회
                     .OrderByDesc("level")
                     .Limit(100)
                     .Get<UserLevelInfo>();
        
        return users.ToList();
    }
}
```

### EXPLAIN을 통한 쿼리 분석
EXPLAIN은 MySQL이 쿼리를 어떻게 실행하는지 보여준다. 인덱스 사용 여부와 성능 문제를 파악할 수 있다.

```sql
EXPLAIN SELECT * FROM users WHERE username = 'hero1';
```

```csharp
// C#에서 EXPLAIN 실행
using (var db = dbContext.CreateQueryFactory())
{
    var explainResult = db.Select("EXPLAIN SELECT * FROM users WHERE username = ?", "hero1");
    
    foreach (var row in explainResult)
    {
        Console.WriteLine($"Type: {row.type}");
        Console.WriteLine($"Key: {row.key}");
        Console.WriteLine($"Rows: {row.rows}");
    }
}
```

**EXPLAIN 결과 해석:**

```
+----+-------------+-------+-------+---------------+----------+---------+-------+------+-------+
| id | select_type | table | type  | possible_keys | key      | key_len | ref   | rows | Extra |
+----+-------------+-------+-------+---------------+----------+---------+-------+------+-------+
|  1 | SIMPLE      | users | ref   | idx_username  | idx_username | 202  | const |    1 | NULL  |
+----+-------------+-------+-------+---------------+----------+---------+-------+------+-------+
```

**주요 컬럼:**

- **type**: 조인 타입 (성능 순서: const > eq_ref > ref > range > index > ALL)
  - `const`: Primary Key나 Unique 인덱스로 단일 행 조회 (가장 빠름)
  - `eq_ref`: JOIN 시 Primary Key나 Unique 인덱스 사용
  - `ref`: Non-Unique 인덱스로 조회
  - `range`: 범위 검색
  - `index`: 인덱스 전체 스캔
  - `ALL`: 테이블 전체 스캔 (가장 느림, 피해야 함)

- **key**: 실제 사용된 인덱스
  - `NULL`이면 인덱스를 사용하지 않음 (문제!)

- **rows**: 검사할 예상 행 수
  - 숫자가 작을수록 좋다

- **Extra**: 추가 정보
  - `Using index`: 커버링 인덱스 사용 (좋음!)
  - `Using where`: WHERE 절 필터링
  - `Using filesort`: 정렬을 위한 추가 작업 필요 (느림)
  - `Using temporary`: 임시 테이블 사용 (느림)

**인덱스 비교 예제:**

```csharp
public void CompareIndexPerformance()
{

    }

    {
        Console.WriteLine($"Type: {row.type}");
        Console.WriteLine($"Key: {row.key ?? "NULL"}");
        Console.WriteLine($"Rows: {row.rows}");

    }
```

## 5.3 게임 랭킹 시스템 구현
실시간으로 업데이트되는 게임 랭킹 시스템을 구현한다. 수백만 명의 유저 중 상위 100명을 빠르게 조회해야 한다.

### 리더보드 테이블 설계

```sql
CREATE TABLE rankings (
    ranking_id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    score BIGINT NOT NULL DEFAULT 0,
    clear_time INT NOT NULL COMMENT '클리어 시간(초)',
    ranking_date DATE NOT NULL,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    
    -- 복합 인덱스: 점수 내림차순, 시간 오름차순
    INDEX idx_score_time (score DESC, clear_time ASC),
    
    -- 일자별 조회용 인덱스
    INDEX idx_date_score (ranking_date, score DESC),
    
    -- 유저별 최고 기록 조회용
    INDEX idx_user_score (user_id, score DESC),
    
    -- 같은 날짜에 같은 유저는 하나의 기록만
    UNIQUE KEY uk_user_date (user_id, ranking_date)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 주간/월간 랭킹을 위한 집계 테이블
CREATE TABLE ranking_summary (
    summary_id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    period_type ENUM('daily', 'weekly', 'monthly') NOT NULL,
    period_start DATE NOT NULL,
    total_score BIGINT NOT NULL DEFAULT 0,
    best_score BIGINT NOT NULL DEFAULT 0,
    play_count INT NOT NULL DEFAULT 0,
    
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    INDEX idx_period_score (period_type, period_start, total_score DESC),
    UNIQUE KEY uk_user_period (user_id, period_type, period_start)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

```csharp
    public MyRankingInfo GetMyRanking(long userId, DateTime date)
    {
        using (var db = _dbContext.CreateQueryFactory())
        {
            // 내 기록 조회
            var myRecord = db.Query("rankings")
                            .Where("user_id", userId)
                            .WhereDate("ranking_date", date)
                            .Select("score", "clear_time")
                            .First<dynamic>();
            
            if (myRecord == null)
            {
                return new MyRankingInfo { rank = -1, message = "오늘 플레이 기록이 없다." };
            }
            
            long myScore = myRecord.score;
            int myClearTime = myRecord.clear_time;
            
            // 나보다 높은 순위의 사람 수 카운트
            var higherRankCount = db.Query("rankings")
                                   .WhereDate("ranking_date", date)
                                   .Where(q => q
                                       .Where("score", ">", myScore)
                                       .OrWhere(sub => sub
                                           .Where("score", "=", myScore)
                                           .Where("clear_time", "<", myClearTime)
                                       )
                                   )
                                   .Count<int>();
            
            return new MyRankingInfo
            {
                rank = higherRankCount + 1,
                score = myScore,
                clear_time = myClearTime,
                message = "순위 조회 완료"

    {
        using (var db = _dbContext.CreateQueryFactory())
        {
            // 1. 내 순위 확인

            {
                return new List<RankingEntry>();

            int endRank = myRanking.rank + range;

            var rankings = db.Query("rankings as r")
                            .Join("users as u", "r.user_id", "u.user_id")
                            .WhereDate("r.ranking_date", DateTime.Today)
                            .Select(
                                "r.user_id",
                                "u.username",

                                "r.clear_time"
                            )
                            .OrderByDesc("r.score")
                            .ThenBy("r.clear_time")
                            .Offset(startRank - 1)
                            .Limit(endRank - startRank + 1)
                            .Get();
            
            var result = new List<RankingEntry>();
            int currentRank = startRank;
            
            foreach (var row in rankings)
            {
                result.Add(new RankingEntry
                {
                    rank = currentRank++,
                    user_id = row.user_id,
                    username = row.username,
                    score = row.score,
                    clear_time = row.clear_time,
                    is_me = row.user_id == userId
                });
            }
            
            return result;
        }
```

```csharp
public class RankingService
{
    // ... 이전 코드 ...
    
    /// <summary>
    /// 기록 등록 및 랭킹 업데이트
    /// </summary>
    public bool SubmitScore(long userId, long score, int clearTime)
    {
        using (var db = _dbContext.CreateQueryFactory())
        {
            db.Connection.Open();
            using (var transaction = db.Connection.BeginTransaction())
            {
                try
                {
                    db.SetTransaction(transaction);
                    
                    var today = DateTime.Today;
                    
                    // 오늘 기존 기록 확인
                    var existingRecord = db.Query("rankings")
                                          .Where("user_id", userId)
                                          .WhereDate("ranking_date", today)
                                          .First<dynamic>();
                    
                    if (existingRecord != null)
                    {
                        // 기존 기록보다 좋으면 업데이트
                        long oldScore = existingRecord.score;
                        int oldTime = existingRecord.clear_time;
                        
                        bool isBetter = score > oldScore || 
                                       (score == oldScore && clearTime < oldTime);
                        
                        if (isBetter)
                        {
                            db.Query("rankings")
                              .Where("user_id", userId)
                              .WhereDate("ranking_date", today)
                              .Update(new
                              {
                                  score = score,
                                  clear_time = clearTime,
                                  updated_at = DateTime.Now
                              });
                            
                            Console.WriteLine("새로운 최고 기록 갱신!");
                        }
                        else
                        {
                            Console.WriteLine("기존 기록이 더 좋다.");
                            transaction.Rollback();
                            return false;
                        }
                    }
                    else
                    {
                        // 새 기록 삽입
                        db.Query("rankings").Insert(new
                        {
                            user_id = userId,
                            score = score,
                            clear_time = clearTime,
                            ranking_date = today,
                            updated_at = DateTime.Now
                        });
                        
                        Console.WriteLine("첫 기록 등록!");
                    }
                    
                    // 주간 집계 업데이트
                    UpdateWeeklySummary(db, userId, score, today);
                    
                    transaction.Commit();
                    return true;
                }
                catch (Exception ex)
                {
                    transaction.Rollback();
                    Console.WriteLine($"기록 등록 실패: {ex.Message}");
                    return false;
                }
            }
        }
    }
    
    /// <summary>
    /// 주간 집계 업데이트
    /// </summary>
    private void UpdateWeeklySummary(QueryFactory db, long userId, long score, DateTime date)
    {
        // 주 시작일 계산 (월요일 기준)
        int daysFromMonday = ((int)date.DayOfWeek - 1 + 7) % 7;
        var weekStart = date.AddDays(-daysFromMonday);
        
        var summary = db.Query("ranking_summary")
                       .Where("user_id", userId)
                       .Where("period_type", "weekly")
                       .Where("period_start", weekStart)
                       .First<dynamic>();
        
        if (summary != null)
        {
            // 기존 집계 업데이트
            db.Query("ranking_summary")
              .Where("user_id", userId)
              .Where("period_type", "weekly")
              .Where("period_start", weekStart)
              .Update(new
              {
                  total_score = (long)summary.total_score + score,
                  best_score = Math.Max((long)summary.best_score, score),
                  play_count = (int)summary.play_count + 1
              });
        }
        else
        {
            // 새 집계 생성
            db.Query("ranking_summary").Insert(new
            {
                user_id = userId,
                period_type = "weekly",
                period_start = weekStart,
                total_score = score,
                best_score = score,
                play_count = 1
            });
        }
    }
}
```

## 정리

Day 5에서는 고급 쿼리 기법과 성능 최적화의 핵심인 인덱스를 배웠다. JOIN을 활용하면 정규화된 테이블에서 효율적으로 데이터를 조회할 수 있다. INNER JOIN, LEFT JOIN의 차이를 이해하고 상황에 맞게 사용해야 한다.

인덱스는 데이터베이스 성능의 핵심이다. B-Tree 구조를 이해하고, 카디널리티가 높은 컬럼에 인덱스를 생성해야 한다. 복합 인덱스는 컬럼 순서가 중요하며, 커버링 인덱스를 활용하면 테이블 접근 없이 빠르게 조회할 수 있다.

EXPLAIN으로 쿼리 실행 계획을 분석하여 인덱스 사용 여부를 확인해야 한다. type이 ALL이면 전체 스캔이므로 인덱스를 추가해야 한다.

게임 랭킹 시스템 구현을 통해 실시간으로 수백만 명의 데이터를 효율적으로 조회하는 방법을 익혔다. 복합 인덱스와 집계 테이블을 활용하면 밀리초 단위로 랭킹을 조회할 수 있다.

다음 Day 6에서는 친구 시스템, 우편함, 던전 보상 등 실전 게임 콘텐츠 시스템을 구현한다.

**핵심 요약:**

- JOIN으로 여러 테이블을 결합하여 조회한다
- INNER JOIN은 일치하는 행만, LEFT JOIN은 왼쪽 테이블 전체를 반환한다
- 인덱스는 B-Tree 구조로 빠른 검색을 제공한다
- 카디널리티가 높은 컬럼에 인덱스를 생성한다
- 복합 인덱스는 컬럼 순서가 중요하다 (WHERE = 조건을 앞에)
- 커버링 인덱스로 테이블 접근을 최소화한다
- EXPLAIN으로 쿼리 실행 계획을 분석한다
