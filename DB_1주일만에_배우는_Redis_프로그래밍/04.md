# Chapter 4. 플레이어 데이터와 인벤토리
온라인 게임에서 플레이어 데이터는 데이터베이스에 영구 저장되지만, 매번 DB에서 읽으면 성능 병목이 발생한다. Redis를 캐시로 활용하면 응답 속도를 수십 배 개선할 수 있다. 이번 장에서는 플레이어 프로필과 인벤토리를 Redis로 효율적으로 관리하는 방법을 다룬다.

## 4.1 플레이어 프로필 캐싱

### DB와 Redis 연동 전략
게임 서버는 일반적으로 MySQL, PostgreSQL 같은 관계형 DB를 사용하여 플레이어 데이터를 영구 저장한다. Redis는 이 데이터의 캐시 계층으로 동작한다.

**계층 구조**

```
┌──────────────┐
│   Client     │
└──────┬───────┘
       │ API Request
       ↓
┌──────────────────┐
│  Game Server     │
└──────┬───────────┘
       │
       ├─→ Redis (Cache)     ← 빠름 (< 1ms)
       │   - 자주 읽는 데이터
       │   - 임시 데이터
       │
       └─→ Database (Primary) ← 느림 (10~100ms)
           - 영구 저장
           - 정합성 보장
```

**캐싱 전략 비교**

게임에서 주로 사용하는 캐싱 패턴은 세 가지다:

1. **Cache-Aside (Lazy Loading)**
   - 요청 시점에 캐시를 확인하고 없으면 DB 조회
   - 가장 일반적인 패턴
   - 읽기가 많은 게임에 적합

2. **Write-Through**
   - 쓰기 시 캐시와 DB 동시 업데이트
   - 캐시가 항상 최신 상태
   - 쓰기 성능이 느림

3. **Write-Behind (Write-Back)**
   - 쓰기를 캐시에만 하고 비동기로 DB 반영
   - 쓰기 성능이 빠름
   - 장애 시 데이터 손실 위험

게임에서는 주로 Cache-Aside를 사용하고, 중요한 데이터(재화, 아이템)는 Write-Through를 병행한다.

### Cache-Aside 패턴 구현
Cache-Aside는 "읽을 때 캐시 확인 → 없으면 DB 조회 → 캐시에 저장" 흐름으로 동작한다.

```
┌─────────────────────────────────┐
│  플레이어 프로필 조회            │
└─────────────────────────────────┘
         │
         ↓
    ┌─────────┐
    │ Redis?  │
    └────┬────┘
         │
    ┌────┴────┐
    │         │
   Yes       No
    │         │
    │         ↓
    │    ┌────────┐
    │    │  DB    │
    │    └───┬────┘
    │        │
    │        ↓
    │   ┌─────────┐
    │   │ Redis에 │
    │   │  저장   │
    │   └───┬─────┘
    │       │
    └───────┴──→ Return
```

```csharp
{
    private readonly RedisConnection _redis;
    private readonly IDbConnection _db; // 실제 DB 연결
    
    public PlayerProfileService(RedisConnection redis, IDbConnection db)
    {
        _redis = redis;
        _db = db;
    }
    
    public async Task<PlayerProfile> GetProfileAsync(string playerId)
    {
        // 1. Redis에서 조회
        var cacheKey = $"player:{playerId}:profile";
        var cache = new RedisString<PlayerProfile>(
            _redis, 
            cacheKey, 
            TimeSpan.FromMinutes(30) // 30분 캐시
        );
        
        var cachedProfile = await cache.GetAsync();
        
        if (cachedProfile.HasValue)
        {
            Console.WriteLine($"Cache HIT: {playerId}");
            return cachedProfile.Value;
        }
        
        // 2. Redis에 없으면 DB 조회

        var profile = await LoadProfileFromDbAsync(playerId);
        
        if (profile == null)
        {
            throw new Exception($"Player {playerId} not found");
        }
        
        // 3. Redis에 저장
        await cache.SetAsync(profile);
        
        return profile;
    }
```

```csharp
public async Task InvalidateProfileCacheAsync(string playerId)
{
    var cacheKey = $"player:{playerId}:profile";
    var db = _redis.GetConnection().GetDatabase();
    await db.KeyDeleteAsync(cacheKey);
    
    Console.WriteLine($"Cache invalidated: {playerId}");
}
```

### 데이터 일관성 유지하기
캐시를 사용하면 DB와 캐시 사이에 데이터 불일치가 발생할 수 있다. 이를 방지하는 전략을 알아본다.

**문제 상황**

```
시간순:
1. 서버 A가 DB에서 gold=1000 읽음
2. 서버 B가 gold를 500 증가 → DB: 1500
3. 서버 B가 캐시 업데이트 → Cache: 1500
4. 서버 A가 gold를 100 증가 → 1100
5. 서버 A가 캐시 업데이트 → Cache: 1100

결과: DB=1500, Cache=1100 (불일치!)
```

**해결 방법 1: 캐시 삭제**

업데이트 후 캐시를 삭제하면 다음 조회 시 DB에서 최신 데이터를 가져온다.

```csharp
public async Task AddGoldSafeAsync(string playerId, long amount)
{
    // 1. DB 업데이트
    await _db.ExecuteAsync(
        "UPDATE players SET gold = gold + @Amount WHERE player_id = @PlayerId",
        new { PlayerId = playerId, Amount = amount }
    );
    
    // 2. 캐시 삭제 (다음 조회 시 DB에서 로드)
    await InvalidateProfileCacheAsync(playerId);
}
```

**해결 방법 2: Redis에서 직접 증감**

Redis에서 원자적 연산으로 증감하고, 비동기로 DB에 반영한다.

```csharp
public async Task AddGoldWithRedisAsync(string playerId, long amount)
{
    var cacheKey = $"player:{playerId}:gold";
    var goldCache = new RedisString<long>(_redis, cacheKey, TimeSpan.FromMinutes(30));
    
    // 1. Redis에서 증감 (원자적 연산)
    var newGold = await goldCache.IncrementAsync(amount);
    
    // 2. 비동기로 DB 업데이트 (큐에 넣거나 주기적으로 flush)
    _ = Task.Run(async () =>
    {
        await _db.ExecuteAsync(
            "UPDATE players SET gold = @Gold WHERE player_id = @PlayerId",
            new { PlayerId = playerId, Gold = newGold }
        );
    });
    
    return newGold;
}
```

**해결 방법 3: 버전 관리**

낙관적 락(Optimistic Locking)으로 버전을 확인한다.

```csharp
public class PlayerProfileWithVersion
{
    public string PlayerId { get; set; }
    public long Gold { get; set; }
    public int Version { get; set; } // 버전 번호
}

public async Task<bool> UpdateGoldWithVersionAsync(
    string playerId, 
    long amount, 
    int expectedVersion)
{
    // 1. 캐시에서 버전 확인
    var profile = await GetProfileWithVersionAsync(playerId);
    
    if (profile.Version != expectedVersion)
    {
        return false; // 버전 불일치 (다른 곳에서 이미 수정됨)
    }
    
    // 2. DB 업데이트 (버전도 증가)
    var updated = await _db.ExecuteAsync(
        @"UPDATE players 
          SET gold = gold + @Amount, version = version + 1 
          WHERE player_id = @PlayerId AND version = @Version",
        new { PlayerId = playerId, Amount = amount, Version = expectedVersion }
    );
    
    if (updated == 0)
    {
        return false; // 동시성 충돌
    }
    
    // 3. 캐시 무효화
    await InvalidateProfileCacheAsync(playerId);
    
    return true;
}
```

**재시도 로직**

```csharp
public async Task AddGoldWithRetryAsync(string playerId, long amount, int maxRetries = 3)
{
    for (int i = 0; i < maxRetries; i++)
    {
        var profile = await GetProfileWithVersionAsync(playerId);

}
```

**분산 락 사용**

Redis 분산 락으로 동시 수정을 방지한다.

```csharp
public async Task AddGoldWithLockAsync(string playerId, long amount)
{
    var lockKey = $"lock:player:{playerId}";
    var db = _redis.GetConnection().GetDatabase();
    
    // 락 획득 시도 (5초 타임아웃, 10초 만료)
    var lockToken = Guid.NewGuid().ToString();
    var acquired = await db.StringSetAsync(
        lockKey, 
        lockToken, 
        TimeSpan.FromSeconds(10), 
        When.NotExists
    );
    
    if (!acquired)
    {
        throw new Exception("Could not acquire lock");
    }
    
    try
    {
        // 락을 획득했으므로 안전하게 업데이트
        var profile = await GetProfileAsync(playerId);
        profile.Gold += amount;
        await UpdateProfileAsync(profile);
    }
    finally
    {
        // 락 해제 (Lua 스크립트로 안전하게)
        var script = @"
            if redis.call('get', KEYS[1]) == ARGV[1] then
                return redis.call('del', KEYS[1])
            else
                return 0
            end
        ";
        await db.ScriptEvaluateAsync(script, new RedisKey[] { lockKey }, new RedisValue[] { lockToken });
    }
}
```

## 4.2 인벤토리 시스템
게임 인벤토리는 플레이어가 보유한 아이템 목록이다. Hash를 사용하면 아이템별로 효율적으로 관리할 수 있다.

### Hash를 활용한 아이템 저장

**인벤토리 데이터 구조**

```
player:1000:inventory (Hash)
  ├─ item:1001 → "5"   (검 5개)
  ├─ item:1002 → "3"   (방패 3개)
  ├─ item:2001 → "100" (포션 100개)
  └─ item:3005 → "1"   (레어템 1개)
```

Hash를 사용하면 전체 인벤토리를 한 키로 관리하면서 개별 아이템에 빠르게 접근할 수 있다.

**인벤토리 아이템 추가**

```csharp
public class InventoryService
{
    private readonly RedisConnection _redis;
    
    public InventoryService(RedisConnection redis)
    {
        _redis = redis;
    }
    
    public async Task AddItemAsync(string playerId, string itemId, int count)
    {
        var inventoryKey = $"player:{playerId}:inventory";
        var db = _redis.GetConnection().GetDatabase();
        
        // Hash 필드 증가 (없으면 0에서 시작)
        var newCount = await db.HashIncrementAsync(inventoryKey, itemId, count);
        
        Console.WriteLine($"Player {playerId} now has {newCount} of item {itemId}");
        
        // 인벤토리 TTL 연장 (플레이어가 활동 중)
        await db.KeyExpireAsync(inventoryKey, TimeSpan.FromHours(24));
    }
    
    public async Task<long> GetItemCountAsync(string playerId, string itemId)
    {
        var inventoryKey = $"player:{playerId}:inventory";
        var db = _redis.GetConnection().GetDatabase();
        
        var count = await db.HashGetAsync(inventoryKey, itemId);
        
        return count.HasValue ? (long)count : 0;
    }
}
```

### 아이템 개수 증감 (HINCRBY)
Redis의 `HINCRBY` 명령은 Hash 필드를 원자적으로 증감한다. 여러 요청이 동시에 들어와도 안전하다.

**아이템 사용**

```csharp
public async Task<bool> UseItemAsync(string playerId, string itemId, int count)
{
    if (count <= 0)
    {
        throw new ArgumentException("Count must be positive");
    }
    
    var inventoryKey = $"player:{playerId}:inventory";
    var db = _redis.GetConnection().GetDatabase();
    
    // 현재 개수 확인
    var currentCount = await db.HashGetAsync(inventoryKey, itemId);
    
    if (!currentCount.HasValue || (long)currentCount < count)
    {
        return false; // 아이템 부족
    }
    
    // 개수 감소
    var newCount = await db.HashIncrementAsync(inventoryKey, itemId, -count);
    
    // 0이 되면 필드 삭제
    if (newCount <= 0)
    {
        await db.HashDeleteAsync(inventoryKey, itemId);
        Console.WriteLine($"Item {itemId} removed from inventory (count reached 0)");
    }
    
    return true;
}
```

**아이템 교환**

두 아이템을 동시에 증감한다.

```csharp
public async Task<bool> ExchangeItemsAsync(
    string playerId, 
    string fromItemId, 
    int fromCount,
    string toItemId, 
    int toCount)
{
    var inventoryKey = $"player:{playerId}:inventory";
    var db = _redis.GetConnection().GetDatabase();
    
    // 트랜잭션으로 원자성 보장
    var tran = db.CreateTransaction();
    
    // 조건: fromItem이 충분한지 확인
    tran.AddCondition(Condition.HashGreaterThanOrEqual(inventoryKey, fromItemId, fromCount));
    
    // 명령: fromItem 감소, toItem 증가
    var decrementTask = tran.HashIncrementAsync(inventoryKey, fromItemId, -fromCount);
    var incrementTask = tran.HashIncrementAsync(inventoryKey, toItemId, toCount);
    
    bool committed = await tran.ExecuteAsync();
    
    if (!committed)
    {
        Console.WriteLine("Exchange failed: insufficient items");
        return false;
    }
    
    Console.WriteLine($"Exchanged {fromCount}x{fromItemId} → {toCount}x{toItemId}");
    return true;
}
```

**아이템 최대 개수 제한**

```csharp
public async Task<bool> AddItemWithMaxAsync(
    string playerId, 
    string itemId, 
    int count, 
    int maxCount = 999)
{
    var inventoryKey = $"player:{playerId}:inventory";
    var db = _redis.GetConnection().GetDatabase();
    
    // Lua 스크립트로 원자적 실행
    var script = @"
        local current = tonumber(redis.call('HGET', KEYS[1], ARGV[1])) or 0
        local toAdd = tonumber(ARGV[2])
        local maxCount = tonumber(ARGV[3])
        
        if current + toAdd > maxCount then
            return -1
        end
        
        return redis.call('HINCRBY', KEYS[1], ARGV[1], toAdd)
    ";
    
    var result = await db.ScriptEvaluateAsync(
        script,
        new RedisKey[] { inventoryKey },
        new RedisValue[] { itemId, count, maxCount }
    );
    
    if ((long)result == -1)
    {
        Console.WriteLine($"Cannot add item: would exceed max count {maxCount}");
        return false;
    }
    
    Console.WriteLine($"Added {count} of {itemId}, new count: {result}");
    return true;
}
```

### 인벤토리 전체 조회 최적화
플레이어 인벤토리가 수백 개 아이템을 가질 수 있다. 조회를 최적화하는 방법을 알아본다.

**페이징 조회**

Hash는 `HSCAN` 명령으로 커서 기반 페이징을 지원한다.

```csharp
public async Task<List<InventoryItem>> GetInventoryPagedAsync(
    string playerId, 
    int cursor = 0, 
    int pageSize = 50)
{
    var inventoryKey = $"player:{playerId}:inventory";
    var db = _redis.GetConnection().GetDatabase();
    
    var result = await db.HashScanAsync(inventoryKey, "*", pageSize, cursor);
    
    var items = new List<InventoryItem>();
    
    foreach (var entry in result)
    {
        items.Add(new InventoryItem
        {
            ItemId = entry.Name,
            Count = (int)entry.Value
        });
    }
    
    return items;
}
```

**특정 카테고리만 조회**

아이템 ID에 카테고리를 포함하면 필터링할 수 있다.

```

item:weapon:1002
```

```csharp
public async Task<Dictionary<string, int>> GetItemsByCategoryAsync(
    string playerId, 
    string category)
{
    var inventoryKey = $"player:{playerId}:inventory";
    var db = _redis.GetConnection().GetDatabase();
    
    var allItems = await db.HashGetAllAsync(inventoryKey);
    var filtered = new Dictionary<string, int>();
    
    var prefix = $"item:{category}:";
    
    foreach (var item in allItems)
    {
        if (item.Name.ToString().StartsWith(prefix))
        {
            filtered[item.Name] = (int)item.Value;
        }
    }
    
    return filtered;
```

**별도의 카테고리별 인덱스**

더 효율적인 방법은 카테고리별로 Set을 유지하는 것이다.

```
player:1000:inventory (Hash)
  ├─ item:weapon:1001 → "5"
  ├─ item:potion:2001 → "100"

player:1000:items:weapon (Set)
  ├─ item:weapon:1001
  ├─ item:weapon:1002

player:1000:items:potion (Set)
  ├─ item:potion:2001
```

```csharp
public async Task AddItemWithCategoryAsync(
    string playerId, 
    string itemId, 
    string category, 
    int count)
{
    var db = _redis.GetConnection().GetDatabase();
    
    // 1. 인벤토리에 아이템 추가
    var inventoryKey = $"player:{playerId}:inventory";
    await db.HashIncrementAsync(inventoryKey, itemId, count);
    
    // 2. 카테고리 인덱스에 추가
    var categoryKey = $"player:{playerId}:items:{category}";
    await db.SetAddAsync(categoryKey, itemId);
}

public async Task<Dictionary<string, int>> GetItemsByCategoryFastAsync(
    string playerId, 
    string category)
{
    var db = _redis.GetConnection().GetDatabase();
    
    // 1. 카테고리의 아이템 ID 목록
    var categoryKey = $"player:{playerId}:items:{category}";
    var itemIds = await db.SetMembersAsync(categoryKey);
    
    // 2. 각 아이템 개수 조회
    var inventoryKey = $"player:{playerId}:inventory";
    var tasks = itemIds.Select(id => db.HashGetAsync(inventoryKey, id)).ToArray();
    var counts = await Task.WhenAll(tasks);
    
    var result = new Dictionary<string, int>();
    for (int i = 0; i < itemIds.Length; i++)
    {
        result[itemIds[i]] = (int)counts[i];
    }
    
    return result;
}
```

**인벤토리 요약 정보 캐싱**

```csharp
{

    public int UniqueItems { get; set; }
    public Dictionary<string, int> CategoryCounts { get; set; }
}

public async Task<InventorySummary> GetInventorySummaryAsync(string playerId)
{
    var summaryKey = $"player:{playerId}:inventory:summary";
    var cache = new RedisString<InventorySummary>(
        _redis, 
        summaryKey, 
        TimeSpan.FromMinutes(5)
    );
    
    var cached = await cache.GetAsync();
    if (cached.HasValue)
    {
        return cached.Value;
    }
    
    // 요약 생성
    var inventory = await GetInventoryAsync(playerId);
    var summary = new InventorySummary
    {
        TotalItems = inventory.Values.Sum(),
        UniqueItems = inventory.Count,
        CategoryCounts = inventory
            .GroupBy(i => i.Key.Split(':')[1]) // "item:weapon:1001" → "weapon"
            .ToDictionary(g => g.Key, g => g.Sum(i => i.Value))
    };
    
    await cache.SetAsync(summary);
    return summary;
}
```

```csharp
public class NoticeService
{
    private readonly RedisConnection _redis;
    
    public NoticeService(RedisConnection redis)
    {
        _redis = redis;
    }
    
    public async Task AddNoticeAsync(Notice notice)
    {
        // 1. 공지 내용 저장
        var noticeKey = $"notice:{notice.NoticeId}";
        var noticeData = new RedisString<Notice>(_redis, noticeKey, null);
        await noticeData.SetAsync(notice);
        
        // 2. 공지 목록에 추가 (우선순위로 정렬)
        var noticesKey = "notices:active";
        var notices = new RedisSortedSet<string>(_redis, noticesKey, null);
        await notices.AddAsync(notice.NoticeId, notice.Priority);
    }
    
    public async Task<List<Notice>> GetActiveNoticesAsync(int limit = 10)
    {
        var noticesKey = "notices:active";
        var notices = new RedisSortedSet<string>(_redis, noticesKey, null);
        
        // 우선순위 높은 순으로 조회
        var noticeIds = await notices.RangeByRankAsync(0, limit - 1, Order.Descending);
        
        var result = new List<Notice>();
        
        foreach (var noticeId in noticeIds)
        {
            var noticeKey = $"notice:{noticeId}";
            var noticeData = new RedisString<Notice>(_redis, noticeKey, null);
            var notice = await noticeData.GetAsync();
            
            if (notice.HasValue)
            {
                // 기간 내 공지만 반환
                var now = DateTime.UtcNow;
                if (now >= notice.Value.StartTime && now <= notice.Value.EndTime)
                {
                    result.Add(notice.Value);
                }
            }
        }
        
        return result;
    }
    
    public async Task RemoveNoticeAsync(string noticeId)
    {
        // 목록에서 제거
        var noticesKey = "notices:active";
        var notices = new RedisSortedSet<string>(_redis, noticesKey, null);
        await notices.RemoveAsync(noticeId);
        
        // 데이터 삭제
        var noticeKey = $"notice:{noticeId}";
        var db = _redis.GetConnection().GetDatabase();
        await db.KeyDeleteAsync(noticeKey);
    }
}
```

## 정리
이번 장에서는 플레이어 데이터와 인벤토리를 Redis로 효율적으로 관리하는 방법을 배웠다.

**핵심 내용**:

1. **플레이어 프로필 캐싱**
   - Cache-Aside 패턴으로 DB 부하 감소
   - TTL로 자동 만료 관리
   - 버전 관리와 분산 락으로 일관성 유지

2. **인벤토리 시스템**
   - Hash로 아이템별 개수 저장
   - HINCRBY로 원자적 증감
   - 카테고리 인덱스로 조회 최적화

3. **게임 메타데이터**
   - 설정값을 Redis에 캐싱
   - Sorted Set으로 공지사항/이벤트 관리
   - 우선순위와 시간 기반 정렬

**실전 팁**:

- 읽기가 많은 데이터는 긴 TTL로 캐싱한다
- 쓰기가 많은 데이터는 Redis에서 증감하고 비동기로 DB 반영한다
- 동시성 문제는 Lua 스크립트나 트랜잭션으로 해결한다
- 대량 조회는 HSCAN으로 페이징한다
- 메타데이터는 서버 시작 시 미리 로드한다

다음 장에서는 Sorted Set을 활용한 실시간 랭킹 시스템을 구축한다. 점수 업데이트, 순위 조회, 시즌별 랭킹 관리 등 경쟁 요소의 핵심 기능을 다룬다.
