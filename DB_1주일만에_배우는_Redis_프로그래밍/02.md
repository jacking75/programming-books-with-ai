# Chapter 2. Redis 내부 구조와 데이터 타입

## 2.1 Redis 아키텍처 개요

### Single-Threaded 모델과 성능
Redis는 단일 스레드(Single-Threaded) 모델로 동작한다. 이는 언뜻 보면 성능에 제약이 있을 것 같지만, 오히려 Redis가 높은 성능을 발휘하는 핵심 요소다.

```
[클라이언트 요청들]
    ↓  ↓  ↓
┌─────────────────┐
│   Event Loop    │  ← 단일 스레드가 순차 처리
│  (epoll/kqueue) │
└─────────────────┘
    ↓
┌─────────────────┐
│  명령어 실행     │
│  (메모리 연산)   │
└─────────────────┘
    ↓
[응답 반환]
```

단일 스레드의 장점은 다음과 같다:

**동시성 문제 없음**: 여러 스레드가 동시에 데이터를 수정할 때 발생하는 락(Lock) 경쟁이 없다. 게임 서버에서 같은 플레이어의 골드를 여러 요청이 동시에 변경하려 할 때도 순차적으로 처리되어 데이터 정합성이 보장된다.

**컨텍스트 스위칭 비용 제로**: CPU가 스레드를 전환하는 비용이 없어 순수하게 명령어 처리에만 집중할 수 있다.

**예측 가능한 성능**: 명령어가 순서대로 처리되므로 성능 예측과 디버깅이 쉽다.

Redis 6.0부터는 I/O 스레드를 도입했다. 하지만 명령어 실행 자체는 여전히 단일 스레드다. I/O 스레드는 네트워크 패킷을 읽고 쓰는 작업만 병렬로 처리한다.

```
[네트워크 I/O]
  ↓  ↓  ↓
┌──────────────┐
│ I/O Threads  │ ← Redis 6.0+ (읽기/쓰기만 병렬)
│ (멀티스레드)  │
└──────────────┘
  ↓
┌──────────────┐
│ Main Thread  │ ← 명령어 실행 (단일스레드)
└──────────────┘
```

이러한 구조 덕분에 Redis는 초당 수십만 건의 명령어를 처리할 수 있다. 게임 서버에서 10만 명의 동시 접속자가 있어도 각 플레이어의 요청이 마이크로초 단위로 처리된다.

### 메모리 관리 방식
Redis는 모든 데이터를 메모리에 저장한다. 디스크 I/O 없이 메모리에서 직접 데이터를 읽고 쓰기 때문에 극도로 빠르다.

**메모리 할당 전략**

Redis는 jemalloc을 기본 메모리 할당자로 사용한다. jemalloc은 메모리 단편화를 최소화하도록 설계되었다.

```
메모리 레이아웃 예시:
┌──────────────────────────────────┐
│         Redis Process            │
├──────────────────────────────────┤
│  Data (실제 키-값 데이터)         │  ← 대부분의 메모리
├──────────────────────────────────┤
│  Buffers (클라이언트 버퍼 등)     │
├──────────────────────────────────┤
│  Replication Backlog             │
├──────────────────────────────────┤
│  AOF Rewrite Buffer              │
└──────────────────────────────────┘
```

게임에서 100만 명의 유저 세션을 저장한다고 가정하자. 각 세션이 평균 1KB라면 약 1GB의 메모리가 필요하다. Redis는 이를 효율적으로 관리하며, `INFO memory` 명령으로 실시간 메모리 사용량을 확인할 수 있다.

**메모리 초과 시 정책**

`maxmemory` 설정으로 최대 메모리를 제한할 수 있다. 한계에 도달하면 `maxmemory-policy`에 따라 동작한다:

- **noeviction**: 새로운 쓰기를 거부한다 (기본값)
- **allkeys-lru**: 가장 오래 사용하지 않은 키를 삭제한다
- **volatile-lru**: TTL이 설정된 키 중 LRU 방식으로 삭제한다
- **allkeys-random**: 무작위로 키를 삭제한다
- **volatile-ttl**: TTL이 가장 짧은 키를 먼저 삭제한다

게임 캐시 서버라면 `volatile-lru`가 적합하다. 세션 데이터는 TTL이 있으므로 자동으로 정리되고, 오래된 캐시부터 제거된다.

```csharp
// redis.conf 설정 예시
maxmemory 2gb
maxmemory-policy volatile-lru
```

### Persistence (RDB, AOF)
메모리 기반 저장소지만 Redis는 데이터를 디스크에 저장하는 기능을 제공한다. 서버가 재시작해도 데이터를 복구할 수 있다.

**RDB (Redis Database)**

특정 시점의 스냅샷을 바이너리 파일로 저장한다.

```
시간 흐름:
  0초        60초       120초      180초
   |          |          |          |
   ├─────────RDB────────RDB────────RDB
              ↓          ↓          ↓
           dump.rdb   dump.rdb   dump.rdb
```

RDB는 컴팩트하고 복원이 빠르다. 하지만 스냅샷 사이의 데이터는 손실될 수 있다. 게임에서 랭킹 데이터처럼 일부 손실이 허용되는 경우 적합하다.

```conf
# redis.conf
save 900 1      # 900초 동안 1개 이상 변경 시 저장
save 300 10     # 300초 동안 10개 이상 변경 시 저장
save 60 10000   # 60초 동안 10000개 이상 변경 시 저장
```

**AOF (Append Only File)**

모든 쓰기 명령을 로그 파일에 순차적으로 기록한다.

```
명령어 실행 순서:
SET user:1000 "player1"  →  AOF에 기록
HSET inventory:1000 gold 100  →  AOF에 기록
INCR login_count  →  AOF에 기록
```

AOF는 데이터 손실이 거의 없지만 파일 크기가 크고 복원이 느리다. 결제 정보나 중요한 게임 재화처럼 손실이 절대 안 되는 데이터에 사용한다.

```conf
# redis.conf
appendonly yes
appendfsync everysec  # 매초 디스크에 동기화 (권장)
# appendfsync always  # 매 명령마다 동기화 (가장 안전, 느림)
# appendfsync no      # OS에 맡김 (빠름, 손실 가능)
```

AOF 파일은 시간이 지나면 거대해진다. Redis는 자동으로 재작성(rewrite)하여 압축한다:

```
원본 AOF:
  SET gold 100
  INCR gold
  INCR gold
  INCR gold
    ↓
재작성 후:
  SET gold 103
```

**RDB vs AOF 선택 가이드**

게임 서버에서는 두 가지를 함께 사용하는 것이 일반적이다:

- RDB: 정기적인 백업용 (4시간마다 스냅샷)
- AOF: 실시간 데이터 보호용 (everysec 모드)

Redis 4.0+부터는 RDB와 AOF를 혼합한 형식도 지원한다. AOF 재작성 시 RDB 스냅샷을 앞부분에 쓰고 이후 증분 명령만 추가한다.

```
┌─────────────────────────────┐
│  RDB 스냅샷 (바이너리)       │
├─────────────────────────────┤
│  이후 변경 명령들 (AOF)      │
└─────────────────────────────┘
```

## 2.2 Redis 핵심 데이터 타입
Redis는 단순한 키-값 저장소가 아니다. 다양한 자료구조를 네이티브로 지원하여 복잡한 데이터를 효율적으로 다룰 수 있다.

### String: 기본 키-값 저장
String은 Redis의 가장 기본적인 타입이다. 텍스트뿐만 아니라 바이너리 데이터, 숫자도 저장할 수 있다. 최대 512MB까지 저장 가능하다.

**기본 사용**

```
SET player:1000:name "DarkKnight"
GET player:1000:name
→ "DarkKnight"
```

**숫자 연산**

String에 정수를 저장하면 원자적(atomic) 증감 연산이 가능하다. 게임에서 동시에 여러 요청이 들어와도 정확한 값이 보장된다.

```
SET player:1000:gold 100
INCR player:1000:gold      → 101
INCRBY player:1000:gold 50 → 151
DECR player:1000:gold      → 150
```

실제 게임 시나리오:

```
플레이어가 아이템 판매 (금화 +100)
동시에 플레이어가 아이템 구매 (금화 -50)

Redis는 순차적으로 처리:
  1. INCRBY gold 100  → 200
  2. DECRBY gold 50   → 150

최종 결과: 150 (정확함!)
```

**만료 시간 설정**

TTL(Time To Live)을 설정하여 자동으로 키가 삭제되게 할 수 있다. 세션 관리에 유용하다.

```
SET session:abc123 "user_data" EX 3600  # 3600초(1시간) 후 삭제
SETEX session:abc123 3600 "user_data"   # 위와 동일

TTL session:abc123  → 3590 (남은 시간 확인)
```

**비트맵 연산**

String은 내부적으로 비트 배열로 다룰 수 있다. 출석 체크나 권한 플래그에 메모리 효율적이다.

```
# 사용자 1000의 1월 출석 체크
SETBIT attendance:202501:1000 0 1  # 1일 출석
SETBIT attendance:202501:1000 1 1  # 2일 출석
SETBIT attendance:202501:1000 5 1  # 6일 출석

BITCOUNT attendance:202501:1000     # 출석 일수 → 3
```

### Hash: 객체 데이터 저장
Hash는 필드-값 쌍의 집합이다. 하나의 키 아래 여러 속성을 저장할 수 있어 객체를 표현하기 적합하다.

```
┌─────────────────────────┐
│  player:1000            │  ← Hash 키
├──────────┬──────────────┤
│  name    │  "DarkKnight"│
│  level   │  "45"        │
│  class   │  "Warrior"   │
│  exp     │  "12350"     │
└──────────┴──────────────┘
```

**기본 명령어**

```
HSET player:1000 name "DarkKnight"
HSET player:1000 level 45
HSET player:1000 class "Warrior"

HGET player:1000 name      → "DarkKnight"
HGETALL player:1000        → name "DarkKnight" level "45" class "Warrior"
```

여러 필드를 한 번에 설정하거나 조회할 수 있다:

```
HMSET player:1000 name "DarkKnight" level 45 class "Warrior"
HMGET player:1000 name level  → ["DarkKnight", "45"]
```

**숫자 필드 증감**

Hash 필드도 숫자 연산이 가능하다. 게임에서 경험치나 재화를 다룰 때 유용하다.

```
HINCRBY player:1000 exp 150     # 경험치 +150
HINCRBY player:1000 gold 1000   # 금화 +1000
HINCRBYFLOAT player:1000 rating 1.5  # 평점 +1.5
```

**Hash의 메모리 효율성**

Hash는 내부적으로 ziplist나 hashtable로 저장된다. 작은 Hash(필드 512개 미만, 값 64바이트 미만)는 ziplist를 사용해 메모리를 절약한다.

100만 플레이어를 String으로 저장 vs Hash로 저장:

```
String 방식 (각 속성이 별도 키):
  player:1000:name    → 오버헤드 큼
  player:1000:level   → 오버헤드 큼
  player:1000:class   → 오버헤드 큼
  
Hash 방식 (하나의 키):
  player:1000 → {name, level, class}  → 오버헤드 작음
```

Hash는 약 5배 이상 메모리 효율적이다.

### List: 순서가 있는 데이터
List는 문자열의 연결 리스트다. 양쪽 끝에서 push/pop이 O(1)이므로 큐나 스택으로 사용하기 좋다.

```
┌────┐    ┌────┐    ┌────┐    ┌────┐
│ A  │ ←→ │ B  │ ←→ │ C  │ ←→ │ D  │
└────┘    └────┘    └────┘    └────┘
 HEAD                           TAIL
```

**기본 명령어**

```
LPUSH messages "msg1"  # 왼쪽에 추가
LPUSH messages "msg2"
RPUSH messages "msg3"  # 오른쪽에 추가

LRANGE messages 0 -1   → ["msg2", "msg1", "msg3"]

LPOP messages          → "msg2" (왼쪽에서 제거)
RPOP messages          → "msg3" (오른쪽에서 제거)
```

**메시지 큐로 활용**

게임에서 우편함이나 알림을 구현할 때 사용한다:

```
# 새 메일 추가 (오른쪽에 push)
RPUSH mailbox:1000 "친구 요청이 도착했습니다"
RPUSH mailbox:1000 "아이템이 만료됩니다"

# 메일 읽기 (왼쪽에서 pop)
LPOP mailbox:1000  → "친구 요청이 도착했습니다"

# 읽지 않은 메일 수
LLEN mailbox:1000  → 1
```

**최근 활동 기록**

로그인 이력이나 최근 플레이 기록을 저장할 때 사용한다:

```
# 최신 로그인 기록 추가 (최대 10개 유지)
LPUSH login_history:1000 "2025-12-26 14:30:00"
LTRIM login_history:1000 0 9  # 10개만 유지

# 최근 3개 조회
LRANGE login_history:1000 0 2
```

**블로킹 pop**

`BLPOP`/`BRPOP`은 리스트가 비어있으면 데이터가 들어올 때까지 대기한다. 작업 큐 시스템에 유용하다:

```
# Worker 프로세스
BLPOP task_queue 0  # 무한 대기

# 다른 프로세스에서
RPUSH task_queue "process_payment"  # Worker가 즉시 깨어남
```

### Set: 중복 없는 집합
Set은 순서가 없는 고유한 문자열의 집합이다. 멤버십 체크가 O(1)이며 집합 연산(교집합, 합집합 등)을 지원한다.

```
┌────────────────┐
│   online_users │
├────────────────┤
│   user:1000    │
│   user:1001    │  ← 중복 없음
│   user:1005    │  ← 순서 없음
└────────────────┘
```

**기본 명령어**

```
SADD online_users "user:1000"
SADD online_users "user:1001"
SADD online_users "user:1000"  # 중복은 무시됨

SMEMBERS online_users  → ["user:1000", "user:1001"]
SISMEMBER online_users "user:1000"  → 1 (존재)
SCARD online_users     → 2 (멤버 수)
```

**집합 연산**

게임에서 친구 관계나 길드 멤버 관리에 활용한다:

```
# A와 B의 친구 목록
SADD friends:A "user:1" "user:2" "user:3"
SADD friends:B "user:2" "user:3" "user:4"

# 공통 친구 (교집합)
SINTER friends:A friends:B  → ["user:2", "user:3"]

# 모든 친구 (합집합)
SUNION friends:A friends:B  → ["user:1", "user:2", "user:3", "user:4"]

# A만 아는 친구 (차집합)
SDIFF friends:A friends:B   → ["user:1"]
```

**랜덤 뽑기**

가챠나 랜덤 보상 시스템에 사용할 수 있다:

```
SADD reward_pool "sword" "shield" "potion" "gold"
SRANDMEMBER reward_pool     → "shield" (랜덤, 제거 안 함)
SPOP reward_pool            → "sword" (랜덤, 제거함)
```

**태그 시스템**

```
# 아이템에 태그 추가
SADD item:sword:1000:tags "legendary" "fire" "two-handed"

# "fire" 태그가 있는지 확인
SISMEMBER item:sword:1000:tags "fire"  → 1
```

### Sorted Set: 점수 기반 정렬 집합
Sorted Set은 Set과 유사하지만 각 멤버에 점수(score)가 있어 자동으로 정렬된다. 게임 랭킹 시스템의 핵심이다.

```
┌──────────────────────────────┐
│      leaderboard             │
├──────────┬───────────────────┤
│  Score   │     Member        │
├──────────┼───────────────────┤
│  9500    │  player:1000      │  ← 높은 점수
│  8800    │  player:1005      │
│  7200    │  player:1002      │
│  5100    │  player:1008      │  ← 낮은 점수
└──────────┴───────────────────┘
```

**기본 명령어**

```
ZADD leaderboard 9500 "player:1000"
ZADD leaderboard 8800 "player:1005"
ZADD leaderboard 7200 "player:1002"

# 점수 순으로 조회 (오름차순)
ZRANGE leaderboard 0 -1 WITHSCORES
→ player:1002 7200, player:1005 8800, player:1000 9500

# 역순으로 조회 (내림차순, 1등부터)
ZREVRANGE leaderboard 0 2 WITHSCORES
→ player:1000 9500, player:1005 8800, player:1002 7200
```

**점수 증감**

```
ZINCRBY leaderboard 100 "player:1002"  # 점수 +100
ZSCORE leaderboard "player:1002"       → 7300
```

**순위 조회**

```
# player:1000의 순위 (0부터 시작)
ZREVRANK leaderboard "player:1000"  → 0 (1등)
ZRANK leaderboard "player:1000"     → 2 (역순 기준 3등)
```

**점수 범위 조회**

특정 점수 구간의 플레이어를 찾을 수 있다:

```
# 8000점 이상 플레이어
ZRANGEBYSCORE leaderboard 8000 +inf
→ player:1005, player:1000

# 5000~8000점 플레이어 수
ZCOUNT leaderboard 5000 8000  → 1
```

**순위 구간 조회**

```
# 상위 10명
ZREVRANGE leaderboard 0 9 WITHSCORES

# 100~110등
ZREVRANGE leaderboard 99 109 WITHSCORES

# 내 주변 순위 (나를 중심으로 ±5명)
# 복잡하므로 Chapter 5에서 자세히 다룬다
```

**시간 기반 정렬**

점수를 타임스탬프로 사용하면 시간 순 정렬이 가능하다:

```
# 최근 접속 순
ZADD recent_login 1703584200 "user:1000"
ZADD recent_login 1703584500 "user:1001"

# 최근 접속 3명
ZREVRANGE recent_login 0 2
```

## 2.3 CloudStructures로 데이터 타입 다루기
CloudStructures는 StackExchange.Redis를 기반으로 한 C# Redis 클라이언트다. 강타입 지원과 비동기 API로 .NET 개발자에게 친숙하다.

### 기본 설정
먼저 NuGet에서 CloudStructures를 설치한다:

```bash
dotnet add package CloudStructures
```

Redis 연결 설정:

```csharp
using CloudStructures;
using CloudStructures.Structures;
using StackExchange.Redis;

// Redis 연결 설정
var config = new RedisConfig("default", "localhost:6379");
var connection = new RedisConnection(config);
```

프로덕션 환경에서는 연결 풀을 관리하는 것이 좋다:

```csharp
public class RedisService
{
    private readonly RedisConnection _connection;
    
    public RedisService(string connectionString)
    {
        var config = new RedisConfig("game-server", connectionString, 
            connectTimeout: 5000, syncTimeout: 3000);
        _connection = new RedisConnection(config);
    }
    
    public RedisConnection Connection => _connection;
}
```

### RedisString 사용법

**기본 Get/Set**


### RedisList 사용법

**메일함 구현**

```csharp
public class MailMessage
{
    public string Title { get; set; }
    public string Content { get; set; }
    public DateTime SentTime { get; set; }
}

var mailbox = new RedisList<MailMessage>(
    connection, 
    "mailbox:1000", 
    TimeSpan.FromDays(30)  // 30일 후 자동 삭제
);

// 메일 추가 (오른쪽에 push)
await mailbox.RightPushAsync(new MailMessage
{
    Title = "이벤트 보상",
    Content = "골드 1000을 지급했습니다",
    SentTime = DateTime.UtcNow
});

// 메일 읽기 (왼쪽에서 pop)
var mail = await mailbox.LeftPopAsync();
if (mail.HasValue)
{
    Console.WriteLine($"제목: {mail.Value.Title}");
}

// 읽지 않은 메일 수
var count = await mailbox.LengthAsync();
Console.WriteLine($"읽지 않은 메일: {count}개");
```

### RedisSortedSet 사용법

**랭킹 시스템**

```csharp
public class RankingEntry
{
    public string PlayerId { get; set; }
    public long Score { get; set; }
}

var leaderboard = new RedisSortedSet<string>(
    connection, 
    "leaderboard:pvp", 
    null
);

await leaderboard.AddAsync("player:1001", 8800);
await leaderboard.AddAsync("player:1002", 7200);

// 점수 증가
await leaderboard.IncrementAsync("player:1002", 100);

// 상위 10명 조회 (내림차순)
var top10 = await leaderboard.RangeByRankWithScoresAsync(
    0, 9, 
    Order.Descending
);

int rank = 1;
foreach (var entry in top10)
{
    Console.WriteLine($"{rank}. {entry.Element} - {entry.Score}점");
    rank++;
}

// 특정 플레이어 순위
var myRank = await leaderboard.RankAsync("player:1000", Order.Descending);
Console.WriteLine($"내 순위: {myRank + 1}등");  // 0-based이므로 +1

// 특정 플레이어 점수
var myScore = await leaderboard.ScoreAsync("player:1000");
Console.WriteLine($"내 점수: {myScore}점");
```

**시간 기반 정렬**

```csharp
// 최근 접속자 (타임스탬프를 점수로 사용)
var recentPlayers = new RedisSortedSet<string>(
    connection, 
    "recent_login", 
    null
);
```

## 정리
이번 장에서는 Redis의 내부 구조와 핵심 데이터 타입을 살펴보았다.

**Redis 아키텍처 핵심**:
- Single-threaded 모델로 동시성 문제 없이 높은 성능 달성
- 메모리 기반 저장으로 초저지연 응답 (마이크로초 단위)
- RDB/AOF를 통한 데이터 영속성 보장

**데이터 타입별 활용**:
- **String**: 세션, 카운터, 간단한 캐시
- **Hash**: 플레이어 프로필, 인벤토리, 객체 데이터
- **List**: 메일함, 활동 로그, 메시지 큐
- **Set**: 온라인 유저, 친구 목록, 태그 시스템
- **Sorted Set**: 랭킹, 시간 기반 정렬, 리더보드

**CloudStructures의 장점**:
- 강타입 지원으로 컴파일 타임 안정성
- 비동기 API로 높은 처리량
- 자동 직렬화/역직렬화

다음 장에서는 이러한 데이터 타입을 활용하여 실제 게임의 플레이어 세션과 인증 시스템을 구현한다. Redis의 TTL 기능과 원자적 연산을 활용하여 안전하고 효율적인 로그인 시스템을 만들어본다.
