## 2.1 hstring - WinRT 문자열 다루기
WinRT를 다룰 때 가장 먼저 마주치는 타입이 바로 **hstring**이다. hstring은 "handle to string"의 약자로, WinRT의 표준 문자열 타입이다.

전통적인 Windows 프로그래밍에서는 BSTR, LPWSTR, std::wstring 등 여러 문자열 타입이 섞여 있었다. 이로 인해 타입 변환 코드가 복잡했다. WinRT의 hstring은 이러한 문제를 해결한다.

**hstring의 특징:**

첫째, hstring은 불변(immutable) 문자열이다. 생성 후 내용을 변경할 수 없다. 이는 멀티스레드 환경에서 안전성을 보장한다.

둘째, hstring은 자동 메모리 관리를 제공한다. 참조 카운팅으로 동작하므로 개발자가 메모리 해제를 신경 쓸 필요가 없다.

셋째, hstring은 COM 인터페이스 매개변수로 직접 사용될 수 있다. BSTR로 자동 변환되므로 호환성이 높다.

**주요 API:**

hstring 생성자는 여러 형태를 지원한다. std::wstring이나 const wchar_t* 포인터로 hstring을 만들 수 있다. hstring을 비어있게 생성할 수도 있다.

to_hstring() 함수는 표준 타입을 hstring으로 변환한다. std::string, std::wstring, const char*, const wchar_t* 모두 지원한다.

to_string() 함수는 hstring을 std::string으로 변환한다. std::wstring으로 변환하려면 to_wstring()을 사용한다.

c_str() 메서드는 hstring의 내부 포인터를 wchar_t*로 반환한다. 하지만 hstring 자체가 살아있는 동안만 유효하다.

size() 메서드는 문자열의 길이를 반환한다. 널 종료 문자는 포함하지 않는다.

empty() 메서드는 문자열이 비어있는지 확인한다.

**실제 사용 예제 (약 20라인):**

```cpp
#include <iostream>
#include <winrt/Windows.Foundation.h>

using namespace winrt;
using namespace Windows::Foundation;

int main() {
    // std::string에서 hstring으로 변환
    std::string message = "Welcome to WinRT";
    hstring wrt_message = to_hstring(message);

    // hstring에서 std::string으로 변환
    std::string back_to_cpp = to_string(wrt_message);
    std::cout << "Message: " << back_to_cpp << std::endl;

    // wchar_t* 포인터로 접근
    const wchar_t* ptr = wrt_message.c_str();
    std::wcout << L"Direct pointer: " << ptr << std::endl;

    // 문자열 길이 확인
    size_t len = wrt_message.size();
    std::cout << "Length: " << len << std::endl;

    // 비어있는지 확인
    hstring empty_str;
    if (empty_str.empty()) {
        std::cout << "String is empty" << std::endl;
    }

    return 0;
}
```

9-10줄에서 std::string을 만들고 to_hstring()으로 hstring으로 변환한다. 이제 wrt_message는 WinRT API에 전달할 수 있다. 
 13줄에서 역으로 변환한다. to_string()은 hstring을 std::string으로 바꾼다. 게임 서버에서는 많은 로직이 std::string 기반이므로 이런 변환이 자주 필요하다.  
16줄의 c_str()은 내부 포인터를 가져온다. 이를 통해 Windows API에 직접 전달할 수 있다.  
19줄의 size()는 문자 개수를 반환한다. 널 종료 문자는 제외된다.  
22-25줄에서 empty() 메서드를 사용한다. 이것으로 문자열이 비어있는지 쉽게 확인할 수 있다.  
  

**GameServer 관점에서의 활용:**  
게임 서버에서 플레이어 이름, 채팅 메시지, 서버 상태 문자열 등을 처리할 때 hstring을 사용한다. 예를 들어 플레이어 접속 로그를 기록할 때 플레이어 이름을 hstring으로 받아서 로그 시스템으로 전달할 수 있다.

hstring은 특히 파일 I/O나 네트워크 통신 시 유용하다. 파일 경로를 hstring으로 전달할 수 있고, 네트워크에서 받은 문자열도 hstring으로 변환하여 WinRT API에 전달할 수 있기 때문이다.
  


## 2.2 winrt::com_ptr - 스마트 포인터
WinRT를 다루면서 가장 중요한 개념 중 하나가 메모리 관리다. winrt::com_ptr은 COM 객체를 안전하게 관리하는 스마트 포인터다.

COM은 참조 카운팅 기반의 메모리 관리를 사용한다. 객체가 생성될 때 참조 카운트가 1로 시작하고, 누군가 그 객체를 참조할 때마다 증가한다. 참조를 포기할 때는 감소한다. 참조 카운트가 0이 되면 객체가 자동으로 삭제된다.

전통적인 COM 프로그래밍에서는 AddRef()와 Release()를 수동으로 호출해야 했다. 이는 매우 번거롭고 버그가 발생하기 쉬웠다. winrt::com_ptr은 RAII(Resource Acquisition Is Initialization) 패턴을 사용하여 이 문제를 해결한다.

**com_ptr의 주요 메서드:**

get() 메서드는 내부 포인터를 반환한다. WinRT API에 포인터가 필요할 때 사용한다.

put() 메서드는 새로운 포인터를 저장할 때 사용한다. 기존 참조는 자동으로 해제되고 새 포인터의 참조 카운트가 증가한다.

as() 메서드는 다른 인터페이스로 형변환한다. QueryInterface를 내부적으로 호출한다.

operator-> 오버로드는 포인터처럼 멤버에 접근할 수 있게 해준다.

operator* 오버로드는 역참조 연산자다.

**실제 사용 예제 :**

```cpp
#include <iostream>
#include <winrt/Windows.Foundation.h>
#include <winrt/Windows.Storage.h>

using namespace winrt;
using namespace Windows::Storage;

int main() {
    winrt::init_apartment();

    // com_ptr로 파일 객체 관리
    StorageFile file = nullptr;
    
    // IStorageFile 인터페이스로도 접근 가능
    // com_ptr은 내부적으로 참조 카운팅을 관리
    
    try {
        // 파일 객체 생성 (간단화된 예제)
        std::cout << "Reference count automatically managed" << std::endl;
        
        // com_ptr 스코프를 벗어나면 자동 정리
        {
            // 파일 시스템 작업...
            std::cout << "File operations completed" << std::endl;
        } // 여기서 자동으로 정리됨
        
        std::cout << "Resources cleaned up" << std::endl;
    }
    catch (const winrt::hresult_error& e) {
        std::cerr << "Error: " << winrt::to_string(e.message()) << std::endl;
    }

    return 0;
}
```

이 예제는 간단해 보이지만 중요한 개념을 담고 있다. 9줄의 init_apartment()는 WinRT를 초기화한다.

12줄에서 StorageFile을 선언한다. StorageFile은 실제로 com_ptr로 감싸져 있다. 개발자는 일반 변수처럼 사용하지만 내부적으로는 스마트 포인터 동작을 한다.  
19-21줄의 스코프 블록을 보자. 이 블록이 끝나면 블록 내에서 생성한 모든 WinRT 객체들이 자동으로 정리된다. 개발자가 명시적으로 delete를 호출할 필요가 없다.  
  
**참조 카운팅 메커니즘 심화:**  
COM 객체가 생성될 때 참조 카운트는 1부터 시작한다. 누군가 그 객체를 참조하면 카운트가 증가한다. 예를 들어 com_ptr로 복사하면 카운트가 증가한다.

```cpp
StorageFile file1 = ...;  // 참조 카운트: 1
StorageFile file2 = file1;  // 참조 카운트: 2
```
  
file2가 스코프를 벗어나면 참조 카운트가 2에서 1로 감소한다. file1도 스코프를 벗어나면 0이 되어 객체가 삭제된다.

**GameServer 관점에서의 활용:**  
게임 서버에서는 네트워크 소켓, 파일 핸들, 타이머 등 여러 리소스를 다룬다. 이 모든 것이 COM 객체로 구현되어 있다. winrt::com_ptr을 사용하면 이 리소스들을 안전하게 관리할 수 있다.

예를 들어 클라이언트별 소켓을 보관할 때 com_ptr을 사용하면 클라이언트 객체가 소멸할 때 소켓도 자동으로 정리된다.
  


## 2.3 winrt::handle - 리소스 관리
com_ptr이 COM 객체를 관리한다면, winrt::handle은 네이티브 Windows 핸들(HANDLE)을 관리한다. 프로세스, 스레드, 이벤트, 뮤텍스, 파일 등을 나타내는 핸들을 안전하게 다룬다.

전통적으로 Windows 핸들은 CloseHandle() 함수로 직접 정리해야 했다. 예외가 발생하면 정리 코드를 실행하지 못할 위험이 있었다. winrt::handle은 이 문제를 RAII 패턴으로 해결한다.

**handle의 주요 특징:**

handle은 스코프를 벗어날 때 자동으로 CloseHandle()을 호출한다. 수동으로 정리할 필요가 없다.

get() 메서드로 내부 핸들을 얻을 수 있다. Windows API에 전달할 때 사용한다.

유효한 핸들인지 확인할 수 있다. nullptr이나 INVALID_HANDLE_VALUE를 검사한다.

이동 의미론(move semantics)을 지원한다. 핸들의 소유권을 이전할 수 있다.

**실제 사용 예제 :**

```cpp
#include <iostream>
#include <windows.h>
#include <winrt/base.h>

using namespace winrt;

int main() {
    // 수동 관리 (위험함)
    HANDLE manual_handle = CreateEventW(nullptr, FALSE, FALSE, nullptr);
    if (manual_handle != INVALID_HANDLE_VALUE) {
        // 사용...
        CloseHandle(manual_handle);  // 반드시 호출해야 함
    }

    // handle로 관리 (안전함)
    handle safe_handle(CreateEventW(nullptr, FALSE, FALSE, nullptr));
    if (safe_handle) {
        // 사용...
        // 스코프를 벗어나면 자동 정리
    }

    std::cout << "Handle automatically cleaned up" << std::endl;

    return 0;
}
```

이 코드는 같은 작업을 두 가지 방식으로 보여준다. 첫 번째 방식(10-14줄)은 수동 관리다. CreateEventW로 핸들을 얻고 사용 후 CloseHandle로 정리한다. 만약 14줄을 빠뜨리면 핸들 누수가 발생한다.  
  
두 번째 방식(17-21줄)은 handle을 사용한다. 18줄에서 handle 객체를 생성할 때 핸들을 넘긴다. 22줄에 도달하면 handle의 소멸자가 자동으로 CloseHandle을 호출한다. 개발자는 정리 코드를 쓸 필요가 없다.  
  
18줄의 if (safe_handle)은 handle이 유효한지 확인한다. bool 변환 연산자가 오버로드되어 있어 이렇게 쓸 수 있다.
  

**RAII 패턴 설명:**  
RAII는 "생성자에서 자원을 획득하고 소멸자에서 자원을 해제한다"는 원칙이다. winrt::handle은 이 패턴을 따른다.

생성자: handle이 생성될 때 내부에 핸들을 저장한다. 이 시점에 자원 획득은 이미 일어난 상태다(CreateEvent 같은 함수가 호출됨).

사용: 중간에 핸들을 사용한다. get() 메서드로 필요할 때 꺼낸다.

소멸자: handle 객체가 스코프를 벗어날 때 자동으로 CloseHandle이 호출된다.

이 패턴의 장점은 예외 안전성이다. 예외가 발생해도 스택 언와인딩 과정에서 소멸자가 호출되므로 자원이 반드시 정리된다.
  
**GameServer 관점에서의 활용:**  
게임 서버는 여러 스레드를 사용한다. 각 스레드는 스레드 핸들로 표현된다. winrt::handle을 사용하면 스레드 종료 시 자동으로 핸들이 정리된다.

또한 게임 서버의 이벤트 기반 동기화에서 handle이 유용하다. 예를 들어 "클라이언트 접속 이벤트"를 나타내는 핸들을 handle로 관리할 수 있다.
  


## 2.4 winrt::fire_and_forget - 비동기 작업
fire_and_forget은 특별한 반환 타입이다. 비동기 함수가 호출자의 대기 없이 실행되도록 한다. "발사하고 잊어버린다"는 뜻의 이름처럼, 결과를 기다리지 않는다.  
  
네트워크 게임 서버에서는 완료를 기다릴 필요가 없는 작업이 많다. 예를 들어 모든 클라이언트에게 게임 상태를 업데이트하는 브로드캐스트는 완료 신호를 기다릴 필요가 없다. fire_and_forget을 사용하면 이런 작업을 효율적으로 처리할 수 있다.  
  
**fire_and_forget의 특징:**  
호출자가 즉시 반환된다. 비동기 작업이 백그라운드에서 실행된다.  
완료를 기다리지 않으므로 예외 처리가 다르다. 예외가 발생해도 호출자는 모른다.  
IAsyncAction과 다르게 co_await로 기다릴 수 없다.  
코루틴이므로 내부에서 co_await를 사용할 수 있다.  
  
**실제 사용 예제 :**

```cpp
#include <iostream>
#include <winrt/Windows.Foundation.h>
#include <chrono>
#include <thread>

using namespace winrt;
using namespace Windows::Foundation;

fire_and_forget BackgroundTask() {
    // 백그라운드에서 실행
    co_await resume_background();
    
    std::cout << "Background task running" << std::endl;
    
    // co_await로 비동기 작업 가능
    std::this_thread::sleep_for(std::chrono::seconds(1));
    
    std::cout << "Background task completed" << std::endl;
}

int main() {
    winrt::init_apartment();
    
    // 함수 호출 후 즉시 반환
    BackgroundTask();
    
    std::cout << "Main thread continues" << std::endl;
    std::this_thread::sleep_for(std::chrono::seconds(2));
    
    return 0;
}
```

9줄의 fire_and_forget이 핵심이다. 이 함수는 비동기 코루틴이지만 호출자가 완료를 기다리지 않는다.  
24줄에서 BackgroundTask()를 호출하면 함수가 즉시 반환된다. 백그라운드 작업은 별도로 실행된다.  
25줄의 "Main thread continues"는 백그라운드 작업이 끝나기 전에 출력될 가능성이 높다. 만약 기다려야 한다면 IAsyncAction을 사용해야 한다.  
  
**fire_and_forget vs IAsyncAction:**  
IAsyncAction을 사용하면 호출자가 co_await로 완료를 기다릴 수 있다. 반환값이 있고 예외도 전파된다.

```cpp
IAsyncAction WaitableTask() {
    co_await resume_background();
    std::cout << "Waitable task" << std::endl;
}

// 사용
auto task = WaitableTask();
co_await task;  // 완료를 기다림
```

fire_and_forget은 기다리지 않는다. 예외가 발생해도 호출자는 모른다. 따라서 fire_and_forget 내부에서는 반드시 try-catch로 예외를 처리해야 한다.  

```cpp
fire_and_forget BackgroundTask() {
    try {
        co_await resume_background();
        // 작업...
    }
    catch (const hresult_error& e) {
        // 로그 기록 등 처리
    }
}
```
  
**GameServer 관점에서의 활용:**  
게임 서버에서 모든 클라이언트에게 메시지를 브로드캐스트할 때 fire_and_forget을 사용한다. 각 클라이언트로의 전송을 기다릴 필요가 없기 때문이다.

또한 서버 상태 저장, 통계 업데이트, 로깅 같은 부가적 작업도 fire_and_forget으로 처리할 수 있다. 주 게임 로직을 블로킹하지 않으면서도 필요한 작업을 수행할 수 있다.
  


## 2.5 컬렉션 타입들 (IVector, IMap)
WinRT는 데이터 컬렉션을 다루기 위해 IVector와 IMap 같은 인터페이스를 제공한다. 이들은 C++ 표준 라이브러리의 vector와 map 같은 역할을 하지만, COM 기반이고 WinRT API와 호환성이 높다.

IVector<T>는 동적 배열이다. 크기가 동적으로 변한다. IMap<K, V>는 키-값 쌍의 컬렉션이다. 게임 서버에서는 클라이언트 목록, 게임 상태 딕셔너리 등을 저장할 때 이 타입들을 사용한다.  

**IVector의 주요 메서드:**

Append(T value) - 끝에 요소 추가
GetAt(uint32_t index) - 인덱스로 요소 접근
RemoveAt(uint32_t index) - 인덱스로 요소 삭제
Clear() - 모든 요소 제거
Size - 현재 크기 속성
First() - 반복자 얻기

**IMap의 주요 메서드:**

Insert(K key, V value) - 키-값 추가 또는 업데이트
Lookup(K key) - 키로 값 찾기
Remove(K key) - 키 제거
Clear() - 모든 항목 제거
Size - 현재 크기 속성
First() - 반복자 얻기

**실제 사용 예제 :**

```cpp
#include <iostream>
#include <winrt/Windows.Foundation.Collections.h>
#include <winrt/base.h> // to_hstring, to_string 등을 위해 추가

// 원본 코드의 std::vector, std::map 헤더는 
// WinRT 컬렉션만 사용한다면 필요하지 않을 수 있습니다.
#include <vector>
#include <map>

using namespace winrt;
using namespace Windows::Foundation::Collections;

int main() {
    winrt::init_apartment();

    // === IVector<int> 예제 ===
    std::cout << "=== IVector Example ===" << std::endl;

    // IVector<int> 인터페이스를 구현하는 single_threaded_vector 생성
    IVector<int> numbers = winrt::single_threaded_vector<int>();
    
    // Append 메서드로 요소 추가
    numbers.Append(10);
    numbers.Append(20);
    numbers.Append(30);

    // C++의 range-based for loop로 표준 컨테이너처럼 순회 가능
    for (int num : numbers) {
        std::cout << num << " ";
    }
    std::cout << std::endl;

    // Size() 메서드로 크기 확인
    std::cout << "Vector size: " << numbers.Size() << std::endl;


    // === IMap<hstring, int> 예제 ===
    std::cout << "\n=== IMap Example ===" << std::endl;
    
    // IMap<hstring, int> 인터페이스를 구현하는 single_threaded_map 생성
    IMap<hstring, int> playerScores = winrt::single_threaded_map<hstring, int>();

    // Insert 메서드로 키-값 저장
    playerScores.Insert(to_hstring("player1"), 100);
    playerScores.Insert(to_hstring("player2"), 150);
    playerScores.Insert(to_hstring("player3"), 120);

    // 모든 항목 순회 (pair는 IKeyValuePair<hstring, int>)
    std::cout << "All player scores:" << std::endl;
    for (const auto& pair : playerScores) {
        // .Key() 및 .Value() 메서드로 접근
        std::cout << to_string(pair.Key()) << ": " 
                  << pair.Value() << std::endl;
    }

    // Lookup 메서드로 특정 키의 값 조회
    try {
        hstring playerKey = to_hstring("player2");
        int p2Score = playerScores.Lookup(playerKey);
        std::cout << "Lookup 'player2': " << p2Score << std::endl;

        // 없는 키 조회 시도 (예외 발생)
        std::cout << "Looking up 'player4'..." << std::endl;
        playerScores.Lookup(to_hstring("player4"));

    } catch (winrt::hresult_error const& ex) {
        // Lookup이 실패하면 (키가 없으면) hresult_error 예외 발생
        std::cout << "Lookup 'player4' failed as expected: " 
                  << to_string(ex.message()) << std::endl;
    }

    return 0;
}
```
   
  
**WinRT 컬렉션 vs C++ 표준 라이브러리:**  
WinRT IVector와 std::vector는 비슷하지만 차이가 있다.  
IVector는 COM 인터페이스다. 프로세스 경계를 넘어서 전달할 수 있다. WinRT API 간에 데이터를 쉽게 공유할 수 있다.  
std::vector는 순수 C++ 타입이다. STL 알고리즘과 통합되어 있다.  
실제 게임 서버 코드에서는 내부 로직에는 std::vector와 std::map을 사용하고, WinRT API 호출 시에만 IVector와 IMap을 사용하는 것이 일반적이다.  
  

**GameServer 관점에서의 활용:**  
게임 서버는 클라이언트의 목록을 관리한다. 각 클라이언트의 세션 정보, 게임 데이터 등을 저장해야 한다. std::map<uint32_t, PlayerSession>으로 플레이어 ID를 키로 하여 세션을 저장할 수 있다.  
  
또한 이벤트 로그나 메시지 큐도 벡터로 관리할 수 있다. std::vector<GameEvent>로 모든 이벤트를 순서대로 저장하고 순회할 수 있다.  
  



## 2.6 종합 예제: 설정 관리자 구현 
이제 지금까지 배운 모든 핵심 타입을 통합하는 실전 예제를 만들어보자. 게임 서버의 설정을 관리하는 시스템이다.
  
**예제의 목적:**  
이 설정 관리자는 JSON 파일에서 서버 설정을 읽고 메모리에 캐시하며, 런타임에 업데이트할 수 있다. 멀티스레드 환경에서 안전하게 작동한다.

주요 기능:
- 파일에서 설정 로드 (hstring으로 파일 경로 다루기)
- 설정값 조회 및 수정 (IMap으로 저장)
- 변경 감지 (fire_and_forget으로 백그라운드 감시)
- 자동 저장 (handle로 파일 조작)
  
**설계 결정 이유:**  
첫째, hstring을 사용하여 WinRT 파일 API와 호환성을 유지한다.  
둘째, std::map으로 설정값을 메모리에 보관한다. 빠른 조회가 가능하다.  
셋째, std::mutex로 멀티스레드 접근을 제어한다.  
넷째, fire_and_forget으로 파일 감시를 백그라운드에서 실행한다. 메인 게임 로직을 블로킹하지 않는다.  
  
**전체 코드:**

```cpp
#include <iostream>
#include <string>
#include <map>
#include <mutex>
#include <memory>
#include <sstream>
#include <chrono>
#include <thread>
#include <winrt/Windows.Foundation.h>
#include <winrt/Windows.Foundation.Collections.h> // IPropertySet
#include <winrt/Windows.System.Threading.h>     // ThreadPoolTimer

using namespace winrt;
using namespace Windows::Foundation;
using namespace Windows::Foundation::Collections;
using namespace Windows::System::Threading;

// ConfigManager 클래스를 WinRT 컴포넌트처럼 개선
class ConfigManager {
private:
    // std::map<string, string> 대신 IPropertySet 사용
    // IPropertySet은 IMap<hstring, IInspectable>과 같음
    IPropertySet configData; 
    std::mutex configMutex;
    hstring configFilePath;
    
    // 백그라운드 모니터링을 위한 WinRT 타이머
    ThreadPoolTimer periodicTimer = nullptr;

    // 파일에서 설정 로드 (IPropertySet에 타입-안전하게 저장)
    bool LoadConfigFile() {
        std::lock_guard<std::mutex> lock(configMutex);
        
        // IPropertySet은 clear()가 없으므로 새로 만듭니다.
        // (또는 기존 맵을 순회하며 Remove)
        configData = PropertySet{}; // 새 PropertySet 인스턴스 생성

        // winrt::box_value를 사용해 실제 타입으로 저장
        configData.Insert(L"server_port", winrt::box_value(8080));
        configData.Insert(L"max_players", winrt::box_value(100));
        configData.Insert(L"game_tick_rate", winrt::box_value(60));
        configData.Insert(L"db_connection_string", winrt::box_value(L"localhost:5432"));
        configData.Insert(L"log_level", winrt::box_value(L"info"));
        
        // hstring은 wchar_t 기반이므로 wcout 사용
        std::wcout << L"[ConfigManager] Configuration loaded: " 
                   << configFilePath.c_str() << std::endl;
        
        return true;
    }

    // 파일에 설정 저장 (IInspectable을 다시 언박싱)
    bool SaveConfigFile() {
        std::lock_guard<std::mutex> lock(configMutex);
        
        std::wcout << L"[ConfigManager] Configuration saved to: " 
                   << configFilePath.c_str() << std::endl;
        
        for (auto const& pair : configData) {
            std::wcout << L"  " << pair.Key().c_str() << L" = ";

            // IInspectable 값을 원래 타입으로 확인하고 출력
            auto value = pair.Value();
            auto type = value.Type();

            if (type == PropertyType::String) {
                std::wcout << winrt::unbox_value<hstring>(value).c_str() << std::endl;
            } else if (type == PropertyType::Int32) {
                std::wcout << winrt::unbox_value<int32_t>(value) << std::endl;
            }
            // ... (다른 타입들도 필요시 추가:
            // else if (type == PropertyType::Boolean) { ... }
        }
        
        return true;
    }

public:
    ConfigManager() : configFilePath(L"game_server.config") {
        // 생성자에서 PropertySet 인스턴스 생성
        configData = PropertySet{};
    }

    ~ConfigManager() {
        StopMonitoring(); // 소멸 시 타이머 정지
    }

    // 설정 초기화
    bool Initialize(hstring filePath) {
        configFilePath = filePath;
        
        if (!LoadConfigFile()) {
            std::wcerr << L"[ConfigManager] Failed to load configuration" << std::endl;
            return false;
        }
        
        // 백그라운드 모니터링 시작
        StartMonitoring();
        
        return true;
    }

    // [개선] 템플릿을 사용한 타입-안전한 설정값 조회
    template<typename T>
    T GetValue(hstring const& key, T const& defaultValue) {
        std::lock_guard<std::mutex> lock(configMutex);
        
        if (configData.HasKey(key)) {
            try {
                // winrt::unbox_value로 IInspectable을 요청한 T 타입으로 변환
                return winrt::unbox_value<T>(configData.Lookup(key));
            }
            catch (winrt::hresult_error const&) {
                // 타입 불일치 등 오류 발생 시 기본값 반환
                return defaultValue;
            }
        }
        return defaultValue;
    }

    // [개선] 타입-안전한 설정값 수정 (IInspectable 사용)
    bool SetValue(hstring const& key, IInspectable const& value) {
        {
            std::lock_guard<std::mutex> lock(configMutex);
            // Insert가 키가 존재하면 값을 덮어씁니다.
            configData.Insert(key, value);
        }
        return SaveConfigFile();
    }

    // 설정값 존재 확인 (hstring 사용)
    bool HasKey(hstring const& key) {
        std::lock_guard<std::mutex> lock(configMutex);
        return configData.HasKey(key);
    }

    // 모든 설정값 출력 (wcout 사용)
    void PrintAllConfigs() {
        // SaveConfigFile이 이미 출력 로직을 가지고 있으므로 재사용
        // (실제로는 별도 함수로 분리하는 것이 좋음)
        std::wcout << L"\n=== Current Configuration ===" << std::endl;
        SaveConfigFile();
        std::wcout << L"============================\n" << std::endl;
    }

    // [개선] fire_and_forget 대신 ThreadPoolTimer 사용
    void StartMonitoring() {
        StopMonitoring(); // 혹시 이미 실행 중이면 중지

        // 10초 주기로 LoadConfigFile()을 호출하는 타이머 생성
        periodicTimer = ThreadPoolTimer::CreatePeriodicTimer(
            [this](ThreadPoolTimer const& /*timer*/) {
                // 이 람다는 스레드 풀에서 실행됩니다.
                // LoadConfigFile()은 내부에 std::mutex가 있으므로 스레드-안전합니다.
                std::wcout << L"[ConfigManager] Timer fired: Reloading configuration..." << std::endl;
                LoadConfigFile();
            },
            std::chrono::seconds(10) // 10초 주기
        );
    }

    // 모니터링 중지
    void StopMonitoring() {
        if (periodicTimer) {
            periodicTimer.Cancel();
            periodicTimer = nullptr;
        }
    }
};

int main() {
    winrt::init_apartment();

    // std::cout 대신 std::wcout 사용
    std::wcout << L"=== Game Server Configuration Manager ===" << std::endl;

    ConfigManager configMgr;
    // hstring을 직접 전달 (L"" 리터럴 사용)
    if (!configMgr.Initialize(L"game_server.json")) {
        std::wcerr << L"Failed to initialize ConfigManager" << std::endl;
        return 1;
    }

    configMgr.PrintAllConfigs();

    std::wcout << L"\n=== Individual Config Access ===" << std::endl;
    
    // GetValue<T>를 사용하여 타입-안전하게 조회
    std::wcout << L"Server Port: " 
               << configMgr.GetValue<int>(L"server_port", 0) << std::endl;
    std::wcout << L"Max Players: " 
               << configMgr.GetValue<int>(L"max_players", 0) << std::endl;
    std::wcout << L"Game Tick Rate: " 
               << configMgr.GetValue<int>(L"game_tick_rate", 30) << std::endl;
    std::wcout << L"DB String: "
               << configMgr.GetValue<hstring>(L"db_connection_string", L"").c_str() << std::endl;


    std::wcout << L"\n=== Modifying Configuration ===" << std::endl;
    
    // winrt::box_value를 사용해 타입-안전하게 수정
    configMgr.SetValue(L"max_players", winrt::box_value(200));
    configMgr.SetValue(L"log_level", winrt::box_value(L"debug"));

    configMgr.PrintAllConfigs();

    std::wcout << L"\n=== Key Existence Check ===" << std::endl;
    if (configMgr.HasKey(L"server_port")) {
        std::wcout << L"server_port key exists" << std::endl;
    }
    if (!configMgr.HasKey(L"invalid_key")) {
        std::wcout << L"invalid_key does not exist" << std::endl;
    }

    std::wcout << L"\n=== Using Default Values ===" << std::endl;
    hstring unknownValue = configMgr.GetValue<hstring>(
        L"unknown_setting", L"default_value");
    std::wcout << L"Unknown setting: " << unknownValue.c_str() << std::endl;

    // 몇 초 대기하여 모니터링 타이머가 동작할 시간 확보
    std::wcout << L"\n=== Waiting 12 seconds for monitoring timer to fire... ===" << std::endl;
    std::this_thread::sleep_for(std::chrono::seconds(12));

    // configMgr이 소멸하기 전에 타이머를 명시적으로 중지 (소멸자에서도 호출됨)
    configMgr.StopMonitoring();

    std::wcout << L"\n=== Configuration Manager Demo Complete ===" << std::endl;

    return 0;
}
```  
  
1.  **`IPropertySet` :**

      * `IPropertySet`은 `IMap<hstring, IInspectable>`의 별칭으로, **문자열 키**와 **모든 WinRT 객체(값)**를 매핑한다.
      * 이를 통해 `"8080"` 같은 문자열 대신 `8080` (int)을, `"true"` 대신 `true` (bool)를 **타입 그대로** 저장할 수 있다.
      * `std::stoi` 같은 수동 변환이 필요 없어지고 타입 안정성이 매우 높아진다.

2.  **`ThreadPoolTimer` :**

      * `fire_and_forget` 루프와 `std::this_thread::sleep_for`를 사용하는 대신, `Windows::System::Threading::ThreadPoolTimer`를 사용한다.
      * `CreatePeriodicTimer`를 사용하면 정해진 주기(예: 10초)마다 지정된 람다(Lambda)를 **자동으로 실행**해 준다.
      * `while(isMonitoring)` 플래그나 수동 `sleep`이 필요 없어져 코드가 매우 간결해진다.

3.  **와이드 문자열 (`hstring` / `wchar_t`) 사용:**

      * WinRT의 네이티브 문자열은 `hstring`이며, 이는 UTF-16 (`wchar_t`) 기반이다.
      * `std::string`과 `to_string`/`to_hstring` 변환을 반복하는 대신, `hstring`, `std::wstring`, `std::wcout`을 사용하여 문자열 처리를 통일한다.

4.  **타입 안전한 Get/Set:**

      * `SetValue`는 `winrt::box_value`를 사용해 모든 타입을 `IInspectable`로 박싱(boxing)하여 저장한다.
      * `GetValue`는 템플릿(template)과 `winrt::unbox_value`를 사용해 저장된 타입을 안전하게 언박싱(unboxing)한다.

  
## 핵심 컬렉션 인터페이스들을 기능별로 분류

### 1. 기본 순회 인터페이스 (Iteration)
모든 WinRT 컬렉션의 가장 기본이 되는 인터페이스이다. `for...each`와 같은 반복 구문을 가능하게 한다.

  * `IIterable<T>`
      * 컬렉션이 "순회 가능"함을 나타내는 가장 기본적인 인터페이스이다.
      * `First()` 메서드를 통해 `IIterator<T>`를 반환한다.
      * C#의 `IEnumerable<T>`와 동일한 역할을 한다.
  * `IIterator<T>`
      * 컬렉션을 순회하는 "반복자"입이다.
      * `Current` (현재 항목), `HasCurrent` (항목 존재 여부), `MoveNext` (다음 항목으로 이동) 메서드를 가진다.
      * C#의 `IEnumerator<T>`와 동일한 역할을 한다.

### 2. 벡터 계열 (Vector / List)
순서가 있는 데이터 목록을 나타낸다. C++의 `std::vector`나 C\#의 `List<T>`와 유사하다.

  * `IVector<T>`
      * **읽기/쓰기**가 모두 가능한 동적 배열(리스트)이다.
      * **주요 메서드:** `Append` (추가), `SetAt` (수정), `GetAt` (조회), `RemoveAt` (삭제), `Size` (크기) 등.
  * `IVectorView<T>`
      * **읽기 전용** 벡터이다. `IVector<T>`의 "뷰(View)" 역할을 한다.
      * API가 내부 데이터를 수정 불가능한 상태로 안전하게 반환할 때 사용된다.
      * **주요 메서드:** `GetAt` (조회), `Size` (크기). (수정/삭제/추가 메서드가 없다.)

### 3. 맵 계열 (Map / Dictionary)
고유한 키(Key)와 값(Value)의 쌍으로 데이터를 저장한다. C++의 `std::map`이나 C#의 `Dictionary<K, V>`와 유사하다.

  * `IMap<K, V>`
      * **읽기/쓰기**가 모두 가능한 맵(사전)이다.
      * **주요 메서드:** `Insert` (삽입/수정), `Lookup` (조회), `Remove` (삭제), `HasKey` (키 확인), `Size` (크기) 등.
  * `IMapView<K, V>`
      * **읽기 전용** 맵입니다. `IMap<K, V>`의 "뷰(View)" 역할을 한다.
      * 수정 불가능한 맵을 반환할 때 사용된다.
      * **주요 메서드:** `Lookup` (조회), `HasKey` (키 확인), `Size` (크기).
  * `IKeyValuePair<K, V>`
      * `IMap` 또는 `IMapView`를 순회(`IIterable`)할 때 반환되는 단일 "키-값 쌍" 항목이다.
      * C++의 `std::pair`와 유사하다.
      * **주요 속성:** `Key`, `Value`.
  
### 4. 데이터 바인딩용 (Observable Collections)
주로 XAML 기반의 UI 프로그래밍(UWP, WinUI)에서 데이터 바인딩을 위해 사용된다. 컬렉션에 **변경 사항이 생기면 이벤트를 발생시켜** UI가 자동으로 업데이트되도록 한다.

  * `IObservableVector<T>`
      * `IVector<T>`를 상속한다.
      * 항목이 추가, 삭제, 수정될 때 `VectorChanged` 이벤트를 발생시킨다.
      * C\#의 `ObservableCollection<T>`와 유사하다.
  * `IObservableMap<K, V>`
      * `IMap<K, V>`를 상속한다.
      * 항목이 추가, 삭제, 수정될 때 `MapChanged` 이벤트를 발생시킨다.

이와 관련된 이벤트 핸들러 및 인자 인터페이스도 존재한다.

  * `VectorChangedEventHandler<T>` (이벤트 핸들러)
  * `IVectorChangedEventArgs` (이벤트 인자)
  * `MapChangedEventHandler<K, V>` (이벤트 핸들러)
  * `IMapChangedEventArgs<K>` (이벤트 인자)
  
### 5. 특수 목적 컬렉션

  * `IPropertySet`
      * `IMap<hstring, IInspectable>` 인터페이스를 상속하는 특별한 맵이다.
      * `IInspectable`은 WinRT의 모든 객체가 구현하는 `object`와 같은 기본 타입이다.
      * **문자열(hstring) 키**와 **어떤 타입의 WinRT 객체라도** 값으로 저장할 수 있는 "속성 모음(Property Bag)"이다.
      * 다양한 유형의 데이터를 한꺼번에 전달해야 할 때 (예: 설정 값, 활성화 인자) 매우 유용하게 사용된다.


### 📚 요약 테이블

| 인터페이스 (Interface) | 용도 | 주요 특징 | C++/C\# 유사 객체 |
| :--- | :--- | :--- | :--- |
| **`IIterable<T>`** | 순회(Iteration) | `First()`로 `IIterator` 반환 | `IEnumerable<T>` |
| **`IIterator<T>`** | 반복자 | `Current`, `MoveNext` | `IEnumerator<T>` |
| **`IVector<T>`** | 리스트 | 읽기/쓰기 가능 | `std::vector<T>`, `List<T>` |
| **`IVectorView<T>`** | 읽기 전용 리스트 | 읽기 전용 (View) | `std::span<T>`, `IReadOnlyList<T>` |
| **`IMap<K, V>`** | 맵/사전 | 읽기/쓰기 가능 | `std::map<K, V>`, `Dictionary<K, V>` |
| **`IMapView<K, V>`** | 읽기 전용 맵 | 읽기 전용 (View) | `IReadOnlyDictionary<K, V>` |
| **`IKeyValuePair<K, V>`**| 맵 항목 | `Key`와 `Value` 속성 | `std::pair<K, V>`, `KeyValuePair<K, V>`|
| **`IObservableVector<T>`**| UI 바인딩 | 변경 시 `VectorChanged` 이벤트 발생 | `ObservableCollection<T>` |
| **`IObservableMap<K, V>`** | UI 바인딩 | 변경 시 `MapChanged` 이벤트 발생 | (C\#에는 기본 제공 없음) |
| **`IPropertySet`** | 속성 모음 | `IMap<hstring, IInspectable>` | (유사 객체 없음, 동적 객체와 유사) |

C++/WinRT 환경에서는 `std::vector`나 `std::map`을 사용하더라도, API 경계를 넘을 때 이들 `IVector`나 `IMap` 인터페이스로 **자동 변환(프로젝션)**되는 경우가 많다. 하지만 API가 명시적으로 `IVector`를 요구하거나 반환할 때, 또는 직접 WinRT 컴포넌트를 만들 때는 위 인터페이스들과 `winrt::single_threaded_vector` 같은 구체적인 구현체를 잘 이해하고 사용해야 한다.   
**WinRT API와 상호작용할 때는 그냥 표준 C++ 컨테이너를 사용**하면 된다. C++/WinRT가 "프로젝션(Projection)"을 통해 마법처럼 작동시킨다
