# Chapter 1. WinRT 시작하기

## 1.1 WinRT가 필요한 이유
여러분이 온라인 게임 서버를 개발할 때 가장 먼저 직면하는 문제는 무엇일까? 바로 네트워킹과 스레딩이다. Windows 플랫폼에서 고성능 서버를 만들려면 현대적이고 효율적인 API가 필요하다.

전통적인 Windows API(예: Winsock, CreateThread)는 강력하지만 매우 저수준이다. 개발자가 직접 메모리 관리, 리소스 정리, 에러 처리를 모두 해야 한다. 이는 코드를 복잡하게 만들고 버그가 발생하기 쉽다.

WinRT(Windows Runtime)는 이러한 문제를 해결한다. WinRT는 Microsoft가 설계한 현대적인 컴포넌트 객체 모델로, 다음과 같은 장점을 제공한다.

첫째, **자동 메모리 관리**이다. WinRT 타입들은 스마트 포인터 기반으로 설계되어 있어 개발자가 new/delete를 직접 호출할 필요가 없다. 참조 카운팅이 자동으로 이루어진다.

둘째, **강타입 시스템**이다. C++/WinRT는 컴파일 시점에 타입을 검증하므로 런타임 에러가 훨씬 적다. 동적 COM 호출 대신 정적 타입 검사를 수행한다.

셋째, **비동기 프로그래밍의 원활한 지원**이다. co_await 키워드를 사용하여 네트워크 I/O를 마치 동기 코드처럼 작성할 수 있다. 콜백 지옥에서 벗어날 수 있다.

넷째, **일관된 API 디자인**이다. 네트워킹, 스레딩, 파일 I/O 모두 비슷한 패턴으로 설계되어 있어 학습 곡선이 낮다.

게임 서버 관점에서 보면 이러한 특징들이 개발 속도와 코드 품질을 크게 향상시킨다. 수천 개의 동시 클라이언트를 처리해야 하는 상황에서 WinRT의 비동기 지원과 메모리 안전성은 매우 중요하다.
  


## 1.2 COM과 WinRT의 관계
WinRT를 제대로 이해하려면 COM(Component Object Model)을 알아야 한다. WinRT는 사실 COM의 진화형이기 때문이다.

COM은 1990년대 Windows에서 컴포넌트 기반 소프트웨어를 만들기 위해 설계되었다. COM의 핵심 아이디어는 인터페이스 기반 프로그래밍이다. 모든 COM 객체는 IUnknown 인터페이스를 구현하고, 이를 통해 QueryInterface, AddRef, Release 메서드를 제공한다.

예를 들어, 전통적인 COM 코드는 다음과 같이 작성되어야 한다:

```cpp
IStream* pStream = nullptr;
HRESULT hr = CreateStream(&pStream);
if (SUCCEEDED(hr)) {
    // 작업 수행
    pStream->Release();  // 수동으로 해제
}
```

이 코드의 문제점은 여러 가지다. 먼저 HRESULT를 계속 확인해야 한다. 또한 리소스 해제를 수동으로 해야 한다. 예외가 발생하면 Release()를 호출하지 못할 수도 있다.

WinRT는 COM의 인터페이스 기반 설계는 유지하되, C++ 현대 기능들을 활용하여 이러한 문제들을 해결한다.

**WinRT의 핵심 개선 사항들:**

첫째, **스마트 포인터 자동화**이다. winrt::com_ptr<T>나 winrt::Windows::Foundation::IUnknown 같은 스마트 포인터 타입을 사용하면 AddRef/Release가 자동으로 호출된다. 개발자는 new/delete를 쓸 일이 거의 없다.

둘째, **예외 기반 에러 처리**이다. HRESULT 대신 표준 C++ 예외를 사용한다. winrt::check_hresult() 함수로 HRESULT를 예외로 변환할 수 있다.

셋째, **현대적인 문법 지원**이다. C++/WinRT는 C++17/20의 기능들을 활용한다. co_await 코루틴, auto 타입 추론, 범위 기반 for 루프 등이 모두 자연스럽게 작동한다.

넷째, **타입 안전성**이다. WinRT API들이 제너릭과 템플릿으로 설계되어 있어 컴파일 시점에 타입을 검증할 수 있다.

간단히 말해, **COM은 기반이고 WinRT는 그 위에 쌓은 현대적인 추상화 계층**이다. 여러분은 COM의 복잡성에 신경 쓰지 않고 깔끔한 C++ 코드를 작성할 수 있다.
  


## 1.3 개발 환경 준비하기
WinRT 프로그래밍을 시작하려면 적절한 개발 환경을 구성해야 한다.

**필수 요구사항:**

첫째, Windows 10 이상의 운영 체제가 필요하다. WinRT는 Windows 플랫폼 고유 API이므로 Windows 환경에서만 개발 가능하다.

둘째, Visual Studio 2019 이상을 설치해야 한다. Community 버전으로도 충분하다. 설치할 때 "C++를 사용한 데스크톱 개발" 워크로드를 반드시 선택해야 한다.

셋째, C++/WinRT NuGet 패키지를 프로젝트에 추가해야 한다. Visual Studio의 NuGet 패키지 관리자에서 "cppwinrt"를 검색하여 설치한다.

넷째, Windows SDK를 최신 버전으로 업데이트해야 한다. WinRT API들이 Windows SDK에 포함되어 있기 때문이다. Visual Studio 설치 관리자에서 "Windows 11 SDK" (또는 사용 중인 Windows 버전의 SDK)를 설치한다.

**프로젝트 생성 단계:**

Visual Studio에서 새 프로젝트를 생성할 때 "C++ 콘솔 앱"을 선택한다. 데스크톱 애플리케이션으로 설정한다.

프로젝트 속성에서 다음을 설정한다:
- C++ 표준: C++17 이상 (설정 > 고급 > C++ 언어 표준)
- 플랫폼 도구 집합: v142 이상 (설정 > 일반 > 플랫폼 도구 집합)

프로젝트 파일(.vcxproj)을 텍스트 편집기로 열어 다음을 추가한다:

```xml
<PropertyGroup>
  <CppWinRTEnabled>true</CppWinRTEnabled>
  <CppWinRTRootNamespaceName>winrt</CppWinRTRootNamespaceName>
</PropertyGroup>
```

이제 빌드하면 WinRT 헤더 파일들이 자동으로 생성된다.

## 1.4 첫 번째 C++/WinRT 프로그램 (10라인)
이제 첫 WinRT 프로그램을 작성해보자. 가장 간단한 예제로 현재 시스템 시간을 가져오는 프로그램을 만들겠다.

```cpp
#include <iostream>
#include <winrt/Windows.Foundation.h>

using namespace winrt;
using namespace Windows::Foundation;

int main() {
    DateTime now = SystemInformation::LocalTime;
    std::cout << "Current time available" << std::endl;
    return 0;
}
```

이 코드를 분석해보자. 첫 번째와 두 번째 줄은 필요한 헤더를 포함한다. iostream은 표준 출력용이고, winrt/Windows.Foundation.h는 WinRT의 기본 타입들을 정의한다.

네 번째와 다섯 번째 줄의 using 선언은 네임스페이스 이름을 줄여준다. 이 없으면 winrt::Windows::Foundation 같이 길게 써야 한다.

main 함수에서 가장 중요한 부분은 DateTime이다. DateTime은 WinRT의 기본 제공 타입이며, 시스템 시간을 나타낸다. SystemInformation::LocalTime을 호출하면 현재 로컬 시간을 얻는다.

**핵심 포인트들:**

DateTime은 여러분이 new로 할당해야 하는 동적 객체가 아니다. WinRT의 기본 타입처럼 스택에 할당된다.

SystemInformation은 정적 API이므로 객체를 생성할 필요가 없다. Windows의 시스템 정보에 바로 접근할 수 있다.

예외 처리가 필요 없다. 이 코드는 거의 항상 성공한다. WinRT의 많은 기본 API들은 예외를 던지지 않는다.

이 간단한 예제가 여러분에게 보여주는 것은 WinRT 프로그래밍이 기존 Windows API보다 훨씬 간결하다는 점이다. 10줄로 완성된 프로그램이다.
  


## 1.5 빌드 과정 이해하기
WinRT 프로그램을 빌드할 때는 일반적인 C++ 프로젝트와 다른 과정이 일어난다. 이를 이해하면 빌드 에러가 발생했을 때 빠르게 대응할 수 있다.

**빌드 단계:**

첫 번째 단계는 메타데이터 생성이다. WinRT의 모든 타입 정보는 Windows 메타데이터 파일(.winmd)에 저장되어 있다. Visual Studio는 이 메타데이터를 읽어 여러분의 프로젝트에 필요한 헤더 파일을 생성한다. 이 파일들은 보통 프로젝트의 임시 디렉토리에 저장된다.

두 번째 단계는 헤더 생성이다. cppwinrt.exe 도구가 .winmd 파일을 읽어 C++ 헤더 파일들을 만든다. 예를 들어 Windows.Networking.Sockets 네임스페이스를 사용하면 해당하는 헤더가 자동으로 생성된다. 이 헤더들에는 클래스 정의, 메서드 서명, 상수들이 포함된다.

세 번째 단계는 일반적인 C++ 컴파일이다. 생성된 헤더를 포함하여 여러분의 소스 코드를 컴파일한다. 이 과정에서 WinRT 타입들이 기계 명령으로 변환된다.

네 번째 단계는 링킹이다. 컴파일된 오브젝트 파일과 Windows 라이브러리들을 연결하여 실행 파일을 만든다.

**생성되는 헤더 파일들:**

project.h 파일이 생성된다. 이것은 프로젝트의 주요 헤더로, winrt/impl/ 디렉토리의 구체적 구현 헤더들을 포함한다.

winrt/impl/Windows.Foundation.*.h 파일들이 생성된다. 각 WinRT 네임스페이스마다 헤더가 만들어진다. 예를 들어 Windows::Networking::Sockets를 사용하면 winrt/impl/Windows.Networking.Sockets.h가 생성된다.

**빌드 에러 해결 팁:**

"winrt/Windows.*.h를 찾을 수 없다"는 에러가 나면, C++/WinRT NuGet 패키지가 제대로 설치되었는지 확인한다.

"cppwinrt.exe를 찾을 수 없다"는 에러가 나면, 프로젝트 파일의 CppWinRTEnabled를 true로 설정했는지 확인한다.

특정 WinRT 타입이 정의되지 않았다는 에러가 나면, 해당 헤더를 include했는지 확인한다. 예를 들어 StreamSocket을 사용하려면 winrt/Windows.Networking.Sockets.h를 include해야 한다.

빌드가 완료되면 출력 디렉토리(보통 Debug 또는 Release)에 실행 파일이 생성된다.
  

## 1.6 종합 예제: WinRT 기반 로그 시스템 
이제 실전적인 예제를 작성해보자. 파일 입출력과 비동기 처리를 통합한 로그 시스템을 구현할 것이다.

**예제의 목적:**

이 예제는 여러 스레드에서 동시에 로그 메시지를 기록하는 상황을 다룬다. 게임 서버에서는 네트워크 이벤트, 게임 로직, 데이터베이스 쿼리 등 여러 곳에서 동시에 로그를 남긴다. 이 예제에서는 파일에 로그를 안전하게 기록하는 방법을 보여준다.

**설계 결정:**

첫째, 비동기 파일 작업을 사용한다. 파일 쓰기가 UI를 블로킹하지 않도록 한다.

둘째, 로그 메시지를 큐에 모았다가 일괄 처리한다. 파일 I/O 횟수를 줄여 성능을 향상시킨다.

셋째, 타임스탬프와 로그 레벨(INFO, WARNING, ERROR)을 함께 기록한다. 실제 게임 서버에서 필요한 정보다.

**전체 코드:**

```cpp
#include <iostream>
#include <string>
#include <vector>
#include <mutex>
#include <sstream>
#include <chrono>
#include <iomanip>

#include <winrt/Windows.Foundation.h>
#include <winrt/Windows.Storage.h>
#include <winrt/Windows.Storage.Streams.h>

using namespace winrt;
using namespace Windows::Foundation;
using namespace Windows::Storage;
using namespace Windows::Storage::Streams;

enum class LogLevel {
    Info,
    Warning,
    Error
};

struct LogEntry {
    std::string timestamp;
    LogLevel level;
    std::string message;
};

class AsyncLogger {
private:
    std::vector<LogEntry> logBuffer;
    std::mutex bufferMutex;
    StorageFile logFile = nullptr;
    bool isInitialized = false;

    std::string GetTimestamp() {
        auto now = std::chrono::system_clock::now();
        auto time = std::chrono::system_clock::to_time_t(now);
        std::ostringstream oss;
        oss << std::put_time(std::localtime(&time), "%Y-%m-%d %H:%M:%S");
        return oss.str();
    }

    std::string LevelToString(LogLevel level) {
        switch (level) {
            case LogLevel::Info:
                return "INFO";
            case LogLevel::Warning:
                return "WARN";
            case LogLevel::Error:
                return "ERROR";
            default:
                return "UNKNOWN";
        }
    }

    fire_and_forget FlushToFileAsync() {
        std::vector<LogEntry> buffer;
        {
            std::lock_guard<std::mutex> lock(bufferMutex);
            buffer = std::move(logBuffer);
            logBuffer.clear();
        }

        if (buffer.empty() || logFile == nullptr) {
            co_return;
        }

        try {
            auto stream = co_await logFile.OpenAsync(
                FileAccessMode::ReadWrite);
            stream.Seek(stream.Size());

            DataWriter writer(stream.GetOutputStreamAt(stream.Size()));

            for (const auto& entry : buffer) {
                std::string line = "[" + entry.timestamp + "] [" + 
                                   LevelToString(entry.level) + "] " + 
                                   entry.message + "\n";
                writer.WriteString(winrt::to_hstring(line));
            }

            co_await writer.StoreAsync();
            co_await stream.FlushAsync();
            stream.Close();
        }
        catch (const hresult_error& e) {
            std::cerr << "File write error: " 
                      << winrt::to_string(e.message()) << std::endl;
        }
    }

public:
    IAsyncAction InitializeAsync(hstring filePath) {
        try {
            auto folder = co_await StorageFolder::GetFolderFromPathAsync(
                Windows::Storage::ApplicationData::Current().LocalFolder().Path());
            
            logFile = co_await folder.CreateFileAsync(
                winrt::hstring(filePath),
                CreationCollisionOption::OpenIfExists);
            
            isInitialized = true;
        }
        catch (const hresult_error& e) {
            std::cerr << "Logger initialization failed: "
                      << winrt::to_string(e.message()) << std::endl;
            isInitialized = false;
        }
    }

    void Log(LogLevel level, const std::string& message) {
        if (!isInitialized) {
            return;
        }

        LogEntry entry{
            GetTimestamp(),
            level,
            message
        };

        {
            std::lock_guard<std::mutex> lock(bufferMutex);
            logBuffer.push_back(entry);

            if (logBuffer.size() >= 10) {
                FlushToFileAsync();
            }
        }
    }

    void LogInfo(const std::string& message) {
        Log(LogLevel::Info, message);
    }

    void LogWarning(const std::string& message) {
        Log(LogLevel::Warning, message);
    }

    void LogError(const std::string& message) {
        Log(LogLevel::Error, message);
    }

    fire_and_forget FlushAsync() {
        co_await FlushToFileAsync();
    }

    ~AsyncLogger() {
        std::lock_guard<std::mutex> lock(bufferMutex);
        if (!logBuffer.empty() && logFile != nullptr) {
            FlushToFileAsync();
        }
    }
};

int main() {
    winrt::init_apartment();

    AsyncLogger logger;

    std::cout << "Initializing logger..." << std::endl;

    return 0;
}
```

**코드 상세 설명:**

먼저 35-38줄을 살펴보자. LogEntry 구조체는 하나의 로그 메시지를 나타낸다. 타임스탬프, 로그 레벨, 메시지를 포함한다.

41-48줄의 AsyncLogger 클래스는 전체 로그 시스템의 핵심이다. logBuffer는 기록할 로그들을 임시로 저장하는 벡터다. bufferMutex는 여러 스레드가 동시에 logBuffer에 접근하는 것을 방지한다. logFile은 실제 파일 객체를 나타낸다.

50-58줄의 GetTimestamp() 메서드는 현재 시간을 "YYYY-MM-DD HH:MM:SS" 형식의 문자열로 반환한다. std::chrono를 사용하여 시간을 얻고 put_time으로 포맷한다.

60-71줄의 LevelToString() 메서드는 LogLevel enum을 문자열로 변환한다. 파일에 기록할 때 사람이 읽기 쉬운 형태로 만든다.

73-98줄의 FlushToFileAsync() 메서드가 가장 중요하다. fire_and_forget 반환 타입은 비동기 작업을 시작한 후 완료를 기다리지 않는다는 뜻이다.

먼저 75-79줄에서 lock_guard로 뮤텍스를 잠근다. 이렇게 하면 logBuffer를 안전하게 이동시킬 수 있다. move()를 사용하면 벡터의 데이터가 buffer로 옮겨지고 logBuffer는 비워진다.

82줄에서 파일을 연다. OpenAsync는 IAsyncOperation<IRandomAccessStream>을 반환한다. co_await로 완료를 기다린다.

85줄에서 파일의 끝으로 이동한다. stream.Size()는 현재 파일 크기를 반환하고 Seek()로 그 위치로 간다.

87-89줄에서 DataWriter를 생성한다. GetOutputStreamAt()은 특정 위치에서 출력 스트림을 만든다.

91-96줄에서 각 로그 항목을 문자열로 포맷하여 파일에 쓴다. WriteString() 메서드가 문자열을 버퍼에 추가하고 StoreAsync()가 실제로 파일에 기록한다.

100-116줄의 InitializeAsync() 메서드는 로거를 초기화한다. ApplicationData::Current().LocalFolder()는 앱 로컬 폴더를 얻는다. CreateFileAsync로 로그 파일을 만들거나 열고 logFile에 저장한다.

118-138줄의 Log() 메서드는 중앙 로깅 함수다. 새 LogEntry를 만들고 logBuffer에 추가한다. 버퍼 크기가 10 이상이면 자동으로 FlushToFileAsync()를 호출한다. 이렇게 하면 로그가 자동으로 일괄 처리되어 성능이 좋아진다.

**개선 방안과 확장 가능성:**

이 로거는 몇 가지 면에서 개선할 수 있다.

첫째, 현재는 FlushAsync()를 명시적으로 호출해야 마지막 로그들이 파일에 쓰인다. 타이머를 추가하여 일정 시간마다 자동으로 flush하는 기능을 추가할 수 있다.

둘째, 로그 파일 크기가 너무 커지면 구 파일을 아카이브하는 로테이션 기능을 추가할 수 있다.

셋째, 네트워크 로깅을 지원할 수 있다. 중요한 로그는 원격 서버로도 전송할 수 있다.

넷째, 로그 필터링을 추가할 수 있다. 특정 레벨 이상의 로그만 저장하도록 설정할 수 있다.

다섯째, 구조화된 로깅을 지원할 수 있다. 현재는 단순 텍스트이지만 JSON 형식으로 저장하면 나중에 파싱하기 쉬워진다.

게임 서버 개발에서 좋은 로깅 시스템은 매우 중요하다. 서버 문제를 디버깅할 때 로그가 가장 중요한 정보원이기 때문이다. 이 예제의 비동기 패턴은 게임 서버의 다른 부분에서도 광범위하게 사용될 수 있다.

## C++/WinRT 설치  (위의 설치 및 시작이 잘 되지 않았다면 이것을 참고한다)
C++/WinRT을 Visual Studio 2022 환경에서 사용하는 기본적인 절차는 다음과 같다. 온라인 게임 서버 개발자라는 상황을 고려하면, 데스크톱/네이티브 C++ 프로젝트에 WinRT API를 끌어오는 시나리오일 가능성이 높으므로 그에 맞춰 정리한다.


### 1. 설치 및 사전 조건 확인

* Visual Studio 2022 설치 시 **“데스크톱 개발용 C++(Desktop development with C++)”** 워크로드를 포함해야 한다.   
* 또한 **“유니버설 Windows 플랫폼 개발(Universal Windows Platform development)”** 워크로드에서 **C++ (v14x) UWP 도구** 옵션이 체크되어 있어야 한다.  
* Windows SDK 버전이 최소 `10.0.17134.0` (Windows 10 버전 1803) 이상이어야 한다.  
* 개발자 모드(Developer mode)가 활성화되어 있으면 빌드/배포에 유리하다.  

---

### 2. 새 프로젝트 만들기 또는 기존 프로젝트에 추가

#### 새 프로젝트

* VS 2022에는 C++/WinRT용 프로젝트 및 항목 템플릿이 기본 제공되어 있다.  
* 예: “콘솔 애플리케이션(C++/WinRT)”, “Windows 데스크톱 애플리케이션(C++/WinRT)” 등의 템플릿을 이용하면 비교적 수월하게 시작 가능하다.  

#### 기존 프로젝트에 C++/WinRT 추가

* 이미 만들어진 Win32 또는 데스크톱 C++ 프로젝트에 WinRT 지원을 추가하고 싶다면, 아래 절차를 따르면 된다.  

  1. 프로젝트 속성 → 일반(General) → Windows SDK 버전을 `10.0.17134.0` 이상으로 설정.  
  2. 프로젝트 속성 → C/C++ → 언어(Language) → C++ 언어 표준을 “ISO C++17 (/std:c++17)” 이상으로 설정.  
  3. NuGet 패키지로 Microsoft.Windows.CppWinRT 설치 (아래 “NuGet 설치” 항목 참조).
  4. 필요 시 `#include <winrt/base.h>` 등 기본 헤더를 미리컴파일 헤더(pch.h 등)에 포함.  
  5. 링커 설정: `windowsapp.lib`를 연결해야 할 수 있는데, NuGet 패키지가 이 부분을 자동 처리해 주기도 한다.  
---

### 3. NuGet 패키지 및 템플릿 활용

* 새 프로젝트로 시작할 경우에는 템플릿이 자동으로 NuGet 패키지 “Microsoft.Windows.CppWinRT”를 참조하도록 구성되어 있을 수 있다. 즉 NuGet으로 설치하지 않아도 된다.  
   
* 기존 프로젝트라면, Visual Studio에서 **프로젝트 → NuGet 패키지 관리… → 찾아보기 → “Microsoft.Windows.CppWinRT” 검색 → 설치** 순서로 추가한다.  

---

### 4. 코드 및 헤더 구조

* C++/WinRT에서 Windows 런타임 API를 사용하려면 해당 네임스페이스 헤더를 명시적으로 포함해야 한다. 예를 들어:

  ```cpp
  #include <winrt/Windows.Foundation.h>
  #include <winrt/Windows.Web.Syndication.h>
  ```

* `winrt::init_apartment()` 호출로 COM 아파트먼트를 초기화해야 하는 경우가 있다.  
* 예제:

  ```cpp
  #include "pch.h"
  using namespace winrt;
  using namespace Windows::Foundation;

  int main()
  {
      init_apartment();
      Uri uri{ L"https://blogs.windows.com/feed" };
      // …
  }
  ```

---

### 5. 주의사항 및 팁

* 빈 프로젝트(Empty Project)에서 C++/WinRT를 사용하면 템플릿이나 설정이 빠져있어 헤더를 찾지 못하는 오류(`#include <winrt/...> 파일을 열 수 없음`)가 발생할 수 있다. 실제로 Stack Overflow에서도 관련 오류가 보고되어 있다.   
* 빌드에 문제가 있다면 아래를 확인해보자:

  * Windows SDK 버전이 너무 낮지는 않은가
  * 프로젝트 속성에서 `/std:c++17` 이상이 설정되어 있는가
  * NuGet 패키지가 정상 설치되었는가
  * `windowsapp.lib`가 링크되어 있는가
* 서버 개발자라면, WinRT API 중 데스크톱/백엔드 영역에서는 제한이 있을 수 있으므로 목적에 맞는 API가 WinRT로 제공되는지 미리 확인하는 것이 좋다.
