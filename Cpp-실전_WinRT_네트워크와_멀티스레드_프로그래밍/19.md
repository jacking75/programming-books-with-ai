# Chapter 19. 메시지 프레이밍
네트워크 통신에서 가장 흔하게 마주치는 문제 중 하나는 **패킷 경계의 손실**이다. TCP는 바이트 스트림 기반이므로, 송신자가 보낸 여러 개의 논리적 메시지가 수신자에게 도착할 때 어디서 나뉘어지는지 알 수 없다. 예를 들어, "HELLO"와 "WORLD"라는 두 메시지를 순차적으로 보냈다면, 수신자는 "HELLOWORLD", "HEL"+"LOWORLD", "HELLO"+"WORLD" 등 다양한 형태로 받을 수 있다. 이러한 상황을 해결하기 위해 메시지 프레이밍 기법이 필요하다.

메시지 프레이밍은 논리적 메시지의 경계를 명확하게 정의하여 수신자가 정확하게 메시지를 분리할 수 있도록 하는 기술이다. 게임 서버 개발에서는 플레이어의 움직임 데이터, 채팅 메시지, 아이템 거래 등 수많은 메시지를 빠르고 안전하게 주고받아야 하므로, 견고한 프레이밍 시스템은 필수적이다.

---

## 19.1 스트림 기반 전송의 문제
TCP를 이용한 통신은 본질적으로 **스트림 기반**이다. 즉, 데이터가 연속적인 바이트 흐름으로 전송되며, 송신자가 보낸 메시지 단위가 그대로 보존되지 않는다. 이를 구체적으로 이해하기 위해 몇 가지 문제 상황을 살펴보자.

**문제 상황 1: 메시지 병합(Message Coalescing)**
송신자가 다음 두 메시지를 순차적으로 전송한다고 하자:
- 메시지 1: "LOGIN:user123" (길이 13)
- 메시지 2: "MOVE:100,200" (길이 12)

TCP 스택은 효율성을 위해 이 두 메시지를 한 번에 전송할 수 있다. 수신자의 입장에서 보면 "LOGIN:user123MOVE:100,200"이라는 25바이트 스트림을 받는다. 수신자는 어디서 첫 번째 메시지가 끝나고 두 번째 메시지가 시작되는지 알 수 없다.

**문제 상황 2: 메시지 분할(Message Fragmentation)**
송신자가 큰 메시지를 보낸다:
- 메시지: "INVENTORY:[item1,item2,item3,...]" (길이 1000)

네트워크 상황에 따라 이 메시지가 여러 번에 나누어 도착할 수 있다. 예를 들어, 첫 번째 recv() 호출에서 512바이트, 두 번째 recv() 호출에서 488바이트를 받는다. 수신자는 512바이트를 받았을 때 메시지가 완전한지 불완전한지 판단할 수 없다.

**문제 상황 3: 혼합(Mixed)**
위의 두 상황이 동시에 발생할 수도 있다. 메시지 A의 일부와 메시지 B의 일부, 그리고 메시지 C의 전체가 한 번의 recv()에서 수신될 수 있다.

이러한 문제를 해결하기 위해, 프로토콜 설계 단계에서부터 메시지의 경계를 명확하게 정의해야 한다. 주요 방식은 다음과 같다:

1. **길이 접두사(Length Prefix)**: 메시지 앞에 메시지의 길이를 포함한다.
2. **구분자(Delimiter)**: 메시지 끝에 특수 문자(예: '\n')를 포함한다.
3. **고정 길이(Fixed Length)**: 모든 메시지가 동일한 길이이다.
4. **헤더-바디(Header-Body)**: 헤더에서 바디의 길이를 명시한다.

각 방식은 장단점이 있으며, 게임의 특성에 따라 선택하거나 조합하여 사용한다.

## 19.2 길이 접두사 방식
길이 접두사 방식은 가장 일반적이고 안전한 방법이다. 각 메시지 앞에 메시지 본체의 길이를 4바이트 정수로 포함하고, 수신자는 먼저 길이를 읽은 후 해당 길이만큼 데이터를 읽는다.

```cpp
#include <cstring>
#include <vector>
#include <stdint.h>

// 길이 접두사 방식의 메시지 전송
class LengthPrefixFramer
{
private:
    std::vector<uint8_t> sendBuffer;

public:
    // 메시지를 길이 접두사 형식으로 인코딩
    void EncodeMessage(const std::vector<uint8_t>& message)
    {
        // 메시지 길이를 네트워크 바이트 순서(빅 엔디안)로 변환
        uint32_t length = htonl(static_cast<uint32_t>(message.size()));
        
        // 길이 4바이트를 버퍼에 추가
        const uint8_t* lengthPtr = reinterpret_cast<const uint8_t*>(&length);
        for (int i = 0; i < 4; i++)
            sendBuffer.push_back(lengthPtr[i]);
        
        // 메시지 본체를 버퍼에 추가
        for (uint8_t byte : message)
            sendBuffer.push_back(byte);
    }

    const uint8_t* GetData() const { return sendBuffer.data(); }
    size_t GetSize() const { return sendBuffer.size(); }
    void Clear() { sendBuffer.clear(); }
};

// 길이 접두사 방식의 메시지 수신
class LengthPrefixDecoder
{
private:
    std::vector<uint8_t> buffer;
    size_t readPos = 0;
    
    static const size_t HEADER_SIZE = 4;

public:
    // 네트워크에서 수신한 데이터를 버퍼에 추가
    void AddData(const uint8_t* data, size_t size)
    {
        for (size_t i = 0; i < size; i++)
            buffer.push_back(data[i]);
    }

    // 완전한 메시지를 읽을 수 있는지 확인 후 읽기
    bool TryGetMessage(std::vector<uint8_t>& outMessage)
    {
        // 헤더를 읽기 위한 최소 바이트 필요
        if (buffer.size() - readPos < HEADER_SIZE)
            return false;

        // 메시지 길이 읽기
        uint32_t netLength = 0;
        std::memcpy(&netLength, &buffer[readPos], HEADER_SIZE);
        uint32_t messageLength = ntohl(netLength);

        // 과도하게 큰 메시지 방지
        if (messageLength > 1000000)
            return false;

        // 전체 메시지(헤더 + 바디)를 읽을 수 있는지 확인
        size_t totalNeeded = readPos + HEADER_SIZE + messageLength;
        if (buffer.size() < totalNeeded)
            return false;

        // 메시지 본체 추출
        outMessage.clear();
        size_t startPos = readPos + HEADER_SIZE;
        for (size_t i = 0; i < messageLength; i++)
            outMessage.push_back(buffer[startPos + i]);

        // 읽은 위치 업데이트
        readPos = totalNeeded;
        return true;
    }

    // 처리된 데이터 정리 (메모리 효율성)
    void CompactBuffer()
    {
        if (readPos > 0)
        {
            buffer.erase(buffer.begin(), buffer.begin() + readPos);
            readPos = 0;
        }
    }
};
```

이 방식의 **장점**은 다음과 같다. 첫째, 메시지 크기에 제한이 없다(4GB까지). 둘째, 이진 데이터를 그대로 전송할 수 있다. 셋째, 프로토콜이 간단하고 빠르다. 네 번째, 대부분의 네트워크 라이브러리와 게임 엔진에서 표준으로 사용한다.

**단점**은 길이가 잘못되면 이후의 모든 메시지가 손상된다는 것이다. 따라서 길이 필드에 대한 검증이 중요하다.

## 19.3 구분자 기반 방식
구분자 기반 방식은 메시지 끝에 특수 문자나 문자열을 추가하여 메시지의 경계를 표시한다. 주로 텍스트 기반 프로토콜(HTTP, SMTP, 채팅 등)에서 사용된다.

```cpp
#include <vector>
#include <string>
#include <cstring>

class DelimiterFramer
{
private:
    std::vector<uint8_t> buffer;
    const uint8_t delimiter = '\n';

public:
    // 메시지 뒤에 구분자를 추가하여 전송
    void EncodeMessage(const std::string& message)
    {
        for (char c : message)
            buffer.push_back(static_cast<uint8_t>(c));
        buffer.push_back(delimiter);
    }

    // 네트워크에서 수신한 데이터 추가
    void AddData(const uint8_t* data, size_t size)
    {
        for (size_t i = 0; i < size; i++)
            buffer.push_back(data[i]);
    }

    // 구분자로 구분된 메시지 추출
    bool TryGetMessage(std::string& outMessage)
    {
        // 구분자 위치 찾기
        size_t delimPos = buffer.size();
        for (size_t i = 0; i < buffer.size(); i++)
        {
            if (buffer[i] == delimiter)
            {
                delimPos = i;
                break;
            }
        }

        // 구분자를 찾지 못했다면 메시지 불완전
        if (delimPos == buffer.size())
            return false;

        // 메시지 추출 (구분자 제외)
        outMessage.clear();
        for (size_t i = 0; i < delimPos; i++)
            outMessage.push_back(static_cast<char>(buffer[i]));

        // 처리된 부분 제거
        buffer.erase(buffer.begin(), buffer.begin() + delimPos + 1);
        return true;
    }

    const uint8_t* GetData() const { return buffer.data(); }
    size_t GetSize() const { return buffer.size(); }
};
```

구분자 방식의 **장점**은 인간이 읽을 수 있는 텍스트 프로토콜을 만들 수 있다는 점이고, **단점**은 메시지 본체가 구분자를 포함할 수 없다는 것이다. 예를 들어, 메시지가 개행 문자를 포함해야 한다면 이스케이핑 처리가 필요하다. 또한 텍스트 데이터 크기가 이진 형식보다 일반적으로 크다.

## 19.4 불완전한 패킷 처리
실제 네트워크 환경에서는 메시지가 여러 번에 나누어 도착하는 것이 일상이다. 프레이밍 시스템은 불완전한 메시지를 버퍼에 저장했다가, 나머지 데이터가 도착하면 완전한 메시지로 조립할 수 있어야 한다.

```cpp
#include <vector>
#include <queue>
#include <cstring>
#include <stdint.h>

class RobustMessageBuffer
{
private:
    std::vector<uint8_t> buffer;
    std::queue<std::vector<uint8_t>> completeMessages;
    
    static const size_t MAX_MESSAGE_SIZE = 1000000;
    static const size_t HEADER_SIZE = 4;

public:
    // 네트워크 소켓에서 수신한 데이터를 처리
    void ProcessReceivedData(const uint8_t* data, size_t size)
    {
        // 수신 데이터를 버퍼에 추가
        for (size_t i = 0; i < size; i++)
            buffer.push_back(data[i]);

        // 완전한 메시지들을 추출
        ExtractMessages();
    }

private:
    void ExtractMessages()
    {
        while (buffer.size() >= HEADER_SIZE)
        {
            // 메시지 길이 읽기
            uint32_t netLength = 0;
            std::memcpy(&netLength, buffer.data(), HEADER_SIZE);
            uint32_t messageLength = ntohl(netLength);

            // 길이 검증
            if (messageLength == 0 || messageLength > MAX_MESSAGE_SIZE)
            {
                // 손상된 메시지: 프로토콜 오류 처리
                buffer.clear();
                return;
            }

            // 전체 메시지를 받을 수 있는지 확인
            size_t totalSize = HEADER_SIZE + messageLength;
            if (buffer.size() < totalSize)
            {
                // 메시지가 아직 불완전함 (더 기다려야 함)
                return;
            }

            // 완전한 메시지 추출
            std::vector<uint8_t> message;
            for (size_t i = HEADER_SIZE; i < totalSize; i++)
                message.push_back(buffer[i]);
            
            completeMessages.push(message);

            // 처리된 바이트 제거
            buffer.erase(buffer.begin(), buffer.begin() + totalSize);
        }
    }

public:
    // 완전하게 처리된 메시지가 있는지 확인
    bool HasCompleteMessage() const
    {
        return !completeMessages.empty();
    }

    // 다음 메시지 가져오기
    std::vector<uint8_t> GetNextMessage()
    {
        std::vector<uint8_t> message = completeMessages.front();
        completeMessages.pop();
        return message;
    }

    // 버퍼의 현재 상태 확인
    size_t GetBufferedSize() const { return buffer.size(); }
    
    // 메시지 큐의 크기
    size_t GetQueuedMessageCount() const { return completeMessages.size(); }
};
```

이 구현의 핵심은 **상태 추적**이다. 불완전한 메시지가 버퍼에 남아있어도 문제없이 처리하며, 다음 recv() 호출에서 추가 데이터를 받으면 자동으로 메시지를 조립한다. 또한 길이 필드 검증을 통해 프로토콜 오류를 조기에 감지한다.

## 19.5 패킷 조립기 구현
복잡한 프로토콜에서는 단순히 메시지를 완전/불완전으로 분류하는 것 이상의 작업이 필요하다. **상태 머신**을 사용하면 프로토콜의 각 단계를 명확하게 표현할 수 있다.

```cpp
#include <vector>
#include <cstring>
#include <queue>
#include <stdint.h>

enum class FrameState
{
    WAITING_HEADER,      // 길이 헤더를 기다리는 중
    WAITING_BODY,        // 메시지 본체를 기다리는 중
    MESSAGE_COMPLETE     // 메시지 완성
};

class StateMachineFramer
{
private:
    std::vector<uint8_t> buffer;
    std::queue<std::vector<uint8_t>> messages;
    
    FrameState state = FrameState::WAITING_HEADER;
    uint32_t expectedMessageLength = 0;
    
    static const size_t HEADER_SIZE = 4;
    static const uint32_t MAX_MESSAGE_SIZE = 1000000;

public:
    void ProcessData(const uint8_t* data, size_t size)
    {
        // 데이터를 버퍼에 추가
        for (size_t i = 0; i < size; i++)
            buffer.push_back(data[i]);

        // 상태 머신 실행
        ProcessStateMachine();
    }

private:
    void ProcessStateMachine()
    {
        while (true)
        {
            switch (state)
            {
            case FrameState::WAITING_HEADER:
                if (!ProcessHeader())
                    return;  // 헤더가 아직 불완전
                break;

            case FrameState::WAITING_BODY:
                if (!ProcessBody())
                    return;  // 바디가 아직 불완전
                break;

            case FrameState::MESSAGE_COMPLETE:
                state = FrameState::WAITING_HEADER;
                break;
            }
        }
    }

    bool ProcessHeader()
    {
        // 헤더 크기만큼 데이터가 있는지 확인
        if (buffer.size() < HEADER_SIZE)
            return false;

        // 길이 해석
        uint32_t netLength = 0;
        std::memcpy(&netLength, buffer.data(), HEADER_SIZE);
        expectedMessageLength = ntohl(netLength);

        // 길이 검증
        if (expectedMessageLength == 0 || expectedMessageLength > MAX_MESSAGE_SIZE)
        {
            // 프로토콜 오류: 버퍼 초기화
            buffer.clear();
            return false;
        }

        // 헤더 제거
        buffer.erase(buffer.begin(), buffer.begin() + HEADER_SIZE);
        state = FrameState::WAITING_BODY;
        return true;
    }

    bool ProcessBody()
    {
        // 바디 크기만큼 데이터가 있는지 확인
        if (buffer.size() < expectedMessageLength)
            return false;

        // 메시지 추출
        std::vector<uint8_t> message;
        for (size_t i = 0; i < expectedMessageLength; i++)
            message.push_back(buffer[i]);
        
        messages.push(message);

        // 처리된 바디 제거
        buffer.erase(buffer.begin(), buffer.begin() + expectedMessageLength);
        state = FrameState::MESSAGE_COMPLETE;
        return true;
    }

public:
    bool HasMessage() const { return !messages.empty(); }
    
    std::vector<uint8_t> GetMessage()
    {
        std::vector<uint8_t> msg = messages.front();
        messages.pop();
        return msg;
    }

    FrameState GetCurrentState() const { return state; }
    size_t GetBufferedBytes() const { return buffer.size(); }
};
```

상태 머신 방식의 **장점**은 각 단계가 명확하고 디버깅이 쉽다는 것이다. 또한 복잡한 프로토콜(예: 여러 헤더가 연속되는 경우)도 상태를 추가하여 처리할 수 있다. **단점**은 코드가 조금 더 복잡해진다는 것이지만, 실전 서버 개발에서는 이 복잡성이 가치가 있다.

## 19.6 종합 예제: 완전한 프레이밍 라이브러리
이제 길이 기반, 구분자 기반, 혼합 방식을 모두 지원하는 범용 프레이밍 라이브러리를 만들어보자. 이 라이브러리는 실제 게임 서버 개발에서 즉시 사용 가능한 수준이다.

**코드의 목적**: 다양한 프로토콜 방식을 하나의 인터페이스로 제공하는 프로덕션급 프레이밍 라이브러리를 구현한다. 사용자는 프레이밍 방식을 선택하여 같은 방식의 메서드들을 사용하면 되므로, 프로토콜 변경 시에도 상위 계층 코드 변경을 최소화할 수 있다.

**설계 결정 이유**:
1. **전략 패턴(Strategy Pattern)**: 다양한 프레이밍 방식을 `IFramingStrategy` 인터페이스로 추상화한다. 런타임에 전략을 변경할 수 있으므로 유연성이 높다.
2. **예외 기반 에러 처리**: C++ 스타일의 명확한 에러 처리로 호출자가 에러 상황을 쉽게 감지할 수 있다.
3. **메모리 효율성**: 불필요한 복사를 피하고, 처리된 데이터를 즉시 제거하여 메모리 사용량을 최소화한다.
4. **스레드 안전성**: 실제 네트워크 서버는 멀티스레드 환경에서 동작하므로, 뮤텍스를 사용한 보호를 고려한다.

```cpp
#include <vector>
#include <queue>
#include <string>
#include <memory>
#include <cstring>
#include <stdint.h>
#include <mutex>

// 프레이밍 방식 정의
enum class FramingMethod
{
    LENGTH_PREFIX,      // 4바이트 길이 + 데이터
    DELIMITER,          // 데이터 + 구분자
    HYBRID              // 길이 + 타입 + 데이터
};

// 프레이밍 전략의 추상 인터페이스
class IFramingStrategy
{
public:
    virtual ~IFramingStrategy() = default;
    virtual void AddData(const uint8_t* data, size_t size) = 0;
    virtual bool TryGetMessage(std::vector<uint8_t>& outMessage) = 0;
    virtual std::vector<uint8_t> EncodeMessage(const std::vector<uint8_t>& message) = 0;
};

// 길이 접두사 전략 구현
class LengthPrefixStrategy : public IFramingStrategy
{
private:
    std::vector<uint8_t> buffer;
    static const size_t HEADER_SIZE = 4;
    static const uint32_t MAX_SIZE = 1000000;

public:
    void AddData(const uint8_t* data, size_t size) override
    {
        for (size_t i = 0; i < size; i++)
            buffer.push_back(data[i]);
    }

    bool TryGetMessage(std::vector<uint8_t>& outMessage) override
    {
        if (buffer.size() < HEADER_SIZE)
            return false;

        uint32_t netLength = 0;
        std::memcpy(&netLength, buffer.data(), HEADER_SIZE);
        uint32_t length = ntohl(netLength);

        if (length == 0 || length > MAX_SIZE)
            throw std::runtime_error("Invalid message length");

        size_t totalSize = HEADER_SIZE + length;
        if (buffer.size() < totalSize)
            return false;

        outMessage.assign(buffer.begin() + HEADER_SIZE, 
                         buffer.begin() + totalSize);
        buffer.erase(buffer.begin(), buffer.begin() + totalSize);
        return true;
    }

    std::vector<uint8_t> EncodeMessage(const std::vector<uint8_t>& message) override
    {
        std::vector<uint8_t> encoded;
        uint32_t length = htonl(static_cast<uint32_t>(message.size()));
        const uint8_t* ptr = reinterpret_cast<const uint8_t*>(&length);
        
        for (int i = 0; i < 4; i++)
            encoded.push_back(ptr[i]);
        for (uint8_t byte : message)
            encoded.push_back(byte);
        
        return encoded;
    }
};

// 구분자 기반 전략 구현
class DelimiterStrategy : public IFramingStrategy
{
private:
    std::vector<uint8_t> buffer;
    const uint8_t delimiter = '\n';
    static const size_t MAX_MESSAGE_SIZE = 100000;

public:
    void AddData(const uint8_t* data, size_t size) override
    {
        for (size_t i = 0; i < size; i++)
            buffer.push_back(data[i]);
    }

    bool TryGetMessage(std::vector<uint8_t>& outMessage) override
    {
        size_t delimPos = 0;
        bool found = false;
        
        for (size_t i = 0; i < buffer.size(); i++)
        {
            if (buffer[i] == delimiter)
            {
                delimPos = i;
                found = true;
                break;
            }
        }

        if (!found)
        {
            if (buffer.size() > MAX_MESSAGE_SIZE)
                throw std::runtime_error("Message too large");
            return false;
        }

        outMessage.assign(buffer.begin(), buffer.begin() + delimPos);
        buffer.erase(buffer.begin(), buffer.begin() + delimPos + 1);
        return true;
    }

    std::vector<uint8_t> EncodeMessage(const std::vector<uint8_t>& message) override
    {
        std::vector<uint8_t> encoded = message;
        encoded.push_back(delimiter);
        return encoded;
    }
};

// 하이브리드 전략 (길이 + 타입 ID)
class HybridStrategy : public IFramingStrategy
{
private:
    std::vector<uint8_t> buffer;
    static const size_t HEADER_SIZE = 8;  // 4바이트 길이 + 4바이트 타입
    static const uint32_t MAX_SIZE = 1000000;

public:
    void AddData(const uint8_t* data, size_t size) override
    {
        for (size_t i = 0; i < size; i++)
            buffer.push_back(data[i]);
    }

    bool TryGetMessage(std::vector<uint8_t>& outMessage) override
    {
        if (buffer.size() < HEADER_SIZE)
            return false;

        uint32_t netLength = 0;
        std::memcpy(&netLength, buffer.data(), 4);
        uint32_t length = ntohl(netLength);

        if (length == 0 || length > MAX_SIZE)
            throw std::runtime_error("Invalid message length");

        size_t totalSize = HEADER_SIZE + length;
        if (buffer.size() < totalSize)
            return false;

        outMessage.assign(buffer.begin() + HEADER_SIZE, 
                         buffer.begin() + totalSize);
        buffer.erase(buffer.begin(), buffer.begin() + totalSize);
        return true;
    }

    std::vector<uint8_t> EncodeMessage(const std::vector<uint8_t>& message) override
    {
        std::vector<uint8_t> encoded;
        
        // 메시지 길이 (4바이트)
        uint32_t length = htonl(static_cast<uint32_t>(message.size()));
        const uint8_t* lengthPtr = reinterpret_cast<const uint8_t*>(&length);
        for (int i = 0; i < 4; i++)
            encoded.push_back(lengthPtr[i]);
        
        // 메시지 타입 ID (4바이트, 이 예제에서는 0)
        uint32_t typeId = htonl(0);
        const uint8_t* typePtr = reinterpret_cast<const uint8_t*>(&typeId);
        for (int i = 0; i < 4; i++)
            encoded.push_back(typePtr[i]);
        
        // 실제 메시지 데이터
        for (uint8_t byte : message)
            encoded.push_back(byte);
        
        return encoded;
    }
};

// 메인 프레이밍 엔진 (퍼블릭 API)
class MessageFramer
{
private:
    std::unique_ptr<IFramingStrategy> strategy;
    mutable std::mutex mutex;

public:
    // 프레이밍 방식 초기화
    MessageFramer(FramingMethod method)
    {
        switch (method)
        {
        case FramingMethod::LENGTH_PREFIX:
            strategy = std::make_unique<LengthPrefixStrategy>();
            break;
        case FramingMethod::DELIMITER:
            strategy = std::make_unique<DelimiterStrategy>();
            break;
        case FramingMethod::HYBRID:
            strategy = std::make_unique<HybridStrategy>();
            break;
        default:
            throw std::runtime_error("Unknown framing method");
        }
    }

    // 네트워크에서 수신한 데이터 처리
    void ProcessReceivedData(const uint8_t* data, size_t size)
    {
        std::lock_guard<std::mutex> lock(mutex);
        strategy->AddData(data, size);
    }

    // 완전한 메시지 추출 시도
    bool TryGetMessage(std::vector<uint8_t>& outMessage)
    {
        std::lock_guard<std::mutex> lock(mutex);
        return strategy->TryGetMessage(outMessage);
    }

    // 메시지 인코딩 (전송 준비)
    std::vector<uint8_t> PrepareMessage(const std::vector<uint8_t>& message)
    {
        std::lock_guard<std::mutex> lock(mutex);
        return strategy->EncodeMessage(message);
    }

    // 문자열 메시지 헬퍼
    void ProcessReceivedString(const std::string& data)
    {
        ProcessReceivedData(
            reinterpret_cast<const uint8_t*>(data.c_str()),
            data.length()
        );
    }

    std::string PrepareStringMessage(const std::string& message)
    {
        auto encoded = PrepareMessage(
            std::vector<uint8_t>(message.begin(), message.end())
        );
        return std::string(encoded.begin(), encoded.end());
    }
};
```

**주요 부분 상세 해설**:

1. **전략 패턴의 적용**: `IFramingStrategy` 인터페이스를 정의하여 다양한 프레이밍 방식을 동일한 인터페이스로 제공한다. 새로운 프레이밍 방식이 필요하면 `IFramingStrategy`를 상속하여 구현하면 된다.

2. **메모리 관리**: 각 전략에서 `buffer.erase()`를 사용하여 처리된 데이터를 즉시 제거한다. 이는 장시간 실행되는 서버에서 메모리 누수를 방지한다.

3. **에러 검증**: 메시지 길이가 비정상적으로 크거나 0이면 예외를 던진다. 이렇게 하면 호출자가 명확하게 에러 상황을 감지할 수 있다.

4. **스레드 안전성**: `std::mutex`와 `std::lock_guard`를 사용하여 멀티스레드 환경에서도 안전하게 동작한다. 수신 스레드와 메시지 처리 스레드가 동시에 접근해도 문제없다.

5. **헬퍼 메서드**: `ProcessReceivedString()`과 `PrepareStringMessage()`는 문자열 기반 프로토콜을 사용할 때 편의를 제공한다.

**개선 방안과 확장 가능성**:

첫째, **압축 기능**을 추가할 수 있다. 큰 메시지의 경우 zlib으로 압축하면 네트워크 대역폭을 절약할 수 있다. `EncodeMessage()` 메서드에서 압축 플래그를 헤더에 추가하고 필요시 데이터를 압축한다.

둘째, **암호화 기능**을 추가할 수 있다. SSL/TLS 또는 커스텀 암호화를 적용하여 메시지 보안을 강화한다. 이 경우 암호화된 데이터 크기도 고려해야 한다.

셋째, **메시지 우선순위**를 지원할 수 있다. 우선순위 큐를 사용하여 중요한 메시지(예: 게임 시작 명령)를 먼저 처리하도록 한다.

네 번째, **적응형 버퍼 크기**를 구현할 수 있다. 버퍼가 자주 가득 찬다면 동적으로 크기를 증가시켜 성능을 개선한다.

다섯 번째, **메시지 통계**를 수집할 수 있다. 처리된 메시지 수, 평균 크기, 에러율 등을 추적하여 서버 모니터링에 활용한다.

마지막으로, **배치 처리 모드**를 추가할 수 있다. 완전한 메시지가 여러 개 있을 때 모두 반환하는 메서드를 추가하면, 메시지 처리 오버헤드를 줄일 수 있다.

이 프레이밍 라이브러리는 다음 장의 세션 관리, 채팅 서버, 게임 로비 등 모든 서버 구현에서 기초가 되므로, 충분히 이해하고 활용하는 것이 중요하다.
