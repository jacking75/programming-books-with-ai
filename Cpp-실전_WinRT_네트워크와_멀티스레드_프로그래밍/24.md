# Chapter 24. 성능 최적화
이제 여러분은 게임 서버의 기본적인 구조를 만들 수 있게 되었다. 하지만 실제 서버를 운영하면서 마주치는 가장 큰 문제는 "성능"이다. 100명의 플레이어를 처리할 수 있는 서버도 1000명이 들어오면 순식간에 느려진다.

이 장에서는 게임 서버를 빠르게 만드는 방법을 배운다. 단순히 "알고리즘을 개선하라"는 추상적인 조언이 아니라, 구체적인 기법들을 다룬다. 메모리 할당을 어떻게 줄이는지, 복사 연산을 어떻게 제거하는지, 네트워크 대역폭을 어떻게 절약하는지 보여줄 것이다.

## 24.1 병목 지점 식별하기
최적화의 첫 번째 원칙은 **"추측하지 마라. 측정하라"**이다. 여러분이 느리다고 생각하는 부분이 실제로 느린 부분이 아닐 수도 있다.

### 프로파일링의 중요성
게임 서버에서 성능을 측정하는 방법은 여러 가지가 있다.

**Windows 성능 분석기 (Windows Performance Analyzer)**

Windows는 무료로 제공하는 매우 강력한 프로파일링 도구가 있다. Visual Studio에 내장된 Profiler는 사용이 간단하지만, 더 깊은 분석이 필요하면 Windows Performance Toolkit (WPT)을 사용해야 한다. 이 도구는 CPU 사용률, 메모리 할당, 디스크 I/O, 네트워크 활동까지 모든 것을 추적할 수 있다.

**Custom Instrumentation**

대부분의 게임 엔진은 성능 측정을 위해 자체 도구를 만든다. 예를 들어, "이 함수가 얼마나 자주 호출되는지", "한 프레임에 몇 밀리초가 걸리는지" 같은 정보를 수집한다.

여기서는 간단한 성능 측정 도구를 만드는 방법을 보여준다.

```cpp
// PerformanceProfiler.h
#pragma once
#include <cstdint>
#include <chrono>
#include <string>
#include <map>
#include <vector>

// 한 번의 함수 호출 시간 측정 데이터
struct TimingData {
    std::string name;
    double totalTimeMs;         // 누적 시간 (밀리초)
    uint64_t callCount;         // 호출 횟수
    double minTimeMs;           // 최소 시간
    double maxTimeMs;           // 최대 시간
    
    double GetAverageTimeMs() const {
        return callCount > 0 ? totalTimeMs / callCount : 0.0;
    }
};

// 프로파일러 - 스코프를 벗어날 때 자동으로 시간을 측정한다
class ScopedTimer {
public:
    ScopedTimer(const std::string& name);
    ~ScopedTimer();

private:
    std::string m_name;
    std::chrono::high_resolution_clock::time_point m_startTime;
};

// 전역 프로파일러
class Profiler {
public:
    static Profiler& GetInstance();
    
    // 새로운 타이밍 데이터를 기록한다
    void RecordTiming(const std::string& name, double timeMs);
    
    // 주어진 이름의 타이밍 데이터를 가져온다
    TimingData GetTiming(const std::string& name) const;
    
    // 모든 타이밍 데이터를 반환한다
    std::vector<TimingData> GetAllTimings() const;
    
    // 모든 데이터를 초기화한다
    void Reset();
    
    // 통계를 출력한다
    void PrintStatistics() const;

private:
    Profiler() = default;
    std::map<std::string, TimingData> m_timings;
};
```

```cpp
// PerformanceProfiler.cpp
#include "PerformanceProfiler.h"
#include <iostream>
#include <iomanip>
#include <algorithm>

ScopedTimer::ScopedTimer(const std::string& name) : m_name(name) {
    m_startTime = std::chrono::high_resolution_clock::now();
}

ScopedTimer::~ScopedTimer() {
    auto endTime = std::chrono::high_resolution_clock::now();
    std::chrono::duration<double, std::milli> elapsed = endTime - m_startTime;
    
    Profiler::GetInstance().RecordTiming(m_name, elapsed.count());
}

Profiler& Profiler::GetInstance() {
    static Profiler instance;
    return instance;
}

void Profiler::RecordTiming(const std::string& name, double timeMs) {
    if (m_timings.find(name) == m_timings.end()) {
        // 처음 기록되는 이름이면 새로 만든다
        TimingData data;
        data.name = name;
        data.totalTimeMs = 0.0;
        data.callCount = 0;
        data.minTimeMs = std::numeric_limits<double>::max();
        data.maxTimeMs = 0.0;
        m_timings[name] = data;
    }
    
    // 데이터를 업데이트한다
    TimingData& data = m_timings[name];
    data.totalTimeMs += timeMs;
    data.callCount++;
    data.minTimeMs = std::min(data.minTimeMs, timeMs);
    data.maxTimeMs = std::max(data.maxTimeMs, timeMs);
}

TimingData Profiler::GetTiming(const std::string& name) const {
    auto it = m_timings.find(name);
    if (it != m_timings.end()) {
        return it->second;
    }
    return TimingData{};
}

std::vector<TimingData> Profiler::GetAllTimings() const {
    std::vector<TimingData> result;
    for (const auto& pair : m_timings) {
        result.push_back(pair.second);
    }
    
    // 총 시간 기준으로 정렬한다 (가장 오래 걸리는 것부터)
    std::sort(result.begin(), result.end(),
        [](const TimingData& a, const TimingData& b) {
            return a.totalTimeMs > b.totalTimeMs;
        });
    
    return result;
}

void Profiler::Reset() {
    m_timings.clear();
}

void Profiler::PrintStatistics() const {
    auto timings = GetAllTimings();
    
    std::cout << "\n=== Performance Statistics ===" << std::endl;
    std::cout << std::left << std::setw(30) << "Function Name"
              << std::setw(15) << "Calls"
              << std::setw(15) << "Total (ms)"
              << std::setw(15) << "Avg (ms)"
              << std::setw(15) << "Min (ms)"
              << std::setw(15) << "Max (ms)" << std::endl;
    std::cout << std::string(90, '-') << std::endl;
    
    for (const auto& timing : timings) {
        std::cout << std::left << std::setw(30) << timing.name
                  << std::setw(15) << timing.callCount
                  << std::setw(15) << std::fixed << std::setprecision(3) << timing.totalTimeMs
                  << std::setw(15) << timing.GetAverageTimeMs()
                  << std::setw(15) << timing.minTimeMs
                  << std::setw(15) << timing.maxTimeMs << std::endl;
    }
}
```

## 24.2 메모리 할당 최소화
게임 서버에서 메모리 할당은 성능을 크게 해친다. 왜냐하면 `new`와 `delete` 연산이 비싸기 때문이다. 특히 게임 루프처럼 매 틱마다 반복되는 코드에서는 메모리 할당을 피해야 한다.

### 오브젝트 풀링 (Object Pooling)
**오브젝트 풀링**은 자주 생성되고 삭제되는 객체를 미리 만들어두고, 필요할 때 재사용하는 기법이다. 예를 들어, 패킷을 나타내는 객체를 생성할 때마다 `new`로 메모리를 할당하는 대신, 미리 정해진 개수의 패킷 객체를 만들어두고 사용이 끝나면 돌려받는다.

```cpp
// ObjectPool.h
#pragma once
#include <vector>
#include <queue>
#include <memory>
#include <stdexcept>

// 제네릭 오브젝트 풀
template<typename T>
class ObjectPool {
public:
    // poolSize개의 객체를 미리 만든다
    explicit ObjectPool(size_t poolSize = 1000) {
        for (size_t i = 0; i < poolSize; ++i) {
            m_available.push(std::make_unique<T>());
        }
    }
    
    // 객체를 꺼낸다. 사용 가능한 객체가 없으면 새로 만든다.
    std::unique_ptr<T> Acquire() {
        if (m_available.empty()) {
            // 풀이 비었으므로 새로 만든다
            return std::make_unique<T>();
        }
        
        auto obj = std::move(m_available.front());
        m_available.pop();
        m_acquired++;
        
        return obj;
    }
    
    // 객체를 반환한다. 풀이 가득 차면 폐기한다.
    void Release(std::unique_ptr<T> obj) {
        if (obj == nullptr) {
            return;
        }
        
        // 객체를 초기화한다 (다음 사용을 위해)
        obj->Reset();
        
        // 풀 크기 제한
        if (m_available.size() < 2000) {  // 최대 2000개까지만 보관
            m_available.push(std::move(obj));
        }
        // 그렇지 않으면 폐기한다 (unique_ptr이 자동으로 삭제)
        
        m_acquired--;
    }
    
    // 현재 풀의 상태
    size_t GetAvailableCount() const { return m_available.size(); }
    size_t GetAcquiredCount() const { return m_acquired; }

private:
    std::queue<std::unique_ptr<T>> m_available;
    size_t m_acquired = 0;
};

// 패킷을 나타낸다
class Packet {
public:
    static constexpr size_t MAX_SIZE = 4096;
    
    Packet() : m_data(new uint8_t[MAX_SIZE]), m_size(0) {}
    ~Packet() { delete[] m_data; }
    
    // 패킷 데이터에 접근한다
    uint8_t* GetData() { return m_data; }
    const uint8_t* GetData() const { return m_data; }
    
    void SetSize(size_t size) { m_size = std::min(size, MAX_SIZE); }
    size_t GetSize() const { return m_size; }
    
    // 패킷을 초기화한다
    void Reset() {
        m_size = 0;
    }

private:
    uint8_t* m_data;
    size_t m_size;
};

// 글로벌 패킷 풀
extern ObjectPool<Packet>& GetPacketPool();
```

```cpp
// ObjectPool.cpp

    static ObjectPool<Packet> pool(1000);  // 1000개의 패킷 미리 할당
```

**오브젝트 풀 사용 예제**

```cpp
// 패킷을 보내는 함수
void SendPacket(uint32_t playerId, const uint8_t* data, size_t size) {
    // 풀에서 패킷을 꺼낸다
    auto packet = GetPacketPool().Acquire();
    
    // 데이터를 복사한다
    std::memcpy(packet->GetData(), data, size);
    packet->SetSize(size);
    
    // 네트워크로 보낸다
    SendToNetwork(playerId, packet.get());
    
    // 함수를 벗어날 때 unique_ptr이 자동으로 Release를 호출한다
    // 이렇게 하려면 Release를 오버로드된 delete처럼 사용해야 한다
    // 더 나은 방법은 custom deleter를 사용하는 것이다
}
```

더 깔끔한 구현을 위해 custom deleter를 사용할 수 있다.

```cpp
// PooledPtr.h - 풀과 자동 연동하는 스마트 포인터
#pragma once
#include <memory>

template<typename T>
class PooledPtr {
public:
    PooledPtr(std::unique_ptr<T> ptr, ObjectPool<T>& pool)
        : m_ptr(std::move(ptr)), m_pool(&pool) {}
    
    ~PooledPtr() {
        if (m_ptr) {
            m_pool->Release(std::move(m_ptr));
        }
    }
    
    T* operator->() { return m_ptr.get(); }
    const T* operator->() const { return m_ptr.get(); }
    T& operator*() { return *m_ptr; }
    const T& operator*() const { return *m_ptr; }
    
    // 이동 생성자와 이동 할당 연산자
    PooledPtr(PooledPtr&& other) noexcept
        : m_ptr(std::move(other.m_ptr)), m_pool(other.m_pool) {}
    
    PooledPtr& operator=(PooledPtr&& other) noexcept {
        if (this != &other) {
            if (m_ptr) {
                m_pool->Release(std::move(m_ptr));
            }
            m_ptr = std::move(other.m_ptr);
            m_pool = other.m_pool;
        }
        return *this;
    }

private:
    std::unique_ptr<T> m_ptr;
    ObjectPool<T>* m_pool;
};
```

**오브젝트 풀의 장점**

오브젝트 풀의 주요 이점은 다음과 같다:

1. **메모리 할당 시간 제거**: 메모리 할당은 `malloc`/`free`를 호출해야 하므로 상대적으로 느리다. 풀에서 꺼내는 것은 단순히 포인터를 반환하므로 매우 빠르다.

2. **메모리 단편화 방지**: 많은 메모리 할당과 해제가 일어나면 힙이 단편화되어 큰 메모리 블록을 할당받기 어려워진다. 풀링은 이를 방지한다.

3. **예측 가능한 성능**: 게임 루프는 일정한 속도를 유지해야 하는데, 메모리 할당은 타이밍을 예측하기 어렵다. 풀링은 성능을 예측 가능하게 만든다.

다만 오브젝트 풀의 크기를 너무 크게 설정하면 불필요한 메모리를 사용하므로, 프로파일링을 통해 적절한 크기를 결정해야 한다.

## 24.3 불필요한 복사 제거
C++에서 흔한 성능 문제는 불필요한 복사다. 함수에 큰 객체를 값으로 넘기거나, 컨테이너에서 임시 객체를 생성하는 등의 상황에서 복사가 일어난다.

### 이동 의미론 (Move Semantics) 활용
C++11부터는 **이동 의미론**이 도입되었다. 이를 통해 복사 대신 소유권 이전을 할 수 있다.

```cpp
// MoveSemantics.h
#pragma once
#include <cstdint>
#include <vector>
#include <utility>

// 게임 상태 스냅샷 - 많은 데이터를 포함한다
class GameStateSnapshot {
public:
    GameStateSnapshot() = default;
    
    // 복사 생성자 (명시적으로 삭제)
    GameStateSnapshot(const GameStateSnapshot&) = delete;
    
    // 이동 생성자 (기본 구현)
    GameStateSnapshot(GameStateSnapshot&&) = default;
    
    // 복사 할당 연산자 (명시적으로 삭제)
    GameStateSnapshot& operator=(const GameStateSnapshot&) = delete;
    
    // 이동 할당 연산자 (기본 구현)
    GameStateSnapshot& operator=(GameStateSnapshot&&) = default;
    
    // 플레이어 데이터를 추가한다 (이동으로)
    void AddPlayerData(std::vector<uint8_t> data) {
        // 복사 대신 이동으로 소유권을 이전한다
        m_playerData = std::move(data);
    }
    
    // 전체 데이터를 설정한다 (이동으로)
    void SetWorldData(std::vector<uint8_t>&& data) {
        // rvalue reference를 사용하면 복사가 일어나지 않는다
        m_worldData = std::move(data);
    }
    
    const std::vector<uint8_t>& GetPlayerData() const {
        return m_playerData;
    }
    
    const std::vector<uint8_t>& GetWorldData() const {
        return m_worldData;
    }

private:
    std::vector<uint8_t> m_playerData;
    std::vector<uint8_t> m_worldData;
};

// 패킷을 나타낸다 - 가변 크기 데이터를 포함한다
class PacketBuffer {
public:
    explicit PacketBuffer(size_t initialCapacity = 256)
        : m_data(initialCapacity), m_size(0) {}
    
    // 이동 생성자
    PacketBuffer(PacketBuffer&& other) noexcept
        : m_data(std::move(other.m_data)), m_size(other.m_size) {
        other.m_size = 0;
    }
    
    // 이동 할당 연산자
    PacketBuffer& operator=(PacketBuffer&& other) noexcept {
        if (this != &other) {
            m_data = std::move(other.m_data);
            m_size = other.m_size;
            other.m_size = 0;
        }
        return *this;
    }
    
    // 데이터를 추가한다
    void AppendData(const uint8_t* data, size_t size) {
        if (m_size + size > m_data.size()) {
            // 용량을 늘린다
            m_data.resize((m_size + size) * 2);
        }
        std::copy(data, data + size, m_data.begin() + m_size);
        m_size += size;
    }
    
    // 버퍼를 비운다 (다시 사용하기 위해)
    void Clear() {
        m_size = 0;
    }
    
    const uint8_t* GetData() const {
        return m_data.data();
    }
    
    size_t GetSize() const {
        return m_size;
    }

private:
    std::vector<uint8_t> m_data;
    size_t m_size;
};
```

```cpp
    // 호출 코드에서 스냅샷을 더 이상 사용할 수 없다.
    
    // 예: auto data = snapshot.GetPlayerData();  // OK, 복사 가능
}

// 효율적인 패킷 전송 함수
void SendPacketEfficient(uint32_t playerId, PacketBuffer&& packet) {
    // packet은 임시 객체이므로, 복사 대신 이동으로 처리한다
    // 함수 내에서 packet의 소유권을 가지고 있다
    
    // 네트워크로 전송
    // NetworkSend(playerId, packet.GetData(), packet.GetSize());
}

// 사용 예제
void GameUpdate() {
    // 스냅샷을 생성한다
    GameStateSnapshot snapshot;
    
    std::vector<uint8_t> playerData;
    playerData.push_back(1);
    playerData.push_back(2);
    playerData.push_back(3);
    
    // 벡터를 이동으로 추가한다 (복사 안 함)
    snapshot.AddPlayerData(std::move(playerData));
    
    // 이 시점에서 playerData는 비어있다
    
    // 스냅샷을 이동으로 전송한다
    ProcessSnapshot(std::move(snapshot));
    
    // 패킷 버퍼를 만들고 이동으로 보낸다
    PacketBuffer buffer;
    buffer.AppendData(nullptr, 0);  // 데이터 추가
    
    SendPacketEfficient(1, std::move(buffer));
}
```

**이동 의미론의 핵심**

이동 의미론은 다음과 같은 상황에서 성능을 크게 개선한다:

1. **큰 컨테이너 전달**: 벡터나 맵 같은 큰 객체를 함수에 넘길 때, 이동을 사용하면 전체 데이터를 복사하지 않는다.

2. **함수 반환값**: 함수가 큰 객체를 반환할 때, 컴파일러는 자동으로 이동을 사용한다 (RVO - Return Value Optimization).

3. **컨테이너 삽입**: `vector::push_back(std::move(obj))`는 원소를 복사하지 않고 이동한다.

주의할 점은, 이동한 객체를 더 이상 사용할 수 없다는 것이다. 따라서 명확한 의도를 가지고 사용해야 한다.

## 24.4 비동기 작업 오버헤드
네트워크 프로그래밍에서는 항상 비동기 작업을 다룬다. 하지만 비동기 작업은 오버헤드가 있다. 작은 작업을 비동기로 만들면 오버헤드가 이득보다 클 수 있다.

### 작업 배칭 (Task Batching)
**작업 배칭**은 여러 작은 작업을 모아서 한 번에 처리하는 기법이다. 예를 들어, 100개의 패킷을 받을 때마다 각각을 비동기 작업으로 처리하는 대신, 100개를 모아서 한 번의 비동기 작업으로 처리한다.

```cpp
// TaskBatching.h
#pragma once
#include <cstdint>
#include <vector>
#include <functional>
#include <mutex>
#include <condition_variable>
#include <thread>

// 처리할 작업의 종류
enum class TaskType {
    ProcessPacket,
    UpdatePlayerState,
    BroadcastMessage
};

// 배치에 포함될 작업
struct Task {
    TaskType type;
    uint32_t playerId;
    std::vector<uint8_t> data;
};

// 작업 배치 처리기
class BatchTaskProcessor {
public:
    BatchTaskProcessor(size_t batchSize = 100, size_t maxWaitMs = 10);
    ~BatchTaskProcessor();
    
    // 작업을 큐에 추가한다
    void EnqueueTask(const Task& task);
    
    // 배치 처리를 시작한다 (별도 스레드에서 실행)
    void Start();
    
    // 배치 처리를 중지한다
    void Stop();
    
    // 현재 큐에 있는 작업 수
    size_t GetPendingTaskCount() const;

private:
    // 배치를 처리한다
    void ProcessBatch(const std::vector<Task>& batch);
    
    // 작업 루프 (별도 스레드에서 실행)
    void WorkerThread();
    
    std::vector<Task> m_taskQueue;
    mutable std::mutex m_queueMutex;
    std::condition_variable m_queueCV;
    
    size_t m_batchSize;         // 한 배치의 최대 크기
    size_t m_maxWaitMs;         // 배치를 보낼 때까지의 최대 대기 시간
    bool m_isRunning;
    std::thread m_workerThread;
};
```

```cpp
// TaskBatching.cpp
#include "TaskBatching.h"
#include <iostream>
#include <chrono>

BatchTaskProcessor::BatchTaskProcessor(size_t batchSize, size_t maxWaitMs)
    : m_batchSize(batchSize), m_maxWaitMs(maxWaitMs), m_isRunning(false) {
}

BatchTaskProcessor::~BatchTaskProcessor() {
    Stop();
}

void BatchTaskProcessor::EnqueueTask(const Task& task) {
    {
        std::lock_guard<std::mutex> lock(m_queueMutex);
        m_taskQueue.push_back(task);
    }
    
    // 배치가 가득 찼으면 신호를 보낸다
    if (m_taskQueue.size() >= m_batchSize) {
        m_queueCV.notify_one();
    }
}

void BatchTaskProcessor::Start() {
    if (m_isRunning) return;
    
    m_isRunning = true;
    m_workerThread = std::thread(&BatchTaskProcessor::WorkerThread, this);
}

void BatchTaskProcessor::Stop() {
    if (!m_isRunning) return;
    
    m_isRunning = false;
    m_queueCV.notify_one();
    
    if (m_workerThread.joinable()) {
        m_workerThread.join();
    }
    
    // 남은 작업들을 처리한다
    std::lock_guard<std::mutex> lock(m_queueMutex);
    if (!m_taskQueue.empty()) {
        ProcessBatch(m_taskQueue);
        m_taskQueue.clear();
    }
}

void BatchTaskProcessor::WorkerThread() {
    while (m_isRunning) {
        std::vector<Task> batch;
        
        {
            std::unique_lock<std::mutex> lock(m_queueMutex);
            
            // 배치 크기만큼 기다리거나, 타임아웃이 될 때까지 기다린다
            // 이렇게 하면 배치가 채워지지 않아도 주기적으로 처리된다
            m_queueCV.wait_for(lock, std::chrono::milliseconds(m_maxWaitMs),
                [this] { return !m_taskQueue.empty() || !m_isRunning; });
            
            // 배치를 모아서 처리한다
            if (!m_taskQueue.empty()) {
                size_t batchCount = std::min(m_batchSize, m_taskQueue.size());
                batch.assign(m_taskQueue.begin(), m_taskQueue.begin() + batchCount);
                m_taskQueue.erase(m_taskQueue.begin(), m_taskQueue.begin() + batchCount);
            }
        }
        
        // 배치를 처리한다 (락을 잡지 않음)
        if (!batch.empty()) {
            ProcessBatch(batch);
        }
    }
}

void BatchTaskProcessor::ProcessBatch(const std::vector<Task>& batch) {
    // 예제 구현: 타입별로 작업을 그룹화해서 처리한다
    
    std::vector<uint32_t> playerIdsToUpdate;
    
    for (const auto& task : batch) {
        switch (task.type) {
            case TaskType::ProcessPacket:
                // 패킷 처리 로직
                std::cout << "Processing packet for player " << task.playerId << std::endl;
                break;
                
            case TaskType::UpdatePlayerState:
                // 플레이어 상태 업데이트 (일괄 처리)
                playerIdsToUpdate.push_back(task.playerId);
                break;
                
            case TaskType::BroadcastMessage:
                // 메시지 브로드캐스트
                std::cout << "Broadcasting to player " << task.playerId << std::endl;
                break;
        }
    }
    
    // 플레이어 상태를 일괄 업데이트한다
    if (!playerIdsToUpdate.empty()) {
        std::cout << "Batch updating " << playerIdsToUpdate.size() << " players" << std::endl;
    }
}

size_t BatchTaskProcessor::GetPendingTaskCount() const {
    std::lock_guard<std::mutex> lock(m_queueMutex);
    return m_taskQueue.size();
}
```

**배칭의 장점**

작업 배칭이 성능을 개선하는 이유는 다음과 같다:

1. **컨텍스트 스위칭 감소**: 작은 작업을 각각 별도 스레드에서 처리하면 스레드 간 컨텍스트 스위칭이 많이 일어난다. 배칭으로 큰 작업 단위를 만들면 스위칭이 줄어든다.

2. **캐시 효율성**: 배치의 모든 작업이 유사한 메모리에 접근한다면, CPU 캐시 미스가 줄어든다.

3. **메모리 할당 감소**: 여러 작업을 모아서 처리하면 임시 객체 생성이 줄어든다.

다만 배칭은 지연시간(latency)을 증가시킨다. 작업이 배치가 될 때까지 기다려야 하기 때문이다. 따라서 실시간 게임에는 적절한 타임아웃을 설정해야 한다.

## 24.5 네트워크 대역폭 절약
게임 서버에서 네트워크 대역폭은 종종 병목이 된다. 많은 플레이어에게 상태를 전송해야 하기 때문이다.

### 압축과 델타 인코딩
이전 장에서 델타 압축을 배웠다. 여기서는 압축(compression)과 함께 사용하는 방법을 보여준다.

```cpp
// NetworkCompression.h
#pragma once
#include <cstdint>
#include <vector>
#include <cstring>

// 압축되지 않은 플레이어 위치 데이터
struct UncompressedPlayerData {
    uint32_t playerId;
    float posX;         // 4바이트
    float posY;         // 4바이트
    uint16_t health;    // 2바이트
    uint8_t direction;  // 1바이트
    // 총 11바이트 (패딩 제외하면 더 클 수 있음)
};

// 압축기/해제기
class DataCompressor {
public:
    // 위치를 양자화한다 (quantize)
    // float 좌표를 정수로 변환해서 데이터 크기를 줄인다
    static uint16_t QuantizeCoordinate(float coord, float min, float max) {
        // coord를 [min, max] 범위에서 [0, 65535]로 변환한다
        float normalized = (coord - min) / (max - min);
        normalized = std::max(0.0f, std::min(1.0f, normalized));
        return static_cast<uint16_t>(normalized * 65535);
    }
    
    // 양자화된 좌표를 다시 float로 변환한다
    static float DeQuantizeCoordinate(uint16_t quantized, float min, float max) {
        float normalized = quantized / 65535.0f;
        return min + normalized * (max - min);
    }
    
    // 플레이어 데이터를 압축한다
    static std::vector<uint8_t> CompressPlayerData(const UncompressedPlayerData& data) {
        std::vector<uint8_t> compressed;
        
        // 플레이어 ID (4바이트)
        AppendUInt32(compressed, data.playerId);
        
        // 위치 (양자화해서 4바이트로 압축)
        uint16_t qx = QuantizeCoordinate(data.posX, 0.0f, 1000.0f);
        uint16_t qy = QuantizeCoordinate(data.posY, 0.0f, 1000.0f);
        AppendUInt16(compressed, qx);
        AppendUInt16(compressed, qy);
        
        // 체력 (2바이트)
        AppendUInt16(compressed, data.health);
        
        // 방향 (1바이트)
        compressed.push_back(data.direction);
        
        // 총 11바이트 (압축 전과 같지만 float 대신 uint16 사용)
        return compressed;
    }
    
    // 압축된 데이터를 해제한다
    static UncompressedPlayerData DecompressPlayerData(const std::vector<uint8_t>& data) {
        UncompressedPlayerData result;
        
        size_t offset = 0;
        
        // 플레이어 ID
        result.playerId = ReadUInt32(data, offset);
        
        // 위치
        uint16_t qx = ReadUInt16(data, offset);
        uint16_t qy = ReadUInt16(data, offset);
        result.posX = DeQuantizeCoordinate(qx, 0.0f, 1000.0f);
        result.posY = DeQuantizeCoordinate(qy, 0.0f, 1000.0f);
        
        // 체력
        result.health = ReadUInt16(data, offset);
        
        // 방향
        result.direction = data[offset];
        
        return result;
    }

private:
    static void AppendUInt32(std::vector<uint8_t>& buf, uint32_t value) {
        buf.push_back((value >> 0) & 0xFF);
        buf.push_back((value >> 8) & 0xFF);
        buf.push_back((value >> 16) & 0xFF);
        buf.push_back((value >> 24) & 0xFF);
    }
    
    static void AppendUInt16(std::vector<uint8_t>& buf, uint16_t value) {
        buf.push_back((value >> 0) & 0xFF);
        buf.push_back((value >> 8) & 0xFF);
    }
    
    static uint32_t ReadUInt32(const std::vector<uint8_t>& buf, size_t& offset) {
        uint32_t value = (buf[offset] << 0) | (buf[offset + 1] << 8) |
                        (buf[offset + 2] << 16) | (buf[offset + 3] << 24);
        offset += 4;
        return value;
    }
    
    static uint16_t ReadUInt16(const std::vector<uint8_t>& buf, size_t& offset) {
        uint16_t value = (buf[offset] << 0) | (buf[offset + 1] << 8);
        offset += 2;
        return value;
    }
};
```

**압축의 효과**

위 예제에서 데이터 크기는 변하지 않지만 (구조체 패딩 때문에 원래도 크고, float를 uint16으로 바꾸는 것도 마찬가지), 실제로는 여러 효과가 있다:

1. **정확도 트레이드오프**: float 32비트를 uint16 16비트로 바꾸면 정확도가 떨어지지만, 게임에서는 ~0.01 픽셀 차이는 눈에 띄지 않는다.

2. **헤더 감소**: 여러 필드를 비트 패킹하면 더 많이 줄일 수 있다. 예를 들어, direction을 2비트로 표현하면 된다.

3. **Huffman 압축**: 자주 나타나는 값(예: 체력 100, 위치 변화 없음)에 더 짧은 코드를 할당한다.

대역폭이 중요한 경우 이러한 기법들을 함께 사용해야 한다.

## 24.6 종합 예제: 고성능 패킷 처리 파이프라인
이제 지금까지 배운 모든 최적화 기법을 통합한 실제 시스템을 만들어본다.

**설계 개요**

이 시스템은 다음과 같은 특징을 가진다:

1. **Zero-Copy 네트워킹**: 패킷을 받을 때 복사하지 않고, 원본 버퍼에서 직접 처리한다.

2. **배치 처리**: 패킷 처리를 배칭으로 일괄 처리한다.

3. **오브젝트 풀**: 자주 할당/해제되는 객체를 재사용한다.

4. **성능 모니터링**: 실시간으로 처리 성능을 추적한다.

이 예제는 약 190줄이며, 게임 서버에서 실제로 사용할 수 있는 수준의 완성도를 가지고 있다.

```cpp
// HighPerformancePacketPipeline.h
#pragma once
#include <cstdint>
#include <vector>
#include <queue>
#include <memory>
#include <mutex>
#include <thread>
#include <functional>
#include <chrono>
#include <array>

// 패킷은 메모리 풀의 버퍼를 가리킨다 (복사 없음)
struct PacketView {
    const uint8_t* data;    // 실제 데이터는 메모리 풀에 있다
    size_t size;
    uint32_t sourcePlayerId;
    uint64_t receiveTime;   // 나노초 단위

// 패킷 처리 함수 타입
using PacketProcessor = std::function<void(const PacketView&)>;

// 메모리 풀 블록
struct MemoryBlock {
    static constexpr size_t BLOCK_SIZE = 4096;
    std::array<uint8_t, BLOCK_SIZE> data;
    size_t usedSize;
    bool inUse;
};

// 메모리 풀 관리자
class MemoryPool {
public:
    explicit MemoryPool(size_t poolSize = 1000);
    
    // 메모리 블록을 할당한다 (벡터 동적 할당 없음)
    MemoryBlock* AllocateBlock();
    
    // 메모리 블록을 해제한다
    void ReleaseBlock(MemoryBlock* block);
    
    // 풀의 통계
    size_t GetAvailableBlockCount() const;
    size_t GetUsedBlockCount() const;

    std::vector<std::unique_ptr<MemoryBlock>> m_blocks;
    std::queue<MemoryBlock*> m_availableBlocks;
    std::mutex m_mutex;

// 고성능 패킷 처리 파이프라인
class HighPerformancePacketPipeline {
public:
    explicit HighPerformancePacketPipeline(size_t maxPacketsPerBatch = 100);

    void Start();
    
    // 파이프라인을 중지한다

    // 성능 통계를 출력한다

private:
    // 패킷 배치 처리
    void ProcessBatch(const std::vector<PacketView>& batch);
    
    // 워커 스레드 메인 루프
    void WorkerThread();
    
    MemoryPool m_memoryPool;
    std::vector<PacketView> m_receivedPackets;
    mutable std::mutex m_packetMutex;
    
    PacketProcessor m_processor;

class PipelineMetrics {
```

```cpp
// HighPerformancePacketPipeline.cpp
#include "HighPerformancePacketPipeline.h"
#include <iostream>
#include <iomanip>
#include <algorithm>
#include <cstring>

// MemoryPool 구현
MemoryPool::MemoryPool(size_t poolSize) {
    for (size_t i = 0; i < poolSize; ++i) {
        auto block = std::make_unique<MemoryBlock>();
        block->usedSize = 0;
        block->inUse = false;
        
        MemoryBlock* ptr = block.get();
        m_blocks.push_back(std::move(block));
        m_availableBlocks.push(ptr);
    }
}

MemoryBlock* MemoryPool::AllocateBlock() {
    std::lock_guard<std::mutex> lock(m_mutex);
    
    if (m_availableBlocks.empty()) {
        // 풀이 비었으므로 새로 할당한다 (비상 상황)
        auto block = std::make_unique<MemoryBlock>();
        block->usedSize = 0;
        block->inUse = true;
        
        MemoryBlock* ptr = block.get();
        m_blocks.push_back(std::move(block));
        return ptr;
    }
    
    MemoryBlock* block = m_availableBlocks.front();
    m_availableBlocks.pop();
    block->usedSize = 0;
    block->inUse = true;
    
    return block;
}

void MemoryPool::ReleaseBlock(MemoryBlock* block) {
    if (block == nullptr) return;
    
    std::lock_guard<std::mutex> lock(m_mutex);
    
    block->inUse = false;
    block->usedSize = 0;
    m_availableBlocks.push(block);
}

size_t MemoryPool::GetAvailableBlockCount() const {
    std::lock_guard<std::mutex> lock(m_mutex);
    return m_availableBlocks.size();
}

size_t MemoryPool::GetUsedBlockCount() const {
    std::lock_guard<std::mutex> lock(m_mutex);
    return m_blocks.size() - m_availableBlocks.size();
}

// HighPerformancePacketPipeline 구현
HighPerformancePacketPipeline::HighPerformancePacketPipeline(size_t maxPacketsPerBatch)
    : m_maxPacketsPerBatch(maxPacketsPerBatch), m_isRunning(false) {
    m_statistics.totalPacketsProcessed = 0;
    m_statistics.totalBatchesProcessed = 0;
    m_statistics.totalProcessingTimeNs = 0;
    m_statistics.peakBatchSize = 0;
}

void HighPerformancePacketPipeline::Start() {
    if (m_isRunning) return;
    
    m_isRunning = true;
    m_workerThread = std::thread(&HighPerformancePacketPipeline::WorkerThread, this);
}

void HighPerformancePacketPipeline::Stop() {
    if (!m_isRunning) return;
    
    m_isRunning = false;
    
    if (m_workerThread.joinable()) {
        m_workerThread.join();
    }
}

void HighPerformancePacketPipeline::ReceivePacket(uint32_t playerId,
                                                  const uint8_t* data,
                                                  size_t size) {
    // 메모리 풀에서 블록을 할당한다
    MemoryBlock* block = m_memoryPool.AllocateBlock();
    
    // 데이터를 복사한다 (유일한 복사 지점)
    size_t copySize = std::min(size, MemoryBlock::BLOCK_SIZE);
    std::memcpy(block->data.data(), data, copySize);
    block->usedSize = copySize;
    
    // 패킷 뷰를 만든다 (복사 없음)
    PacketView view;
    view.data = block->data.data();
    view.size = copySize;
    view.sourcePlayerId = playerId;
    view.receiveTime = std::chrono::high_resolution_clock::now().time_since_epoch().count();
    
    // 패킷을 큐에 추가한다
    {
        std::lock_guard<std::mutex> lock(m_packetMutex);
        m_receivedPackets.push_back(view);
    }
}

void HighPerformancePacketPipeline::RegisterProcessor(PacketProcessor processor) {
    m_processor = processor;
}

void HighPerformancePacketPipeline::WorkerThread() {
    while (m_isRunning) {
        std::vector<PacketView> batch;
        
        {
            std::lock_guard<std::mutex> lock(m_packetMutex);
            
            if (m_receivedPackets.empty()) {
                // 패킷이 없으면 짧은 시간 기다린다
                std::this_thread::sleep_for(std::chrono::milliseconds(1));
                continue;
            }
            
            // 배치 크기만큼 패킷을 모은다
            size_t batchCount = std::min(m_maxPacketsPerBatch, m_receivedPackets.size());
            batch.assign(m_receivedPackets.begin(),
                        m_receivedPackets.begin() + batchCount);
            m_receivedPackets.erase(m_receivedPackets.begin(),
                                   m_receivedPackets.begin() + batchCount);
        }
        
        // 배치를 처리한다 (락을 잡지 않음)
        if (!batch.empty()) {
            ProcessBatch(batch);
        }
    }
}

void HighPerformancePacketPipeline::ProcessBatch(const std::vector<PacketView>& batch) {
    auto startTime = std::chrono::high_resolution_clock::now();
    
    // 배치의 모든 패킷을 처리한다
    for (const auto& packet : batch) {
        if (m_processor) {
            m_processor(packet);
        }
    }
    
    auto endTime = std::chrono::high_resolution_clock::now();
    uint64_t processingTimeNs = 
        (endTime - startTime).count();
    
    // 통계를 업데이트한다
    m_statistics.totalPacketsProcessed += batch.size();
    m_statistics.totalBatchesProcessed++;
    m_statistics.totalProcessingTimeNs += processingTimeNs;
    m_statistics.peakBatchSize = std::max(m_statistics.peakBatchSize,
                                          static_cast<uint64_t>(batch.size()));
}

void HighPerformancePacketPipeline::PrintStatistics() const {
    std::cout << "\n=== Packet Pipeline Statistics ===" << std::endl;
    std::cout << "Total packets processed: " << m_statistics.totalPacketsProcessed << std::endl;
    std::cout << "Total batches processed: " << m_statistics.totalBatchesProcessed << std::endl;
    
    if (m_statistics.totalBatchesProcessed > 0) {
        double avgTimePerBatch = static_cast<double>(m_statistics.totalProcessingTimeNs) /
                                 m_statistics.totalBatchesProcessed;
        std::cout << "Avg processing time per batch: " << std::fixed << std::setprecision(3)
                  << avgTimePerBatch / 1000000.0 << " ms" << std::endl;
    }
    
    if (m_statistics.totalPacketsProcessed > 0) {
        double avgTimePerPacket = static_cast<double>(m_statistics.totalProcessingTimeNs) /
                                  m_statistics.totalPacketsProcessed;
        std::cout << "Avg processing time per packet: " << std::fixed << std::setprecision(3)
                  << avgTimePerPacket / 1000.0 << " μs" << std::endl;
    }
    
    std::cout << "Peak batch size: " << m_statistics.peakBatchSize << " packets" << std::endl;
    std::cout << "Memory pool - Available blocks: " << m_memoryPool.GetAvailableBlockCount()
              << ", Used blocks: " << m_memoryPool.GetUsedBlockCount() << std::endl;
}
```

**이 구현의 설계 결정**

이 시스템의 핵심 설계 결정사항은 다음과 같다:

1. **Zero-Copy 원칙**: 패킷 데이터는 네트워크 계층에서 메모리 풀로 한 번만 복사되고, 이후는 모두 포인터로 참조한다. 이는 대량의 패킷을 처리할 때 성능을 크게 향상시킨다.

2. **메모리 풀의 고정 크기**: 메모리 블록의 크기(4KB)와 개수(1000개)를 미리 정해서, 실행 중에 메모리 할당/해제가 일어나지 않도록 한다. 만약 풀이 비어도 새로 할당하므로 안전하지만, 성능이 떨어진다.

3. **배치 크기 동적 조정**: 패킷 처리를 배칭으로 하면서도, 지연을 줄이기 위해 배치 크기를 제한한다(기본값 100개). 너무 크면 지연이 증가하고, 너무 작으면 배칭의 이점이 없다.

4. **성능 모니터링**: 시스템이 어떻게 작동하는지 실시간으로 추적할 수 있도록 통계를 수집한다. 이를 통해 병목지점을 쉽게 찾을 수 있다.

**주요 부분 상세 해설**

`ReceivePacket` 함수를 보자. 이 함수는 네트워크 계층에서 호출되는데, 데이터를 메모리 풀의 블록으로 복사한다. 왜 복사를 할까? 네트워크 라이브러리(예: WinRT의 DatagramSocket)는 콜백에서만 데이터가 유효하기 때문이다. 따라서 우리가 소유할 수 있는 메모리로 복사해야 한다.

`ReceivePacket`에서 `PacketView`는 데이터의 포인터만 저장한다. 이후 모든 처리는 이 포인터를 사용하므로, 추가 복사가 일어나지 않는다. 이것이 "Zero-Copy" 개념이다.

`WorkerThread` 함수는 배치 처리를 한다. 패킷이 도착할 때마다 처리하는 대신, 일정 개수를 모아서 한 번에 처리한다. 이렇게 하면 CPU 캐시 효율성이 높아지고, 컨텍스트 스위칭이 줄어든다.

**개선 방안과 확장 가능성**

현재 구현은 기본 구조만 제공한다. 실제 서버를 만들 때는 다음과 같은 개선이 필요하다:

1. **메모리 풀 자동 확장**: 현재는 풀이 비면 새로 할당하지만, 더 나은 방법은 필요에 따라 풀의 크기를 자동으로 늘리는 것이다. 다만 synchronization이 필요하다.

2. **우선순위 처리**: 모든 패킷을 같은 우선순위로 처리하지만, 실제 게임 서버에서는 중요한 패킷(입력)을 먼저 처리하고, 덜 중요한 패킷(상태 동기화)은 나중에 처리해야 할 수 있다.

3. **적응형 배치 크기**: 시스템 부하에 따라 배치 크기를 자동으로 조정할 수 있다. 패킷이 많이 들어오면 배치를 크게 해서 처리량을 높이고, 적으면 작게 해서 지연을 줄인다.

4. **멀티 워커 스레드**: 현재는 단일 워커 스레드만 사용하지만, 멀티코어 시스템에서는 여러 워커 스레드를 사용해서 병렬 처리할 수 있다. 다만 동시성 제어가 복잡해진다.

5. **패킷 필터링**: 악의적인 패킷(너무 크거나, 많은 패킷을 한 번에 보내는 클라이언트)을 조기에 필터링해서 리소스 낭비를 줄일 수 있다.

6. **압축 지원**: 대역폭이 제약이 되는 경우, 패킷 처리 파이프라인에 압축/해제 단계를 추가할 수 있다.

이 기초 위에 이러한 개선사항들을 추가하면 매우 효율적인 게임 서버를 만들 수 있다.
