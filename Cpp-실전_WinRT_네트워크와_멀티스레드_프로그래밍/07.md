# Chapter 7. 동기화 객체
멀티스레드 프로그래밍에서 동기화는 필수다. 여러 스레드가 같은 데이터에 접근할 때, 한 스레드가 데이터를 수정하는 동안 다른 스레드가 그 데이터를 읽으면 예측 불가능한 결과가 발생한다. 이를 방지하기 위해 C++ 표준 라이브러리는 다양한 동기화 객체를 제공한다. 이 장에서는 뮤텍스, 락, 조건 변수, 원자적 연산 등을 차례대로 배우고, 게임 서버에서 스레드 안전하게 데이터를 공유하는 방법을 익힌다.
  

## 7.1 경쟁 조건과 데이터 레이스

### 문제의 본질
멀티스레드 환경에서 여러 스레드가 동시에 같은 변수에 접근할 때 발생하는 문제를 "경쟁 조건(Race Condition)"이라고 부른다. 더 정확히는, 동기화 없이 여러 스레드가 공유 데이터를 접근하는 것을 "데이터 레이스(Data Race)"라고 한다.

예를 들어 플레이어 점수를 나타내는 정수형 변수 `score`가 있고, 두 개의 스레드가 동시에 `score++`를 실행한다고 하자. 생각하기에는 점수가 2씩 증가해야 하지만, 실제로는 예측 불가능한 결과가 나올 수 있다.

왜 그럴까? `score++`는 내부적으로 다음 세 가지 작업으로 나뉜다:
1. score의 값을 읽는다 (예: 100)
2. 1을 더한다 (100 + 1 = 101)
3. 결과를 score에 저장한다

스레드 A와 B가 동시에 이 작업을 수행하면:
- 시간 1: A가 score 읽음 (100)
- 시간 2: B가 score 읽음 (100) - 아직 A의 쓰기가 반영 안 됨
- 시간 3: A가 101을 저장
- 시간 4: B가 101을 저장

최종 결과는 102가 아니라 101이 된다. B의 쓰기가 A의 쓰기를 덮어씌웠기 때문이다.

### 문제 상황 재현 예제 

```cpp
#include <iostream>
#include <thread>
#include <vector>

int sharedCounter = 0;

void IncrementCounter(int iterations)
{
    for (int i = 0; i < iterations; ++i)
    {
        // 동기화 없이 카운터 증가
        sharedCounter++;
    }
}

int main()
{
    const int ITERATIONS = 100000;
    const int NUM_THREADS = 4;
    
    std::cout << "=== 데이터 레이스 데모 ===" << std::endl;
    std::cout << "4개 스레드, 각각 100,000번 증가" << std::endl;
    std::cout << "예상 결과: " << (ITERATIONS * NUM_THREADS) << std::endl;
    
    std::vector<std::thread> threads;
    
    for (int i = 0; i < NUM_THREADS; ++i)
    {
        threads.emplace_back(IncrementCounter, ITERATIONS);
    }
    
    for (auto& t : threads)
    {
        t.join();
    }
    
    std::cout << "실제 결과: " << sharedCounter << std::endl;
    
    if (sharedCounter != ITERATIONS * NUM_THREADS)
    {
        std::cout << "오류: 데이터 레이스가 발생했습니다!" << std::endl;
        std::cout << "손실된 업데이트: " 
                 << (ITERATIONS * NUM_THREADS - sharedCounter) << std::endl;
    }
    
    return 0;
}
```

이 프로그램을 여러 번 실행하면 매번 다른 결과가 나올 것이다. 어떨 때는 400,000에 가깝지만, 어떨 때는 훨씬 작은 값이 나온다. 이는 OS의 스레드 스케줄링이 매번 다르기 때문이다.
  
   

## 7.2 std::mutex 기초

### API 설명
`std::mutex`는 "상호 배제(Mutual Exclusion)"를 구현한 동기화 객체다. 어느 시점에 최대 하나의 스레드만 뮤텍스를 보유할 수 있다. 임계 영역(critical section)에 들어가기 전에 뮤텍스를 획득하고, 나가면서 해제한다.

뮤텍스의 기본 메서드는:
- `lock()`: 뮤텍스 획득. 다른 스레드가 보유 중이면 그 스레드가 해제할 때까지 대기한다.
- `unlock()`: 뮤텍스 해제. 대기 중인 스레드 중 하나가 깨어나 뮤텍스를 획득한다.
- `try_lock()`: 뮤텍스 획득 시도. 성공하면 true, 실패하면 false를 반환한다.

뮤텍스를 직접 `lock()`/`unlock()`으로 관리하는 것은 위험하다. 만약 `lock()` 후 예외가 발생하면 `unlock()`이 실행되지 않아 데드락에 빠질 수 있다. 따라서 RAII 패턴을 사용하는 것이 좋다.

### 간단한 카운터 보호 예제 

```cpp
#include <iostream>
#include <thread>
#include <mutex>
#include <vector>

std::mutex counterMutex;
int protectedCounter = 0;

void IncrementCounterSafe(int iterations)
{
    for (int i = 0; i < iterations; ++i)
    {
        counterMutex.lock();
        protectedCounter++;
        counterMutex.unlock();
    }
}

int main()
{
    const int ITERATIONS = 100000;
    const int NUM_THREADS = 4;
    
    std::cout << "=== 뮤텍스로 보호된 카운터 ===" << std::endl;
    
    std::vector<std::thread> threads;
    
    for (int i = 0; i < NUM_THREADS; ++i)
    {
        threads.emplace_back(IncrementCounterSafe, ITERATIONS);
    }
    
    for (auto& t : threads)
    {
        t.join();
    }
    
    std::cout << "최종 결과: " << protectedCounter << std::endl;
    std::cout << "예상 결과: " << (ITERATIONS * NUM_THREADS) << std::endl;
    
    if (protectedCounter == ITERATIONS * NUM_THREADS)
    {
        std::cout << "성공: 데이터가 올바르게 보호되었습니다!" << std::endl;
    }
    
    return 0;
}
```

이 코드는 이전의 데이터 레이스 예제와 동일하지만, `std::mutex`로 보호된다. 이제 `protectedCounter++`가 실행될 때 다른 스레드는 뮤텍스를 획득하지 못하므로, 항상 올바른 결과가 나온다.

그러나 여전히 위험이 있다. 예외가 발생하면 `unlock()`이 호출되지 않을 수 있다.
  


## 7.3 std::lock_guard와 std::unique_lock

### API 설명
`std::lock_guard`와 `std::unique_lock`은 RAII 패턴을 사용해서 뮤텍스를 자동으로 관리한다.

**std::lock_guard**는 매우 간단하다. 생성자에서 뮤텍스를 `lock()`하고, 소멸자에서 `unlock()`한다. 블록을 벗어나면 자동으로 뮤텍스가 해제되므로, 예외 안전성이 보장된다.

```cpp
{
    std::lock_guard<std::mutex> lock(mutex);
    // 임계 영역
}  // lock 소멸, 뮤텍스 자동 해제
```

**std::unique_lock**은 `lock_guard`보다 유연하다. 명시적으로 `lock()`, `unlock()`을 호출할 수 있고, 뮤텍스를 다른 객체로 이동시킬 수도 있다. 또한 조건 변수와 함께 사용해야 한다.

### RAII 기반 잠금 패턴 예제 

```cpp
#include <iostream>
#include <thread>
#include <mutex>
#include <vector>

std::mutex dataLock;
int safeData = 0;

void ModifyDataWithLockGuard(int value)
{
    std::lock_guard<std::mutex> lock(dataLock);
    // lock 생성자에서 lock()이 호출됨
    
    std::cout << "스레드 " << std::this_thread::get_id() 
              << ": 데이터 수정" << std::endl;
    safeData += value;
    
    // 블록 벗어남 -> lock 소멸자에서 unlock()이 자동 호출
}

void ModifyDataWithUniqueLock(int value)
{
    std::unique_lock<std::mutex> lock(dataLock);
    
    std::cout << "스레드 " << std::this_thread::get_id() 
              << ": unique_lock으로 데이터 수정" << std::endl;
    safeData += value;
    
    // 명시적 unlock 가능
    lock.unlock();
    
    // 일부 작업은 락 없이 수행 가능
    std::cout << "락 해제 후 작업" << std::endl;
}

int main()
{
    std::cout << "=== 스마트 락 패턴 ===" << std::endl;
    
    std::vector<std::thread> threads;
    
    for (int i = 0; i < 3; ++i)
    {
        threads.emplace_back(ModifyDataWithLockGuard, 10);
        threads.emplace_back(ModifyDataWithUniqueLock, 5);
    }
    
    for (auto& t : threads)
    {
        t.join();
    }
    
    std::cout << "최종 데이터 값: " << safeData << std::endl;
    
    return 0;
}
```

`lock_guard`는 간단하고 성능이 좋지만, 락을 임의로 해제할 수 없다. `unique_lock`은 더 유연하지만 약간의 오버헤드가 있다. 게임 서버에서는 보통 간단한 상황에 `lock_guard`, 복잡한 제어가 필요한 상황에 `unique_lock`을 사용한다.
  


## 7.4 std::shared_mutex

### API 설명
`std::shared_mutex`는 읽기-쓰기 잠금(Reader-Writer Lock)을 구현한다. 여러 스레드가 동시에 데이터를 읽을 수 있지만, 한 스레드가 쓰고 있으면 다른 스레드는 읽거나 쓸 수 없다.

게임 서버에서는 플레이어 데이터를 자주 읽지만 가끔 수정한다. 이런 경우 `shared_mutex`가 유용하다. 여러 스레드가 동시에 플레이어 정보를 읽을 수 있지만, 플레이어가 레벨업하면 쓰기 락으로 배타적 접근을 보장한다.

관련 메서드는:
- `lock()` / `unlock()`: 배타적 쓰기 잠금
- `lock_shared()` / `unlock_shared()`: 공유 읽기 잠금

`std::shared_lock`은 `shared_mutex`의 `lock_shared()`/`unlock_shared()`를 RAII로 관리한다.

### 읽기-쓰기 잠금 예제 

```cpp
#include <iostream>
#include <thread>
#include <shared_mutex>
#include <vector>
#include <chrono>

std::shared_mutex playerDataLock;
int playerLevel = 1;
int playerExperience = 0;

void ReadPlayerData(int threadId)
{
    for (int i = 0; i < 5; ++i)
    {
        std::shared_lock<std::shared_mutex> lock(playerDataLock);
        
        std::cout << "[읽기 " << threadId << "] "
                 << "레벨: " << playerLevel 
                 << ", 경험치: " << playerExperience << std::endl;
        
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
    }
}

void WritePlayerData(int experience)
{
    std::unique_lock<std::shared_mutex> lock(playerDataLock);
    
    std::cout << "[쓰기] 경험치 " << experience << " 획득" << std::endl;
    playerExperience += experience;
    
    if (playerExperience >= 100)
    {
        playerLevel++;
        playerExperience = 0;
        std::cout << "[쓰기] 레벨업! 새 레벨: " << playerLevel << std::endl;
    }
    
    std::this_thread::sleep_for(std::chrono::milliseconds(500));
}

int main()
{
    std::cout << "=== 읽기-쓰기 잠금 데모 ===" << std::endl;
    
    std::vector<std::thread> threads;
    
    // 읽기 스레드 3개
    for (int i = 0; i < 3; ++i)
    {
        threads.emplace_back(ReadPlayerData, i);
    }
    
    // 쓰기 스레드 2개
    for (int i = 0; i < 2; ++i)
    {
        threads.emplace_back(WritePlayerData, 60);
    }
    
    for (auto& t : threads)
    {
        t.join();
    }
    
    std::cout << "최종 상태: 레벨 " << playerLevel 
             << ", 경험치 " << playerExperience << std::endl;
    
    return 0;
}
```

이 예제에서 여러 읽기 스레드는 동시에 플레이어 데이터를 읽을 수 있다. 하지만 쓰기 스레드가 실행되면 모든 읽기 스레드는 대기해야 하고, 다른 쓰기 스레드도 대기한다.
  


## 7.5 조건 변수 (std::condition_variable)

### API 설명
조건 변수는 스레드가 특정 조건이 만족될 때까지 대기하는 메커니즘이다. 예를 들어 "작업 큐가 비어있으면 대기하고, 새 작업이 들어오면 깨어난다"는 패턴이다.

조건 변수의 기본 메서드는:
- `wait()`: 조건이 만족될 때까지 대기. 뮤텍스를 자동으로 해제했다가 깨어나면 다시 획득한다.
- `notify_one()`: 대기 중인 스레드 중 하나를 깨운다.
- `notify_all()`: 대기 중인 모든 스레드를 깨운다.

조건 변수는 항상 뮤텍스와 함께 사용된다. `wait()`는 `unique_lock` 객체를 매개변수로 받는다.

### 대기와 알림 메커니즘 예제 

```cpp
#include <iostream>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>
#include <vector>

std::mutex queueLock;
std::condition_variable queueCV;
std::queue<int> taskQueue;
bool shouldExit = false;

void Producer()
{
    for (int i = 1; i <= 5; ++i)
    {
        {
            std::unique_lock<std::mutex> lock(queueLock);
            taskQueue.push(i);
            std::cout << "[생산] 작업 " << i << " 추가" << std::endl;
        }
        queueCV.notify_one();  // 대기 중인 소비자를 깨움
        
        std::this_thread::sleep_for(std::chrono::milliseconds(500));
    }
    
    {
        std::unique_lock<std::mutex> lock(queueLock);
        shouldExit = true;
    }
    queueCV.notify_all();  // 모든 소비자를 깨움
}

void Consumer(int id)
{
    while (true)
    {
        std::unique_lock<std::mutex> lock(queueLock);
        
        // wait는 조건이 true가 될 때까지 대기
        queueCV.wait(lock, []() { return !taskQueue.empty() || shouldExit; });
        
        if (taskQueue.empty() && shouldExit)
        {
            std::cout << "[소비자 " << id << "] 종료" << std::endl;
            break;
        }
        
        if (!taskQueue.empty())
        {
            int task = taskQueue.front();
            taskQueue.pop();
            std::cout << "[소비자 " << id << "] 작업 " << task << " 처리" << std::endl;
        }
    }
}

int main()
{
    std::cout << "=== 조건 변수 데모 (생산자-소비자) ===" << std::endl;
    
    std::thread producer(Producer);
    
    std::vector<std::thread> consumers;
    for (int i = 0; i < 2; ++i)
    {
        consumers.emplace_back(Consumer, i);
    }
    
    producer.join();
    for (auto& t : consumers)
    {
        t.join();
    }
    
    std::cout << "모든 스레드 종료" << std::endl;
    
    return 0;
}
```

이 예제는 전형적인 생산자-소비자 패턴이다. 생산자는 작업을 큐에 추가하고 소비자를 깨운다. 소비자는 큐가 비어있으면 조건 변수에서 대기했다가, 생산자가 깨우면 작업을 처리한다.
  


## 7.6 std::atomic 기초

### API 설명
`std::atomic<T>`는 락을 사용하지 않고 원자적(atomic) 연산을 보장하는 템플릿이다. 여러 스레드가 동시에 접근해도 데이터 레이스가 발생하지 않는다.

원자적 연산이란 "모두 실행되거나 모두 실행되지 않는" 연산을 의미한다. 중간 상태가 노출되지 않는다.

`std::atomic<T>`의 주요 메서드는:
- `load()`: 값을 읽는다
- `store()`: 값을 쓴다
- `exchange()`: 값을 바꾸고 이전 값을 반환한다
- `compare_exchange_strong()`: 값이 예상값이면 새 값으로 바꾼다

일반적인 정수 타입에 대해 `++`, `--`, `+=` 등의 연산자도 오버로딩되어 있다.

뮤텍스보다 빠르지만, 복잡한 동기화에는 적합하지 않다. 간단한 플래그나 카운터에 사용한다.

### 락프리 카운터 예제 

```cpp
#include <iostream>
#include <thread>
#include <atomic>
#include <vector>

std::atomic<int> atomicCounter(0);

void IncrementWithAtomic(int iterations)
{
    for (int i = 0; i < iterations; ++i)
    {
        atomicCounter++;  // 원자적 증가
    }
}

int main()
{
    const int ITERATIONS = 100000;
    const int NUM_THREADS = 4;
    
    std::cout << "=== std::atomic 카운터 ===" << std::endl;
    
    std::vector<std::thread> threads;
    
    for (int i = 0; i < NUM_THREADS; ++i)
    {
        threads.emplace_back(IncrementWithAtomic, ITERATIONS);
    }
    
    for (auto& t : threads)
    {
        t.join();
    }
    
    std::cout << "최종 결과: " << atomicCounter.load() << std::endl;
    std::cout << "예상 결과: " << (ITERATIONS * NUM_THREADS) << std::endl;
    
    if (atomicCounter == ITERATIONS * NUM_THREADS)
    {
        std::cout << "성공: 원자적 연산이 올바르게 작동!" << std::endl;
    }
    
    return 0;
}
```

`std::atomic`을 사용한 카운터는 뮤텍스보다 훨씬 빠르다. 개별 증가 연산이 원자적으로 보장되므로 데이터 레이스가 없다.
  


## 7.7 종합 예제: 스레드 안전 큐

### 코드의 목적
이 예제는 멀티스레드 환경에서 안전하게 사용할 수 있는 큐(Queue) 자료구조를 구현한다. 생산자 스레드가 아이템을 추가하고, 소비자 스레드가 가져간다. 온라인 게임 서버에서는 이런 큐가 네트워크 패킷 처리, 작업 분배 등에 자주 사용된다.

### 설계 결정 이유
첫 번째, 제네릭 템플릿으로 설계해서 모든 타입을 지원한다.

두 번째, `std::unique_lock`과 조건 변수를 사용해서 효율적인 대기-깨우기 메커니즘을 구현한다.

세 번째, 이동 의미론(move semantics)을 지원해서 복사 오버헤드를 줄인다.

네 번째, 최대 크기 제한을 설정할 수 있어서, 메모리 폭증을 방지한다.

다섯 번째, 안전한 종료를 위해 `Stop()` 메서드를 제공한다.

### 주요 부분 상세 해설
큐의 내부 구현은 `std::deque<T>`를 사용한다. `deque`는 양쪽 끝에서 효율적으로 삽입/삭제가 가능하다.

`Push()`는 아이템을 큐에 추가하고 대기 중인 소비자를 깨운다. 최대 크기에 도달하면 대기한다.

`Pop()`은 큐가 비어있으면 대기했다가, 아이템이 들어오면 가져간다.

이동 의미론은 `std::move()`를 사용해서, 아이템을 복사하지 않고 이동한다. 큰 객체의 경우 성능 차이가 크다.

### 전체 코드   

```cpp
#include <iostream>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <deque>
#include <memory>
#include <vector>
#include <chrono>

template<typename T>
class ThreadSafeQueue
{
private:
    mutable std::mutex m_lock;
    std::condition_variable m_notEmpty;
    std::condition_variable m_notFull;
    std::deque<T> m_data;
    size_t m_maxSize;
    bool m_stopped = false;
    
public:
    explicit ThreadSafeQueue(size_t maxSize = 1000)
        : m_maxSize(maxSize)
    {
    }
    
    ~ThreadSafeQueue()
    {
        Stop();
    }
    
    // 아이템을 큐에 추가 (복사)
    void Push(const T& value)
    {
        std::unique_lock<std::mutex> lock(m_lock);
        
        // 최대 크기에 도달하면 대기
        m_notFull.wait(lock, [this]() { 
            return m_data.size() < m_maxSize || m_stopped; 
        });
        
        if (m_stopped)
            return;
        
        m_data.push_back(value);
        m_notEmpty.notify_one();
    }
    
    // 아이템을 큐에 추가 (이동)
    void Push(T&& value)
    {
        std::unique_lock<std::mutex> lock(m_lock);
        
        m_notFull.wait(lock, [this]() { 
            return m_data.size() < m_maxSize || m_stopped; 
        });
        
        if (m_stopped)
            return;
        
        m_data.push_back(std::move(value));
        m_notEmpty.notify_one();
    }
    
    // 아이템을 큐에서 가져옴
    bool Pop(T& value, int timeoutMs = -1)
    {
        std::unique_lock<std::mutex> lock(m_lock);
        
        if (timeoutMs < 0)
        {
            // 무한 대기
            m_notEmpty.wait(lock, [this]() { 
                return !m_data.empty() || m_stopped; 
            });
        }
        else
        {
            // 타임아웃 있음
            if (!m_notEmpty.wait_for(lock, 
                std::chrono::milliseconds(timeoutMs),
                [this]() { return !m_data.empty() || m_stopped; }))
            {
                return false;  // 타임아웃
            }
        }
        
        if (m_data.empty())
        {
            return false;
        }
        
        value = std::move(m_data.front());
        m_data.pop_front();
        m_notFull.notify_one();
        
        return true;
    }
    
    // 큐의 크기 반환
    size_t Size() const
    {
        std::unique_lock<std::mutex> lock(m_lock);
        return m_data.size();
    }
    
    // 큐가 비어있는지 확인
    bool Empty() const
    {
        std::unique_lock<std::mutex> lock(m_lock);
        return m_data.empty();
    }
    
    // 큐 비우기
    void Clear()
    {
        std::unique_lock<std::mutex> lock(m_lock);
        m_data.clear();
    }
    
    // 큐 중지 (생산/소비 중단)
    void Stop()
    {
        {
            std::unique_lock<std::mutex> lock(m_lock);
            m_stopped = true;
        }
        m_notEmpty.notify_all();
        m_notFull.notify_all();
    }
    
    // 중지 여부 확인
    bool IsStopped() const
    {
        std::unique_lock<std::mutex> lock(m_lock);
        return m_stopped;
    }
};

// 테스트용 메시지 구조
struct Message
{
    int id;
    std::string content;
    
    Message() = default;
    Message(int id, const std::string& content)
        : id(id), content(content)
    {
    }
};

class GameServer
{
private:
    ThreadSafeQueue<Message> m_messageQueue;
    std::vector<std::thread> m_workerThreads;
    bool m_running = false;
    
public:
    GameServer(int numWorkers = 2)
        : m_messageQueue(100)
    {
        m_running = true;
        
        for (int i = 0; i < numWorkers; ++i)
        {
            m_workerThreads.emplace_back(
                [this, i]() { ProcessMessages(i); });
        }
    }
    
    ~GameServer()
    {
        Stop();
    }
    
    void EnqueueMessage(int id, const std::string& content)
    {
        Message msg(id, content);
        m_messageQueue.Push(std::move(msg));
    }
    
    void ProcessMessages(int workerId)
    {
        Message msg;
        while (m_running)
        {
            if (m_messageQueue.Pop(msg, 500))  // 500ms 타임아웃
            {
                std::cout << "[워커 " << workerId << "] "
                         << "메시지 처리 - ID: " << msg.id
                         << ", 내용: " << msg.content << std::endl;
                
                std::this_thread::sleep_for(
                    std::chrono::milliseconds(200));
            }
        }
    }
    
    void Stop()
    {
        m_running = false;
        m_messageQueue.Stop();
        
        for (auto& t : m_workerThreads)
        {
            if (t.joinable())
                t.join();
        }
    }
    
    size_t GetQueueSize() const
    {
        return m_messageQueue.Size();
    }
};

int main()
{
    std::cout << "=== 스레드 안전 큐 데모 ===" << std::endl;
    
    GameServer server(2);  // 워커 스레드 2개
    
    // 메인 스레드에서 메시지 추가
    std::cout << "\n[메인] 메시지 5개 추가" << std::endl;
    for (int i = 1; i <= 5; ++i)
    {
        server.EnqueueMessage(i, "메시지_" + std::to_string(i));
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
    }
    
    std::cout << "[메인] 대기 중..." << std::endl;
    std::this_thread::sleep_for(std::chrono::seconds(3));
    
    std::cout << "\n[메인] 메시지 3개 추가" << std::endl;
    for (int i = 6; i <= 8; ++i)
    {
        server.EnqueueMessage(i, "메시지_" + std::to_string(i));
    }
    
    std::this_thread::sleep_for(std::chrono::seconds(2));
    
    std::cout << "\n[메인] 서버 종료" << std::endl;
    
    return 0;
}
```
  

### 코드 구성 설명

**ThreadSafeQueue 클래스:**
- 템플릿 매개변수 `T`로 모든 타입을 지원한다.
- `Push()`는 복사와 이동 버전 두 가지를 제공한다.
- `Pop()`은 타임아웃을 지원해서, 무한 대기를 방지한다.
- 조건 변수 두 개: `m_notEmpty`(아이템 도착 신호), `m_notFull`(공간 여유 신호)
- `Stop()`로 안전하게 종료할 수 있다.

**GameServer 클래스:**
- 큐를 사용해서 메시지 처리 시뮬레이션한다.
- 워커 스레드들이 `ProcessMessages()`에서 큐에서 메시지를 꺼내 처리한다.
- 메인 스레드는 `EnqueueMessage()`로 메시지를 추가한다.

### 개선 방안과 확장 가능성

**현재 한계:**

첫 번째, 우선순위 큐가 아니다. FIFO(First In First Out)만 지원한다.

두 번째, 메시지 처리 콜백을 지정할 수 없다. 모든 메시지가 같은 방식으로 처리된다.

세 번째, 통계 정보가 없다. 처리된 메시지 수, 평균 대기 시간 등을 추적하지 않는다.

**개선 방안:**

첫 번째, 우선순위를 추가할 수 있다. `Priority` 필드를 메시지에 추가하고, `std::priority_queue`를 사용한다.

두 번째, 처리 콜백을 지원할 수 있다. 함수 포인터나 람다를 저장해서 메시지 처리 방식을 커스터마이징한다.

세 번째, 통계를 추가할 수 있다. 처리된 메시지 수, 큐의 최대 크기, 평균 대기 시간 등을 추적한다.

네 번째, 배치 처리를 지원할 수 있다. 여러 메시지를 한 번에 꺼내서 처리한다.

**게임 서버 적용 예시:**

이 패턴은 게임 서버의 네트워크 계층에서 다음과 같이 사용된다:

```cpp
class PlayerConnectionHandler
{
private:
    ThreadSafeQueue<NetworkPacket> m_packetQueue;
    
public:
    void OnPacketReceived(const NetworkPacket& packet)
    {
        m_packetQueue.Push(packet);
    }
    
    void ProcessPackets()
    {
        NetworkPacket pkt;
        while (m_packetQueue.Pop(pkt, 10))
        {
            HandlePacket(pkt);
        }
    }
};
```

또는 작업 분배에도 사용될 수 있다:

```cpp
ThreadSafeQueue<GameTask> taskQueue;

// 게임 로직 스레드
void GameLogicThread()
{
    GameTask task;
    while (taskQueue.Pop(task, 16))  // 16ms 타임아웃 (60FPS)
    {
        task.Execute();
    }
}

// 게임 틱에서 작업 추가
void GameTick()
{
    taskQueue.Push(UpdatePlayerPositions());
    taskQueue.Push(CheckCollisions());
    taskQueue.Push(BroadcastGameState());
}
```

---

이 장에서 배운 내용을 정리하면:

1. **경쟁 조건**: 동기화 없이 여러 스레드가 공유 데이터에 접근하면 예측 불가능한 결과가 발생한다.

2. **std::mutex**: 상호 배제를 구현해서 한 번에 하나의 스레드만 임계 영역에 접근하도록 한다.

3. **RAII 패턴**: `lock_guard`와 `unique_lock`으로 뮤텍스를 안전하게 관리한다.

4. **shared_mutex**: 읽기-쓰기 잠금으로 여러 읽기 스레드를 허용하면서 쓰기는 배타적으로 한다.

5. **조건 변수**: 특정 조건을 기다리는 스레드를 효율적으로 대기-깨우기한다.

6. **std::atomic**: 락 없이 원자적 연산을 보장해서 간단한 동기화에 성능을 제공한다.

7. **스레드 안전 큐**: 멀티스레드 환경에서 안전하게 데이터를 주고받는 실전 패턴이다.

다음 장부터는 네트워킹을 다룬다. 동기화 기술과 비동기 프로그래밍을 조합해서, 고성능 네트워크 서버를 구축하는 방법을 배운다.
