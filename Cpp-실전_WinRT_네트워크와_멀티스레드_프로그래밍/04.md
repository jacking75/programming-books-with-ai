# Chapter 4. WinRT 스레딩 기초

스레딩은 비동기 프로그래밍과 함께 고성능 네트워크 서버를 구축하는 또 다른 핵심 기술이다. 비동기 프로그래밍이 한 스레드에서 여러 작업을 번갈아가며 처리한다면, 스레딩은 여러 스레드를 활용해서 진정한 병렬 처리를 가능하게 한다. 이 장에서는 WinRT의 스레딩 모델을 깊이 있게 학습하고, 스레드 간 안전한 통신 방법을 익힌다.
  


## 4.1 스레드가 필요한 이유
단일 스레드에서만 비동기 프로그래밍을 수행하면, 결국 한 번에 한 작업씩만 실행된다. 예를 들어 4코어 프로세서에서 한 스레드만 사용한다면, 나머지 3개 코어는 놀게 된다. 이는 CPU 리소스를 낭비하는 것이다.

멀티코어 환경에서 진정한 병렬 처리를 하려면 여러 스레드가 필요하다. 각 스레드는 다른 코어에서 독립적으로 실행되므로, 여러 작업을 동시에 처리할 수 있다. 온라인 게임 서버에서 클라이언트 A의 네트워크 I/O를 처리하는 스레드 1과, 클라이언트 B의 게임 로직을 처리하는 스레드 2가 동시에 실행될 수 있다는 뜻이다.

그러나 스레드 관리는 복잡하다. 스레드들이 같은 메모리에 접근할 때 데이터 경쟁이 발생할 수 있고, 스레드 간 동기화가 필요하며, 데드락을 조심해야 한다. WinRT의 코루틴 기반 비동기 모델은 이러한 복잡성을 줄여준다. 스레드 풀을 내부적으로 관리하고, 코루틴 문법으로 간단하게 멀티스레딩 코드를 작성할 수 있게 해준다.
  


## 4.2 아파트먼트 모델 이해하기 (STA/MTA)

### API 설명
WinRT는 COM(Component Object Model)을 기반으로 한다. COM은 객체의 메서드를 어느 스레드에서 호출할 수 있는지를 규정하는 "아파트먼트 모델(Apartment Model)"이라는 개념을 제공한다. 이는 Windows 프로그래밍에서 오래된 개념이지만, WinRT 코드 작성 시 여전히 중요하다.  
  
**STA(Single-Threaded Apartment)** 는 각 스레드가 자신의 메시지 큐를 가지고 있으며, 객체는 생성된 스레드에서만 메서드가 호출될 수 있다. UI 스레드는 대부분 STA로 설정된다. STA 객체를 다른 스레드에서 호출하려면 특별한 마샬링(marshaling) 처리가 필요하다.

**MTA(Multi-Threaded Apartment)** 는 여러 스레드가 동시에 같은 객체의 메서드를 호출할 수 있다. 더 높은 동시성을 제공하지만, 스레드 안전성은 개발자가 책임져야 한다.

WinRT의 코루틴은 이 문제를 우아하게 해결한다. `co_await resume_background()`로 백그라운드 스레드로 이동하고, `co_await resume_foreground()`로 UI 스레드로 돌아올 수 있다. 컴파일러가 자동으로 필요한 마샬링을 처리한다.

### 모델별 특성과 사용 시나리오

**STA를 사용하는 경우:**
- UI 컴포넌트와 상호작용할 때. Windows의 UI 라이브러리(XAML 등)는 STA를 요구한다.
- 단순한 순차 처리 로직. 복잡한 스레드 동기화가 필요 없을 때.
- 기존 COM 컴포넌트와 통합할 때.

**MTA를 사용하는 경우:**
- 높은 동시성이 필요한 서버 로직. 예를 들어 게임 서버의 메인 로직.
- 많은 수의 비동기 작업을 동시에 처리해야 할 때.
- UI와 무관한 순수 계산 작업.

온라인 게임 서버는 보통 MTA로 초기화되어 시작한다. 그러나 UI가 있는 게임 클라이언트나 로비 서버의 관리 인터페이스는 STA가 필요할 수 있다.

다음 예제를 통해 어느 스레드에서 코드가 실행되는지 확인할 수 있다.

```cpp
#include <winrt/Windows.Foundation.h>
#include <iostream>
#include <thread>

using namespace winrt;
using namespace Windows::Foundation;

IAsyncAction DemonstrateApartments()
{
    // 현재 스레드 ID 출력
    std::cout << "초기 스레드 ID: " << std::this_thread::get_id() << std::endl;
    
    // 백그라운드 스레드로 이동
    co_await resume_background();
    std::cout << "백그라운드 스레드 ID: " << std::this_thread::get_id() << std::endl;
    
    // 다시 원래 스레드로 돌아올 수 없다 (아파트먼트 전환은 일방향)
    // 대신 다른 스레드에서 계속 실행된다
    
    co_await resume_after(std::chrono::milliseconds(100));
    std::cout << "100ms 후 스레드 ID: " << std::this_thread::get_id() << std::endl;
}

int main()
{
    init_apartment();
    
    auto action = DemonstrateApartments();
    action.get();
    
    return 0;
}
```

이 예제를 실행하면 초기 스레드와 백그라운드 스레드의 ID가 다름을 확인할 수 있다. 이는 `resume_background()`가 실제로 다른 스레드로 작업을 옮긴다는 뜻이다.  
  


## 4.3 resume_background() 사용법

### API 상세 설명
`co_await resume_background()`는 현재 코루틴의 실행을 스레드 풀의 백그라운드 스레드로 옮기는 함수다. 이 함수는 CPU 집약적인 작업이나 I/O 작업을 블로킹 없이 처리할 수 있게 해준다.

`resume_background()`의 시그니처는 단순하다. 매개변수가 없고, 코루틴이 일시 중단되었다가 백그라운드 스레드에서 재개된다. 재개될 때는 스레드 풀의 어느 스레드가 할당되는지는 OS가 결정한다.

이 함수는 UI 스레드에서 오래 걸리는 작업을 할 때 매우 유용하다. UI 스레드가 블로킹되면 사용자 인터페이스가 응답하지 않는 "프리징" 현상이 발생한다. `resume_background()`를 사용하면 UI는 반응적으로 유지하면서 백그라운드에서 작업을 계속할 수 있다.

### 백그라운드 작업 예제 

```cpp
#include <winrt/Windows.Foundation.h>
#include <iostream>
#include <chrono>
#include <thread>

using namespace winrt;
using namespace Windows::Foundation;

// CPU 집약적인 작업을 시뮬레이션
int HeavyCalculation()
{
    int result = 0;
    for (int i = 0; i < 1000000000; ++i)
        result += i % 7;
    return result;
}

IAsyncOperation<int> CalculateAsync()
{
    std::cout << "계산 시작..." << std::endl;
    std::cout << "UI 스레드 ID: " << std::this_thread::get_id() << std::endl;
    
    // 백그라운드 스레드로 이동
    co_await resume_background();
    
    std::cout << "계산 스레드 ID: " << std::this_thread::get_id() << std::endl;
    
    // 오래 걸리는 계산 수행
    int result = HeavyCalculation();
    
    std::cout << "계산 완료: " << result << std::endl;
    
    co_return result;
}

int main()
{
    init_apartment();
    
    auto calcOp = CalculateAsync();
    int result = calcOp.get();
    
    std::cout << "최종 결과: " << result << std::endl;
    
    return 0;
}
```

이 예제에서 `HeavyCalculation()`은 10억 번의 반복으로 약 몇 초가 걸리는 작업이다. 만약 UI 스레드에서 직접 호출하면 UI가 프리징될 것이다. 그러나 `resume_background()`를 사용하면 백그라운드 스레드에서 계산이 진행되므로, UI 스레드는 계속 다른 작업을 할 수 있다.
  


## 4.4 resume_foreground() 사용법

### API 설명
`resume_foreground()`는 `resume_background()`의 반대다. 백그라운드 스레드에서 실행 중인 코루틴을 UI 스레드(또는 지정된 디스패처 스레드)로 돌려보낸다.

게임 서버 개발에서는 UI가 필요 없을 수 있지만, 게임 클라이언트나 서버 관리 도구에서는 UI 업데이트가 필수다. 예를 들어 서버 상태 모니터링 애플리케이션에서 백그라운드에서 통계를 계산하고, 계산 결과를 UI에 표시해야 한다.

`resume_foreground()`를 사용하려면 `DispatcherQueue`가 필요하다. `DispatcherQueue`는 특정 스레드에서 작업을 실행하는 큐다. 보통 UI 스레드는 이미 디스패처 큐를 가지고 있으므로, `DispatcherQueue::GetForCurrentThread()`로 얻을 수 있다.

### DispatcherQueue 설명과 UI 업데이트 예제 

```cpp
#include <winrt/Windows.Foundation.h>
#include <winrt/Windows.System.h>
#include <iostream>
#include <thread>

using namespace winrt;
using namespace Windows::Foundation;
using namespace Windows::System;

// 현재 스레드의 디스패처 큐 저장 (UI 스레드에서 호출되어야 함)
DispatcherQueue g_uiDispatcher = nullptr;

IAsyncAction DataProcessingWorkflow()
{
    std::cout << "작업 시작" << std::endl;
    std::cout << "스레드 ID: " << std::this_thread::get_id() << std::endl;
    
    // 백그라운드에서 작업 수행
    co_await resume_background();
    
    std::cout << "백그라운드 처리 중... 스레드 ID: " << std::this_thread::get_id() << std::endl;
    
    // 오래 걸리는 데이터 처리
    co_await resume_after(std::chrono::milliseconds(500));
    
    int processedData = 42;
    
    // UI 스레드로 돌아가기
    if (g_uiDispatcher)
    {
        co_await resume_foreground(g_uiDispatcher);
        
        std::cout << "UI 업데이트 (스레드 ID: " << std::this_thread::get_id() << ")" << std::endl;
        std::cout << "처리된 데이터: " << processedData << std::endl;
    }
}

int main()
{
    init_apartment();
    
    // UI 스레드의 디스패처 큐 저장
    g_uiDispatcher = DispatcherQueue::GetForCurrentThread();
    
    if (!g_uiDispatcher)
    {
        std::cout << "디스패처 큐 생성 실패" << std::endl;
        return 1;
    }
    
    // 비동기 작업 시작
    auto workOp = DataProcessingWorkflow();
    workOp.get();
    
    std::cout << "모든 작업 완료" << std::endl;
    
    return 0;
}
```
  
이 예제에서 중요한 점은 `DispatcherQueue::GetForCurrentThread()`를 main 함수에서 호출한다는 것이다. main 함수는 UI 스레드에서 실행되므로, 이때 얻은 디스패처 큐는 UI 스레드를 나타낸다. 나중에 백그라운드 스레드에서 `resume_foreground(g_uiDispatcher)`를 호출하면, 실행이 UI 스레드로 복귀한다.  

  
### DispatcherQueue

* **DispatcherQueue**는 특정 스레드에 귀속된 작업 큐로, 해당 스레드에서 실행되어야 하는 콜백을 **순서대로 디스패치**해 주는 런루프 기반 인프라다
* UI 스레드처럼 “단일 스레드 전용” 상태를 갖는 컴포넌트에 안전하게 접근하기 위해 사용하며, `TryEnqueue`로 작업을 큐잉하면 **그 스레드 컨텍스트**에서 콜백이 실행된다
* UWP의 `CoreDispatcher`와 유사하지만 **윈도우/뷰에 종속되지 않고** 임의의 스레드에도 큐를 붙일 수 있다는 점이 핵심이다

#### 주요 타입

* `Windows.System.DispatcherQueue`

  * `TryEnqueue(…)`, `HasThreadAccess`, `CreateTimer()` 등을 제공한다
* `Windows.System.DispatcherQueueController`

  * 큐가 없는 스레드에 **디스패처 큐를 생성**하고 수명(`ShutdownQueueAsync`)을 관리한다
* `DispatcherQueueTimer`

  * 큐 스레드에서 동작하는 타이머를 제공한다
* WinUI 3을 쓴다면 네임스페이스가 `Microsoft.UI.Dispatching.DispatcherQueue`로 다를 수 있으나 개념과 API 패턴은 동일하다

#### 언제 쓰나

* 백그라운드 전용 작업 스레드에 **메시지 루프/디스패처**를 붙여 해당 스레드에서 안전하게 객체를 다루어야 할 때
* UI 스레드 접근이 필요한 코드를 **다른 스레드에서 큐잉**할 때
* 타이머 콜백을 스레드의 **디스패처 컨텍스트**에서 실행하고 싶을 때

#### 기본 사용 패턴
1) 현재 스레드에 큐 얻기/사용    
UI 스레드(WinUI 3 등)라면 이미 큐가 존재하므로 바로 가져와서 큐잉하면 된다.
  
```cpp
// C++/WinRT
auto dq = Windows::System::DispatcherQueue::GetForCurrentThread();
dq.TryEnqueue([] { /* 이 스레드 컨텍스트에서 실행 */ });
```
  
2) 전용 스레드와 큐 만들기  
UI가 아닌 **전용 스레드**를 하나 띄우고 그 스레드에 큐를 붙여 사용한다.
  
```cpp
// C++/WinRT (데스크톱 헬퍼 API 사용 예)
Windows::System::DispatcherQueueController controller{ nullptr };
controller = Windows::System::DispatcherQueueController::CreateOnDedicatedThread();
auto dq = controller.DispatcherQueue();
dq.TryEnqueue([] { /* 전용 스레드 작업 */ });
controller.ShutdownQueueAsync();
```
  
#### 우선순위와 실행 보장

* `DispatcherQueuePriority`로 `Low/Normal/High`를 지정할 수 있다
* 우선순위는 **큐 내부 스케줄링 힌트**일 뿐, 스레드 선점 우선순위와는 다르다
* `TryEnqueue`는 큐에 성공적으로 추가되었음을 의미하지만, 스레드가 살아 있어야 실제로 실행된다

#### 생명주기와 종료

* `DispatcherQueueController.ShutdownQueueAsync()`를 호출하면 더 이상 새 작업을 받지 않으며, 대기 중인 작업을 처리한 뒤 종료 이벤트(`ShutdownStarting/Completed`)를 발행한다
* 전용 스레드를 만들었을 경우 **컨트롤러가 수명 관리자** 역할을 하므로 확실히 종료해 리소스 누수를 막아야 한다

#### CoreDispatcher와의 비교

* `CoreDispatcher`는 보통 **UI 창/스레드**에 결합되어 있고 UWP/WinUI 2 맥락에서 사용한다
* `DispatcherQueue`는 윈도우 개념과 분리되어 **임의의 스레드**에 붙일 수 있어, Win32/데스크톱·WinUI 3·백그라운드 컴포넌트에서 범용적으로 쓰기 좋다

#### 실전 팁

* `HasThreadAccess`로 현재 스레드가 해당 리소스의 소유 스레드인지 **항상 검사**한 뒤, 아니면 `TryEnqueue`로 건네주는 패턴을 습관화한다
* 장시간 블로킹 작업을 큐 콜백 안에서 수행하지 말고, **백그라운드 작업 → 결과만 큐잉** 구조로 설계한다
* 전용 스레드+큐를 만들었다면 종료 경로에서 `ShutdownQueueAsync`와 객체 해제를 **명시적으로** 처리한다
* WinUI 3를 쓰는 경우 네임스페이스만 다를 뿐 개념은 동일하므로, 프레임워크가 제공하는 `Window.DispatcherQueue` 등을 우선 활용한다
  
  
  
## 4.5 스레드 컨텍스트 전환 실습

### API 설명
코루틴은 여러 스레드를 오가며 실행될 수 있다. 이를 "스레드 컨텍스트 전환"이라고 부른다. 이런 유연성이 있으면 각 작업에 가장 적합한 스레드에서 실행할 수 있다.

예를 들어, 네트워크 I/O는 스레드 풀에서 처리하고, 게임 로직 계산은 백그라운드에서, 최종 결과를 UI에 반영하는 것은 UI 스레드에서 처리하는 식이다.

스레드를 전환할 때 주의해야 할 점이 있다. 어떤 객체는 특정 스레드에서만 접근할 수 있다. UI 객체는 UI 스레드에서만 접근해야 하고, STA COM 객체는 생성된 스레드에서만 접근해야 한다. 반면 일반 C++ 객체나 std::atomic은 어느 스레드에서나 접근 가능하다(물론 스레드 안전성을 지켜야 함).

### 코루틴 내 스레드 이동 예제 

```cpp
#include <winrt/Windows.Foundation.h>
#include <winrt/Windows.System.h>
#include <iostream>
#include <thread>
#include <vector>

using namespace winrt;
using namespace Windows::Foundation;
using namespace Windows::System;

DispatcherQueue g_uiDispatcher = nullptr;

// 각 단계별 스레드 ID를 기록하는 구조체
struct ThreadLog
{
    std::string stageName;
    std::thread::id threadId;
    std::string description;
};

std::vector<ThreadLog> g_threadLogs;

IAsyncOperation<int> ComplexWorkflow()
{
    // 1단계: 초기화 (현재 스레드에서)
    g_threadLogs.push_back({
        "초기화",
        std::this_thread::get_id(),
        "작업 준비"
    });
    
    // 2단계: 백그라운드에서 I/O 작업
    co_await resume_background();
    g_threadLogs.push_back({
        "I/O 작업",
        std::this_thread::get_id(),
        "네트워크 또는 파일 I/O"
    });
    co_await resume_after(std::chrono::milliseconds(100));
    
    // 3단계: 더 많은 계산 (여전히 백그라운드)
    int calculatedValue = 100 * 2 + 50;
    g_threadLogs.push_back({
        "계산",
        std::this_thread::get_id(),
        "데이터 처리"
    });
    
    // 4단계: 백그라운드에서 또 다른 I/O
    co_await resume_background();
    g_threadLogs.push_back({
        "추가 I/O",
        std::this_thread::get_id(),
        "추가 네트워크 요청"
    });
    co_await resume_after(std::chrono::milliseconds(50));
    
    // 5단계: UI 스레드로 돌아가서 결과 표시
    if (g_uiDispatcher)
    {
        co_await resume_foreground(g_uiDispatcher);
        g_threadLogs.push_back({
            "UI 업데이트",
            std::this_thread::get_id(),
            "최종 결과 표시"
        });
    }
    
    co_return calculatedValue;
}

int main()
{
    init_apartment();
    
    g_uiDispatcher = DispatcherQueue::GetForCurrentThread();
    
    std::cout << "=== 스레드 컨텍스트 전환 실습 ===" << std::endl;
    std::cout << "메인 스레드 ID: " << std::this_thread::get_id() << std::endl;
    std::cout << std::endl;
    
    auto workOp = ComplexWorkflow();
    int result = workOp.get();
    
    std::cout << "=== 스레드 이동 로그 ===" << std::endl;
    for (const auto& log : g_threadLogs)
    {
        std::cout << "[" << log.stageName << "] "
                  << "스레드 ID: " << log.threadId << " - "
                  << log.description << std::endl;
    }
    
    std::cout << std::endl << "최종 결과: " << result << std::endl;
    
    return 0;
}
```

이 예제는 한 코루틴이 여러 스레드를 오가는 모습을 보여준다. 각 단계마다 스레드 ID를 기록하므로, 실행 결과를 보면 어느 단계가 어느 스레드에서 실행되었는지 명확하게 확인할 수 있다.

중요한 패턴은 다음과 같다:
- I/O 작업은 백그라운드에서 (`resume_background()`)
- 계산 작업도 백그라운드에서 실행 가능
- UI 업데이트는 반드시 UI 스레드에서 (`resume_foreground()`)

이 패턴을 따르면 UI는 반응적으로 유지되면서도 오래 걸리는 작업을 모두 처리할 수 있다.
  


## 4.6 종합 예제: 멀티스레드 이미지 프로세서

### 코드의 목적
이 예제는 대용량 이미지 파일을 여러 스레드에서 동시에 처리하고, 진행 상황을 UI에 실시간으로 업데이트하는 시스템이다. 온라인 게임 서버에서 게임 리소스를 처리할 때, 또는 게임 클라이언트에서 자산을 로딩할 때 이런 패턴이 사용된다.

### 설계 결정 이유
첫 번째, 작업을 여러 청크로 나누어 처리한다. 큰 이미지를 한 번에 처리하면 한 스레드가 오래 블로킹된다. 청크로 나누면 각 청크를 서로 다른 스레드에서 동시에 처리할 수 있다.

두 번째, 진행률 콜백을 사용한다. UI는 실시간으로 처리 상황을 알아야 한다. 콜백을 통해 각 청크 완료 시마다 UI를 업데이트할 수 있다.

세 번째, 스레드 컨텍스트를 명확하게 관리한다. 이미지 처리는 백그라운드에서, UI 업데이트는 UI 스레드에서 한다.

네 번째, 에러 처리와 부분 실패를 고려한다. 청크 중 하나가 실패해도 전체 작업이 중단되지 않도록 한다.

### 주요 부분 상세 해설
이미지 처리 로직은 실제로는 이미지 필터(밝기 조절, 회전 등)를 적용하는 것이지만, 예제를 단순하게 하기 위해 계산 작업으로 대체했다.

진행률 추적은 원자적 정수(`std::atomic<int>`)를 사용한다. 여러 스레드가 동시에 업데이트하므로 동기화가 필요한데, `std::atomic`은 락 없이 스레드 안전하게 업데이트할 수 있다.

UI 업데이트는 주기적으로만 한다. 매 청크마다 UI를 업데이트하면 오버헤드가 크므로, 진행률이 충분히 변했을 때만 UI를 갱신한다.

### 전체 코드 

```cpp
#include <winrt/Windows.Foundation.h>
#include <winrt/Windows.System.h>
#include <iostream>
#include <thread>
#include <atomic>
#include <vector>
#include <memory>
#include <cmath>

using namespace winrt;
using namespace Windows::Foundation;
using namespace Windows::System;

// 진행률 업데이트 콜백
using ProgressCallback = std::function<void(int progressPercent, int completedChunks, int totalChunks)>;

DispatcherQueue g_uiDispatcher = nullptr;

class ImageProcessor
{
private:
    static const int CHUNK_SIZE = 1024 * 1024;  // 1MB per chunk
    static const int NUM_CHUNKS = 10;
    static const int UPDATE_INTERVAL = 1;  // 1% 마다 UI 업데이트
    
    std::atomic<int> m_completedChunks = 0;
    int m_lastReportedProgress = 0;
    
public:
    // 청크 처리 시뮬레이션
    void ProcessChunk(int chunkIndex)
    {
        // 실제로는 이미지 필터 연산을 수행
        // 예제에서는 계산 작업으로 대체
        int result = 0;
        for (int i = 0; i < 100000000; ++i)
        {
            result += (i * chunkIndex) % 1000000;
        }
        
        // 청크 완료 카운트 증가
        m_completedChunks++;
    }
    
    IAsyncAction ProcessImageAsync(
        const winrt::hstring& imagePath,
        const ProgressCallback& onProgress)
    {
        std::cout << "이미지 처리 시작: " << imagePath.c_str() << std::endl;
        
        // 초기화
        m_completedChunks = 0;
        m_lastReportedProgress = 0;
        
        // 모든 청크를 병렬로 처리
        std::vector<IAsyncAction> tasks;
        
        for (int i = 0; i < NUM_CHUNKS; ++i)
        {
            // 각 청크를 비동기로 처리
            auto processChunkAsync = [this, i]() -> IAsyncAction
            {
                // 백그라운드에서 처리
                co_await resume_background();
                
                // CPU 집약적인 청크 처리
                ProcessChunk(i);
                
                // 진행률 계산 및 갱신
                int completed = m_completedChunks.load();
                int percent = (completed * 100) / NUM_CHUNKS;
                
                // 충분한 진행이 있을 때만 UI 업데이트
                if (percent - m_lastReportedProgress >= UPDATE_INTERVAL)
                {
                    // UI 스레드로 전환
                    if (g_uiDispatcher)
                    {
                        co_await resume_foreground(g_uiDispatcher);
                        
                        m_lastReportedProgress = percent;
                        if (onProgress)
                        {
                            onProgress(percent, completed, NUM_CHUNKS);
                        }
                    }
                }
            };
            
            tasks.push_back(processChunkAsync());
        }
        
        // 모든 청크 처리 대기
        for (auto& task : tasks)
        {
            try
            {
                co_await task;
            }
            catch (const std::exception& e)
            {
                std::cout << "청크 처리 오류: " << e.what() << std::endl;
            }
        }
        
        // 최종 진행률 업데이트
        if (g_uiDispatcher && m_lastReportedProgress < 100)
        {
            co_await resume_foreground(g_uiDispatcher);
            
            int completed = m_completedChunks.load();
            if (onProgress)
            {
                onProgress(100, completed, NUM_CHUNKS);
            }
        }
    }
};

void OnProgressUpdate(int percent, int completed, int total)
{
    std::cout << "\r처리 진행률: " << percent << "% ("
              << completed << "/" << total << " 청크)";
    std::cout.flush();
}

int main()
{
    init_apartment();
    
    g_uiDispatcher = DispatcherQueue::GetForCurrentThread();
    
    if (!g_uiDispatcher)
    {
        std::cout << "디스패처 큐 초기화 실패" << std::endl;
        return 1;
    }
    
    std::cout << "=== 멀티스레드 이미지 프로세서 ===" << std::endl;
    std::cout << "이미지 처리를 여러 스레드에서 동시에 수행합니다." << std::endl << std::endl;
    
    ImageProcessor processor;
    
    auto processOp = processor.ProcessImageAsync(
        winrt::hstring(L"sample_image.jpg"),
        OnProgressUpdate);
    
    try
    {
        processOp.get();
        std::cout << "\n\n이미지 처리 완료!" << std::endl;
    }
    catch (const std::exception& e)
    {
        std::cout << "\n\n처리 중 오류 발생: " << e.what() << std::endl;
        return 1;
    }
    
    return 0;
}
```

### 코드 실행 흐름 설명

1. `ProcessImageAsync`가 호출되면 10개의 청크를 각각 처리하는 비동기 작업들을 생성한다.
2. 각 청크 처리 작업은 백그라운드 스레드에서 실행되므로, 여러 청크가 동시에 처리된다.
3. 청크 완료 후 진행률을 계산하고, 1% 이상 진행되었으면 UI 스레드로 전환해서 UI를 업데이트한다.
4. 모든 청크가 처리될 때까지 대기하고, 최종 결과를 UI에 표시한다.

### 개선 방안과 확장 가능성

**현재 한계:**

첫 번째, 청크 처리가 동기적이다. `ProcessChunk()` 함수가 블로킹하므로, 진정한 의미의 병렬 처리는 아니다. 실제로는 여러 스레드가 동시에 이 함수를 호출하게 되지만, 함수 내부의 계산은 순차적이다.

두 번째, 진행률 갱신이 각 청크 이후에만 일어난다. 더 세밀한 진행률 추적이 필요하면 청크 내부에도 체크포인트를 추가해야 한다.

**개선 방안:**

첫 번째, 스레드 풀을 명시적으로 사용할 수 있다. Chapter 5에서 다룰 `ThreadPool::RunAsync()`를 사용하면 더 세밀한 제어가 가능하다.

두 번째, 취소 토큰을 추가할 수 있다. 사용자가 처리를 중단하고 싶을 때 빠르게 종료하도록 한다.

세 번째, 에러 처리를 더 강화할 수 있다. 특정 청크가 실패했을 때 재시도 로직이나 부분 롤백이 필요할 수 있다.

네 번째, 메모리 효율성을 개선할 수 있다. 청크의 결과를 메모리에 계속 보관하는 대신, 처리 후 즉시 저장하는 스트리밍 방식을 사용할 수 있다.

**게임 서버 적용 예시:**

이 패턴은 게임 서버에서 여러 곳에 적용할 수 있다:
- 게임 월드 로딩: 광대한 맵을 섹션별로 분할해서 동시에 로드
- 플레이어 데이터 처리: 로그인한 여러 플레이어의 데이터를 동시에 처리
- 게임 이벤트 실행: 많은 수의 게임 이벤트를 병렬로 처리

다음 장에서는 ThreadPool을 더 자세히 살펴본다. 스레드 풀은 스레드 생성 오버헤드를 줄이고, 리소스를 더 효율적으로 관리할 수 있게 해준다.

---

이 장에서 배운 내용을 정리하면:

1. **아파트먼트 모델**: WinRT는 COM 기반이므로, 어느 스레드에서 객체 메서드를 호출할 수 있는지 명확히 해야 한다.

2. **resume_background()**: UI 스레드를 블로킹하지 않고 오래 걸리는 작업을 백그라운드에서 처리한다.

3. **resume_foreground()**: 백그라운드에서 완료된 결과를 UI 스레드로 복귀해서 UI를 업데이트한다.

4. **스레드 컨텍스트 전환**: 코루틴은 여러 스레드를 오가며 실행되므로, 각 부분에 적합한 스레드를 선택해야 한다.

5. **병렬 처리**: 멀티코어 시스템에서 여러 스레드를 활용하면 진정한 병렬 처리가 가능해진다.

다음 장에서는 WinRT의 ThreadPool API를 학습한다. ThreadPool을 사용하면 스레드를 더 효율적으로 관리할 수 있고, 작업의 우선순위를 설정할 수 있다.
