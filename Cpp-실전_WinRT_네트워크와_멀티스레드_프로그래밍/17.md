# Chapter 17. 패킷 프로토콜 설계

## 17.1 프로토콜 설계 원칙

### 효율성, 확장성, 호환성
패킷 프로토콜은 서버와 클라이언트 사이에서 데이터를 주고받기 위한 규약이다. 게임 서버를 개발할 때 프로토콜 설계는 핵심적인 결정 중 하나다. 잘못된 프로토콜 설계는 나중에 수정하기 어렵고, 성능 문제로까지 이어질 수 있다.

프로토콜 설계의 세 가지 핵심 원칙을 살펴보자.

**효율성**은 네트워크 대역폭과 처리 속도를 의미한다. 같은 정보를 보낼 때, 적은 바이트로 표현할수록 네트워크 효율이 높다. 게임 서버에서는 매초 수천 개의 메시지가 오가므로, 작은 최적화도 큰 차이가 난다. 예를 들어, 플레이어의 위치를 float 3개(12바이트)로 보낼 수도 있고, 양자화(quantization)하여 short 3개(6바이트)로 보낼 수도 있다. 후자는 네트워크 대역폭을 절반으로 줄일 수 있다.

**확장성**은 프로토콜이 시간이 지나면서 변해야 한다는 사실을 반영한다. 게임이 업데이트되면서 새로운 메시지 타입이 추가되거나, 기존 메시지에 새로운 필드가 추가될 수 있다. 좋은 프로토콜은 이러한 변화를 버전 호환성을 유지하면서 처리할 수 있어야 한다.

**호환성**은 다양한 플랫폼과 언어에서 통신할 수 있어야 한다는 의미다. C++ 서버와 C# 클라이언트, JavaScript 클라이언트 등이 같은 프로토콜로 통신해야 한다. 따라서 바이트 순서(endianness), 데이터 정렬(alignment) 등을 명확히 정의해야 한다.

## 17.2 고정 길이 vs 가변 길이 헤더

### 각 방식의 장단점
패킷의 헤더 구조를 설계할 때, 모든 패킷이 같은 크기의 헤더를 가질 것인지(고정 길이), 아니면 각 패킷이 다른 크기의 헤더를 가질 것인지(가변 길이) 결정해야 한다.

**고정 길이 헤더**는 모든 패킷의 헤더 크기가 같다. 예를 들어, 항상 8바이트의 헤더를 가질 수 있다: [MessageType(1)] [Flags(1)] [SequenceNumber(2)] [PayloadSize(4)].

고정 길이 헤더의 장점은 다음과 같다. 첫째, 파싱이 빠르다. 위치가 고정되어 있으므로 특정 필드를 읽을 때 오프셋 계산이 필요 없다. 둘째, 구현이 간단하다. 항상 같은 크기만큼 읽으면 되므로 코드가 직관적이다. 셋째, 메모리 할당이 예측 가능하다.

단점은 다음과 같다. 첫째, 공간 낭비가 있을 수 있다. 작은 메시지도 큰 헤더를 가져야 한다. 둘째, 확장이 어렵다. 새로운 필드를 추가하려면 헤더 크기를 늘려야 하는데, 이는 호환성 문제를 일으킨다.

**가변 길이 헤더**는 메시지 타입에 따라 헤더 크기가 다르다. 간단한 메시지는 작은 헤더를, 복잡한 메시지는 큰 헤더를 가질 수 있다.

장점은 다음과 같다. 첫째, 공간 효율성이 좋다. 필요한 정보만 보낼 수 있다. 둘째, 확장이 유연하다. 새로운 필드를 추가해도 기존 메시지에 영향을 주지 않는다.

단점은 다음과 같다. 첫째, 파싱이 복잡하다. 각 필드의 위치가 고정되지 않아 순차적으로 읽어야 한다. 둘째, 구현이 어렵다. 각 메시지 타입별로 별도의 파싱 로직이 필요하다.

실전에서는 하이브리드 방식을 자주 사용한다. 고정 길이의 기본 헤더 + 메시지 타입별 추가 헤더 구조가 그것이다.

다음은 고정 길이 vs 가변 길이 헤더의 비교 예제다.

```cpp
#include <cstdint>
#include <iostream>

// 방식 1: 고정 길이 헤더
struct FixedHeaderPacket
{
    uint8_t messageType;
    uint8_t flags;
    uint16_t sequenceNumber;
    uint32_t payloadSize;
    // 항상 8바이트
};

// 방식 2: 가변 길이 헤더 (메시지 타입에 따라 다름)
struct PlayerPositionMessage
{
    uint8_t messageType;        // 1바이트
    uint32_t playerId;          // 4바이트
    float x, y, z;              // 12바이트
    // 총 17바이트
};

struct ChatMessage
{
    uint8_t messageType;        // 1바이트
    uint32_t senderId;          // 4바이트
    uint16_t textLength;        // 2바이트
    // 텍스트는 가변 길이
    // 최소 7바이트
};

// 하이브리드 방식: 기본 헤더 + 타입별 추가 헤더
struct HybridPacket
{
    // 기본 헤더 (고정 길이)
    uint8_t messageType;
    uint16_t payloadSize;
    
    // 페이로드는 messageType에 따라 달라짐
};

void ComparePacketFormats()
{
    std::wcout << L"패킷 형식 크기 비교:" << std::endl;
    std::wcout << L"================" << std::endl;
    
    std::wcout << L"고정 헤더 (FixedHeaderPacket): " 
              << sizeof(FixedHeaderPacket) << L" bytes" << std::endl;
    
    std::wcout << L"위치 메시지 (PlayerPositionMessage): " 
              << sizeof(PlayerPositionMessage) << L" bytes" << std::endl;
    
    std::wcout << L"채팅 메시지 헤더 (ChatMessage): " 
              << sizeof(ChatMessage) << L" bytes" << std::endl;
    
    std::wcout << L"하이브리드 헤더 (HybridPacket): " 
              << sizeof(HybridPacket) << L" bytes" << std::endl;
    
    std::wcout << std::endl;
    std::wcout << L"분석:" << std::endl;
    std::wcout << L"- 고정 길이: 모든 패킷이 8바이트 헤더" << std::endl;
    std::wcout << L"- 가변 길이: 메시지마다 다른 크기" << std::endl;
    std::wcout << L"- 하이브리드: 유연성과 성능의 균형" << std::endl;
}
```

## 17.3 헤더 구조 정의하기

### 구조체와 패킹
패킷 헤더는 보통 C++ 구조체로 정의한다. 하지만 컴파일러의 자동 패딩(padding) 때문에 예상과 다른 크기가 될 수 있다. 이를 제어하려면 구조체 패킹(packing)을 명시적으로 설정해야 한다.

다음은 헤더 구조 정의의 올바른 방법을 보여주는 예제다.

```cpp
#include <cstdint>
#include <iostream>

// 패킹 없음 (위험)
struct UnnpackedHeader
{
    uint8_t type;           // 1바이트
    // 패딩: 3바이트
    uint32_t size;          // 4바이트
    uint16_t sequence;      // 2바이트
    // 패딩: 2바이트
    // 예상: 7바이트, 실제: 12바이트
};

// 패킹 적용 (올바름)
#pragma pack(push, 1)
struct PackedHeader
{
    uint8_t type;           // 1바이트
    uint32_t size;          // 4바이트
    uint16_t sequence;      // 2바이트
    // 총 7바이트
};
#pragma pack(pop)

// 더 나은 방법: __declspec
__declspec(align(1))
struct AlignedHeader
{
    uint8_t type;
    uint32_t size;
    uint16_t sequence;
};

// 게임 프로토콜용 기본 헤더
#pragma pack(push, 1)
struct GamePacketHeader
{
    uint8_t version;        // 프로토콜 버전
    uint8_t messageType;    // 메시지 타입
    uint16_t messageId;     // 메시지 ID
    uint32_t payloadSize;   // 페이로드 크기
    uint32_t timestamp;     // 타임스탐프
};
#pragma pack(pop)

void DemonstrateHeaderPacking()
{
    std::wcout << L"구조체 패킹 데모:" << std::endl;
    std::wcout << L"==================" << std::endl;
    
    std::wcout << L"패킹 없음 (UnnpackedHeader): " 
              << sizeof(UnnpackedHeader) << L" bytes" << std::endl;
    std::wcout << L"예상: 7 bytes, 실제: 12 bytes" << std::endl;
    
    std::wcout << std::endl;
    
    std::wcout << L"패킹 적용 (PackedHeader): " 
              << sizeof(PackedHeader) << L" bytes" << std::endl;
    std::wcout << L"예상: 7 bytes, 실제: 7 bytes" << std::endl;
    
    std::wcout << std::endl;
    
    std::wcout << L"게임 패킷 헤더 (GamePacketHeader): " 
              << sizeof(GamePacketHeader) << L" bytes" << std::endl;
    
    // 필드 오프셋 확인
    std::wcout << std::endl;
    std::wcout << L"필드 오프셋:" << std::endl;
    
    GamePacketHeader* ptr = nullptr;
    std::wcout << L"version offset: " 
              << (uintptr_t)&ptr->version << std::endl;
    std::wcout << L"messageType offset: " 
              << (uintptr_t)&ptr->messageType << std::endl;
    std::wcout << L"messageId offset: " 
              << (uintptr_t)&ptr->messageId << std::endl;
    std::wcout << L"payloadSize offset: " 
              << (uintptr_t)&ptr->payloadSize << std::endl;
    std::wcout << L"timestamp offset: " 
              << (uintptr_t)&ptr->timestamp << std::endl;
}
```

## 17.4 메시지 타입과 ID 체계

### enum class 활용
패킷의 메시지 타입을 정의할 때, 명확하고 확장 가능한 방식을 사용해야 한다. enum class는 타입 안전성을 제공하므로 권장된다.

다음은 메시지 타입 정의의 좋은 예제다.

```cpp
#include <cstdint>
#include <string>
#include <map>

// 메시지 카테고리별 enum class
enum class MessageCategory : uint8_t
{
    System = 0,
    Authentication = 1,
    GameState = 2,
    Player = 3,
    Combat = 4,
    Chat = 5,
    Item = 6,
    Dungeon = 7
};

// 시스템 메시지
enum class SystemMessage : uint8_t
{
    Heartbeat = 0,
    ServerInfo = 1,
    Shutdown = 2,
    Configuration = 3
};

// 인증 메시지
enum class AuthMessage : uint8_t
{
    Login = 0,
    LoginResponse = 1,
    Logout = 2,
    TokenRefresh = 3
};

// 게임 상태 메시지
enum class GameStateMessage : uint8_t
{
    PlayerJoined = 0,
    PlayerLeft = 1,
    MapChange = 2,
    TimeUpdate = 3
};

// 플레이어 메시지
enum class PlayerMessage : uint8_t
{
    PositionUpdate = 0,
    StatusUpdate = 1,
    InventoryUpdate = 2,
    LevelUp = 3
};

// 메시지 타입을 계산하는 함수
constexpr uint16_t MakeMessageType(
    MessageCategory category, 
    uint8_t subType)
{
    return (static_cast<uint16_t>(category) << 8) | subType;
}

// 메시지 타입에서 카테고리와 서브타입 추출
constexpr MessageCategory GetCategory(uint16_t messageType)
{
    return static_cast<MessageCategory>(messageType >> 8);
}

constexpr uint8_t GetSubType(uint16_t messageType)
{
    return static_cast<uint8_t>(messageType & 0xFF);
}

// 메시지 타입을 문자열로 변환 (디버깅용)
std::string MessageTypeToString(uint16_t messageType)
{
    MessageCategory category = GetCategory(messageType);
    uint8_t subType = GetSubType(messageType);
    
    std::string categoryStr;
    switch (category)
    {
    case MessageCategory::System:
        categoryStr = "System";
        break;
    case MessageCategory::Authentication:
        categoryStr = "Authentication";
        break;
    case MessageCategory::GameState:
        categoryStr = "GameState";
        break;
    case MessageCategory::Player:
        categoryStr = "Player";
        break;
    default:
        categoryStr = "Unknown";
        break;
    }
    
    return categoryStr + "::" + std::to_string(static_cast<int>(subType));
}

void DemonstrateMessageTypes()
{
    std::wcout << L"메시지 타입 체계:" << std::endl;
    std::wcout << L"==================" << std::endl;
    
    // 메시지 타입 구성
    uint16_t loginMsg = MakeMessageType(
        MessageCategory::Authentication,
        static_cast<uint8_t>(AuthMessage::Login)
    );
    
    std::wcout << L"로그인 메시지 타입: " << loginMsg << std::endl;
    std::wcout << L"카테고리: " 
              << static_cast<int>(GetCategory(loginMsg)) << std::endl;
    std::wcout << L"서브타입: " 
              << static_cast<int>(GetSubType(loginMsg)) << std::endl;
    
    // 플레이어 위치 업데이트
    uint16_t posMsg = MakeMessageType(
        MessageCategory::Player,
        static_cast<uint8_t>(PlayerMessage::PositionUpdate)
    );
    
    std::wcout << std::endl;
    std::wcout << L"플레이어 위치 메시지: " << posMsg << std::endl;
    
    // 여러 메시지 타입 출력
    std::wcout << std::endl;
    std::wcout << L"메시지 타입 목록:" << std::endl;
    
    std::vector<uint16_t> messageTypes = {
        MakeMessageType(MessageCategory::System, 
                       static_cast<uint8_t>(SystemMessage::Heartbeat)),
        MakeMessageType(MessageCategory::Authentication, 
                       static_cast<uint8_t>(AuthMessage::Login)),
        MakeMessageType(MessageCategory::Player, 
                       static_cast<uint8_t>(PlayerMessage::PositionUpdate))
    };
    
    for (auto msgType : messageTypes)
    {
        std::wcout << L"- " << msgType << std::endl;
    }
}
```

## 17.5 바이트 순서 (Endianness)

### 네트워크 바이트 순서 변환
서로 다른 아키텍처의 시스템이 통신할 때 바이트 순서 문제가 발생할 수 있다. Intel x86은 리틀 엔디안을, 대형 컴퓨터는 빅 엔디안을 사용한다. 네트워크 통신에서는 표준으로 빅 엔디안(네트워크 바이트 순서)을 사용한다.

다음은 바이트 순서 변환을 다루는 예제다.

```cpp
#include <cstdint>
#include <iostream>
#include <cstring>

// 플랫폼 바이트 순서 확인
bool IsLittleEndian()
{
    uint16_t value = 0x0102;
    uint8_t* ptr = reinterpret_cast<uint8_t*>(&value);
    return ptr[0] == 0x02;  // 리틀 엔디안이면 0x02가 먼저 나옴
}

// 16비트 호스트 순서 → 네트워크 순서
uint16_t HostToNetworkShort(uint16_t value)
{
    if (IsLittleEndian())
    {
        return ((value & 0xFF) << 8) | ((value >> 8) & 0xFF);
    }
    return value;
}

// 16비트 네트워크 순서 → 호스트 순서
uint16_t NetworkToHostShort(uint16_t value)
{
    return HostToNetworkShort(value);  // 변환은 양방향 동일
}

// 32비트 호스트 순서 → 네트워크 순서
uint32_t HostToNetworkLong(uint32_t value)
{
    if (IsLittleEndian())
    {
        return ((value & 0x000000FF) << 24) |
               ((value & 0x0000FF00) << 8) |
               ((value & 0x00FF0000) >> 8) |
               ((value & 0xFF000000) >> 24);
    }
    return value;
}

// 32비트 네트워크 순서 → 호스트 순서
uint32_t NetworkToHostLong(uint32_t value)
{
    return HostToNetworkLong(value);  // 변환은 양방향 동일
}

// 부동소수점 변환
uint32_t FloatToNetworkBytes(float value)
{
    uint32_t intValue;
    std::memcpy(&intValue, &value, sizeof(float));
    return HostToNetworkLong(intValue);
}

float NetworkBytesToFloat(uint32_t networkValue)
{
    uint32_t intValue = NetworkToHostLong(networkValue);
    float value;
    std::memcpy(&value, &intValue, sizeof(float));
    return value;
}

void DemonstrateEndianness()
{
    std::wcout << L"바이트 순서 변환:" << std::endl;
    std::wcout << L"==================" << std::endl;
    
    std::wcout << L"현재 시스템: " 
              << (IsLittleEndian() ? L"Little Endian" : L"Big Endian") 
              << std::endl;
    
    // 16비트 변환
    uint16_t value16 = 0x1234;
    uint16_t networked16 = HostToNetworkShort(value16);
    
    std::wcout << std::endl;
    std::wcout << L"16비트 변환:" << std::endl;
    std::wcout << L"호스트 순서: 0x" << std::hex << value16 << std::endl;
    std::wcout << L"네트워크 순서: 0x" << std::hex << networked16 
              << std::endl;
    
    // 32비트 변환
    uint32_t value32 = 0x12345678;
    uint32_t networked32 = HostToNetworkLong(value32);
    
    std::wcout << std::endl;
    std::wcout << L"32비트 변환:" << std::endl;
    std::wcout << L"호스트 순서: 0x" << std::hex << value32 << std::endl;
    std::wcout << L"네트워크 순서: 0x" << std::hex << networked32 
              << std::endl;
    
    // 실수 변환
    float floatValue = 3.14159f;
    uint32_t networkedFloat = FloatToNetworkBytes(floatValue);
    float recoveredFloat = NetworkBytesToFloat(networkedFloat);
    
    std::wcout << std::endl;
    std::wcout << L"실수 변환:" << std::endl;
    std::wcout << L"원본: " << std::dec << floatValue << std::endl;
    std::wcout << L"복원: " << recoveredFloat << std::endl;
}
```

## 17.6 종합 예제: 범용 패킷 프로토콜

### 코드의 목적
이 예제는 TLV(Type-Length-Value) 형식을 사용한 확장 가능한 패킷 프로토콜을 구현한다. TLV는 각 데이터가 세 부분으로 구성된다: 데이터의 타입, 데이터의 길이, 실제 데이터 값. 이 구조는 매우 유연해서 새로운 필드를 쉽게 추가할 수 있다.

이 프로토콜은 실제 게임 서버에서 사용할 수 있는 완성도 높은 구현이다. 버전 호환성을 유지하면서도 기능을 확장할 수 있다.

### 설계 결정 이유
TLV 형식을 선택한 이유는 다음과 같다.

첫 번째는 호환성이다. 클라이언트와 서버의 버전이 다를 수 있다. 클라이언트는 모르는 필드를 그냥 건너뛸 수 있으므로, 하위 호환성이 자동으로 보장된다.

두 번째는 확장성이다. 새로운 필드를 추가할 때 기존 코드를 수정할 필요가 없다. 그냥 새로운 TLV 항목을 추가하면 된다.

세 번째는 명확성이다. 각 필드가 자신의 길이를 명시적으로 지정하므로, 파싱이 명확하다. 고정 길이 헤더의 단점을 보완한다.

### 코드 구현

```cpp
#include <cstdint>
#include <vector>
#include <map>
#include <string>
#include <iostream>
#include <cstring>
#include <memory>

// TLV 필드 타입
enum class TLVFieldType : uint16_t
{
    // 게임 상태 정보
    PlayerId = 1,
    PlayerName = 2,
    PlayerLevel = 3,
    PlayerPosition = 4,
    PlayerHealth = 5,
    
    // 채팅 정보
    ChatMessage = 100,
    ChatRoom = 101,
    
    // 게임 이벤트
    EventType = 200,
    EventData = 201,
    
    // 확장용
    CustomData = 1000
};

// TLV 항목 구조
#pragma pack(push, 1)
struct TLVItem
{
    uint16_t type;      // Type
    uint32_t length;    // Length
    // Value는 이 구조체 뒤에 위치
};
#pragma pack(pop)

// 범용 패킷 프로토콜
class TLVPacket
{
private:
    std::vector<uint8_t> m_buffer;
    
public:
    // 빈 패킷 생성
    TLVPacket() = default;
    
    // 버퍼에서 패킷 생성
    TLVPacket(std::vector<uint8_t> const& data)
        : m_buffer(data)
    {
    }
    
    // 필드 추가 (정수형)
    void AddField(TLVFieldType type, uint32_t value)
    {
        AddFieldRaw(type, 
                   reinterpret_cast<uint8_t*>(&value), 
                   sizeof(uint32_t));
    }
    
    // 필드 추가 (문자열)
    void AddField(TLVFieldType type, std::string const& value)
    {
        AddFieldRaw(type,
                   reinterpret_cast<uint8_t const*>(value.c_str()),
                   value.length());
    }
    
    // 필드 추가 (바이트 배열)
    void AddFieldRaw(TLVFieldType type, 
                    uint8_t const* data, 
                    uint32_t length)
    {
        // TLV 헤더 추가
        TLVItem item;
        item.type = static_cast<uint16_t>(type);
        item.length = length;
        
        // 바이트로 변환하여 버퍼에 추가
        uint8_t* itemPtr = reinterpret_cast<uint8_t*>(&item);
        m_buffer.insert(m_buffer.end(), 
                       itemPtr, 
                       itemPtr + sizeof(TLVItem));
        
        // 데이터 추가
        m_buffer.insert(m_buffer.end(), data, data + length);
    }
    
    // 필드 읽기
    bool GetField(TLVFieldType type, uint32_t& value)
    {
        uint8_t const* data = nullptr;
        uint32_t length = 0;
        
        if (!GetFieldRaw(type, data, length))
            return false;
        
        if (length != sizeof(uint32_t))
            return false;
        
        std::memcpy(&value, data, sizeof(uint32_t));
        return true;
    }
    
    // 필드 읽기 (문자열)
    bool GetField(TLVFieldType type, std::string& value)
    {
        uint8_t const* data = nullptr;
        uint32_t length = 0;
        
        if (!GetFieldRaw(type, data, length))
            return false;
        
        value.assign(reinterpret_cast<char const*>(data), length);
        return true;
    }
    
    // 필드 읽기 (원본)
    bool GetFieldRaw(TLVFieldType type, 
                    uint8_t const*& outData, 
                    uint32_t& outLength)
    {
        uint16_t targetType = static_cast<uint16_t>(type);
        size_t offset = 0;
        
        while (offset + sizeof(TLVItem) <= m_buffer.size())
        {
            TLVItem const* item = 
                reinterpret_cast<TLVItem const*>(
                    m_buffer.data() + offset
                );
            
            if (item->type == targetType)
            {
                outData = m_buffer.data() + offset + sizeof(TLVItem);
                outLength = item->length;
                return true;
            }
            
            offset += sizeof(TLVItem) + item->length;
        }
        
        return false;
    }
    
    // 패킷 직렬화
    std::vector<uint8_t> const& Serialize() const
    {
        return m_buffer;
    }
    
    // 패킷 크기
    size_t GetSize() const
    {
        return m_buffer.size();
    }
    
    // 모든 필드 순회 (디버깅용)
    void IterateFields(
        std::function<void(TLVFieldType, 
                         uint8_t const*, 
                         uint32_t)> callback) const
    {
        size_t offset = 0;
        
        while (offset + sizeof(TLVItem) <= m_buffer.size())
        {
            TLVItem const* item = 
                reinterpret_cast<TLVItem const*>(
                    m_buffer.data() + offset
                );
            
            uint8_t const* data = 
                m_buffer.data() + offset + sizeof(TLVItem);
            
            callback(static_cast<TLVFieldType>(item->type), 
                    data, 
                    item->length);
            
            offset += sizeof(TLVItem) + item->length;
        }
    }
};

// 사용 예제
void DemonstrateTLVProtocol()
{
    std::wcout << L"TLV 기반 패킷 프로토콜:" << std::endl;
    std::wcout << L"======================" << std::endl;
    
    // 패킷 생성
    TLVPacket packet;
    
    // 필드 추가
    packet.AddField(TLVFieldType::PlayerId, 12345u);
    packet.AddField(TLVFieldType::PlayerName, std::string("Hero"));
    packet.AddField(TLVFieldType::PlayerLevel, 50u);
    
    // 3D 좌표 추가
    struct Position
    {
        float x, y, z;
    } pos = {100.5f, 200.3f, 50.1f};
    
    packet.AddFieldRaw(TLVFieldType::PlayerPosition,
                      reinterpret_cast<uint8_t const*>(&pos),
                      sizeof(Position));
    
    std::wcout << L"\n패킷 생성 완료:" << std::endl;
    std::wcout << L"- 패킷 크기: " << packet.GetSize() 
              << L" bytes" << std::endl;
    
    // 필드 읽기
    std::wcout << L"\n필드 읽기:" << std::endl;
    
    uint32_t playerId = 0;
    if (packet.GetField(TLVFieldType::PlayerId, playerId))
    {
        std::wcout << L"- PlayerId: " << playerId << std::endl;
    }
    
    std::string playerName;
    if (packet.GetField(TLVFieldType::PlayerName, playerName))
    {
        std::wcout << L"- PlayerName: " << playerName.c_str() 
                  << std::endl;
    }
    
    uint32_t level = 0;
    if (packet.GetField(TLVFieldType::PlayerLevel, level))
    {
        std::wcout << L"- PlayerLevel: " << level << std::endl;
    }
    
    // 모든 필드 순회
    std::wcout << L"\n모든 필드:" << std::endl;
    
    packet.IterateFields(
        [](TLVFieldType type, uint8_t const* data, uint32_t length)
    {
        std::wcout << L"- Type: " << static_cast<int>(type)
                  << L", Length: " << length << L" bytes" << std::endl;
    });
    
    // 직렬화
    auto serialized = packet.Serialize();
    
    std::wcout << L"\n직렬화된 패킷 (16진수):" << std::endl;
    std::wcout << L"Size: " << serialized.size() << std::endl;
    
    // 처음 32바이트만 출력
    std::wcout << L"Data: ";
    for (size_t i = 0; i < std::min(size_t(32), serialized.size()); i++)
    {
        std::wcout << std::hex << std::setw(2) << std::setfill(L'0')
                  << static_cast<int>(serialized[i]) << L" ";
    }
    std::wcout << std::endl;
}
```

### 주요 부분 상세 해설

**TLVItem 구조체**는 각 필드의 헤더를 정의한다. type은 필드의 종류, length는 뒤따르는 데이터의 크기다. pragma pack(push, 1)을 사용하여 패딩이 없도록 한다.

**AddField 메서드**는 다양한 타입의 데이터를 추가한다. 오버로딩을 통해 정수, 문자열 등을 다룰 수 있다. 내부적으로는 AddFieldRaw()에 위임하여 중복 코드를 피한다.

**GetFieldRaw 메서드**는 버퍼를 순회하며 일치하는 타입의 필드를 찾는다. 이를 통해 필드의 순서와 무관하게 접근할 수 있다.

**IterateFields 메서드**는 모든 필드를 순회한다. 콜백 함수를 사용하여 유연한 처리를 가능하게 한다.

### 개선 방안과 확장 가능성
첫 번째 개선점은 필드 캐싱이다. 같은 필드를 반복해서 조회하면 비효율적이므로, 처음 파싱 시 필드를 맵으로 캐싱할 수 있다.

두 번째는 중첩된 TLV다. TLVPacket이 또 다른 TLVPacket을 포함할 수 있도록 확장할 수 있다. 이를 통해 복잡한 데이터 구조를 표현할 수 있다.

세 번째는 데이터 검증이다. 필드를 읽을 때 타입과 길이가 예상대로인지 검증하는 로직을 추가할 수 있다.

네 번째는 압축이다. 자주 사용되는 필드는 인코딩하여 크기를 줄일 수 있다.

다섯 번째는 버전 정보다. 패킷의 시작에 프로토콜 버전을 포함시켜, 여러 버전을 동시에 지원할 수 있다.

실전 게임 서버에서는 이러한 확장 기능들을 조합하여 강력하고 유연한 통신 프로토콜을 구축할 수 있다.
