# Chapter 3. 비동기 프로그래밍 입문
비동기 프로그래밍은 네트워크 애플리케이션 개발에서 가장 중요한 개념이다. 특히 온라인 게임 서버처럼 수많은 클라이언트의 요청을 동시에 처리해야 하는 환경에서는 비동기 프로그래밍 없이는 효율적인 시스템을 만들 수 없다. 이 장에서는 WinRT의 비동기 API를 차근차근 학습하고, 각 개념을 명확히 이해할 수 있는 실전 예제들을 다룬다.
  

## 3.1 왜 비동기 프로그래밍인가
동기 프로그래밍에서는 함수가 완료될 때까지 실행 흐름이 멈춘다. 예를 들어 네트워크에서 데이터를 받을 때까지 프로그램 전체가 대기하는 것이다. 만약 한 클라이언트의 요청 처리에 100밀리초가 걸린다면, 10개의 요청을 처리하려면 총 1초가 필요하다.

비동기 프로그래밍은 이 문제를 해결한다. 작업을 시작하고 완료될 때까지 기다리지 않으며, 그 사이에 다른 작업을 계속 진행한다. 같은 스레드에서 여러 작업을 번갈아가며 처리하므로, 10개의 요청을 거의 동시에 처리할 수 있다. 이는 리소스를 효율적으로 사용하면서도 높은 처리량을 달성하는 핵심 기술이다.

WinRT는 C++17의 코루틴(coroutine) 기능을 기반으로 한 우아한 비동기 프로그래밍 모델을 제공한다. `co_await` 키워드를 사용하면 비동기 작업을 마치 동기 코드처럼 자연스럽게 작성할 수 있다.
  


## 3.2 IAsyncAction 이해하기

### API 상세 설명
`IAsyncAction`은 반환값이 없는 비동기 작업을 나타내는 인터페이스다. 네트워크 데이터를 보내기, 파일에 쓰기, 또는 시간이 걸리는 계산을 수행하되 결과를 반환할 필요가 없을 때 사용한다.

`IAsyncAction`의 주요 메서드는 다음과 같다:

`GetResults()`는 비동기 작업의 완료를 기다리고 작업이 정상 완료되었는지 확인한다. 작업 중 예외가 발생했다면 여기서 예외가 발생한다.

`Completed` 속성은 이벤트 핸들러를 등록하여 작업이 완료되었을 때 알림을 받을 수 있게 한다.

`Cancel()` 메서드는 진행 중인 작업을 취소한다. 취소 가능 여부는 작업의 구현에 따라 결정된다.

`Status` 속성은 작업의 현재 상태를 반환한다. 값은 `AsyncStatus::Started`, `AsyncStatus::Completed`, `AsyncStatus::Canceled`, `AsyncStatus::Error` 중 하나다.
  
### 기본 예제 

```cpp
#include <winrt/Windows.Foundation.h>
#include <iostream>

using namespace winrt;
using namespace Windows::Foundation;

// 시간이 걸리는 작업을 시뮬레이션하는 비동기 함수
IAsyncAction SampleAsyncWork()
{
    co_await resume_after(std::chrono::milliseconds(500));
    std::cout << "비동기 작업 완료!" << std::endl;
}

int main()
{
    init_apartment();
    
    // 비동기 작업 시작
    IAsyncAction action = SampleAsyncWork();
    
    // 작업이 완료될 때까지 대기
    action.get();
    
    std::cout << "메인 함수 종료" << std::endl;
    
    return 0;
}
```

이 예제에서 `co_await resume_after()`는 지정된 시간만큼 대기하는 코루틴 suspend 지점이다. `SampleAsyncWork()` 함수는 `co_await`를 사용하므로 자동으로 비동기 함수가 된다. 메인 함수에서 `action.get()`을 호출하면 비동기 작업이 완료될 때까지 대기한다.
  


## 3.3 IAsyncOperation<T> 사용하기

### API 설명과 반환값 처리
`IAsyncOperation<T>`는 `IAsyncAction`과 유사하지만, 작업이 완료될 때 특정 타입 T의 값을 반환한다는 점이 다르다. 네트워크에서 데이터를 받으면 그 데이터를 반환하거나, 계산 결과를 반환할 때 사용한다.

주요 메서드는 `IAsyncAction`과 거의 동일하지만, `GetResults()`가 T 타입의 값을 반환한다는 것이 다르다.

템플릿 매개변수 T는 기본 타입(int, double), 문자열, WinRT 인터페이스 등 대부분의 타입이 될 수 있다. 그러나 포인터는 안 된다. 포인터가 필요하면 `Windows::Foundation::Collections::IVector<T>` 같은 컨테이너를 사용해야 한다.

### 반환값 처리 예제 

```cpp
#include <winrt/Windows.Foundation.h>
#include <iostream>
#include <string>

using namespace winrt;
using namespace Windows::Foundation;

// 정수를 계산해서 반환하는 비동기 함수
IAsyncOperation<int> CalculateAsync()
{
    co_await resume_after(std::chrono::milliseconds(300));
    
    int result = 42;
    co_return result;  // 값을 반환
}

// 문자열을 반환하는 비동기 함수
IAsyncOperation<winrt::hstring> GetNameAsync()
{
    co_await resume_after(std::chrono::milliseconds(200));
    co_return winrt::hstring(L"게임 서버");
}

int main()
{
    init_apartment();
    
    // 정수 반환값 처리
    IAsyncOperation<int> calcOp = CalculateAsync();
    int value = calcOp.get();
    std::wcout << L"계산 결과: " << value << std::endl;
    
    // 문자열 반환값 처리
    IAsyncOperation<winrt::hstring> nameOp = GetNameAsync();
    winrt::hstring name = nameOp.get();
    std::wcout << L"이름: " << name.c_str() << std::endl;
    
    return 0;
}
```

`co_return`은 C++20 코루틴 문법으로, 비동기 함수에서 값을 반환하는 방법이다. 반환값은 `get()`을 호출할 때 얻을 수 있다. `winrt::hstring`은 WinRT의 문자열 타입으로, 유니코드를 기본으로 지원한다.
  


## 3.4 co_await의 의미와 동작 원리

### 코루틴 상태 머신 설명
`co_await`는 C++20 코루틴의 핵심 키워드다. 이 키워드를 만나면 컴파일러는 함수를 상태 머신으로 변환한다. 상태 머신은 현재 실행 위치, 지역 변수의 상태 등을 저장했다가, 비동기 작업이 완료되면 그 지점부터 계속 실행한다.

`co_await` 이전의 코드는 상태 0에서 실행되고, `co_await` 지점에서 실행이 일시 중단된다. 그 동안 스레드는 다른 작업을 할 수 있다. 비동기 작업이 완료되면 상태 머신은 상태 1로 전환되고, `co_await` 이후의 코드가 실행된다.

이 방식의 장점은 동기 코드처럼 보이면서도 블로킹 없이 비동기로 동작한다는 것이다. 이를 "structured concurrency"라고 부른다.

### 코루틴 상태 머신 예제 

```cpp
#include <winrt/Windows.Foundation.h>
#include <iostream>

using namespace winrt;
using namespace Windows::Foundation;

IAsyncAction DemonstrateAwait()
{
    std::cout << "상태 0: 시작" << std::endl;
    
    // 첫 번째 co_await 지점
    co_await resume_after(std::chrono::milliseconds(100));
    std::cout << "상태 1: 100ms 후 실행" << std::endl;
    
    // 두 번째 co_await 지점
    co_await resume_after(std::chrono::milliseconds(100));
    std::cout << "상태 2: 추가 100ms 후 실행" << std::endl;
}

int main()
{
    init_apartment();
    
    auto action = DemonstrateAwait();
    action.get();  // 모든 상태가 완료될 때까지 대기
    
    std::cout << "모든 작업 완료" << std::endl;
    return 0;
}
```

이 예제에서 컴파일러는 두 개의 `co_await` 지점을 감지하고 최소 3개의 상태를 가진 상태 머신을 생성한다. 첫 번째 `co_await`에서 일시 중단되었다가 100ms 후에 재개되고, 두 번째 `co_await`에서 다시 일시 중단되었다가 100ms 후에 재개된다.
  


## 3.5 예외 처리하기

### API 설명
비동기 함수에서 발생한 예외는 어떻게 처리할까? 비동기 함수를 호출했을 때 즉시 예외가 발생하지 않는다. 대신 `get()`을 호출할 때 또는 `co_await`로 결과를 기다릴 때 예외가 발생한다.

`IAsyncAction`이나 `IAsyncOperation<T>`에서 `get()`을 호출하면, 만약 비동기 작업 중에 예외가 발생했다면 그 예외가 현재 스레드에서 재발생한다. 이를 통해 일반적인 try-catch 문법으로 예외를 처리할 수 있다.

WinRT 함수들은 종종 `hresult` 기반 에러를 반환한다. 이들은 자동으로 C++ 예외로 변환되므로, C++의 표준 예외 처리 메커니즘을 사용할 수 있다.

### try-catch와 비동기 예외 전파 예제 

```cpp
#include <winrt/Windows.Foundation.h>
#include <iostream>

using namespace winrt;
using namespace Windows::Foundation;

enum class ErrorType
{
    None,
    Validation,
    Timeout,
    Network
};

IAsyncOperation<int> RiskyOperationAsync(ErrorType errorType)
{
    co_await resume_after(std::chrono::milliseconds(100));
    
    if (errorType == ErrorType::Validation)
        throw std::invalid_argument("입력 값이 유효하지 않습니다");
    
    if (errorType == ErrorType::Network)
        throw std::runtime_error("네트워크 연결 실패");
    
    co_return 100;
}

int main()
{
    init_apartment();
    
    // 정상 경우
    try
    {
        auto op = RiskyOperationAsync(ErrorType::None);
        int result = op.get();
        std::cout << "결과: " << result << std::endl;
    }
    catch (const std::exception& e)
    {
        std::cout << "오류: " << e.what() << std::endl;
    }
    
    // 검증 오류 경우
    try
    {
        auto op = RiskyOperationAsync(ErrorType::Validation);
        int result = op.get();
    }
    catch (const std::invalid_argument& e)
    {
        std::cout << "검증 오류: " << e.what() << std::endl;
    }
    
    // 네트워크 오류 경우
    try
    {
        auto op = RiskyOperationAsync(ErrorType::Network);
        int result = op.get();
    }
    catch (const std::runtime_error& e)
    {
        std::cout << "실행 오류: " << e.what() << std::endl;
    }
    
    return 0;
}
```

비동기 함수 내에서 예외를 던지면, `get()`을 호출할 때 그 예외가 발생한다. 각 오류 타입별로 서로 다른 예외를 처리할 수 있으며, 기본 예외 클래스인 `std::exception`으로 모든 예외를 잡을 수도 있다. 이는 네트워크 작업에서 매우 중요하다. 연결 실패, 타임아웃, 데이터 오류 등을 각각 다르게 처리할 수 있기 때문이다. 
  
  

## 3.6 작업 취소와 타임아웃

### API 설명
비동기 작업을 중간에 취소해야 할 경우가 있다. 사용자가 다운로드를 중단하거나, 응답이 너무 오래 걸리거나, 시스템 종료 시에 말이다. WinRT는 `CancellationToken`을 통해 이를 지원한다.

`CancellationToken`은 작업 취소를 신호하는 토큰이다. `CancellationTokenSource`에서 생성하며, `IsCancellationRequested` 속성으로 취소 여부를 확인할 수 있다. 비동기 함수는 정기적으로 이 속성을 확인하고, 취소가 요청되었으면 빠르게 종료해야 한다.

타임아웃은 `co_await resume_after()`를 사용해서 직접 구현할 수 있다. 타임아웃 시간 동안 작업이 완료되지 않으면 특정 동작을 수행하도록 코드를 작성하면 된다.

### CancellationToken 사용법 예제 

```cpp
#include <winrt/Windows.Foundation.h>
#include <iostream>
#include <chrono>

using namespace winrt;
using namespace Windows::Foundation;

IAsyncOperation<int> LongRunningTaskAsync(CancellationToken ct)
{
    // 긴 작업을 100ms의 청크로 나누어 처리
    for (int i = 0; i < 10; ++i)
    {
        // 취소 요청 확인
        if (ct.get().IsCancellationRequested())
        {
            std::cout << "작업 취소됨 (진행률: " << (i * 10) << "%)" << std::endl;
            throw OperationCanceledException();
        }
        
        // 100ms 작업 시뮬레이션
        co_await resume_after(std::chrono::milliseconds(100));
        std::cout << "진행률: " << ((i + 1) * 10) << "%" << std::endl;
    }
    
    co_return 42;
}

int main()
{
    init_apartment();
    
    // 취소 토큰 생성
    CancellationTokenSource cts;
    CancellationToken ct = cts.Token();
    
    // 비동기 작업 시작
    auto task = LongRunningTaskAsync(ct);
    
    // 300ms 후에 취소 요청
    std::cout << "작업 시작, 300ms 후 취소 예정" << std::endl;
    
    auto cancelTask = [](CancellationTokenSource cts) -> IAsyncAction
    {
        co_await resume_after(std::chrono::milliseconds(300));
        std::cout << "취소 신호 전송" << std::endl;
        cts.Cancel();
    };
    
    try
    {
        auto cancelOp = cancelTask(cts);
        // 두 작업이 병렬로 진행됨
        int result = task.get();
        std::cout << "작업 완료, 결과: " << result << std::endl;
    }
    catch (const hresult_canceled&)
    {
        std::cout << "취소됨" << std::endl;
    }
    
    return 0;
}
```

이 예제에서는 10개의 청크로 나누어진 작업을 수행한다. 각 청크마다 `IsCancellationRequested()`를 확인해서 취소 요청이 있으면 즉시 `OperationCanceledException`을 던진다. 취소 요청은 별도로 시뮬레이션되어 300ms 후에 전송된다. 실전 코드에서는 사용자 입력, 타임아웃 등 다양한 이유로 취소를 요청할 수 있다.
  
   

## 3.7 종합 예제: 비동기 파일 다운로더

### 코드의 목적
이 예제는 인터넷에서 파일을 다운로드하는 완전한 비동기 시스템이다. 진행률 보고, 취소 기능, 재시도 로직을 모두 포함하고 있다. 온라인 게임 서버에서는 게임 패치 다운로드, 리소스 로딩 등에 이런 패턴이 필수적이다.

### 설계 결정 이유
첫 번째, 진행률을 보고하기 위해 콜백 함수를 매개변수로 받는다. 이렇게 하면 호출자가 진행 상황을 UI에 반영하거나 로깅할 수 있다.

두 번째, 청크 단위로 다운로드한다. 한 번에 전체 파일을 다운로드하려고 하면 메모리 부담이 크고 네트워크 오류 시 처음부터 다시 해야 한다. 청크 단위로 나누면 중간에 실패해도 재시도가 쉽다.

세 번째, 취소 토큰을 사용한다. 시스템이 종료되거나 사용자가 다운로드를 중단할 수 있어야 한다.

네 번째, 재시도 로직을 내장한다. 네트워크 오류는 일시적일 수 있으므로, 일정 횟수까지 자동으로 재시도하는 것이 좋다.

### 주요 부분 상세 해설
다운로드 함수는 파일 크기를 먼저 파악한다. HTTP HEAD 요청 또는 GET 응답 헤더에서 `Content-Length`를 읽으면 된다. 파일 크기를 알면 진행률을 정확하게 계산할 수 있다.

청크 다운로드 루프에서는 각 청크마다 취소 여부를 확인한다. 또한 네트워크 오류가 발생하면 최대 3회까지 재시도한다. 지수 백오프를 사용해서 재시도 간격을 점점 늘린다. 첫 번째 재시도는 100ms 후, 두 번째는 200ms 후, 세 번째는 400ms 후에 시도한다.

진행률 콜백은 다운로드한 바이트 수와 전체 바이트 수를 전달한다. 호출자는 이를 기반으로 진행률 표시줄을 업데이트할 수 있다.

### 전체 코드 

```cpp
// C++/WinRT의 핵심 헤더 및 비동기 작업을 위한 헤더
#include <winrt/Windows.Foundation.h>
// 파일 및 폴더 작업을 위한 헤더 (StorageFolder, StorageFile)
#include <winrt/Windows.Storage.h>
// HTTP 클라이언트 (HttpClient) - (이 예제에서는 m_client를 선언만 하고 사용하지 않음)
#include <winrt/Windows.Web.Http.h>
// 파일 스트림 및 버퍼 작업을 위한 헤더 (IRandomAccessStream, Buffer)
#include <winrt/Windows.Storage.Streams.h>
#include <iostream>     // 콘솔 입출력
#include <functional>   // std::function (콜백용)
#include <string>       // std::string

using namespace winrt;
using namespace Windows::Foundation;
using namespace Windows::Storage;
using namespace Windows::Web::Http;
using namespace Windows::Storage.Streams;

// 진행률 콜백을 위한 C++ 표준 함수 템플릿 정의
// (다운로드된 바이트, 총 바이트)를 인자로 받음
using ProgressCallback = std::function<void(uint64_t downloaded, uint64_t total)>;

/**
 * @class AsyncDownloader
 * @brief 청크 다운로드, 재시도, 취소, 진행률 보고 기능을 갖춘 비동기 다운로더 클래스 (시뮬레이션)
 */
class AsyncDownloader
{
private:
    // 한 번에 다운로드할 청크(조각) 크기 (64KB)
    static const uint32_t CHUNK_SIZE = 65536;
    // 특정 청크 다운로드 실패 시 최대 재시도 횟수
    static const uint32_t MAX_RETRIES = 3;
    
    // HTTP 요청을 보내기 위한 클라이언트 객체
    // (참고: 이 예제 코드에서는 m_client가 실제로 사용되지 않았습니다.)
    HttpClient m_client;
    
public:
    AsyncDownloader()
    {
        // HTTP 클라이언트 초기화 (필요시 필터 등을 설정)
    }
    
    /**
     * @brief 파일을 비동기적으로 다운로드하는 메인 코루틴 함수
     * @param url 다운로드할 URL (현재 예제에서는 사용 안 함)
     * @param savePath 저장할 파일 이름 (예: "my_file.bin")
     * @param onProgress 진행률 보고 콜백 함수
     * @param ct 외부에서 작업을 취소하기 위한 취소 토큰
     * @return IAsyncOperation<bool> - 작업 완료 시 성공(true) 또는 실패(false) 반환
     */
    IAsyncOperation<bool> DownloadFileAsync(
        const winrt::hstring& url,       // (사용 안 됨)
        const winrt::hstring& savePath,  // 저장될 파일명
        const ProgressCallback& onProgress, // 진행률 콜백
        CancellationToken ct)            // 취소 토큰
    {
        try
        {
            // --- 다운로드 시뮬레이션 준비 ---
            // (실제 코드: 여기서 HTTP HEAD 요청을 보내 파일 크기를 알아내야 함)
            uint64_t totalSize = 1024 * 1024; // 예제를 위해 1MB로 고정
            uint64_t downloadedSize = 0;      // 현재까지 다운로드된 크기
            
            // --- 저장할 파일 준비 (비동기) ---
            // (참고: "C:\Downloads"는 권한 문제로 실패할 수 있습니다. 
            //  보통은 KnownFolders::DownloadsFolder 등을 사용해야 합니다.)
            StorageFolder folder = co_await StorageFolder::GetFolderFromPathAsync(
                hstring(L"C:\\Downloads"));
            
            // 지정된 파일명으로 파일을 생성 (이미 있으면 덮어쓰기)
            StorageFile file = co_await folder.CreateFileAsync(
                savePath, CreationCollisionOption::ReplaceExisting);
            
            // 생성된 파일에 데이터를 쓸 수 있도록 스트림 열기
            IRandomAccessStream stream = 
                co_await file.OpenAsync(FileAccessMode::ReadWrite);
            
            // --- 청크 단위 다운로드 루프 ---
            // totalSize에 도달할 때까지 CHUNK_SIZE만큼 offset을 증가시키며 반복
            for (uint64_t offset = 0; offset < totalSize; offset += CHUNK_SIZE)
            {
                // [1] 작업 취소 확인 (CancellationToken)
                // 루프가 돌 때마다 외부에서 취소 요청이 있었는지 확인
                if (ct.get().IsCancellationRequested())
                {
                    std::cout << "\n[AsyncDownloader] 취소 감지됨. 스트림 닫는 중..." << std::endl;
                    stream.Close();
                    co_return false; // 코루틴 종료 (실패)
                }
                
                // [2] 이번 청크 크기 계산
                // (마지막 청크는 CHUNK_SIZE보다 작을 수 있음)
                uint32_t chunkSize = static_cast<uint32_t>(
                    std::min((uint64_t)CHUNK_SIZE, totalSize - offset));
                
                // [3] 청크 다운로드 및 재시도 루프
                bool success = false; // 현재 청크 성공 여부
                for (uint32_t retry = 0; retry < MAX_RETRIES && !success; ++retry)
                {
                    try
                    {
                        // --- 실제 다운로드 시뮬레이션 ---
                        // (실제 코드: m_client.GetAsync()로 Range 헤더와 함께 요청)
                        
                        // 10ms간 비동기적으로 대기하여 네트워크 지연 시뮬레이션
                        co_await resume_after(std::chrono::milliseconds(10));
                        
                        // (실제 코드: 응답 스트림에서 데이터를 읽어 버퍼에 채움)
                        // 여기서는 가짜 데이터가 채워진 버퍼를 생성
                        Buffer buffer(chunkSize);
                        
                        // 파일 스트림에 버퍼의 데이터를 비동기적으로 쓰기
                        co_await stream.WriteAsync(buffer);
                        
                        // 다운로드된 크기 업데이트 및 성공 플래그 설정
                        downloadedSize += chunkSize;
                        success = true;
                        
                        // [4] 진행률 보고
                        // 콜백 함수가 등록되어 있으면 호출
                        if (onProgress)
                        {
                            onProgress(downloadedSize, totalSize);
                        }
                    }
                    catch (const std::exception& e)
                    {
                        std::cout << "\n[AsyncDownloader] 청크 오류: " << e.what() << std::endl;
                        
                        // [5] 재시도 및 지수 백오프(Exponential Backoff)
                        if (retry < MAX_RETRIES - 1) // 아직 재시도 기회가 남았다면
                        {
                            // 100ms, 200ms, 400ms... (100 * 2^retry) 만큼 대기
                            uint32_t delayMs = 100 * (1 << retry);
                            std::cout << "[AsyncDownloader] " << delayMs << "ms 후 재시도..." << std::endl;
                            co_await resume_after(std::chrono::milliseconds(delayMs));
                        }
                        else // 최대 재시도 횟수 도달
                        {
                            std::cout << "[AsyncDownloader] 최대 재시도 실패" << std::endl;
                            stream.Close();
                            co_return false; // 코루틴 종료 (실패)
                        }
                    }
                } // end of retry loop
            } // end of chunk loop
            
            // 모든 루프가 성공적으로 끝나면
            stream.Close();
            co_return true; // 코루틴 종료 (성공)
        }
        catch (const std::exception& e) // (예: 파일 생성/폴더 접근 실패 등)
        {
            std::cout << "다운로드 중 치명적 오류: " << e.what() << std::endl;
            co_return false; // 코루틴 종료 (실패)
        }
    }
};

/**
 * @brief 진행률 콜백 함수 (ProgressCallback 타입과 일치)
 * @details \r (캐리지 리턴)을 사용해 현재 줄에 덮어쓰며 진행률을 표시
 */
void PrintProgress(uint64_t downloaded, uint64_t total)
{
    if (total == 0) return;
    
    int percent = static_cast<int>((downloaded * 100) / total);
    // \r: 줄 바꿈 없이 현재 줄의 맨 앞으로 커서 이동
    std::cout << "\r진행률: " << percent << "% (" 
              << (downloaded / 1024) << "KB / " 
              << (total / 1024) << "KB)";
    std::cout.flush(); // 출력 버퍼를 즉시 비움
}

int main()
{
    // WinRT COM 아파트먼트 초기화 (필수)
    init_apartment();
    
    AsyncDownloader downloader;
    
    // 취소 토큰 소스 생성 (작업 취소 신호를 보낼 수 있음)
    CancellationTokenSource cts;
    
    std::cout << "파일 다운로드 시작... (1MB 시뮬레이션)" << std::endl;
    
    // 비동기 다운로드 작업 시작 (이 시점에 코루틴이 실행됨)
    // .get()을 호출하기 전까지는 main 스레드를 차단하지 않음
    auto downloadTask = downloader.DownloadFileAsync(
        hstring(L"http://example.com/largefile.bin"), // (사용 안 됨)
        hstring(L"downloaded_file.bin"),              // 저장될 파일명
        PrintProgress,                               // 진행률 콜백 함수 전달
        cts.Token());                                // 취소 토큰 전달
    
    // (테스트용) 5초 후에 다운로드를 자동으로 취소하는 별도의 코루틴
    auto autoCancel = [](CancellationTokenSource cts) -> IAsyncAction
    {
        // 5초간 비동기 대기
        co_await resume_after(std::chrono::seconds(5));
        std::cout << "\n[main] 5초 경과. 취소 신호 전송!" << std::endl;
        // 취소 소스(cts)에 취소 명령
        cts.Cancel();
    };
    
    try
    {
        // 취소 타이머 작업 시작
        auto cancelTask = autoCancel(cts);
        
        // --- 메인 스레드 대기 ---
        // downloadTask.get()을 호출하면 해당 작업(코루틴)이
        // 완료(성공, 실패, 취소)될 때까지 main 스레드가 여기서 대기(Block)함
        bool result = downloadTask.get();
        
        // .get()이 완료된 후 결과 처리
        if (result)
            std::cout << "\n다운로드 완료!" << std::endl;
        else
            std::cout << "\n다운로드 실패 (재시도 실패 또는 내부 오류)" << std::endl;
    }
    catch (const hresult_canceled&) // CancellationToken에 의해 취소된 경우
    {
        // downloadTask.get()이 이 예외를 발생시킴
        std::cout << "\n[main] 다운로드가 사용자에 의해 취소되었습니다" << std::endl;
    }
    catch (const std::exception& e) // 기타 예외
    {
        std::cout << "\n[main] 오류: " << e.what() << std::endl;
    }
    
    return 0;
}
```

### 개선 방안과 확장 가능성
현재 예제는 교육 목적으로 단순화되어 있다. 실전 환경에서는 다음과 같은 개선이 필요하다.

첫 번째, 실제 HTTP 클라이언트 구현이 필요하다. 현재는 청크 다운로드를 시뮬레이션하지만, 실제로는 WinRT의 `HttpClient`를 사용해서 데이터를 받아야 한다. `HttpClient.GetAsync()` 또는 `HttpClient.GetInputStreamAsync()`를 사용한다.

두 번째, 파일 시스템 접근을 더 강력하게 해야 한다. 현재 예제는 `StorageFolder`를 직접 생성하지만, 실제로는 사용자가 선택한 폴더에 저장해야 한다. UWP 환경이라면 `FileSavePicker`를 사용하는 것이 좋다.

세 번째, 부분 다운로드 재개를 구현할 수 있다. 만약 다운로드 중간에 연결이 끊어졌다면, HTTP Range 헤더를 사용해서 남은 부분만 다운로드하도록 개선할 수 있다. 이는 대용량 파일의 경우 사용자 경험을 크게 향상시킨다.

네 번째, 암호화와 무결성 검증을 추가할 수 있다. 파일의 SHA256 해시를 비교해서 다운로드 중 손상이 없었는지 확인한다. 게임 서버에서는 게임 패치 파일이 변조되지 않았는지 확인해야 한다.

다섯 번째, 대역폭 제한 기능을 추가할 수 있다. 다운로드 속도를 의도적으로 제한해서 네트워크 전체 대역폭을 독점하지 않도록 한다. 이는 특히 백그라운드 다운로드에서 중요하다.

이러한 개선 사항들은 Chapter 14의 "비동기 패턴 심화"에서 다루게 될 것이다. 지금은 기본적인 구조와 패턴을 이해하는 것이 중요하다.
