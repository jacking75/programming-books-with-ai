# Chapter 12. 소켓 옵션과 제어

## 12.1 StreamSocketControl API 설명

### 사용 가능한 모든 옵션 개요
소켓을 생성했다고 해서 바로 최적의 성능을 낼 수 있는 것은 아니다. 네트워크의 특성, 애플리케이션의 요구사항, 그리고 사용 환경에 따라 소켓의 동작을 세밀하게 조정해야 한다. WinRT의 StreamSocketControl 클래스는 이러한 소켓 옵션들을 제어하기 위한 인터페이스를 제공한다.

StreamSocket을 생성한 후 Control 속성에 접근하면 StreamSocketControl 객체를 얻을 수 있다. 이 객체를 통해 소켓의 동작 방식을 변경할 수 있다. 중요한 점은 소켓 연결 이전에 이러한 옵션들을 설정해야 한다는 것이다. 연결 후에 옵션을 변경하면 일부 옵션은 적용되지 않을 수 있다.

StreamSocketControl에서 제공하는 주요 옵션들을 살펴보자. 첫 번째는 KeepAliveIntervalInMilliseconds로, 연결을 유지하기 위해 킵얼라이브 패킷을 보내는 간격을 설정한다. 두 번째는 NoDelay로, Nagle 알고리즘을 비활성화하여 작은 패킷도 지연 없이 전송한다. 세 번째는 QualityOfService로, 네트워크 우선순위를 설정한다. 네 번째는 OutboundBufferSizeInBytes로, 송신 버퍼의 크기를 조정한다.

추가로 ClientCertificate는 TLS/SSL 연결에 사용할 클라이언트 인증서를 설정한다. InboundBufferSizeInBytes는 수신 버퍼의 크기를 조정한다. TransferOwnershipOfSocket 메서드는 소켓의 소유권을 다른 객체에 양도할 때 사용한다.

이 장에서는 각 옵션의 의미를 이해하고, 실제로 언제 어떤 옵션을 설정해야 하는지 배우게 된다. 게임 서버 개발자로서 너는 이러한 옵션들을 적절히 조합하여 최상의 성능을 끌어내야 한다.

## 12.2 KeepAlive 설정하기

### 연결 유지 메커니즘
네트워크 연결이 오랫동안 유휴 상태에 있으면 중간의 라우터나 방화벽이 연결을 종료할 수 있다. 이를 방지하기 위해 주기적으로 작은 패킷(킵얼라이브 패킷)을 보내서 연결을 살려두는 메커니즘이 킵얼라이브다.

KeepAlive를 설정하면, 지정한 간격마다 자동으로 작은 패킷이 전송되어 연결을 유지한다. 이는 특히 게임 서버처럼 장시간 연결을 유지해야 하는 애플리케이션에서 중요하다.

다음은 KeepAlive를 설정하는 예제다.

```cpp
#include <winrt/Windows.Networking.Sockets.h>
#include <iostream>

using namespace winrt;
using namespace Windows::Networking::Sockets;

IAsyncAction SetupKeepAliveAsync()
{
    StreamSocket socket;
    
    try
    {
        // KeepAlive 활성화
        auto control = socket.Control();
        control.KeepAlive(true);
        
        // 킵얼라이브 간격을 30초로 설정
        control.KeepAliveIntervalInMilliseconds(30000);
        
        std::wcout << L"KeepAlive 설정 완료" << std::endl;
        std::wcout << L"- KeepAlive: " 
                  << (control.KeepAlive() ? L"활성화" : L"비활성화") 
                  << std::endl;
        std::wcout << L"- 간격: " << control.KeepAliveIntervalInMilliseconds() 
                  << L"ms" << std::endl;
    }
    catch (hresult_error const& ex)
    {
        std::wcout << L"오류: " << ex.message().c_str() << std::endl;
    }
}
```

이 코드를 설명하자면, 먼저 StreamSocket 객체의 Control() 메서드를 호출하여 StreamSocketControl 객체를 얻는다. 그 다음 KeepAlive() 메서드에 true를 전달하여 킵얼라이브를 활성화한다.

KeepAliveIntervalInMilliseconds()는 킵얼라이브 패킷을 보내는 간격을 밀리초 단위로 설정한다. 30000은 30초를 의미한다. 게임 서버의 경우 보통 30초에서 60초 사이의 값을 사용한다.

킵얼라이브가 활성화되면, 소켓이 유휴 상태일 때 지정한 간격마다 자동으로 패킷이 전송되므로 연결이 유지된다. 만약 네트워크가 단절되었다면, 킵얼라이브 응답이 오지 않아 애플리케이션에서 이를 감지할 수 있다.

## 12.3 NoDelay (Nagle 알고리즘)

### 지연 최소화 설정
TCP 프로토콜에는 Nagle 알고리즘이라는 최적화 기법이 있다. 이 알고리즘은 작은 패킷 여러 개를 모아서 큰 패킷 하나로 보내는 방식으로, 네트워크 효율성을 높인다. 하지만 게임처럼 지연 시간이 중요한 애플리케이션에서는 이러한 버퍼링이 눈에 띄는 지연을 유발할 수 있다.

NoDelay 옵션을 활성화하면 Nagle 알고리즘을 비활성화하여, 데이터가 준비되는 즉시 전송된다. 이를 통해 지연을 최소화할 수 있다.

다음은 NoDelay를 설정하는 예제다.

```cpp
#include <winrt/Windows.Networking.Sockets.h>
#include <iostream>

using namespace winrt;
using namespace Windows::Networking::Sockets;

IAsyncAction SetupNoDelayAsync()
{
    StreamSocket socket;
    
    try
    {
        // NoDelay 활성화 (Nagle 알고리즘 비활성화)
        auto control = socket.Control();
        control.NoDelay(true);
        
        std::wcout << L"NoDelay 설정 완료" << std::endl;
        std::wcout << L"- NoDelay: " 
                  << (control.NoDelay() ? L"활성화" : L"비활성화") 
                  << std::endl;
        std::wcout << L"Nagle 알고리즘이 비활성화되었습니다." << std::endl;
    }
    catch (hresult_error const& ex)
    {
        std::wcout << L"오류: " << ex.message().c_str() << std::endl;
    }
}
```

이 코드는 간단하다. NoDelay() 메서드에 true를 전달하면 Nagle 알고리즘이 비활성화된다. 그러면 데이터가 준비되는 즉시 네트워크로 전송된다.

NoDelay의 트레이드오프를 이해하는 것이 중요하다. Nagle 알고리즘을 비활성화하면 지연 시간은 감소하지만, 작은 패킷들이 네트워크를 통해 전송되므로 전체 처리량은 약간 감소할 수 있다. 게임처럼 지연 시간이 중요한 경우에는 이 트레이드오프가 양호하지만, 대용량 데이터 전송에는 적절하지 않을 수 있다.

실전에서는 애플리케이션의 특성에 따라 결정해야 한다. 액션 게임은 NoDelay를 활성화하고, 대용량 파일 전송은 Nagle 알고리즘을 유지하는 방식을 권장한다.

## 12.4 QualityOfService 설정

### 네트워크 우선순위
QualityOfService(QoS)는 네트워크 트래픽의 우선순위를 설정하는 기능이다. 시스템 관리자가 네트워크 정책을 설정한 경우, 애플리케이션이 자신의 트래픽에 대한 우선순위를 지정할 수 있다. 이를 통해 중요한 트래픽이 덜 중요한 트래픽보다 먼저 처리되도록 할 수 있다.

다음은 QoS를 설정하는 예제다.

```cpp
#include <winrt/Windows.Networking.Sockets.h>
#include <iostream>

using namespace winrt;
using namespace Windows::Networking::Sockets;
using namespace Windows::Networking::Core;

IAsyncAction SetupQualityOfServiceAsync()
{
    StreamSocket socket;
    
    try
    {
        auto control = socket.Control();
        
        // QualityOfService를 "LowLatency"로 설정
        // 다른 옵션: Default, LowLatency, NetworkControl, Throughput
        control.QualityOfService(SocketQualityOfService::LowLatency);
        
        std::wcout << L"QualityOfService 설정 완료" << std::endl;
        std::wcout << L"- QoS: LowLatency (낮은 지연)" << std::endl;
        
        // 다른 예제: 높은 처리량이 필요한 경우
        control.QualityOfService(SocketQualityOfService::Throughput);
        std::wcout << L"- QoS를 Throughput으로 변경" << std::endl;
    }
    catch (hresult_error const& ex)
    {
        std::wcout << L"오류: " << ex.message().c_str() << std::endl;
    }
}
```

SocketQualityOfService enum은 여러 값을 제공한다. LowLatency는 게임이나 실시간 통신에 적합하며, 지연 시간을 최소화하도록 네트워크를 설정한다. Throughput은 대용량 파일 전송처럼 처리량이 중요한 경우에 사용된다. NetworkControl은 네트워크 관리 트래픽의 우선순위를 높인다. Default는 기본값으로 특별한 우선순위를 설정하지 않는다.

실제로 QoS가 적용되려면 네트워크 관리자가 시스템에 QoS 정책을 설정해야 한다. 따라서 모든 환경에서 효과가 있는 것은 아니지만, 설정해두면 적절한 환경에서 성능 향상을 기대할 수 있다.

## 12.5 OutboundBufferSizeInBytes

### 송신 버퍼 크기 조정
송신 버퍼는 애플리케이션이 보내는 데이터를 일시적으로 저장하는 메모리 영역이다. 버퍼가 크면 더 많은 데이터를 한 번에 보낼 수 있지만 메모리를 더 사용한다. 버퍼가 작으면 메모리는 절약하지만 처리량이 감소할 수 있다.

OutboundBufferSizeInBytes로 송신 버퍼의 크기를 조정할 수 있다. 이는 소켓 연결 전에 설정해야 한다.

다음은 송신 버퍼 크기를 조정하는 예제다.

```cpp
#include <winrt/Windows.Networking.Sockets.h>
#include <iostream>

using namespace winrt;
using namespace Windows::Networking::Sockets;

IAsyncAction ConfigureOutboundBufferAsync()
{
    StreamSocket socket;
    
    try
    {
        auto control = socket.Control();
        
        // 송신 버퍼를 256KB로 설정
        uint32_t bufferSize = 256 * 1024; // 256KB
        control.OutboundBufferSizeInBytes(bufferSize);
        
        std::wcout << L"송신 버퍼 설정 완료" << std::endl;
        std::wcout << L"- 버퍼 크기: " 
                  << control.OutboundBufferSizeInBytes() / 1024 
                  << L"KB" << std::endl;
        
        // 대용량 데이터 전송이 필요한 경우 버퍼를 크게 설정
        control.OutboundBufferSizeInBytes(1024 * 1024); // 1MB
        
        std::wcout << L"- 버퍼 크기를 1MB로 증가" << std::endl;
    }
    catch (hresult_error const& ex)
    {
        std::wcout << L"오류: " << ex.message().c_str() << std::endl;
    }
}
```

버퍼 크기를 설정할 때는 용도를 고려해야 한다. 게임 서버의 플레이어 위치 업데이트 같은 작은 메시지를 자주 보내는 경우, 64KB 정도의 작은 버퍼로도 충분하다. 반면 영상 스트리밍처럼 큰 데이터를 보내야 하는 경우, 1MB 이상의 큰 버퍼를 설정하는 것이 좋다.

또한 서버가 많은 클라이언트를 연결해야 하는 경우, 클라이언트당 버퍼가 차지하는 메모리의 합계가 시스템 메모리를 초과하지 않도록 주의해야 한다. 예를 들어 10,000명의 클라이언트 각각에 1MB 버퍼를 할당하면 10GB의 메모리가 필요하다.

## 12.6 ClientCertificate와 보안

### TLS/SSL 설정 기초
인터넷을 통해 민감한 정보(계정 정보, 결제 정보 등)를 전송해야 하는 경우, 암호화를 통해 보호해야 한다. TLS(Transport Layer Security)는 이를 위한 표준 프로토콜이다. ClientCertificate 옵션을 사용하면 클라이언트 인증서를 설정하여 상호 인증(mutual authentication)을 구현할 수 있다.

다음은 클라이언트 인증서를 설정하는 예제다.

```cpp
#include <winrt/Windows.Networking.Sockets.h>
#include <winrt/Windows.Security.Cryptography.Certificates.h>
#include <iostream>

using namespace winrt;
using namespace Windows::Networking::Sockets;
using namespace Windows::Security::Cryptography::Certificates;

IAsyncAction SetupClientCertificateAsync()
{
    StreamSocket socket;
    
    try
    {
        auto control = socket.Control();
        
        // 클라이언트 인증서 설정 (실제로는 인증서 파일에서 로드)
        // 여기서는 개념 설명을 위한 예제임
        
        // TLS를 통해 연결 시 클라이언트 인증서 사용
        // 실제 환경에서는 다음과 같이 인증서를 로드하고 설정
        /*
        auto cert = co_await Certificate::ImportPfxDataAsync(
            certData,
            L"password"
        );
        control.ClientCertificate(cert);
        */
        
        std::wcout << L"클라이언트 인증서 설정 예제" << std::endl;
        std::wcout << L"- TLS/SSL을 사용한 보안 연결" << std::endl;
        
        // 실전 예제: HTTPS 서버에 연결할 때
        std::wcout << L"- 서버 인증서 검증을 설정합니다." << std::endl;
        
        // SSL 오류를 무시하지 않음 (보안상 중요)
        control.IgnorableServerCertificateErrors().Clear();
        
        std::wcout << L"- 서버 인증서 검증: 활성화됨" << std::endl;
    }
    catch (hresult_error const& ex)
    {
        std::wcout << L"오류: " << ex.message().c_str() << std::endl;
    }
}
```

ClientCertificate를 설정하는 것은 상호 인증이 필요한 경우에 사용된다. 예를 들어, 게임 서버가 클라이언트의 신분을 확인하거나, 클라이언트가 서버의 신분을 확인할 때 필요하다.

중요한 보안 원칙은 다음과 같다. 첫째, 인증서는 안전한 경로에서 로드해야 한다. 둘째, 서버 인증서 검증은 항상 활성화해야 한다. 셋째, 자체 서명된 인증서(self-signed certificate)는 개발 단계에서만 사용해야 한다.

IgnorableServerCertificateErrors()는 서버 인증서 검증 실패를 무시할 수 있게 한다. 개발 중에는 자체 서명된 인증서를 사용해야 하므로 이를 설정할 수 있지만, 프로덕션 환경에서는 절대 사용해서는 안 된다.

## 12.7 종합 예제: 성능 벤치마크 도구

### 코드의 목적
이 예제는 다양한 소켓 옵션 조합이 네트워크 성능에 미치는 영향을 측정하는 벤치마크 도구를 구현한다. 서로 다른 설정으로 데이터를 전송하고, 처리량(throughput)과 지연 시간(latency)을 측정하여 어떤 옵션 조합이 가장 효율적인지 비교할 수 있다.

이 도구는 게임 서버 개발자가 프로덕션 환경에 배포하기 전에 최적의 소켓 설정을 찾을 수 있게 도와준다.

### 설계 결정 이유
벤치마크는 여러 프로필(Profile)을 정의하고 각각을 테스트한다. "낮은 지연(LowLatency)" 프로필은 게임처럼 응답성이 중요한 경우에, "높은 처리량(HighThroughput)" 프로필은 영상 스트리밍처럼 대역폭이 중요한 경우에 사용된다.

각 프로필마다 여러 작은 패킷을 보내고, 왕복 시간(RTT)과 초당 전송량(packets per second)을 측정한다. 이를 통해 각 프로필의 장단점을 객관적으로 비교할 수 있다.

### 코드 구현

```cpp
#include <winrt/Windows.Networking.Sockets.h>
#include <winrt/Windows.Networking.h>
#include <winrt/Windows.Storage.Streams.h>
#include <iostream>
#include <chrono>
#include <vector>
#include <iomanip>

using namespace winrt;
using namespace Windows::Networking::Sockets;
using namespace Windows::Networking;
using namespace Windows::Storage::Streams;

// 벤치마크 프로필 정의
struct BenchmarkProfile
{
    std::wstring name;
    bool keepAlive;
    bool noDelay;
    uint32_t bufferSize;
    SocketQualityOfService qos;
};

// 벤치마크 결과
struct BenchmarkResult
{
    std::wstring profileName;
    double averageLatencyMs;
    int packetsPerSecond;
    double totalThroughputMBps;
};

// 벤치마크 도구 클래스
class SocketBenchmark
{
private:
    static constexpr int PACKET_SIZE = 1024;        // 1KB 패킷
    static constexpr int TOTAL_PACKETS = 1000;      // 1000개 패킷 전송
    static constexpr int WARMUP_PACKETS = 100;      // 사전 테스트 100개
    
public:
    IAsyncAction RunBenchmarkAsync(std::wstring const& serverAddress, 
                                    std::wstring const& serverPort)
    {
        std::vector<BenchmarkProfile> profiles = CreateProfiles();
        std::vector<BenchmarkResult> results;
        
        for (auto const& profile : profiles)
        {
            std::wcout << L"\n테스트 중: " << profile.name << std::endl;
            auto result = co_await RunProfileAsync(
                profile, 
                serverAddress, 
                serverPort
            );
            results.push_back(result);
        }
        
        PrintResults(results);
    }
    
private:
    std::vector<BenchmarkProfile> CreateProfiles()
    {
        return {
            {
                L"기본 설정",
                false,                                  // keepAlive
                false,                                  // noDelay
                65536,                                  // 64KB 버퍼
                SocketQualityOfService::Default
            },
            {
                L"낮은 지연 (GameServer)",
                true,                                   // keepAlive
                true,                                   // noDelay
                131072,                                 // 128KB 버퍼
                SocketQualityOfService::LowLatency
            },
            {
                L"높은 처리량 (Streaming)",
                true,                                   // keepAlive
                false,                                  // Nagle 활성화
                1048576,                                // 1MB 버퍼
                SocketQualityOfService::Throughput
            },
            {
                L"작은 메모리 풋프린트",
                true,                                   // keepAlive
                true,                                   // noDelay
                32768,                                  // 32KB 버퍼
                SocketQualityOfService::LowLatency
            }
        };
    }
    
    IAsyncAction<BenchmarkResult> RunProfileAsync(
        BenchmarkProfile const& profile,
        std::wstring const& serverAddress,
        std::wstring const& serverPort)
    {
        StreamSocket socket;
        BenchmarkResult result;
        result.profileName = profile.name;
        
        try
        {
            // 소켓 옵션 적용
            auto control = socket.Control();
            control.KeepAlive(profile.keepAlive);
            control.NoDelay(profile.noDelay);
            control.OutboundBufferSizeInBytes(profile.bufferSize);
            control.QualityOfService(profile.qos);
            
            // 서버에 연결
            HostName host(serverAddress);
            co_await socket.ConnectAsync(host, serverPort);
            
            // 사전 테스트 (워밍업)
            for (int i = 0; i < WARMUP_PACKETS; i++)
            {
                co_await SendPacketAsync(socket);
            }
            
            // 실제 벤치마크 측정
            std::vector<double> latencies;
            auto startTime = std::chrono::high_resolution_clock::now();
            
            for (int i = 0; i < TOTAL_PACKETS; i++)
            {
                auto packetStartTime = 
                    std::chrono::high_resolution_clock::now();
                
                co_await SendPacketAsync(socket);
                
                auto packetEndTime = 
                    std::chrono::high_resolution_clock::now();
                auto packetLatency = std::chrono::duration_cast<
                    std::chrono::microseconds>(
                    packetEndTime - packetStartTime).count() / 1000.0;
                
                latencies.push_back(packetLatency);
            }
            
            auto endTime = std::chrono::high_resolution_clock::now();
            auto totalTime = std::chrono::duration_cast<
                std::chrono::milliseconds>(endTime - startTime).count();
            
            // 결과 계산
            double sumLatency = 0;
            for (auto lat : latencies)
            {
                sumLatency += lat;
            }
            result.averageLatencyMs = sumLatency / latencies.size();
            
            result.packetsPerSecond = 
                static_cast<int>((TOTAL_PACKETS * 1000.0) / totalTime);
            
            result.totalThroughputMBps = 
                (result.packetsPerSecond * PACKET_SIZE) / (1024.0 * 1024.0);
            
            std::wcout << L"  완료!" << std::endl;
        }
        catch (hresult_error const& ex)
        {
            std::wcout << L"  오류: " << ex.message().c_str() << std::endl;
            result.averageLatencyMs = -1;
            result.packetsPerSecond = 0;
            result.totalThroughputMBps = 0;
        }
        
        co_return result;
    }
    
    IAsyncAction SendPacketAsync(StreamSocket const& socket)
    {
        try
        {
            auto outputStream = socket.OutputStream();
            DataWriter writer(outputStream);
            
            // PACKET_SIZE 바이트의 데이터 전송
            for (int i = 0; i < PACKET_SIZE; i++)
            {
                writer.WriteByte(static_cast<uint8_t>(i % 256));
            }
            
            co_await writer.StoreAsync();
        }
        catch (hresult_error const&)
        {
            // 패킷 전송 실패는 무시
        }
    }
    
    void PrintResults(std::vector<BenchmarkResult> const& results)
    {
        std::wcout << L"\n" << std::wstring(80, L'=') << std::endl;
        std::wcout << L"벤치마크 결과" << std::endl;
        std::wcout << std::wstring(80, L'=') << std::endl;
        
        std::wcout << std::left << std::setw(30) << L"프로필"
                  << std::setw(20) << L"평균 지연(ms)"
                  << std::setw(20) << L"패킷/초"
                  << std::setw(20) << L"처리량(MB/s)"
                  << std::endl;
        
        std::wcout << std::wstring(80, L'-') << std::endl;
        
        for (auto const& result : results)
        {
            std::wcout << std::left << std::setw(30) << result.profileName
                      << std::setw(20) << std::fixed << std::setprecision(3)
                      << result.averageLatencyMs
                      << std::setw(20) << result.packetsPerSecond
                      << std::setw(20) << std::fixed << std::setprecision(3)
                      << result.totalThroughputMBps
                      << std::endl;
        }
        
        std::wcout << std::wstring(80, L'=') << std::endl;
        
        // 최적의 프로필 출력
        BenchmarkResult bestLatency = results[0];
        BenchmarkResult bestThroughput = results[0];
        
        for (auto const& result : results)
        {
            if (result.averageLatencyMs > 0 && 
                result.averageLatencyMs < bestLatency.averageLatencyMs)
            {
                bestLatency = result;
            }
            if (result.totalThroughputMBps > bestThroughput.totalThroughputMBps)
            {
                bestThroughput = result;
            }
        }
        
        std::wcout << L"\n권장 사항:" << std::endl;
        std::wcout << L"- 낮은 지연이 필요한 경우: " 
                  << bestLatency.profileName << std::endl;
        std::wcout << L"- 높은 처리량이 필요한 경우: " 
                  << bestThroughput.profileName << std::endl;
    }
};

// 사용 예제
IAsyncAction RunBenchmarkExampleAsync()
{
    SocketBenchmark benchmark;
    co_await benchmark.RunBenchmarkAsync(L"localhost", L"8080");
}
```

### 주요 부분 상세 해설
BenchmarkProfile 구조체는 소켓의 여러 설정을 한 번에 관리한다. 이를 통해 서로 다른 설정 조합을 쉽게 비교할 수 있다.

CreateProfiles() 메서드는 실전에서 자주 사용되는 네 가지 프로필을 정의한다. "낮은 지연" 프로필은 게임 서버처럼 빠른 응답이 필요한 경우에, "높은 처리량" 프로필은 영상 스트리밍처럼 대역폭이 중요한 경우에 사용된다.

RunProfileAsync() 메서드에서는 먼저 워밍업 단계를 거친다. 이는 JIT 컴파일, 메모리 할당 등의 초기 오버헤드를 제거하기 위함이다. 그 다음 실제 벤치마크를 실행하고, 각 패킷의 왕복 시간을 기록한다.

결과 계산 부분에서는 평균 지연, 초당 패킷 수, 처리량 등 세 가지 메트릭을 계산한다. 이를 통해 성능의 여러 측면을 종합적으로 평가할 수 있다.

PrintResults() 메서드는 결과를 표 형식으로 출력하고, 자동으로 최적의 프로필을 추천한다.

### 개선 방안과 확장 가능성
현재 구현은 단방향(클라이언트에서 서버로)으로만 데이터를 전송한다. 실제 네트워크 통신은 양방향이므로, 서버의 응답 시간도 포함하는 왕복 시간(RTT) 측정을 추가할 수 있다.

두 번째는 다양한 패킷 크기에 대한 테스트다. 현재는 1KB 패킷만 테스트하지만, 작은 패킷(100B)부터 큰 패킷(10MB)까지 다양한 크기로 테스트하면 더 정확한 결과를 얻을 수 있다.

세 번째는 네트워크 상태 시뮬레이션이다. 지연 시간, 패킷 손실, 대역폭 제한 등의 네트워크 조건을 인위적으로 만들어 벤치마크를 실행할 수 있다.

네 번째는 멀티스레드 벤치마킹이다. 여러 클라이언트가 동시에 연결했을 때 성능이 어떻게 변하는지 측정할 수 있다. 이는 게임 서버의 부하 테스트와 유사하다.

다섯 번째는 메모리 사용량 추적이다. 벤치마크와 함께 메모리 할당, 해제 패턴을 기록하여 메모리 누수가 있는지 확인할 수 있다.

실전에서는 이러한 개선 사항들을 적용하여 더욱 정확하고 실용적인 벤치마크 도구를 만들 수 있다.
