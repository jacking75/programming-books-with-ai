# Chapter 5. ThreadPool 활용하기
ThreadPool은 WinRT의 스레드 관리 시스템의 핵심이다. 이전 장에서 배운 `resume_background()`와 `resume_foreground()`는 자동으로 ThreadPool을 사용하지만, ThreadPool API를 직접 사용하면 작업 우선순위, 작업 취소, 작업 간 의존성 등을 더 세밀하게 제어할 수 있다. 이 장에서는 ThreadPool을 마스터해서 고성능 멀티스레드 애플리케이션을 구축하는 방법을 배운다.
  


## 5.1 ThreadPool.RunAsync() API 상세 설명

### API 설명
`ThreadPool::RunAsync()`는 작업을 스레드 풀에 제출하고 비동기 작업 객체를 즉시 반환하는 함수다. 작업이 실제로 실행될 때까지 기다리지 않으므로, 호출 스레드는 계속 다른 작업을 진행할 수 있다.

함수의 시그니처는 다음과 같다:
```cpp
IAsyncAction RunAsync(WorkItemHandler const& handler)
IAsyncAction RunAsync(WorkItemHandler const& handler, WorkItemPriority priority)
```

첫 번째 매개변수 `handler`는 `WorkItemHandler` 타입이며, 이는 `IAsyncAction`을 반환하는 함수다. 람다 함수, 함수 포인터, 또는 함수 객체를 전달할 수 있다.

두 번째 매개변수 `priority`는 선택 사항이며, 작업의 우선순위를 나타낸다. 기본값은 `WorkItemPriority::Normal`이다.

`RunAsync()`가 반환하는 `IAsyncAction`을 사용하면 작업의 완료를 기다리거나, 진행 상황을 모니터링하거나, 작업을 취소할 수 있다.

ThreadPool은 시스템의 CPU 코어 수에 따라 스레드를 자동으로 생성하고 관리한다. 스레드를 명시적으로 생성할 필요가 없으며, 과도한 스레드 생성으로 인한 리소스 낭비도 없다.

### 작업 제출과 완료 처리 예제 

```cpp
#include <winrt/Windows.System.Threading.h>
#include <iostream>

using namespace winrt;
using namespace Windows::System::Threading;

int main()
{
    init_apartment();
    
    // 간단한 작업 제출
    auto workItem = ThreadPool::RunAsync(
        [](IAsyncAction)
        {
            std::cout << "스레드 풀에서 작업 실행 중..." << std::endl;
            // 실제 작업 수행
        });
    
    // 작업 완료 대기
    workItem.get();
    
    std::cout << "작업 완료!" << std::endl;
    
    return 0;
}
```

람다 함수의 매개변수 `IAsyncAction`은 현재 작업 자신을 나타낸다. 작업 내에서 자신을 취소하거나 상태를 확인할 때 사용할 수 있다.
  


## 5.2 WorkItemPriority 이해하기

### API 설명
WinRT의 ThreadPool은 작업에 우선순위를 부여할 수 있다. 우선순위가 높은 작업은 우선순위가 낮은 작업보다 먼저 실행된다. 이는 게임 서버에서 중요한 기능인데, 예를 들어 플레이어의 입력 처리는 통계 로깅보다 높은 우선순위를 가져야 한다.

WinRT는 세 가지 우선순위를 정의한다:

**WorkItemPriority::High**: 최고 우선순위. 중요한 작업이나 시간이 중요한 작업에 사용한다. 게임 서버에서는 플레이어 입력 처리, 게임 상태 업데이트 등이 High 우선순위다.

**WorkItemPriority::Normal**: 일반적인 우선순위. 대부분의 작업이 이 우선순위를 사용한다.

**WorkItemPriority::Low**: 낮은 우선순위. 백그라운드 작업, 통계 수집, 정기적인 정리 작업 등에 사용한다.

우선순위는 작업의 스케줄링에만 영향을 미치며, 작업을 즉시 취소하거나 중단시키지는 않는다. 높은 우선순위 작업이 낮은 우선순위 작업을 선점(preempt)하지 않는다는 뜻이다. 대신 스레드 풀의 큐에서 높은 우선순위 작업이 먼저 선택된다.

### 우선순위별 실행 순서 예제 

```cpp
#include <winrt/Windows.System.Threading.h>
#include <iostream>
#include <queue>
#include <mutex>
#include <chrono>

using namespace winrt;
using namespace Windows::System::Threading;

// 작업 실행 순서를 기록
std::queue<std::string> g_executionOrder;
std::mutex g_orderMutex;

void RecordExecution(const std::string& taskName)
{
    std::lock_guard<std::mutex> lock(g_orderMutex);
    g_executionOrder.push(taskName);
    std::cout << "[" << taskName << "] 실행됨" << std::endl;
}

IAsyncAction DemonstratePriorities()
{
    std::cout << "=== 우선순위 데모 ===" << std::endl;
    
    // 낮은 우선순위 작업 제출
    auto lowPriorityTask = ThreadPool::RunAsync(
        [](IAsyncAction)
        {
            RecordExecution("Low Priority Task");
        },
        WorkItemPriority::Low);
    
    // 높은 우선순위 작업 제출
    auto highPriorityTask = ThreadPool::RunAsync(
        [](IAsyncAction)
        {
            RecordExecution("High Priority Task");
        },
        WorkItemPriority::High);
    
    // 일반 우선순위 작업 제출
    auto normalPriorityTask = ThreadPool::RunAsync(
        [](IAsyncAction)
        {
            RecordExecution("Normal Priority Task");
        },
        WorkItemPriority::Normal);
    
    // 모든 작업 완료 대기
    co_await lowPriorityTask;
    co_await highPriorityTask;
    co_await normalPriorityTask;
    
    std::cout << "\n=== 실행 순서 ===" << std::endl;
    while (!g_executionOrder.empty())
    {
        std::cout << g_executionOrder.front() << std::endl;
        g_executionOrder.pop();
    }
}

int main()
{
    init_apartment();
    
    auto demo = DemonstratePriorities();
    demo.get();
    
    return 0;
}
```

이 예제를 실행하면 실행 순서가 항상 일정하지 않을 수 있다. 왜냐하면 작업들이 동시에 여러 스레드에서 실행되기 때문이다. 그러나 일반적으로 높은 우선순위 작업이 먼저 선택될 가능성이 높다. 실행 순서를 추적하기 위해 뮤텍스를 사용했다.
  


## 5.3 WorkItemHandler와 람다 함수

### API 설명
`WorkItemHandler`는 함수 시그니처로, `IAsyncAction`을 반환하고 `IAsyncAction` 매개변수를 받는 함수를 나타낸다. 이는 C++에서 `std::function<IAsyncAction(IAsyncAction)>`과 유사하지만, WinRT 타입이다.

ThreadPool에 제출하는 작업은 이 시그니처를 따라야 한다. 가장 간단한 방법은 람다 함수를 사용하는 것이다. 람다 함수는 캡처(capture)를 지원하므로, 외부 변수에 접근할 수 있다.

작업 함수 내에서 가능한 작업들은:
- 동기 작업: CPU 계산, 메모리 접근 등
- 비동기 작업: `co_await`를 사용한 다른 비동기 작업 호출
- 취소 확인: 매개변수로 받은 `IAsyncAction`을 사용

### 다양한 작업 정의 방법 예제 

```cpp
#include <winrt/Windows.System.Threading.h>
#include <iostream>
#include <string>

using namespace winrt;
using namespace Windows::System::Threading;

// 1. 람다 함수 (캡처 없음)
void Example1_SimpleLambda()
{
    auto task = ThreadPool::RunAsync(
        [](IAsyncAction)
        {
            std::cout << "간단한 람다 함수 작업" << std::endl;
        });
    task.get();
}

// 2. 람다 함수 (값 캡처)
void Example2_LambdaWithCapture()
{
    int sharedValue = 42;
    std::string taskName = "작업1";
    
    auto task = ThreadPool::RunAsync(
        [sharedValue, taskName](IAsyncAction)
        {
            std::cout << "캡처된 값: " << sharedValue 
                      << ", 이름: " << taskName << std::endl;
        });
    task.get();
}

// 3. 람다 함수 (참조 캡처 - 주의 필요)
void Example3_LambdaWithReferenceCapture()
{
    int counter = 0;
    
    auto task = ThreadPool::RunAsync(
        [&counter](IAsyncAction)
        {
            // 참조로 캡처되었으므로 counter를 직접 수정 가능
            // 하지만 스레드 안전성 주의 필요!
            counter++;
            std::cout << "카운터: " << counter << std::endl;
        });
    task.get();
}

// 4. 함수 포인터
void MyTaskFunction(IAsyncAction)
{
    std::cout << "함수 포인터로 정의된 작업" << std::endl;
}

void Example4_FunctionPointer()
{
    auto task = ThreadPool::RunAsync(MyTaskFunction);
    task.get();
}

int main()
{
    init_apartment();
    
    Example1_SimpleLambda();
    Example2_LambdaWithCapture();
    Example3_LambdaWithReferenceCapture();
    Example4_FunctionPointer();
    
    return 0;
}
```

각 방법의 장단점을 이해하는 것이 중요하다. 람다 함수의 값 캡처는 안전하지만, 변수가 복사되므로 메모리 오버헤드가 있다. 참조 캡처는 메모리 효율적이지만, 캡처된 변수가 작업 실행 중에 삭제되지 않도록 주의해야 한다.
  


## 5.4 작업 취소하기

### API 설명
작업을 제출한 후, 상황에 따라 그 작업을 취소해야 할 수 있다. 예를 들어 사용자가 버튼을 눌러서 실행 중인 작업을 중단하거나, 타임아웃이 발생했거나, 시스템 종료 시에 말이다.

`ThreadPool::RunAsync()`가 반환하는 `IAsyncAction`은 취소 가능하다. `Cancel()` 메서드를 호출하면 작업 취소를 요청한다. 그러나 이미 실행 중인 작업은 즉시 중단되지 않는다. 대신 `CancellationToken`을 통해 취소 여부를 확인하고, 작업 내에서 정리 작업을 수행할 수 있다.

작업이 취소되면 완료 상태는 `AsyncStatus::Canceled`가 되며, `get()`을 호출하면 `hresult_canceled` 예외가 발생한다.

### CancellationToken 통합 예제 

```cpp
#include <winrt/Windows.System.Threading.h>
#include <winrt/Windows.Foundation.h>
#include <iostream>
#include <thread>
#include <chrono>

using namespace winrt;
using namespace Windows::System::Threading;
using namespace Windows::Foundation;

IAsyncAction CancelableTaskDemo()
{
    std::cout << "=== 취소 가능한 작업 데모 ===" << std::endl;
    
    // 취소 토큰 생성
    CancellationTokenSource cts;
    CancellationToken ct = cts.Token();
    
    // 취소 가능한 작업 제출
    auto task = ThreadPool::RunAsync(
        [ct](IAsyncAction)
        {
            std::cout << "작업 시작" << std::endl;
            
            // 10개의 청크로 작업 수행
            for (int i = 0; i < 10; ++i)
            {
                // 취소 확인
                if (ct.IsCancellationRequested())
                {
                    std::cout << "작업 취소됨 (청크 " << i << "/10)" << std::endl;
                    co_return;
                }
                
                // 각 청크마다 100ms 작업
                std::cout << "청크 " << (i + 1) << "/10 처리 중..." << std::endl;
                
                // 실제로는 co_await resume_after()를 사용하지만,
                // 이 예제에서는 동기 sleep으로 시뮬레이션
                Sleep(100);
            }
            
            std::cout << "작업 완료" << std::endl;
        });
    
    // 별도의 코루틴에서 500ms 후 취소 신호 전송
    auto cancelTask = [](CancellationTokenSource cts) -> IAsyncAction
    {
        co_await resume_after(std::chrono::milliseconds(500));
        std::cout << "\n취소 신호 전송" << std::endl;
        cts.Cancel();
    };
    
    try
    {
        co_await cancelTask(cts);
        
        // 작업이 취소될 때까지 대기
        co_await task;
        std::cout << "작업이 정상 완료됨" << std::endl;
    }
    catch (const hresult_canceled&)
    {
        std::cout << "작업이 취소되었습니다" << std::endl;
    }
    catch (const std::exception& e)
    {
        std::cout << "오류: " << e.what() << std::endl;
    }
}

int main()
{
    init_apartment();
    
    auto demo = CancelableTaskDemo();
    demo.get();
    
    return 0;
}
```

중요한 점은 취소는 "요청"이며, 작업이 이를 "확인"하고 "응답"해야 한다는 것이다. 무한 루프에 빠진 작업은 취소 신호를 절대 확인하지 않으므로, 항상 정기적으로 취소 여부를 확인해야 한다.
  


## 5.5 여러 작업 대기하기

### API 설명
경우에 따라 여러 작업을 동시에 제출하고, 모든 작업이 완료될 때까지 기다려야 한다. 예를 들어 게임 서버에서 여러 클라이언트의 데이터를 동시에 로드하고, 모두 완료된 후에 게임을 시작해야 할 수 있다.

WinRT의 비동기 라이브러리는 여러 작업을 조합하는 유틸리티를 제공한다. `when_all()`은 여러 개의 `IAsyncAction` 또는 `IAsyncOperation<T>`을 받아서, 모두 완료될 때까지 기다리는 새로운 비동기 작업을 반환한다.

### when_all 패턴 구현 예제 

```cpp
#include <winrt/Windows.System.Threading.h>
#include <winrt/Windows.Foundation.h>
#include <iostream>
#include <vector>

using namespace winrt;
using namespace Windows::System::Threading;
using namespace Windows::Foundation;

// 간단한 작업을 만드는 헬퍼 함수
IAsyncAction CreateSimpleTask(int taskId, int durationMs)
{
    return ThreadPool::RunAsync(
        [taskId, durationMs](IAsyncAction)
        {
            std::cout << "작업 " << taskId << " 시작" << std::endl;
            Sleep(durationMs);
            std::cout << "작업 " << taskId << " 완료" << std::endl;
        });
}

IAsyncAction MultipleTasksDemo()
{
    std::cout << "=== 여러 작업 대기 데모 ===" << std::endl;
    
    // 3개의 작업 생성
    std::vector<IAsyncAction> tasks;
    tasks.push_back(CreateSimpleTask(1, 300));
    tasks.push_back(CreateSimpleTask(2, 200));
    tasks.push_back(CreateSimpleTask(3, 400));
    
    std::cout << "모든 작업 시작, 완료 대기 중..." << std::endl;
    
    // 모든 작업이 완료될 때까지 대기
    for (auto& task : tasks)
    {
        try
        {
            co_await task;
        }
        catch (const std::exception& e)
        {
            std::cout << "작업 오류: " << e.what() << std::endl;
        }
    }
    
    std::cout << "모든 작업 완료!" << std::endl;
}

// 동적으로 생성되는 작업들을 처리하는 예제
IAsyncAction DynamicTasksDemo()
{
    std::cout << "\n=== 동적 작업 생성 데모 ===" << std::endl;
    
    const int NUM_TASKS = 5;
    std::vector<IAsyncAction> tasks;
    
    // 동적으로 여러 작업 생성
    for (int i = 0; i < NUM_TASKS; ++i)
    {
        tasks.push_back(CreateSimpleTask(i + 1, 100 + (i * 50)));
    }
    
    std::cout << "5개의 작업 시작" << std::endl;
    
    // 모든 작업 완료 대기
    for (auto& task : tasks)
    {
        co_await task;
    }
    
    std::cout << "모든 동적 작업 완료!" << std::endl;
}

int main()
{
    init_apartment();
    
    auto demo1 = MultipleTasksDemo();
    demo1.get();
    
    auto demo2 = DynamicTasksDemo();
    demo2.get();
    
    return 0;
}
```

이 패턴은 매우 유용하다. 게임 서버에서는 여러 플레이어의 데이터를 병렬로 로드하고, 모두 완료되면 게임을 시작하는 식으로 사용할 수 있다.
  


## 5.6 종합 예제: 작업 스케줄러 시스템

### 코드의 목적
이 예제는 우선순위 기반 작업 스케줄러를 구현한다. 작업들이 우선순위에 따라 실행되고, 의존성을 가질 수 있으며, 동시에 실행되는 작업 수를 제한할 수 있다. 게임 서버에서는 플레이어 입력 처리(High), 게임 로직(Normal), 통계 수집(Low) 등을 이런 방식으로 관리할 수 있다.

### 설계 결정 이유

첫 번째, 우선순위 큐를 사용해서 작업을 우선순위 순서대로 저장한다. C++의 `std::priority_queue`를 활용한다.

두 번째, 동시 작업 수를 제한한다. ThreadPool이 무제한으로 많은 작업을 처리하게 하면 시스템 리소스가 낭비될 수 있다. 예를 들어 최대 4개 작업만 동시에 실행하도록 제한한다.

세 번째, 작업 간 의존성을 지원한다. 어떤 작업은 다른 작업이 완료된 후에야 시작할 수 있다.

네 번째, 작업 상태를 추적한다. 대기 중, 실행 중, 완료 등의 상태를 기록한다.

### 주요 부분 상세 해설

작업은 구조체로 정의된다. 각 작업은 ID, 이름, 우선순위, 의존성 리스트, 실행 함수를 가진다.

스케줄러는 작업을 우선순위 큐에 추가하고, 주기적으로 큐에서 작업을 꺼내서 실행한다. 동시 실행 작업 수가 제한을 초과하지 않을 때만 새 작업을 시작한다.

의존성 처리는 다음과 같다: 의존하는 작업이 모두 완료되어야 작업이 실행 가능 상태가 된다. 스케줄러는 정기적으로 완료된 작업을 확인하고, 대기 중인 작업의 의존성을 검사한다.

### 전체 코드 (190라인)

```cpp
#include <winrt/Windows.System.Threading.h>
#include <winrt/Windows.Foundation.h>
#include <iostream>
#include <queue>
#include <map>
#include <vector>
#include <memory>
#include <mutex>
#include <set>

using namespace winrt;
using namespace Windows::System::Threading;
using namespace Windows::Foundation;

// 작업 상태
enum class TaskState
{
    Waiting,     // 의존성 대기 중
    Ready,       // 실행 가능
    Running,     // 실행 중
    Completed,   // 완료됨
    Failed       // 실패함
};

// 작업 정의
struct ScheduledTask
{
    int id;
    std::string name;
    WorkItemPriority priority;
    std::vector<int> dependencies;  // 이 작업이 의존하는 작업 ID들
    std::function<void()> work;      // 실행할 작업
    TaskState state = TaskState::Waiting;
    
    bool operator>(const ScheduledTask& other) const
    {
        // 우선순위가 높을수록 먼저 실행 (우선순위 큐용)
        return priority < other.priority;
    }
};

class TaskScheduler
{
private:
    static const int MAX_CONCURRENT_TASKS = 4;
    
    std::priority_queue<ScheduledTask, std::vector<ScheduledTask>, 
                       std::greater<ScheduledTask>> m_taskQueue;
    std::map<int, ScheduledTask> m_allTasks;
    std::set<int> m_runningTasks;
    std::set<int> m_completedTasks;
    std::mutex m_mutex;
    
public:
    void AddTask(const ScheduledTask& task)
    {
        std::lock_guard<std::mutex> lock(m_mutex);
        m_allTasks[task.id] = task;
        m_taskQueue.push(task);
    }
    
    bool AreAllDependenciesComplete(int taskId)
    {
        const auto& task = m_allTasks[taskId];
        for (int depId : task.dependencies)
        {
            if (m_completedTasks.find(depId) == m_completedTasks.end())
                return false;
        }
        return true;
    }
    
    IAsyncAction RunScheduler()
    {
        std::cout << "=== 작업 스케줄러 시작 ===" << std::endl;
        
        while (!m_allTasks.empty())
        {
            std::vector<int> tasksToRun;
            
            {
                std::lock_guard<std::mutex> lock(m_mutex);
                
                // 현재 실행 중인 작업이 완료되었는지 확인
                // (실제로는 각 작업의 IAsyncAction을 추적해야 함)
                
                // 실행 가능한 작업 찾기
                for (auto& [id, task] : m_allTasks)
                {
                    if (task.state == TaskState::Waiting && 
                        AreAllDependenciesComplete(id) &&
                        m_runningTasks.size() < MAX_CONCURRENT_TASKS)
                    {
                        tasksToRun.push_back(id);
                        task.state = TaskState::Ready;
                    }
                }
            }
            
            // 준비된 작업들을 실행
            for (int taskId : tasksToRun)
            {
                auto& task = m_allTasks[taskId];
                std::cout << "[" << task.name << "] 시작 (ID: " << task.id << ")" 
                         << std::endl;
                
                task.state = TaskState::Running;
                m_runningTasks.insert(taskId);
                
                auto asyncTask = ThreadPool::RunAsync(
                    [this, taskId](IAsyncAction)
                    {
                        try
                        {
                            auto& task = m_allTasks[taskId];
                            task.work();
                            
                            {
                                std::lock_guard<std::mutex> lock(m_mutex);
                                task.state = TaskState::Completed;
                                m_completedTasks.insert(taskId);
                                m_runningTasks.erase(taskId);
                            }
                            
                            std::cout << "[" << task.name << "] 완료" << std::endl;
                        }
                        catch (const std::exception& e)
                        {
                            std::lock_guard<std::mutex> lock(m_mutex);
                            auto& task = m_allTasks[taskId];
                            task.state = TaskState::Failed;
                            m_runningTasks.erase(taskId);
                            std::cout << "[" << task.name << "] 실패: " 
                                     << e.what() << std::endl;
                        }
                    },
                    task.priority);
                
                co_await asyncTask;
            }
            
            // 모든 작업이 완료되었는지 확인
            {
                std::lock_guard<std::mutex> lock(m_mutex);
                if (m_completedTasks.size() == m_allTasks.size())
                    break;
            }
            
            // 짧은 대기 후 다시 확인
            co_await resume_after(std::chrono::milliseconds(100));
        }
        
        std::cout << "=== 스케줄러 종료 ===" << std::endl;
    }
};

int main()
{
    init_apartment();
    
    TaskScheduler scheduler;
    
    // 테스트 작업들 추가
    scheduler.AddTask({
        1, "로드 데이터", WorkItemPriority::High,
        {}, [](){ Sleep(200); }
    });
    
    scheduler.AddTask({
        2, "초기화", WorkItemPriority::High,
        {}, [](){ Sleep(150); }
    });
    
    scheduler.AddTask({
        3, "검증", WorkItemPriority::Normal,
        {1, 2}, [](){ Sleep(100); }  // 작업 1, 2 완료 후 실행
    });
    
    scheduler.AddTask({
        4, "처리", WorkItemPriority::Normal,
        {3}, [](){ Sleep(200); }  // 작업 3 완료 후 실행
    });
    
    scheduler.AddTask({
        5, "로깅", WorkItemPriority::Low,
        {4}, [](){ Sleep(50); }  // 작업 4 완료 후 실행
    });
    
    scheduler.AddTask({
        6, "정리", WorkItemPriority::Low,
        {}, [](){ Sleep(75); }
    });
    
    // 스케줄러 실행
    auto schedulerTask = scheduler.RunScheduler();
    schedulerTask.get();
    
    std::cout << "\n모든 작업 완료!" << std::endl;
    
    return 0;
}
```

### 코드 실행 흐름 설명

1. 여러 작업이 스케줄러에 추가된다.
2. 스케줄러는 의존성을 확인하고, 의존성이 없거나 모두 완료된 작업부터 실행한다.
3. 동시에 실행되는 작업이 최대 4개를 초과하지 않도록 제한한다.
4. 각 작업이 완료되면 상태를 업데이트하고, 대기 중인 작업의 의존성을 재확인한다.
5. 모든 작업이 완료되면 스케줄러가 종료한다.

### 개선 방안과 확장 가능성

**현재 한계:**

첫 번째, 모든 작업을 동기적으로 처리한다. 즉, 한 작업이 완료될 때까지 다음 작업을 시작하지 않는다. 이는 단순하지만 비효율적이다.

두 번째, 작업의 실행 결과를 반환하지 않는다. 실제로는 작업이 계산 결과를 생성할 수 있고, 이를 의존하는 작업이 사용해야 할 수도 있다.

세 번째, 작업 추가 후 스케줄러를 시작하는 방식이다. 동적으로 작업을 추가해야 할 경우 불편하다.

**개선 방안:**

첫 번째, 비동기 스케줄링을 구현할 수 있다. 각 작업의 `IAsyncAction`을 별도로 추적하면서, 완료 여부를 비동기로 확인한다.

두 번째, 제네릭 템플릿을 사용해서 작업의 반환값을 처리할 수 있다. 이렇게 하면 작업 간 데이터 전달이 가능해진다.

세 번째, 스케줄러를 계속 실행하면서 새 작업을 추가할 수 있는 동적 스케줄링을 구현할 수 있다.

네 번째, 작업 그룹(batch)을 지원할 수 있다. 여러 작업이 하나의 논리적 단위로 처리되도록 한다.

**게임 서버 적용 예시:**

이 패턴은 게임 서버의 다양한 부분에 적용될 수 있다:
- 게임 월드 초기화: 여러 섹션을 병렬로 로드하되, 특정 의존성을 유지
- 플레이어 로그인: 계정 확인 → 데이터 로드 → 검증 → 게임월드 입장
- 배치 작업: 매시간 통계 수집 → 처리 → 저장 → 보고서 생성

다음 장에서는 타이머를 다룬다. 타이머는 스케줄러와 함께 사용되어 정기적인 작업을 효과적으로 관리할 수 있게 해준다.

---

이 장에서 배운 내용을 정리하면:

1. **ThreadPool::RunAsync()**: 작업을 스레드 풀에 제출하고 비동기 작업 객체를 반환한다.

2. **WorkItemPriority**: 작업의 우선순위를 설정해서 중요한 작업이 먼저 실행되도록 한다.

3. **WorkItemHandler**: 작업 함수의 시그니처로, 람다 함수나 함수 포인터로 구현할 수 있다.

4. **작업 취소**: CancellationToken을 사용해서 작업을 취소할 수 있으며, 작업 내에서 취소 여부를 정기적으로 확인해야 한다.

5. **여러 작업 대기**: 여러 작업을 동시에 실행하고, 모두 완료될 때까지 기다릴 수 있다.

6. **스케줄러 패턴**: 우선순위, 의존성, 동시 실행 제한을 조합해서 복잡한 작업 흐름을 관리할 수 있다.

다음 장에서는 타이머를 배운다. ThreadPool 위에 타이머를 구축하면, 게임의 주기적인 이벤트(틱, 업데이트, 저장 등)를 효율적으로 처리할 수 있다.
