## Appendix A. WinRT 네트워킹 API 레퍼런스
WinRT 네트워킹은 매우 강력하지만 API가 많아서 헷갈리기 쉽다. 여기서는 게임 서버 개발에 필요한 핵심 API들을 정리한다.

### StreamSocket (TCP)

**목적**: 신뢰성 있는 양방향 통신이 필요할 때 사용한다.

**주요 메서드**

| 메서드 | 목적 | 반환값 |
|--------|------|--------|
| `ConnectAsync(...)` | 서버에 연결 | `IAsyncAction` |
| `InputStream` | 수신 스트림 접근 | `IInputStream` |
| `OutputStream` | 송신 스트림 접근 | `IOutputStream` |
| `Control` | 소켓 옵션 제어 | `StreamSocketControl` |
| `Information` | 소켓 정보 조회 | `StreamSocketInformation` |

**핵심 프로퍼티**

```cpp
// 소켓 타임아웃 설정
socket.Control().KeepAliveEnabled(true);
socket.Control().NoDelay(true);  // Nagle 알고리즘 비활성화

// 원격 주소/포트 확인
auto remoteAddress = socket.Information().RemoteAddress();
auto remotePort = socket.Information().RemotePort();
```

**일반적인 사용 패턴**

```cpp
StreamSocket socket;

// 연결
co_await socket.ConnectAsync(hostName, serviceNameOrPort);

// 송신
DataWriter writer(socket.OutputStream());
writer.WriteString(L"Hello");
co_await writer.StoreAsync();

// 수신
DataReader reader(socket.InputStream());
co_await reader.LoadAsync(1024);
auto message = reader.ReadString(reader.UnconsumedBufferLength());
```

### StreamSocketListener (TCP 서버)

**목적**: TCP 서버를 만들 때 사용한다. 클라이언트의 연결을 수락한다.

**주요 메서드**

| 메서드 | 목적 | 반환값 |
|--------|------|--------|
| `BindServiceNameAsync(...)` | 포트에 바인드 | `IAsyncAction` |
| `ConnectionReceived` | 이벤트 (클라이언트 연결됨) | `TypedEventHandler` |
| `Close()` | 서버 종료 | `void` |

**일반적인 사용 패턴**

```cpp
StreamSocketListener listener;

// 이벤트 핸들러 등록
listener.ConnectionReceived([this](StreamSocketListener sender, 
                                   StreamSocketListenerConnectionReceivedEventArgs e) {
    // 새로운 클라이언트 연결
    auto socket = e.Socket();
    HandleNewClient(socket);
});

// 포트에 바인드
co_await listener.BindServiceNameAsync(L"8080");
```

### DatagramSocket (UDP)

**목적**: 빠른 통신이 필요하지만 신뢰성은 필요 없을 때 사용한다.

**주요 메서드**

| 메서드 | 목적 | 반환값 |
|--------|------|--------|
| `BindServiceNameAsync(...)` | 포트에 바인드 | `IAsyncAction` |
| `SendToAsync(...)` | 특정 주소로 전송 | `IAsyncOperation<uint>` |
| `MessageReceived` | 이벤트 (메시지 수신) | `TypedEventHandler` |

**일반적인 사용 패턴**

```cpp
DatagramSocket socket;

// 메시지 수신 이벤트 등록
socket.MessageReceived([this](DatagramSocket sender, DatagramSocketMessageReceivedEventArgs e) {
    auto reader = e.GetDataReader();
    auto data = reader.ReadBuffer(reader.UnconsumedBufferLength());
    HandleMessage(data);
});

// 포트에 바인드
co_await socket.BindServiceNameAsync(L"9090");

// 메시지 전송
DataWriter writer(socket.OutputStream());
writer.WriteBuffer(myBuffer);
co_await socket.SendToAsync(remoteHostName, L"9090");
```

### DataReader / DataWriter

**목적**: 바이트 스트림과 고수준 데이터 타입 사이의 변환을 제공한다.

**DataWriter 메서드**

| 메서드 | 목적 |
|--------|------|
| `WriteByte(uint8_t)` | 1바이트 쓰기 |
| `WriteUInt16(uint16_t)` | 2바이트 정수 쓰기 |
| `WriteUInt32(uint32_t)` | 4바이트 정수 쓰기 |
| `WriteString(hstring)` | 문자열 쓰기 (길이 프리픽스 포함) |
| `WriteBuffer(IBuffer)` | 버퍼 전체 쓰기 |
| `StoreAsync()` | 버퍼를 스트림에 쓰고 플러시 |

**DataReader 메서드**

| 메서드 | 목적 | 반환값 |
|--------|------|--------|
| `LoadAsync(uint)` | 스트림에서 바이트 읽기 | `IAsyncOperation<uint>` |
| `ReadByte()` | 1바이트 읽기 | `uint8_t` |
| `ReadUInt16()` | 2바이트 정수 읽기 | `uint16_t` |
| `ReadUInt32()` | 4바이트 정수 읽기 | `uint32_t` |
| `ReadString(uint)` | 문자열 읽기 | `hstring` |
| `ReadBuffer(uint)` | 버퍼 읽기 | `IBuffer` |
| `UnconsumedBufferLength` | 읽지 않은 바이트 수 | `uint` |

**사용 예제**

```cpp
// 쓰기
DataWriter writer(socket.OutputStream());
writer.WriteUInt32(1000);           // 플레이어 ID
writer.WriteSingle(123.456f);       // X 좌표
writer.WriteSingle(789.012f);       // Y 좌표
co_await writer.StoreAsync();

// 읽기
DataReader reader(socket.InputStream());
co_await reader.LoadAsync(12);      // 12바이트 읽기
uint32_t playerId = reader.ReadUInt32();
float x = reader.ReadSingle();
float y = reader.ReadSingle();
```

### 버퍼 관련 API

**IBuffer 생성**

| 메서드 | 목적 | 반환값 |
|--------|------|--------|
| `CryptographicBuffer::GenerateRandomBuffer(uint)` | 랜덤 버퍼 생성 | `IBuffer` |
| `CryptographicBuffer::ConvertStringToBinary(...)` | 문자열을 바이너리로 변환 | `IBuffer` |
| `CryptographicBuffer::ConvertBinaryToString(...)` | 바이너리를 문자열로 변환 | `hstring` |

**BufferFactory (WinRT에서 제공)**

```cpp
// 특정 크기의 버퍼 생성
auto buffer = CryptographicBuffer::GenerateRandomBuffer(4096);

// 또는 바이트 배열로부터 생성
uint8_t data[] = {1, 2, 3, 4};
// ... 별도 구현 필요
```

## Appendix B. 스레딩 API 레퍼런스
게임 서버에서 멀티스레딩은 필수다. WinRT는 다양한 스레딩 도구를 제공한다.

### ThreadPool

**목적**: 비동기 작업을 스레드 풀에서 실행한다.

**주요 메서드**

| 메서드 | 목적 | 반환값 |
|--------|------|--------|
| `RunAsync(WorkItemHandler)` | 작업 실행 | `IAsyncAction` |
| `RunAsync(priority, WorkItemHandler)` | 우선순위 지정해서 실행 | `IAsyncAction` |

**우선순위**

| 값 | 의미 |
|----|------|
| `WorkItemPriority::Low` | 낮은 우선순위 (백그라운드 작업) |
| `WorkItemPriority::Normal` | 일반 우선순위 |
| `WorkItemPriority::High` | 높은 우선순위 (긴급 작업) |

**사용 예제**

```cpp
// 간단한 비동기 작업
ThreadPool::RunAsync([this](IAsyncAction action) {
    // 이 코드는 스레드 풀에서 실행된다
    DoSomeWork();
});

// 우선순위 지정
ThreadPool::RunAsync(WorkItemPriority::High, [this](IAsyncAction action) {
    // 높은 우선순위로 실행
    HandleUrgentTask();
});
```

### Mutex (뮤텍스)

**목적**: 여러 스레드가 공유 자원에 접근할 때 동기화한다.

**일반적인 사용 패턴 (C++ 표준 라이브러리)**

```cpp
#include <mutex>

std::mutex mtx;
std::lock_guard<std::mutex> lock(mtx);
// lock 범위에서 상호 배제 보장
```

**WinRT Slim Reader/Writer Lock**

```cpp
// Slim Reader/Writer Lock는 WinRT에서 제공되지 않으므로
// C++ 표준 라이브러리의 std::shared_mutex 사용
#include <shared_mutex>

std::shared_mutex rwLock;

// 읽기 (여러 스레드 동시 허용)
{
    std::shared_lock<std::shared_mutex> lock(rwLock);
    // 읽기 작업
}

// 쓰기 (하나의 스레드만 허용)
{
    std::unique_lock<std::shared_mutex> lock(rwLock);
    // 쓰기 작업
}
```

### Event (이벤트)

**목적**: 스레드 간 신호 전달이 필요할 때 사용한다.

**Windows API 사용**

```cpp
#include <windows.h>

// 이벤트 생성 (자동 리셋)
HANDLE hEvent = CreateEventW(NULL, FALSE, FALSE, NULL);

// 이벤트 대기 (신호 대기)
WaitForSingleObject(hEvent, INFINITE);

// 이벤트 신호 설정
SetEvent(hEvent);

// 이벤트 리셋
ResetEvent(hEvent);

// 정리
CloseHandle(hEvent);
```

**C++ 표준 라이브러리로 구현**

```cpp
#include <condition_variable>
#include <mutex>

std::condition_variable cv;
std::mutex mtx;
bool isSignaled = false;

// 신호 대기
std::unique_lock<std::mutex> lock(mtx);
cv.wait(lock, [] { return isSignaled; });

// 신호 설정
{
    std::lock_guard<std::mutex> lock(mtx);
    isSignaled = true;
}
cv.notify_one();
```

### TimerQueueTimer (타이머)

**목적**: 일정 시간 간격으로 콜백을 실행한다.

**Windows API 사용**

```cpp
#include <windows.h>

HANDLE hTimerQueue = CreateTimerQueue();
HANDLE hTimer = NULL;

// 타이머 생성 (1초 주기, 처음 실행까지 500ms)
CreateTimerQueueTimer(&hTimer, hTimerQueue,
    [](PVOID param, BOOLEAN timerOrWaitFired) {
        // 타이머 콜백
        OnTimerTick();
    },
    NULL,      // 파라미터
    500,       // 초기 지연 (ms)
    1000,      // 주기 (ms)
    0);        // 플래그

// 타이머 삭제
DeleteTimerQueueTimer(hTimerQueue, hTimer, INVALID_HANDLE_VALUE);

// 타이머 큐 삭제
DeleteTimerQueue(hTimerQueue);
```

## Appendix C. 에러 코드와 해결 방법
게임 서버 개발 중에 자주 마주치는 에러들과 그 해결법을 정리한다.

### SocketErrorStatus (네트워크 에러)

| 에러 | 의미 | 해결법 |
|-----|------|--------|
| `Unknown` | 알 수 없는 에러 | 시스템 로그 확인, 네트워크 연결 테스트 |
| `OperationAborted` | 작업이 취소됨 | 일반적인 종료 상황, 재연결 로직 추가 |
| `HttpInvalidServerResponse` | HTTP 응답 잘못됨 | 서버 구현 확인, HTTP 헤더 형식 확인 |
| `ConnectionRefused` | 연결 거부 | 서버가 실행 중인지 확인, 방화벽 설정 확인 |
| `ConnectionReset` | 연결 초기화됨 | 네트워크 불안정, 타임아웃 처리 구현 |
| `NetworkUnreachable` | 네트워크 도달 불가 | 인터넷 연결 확인, IP 주소 확인 |
| `TimeoutExpired` | 타임아웃 | 타임아웃 값 증가, 네트워크 지연 확인 |
| `HostNameNotResolved` | 호스트명 해석 실패 | DNS 설정 확인, 호스트명 확인 |

**일반적인 에러 처리 패턴**

```cpp
try {
    co_await socket.ConnectAsync(hostName, L"8080");
} catch (hresult_error const& ex) {
    auto status = SocketError::GetStatus(ex.code());
    
    switch (status) {
        case SocketErrorStatus::ConnectionRefused:
            // 서버가 실행 중이지 않음
            std::wcout << L"Server not running\n";
            break;
            
        case SocketErrorStatus::NetworkUnreachable:
            // 네트워크 문제
            std::wcout << L"Network unreachable\n";
            break;
            
        case SocketErrorStatus::TimeoutExpired:
            // 타임아웃
            std::wcout << L"Connection timeout\n";
            break;
            
        default:
            std::wcout << L"Connection failed: " << ex.message().c_str() << L"\n";
    }
}
```

### HRESULT 에러

| 에러 코드 | 의미 | 해결법 |
|---------|------|--------|
| `S_OK (0x00000000)` | 성공 | - |
| `E_INVALIDARG (0x80070057)` | 잘못된 인자 | 함수 호출 파라미터 확인 |
| `E_OUTOFMEMORY (0x8007000E)` | 메모리 부족 | 메모리 누수 확인, 할당 크기 검증 |
| `E_FAIL (0x80004005)` | 일반 실패 | 자세한 에러 정보 조회 |
| `E_PENDING (0x8000000A)` | 작업 대기 중 | 비동기 작업 완료 대기 |
| `RPC_E_SERVERFAULT (0x80010105)` | 서버 에러 | 원격 서버 문제, 로그 확인 |

**에러 코드를 문자열로 변환**

```cpp
std::wstring GetHRESULTErrorMessage(HRESULT hr) {
    switch (hr) {
        case S_OK:
            return L"Success";
        case E_INVALIDARG:
            return L"Invalid argument";
        case E_OUTOFMEMORY:
            return L"Out of memory";
        case E_FAIL:
            return L"General failure";
        default:
            return L"Unknown error: 0x" + std::to_wstring(static_cast<uint32_t>(hr));
    }
}
```

### 일반적인 런타임 에러

| 상황 | 원인 | 해결법 |
|-----|------|--------|
| "Access denied" | 권한 부족 | 관리자 권한으로 실행, 포트 권한 확인 |
| "Address already in use" | 포트 이미 사용 중 | 다른 포트 사용, 시간 대기 후 재시작 |
| "Too many open files" | 파일 디스크립터 부족 | 소켓 누수 확인, 연결 해제 확인 |
| "Broken pipe" | 원격 연결 끊김 | 재연결 로직 구현, 하트비트 구현 |
| "Memory corruption" | 메모리 손상 | 버퍼 오버플로우 확인, 메모리 할당 검증 |

## Appendix E. 프로토콜 설계 패턴
재사용 가능한 프로토콜 패턴들을 정리한다.

### 패턴 1: 헤더 + 페이로드 구조

가장 일반적인 패턴이다. 메시지 헤더에 타입과 길이를 포함한다.

**구조**

```
[MessageType(1byte)] [Length(2bytes)] [Payload(variable)]
```

**구현**

```cpp
struct MessageHeader {
    uint8_t type;       // 메시지 타입
    uint16_t length;    // 페이로드 길이
};

void SendMessage(const uint8_t* payload, uint16_t payloadLength, uint8_t type) {
    MessageHeader header;
    header.type = type;
    header.length = payloadLength;
    
    // 헤더 + 페이로드 전송
    SendData((uint8_t*)&header, sizeof(header));
    SendData(payload, payloadLength);
}

bool ReceiveMessage(uint8_t* outPayload, uint16_t& outLength) {
    // 헤더 수신
    MessageHeader header;
    if (!ReceiveData((uint8_t*)&header, sizeof(header))) {
        return false;
    }
    
    // 페이로드 수신
    if (!ReceiveData(outPayload, header.length)) {
        return false;
    }
    
    outLength = header.length;
    return true;
}
```

### 패턴 2: 길이-값 (TLV: Type-Length-Value)

더 유연한 구조로, 여러 필드를 하나의 메시지에 포함할 수 있다.

**구조**

```
[Type(1)] [Length(2)] [Value(variable)] 
[Type(1)] [Length(2)] [Value(variable)] 
...
```

**구현**

```cpp
struct TLVField {
    uint8_t type;
    uint16_t length;
    uint8_t value[256];
};

void AddTLVField(std::vector<uint8_t>& message, uint8_t type, 
                 const uint8_t* value, uint16_t length) {
    message.push_back(type);
    message.push_back((length >> 8) & 0xFF);
    message.push_back(length & 0xFF);
    message.insert(message.end(), value, value + length);
}

bool ParseTLVFields(const uint8_t* data, size_t dataLength,
                    std::vector<TLVField>& outFields) {
    size_t offset = 0;
    while (offset < dataLength) {
        if (offset + 3 > dataLength) break;
        
        TLVField field;
        field.type = data[offset++];
        field.length = (data[offset] << 8) | data[offset + 1];
        offset += 2;
        
        if (offset + field.length > dataLength) break;
        
        std::memcpy(field.value, data + offset, field.length);
        offset += field.length;
        
        outFields.push_back(field);
    }
    return true;
}
```

### 패턴 5: 실시간 스트림 (Real-time Stream)
게임 상태를 연속적으로 전송한다.

**구조**

```
[SequenceNumber(4)] [Timestamp(8)] [PlayerCount(2)]
[PlayerID(4)] [PosX(4)] [PosY(4)] ...
[PlayerID(4)] [PosX(4)] [PosY(4)] ...
```

**구현**

```cpp
void SendGameState(uint32_t sequenceNumber, const std::vector<PlayerState>& players) {
    std::vector<uint8_t> message;
    
    // 시퀀스 번호
    AppendUInt32(message, sequenceNumber);
    
    // 타임스탬프
    auto now = std::chrono::system_clock::now().time_since_epoch().count();
    AppendUInt64(message, now);
    
    // 플레이어 수
    AppendUInt16(message, players.size());
    
    // 플레이어 데이터
    for (const auto& player : players) {
        AppendUInt32(message, player.id);
        AppendFloat(message, player.posX);
        AppendFloat(message, player.posY);
    }
    
    SendData(message.data(), message.size());
}
```

### 패턴 6: 요청-응답 (Request-Response)
클라이언트 요청에 서버가 응답한다.

**구조**

**요청**
```
[RequestID(4)] [RequestType(1)] [Payload(variable)]
```

**응답**
```
[RequestID(4)] [ResponseCode(1)] [Payload(variable)]
```

**구현**

```cpp
struct Request {
    uint32_t requestId;
    uint8_t type;
    std::vector<uint8_t> payload;
};

struct Response {
    uint32_t requestId;  // 요청과 같은 ID
    uint8_t responseCode; // 0 = 성공, 1 = 실패
    std::vector<uint8_t> payload;
};

void HandleRequest(const Request& req) {
    Response resp;
    resp.requestId = req.requestId;
    
    switch (req.type) {
        case 0x01:  // 로그인 요청
            resp.responseCode = ProcessLogin(req.payload) ? 0 : 1;
            break;
        case 0x02:  // 게임 참여 요청
            resp.responseCode = ProcessJoinGame(req.payload) ? 0 : 1;
            break;
        default:
            resp.responseCode = 1;
    }
    
    SendResponse(resp);
}
```

### 패턴 7: 하트비트 (Heartbeat)
연결 유지를 확인한다.

**구조**

```
[Type: HEARTBEAT(1)] [Timestamp(8)]
```

**구현**

```cpp
void SendHeartbeat() {
    std::vector<uint8_t> message;
    message.push_back(0xFF);  // HEARTBEAT 타입
    
    auto now = std::chrono::system_clock::now().time_since_epoch().count();
    uint8_t* ptr = (uint8_t*)&now;
    message.insert(message.end(), ptr, ptr + 8);
    
    SendData(message.data(), message.size());
}

bool CheckHeartbeat(const std::vector<uint8_t>& lastHeartbeat, 
                    int timeoutSeconds) {
    auto now = std::chrono::system_clock::now().time_since_epoch().count();
    
    if (lastHeartbeat.size() < 9) return false;
    
    int64_t lastTime;
    std::memcpy(&lastTime, lastHeartbeat.data() + 1, 8);
    
    int elapsedSeconds = (now - lastTime) / 1000000000;  // 나노초를 초로 변환
    
    return elapsedSeconds < timeoutSeconds;
}
```

### 프로토콜 선택 가이드

| 패턴 | 장점 | 단점 | 사용 시기 |
|-----|------|------|----------|
| 헤더+페이로드 | 간단함 | 유연성 부족 | 메시지 타입 수가 적을 때 |
| TLV | 유연함 | 파싱 복잡 | 필드가 자주 변할 때 |
| JSON/Protobuf | 표준화, 가독성 | 오버헤드 큼 | 복잡한 구조, 디버깅 중요 |
| 고정 크기 | 빠름 | 낭비 가능 | 실시간 게임 |
| 실시간 스트림 | 효율적 | 복잡함 | 상태 동기화 많음 |
| 요청-응답 | 명확함 | 지연 발생 | 로그인, 구매 등 |
| 하트비트 | 신뢰성 | 오버헤드 | 연결 추적 필요 |
