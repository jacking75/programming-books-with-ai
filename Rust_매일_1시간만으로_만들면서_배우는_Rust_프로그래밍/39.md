# Day 39: unsafe와 FFI 맛보기

지금까지 우리는 Rust의 안전성 보장 안에서 프로그래밍을 해왔다. 컴파일러가 소유권과 빌림 규칙을 검사하여 메모리 안전성과 데이터 경쟁을 컴파일 타임에 방지해주었다. 하지만 실무에서는 때때로 이러한 안전망을 벗어나야 하는 상황이 발생한다. 레거시 C 라이브러리를 사용해야 하거나, 하드웨어를 직접 제어해야 하거나, 극한의 성능 최적화가 필요한 경우다. 오늘은 이러한 상황에서 사용하는 `unsafe` 코드와 FFI(Foreign Function Interface)를 살펴본다.

## unsafe의 의미와 필요성

**unsafe가 무엇인가:**

Rust에서 `unsafe` 키워드는 "이 코드는 컴파일러가 안전성을 보장할 수 없으니, 프로그래머가 직접 정확성을 책임진다"는 표시다. 이것이 Rust의 안전성을 완전히 무력화하는 것은 아니다. 오히려 위험한 작업을 명시적으로 표시하여 코드 리뷰와 디버깅 시 집중해야 할 부분을 명확히 한다.

unsafe 블록이나 함수 안에서는 다음 다섯 가지 작업을 추가로 할 수 있다:

1. 원시 포인터(raw pointer)를 역참조하기
2. unsafe 함수나 메서드 호출하기
3. 가변 정적 변수에 접근하거나 수정하기
4. unsafe trait 구현하기
5. union의 필드에 접근하기

중요한 점은 unsafe 블록 안에서도 소유권 규칙은 여전히 적용된다는 것이다. unsafe는 컴파일러가 검증할 수 없는 추가적인 작업을 허용할 뿐이다.

**왜 unsafe가 필요한가:**

실무에서 가장 흔한 경우는 C/C++ 라이브러리와의 상호운용이다. 수십 년간 축적된 네트워크 스택, 그래픽 라이브러리, 데이터베이스 드라이버 등을 Rust에서 활용하려면 FFI를 통해 C 함수를 호출해야 한다. 또한 운영체제의 시스템 콜을 직접 호출하거나, 메모리 매핑 I/O를 사용하거나, 극도로 최적화된 자료구조를 구현할 때도 unsafe가 필요하다.

## 원시 포인터 다루기

**원시 포인터의 기초:**

Rust에는 두 종류의 원시 포인터가 있다:

- `*const T`: 불변 원시 포인터
- `*mut T`: 가변 원시 포인터

참조(`&T`, `&mut T`)와 달리 원시 포인터는 다음과 같은 특징이 있다:

- null일 수 있다
- 자동으로 정리되지 않는다
- 빌림 규칙을 무시할 수 있다
- 생성은 safe하지만 역참조는 unsafe다

간단한 예제를 통해 살펴보자:

```rust
fn demonstrate_raw_pointers() {
    let mut value = 42;
    
    // 참조에서 원시 포인터 생성 (safe)
    let raw_const = &value as *const i32;
    let raw_mut = &mut value as *mut i32;
    
    // 원시 포인터 역참조 (unsafe)
    unsafe {
        println!("raw_const points to: {}", *raw_const);
        *raw_mut = 100;
        println!("after mutation: {}", *raw_const);
    }
}
```

이 코드는 안전한 참조에서 원시 포인터를 만들고, unsafe 블록 안에서 역참조한다. 원시 포인터를 만드는 것 자체는 안전하다. 메모리에 접근하지 않기 때문이다. 하지만 역참조(`*raw_const`)는 unsafe 블록 안에서만 가능하다.

**실전 예제: 링 버퍼 구현:**

원시 포인터가 실제로 유용한 경우를 보기 위해 고성능 링 버퍼를 구현해보자. 링 버퍼는 네트워크 프로그래밍에서 자주 사용되는 자료구조로, 고정 크기 버퍼를 순환하며 사용한다.

```rust
use std::alloc::{alloc, dealloc, Layout};
use std::ptr;

pub struct RingBuffer {
    buffer: *mut u8,
    capacity: usize,
    read_pos: usize,
    write_pos: usize,
}

impl RingBuffer {
    pub fn new(capacity: usize) -> Self {
        unsafe {
            let layout = Layout::array::<u8>(capacity).unwrap();
            let buffer = alloc(layout);
            
            if buffer.is_null() {
                panic!("메모리 할당 실패");
            }
            
            RingBuffer {
                buffer,
                capacity,
                read_pos: 0,
                write_pos: 0,
            }
        }
    }
    
    pub fn write(&mut self, data: &[u8]) -> usize {
        let available = self.available_write();
        let to_write = data.len().min(available);
        
        unsafe {
            for i in 0..to_write {
                let pos = (self.write_pos + i) % self.capacity;
                ptr::write(self.buffer.add(pos), data[i]);
            }
        }
        
        self.write_pos = (self.write_pos + to_write) % self.capacity;
        to_write
    }
    
    pub fn read(&mut self, data: &mut [u8]) -> usize {
        let available = self.available_read();
        let to_read = data.len().min(available);
        
        unsafe {
            for i in 0..to_read {
                let pos = (self.read_pos + i) % self.capacity;
                data[i] = ptr::read(self.buffer.add(pos));
            }
        }
        
        self.read_pos = (self.read_pos + to_read) % self.capacity;
        to_read
    }
    
    fn available_write(&self) -> usize {
        if self.write_pos >= self.read_pos {
            self.capacity - (self.write_pos - self.read_pos) - 1
        } else {
            self.read_pos - self.write_pos - 1
        }
    }
    
    fn available_read(&self) -> usize {
        if self.write_pos >= self.read_pos {
            self.write_pos - self.read_pos
        } else {
            self.capacity - (self.read_pos - self.write_pos)
        }
    }
}

impl Drop for RingBuffer {
    fn drop(&mut self) {
        unsafe {
            let layout = Layout::array::<u8>(self.capacity).unwrap();
            dealloc(self.buffer, layout);
        }
    }
}

// Send와 Sync는 자동으로 구현되지 않으므로 명시적으로 구현
unsafe impl Send for RingBuffer {}
unsafe impl Sync for RingBuffer {}
```

이 구현에서 주목할 점은 unsafe 코드를 최소화하고 안전한 인터페이스로 감쌌다는 것이다. 외부에서는 `write()`와 `read()` 같은 안전한 메서드만 사용하면 되고, 내부의 unsafe 작업은 라이브러리 작성자가 책임진다.

```rust
fn test_ring_buffer() {
    let mut rb = RingBuffer::new(10);
    
    // 데이터 쓰기
    let written = rb.write(b"Hello");
    println!("written: {} bytes", written);
    
    // 데이터 읽기
    let mut buffer = [0u8; 5];
    let read = rb.read(&mut buffer);
    println!("read: {} bytes, data: {}", read, String::from_utf8_lossy(&buffer));
}
```

## FFI로 C 라이브러리 사용하기

**FFI의 기본 개념:**

FFI(Foreign Function Interface)는 다른 언어로 작성된 코드를 호출하는 메커니즘이다. Rust는 C ABI(Application Binary Interface)와 호환되므로 C로 작성된 라이브러리를 직접 호출할 수 있다. 이것이 중요한 이유는 대부분의 시스템 라이브러리와 레거시 코드가 C API를 제공하기 때문이다.

**간단한 예제: C 표준 라이브러리 사용:**

가장 기본적인 예제로 C의 `strlen` 함수를 호출해보자:

```rust
use std::ffi::CStr;
use std::os::raw::c_char;

// C 함수 선언
extern "C" {
    fn strlen(s: *const c_char) -> usize;
}

fn safe_strlen(s: &str) -> usize {
    // Rust 문자열을 C 문자열로 변환
    let c_str = std::ffi::CString::new(s).expect("CString 변환 실패");
    
    unsafe {
        // C 함수 호출
        strlen(c_str.as_ptr())
    }
}

fn test_strlen() {
    let length = safe_strlen("Hello, FFI!");
    println!("문자열 길이: {}", length);
}
```

여기서 `extern "C"` 블록은 C ABI를 사용하는 외부 함수를 선언한다. `strlen` 호출은 unsafe 블록 안에서만 가능하다. 컴파일러가 C 코드의 안전성을 검증할 수 없기 때문이다.

**실전 예제: 시스템 정보 가져오기:**

실무에서 자주 사용되는 패턴으로, Linux의 `sysinfo` 시스템 콜을 호출하여 시스템 정보를 가져와보자:

```rust
use std::mem;

#[repr(C)]
#[derive(Debug)]
struct SysInfo {
    uptime: i64,
    loads: [u64; 3],
    totalram: u64,
    freeram: u64,
    sharedram: u64,
    bufferram: u64,
    totalswap: u64,
    freeswap: u64,
    procs: u16,
    pad: u16,
    totalhigh: u64,
    freehigh: u64,
    mem_unit: u32,
    _padding: [u8; 0],
}

extern "C" {
    fn sysinfo(info: *mut SysInfo) -> i32;
}

pub fn get_system_info() -> Result<SystemInfo, String> {
    let mut info: SysInfo = unsafe { mem::zeroed() };
    
    let result = unsafe {
        sysinfo(&mut info as *mut SysInfo)
    };
    
    if result != 0 {
        return Err("sysinfo 호출 실패".to_string());
    }
    
    Ok(SystemInfo {
        uptime_seconds: info.uptime,
        total_ram_mb: (info.totalram * info.mem_unit as u64) / (1024 * 1024),
        free_ram_mb: (info.freeram * info.mem_unit as u64) / (1024 * 1024),
        process_count: info.procs,
    })
}

#[derive(Debug)]
pub struct SystemInfo {
    pub uptime_seconds: i64,
    pub total_ram_mb: u64,
    pub free_ram_mb: u64,
    pub process_count: u16,
}
```

이 코드의 핵심 요소들을 살펴보자:

**#[repr(C)]** 속성은 구조체의 메모리 레이아웃을 C와 호환되게 만든다. Rust는 기본적으로 구조체 필드를 재배열하여 최적화하는데, C 구조체와 통신할 때는 정확히 같은 레이아웃이 필요하다.

**mem::zeroed()** 는 메모리를 0으로 초기화한다. 이것은 unsafe 함수인데, 모든 비트가 0인 것이 유효한 값이 아닌 타입도 있기 때문이다. 하지만 SysInfo는 POD(Plain Old Data) 구조체이므로 안전하다.

실제 사용은 이렇게 한다:

```rust
fn main() {
    match get_system_info() {
        Ok(info) => {
            println!("시스템 정보:");
            println!("  가동 시간: {} 초", info.uptime_seconds);
            println!("  전체 RAM: {} MB", info.total_ram_mb);
            println!("  여유 RAM: {} MB", info.free_ram_mb);
            println!("  프로세스 수: {}", info.process_count);
        }
        Err(e) => eprintln!("에러: {}", e),
    }
}
```

**C 라이브러리 래핑: OpenSSL 예제:**

더 복잡한 예제로, 실무에서 자주 사용하는 OpenSSL 라이브러리를 사용하여 SHA-256 해시를 계산해보자. 실제로는 `openssl` 크레이트를 사용하는 것이 좋지만, FFI 패턴을 이해하기 위해 직접 구현한다.

```rust
use std::ffi::c_void;

const SHA256_DIGEST_LENGTH: usize = 32;

#[repr(C)]
struct SHA256_CTX {
    h: [u32; 8],
    nl: u32,
    nh: u32,
    data: [u8; 64],
    num: u32,
}

extern "C" {
    fn SHA256_Init(ctx: *mut SHA256_CTX) -> i32;
    fn SHA256_Update(ctx: *mut SHA256_CTX, data: *const c_void, len: usize) -> i32;
    fn SHA256_Final(md: *mut u8, ctx: *mut SHA256_CTX) -> i32;
}

pub struct Sha256 {
    ctx: SHA256_CTX,
}

impl Sha256 {
    pub fn new() -> Self {
        unsafe {
            let mut ctx = std::mem::zeroed();
            SHA256_Init(&mut ctx);
            Sha256 { ctx }
        }
    }
    
    pub fn update(&mut self, data: &[u8]) {
        unsafe {
            SHA256_Update(
                &mut self.ctx,
                data.as_ptr() as *const c_void,
                data.len()
            );
        }
    }
    
    pub fn finalize(mut self) -> [u8; SHA256_DIGEST_LENGTH] {
        let mut hash = [0u8; SHA256_DIGEST_LENGTH];
        unsafe {
            SHA256_Final(hash.as_mut_ptr(), &mut self.ctx);
        }
        hash
    }
}

fn test_sha256() {
    let mut hasher = Sha256::new();
    hasher.update(b"Hello, ");
    hasher.update(b"World!");
    
    let hash = hasher.finalize();
    
    print!("SHA-256: ");
    for byte in hash.iter() {
        print!("{:02x}", byte);
    }
    println!();
}
```

이 패턴은 실무에서 매우 흔하다. unsafe한 C API를 안전한 Rust 인터페이스로 감싸서 사용자가 메모리 안전성을 걱정하지 않고 사용할 수 있게 한다.

## 안전 추상화 만들기

**unsafe 코드의 원칙:**

unsafe 코드를 작성할 때는 다음 원칙을 따른다:

1. unsafe 코드를 최소화하고 명확히 경계를 정한다
2. 안전한 공개 API 뒤에 unsafe를 숨긴다
3. 불변식(invariant)을 문서화하고 유지한다
4. 철저히 테스트한다

**실전 예제: 안전한 버퍼 관리:**

네트워크 서버에서 자주 사용되는 패턴으로, 메모리 풀을 구현해보자. 객체를 매번 할당/해제하는 대신 재사용하여 성능을 향상시킨다:

```rust
use std::ptr;
use std::mem;

pub struct BufferPool {
    buffers: Vec<*mut Vec<u8>>,
    buffer_size: usize,
}

impl BufferPool {
    pub fn new(capacity: usize, buffer_size: usize) -> Self {
        let mut buffers = Vec::with_capacity(capacity);
        
        for _ in 0..capacity {
            let buffer = Box::new(Vec::with_capacity(buffer_size));
            buffers.push(Box::into_raw(buffer));
        }
        
        BufferPool {
            buffers,
            buffer_size,
        }
    }
    
    pub fn acquire(&mut self) -> Option<PooledBuffer> {
        self.buffers.pop().map(|ptr| {
            let vec = unsafe {
                // 소유권 되찾기
                let mut vec = Box::from_raw(ptr);
                vec.clear();
                vec
            };
            
            PooledBuffer {
                buffer: vec,
                pool_ptr: self as *mut BufferPool,
            }
        })
    }
    
    unsafe fn release(&mut self, buffer: Vec<u8>) {
        if buffer.capacity() == self.buffer_size {
            let ptr = Box::into_raw(Box::new(buffer));
            self.buffers.push(ptr);
        }
        // 용량이 다르면 버린다 (비정상 상황)
    }
}

impl Drop for BufferPool {
    fn drop(&mut self) {
        for ptr in self.buffers.drain(..) {
            unsafe {
                // 메모리 해제
                let _ = Box::from_raw(ptr);
            }
        }
    }
}

pub struct PooledBuffer {
    buffer: Box<Vec<u8>>,
    pool_ptr: *mut BufferPool,
}

impl PooledBuffer {
    pub fn as_mut(&mut self) -> &mut Vec<u8> {
        &mut self.buffer
    }
}

impl Drop for PooledBuffer {
    fn drop(&mut self) {
        unsafe {
            let pool = &mut *self.pool_ptr;
            let buffer = mem::replace(&mut *self.buffer, Vec::new());
            pool.release(buffer);
        }
    }
}

unsafe impl Send for BufferPool {}
```

사용 예제:

```rust
fn process_request(pool: &mut BufferPool) {
    if let Some(mut buf) = pool.acquire() {
        let buffer = buf.as_mut();
        buffer.extend_from_slice(b"HTTP/1.1 200 OK\r\n");
        buffer.extend_from_slice(b"Content-Length: 13\r\n\r\n");
        buffer.extend_from_slice(b"Hello, World!");
        
        // 여기서 네트워크로 전송...
        println!("버퍼 사용: {} bytes", buffer.len());
        
        // buf가 스코프를 벗어나면 자동으로 풀로 반환된다
    }
}
```

이 패턴은 강력하다. 사용자는 안전한 API만 사용하고, Drop trait이 자동으로 리소스를 정리하므로 누수 걱정이 없다.

## FFI와 빌림의 상호작용

**라이프타임과 FFI:**

FFI를 사용할 때 가장 까다로운 부분은 Rust의 라이프타임 시스템과 C의 포인터를 조화시키는 것이다. C 함수는 포인터가 얼마나 오래 유효한지 알 수 없으므로, 프로그래머가 직접 관리해야 한다.

```rust
use std::ffi::{CString, CStr};
use std::os::raw::c_char;

extern "C" {
    // C 함수가 문자열을 저장한다고 가정
    fn store_string(s: *const c_char);
    fn get_stored_string() -> *const c_char;
}

// 위험한 예제 - 하지 말 것!
fn dangerous_pattern() {
    let s = CString::new("temporary").unwrap();
    unsafe {
        store_string(s.as_ptr());
        // s가 여기서 drop되면 C 쪽 포인터는 댕글링 포인터가 된다!
    }
}

// 안전한 패턴
fn safe_pattern() -> Result<String, std::str::Utf8Error> {
    // CString을 Box로 감싸서 소유권 유지
    let s = Box::new(CString::new("persistent").unwrap());
    let ptr = s.as_ptr();
    
    unsafe {
        store_string(ptr);
        // s를 leak하여 메모리를 영구적으로 유지
        // (실제로는 나중에 정리하는 메커니즘이 필요)
        Box::leak(s);
        
        // 나중에 읽기
        let c_str = CStr::from_ptr(get_stored_string());
        c_str.to_str().map(|s| s.to_owned())
    }
}
```

**콜백 함수 전달하기:**

C 라이브러리에 Rust 클로저를 콜백으로 전달하는 것은 흔한 패턴이다. 하지만 클로저는 환경을 캡처하므로 조심해야 한다:

```rust
use std::os::raw::{c_int, c_void};

extern "C" {
    fn process_data(
        data: *const u8,
        len: usize,
        callback: extern "C" fn(*const u8, usize, *mut c_void),
        user_data: *mut c_void
    );
}

// C 호환 콜백 함수
extern "C" fn callback_wrapper(
    data: *const u8,
    len: usize,
    user_data: *mut c_void
) {
    unsafe {
        let slice = std::slice::from_raw_parts(data, len);
        let closure = &mut *(user_data as *mut &mut dyn FnMut(&[u8]));
        closure(slice);
    }
}

pub fn process_with_callback<F>(data: &[u8], mut callback: F)
where
    F: FnMut(&[u8])
{
    let mut closure: &mut dyn FnMut(&[u8]) = &mut callback;
    let user_data = &mut closure as *mut _ as *mut c_void;
    
    unsafe {
        process_data(
            data.as_ptr(),
            data.len(),
            callback_wrapper,
            user_data
        );
    }
}
```

## 실무에서의 사용 사례

**시스템 프로그래밍:**

운영체제와 직접 상호작용하는 코드를 작성할 때 unsafe가 필요하다. 예를 들어 메모리 매핑 파일을 사용하는 경우:

```rust
use std::fs::File;
use std::os::unix::io::AsRawFd;

#[cfg(unix)]
pub struct MemoryMappedFile {
    ptr: *mut u8,
    len: usize,
}

#[cfg(unix)]
impl MemoryMappedFile {
    pub fn open(path: &str) -> std::io::Result<Self> {
        use std::os::raw::c_void;
        
        let file = File::open(path)?;
        let len = file.metadata()?.len() as usize;
        
        let ptr = unsafe {
            let fd = file.as_raw_fd();
            let ptr = libc::mmap(
                std::ptr::null_mut(),
                len,
                libc::PROT_READ,
                libc::MAP_PRIVATE,
                fd,
                0
            );
            
            if ptr == libc::MAP_FAILED {
                return Err(std::io::Error::last_os_error());
            }
            
            ptr as *mut u8
        };
        
        Ok(MemoryMappedFile { ptr, len })
    }
    
    pub fn as_slice(&self) -> &[u8] {
        unsafe {
            std::slice::from_raw_parts(self.ptr, self.len)
        }
    }
}

#[cfg(unix)]
impl Drop for MemoryMappedFile {
    fn drop(&mut self) {
        unsafe {
            libc::munmap(self.ptr as *mut libc::c_void, self.len);
        }
    }
}
```

**성능 최적화:**

벡터화된 연산이나 SIMD 명령어를 사용할 때도 unsafe가 필요하다:

```rust
#[cfg(target_arch = "x86_64")]
use std::arch::x86_64::*;

pub fn sum_f32_simd(data: &[f32]) -> f32 {
    #[cfg(target_arch = "x86_64")]
    {
        if is_x86_feature_detected!("avx2") {
            return unsafe { sum_f32_avx2(data) };
        }
    }
    
    // 폴백: 일반 구현
    data.iter().sum()
}

#[cfg(target_arch = "x86_64")]
#[target_feature(enable = "avx2")]
unsafe fn sum_f32_avx2(data: &[f32]) -> f32 {
    let mut sum = _mm256_setzero_ps();
    let chunks = data.chunks_exact(8);
    let remainder = chunks.remainder();
    
    for chunk in chunks {
        let values = _mm256_loadu_ps(chunk.as_ptr());
        sum = _mm256_add_ps(sum, values);
    }
    
    // 수평 합산
    let mut result = [0f32; 8];
    _mm256_storeu_ps(result.as_mut_ptr(), sum);
    let total: f32 = result.iter().sum();
    
    // 나머지 처리
    total + remainder.iter().sum::<f32>()
}
```

## 디버깅과 테스팅

**unsafe 코드의 테스트:**

unsafe 코드는 특히 철저한 테스트가 필요하다. 컴파일러가 안전성을 보장하지 않으므로 프로그래머가 모든 경우를 검증해야 한다:

```rust
#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_ring_buffer_basic() {
        let mut rb = RingBuffer::new(10);
        
        // 쓰기
        let data = b"hello";
        assert_eq!(rb.write(data), 5);
        
        // 읽기
        let mut buf = [0u8; 5];
        assert_eq!(rb.read(&mut buf), 5);
        assert_eq!(&buf, b"hello");
    }
    
    #[test]
    fn test_ring_buffer_wraparound() {
        let mut rb = RingBuffer::new(5);
        
        // 거의 채우기
        assert_eq!(rb.write(b"abc"), 3);
        
        // 일부 읽기
        let mut buf = [0u8; 2];
        assert_eq!(rb.read(&mut buf), 2);
        assert_eq!(&buf, b"ab");
        
        // 다시 쓰기 (wrap around)
        assert_eq!(rb.write(b"de"), 2);
        
        // 모두 읽기
        let mut buf = [0u8; 3];
        assert_eq!(rb.read(&mut buf), 3);
        assert_eq!(&buf, b"cde");
    }
    
    #[test]
    fn test_buffer_pool() {
        let mut pool = BufferPool::new(2, 1024);
        
        {
            let mut buf1 = pool.acquire().unwrap();
            let mut buf2 = pool.acquire().unwrap();
            assert!(pool.acquire().is_none()); // 모두 사용중
            
            buf1.as_mut().extend_from_slice(b"test");
            buf2.as_mut().extend_from_slice(b"data");
        }
        
        // 반환되었으므로 다시 사용 가능
        let buf = pool.acquire().unwrap();
        assert_eq!(buf.as_mut().len(), 0); // 깨끗한 상태
    }
}
```

**Miri로 unsafe 검증하기:**

Miri는 Rust의 중간 표현을 해석하는 도구로, undefined behavior를 감지한다. unsafe 코드를 검증하는 데 매우 유용하다:

```bash
# Miri 설치
rustup component add miri

# 테스트 실행
cargo miri test
```

Miri는 다음과 같은 문제를 찾아낸다:

- 초기화되지 않은 메모리 읽기
- 해제된 메모리 접근
- 잘못된 정렬의 포인터 역참조
- 데이터 경쟁

## unsafe 사용 시 주의사항

**일반적인 함정들:**

1. **정렬 문제**: 포인터 캐스팅 시 정렬을 보장해야 한다. `std::ptr::read_unaligned`를 사용하거나 정렬을 확인한다.

2. **초기화되지 않은 메모리**: `MaybeUninit<T>`를 사용하여 명시적으로 관리한다.

3. **소유권 혼동**: `Box::from_raw`와 `Box::into_raw`를 쌍으로 사용해야 한다. 한쪽만 호출하면 메모리 누수나 이중 해제가 발생한다.

4. **C 문자열 관리**: `CString`이 스코프를 벗어나면 메모리가 해제되므로, C 함수에 전달한 포인터가 무효화된다.

```rust
use std::mem::MaybeUninit;

// 올바른 패턴: 초기화되지 않은 배열
fn create_initialized_array() -> [i32; 1000] {
    let mut arr: [MaybeUninit<i32>; 1000] = unsafe {
        MaybeUninit::uninit().assume_init()
    };
    
    for elem in &mut arr[..] {
        elem.write(0);
    }
    
    unsafe {
        // 모두 초기화되었음을 보장
        std::mem::transmute(arr)
    }
}
```

## 안전한 대안 고려하기

**unsafe 없이 해결하기:**

많은 경우 unsafe 없이도 동일한 기능을 구현할 수 있다. 성능 차이가 미미하다면 안전한 방법을 선택한다:

```rust
// unsafe 사용
fn sum_unsafe(data: &[i32]) -> i32 {
    let mut sum = 0;
    unsafe {
        let ptr = data.as_ptr();
        for i in 0..data.len() {
            sum += *ptr.add(i);
        }
    }
    sum
}

// 안전한 방법 (거의 동일한 성능)
fn sum_safe(data: &[i32]) -> i32 {
    data.iter().sum()
}
```

**크레이트 활용하기:**

실무에서는 검증된 크레이트를 사용하는 것이 좋다. 예를 들어:

- `libc`: 시스템 콜 래퍼
- `nix`: Unix API의 안전한 래퍼
- `winapi`: Windows API 바인딩
- `cc`: C 코드 컴파일 도구

이들은 이미 수많은 프로젝트에서 사용되고 검증되었다.

## 메모리 레이아웃 다이어그램

```
참조와 원시 포인터의 차이:

참조 (&T):
┌─────────────────────────────────┐
│ Rust 컴파일러가 관리            │
├─────────────────────────────────┤
│ ✓ 항상 유효함                   │
│ ✓ 빌림 규칙 적용                │
│ ✓ null 불가                     │
│ ✓ 자동 역참조                   │
└─────────────────────────────────┘

원시 포인터 (*const T, *mut T):
┌─────────────────────────────────┐
│ 프로그래머가 관리               │
├─────────────────────────────────┤
│ ✗ 유효성 보장 없음              │
│ ✗ 빌림 규칙 무시 가능           │
│ ✓ null 가능                     │
│ ✗ unsafe 블록 필요              │
└─────────────────────────────────┘

FFI 경계:
┌──────────────┐        ┌──────────────┐
│  Rust 코드   │◄──────►│   C 코드     │
├──────────────┤        ├──────────────┤
│ 소유권 있음  │        │ 소유권 없음  │
│ 빌림 규칙    │        │ 수동 관리    │
│ 안전 보장    │        │ 보장 없음    │
└──────────────┘        └──────────────┘
      ▲                        ▲
      │   extern "C" { }       │
      └────────────────────────┘
```

## 다음 단계

오늘 우리는 Rust의 안전망을 벗어나는 방법을 배웠다. unsafe는 강력하지만 책임이 따른다. 실무에서는 대부분의 코드를 안전한 Rust로 작성하고, 정말 필요한 경우에만 unsafe를 사용한다. 그리고 unsafe 코드는 항상 안전한 추상화 뒤에 숨겨서 사용자가 메모리 안전성을 걱정하지 않게 만든다.

내일은 이 책의 마지막 날로, Rust 생태계와 계속 학습하는 방법을 다룬다. 40일간의 여정을 마무리하며 실전에서 활용할 수 있는 리소스와 다음 단계를 살펴본다.

**오늘의 핵심:**
- unsafe는 컴파일러가 검증할 수 없는 작업을 허용한다
- 원시 포인터로 저수준 메모리 조작이 가능하다
- FFI로 C 라이브러리와 상호운용할 수 있다
- 안전한 추상화로 unsafe를 감싸는 것이 모범 사례다
- 철저한 테스트와 문서화가 필수다
