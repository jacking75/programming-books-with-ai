# Day 24: select!로 여러 작업 동시 대기

비동기 프로그래밍에서 여러 작업을 동시에 기다리다가 그중 하나가 먼저 완료되면 바로 반응해야 하는 상황이 자주 발생한다. 예를 들어 클라이언트로부터 메시지를 기다리면서 동시에 일정 시간이 지나면 타임아웃 처리를 해야 하거나, 여러 채널에서 오는 메시지를 동시에 받아야 하는 경우가 그렇다. 오늘은 tokio의 `select!` 매크로를 활용해서 이런 상황을 우아하게 처리하는 방법을 배운다.

## **select!가 필요한 이유**

지금까지 만든 채팅 서버를 생각해보면, 클라이언트와의 연결이 맺어진 후 계속해서 메시지를 읽어들인다. 하지만 실제 서비스에서는 클라이언트가 아무런 동작 없이 연결만 유지하는 경우가 있다. 이런 유휴(idle) 연결들이 쌓이면 서버 리소스가 낭비된다. 이를 방지하려면 "5분 동안 메시지가 없으면 연결을 끊는다"는 로직이 필요하다.

단순하게 생각하면 타임아웃 태스크와 메시지 수신 태스크를 따로 만들 수 있겠지만, 이렇게 하면 두 작업 간의 조율이 복잡해진다. `select!` 매크로는 이런 문제를 간결하게 해결해준다.

```
┌─────────────────────────────────────┐
│        select! 매크로               │
├─────────────────────────────────────┤
│                                     │
│  ┌──────────┐      ┌──────────┐   │
│  │  작업 A  │      │  작업 B  │   │
│  │ (메시지) │      │(타임아웃)│   │
│  └────┬─────┘      └─────┬────┘   │
│       │                  │         │
│       └────────┬─────────┘         │
│                ▼                    │
│        먼저 완료된 작업 처리        │
│                                     │
└─────────────────────────────────────┘
```

## **기본 select! 사용법**

먼저 간단한 예제로 `select!`의 작동 방식을 이해해보자. 두 개의 채널에서 메시지를 받는데, 어느 쪽이 먼저 도착하든 그것을 처리하는 코드다.

```rust
use tokio::sync::mpsc;
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    let (tx1, mut rx1) = mpsc::channel::<String>(10);
    let (tx2, mut rx2) = mpsc::channel::<String>(10);

    // 첫 번째 채널에 1초 후 메시지 전송
    tokio::spawn(async move {
        sleep(Duration::from_secs(1)).await;
        tx1.send("첫 번째 채널 메시지".to_string()).await.unwrap();
    });

    // 두 번째 채널에 2초 후 메시지 전송
    tokio::spawn(async move {
        sleep(Duration::from_secs(2)).await;
        tx2.send("두 번째 채널 메시지".to_string()).await.unwrap();
    });

    // 둘 중 먼저 도착하는 메시지 처리
    tokio::select! {
        Some(msg) = rx1.recv() => {
            println!("rx1에서 받음: {}", msg);
        }
        Some(msg) = rx2.recv() => {
            println!("rx2에서 받음: {}", msg);
        }
    }
}
```

이 코드를 실행하면 1초 후에 "rx1에서 받음: 첫 번째 채널 메시지"가 출력된다. `select!` 매크로는 두 개의 `recv()` 작업을 동시에 대기하다가 먼저 완료되는 것을 실행한다. 중요한 점은 한 번 실행되면 끝난다는 것이다. 만약 계속해서 메시지를 받고 싶다면 루프 안에서 `select!`를 사용해야 한다.

## **타임아웃 구현하기**

이제 실제로 유용한 패턴인 타임아웃을 구현해보자. 클라이언트로부터 메시지를 기다리되, 30초 동안 아무 메시지가 없으면 연결을 끊는 로직이다.

```rust
use tokio::io::{AsyncBufReadExt, BufReader};
use tokio::net::TcpStream;
use tokio::time::{sleep, Duration};

async fn handle_client(socket: TcpStream) {
    let mut reader = BufReader::new(socket);
    let mut line = String::new();

    loop {
        line.clear();

        tokio::select! {
            result = reader.read_line(&mut line) => {
                match result {
                    Ok(0) => {
                        println!("클라이언트 연결 종료");
                        break;
                    }
                    Ok(n) => {
                        println!("받은 메시지 ({} 바이트): {}", n, line.trim());
                        // 메시지를 받았으므로 타임아웃 리셋됨
                    }
                    Err(e) => {
                        eprintln!("읽기 에러: {}", e);
                        break;
                    }
                }
            }
            _ = sleep(Duration::from_secs(30)) => {
                println!("30초 동안 활동 없음, 연결 종료");
                break;
            }
        }
    }
}
```

이 코드의 핵심은 루프 안에서 `select!`를 사용한다는 점이다. 매 반복마다 두 가지 작업을 동시에 기다린다. 첫 번째는 `reader.read_line()`으로 클라이언트로부터 한 줄을 읽는 것이고, 두 번째는 `sleep(30초)`으로 타임아웃을 기다리는 것이다. 메시지가 도착하면 첫 번째 브랜치가 실행되고 루프가 다시 시작되면서 타임아웃이 리셋된다. 반대로 30초 동안 메시지가 없으면 두 번째 브랜치가 실행되어 루프를 빠져나온다.

여기서 빌림과 관련된 중요한 포인트가 있다. `line` 버퍼는 `read_line()`에 가변 참조로 전달된다. 하지만 `select!` 블록 전체가 끝날 때까지 이 참조가 유효해야 하므로, 다른 브랜치에서는 `line`을 사용할 수 없다. 만약 타임아웃 브랜치에서 `line`을 읽으려고 하면 컴파일 에러가 발생한다.

## **여러 채널 동시에 처리하기**

채팅 서버에서는 여러 소스로부터 메시지를 받아야 한다. 클라이언트로부터 오는 메시지, 다른 클라이언트들에게 브로드캐스트할 메시지, 그리고 서버 종료 신호 등이다. 이 모든 것을 `select!`로 우아하게 처리할 수 있다.

```rust
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use tokio::net::TcpStream;
use tokio::sync::mpsc;
use tokio::time::{sleep, Duration};

async fn handle_client_advanced(
    socket: TcpStream,
    mut broadcast_rx: mpsc::Receiver<String>,
    mut shutdown_rx: mpsc::Receiver<()>,
) {
    let (reader, mut writer) = socket.into_split();
    let mut reader = BufReader::new(reader);
    let mut line = String::new();

    loop {
        line.clear();

        tokio::select! {
            // 클라이언트로부터 메시지 읽기
            result = reader.read_line(&mut line) => {
                match result {
                    Ok(0) => {
                        println!("클라이언트 연결 종료");
                        break;
                    }
                    Ok(_) => {
                        println!("클라이언트: {}", line.trim());
                        // 여기서 브로드캐스트 채널로 전송할 수 있음
                    }
                    Err(e) => {
                        eprintln!("읽기 에러: {}", e);
                        break;
                    }
                }
            }

            // 다른 클라이언트로부터의 브로드캐스트 메시지
            Some(msg) = broadcast_rx.recv() => {
                if let Err(e) = writer.write_all(msg.as_bytes()).await {
                    eprintln!("쓰기 에러: {}", e);
                    break;
                }
            }

            // 서버 종료 신호
            Some(_) = shutdown_rx.recv() => {
                println!("종료 신호 받음, 연결 정리");
                let _ = writer.write_all(b"Server shutting down\n").await;
                break;
            }

            // 유휴 타임아웃
            _ = sleep(Duration::from_secs(300)) => {
                println!("5분 동안 활동 없음, 연결 종료");
                let _ = writer.write_all(b"Timeout\n").await;
                break;
            }
        }
    }
}
```

이 코드는 네 가지 작업을 동시에 대기한다. 각 브랜치는 독립적이지만 하나가 완료되면 그것을 처리하고 다시 루프로 돌아간다. 여기서 주목할 점은 소유권이 어떻게 관리되는가다.

`socket.into_split()`을 호출하면 소켓이 읽기용과 쓰기용으로 분리된다. 이렇게 하면 `reader`는 첫 번째 브랜치에서, `writer`는 다른 브랜치들에서 독립적으로 사용할 수 있다. 만약 분리하지 않았다면 하나의 소켓에 대해 동시에 여러 참조가 필요해서 빌림 검사기가 에러를 낸다.

채널 수신자들(`broadcast_rx`, `shutdown_rx`)은 함수 파라미터로 받아서 이동되었다. 이들은 루프 안에서 계속 사용되는데, `select!` 매크로가 내부적으로 각 브랜치를 퓨처로 만들 때 적절하게 참조를 관리해준다.

## **select! 내부 동작과 빌림**

`select!` 매크로가 내부적으로 어떻게 작동하는지 이해하면 빌림 관련 에러를 피하는 데 도움이 된다. 매크로는 각 브랜치의 퓨처를 동시에 폴링하다가 하나가 준비되면 그 브랜치를 실행한다. 이 과정에서 모든 브랜치가 동시에 살아있어야 하므로, 공유 데이터에 대한 빌림 규칙이 엄격하게 적용된다.

예를 들어 이런 코드는 컴파일되지 않는다:

```rust
// ❌ 컴파일 에러
let mut counter = 0;

tokio::select! {
    _ = async {
        counter += 1;  // 가변 빌림
    } => {}
    _ = async {
        counter += 1;  // 또 다른 가변 빌림
    } => {}
}
```

두 브랜치가 동시에 `counter`를 가변으로 빌리려고 하기 때문에 에러가 발생한다. 이런 상황에서는 공유 상태를 `Mutex`로 감싸거나, 각 브랜치가 독립적인 데이터를 다루도록 설계를 변경해야 한다.

반면 이렇게 각 브랜치가 다른 변수를 다루면 문제없다:

```rust
// ✅ 정상 작동
let mut counter1 = 0;
let mut counter2 = 0;

tokio::select! {
    _ = async {
        counter1 += 1;
    } => {}
    _ = async {
        counter2 += 1;
    } => {}
}
```

## **실전: 유휴 연결 관리 시스템**

이제 배운 내용을 종합해서 실제로 동작하는 채팅 서버에 유휴 연결 관리 기능을 추가해보자. 전체 구조를 먼저 살펴보겠다.

```
┌──────────────────────────────────────────────┐
│              메인 서버 루프                  │
│                                              │
│  ┌────────────────────────────────────────┐ │
│  │  새 연결 수락 → 클라이언트 핸들러 생성 │ │
│  └────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
                    │
                    ▼
┌──────────────────────────────────────────────┐
│         클라이언트 핸들러 (각 연결마다)      │
│                                              │
│  ┌────────────────────────────────────────┐ │
│  │           select! 루프                 │ │
│  │                                        │ │
│  │  ┌──────────────┐  ┌──────────────┐  │ │
│  │  │클라이언트 메시지│ │브로드캐스트  │  │ │
│  │  │     수신      │  │  메시지 수신 │  │ │
│  │  └──────────────┘  └──────────────┘  │ │
│  │                                        │ │
│  │  ┌──────────────┐  ┌──────────────┐  │ │
│  │  │  종료 신호   │  │ 타임아웃     │  │ │
│  │  │     수신      │  │  (5분)       │  │ │
│  │  └──────────────┘  └──────────────┘  │ │
│  └────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
```

전체 코드를 구현해보자:

```rust
use std::sync::Arc;
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use tokio::net::{TcpListener, TcpStream};
use tokio::sync::{broadcast, mpsc, Mutex};
use tokio::time::{sleep, Duration};

// 공유 상태: 브로드캐스트 채널
struct ChatServer {
    broadcast_tx: broadcast::Sender<String>,
    active_connections: Arc<Mutex<usize>>,
}

impl ChatServer {
    fn new() -> Self {
        let (broadcast_tx, _) = broadcast::channel(100);
        Self {
            broadcast_tx,
            active_connections: Arc::new(Mutex::new(0)),
        }
    }

    async fn handle_client(
        &self,
        socket: TcpStream,
        addr: std::net::SocketAddr,
        mut shutdown_rx: mpsc::Receiver<()>,
    ) {
        // 연결 수 증가
        {
            let mut count = self.active_connections.lock().await;
            *count += 1;
            println!("새 연결: {} (총 {}개)", addr, *count);
        }

        let (reader, mut writer) = socket.into_split();
        let mut reader = BufReader::new(reader);
        let mut line = String::new();
        let mut broadcast_rx = self.broadcast_tx.subscribe();

        // 환영 메시지
        let welcome = format!("채팅 서버에 오신 것을 환영한다 ({})\n", addr);
        let _ = writer.write_all(welcome.as_bytes()).await;

        loop {
            line.clear();

            tokio::select! {
                // 클라이언트로부터 메시지 읽기
                result = reader.read_line(&mut line) => {
                    match result {
                        Ok(0) => {
                            println!("{}: 연결 종료", addr);
                            break;
                        }
                        Ok(_) => {
                            let trimmed = line.trim();
                            if trimmed.is_empty() {
                                continue;
                            }

                            println!("{}: {}", addr, trimmed);

                            // 모든 클라이언트에게 브로드캐스트
                            let broadcast_msg = format!("{}: {}\n", addr, trimmed);
                            let _ = self.broadcast_tx.send(broadcast_msg);
                        }
                        Err(e) => {
                            eprintln!("{}: 읽기 에러 - {}", addr, e);
                            break;
                        }
                    }
                }

                // 브로드캐스트 메시지 받기
                result = broadcast_rx.recv() => {
                    match result {
                        Ok(msg) => {
                            // 자기 자신의 메시지는 제외하고 전송
                            if !msg.starts_with(&format!("{}:", addr)) {
                                if let Err(e) = writer.write_all(msg.as_bytes()).await {
                                    eprintln!("{}: 쓰기 에러 - {}", addr, e);
                                    break;
                                }
                            }
                        }
                        Err(e) => {
                            eprintln!("{}: 브로드캐스트 수신 에러 - {}", addr, e);
                            break;
                        }
                    }
                }

                // 서버 종료 신호
                _ = shutdown_rx.recv() => {
                    println!("{}: 종료 신호 받음", addr);
                    let _ = writer.write_all(b"서버가 종료된다\n").await;
                    break;
                }

                // 5분 유휴 타임아웃
                _ = sleep(Duration::from_secs(300)) => {
                    println!("{}: 5분 동안 활동 없음, 연결 종료", addr);
                    let _ = writer.write_all(b"타임아웃으로 연결이 종료된다\n").await;
                    break;
                }
            }
        }

        // 연결 수 감소
        {
            let mut count = self.active_connections.lock().await;
            *count -= 1;
            println!("{}: 연결 해제 (총 {}개)", addr, *count);
        }
    }
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("채팅 서버가 8080 포트에서 시작되었다");

    let server = Arc::new(ChatServer::new());
    let (shutdown_tx, _) = broadcast::channel::<()>(1);

    loop {
        let (socket, addr) = listener.accept().await?;
        let server = Arc::clone(&server);
        let shutdown_rx = shutdown_tx.subscribe();

        tokio::spawn(async move {
            server.handle_client(socket, addr, shutdown_rx).await;
        });
    }
}
```

이 구현에서 주목할 점들을 살펴보자.

첫째, `ChatServer` 구조체가 `Arc`로 감싸져서 여러 태스크 간에 공유된다. 내부의 `broadcast_tx`는 Clone이 가능하고 `active_connections`는 `Arc<Mutex<T>>`로 안전하게 공유된다.

둘째, 각 클라이언트 핸들러는 독립적인 `shutdown_rx`를 받는다. 이것은 `broadcast::channel`의 구독자인데, 이렇게 하면 메인 루프에서 `shutdown_tx.send(())`를 호출하면 모든 클라이언트가 동시에 종료 신호를 받을 수 있다.

셋째, `select!` 루프 안에서 소켓이 읽기와 쓰기로 분리되어 있다. `reader`는 첫 번째 브랜치에서만, `writer`는 나머지 브랜치에서 사용된다. 이렇게 하면 동시에 읽기와 쓰기가 가능하면서도 빌림 검사를 통과한다.

넷째, 타임아웃이 매 루프마다 리셋된다. 메시지를 받으면 루프가 다시 시작되면서 새로운 `sleep(300초)`가 시작된다. 이것은 idle timeout 패턴의 정석적인 구현 방식이다.

## **select!의 고급 기능들**

`select!` 매크로에는 몇 가지 유용한 고급 기능이 있다.

**biased 선택**: 기본적으로 `select!`는 랜덤하게 브랜치를 선택하지만, `biased;`를 추가하면 위에서부터 순서대로 확인한다.

```rust
tokio::select! {
    biased;
    
    // 이 브랜치가 준비되었다면 항상 먼저 실행
    _ = high_priority_task() => {
        println!("높은 우선순위");
    }
    
    _ = normal_task() => {
        println!("일반 우선순위");
    }
}
```

**조건부 브랜치**: `if` 조건을 붙여서 특정 상황에만 브랜치를 활성화할 수 있다.

```rust
let mut should_timeout = true;

tokio::select! {
    result = reader.read_line(&mut line) => {
        // 처리
    }
    
    _ = sleep(Duration::from_secs(30)), if should_timeout => {
        println!("타임아웃");
    }
}
```

**else 브랜치**: 모든 브랜치가 준비되지 않았을 때 즉시 실행되는 브랜치다. 이것은 논블로킹 동작을 구현할 때 유용하다.

```rust
tokio::select! {
    Some(msg) = rx.recv() => {
        println!("메시지: {}", msg);
    }
    else => {
        println!("메시지가 없다, 다른 작업 수행");
    }
}
```

## **디버깅과 모니터링**

실제 운영 환경에서는 `select!`로 처리되는 각 브랜치의 실행 빈도를 모니터링하는 것이 중요하다. 어느 브랜치가 얼마나 자주 실행되는지, 타임아웃이 얼마나 자주 발생하는지 등의 정보가 성능 튜닝에 도움이 된다.

```rust
use std::sync::atomic::{AtomicU64, Ordering};

struct Statistics {
    messages_received: AtomicU64,
    broadcasts_sent: AtomicU64,
    timeouts: AtomicU64,
    shutdowns: AtomicU64,
}

impl Statistics {
    fn new() -> Self {
        Self {
            messages_received: AtomicU64::new(0),
            broadcasts_sent: AtomicU64::new(0),
            timeouts: AtomicU64::new(0),
            shutdowns: AtomicU64::new(0),
        }
    }

    fn print(&self) {
        println!("=== 통계 ===");
        println!("수신 메시지: {}", self.messages_received.load(Ordering::Relaxed));
        println!("브로드캐스트: {}", self.broadcasts_sent.load(Ordering::Relaxed));
        println!("타임아웃: {}", self.timeouts.load(Ordering::Relaxed));
        println!("종료: {}", self.shutdowns.load(Ordering::Relaxed));
    }
}

// select! 루프 안에서
tokio::select! {
    result = reader.read_line(&mut line) => {
        stats.messages_received.fetch_add(1, Ordering::Relaxed);
        // ...
    }
    _ = sleep(Duration::from_secs(300)) => {
        stats.timeouts.fetch_add(1, Ordering::Relaxed);
        // ...
    }
}
```

`AtomicU64`를 사용하면 Mutex 없이도 여러 태스크에서 안전하게 카운터를 증가시킬 수 있다. 이것은 고성능 모니터링에 적합한 패턴이다.

## **실습 과제**

오늘 배운 내용을 바탕으로 다음 기능을 추가해보자:

1. 클라이언트가 "/ping" 명령을 보내면 "/pong"으로 응답하고, 이것을 타임아웃 리셋으로 간주하도록 수정한다.

2. 타임아웃 시간을 설정 파일에서 읽어오도록 변경한다. (예: 개발 환경에서는 30초, 프로덕션에서는 5분)

3. 통계 정보를 주기적으로 (예: 1분마다) 출력하는 별도의 태스크를 추가한다.

힌트: 통계 출력 태스크도 `select!`를 사용할 수 있다. 타이머와 종료 신호를 동시에 대기하면 된다.

```rust
tokio::spawn(async move {
    let mut interval = tokio::time::interval(Duration::from_secs(60));
    
    loop {
        tokio::select! {
            _ = interval.tick() => {
                stats.print();
            }
            _ = shutdown_rx.recv() => {
                println!("통계 태스크 종료");
                break;
            }
        }
    }
});
```

## **정리**

오늘은 `tokio::select!` 매크로를 통해 여러 비동기 작업을 동시에 대기하는 방법을 배웠다. 이것은 타임아웃 구현, 다중 채널 처리, 우아한 종료 등 실전에서 자주 필요한 패턴들을 간결하게 표현할 수 있게 해준다.

핵심 포인트를 정리하면 다음과 같다. `select!`는 여러 퓨처를 동시에 폴링하다가 하나가 준비되면 해당 브랜치를 실행한다. 루프 안에서 사용하면 계속해서 여러 소스를 모니터링할 수 있다. 빌림 규칙은 모든 브랜치에 걸쳐 적용되므로, 공유 데이터는 적절히 분리하거나 동기화 프리미티브를 사용해야 한다. 소켓을 읽기/쓰기로 분리하면 동시에 읽고 쓸 수 있다.

내일은 비동기 채널과 스트림을 더 깊이 다루면서, 오늘 만든 채팅 서버를 더욱 확장해보겠다. `tokio::sync` 모듈의 다양한 채널 타입들과 `futures::Stream` trait을 활용하면 더 복잡한 메시지 라우팅 패턴도 우아하게 구현할 수 있다.
