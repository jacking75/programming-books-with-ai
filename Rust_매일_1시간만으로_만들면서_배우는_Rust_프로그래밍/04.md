# Day 4: 구조체로 데이터 모델링하기
오늘은 Rust의 구조체(struct)를 사용해서 관련된 데이터를 하나로 묶고, 메서드를 추가해서 객체지향적인 코드를 작성하는 방법을 배운다. Day 3까지 만든 설정 파서를 구조체로 재구성하면서 코드의 구조와 가독성을 크게 개선한다.

## struct 정의와 사용
구조체는 여러 개의 관련된 값을 하나의 의미 있는 단위로 묶는 커스텀 데이터 타입이다. 서버 설정을 예로 들어보자. 지금까지는 개별 변수나 튜플로 데이터를 다뤘지만, 구조체를 사용하면 훨씬 명확하고 관리하기 쉬워진다.

```rust
// 구조체 정의
struct ServerConfig {
    host: String,
    port: u16,
    max_connections: u32,
    timeout_seconds: u32,
}

fn main() {
    // 구조체 인스턴스 생성
    let config = ServerConfig {
        host: String::from("localhost"),
        port: 8080,
        max_connections: 100,
        timeout_seconds: 30,
    };
    
    // 필드 접근
    println!("서버 주소: {}:{}", config.host, config.port);
    println!("최대 연결: {}", config.max_connections);
}
```

구조체의 각 요소를 필드(field)라고 부른다. 필드는 이름과 타입을 가지며, 점(`.`) 연산자로 접근한다. 구조체를 정의할 때는 `struct` 키워드를 사용하고, 중괄호 안에 필드들을 나열한다.

구조체도 기본적으로 불변이다. 필드를 수정하려면 인스턴스 전체를 `mut`로 선언해야 한다.

```rust
fn main() {
    let mut config = ServerConfig {
        host: String::from("localhost"),
        port: 8080,
        max_connections: 100,
        timeout_seconds: 30,
    };
    
    config.port = 9000;  // 가능하다
    config.timeout_seconds = 60;  // 가능하다
    
    println!("변경된 포트: {}", config.port);
}
```

Rust는 특정 필드만 가변으로 만들 수 없다. 구조체 전체가 가변이거나 불변이다. 이는 일관성을 유지하고 예측 가능한 코드를 작성하는 데 도움이 된다.

```
    ServerConfig
    ┌─────────────────────────┐
    │ host: String            │
    │ port: u16               │
    │ max_connections: u32    │
    │ timeout_seconds: u32    │
    └─────────────────────────┘
           하나의 단위로
           다루어진다
```

필드 초기화 단축 문법도 있다. 변수명과 필드명이 같으면 중복해서 쓰지 않아도 된다.

```rust
fn create_config(host: String, port: u16) -> ServerConfig {
    ServerConfig {
        host,  // host: host와 같다
        port,  // port: port와 같다
        max_connections: 100,
        timeout_seconds: 30,
    }
}
```

구조체 업데이트 문법을 사용하면 기존 인스턴스의 값을 대부분 재사용하면서 일부만 변경할 수 있다.

```rust
fn main() {
    let config1 = ServerConfig {
        host: String::from("localhost"),
        port: 8080,
        max_connections: 100,
        timeout_seconds: 30,
    };
    
    // config1의 값을 기반으로 port만 변경한 새 인스턴스
    let config2 = ServerConfig {
        port: 9000,
        ..config1  // 나머지 필드는 config1에서 가져온다
    };
    
    println!("config2 포트: {}", config2.port);
    // 주의: config1의 host가 이동되었으므로 config1은 더 이상 사용할 수 없다
}
```
  

## impl 블록으로 메서드 추가
구조체에 관련된 동작을 함수로 정의할 수 있다. `impl` 블록을 사용하면 구조체에 메서드를 추가할 수 있다. 메서드는 첫 번째 매개변수로 `self`를 받는다는 점만 빼면 일반 함수와 같다.

```rust
struct ServerConfig {
    host: String,
    port: u16,
    max_connections: u32,
    timeout_seconds: u32,
}

impl ServerConfig {
    // 연관 함수 (associated function) - self가 없다
    // 다른 언어의 static 메서드나 생성자와 비슷하다
    fn new(host: String, port: u16) -> Self {
        Self {
            host,
            port,
            max_connections: 100,
            timeout_seconds: 30,
        }
    }
    
    // 기본값으로 생성하는 연관 함수
    fn default() -> Self {
        Self {
            host: String::from("localhost"),
            port: 8080,
            max_connections: 100,
            timeout_seconds: 30,
        }
    }
    
    // 불변 참조를 받는 메서드
    fn address(&self) -> String {
        format!("{}:{}", self.host, self.port)
    }
    
    // 가변 참조를 받는 메서드
    fn set_port(&mut self, port: u16) {
        self.port = port;
    }
    
    // 값을 소비하는 메서드 (self를 받는다)
    fn validate(self) -> Result<Self, String> {
        if self.port == 0 {
            return Err("포트는 0일 수 없다".to_string());
        }
        if self.max_connections == 0 {
            return Err("최대 연결 수는 0일 수 없다".to_string());
        }
        Ok(self)
    }
}

fn main() {
    // 연관 함수는 ::로 호출한다
    let mut config = ServerConfig::new(
        String::from("0.0.0.0"),
        8080
    );
    
    // 메서드는 .로 호출한다
    println!("서버 주소: {}", config.address());
    
    config.set_port(9000);
    println!("변경된 주소: {}", config.address());
    
    // validate는 self를 소비하므로 이후 config를 사용할 수 없다
    match config.validate() {
        Ok(validated_config) => {
            println!("검증 완료: {}", validated_config.address());
        }
        Err(e) => println!("검증 실패: {}", e),
    }
}
```

메서드의 `self` 매개변수는 세 가지 형태로 받을 수 있다.

**불변 참조 `&self`:** 데이터를 읽기만 할 때 사용한다. 가장 흔하게 쓰인다. 호출 후에도 인스턴스를 계속 사용할 수 있다.

**가변 참조 `&mut self`:** 데이터를 수정할 때 사용한다. 호출 후에도 인스턴스를 계속 사용할 수 있다.

**값 `self`:** 소유권을 가져갈 때 사용한다. 주로 인스턴스를 변환하거나 소비할 때 쓴다. 호출 후 원래 인스턴스는 사용할 수 없다.

```
    메서드 호출 흐름
    
    config.address()
       │
       ├─> &self를 받는다
       │   (읽기만 한다)
       │
       └─> 호출 후 config 계속 사용 가능
    
    config.set_port(9000)
       │
       ├─> &mut self를 받는다
       │   (수정한다)
       │
       └─> 호출 후 config 계속 사용 가능
    
    config.validate()
       │
       ├─> self를 받는다
       │   (소유권을 가져간다)
       │
       └─> 호출 후 config 사용 불가
```

`Self` 타입은 구현하고 있는 타입의 별칭이다. `ServerConfig`를 `Self`로 쓸 수 있어서 타입명이 바뀌어도 코드를 수정할 곳이 줄어든다.
  

## 설정을 구조체로 표현하기
이제 Day 2와 Day 3에서 만든 설정 파서를 구조체로 재구성한다. 개별 키-값 쌍 대신 의미 있는 구조체로 데이터를 모델링하면 타입 안전성이 높아지고 코드가 명확해진다.

```rust
#[derive(Debug)]
enum ConfigError {
    ParseError(String),
    InvalidValue { key: String, value: String, reason: String },
    MissingRequired(String),
}

struct AppConfig {
    server: ServerConfig,
    database: DatabaseConfig,
}

struct ServerConfig {
    host: String,
    port: u16,
    max_connections: u32,
    timeout_seconds: u32,
}

struct DatabaseConfig {
    connection_string: String,
    pool_size: u32,
}

impl AppConfig {
    fn from_text(text: &str) -> Result<Self, ConfigError> {
        let mut host = None;
        let mut port = None;
        let mut max_connections = None;
        let mut timeout_seconds = None;
        let mut connection_string = None;
        let mut pool_size = None;
        
        for line in text.lines() {
            let line = line.trim();
            
            // 빈 줄이나 주석 건너뛰기
            if line.is_empty() || line.starts_with('#') {
                continue;
            }
            
            // 키=값 파싱
            let parts: Vec<&str> = line.split('=').collect();
            if parts.len() != 2 {
                return Err(ConfigError::ParseError(line.to_string()));
            }
            
            let key = parts[0].trim();
            let value = parts[1].trim();
            
            // 각 키에 따라 적절히 파싱
            match key {
                "host" => host = Some(value.to_string()),
                "port" => {
                    port = Some(value.parse::<u16>().map_err(|_| {
                        ConfigError::InvalidValue {
                            key: key.to_string(),
                            value: value.to_string(),
                            reason: "유효한 포트 번호가 아니다".to_string(),
                        }
                    })?);
                }
                "max_connections" => {
                    max_connections = Some(value.parse::<u32>().map_err(|_| {
                        ConfigError::InvalidValue {
                            key: key.to_string(),
                            value: value.to_string(),
                            reason: "유효한 숫자가 아니다".to_string(),
                        }
                    })?);
                }
                "timeout_seconds" => {
                    timeout_seconds = Some(value.parse::<u32>().map_err(|_| {
                        ConfigError::InvalidValue {
                            key: key.to_string(),
                            value: value.to_string(),
                            reason: "유효한 숫자가 아니다".to_string(),
                        }
                    })?);
                }
                "db_connection" => connection_string = Some(value.to_string()),
                "db_pool_size" => {
                    pool_size = Some(value.parse::<u32>().map_err(|_| {
                        ConfigError::InvalidValue {
                            key: key.to_string(),
                            value: value.to_string(),
                            reason: "유효한 숫자가 아니다".to_string(),
                        }
                    })?);
                }
                _ => {
                    // 알 수 없는 키는 무시한다
                    println!("경고: 알 수 없는 설정 키 '{}'", key);
                }
            }
        }
        
        // 필수 값 확인 및 기본값 적용
        let server = ServerConfig {
            host: host.unwrap_or_else(|| "localhost".to_string()),
            port: port.ok_or_else(|| ConfigError::MissingRequired("port".to_string()))?,
            max_connections: max_connections.unwrap_or(100),
            timeout_seconds: timeout_seconds.unwrap_or(30),
        };
        
        let database = DatabaseConfig {
            connection_string: connection_string.ok_or_else(|| {
                ConfigError::MissingRequired("db_connection".to_string())
            })?,
            pool_size: pool_size.unwrap_or(10),
        };
        
        Ok(AppConfig { server, database })
    }
}

impl ServerConfig {
    fn address(&self) -> String {
        format!("{}:{}", self.host, self.port)
    }
    
    fn validate(&self) -> Result<(), ConfigError> {
        if self.port == 0 {
            return Err(ConfigError::InvalidValue {
                key: "port".to_string(),
                value: self.port.to_string(),
                reason: "포트는 0일 수 없다".to_string(),
            });
        }
        
        if self.max_connections == 0 {
            return Err(ConfigError::InvalidValue {
                key: "max_connections".to_string(),
                value: self.max_connections.to_string(),
                reason: "최대 연결 수는 0보다 커야 한다".to_string(),
            });
        }
        
        Ok(())
    }
}

impl DatabaseConfig {
    fn validate(&self) -> Result<(), ConfigError> {
        if self.connection_string.is_empty() {
            return Err(ConfigError::InvalidValue {
                key: "db_connection".to_string(),
                value: String::new(),
                reason: "연결 문자열이 비어있다".to_string(),
            });
        }
        
        if self.pool_size == 0 {
            return Err(ConfigError::InvalidValue {
                key: "db_pool_size".to_string(),
                value: self.pool_size.to_string(),
                reason: "풀 크기는 0보다 커야 한다".to_string(),
            });
        }
        
        Ok(())
    }
}

fn main() {
    let config_text = r#"
# 서버 설정
host=0.0.0.0
port=8080
max_connections=200
timeout_seconds=60

# 데이터베이스 설정
db_connection=postgresql://localhost/mydb
db_pool_size=20
"#;
    
    match AppConfig::from_text(config_text) {
        Ok(config) => {
            println!("설정 로드 성공!");
            println!("서버 주소: {}", config.server.address());
            println!("최대 연결: {}", config.server.max_connections);
            println!("DB 풀 크기: {}", config.database.pool_size);
            
            // 유효성 검증
            if let Err(e) = config.server.validate() {
                println!("서버 설정 오류: {:?}", e);
            }
            if let Err(e) = config.database.validate() {
                println!("데이터베이스 설정 오류: {:?}", e);
            }
        }
        Err(e) => {
            println!("설정 로드 실패: {:?}", e);
        }
    }
}
```

이 코드에서 주목할 점은 설정을 계층적으로 구조화했다는 것이다. `AppConfig`는 `ServerConfig`와 `DatabaseConfig`를 포함한다. 이렇게 하면 관련된 설정을 그룹화하고, 각각 독립적으로 검증할 수 있다.

```
         AppConfig
            │
     ┌──────┴──────┐
     │             │
ServerConfig  DatabaseConfig
     │             │
  ┌──┴──┐       ┌──┴──┐
 host port    conn  pool
 max  timeout  string size
```

`Option`을 활용해서 선택적 값과 기본값을 우아하게 처리했다. 파싱 중에는 `Option`으로 저장하고, 마지막에 `unwrap_or`로 기본값을 적용하거나 `ok_or_else`로 필수 값을 확인한다.
  

## Debug trait으로 출력 개선
구조체를 `println!`로 출력하려고 하면 기본적으로 에러가 발생한다. Rust는 어떻게 출력할지 모르기 때문이다. `Debug` trait을 구현하면 디버깅용 출력이 가능하다.

```rust
// 수동으로 Debug를 구현하는 방법
impl std::fmt::Debug for ServerConfig {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        f.debug_struct("ServerConfig")
            .field("host", &self.host)
            .field("port", &self.port)
            .field("max_connections", &self.max_connections)
            .field("timeout_seconds", &self.timeout_seconds)
            .finish()
    }
}
```

하지만 대부분의 경우 자동 구현으로 충분하다. `#[derive(Debug)]` 속성을 추가하면 컴파일러가 자동으로 `Debug`를 구현해준다.

```rust
#[derive(Debug)]
struct ServerConfig {
    host: String,
    port: u16,
    max_connections: u32,
    timeout_seconds: u32,
}

#[derive(Debug)]
struct DatabaseConfig {
    connection_string: String,
    pool_size: u32,
}

#[derive(Debug)]
struct AppConfig {
    server: ServerConfig,
    database: DatabaseConfig,
}

fn main() {
    let config = AppConfig {
        server: ServerConfig {
            host: String::from("localhost"),
            port: 8080,
            max_connections: 100,
            timeout_seconds: 30,
        },
        database: DatabaseConfig {
            connection_string: String::from("postgresql://localhost/mydb"),
            pool_size: 10,
        },
    };
    
    // {:?}로 한 줄에 출력
    println!("{:?}", config);
    
    // {:#?}로 예쁘게 정렬해서 출력
    println!("{:#?}", config);
}
```

`{:#?}` 형식은 중첩된 구조체를 들여쓰기와 함께 출력해서 가독성이 좋다.

```
AppConfig {
    server: ServerConfig {
        host: "localhost",
        port: 8080,
        max_connections: 100,
        timeout_seconds: 30,
    },
    database: DatabaseConfig {
        connection_string: "postgresql://localhost/mydb",
        pool_size: 10,
    },
}
```

`dbg!` 매크로는 디버깅을 더욱 쉽게 만들어준다. 표현식의 값과 파일명, 라인 번호를 출력하고 값을 반환한다.

```rust
fn main() {
    let config = AppConfig::from_text("port=8080\ndb_connection=test").unwrap();
    
    // dbg!는 값을 출력하고 소유권을 반환한다
    let port = dbg!(config.server.port);
    
    // 참조로 전달하면 원본을 유지할 수 있다
    dbg!(&config.server);
    
    // 여전히 config를 사용할 수 있다
    println!("포트: {}", config.server.port);
}
```
  

## 빌더 패턴으로 가독성 높이기
설정 항목이 많아지면 구조체 초기화가 복잡해진다. 빌더 패턴을 사용하면 유창한(fluent) API로 인스턴스를 생성할 수 있다.

```rust
#[derive(Debug)]
struct ServerConfig {
    host: String,
    port: u16,
    max_connections: u32,
    timeout_seconds: u32,
}

struct ServerConfigBuilder {
    host: String,
    port: u16,
    max_connections: u32,
    timeout_seconds: u32,
}

impl ServerConfigBuilder {
    fn new() -> Self {
        Self {
            host: String::from("localhost"),
            port: 8080,
            max_connections: 100,
            timeout_seconds: 30,
        }
    }
    
    fn host(mut self, host: String) -> Self {
        self.host = host;
        self
    }
    
    fn port(mut self, port: u16) -> Self {
        self.port = port;
        self
    }
    
    fn max_connections(mut self, max_connections: u32) -> Self {
        self.max_connections = max_connections;
        self
    }
    
    fn timeout_seconds(mut self, timeout_seconds: u32) -> Self {
        self.timeout_seconds = timeout_seconds;
        self
    }
    
    fn build(self) -> Result<ServerConfig, ConfigError> {
        if self.port == 0 {
            return Err(ConfigError::InvalidValue {
                key: "port".to_string(),
                value: self.port.to_string(),
                reason: "포트는 0일 수 없다".to_string(),
            });
        }
        
        Ok(ServerConfig {
            host: self.host,
            port: self.port,
            max_connections: self.max_connections,
            timeout_seconds: self.timeout_seconds,
        })
    }
}

fn main() {
    let config = ServerConfigBuilder::new()
        .host(String::from("0.0.0.0"))
        .port(9000)
        .max_connections(200)
        .build()
        .unwrap();
    
    println!("{:#?}", config);
}
```

빌더 패턴에서 각 메서드는 `self`를 받아서 수정한 후 다시 반환한다. 이렇게 하면 메서드 호출을 체이닝할 수 있다. `build()` 메서드에서 최종 검증을 수행하고 실제 구조체를 생성한다.

```
    new()
     │
     v
  Builder (기본값)
```

## 튜플 구조체와 유닛 구조체
Rust는 몇 가지 특별한 형태의 구조체도 제공한다. 튜플 구조체는 필드명 없이 타입만 정의한다. 의미를 부여하기 위해 타입을 감쌀 때 유용하다.

```rust
struct Port(u16);
struct IpAddress(String);

impl Port {
    fn new(port: u16) -> Result<Self, String> {
        if port == 0 {
            return Err("포트는 0일 수 없다".to_string());
        }
        Ok(Port(port))
    }
    
    fn value(&self) -> u16 {
        self.0  // 첫 번째 필드는 .0으로 접근한다
    }
}

fn main() {
    let port = Port::new(8080).unwrap();
    println!("포트: {}", port.value());
    
    // u16과 Port는 서로 다른 타입이다
    let raw_port: u16 = 8080;
    // let p: Port = raw_port;  // 컴파일 에러! 타입이 다르다
}
```

튜플 구조체는 타입 안전성을 높인다. `Port`와 일반 `u16`을 실수로 섞어 쓸 수 없다. 이를 뉴타입 패턴(newtype pattern)이라고 부른다.

유닛 구조체는 필드가 전혀 없는 구조체다. 주로 trait을 구현하기 위한 타입으로 사용한다.

```rust
struct DatabaseConnection;

impl DatabaseConnection {
    fn connect() -> Self {
        println!("데이터베이스 연결 중...");
        DatabaseConnection
    }
    
    fn query(&self, sql: &str) {
        println!("쿼리 실행: {}", sql);
    }
}

fn main() {
    let db = DatabaseConnection::connect();
    db.query("SELECT * FROM users");
}
```
  

## 실전 예제: 로그 엔트리 파서
배운 내용을 종합해서 로그 파일의 엔트리를 파싱하는 구조체를 만들어본다. 앞으로 Day 6-10에서 로그 처리 시스템을 계속 발전시킬 것이다.

```rust
#[derive(Debug)]
enum LogLevel {
    Debug,
    Info,
    Warning,
    Error,
}

impl LogLevel {
    fn from_str(s: &str) -> Option<Self> {
        match s.to_uppercase().as_str() {
            "DEBUG" => Some(LogLevel::Debug),
            "INFO" => Some(LogLevel::Info),
            "WARNING" | "WARN" => Some(LogLevel::Warning),
            "ERROR" => Some(LogLevel::Error),
            _ => None,
        }
    }
}

#[derive(Debug)]
struct LogEntry {
    timestamp: String,
    level: LogLevel,
    message: String,
}

impl LogEntry {
    fn parse(line: &str) -> Option<Self> {
        // 형식: [2024-01-01 10:30:45] INFO: 서버 시작
        let line = line.trim();
        
        // 타임스탬프 추출
        if !line.starts_with('[') {
            return None;
        }
        
        let close_bracket = line.find(']')?;
        let timestamp = line[1..close_bracket].to_string();
        
        // 나머지 부분 파싱
        let rest = line[close_bracket + 1..].trim();
        let parts: Vec<&str> = rest.splitn(2, ':').collect();
        
        if parts.len() != 2 {
            return None;
        }
        
        let level = LogLevel::from_str(parts[0].trim())?;
        let message = parts[1].trim().to_string();
        
        Some(LogEntry {
            timestamp,
            level,
            message,
        })
    }
    
    fn is_error(&self) -> bool {
        matches!(self.level, LogLevel::Error)
    }
    
    fn is_warning_or_error(&self) -> bool {
        matches!(self.level, LogLevel::Warning | LogLevel::Error)
    }
}

fn main() {
    let log_lines = vec![
        "[2024-01-01 10:30:45] INFO: 서버 시작",
        "[2024-01-01 10:30:46] DEBUG: 설정 로드 완료",
        "[2024-01-01 10:30:50] WARNING: 연결 시도 실패, 재시도 중",
        "[2024-01-01 10:31:00] ERROR: 데이터베이스 연결 실패",
        "잘못된 형식의 로그",
    ];
    
    for line in log_lines {
        match LogEntry::parse(line) {
            Some(entry) => {
                if entry.is_warning_or_error() {
                    println!("⚠️  {:#?}", entry);
                } else {
                    println!("{:?}", entry);
                }
            }
            None => {
                println!("파싱 실패: {}", line);
            }
        }
    }
}
```

이 예제는 실제 로그 파싱의 단순화된 버전이다. 구조체를 사용해서 로그 엔트리를 명확한 타입으로 표현하고, 메서드로 의미 있는 동작을 제공한다. `matches!` 매크로는 패턴 매칭을 간결하게 만들어준다.
  

## 정리하며
오늘은 구조체로 데이터를 모델링하는 방법을 배웠다. 관련된 데이터를 하나로 묶고, 메서드를 추가해서 객체지향적인 코드를 작성할 수 있다.

핵심 포인트를 정리하면 다음과 같다.

구조체는 `struct` 키워드로 정의하고 여러 필드를 가질 수 있다. 각 필드는 이름과 타입을 명시한다.

`impl` 블록으로 구조체에 메서드를 추가한다. `self`, `&self`, `&mut self` 중 적절한 형태를 선택해서 소유권과 가변성을 제어한다.

연관 함수는 `self`를 받지 않으며 `::` 연산자로 호출한다. 주로 생성자나 팩토리 함수로 사용한다.

`#[derive(Debug)]`를 추가하면 디버깅 출력을 자동으로 구현할 수 있다. `{:?}`와 `{:#?}` 형식 지정자로 구조체를 출력한다.

빌더 패턴은 복잡한 구조체 생성을 가독성 좋게 만들어준다. 메서드 체이닝으로 유창한 API를 제공한다.

튜플 구조체는 타입 안전성을 높이는 뉴타입 패턴에 유용하다. 필드명 없이 타입만 정의한다.

내일은 컬렉션을 배운다. `Vec`, `String`, `HashMap` 같은 동적 데이터 구조로 설정 파서를 더욱 유연하게 만든다. 구조체와 컬렉션을 조합하면 복잡한 데이터도 효과적으로 다룰 수 있다.
