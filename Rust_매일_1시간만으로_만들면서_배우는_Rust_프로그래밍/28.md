# Day 28: 테스트 작성하기

## 실전 시나리오: 신뢰할 수 있는 코드 만들기

지금까지 우리는 TCP 서버, 비동기 프로그래밍, 설정 관리 시스템을 구축했다. 코드가 점점 복잡해지면서 한 가지 질문이 생긴다. "이 코드가 정말 제대로 작동할까?" 수동으로 서버를 실행하고 텔넷으로 연결해서 테스트하는 것은 시간이 오래 걸리고 실수하기 쉽다. 더 나쁜 것은 코드를 수정할 때마다 모든 기능이 여전히 작동하는지 확인하기 어렵다는 점이다.

Rust는 테스트를 언어와 도구 체인에 깊이 통합했다. 테스트 작성이 간단하고, 실행이 빠르며, 실패 원인을 명확하게 보여준다. 오늘은 우리가 만든 프로토콜 파서와 설정 시스템에 테스트를 작성하면서, 테스트가 어떻게 코드의 품질을 높이고 리팩토링을 안전하게 만드는지 경험한다.

## 첫 단위 테스트 작성하기

Day 13에서 만든 프로토콜 파서부터 시작한다. 먼저 테스트할 코드를 간단히 복습한다.

```rust
// src/protocol.rs
#[derive(Debug, PartialEq)]
pub enum Message {
    Ping,
    Pong,
    Echo(String),
    Unknown(String),
}

impl Message {
    pub fn parse(input: &str) -> Self {
        let trimmed = input.trim();
        
        if trimmed.eq_ignore_ascii_case("PING") {
            Message::Ping
        } else if trimmed.eq_ignore_ascii_case("PONG") {
            Message::Pong
        } else if let Some(content) = trimmed.strip_prefix("ECHO ") {
            Message::Echo(content.to_string())
        } else {
            Message::Unknown(trimmed.to_string())
        }
    }
    
    pub fn to_string(&self) -> String {
        match self {
            Message::Ping => "PING".to_string(),
            Message::Pong => "PONG".to_string(),
            Message::Echo(content) => format!("ECHO {}", content),
            Message::Unknown(content) => content.clone(),
        }
    }
}
```

이제 이 코드가 제대로 작동하는지 테스트를 작성한다. Rust에서는 테스트를 같은 파일에 작성하는 것이 일반적이다.

```rust
// src/protocol.rs (계속)

#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_parse_ping() {
        let msg = Message::parse("PING");
        assert_eq!(msg, Message::Ping);
    }
    
    #[test]
    fn test_parse_ping_case_insensitive() {
        assert_eq!(Message::parse("ping"), Message::Ping);
        assert_eq!(Message::parse("PiNg"), Message::Ping);
        assert_eq!(Message::parse("PING"), Message::Ping);
    }
    
    #[test]
    fn test_parse_ping_with_whitespace() {
        assert_eq!(Message::parse("  PING  "), Message::Ping);
        assert_eq!(Message::parse("\tPING\n"), Message::Ping);
    }
    
    #[test]
    fn test_parse_pong() {
        let msg = Message::parse("PONG");
        assert_eq!(msg, Message::Pong);
    }
    
    #[test]
    fn test_parse_echo() {
        let msg = Message::parse("ECHO hello world");
        assert_eq!(msg, Message::Echo("hello world".to_string()));
    }
    
    #[test]
    fn test_parse_echo_empty() {
        let msg = Message::parse("ECHO ");
        assert_eq!(msg, Message::Echo("".to_string()));
    }
    
    #[test]
    fn test_parse_unknown() {
        let msg = Message::parse("INVALID COMMAND");
        assert_eq!(msg, Message::Unknown("INVALID COMMAND".to_string()));
    }
}
```

여기서 중요한 부분들을 살펴본다.

**`#[cfg(test)]`**: 이 애트리뷰트는 테스트를 빌드할 때만 이 모듈을 컴파일하라고 지시한다. 일반 빌드에서는 테스트 코드가 포함되지 않아 바이너리 크기가 작아진다.

**`mod tests`**: 테스트를 별도 모듈로 분리한다. 이렇게 하면 테스트 코드가 깔끔하게 정리된다.

**`use super::*`**: 부모 모듈의 모든 항목을 가져온다. 테스트에서 `Message`를 사용하기 위해 필요하다.

**`#[test]`**: 이 함수가 테스트라는 것을 표시한다. `cargo test`를 실행하면 이 함수들이 자동으로 실행된다.

테스트를 실행해본다.

```bash
cargo test
```

출력:

```
running 7 tests
test protocol::tests::test_parse_echo ... ok
test protocol::tests::test_parse_echo_empty ... ok
test protocol::tests::test_parse_ping ... ok
test protocol::tests::test_parse_ping_case_insensitive ... ok
test protocol::tests::test_parse_ping_with_whitespace ... ok
test protocol::tests::test_parse_pong ... ok
test protocol::tests::test_parse_unknown ... ok

test result: ok. 7 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

모든 테스트가 통과했다. 각 테스트는 독립적으로 실행되며, 하나가 실패해도 다른 테스트는 계속 실행된다.

## assert 매크로 패밀리

Rust는 다양한 assert 매크로를 제공한다. 각각 언제 사용하는지 알아본다.

```rust
#[cfg(test)]
mod assert_examples {
    use super::*;
    
    #[test]
    fn test_assert_eq() {
        // 두 값이 같은지 확인 (가장 흔히 사용)
        let result = 2 + 2;
        assert_eq!(result, 4);
    }
    
    #[test]
    fn test_assert_ne() {
        // 두 값이 다른지 확인
        let result = 2 + 2;
        assert_ne!(result, 5);
    }
    
    #[test]
    fn test_assert() {
        // 조건이 참인지 확인
        let is_valid = true;
        assert!(is_valid);
        
        // 복잡한 조건도 가능
        let value = 42;
        assert!(value > 0 && value < 100);
    }
    
    #[test]
    fn test_assert_with_message() {
        // 실패 시 커스텀 메시지 출력
        let result = calculate_something();
        assert_eq!(result, 42, "계산 결과가 예상과 다름: {}", result);
    }
    
    #[test]
    #[should_panic]
    fn test_should_panic() {
        // 이 테스트는 패닉이 발생해야 통과
        panic!("의도적인 패닉");
    }
    
    #[test]
    #[should_panic(expected = "overflow")]
    fn test_should_panic_with_message() {
        // 특정 메시지를 포함한 패닉이 발생해야 통과
        panic!("integer overflow detected");
    }
}

fn calculate_something() -> i32 {
    42
}
```

실제로 실패하는 테스트를 보면 어떤 정보를 제공하는지 알 수 있다.

```rust
#[test]
fn test_deliberately_failing() {
    let msg = Message::parse("ECHO test");
    assert_eq!(msg, Message::Ping); // 의도적으로 잘못된 기대값
}
```

실행하면:

```
---- protocol::tests::test_deliberately_failing stdout ----
thread 'protocol::tests::test_deliberately_failing' panicked at 'assertion failed: `(left == right)`
  left: `Echo("test")`,
 right: `Ping`', src/protocol.rs:XX:X
```

좌변과 우변을 명확하게 보여주므로 무엇이 잘못되었는지 즉시 알 수 있다. 이것이 `assert_eq!`가 단순한 `assert!(a == b)`보다 선호되는 이유다.

## Result를 반환하는 테스트

에러 처리를 하는 코드를 테스트할 때는 테스트 함수가 `Result`를 반환하게 할 수 있다. `?` 연산자를 사용해서 간결하게 작성할 수 있다.

```rust
// src/config.rs에서 설정 파싱 테스트
#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_config_from_valid_json() -> Result<(), Box<dyn std::error::Error>> {
        let json = r#"
        {
            "server": {
                "host": "127.0.0.1",
                "port": 8080,
                "max_connections": 100
            },
            "timeout": {
                "idle_seconds": 300,
                "shutdown_seconds": 10
            },
            "logging": {
                "level": "info",
                "format": "json"
            }
        }
        "#;
        
        let config: ServerConfig = serde_json::from_str(json)?;
        
        assert_eq!(config.server.host, "127.0.0.1");
        assert_eq!(config.server.port, 8080);
        assert_eq!(config.timeout.idle_seconds, 300);
        assert_eq!(config.logging.level, "info");
        
        Ok(())
    }
    
    #[test]
    fn test_config_from_invalid_json() {
        let json = r#"{ "invalid": json }"#;
        let result: Result<ServerConfig, _> = serde_json::from_str(json);
        
        assert!(result.is_err(), "잘못된 JSON은 에러를 반환해야 함");
    }
    
    #[test]
    fn test_config_validation_fails_on_zero_max_connections() {
        let mut config = ServerConfig::default();
        config.server.max_connections = 0;
        
        let result = config.validate();
        assert!(result.is_err());
        
        // 에러 메시지 확인
        if let Err(e) = result {
            let error_msg = e.to_string();
            assert!(error_msg.contains("최대 연결 수"));
        }
    }
    
    #[test]
    fn test_config_validation_fails_on_invalid_log_level() {
        let mut config = ServerConfig::default();
        config.logging.level = "invalid_level".to_string();
        
        match config.validate() {
            Err(ConfigError::ValidationError(msg)) => {
                assert!(msg.contains("잘못된 로그 레벨"));
            }
            _ => panic!("ValidationError가 발생해야 함"),
        }
    }
}
```

여기서 주목할 점은 에러 케이스도 적극적으로 테스트한다는 것이다. 올바른 입력뿐만 아니라 잘못된 입력에 대해서도 코드가 예상대로 동작하는지 확인한다.

## 테스트 헬퍼 함수와 픽스처

테스트에서 반복되는 설정 코드는 헬퍼 함수로 추출한다. 이렇게 하면 테스트가 더 읽기 쉬워진다.

```rust
#[cfg(test)]
mod tests {
    use super::*;
    
    // 테스트용 설정 생성 헬퍼
    fn create_test_config() -> ServerConfig {
        ServerConfig {
            server: NetworkConfig {
                host: "127.0.0.1".to_string(),
                port: 9999,
                max_connections: 10,
            },
            timeout: TimeoutConfig {
                idle_seconds: 60,
                shutdown_seconds: 5,
            },
            logging: LoggingConfig {
                level: "debug".to_string(),
                format: "text".to_string(),
            },
        }
    }
    
    // 임시 설정 파일 생성 헬퍼
    fn create_temp_config_file(content: &str) -> std::io::Result<tempfile::NamedTempFile> {
        use std::io::Write;
        let mut file = tempfile::NamedTempFile::new()?;
        file.write_all(content.as_bytes())?;
        file.flush()?;
        Ok(file)
    }
    
    #[test]
    fn test_config_with_helper() {
        let config = create_test_config();
        assert_eq!(config.server.port, 9999);
    }
    
    #[test]
    fn test_config_file_loading() -> Result<(), Box<dyn std::error::Error>> {
        let json_content = r#"
        {
            "server": {"host": "0.0.0.0", "port": 8080, "max_connections": 100},
            "timeout": {"idle_seconds": 300, "shutdown_seconds": 10},
            "logging": {"level": "info", "format": "json"}
        }
        "#;
        
        let temp_file = create_temp_config_file(json_content)?;
        let config = ServerConfig::from_file(temp_file.path())?;
        
        assert_eq!(config.server.host, "0.0.0.0");
        assert_eq!(config.server.port, 8080);
        
        // temp_file은 스코프를 벗어나면 자동으로 삭제됨
        Ok(())
    }
}
```

`tempfile` 크레이트를 사용하려면 `Cargo.toml`에 추가한다.

```toml
[dev-dependencies]
tempfile = "3"
```

`[dev-dependencies]`는 테스트와 벤치마크에만 필요한 의존성을 지정한다. 일반 빌드에서는 포함되지 않는다.

## 통합 테스트

지금까지 작성한 테스트는 "단위 테스트"다. 개별 함수나 모듈을 테스트한다. "통합 테스트"는 여러 모듈이 함께 작동하는지 확인한다. 프로젝트 루트의 `tests/` 디렉토리에 작성한다.

```
my_project/
├── Cargo.toml
├── src/
│   ├── lib.rs
│   ├── protocol.rs
│   └── config.rs
└── tests/
    └── integration_test.rs
```

먼저 라이브러리로 사용할 수 있도록 `src/lib.rs`를 만든다.

```rust
// src/lib.rs
pub mod protocol;
pub mod config;
```

이제 통합 테스트를 작성한다.

```rust
// tests/integration_test.rs
use my_project::protocol::Message;
use my_project::config::ServerConfig;

#[test]
fn test_protocol_and_config_together() {
    // 설정 로드
    let config = ServerConfig::default();
    
    // 프로토콜 메시지 파싱
    let msg = Message::parse("ECHO hello");
    
    // 설정과 메시지가 함께 작동하는 시나리오
    assert_eq!(config.server.port, 8080);
    assert_eq!(msg, Message::Echo("hello".to_string()));
}

#[test]
fn test_full_message_roundtrip() {
    // 메시지를 파싱하고 다시 문자열로 변환했을 때 일관성 유지
    let original = "ECHO test message";
    let parsed = Message::parse(original);
    let serialized = parsed.to_string();
    
    // 다시 파싱하면 같은 결과가 나와야 함
    let reparsed = Message::parse(&serialized);
    assert_eq!(parsed, reparsed);
}
```

통합 테스트는 마치 외부 사용자가 라이브러리를 사용하는 것처럼 작성한다. `pub`으로 공개된 API만 접근할 수 있다.

## 비동기 코드 테스트하기

비동기 함수를 테스트하려면 `tokio::test` 매크로를 사용한다.

```rust
// src/server.rs
#[cfg(test)]
mod tests {
    use super::*;
    use tokio::io::{AsyncReadExt, AsyncWriteExt};
    
    #[tokio::test]
    async fn test_echo_server_single_message() {
        // 테스트용 서버를 랜덤 포트에서 시작
        let listener = tokio::net::TcpListener::bind("127.0.0.1:0").await.unwrap();
        let addr = listener.local_addr().unwrap();
        
        // 백그라운드에서 서버 실행
        tokio::spawn(async move {
            let (mut socket, _) = listener.accept().await.unwrap();
            let mut buf = [0; 1024];
            let n = socket.read(&mut buf).await.unwrap();
            socket.write_all(&buf[..n]).await.unwrap();
        });
        
        // 클라이언트로 연결
        let mut client = tokio::net::TcpStream::connect(addr).await.unwrap();
        
        // 메시지 전송
        client.write_all(b"ECHO hello").await.unwrap();
        
        // 응답 수신
        let mut response = [0; 1024];
        let n = client.read(&mut response).await.unwrap();
        
        assert_eq!(&response[..n], b"ECHO hello");
    }
    
    #[tokio::test]
    async fn test_concurrent_connections() {
        let listener = tokio::net::TcpListener::bind("127.0.0.1:0").await.unwrap();
        let addr = listener.local_addr().unwrap();
        
        // 서버 태스크
        tokio::spawn(async move {
            loop {
                let (mut socket, _) = listener.accept().await.unwrap();
                tokio::spawn(async move {
                    let mut buf = [0; 1024];
                    if let Ok(n) = socket.read(&mut buf).await {
                        let _ = socket.write_all(&buf[..n]).await;
                    }
                });
            }
        });
        
        // 여러 클라이언트 동시 연결
        let mut handles = vec![];
        for i in 0..10 {
            let handle = tokio::spawn(async move {
                let mut client = tokio::net::TcpStream::connect(addr).await.unwrap();
                let msg = format!("message {}", i);
                client.write_all(msg.as_bytes()).await.unwrap();
                
                let mut response = vec![0; msg.len()];
                client.read_exact(&mut response).await.unwrap();
                
                assert_eq!(response, msg.as_bytes());
            });
            handles.push(handle);
        }
        
        // 모든 클라이언트가 완료될 때까지 대기
        for handle in handles {
            handle.await.unwrap();
        }
    }
}
```

비동기 테스트는 실제 네트워크 연결을 만들고 데이터를 주고받는다. 테스트 격리를 위해 포트 0을 바인딩하면 OS가 사용 가능한 랜덤 포트를 할당해준다.

## 소유권과 빌림을 고려한 테스트

테스트에서도 소유권 규칙이 적용된다. 이것이 테스트 작성 방식에 영향을 미친다.

```rust
#[cfg(test)]
mod ownership_tests {
    use super::*;
    
    #[test]
    fn test_message_ownership() {
        let input = "ECHO test".to_string();
        
        // parse는 &str를 받으므로 참조를 전달
        let msg = Message::parse(&input);
        
        // input은 여전히 유효하고 다시 사용할 수 있음
        assert_eq!(input, "ECHO test");
        assert_eq!(msg, Message::Echo("test".to_string()));
    }
    
    #[test]
    fn test_config_cloning() {
        let config1 = create_test_config();
        
        // Clone을 구현했으므로 복사 가능
        let config2 = config1.clone();
        
        // 둘 다 독립적으로 사용 가능
        assert_eq!(config1.server.port, config2.server.port);
        
        // 하나를 변경해도 다른 하나는 영향 없음
        let mut config3 = config2.clone();
        config3.server.port = 9999;
        assert_eq!(config2.server.port, 8080);
        assert_eq!(config3.server.port, 9999);
    }
    
    #[test]
    fn test_borrowing_in_helper() {
        fn validate_message(msg: &Message) -> bool {
            match msg {
                Message::Ping | Message::Pong => true,
                Message::Echo(content) => !content.is_empty(),
                Message::Unknown(_) => false,
            }
        }
        
        let msg = Message::parse("ECHO hello");
        
        // 참조로 전달하므로 소유권 유지
        assert!(validate_message(&msg));
        
        // msg를 계속 사용 가능
        assert_eq!(msg, Message::Echo("hello".to_string()));
    }
}
```

테스트 헬퍼 함수를 작성할 때 소유권을 어떻게 처리할지 신중하게 결정한다. 일반적으로 참조(`&T`)를 받는 것이 유연하지만, 때로는 소유권을 가져가는 것(`T`)이 더 간단할 수 있다.

## 테스트 조직화 전략

프로젝트가 커지면 테스트도 조직화가 필요하다.

```rust
// src/protocol.rs
#[cfg(test)]
mod tests {
    use super::*;
    
    // 파싱 관련 테스트를 그룹화
    mod parsing {
        use super::*;
        
        #[test]
        fn ping_variants() {
            assert_eq!(Message::parse("PING"), Message::Ping);
            assert_eq!(Message::parse("ping"), Message::Ping);
            assert_eq!(Message::parse("  PING  "), Message::Ping);
        }
        
        #[test]
        fn echo_variants() {
            assert_eq!(
                Message::parse("ECHO hello"), 
                Message::Echo("hello".to_string())
            );
            assert_eq!(
                Message::parse("ECHO "), 
                Message::Echo("".to_string())
            );
        }
    }
    
    // 직렬화 관련 테스트를 그룹화
    mod serialization {
        use super::*;
        
        #[test]
        fn to_string_ping() {
            assert_eq!(Message::Ping.to_string(), "PING");
        }
        
        #[test]
        fn to_string_echo() {
            let msg = Message::Echo("test".to_string());
            assert_eq!(msg.to_string(), "ECHO test");
        }
        
        #[test]
        fn roundtrip() {
            let original = Message::Echo("data".to_string());
            let serialized = original.to_string();
            let deserialized = Message::parse(&serialized);
            assert_eq!(original, deserialized);
        }
    }
}
```

이렇게 하면 테스트를 논리적으로 그룹화하고, `cargo test parsing`처럼 특정 그룹만 실행할 수 있다.

## 테스트 실행 옵션

Cargo는 다양한 테스트 실행 옵션을 제공한다.

```bash
# 모든 테스트 실행
cargo test

# 특정 이름을 포함한 테스트만 실행
cargo test parse

# 단일 테스트 실행
cargo test test_parse_ping

# 무시된 테스트 실행
cargo test -- --ignored

# 출력을 캡처하지 않고 표시 (println! 보기)
cargo test -- --nocapture

# 병렬 실행 제한 (또는 순차 실행)
cargo test -- --test-threads=1

# 테스트 바이너리를 빌드만 하고 실행하지 않음
cargo test --no-run

# 릴리스 모드로 테스트 (최적화 적용)
cargo test --release
```

## 모킹과 테스트 더블

실제 네트워크나 파일 시스템에 의존하지 않고 테스트하려면 trait를 사용한 추상화가 도움이 된다.

```rust
// 실제 구현과 테스트 구현을 모두 지원하는 trait
pub trait ConfigLoader {
    fn load(&self, path: &str) -> Result<ServerConfig, ConfigError>;
}

// 실제 파일 시스템 사용
pub struct FileConfigLoader;

impl ConfigLoader for FileConfigLoader {
    fn load(&self, path: &str) -> Result<ServerConfig, ConfigError> {
        ServerConfig::from_file(path)
    }
}

// 테스트용 모킹 구현
#[cfg(test)]
pub struct MockConfigLoader {
    pub config: ServerConfig,
}

#[cfg(test)]
impl ConfigLoader for MockConfigLoader {
    fn load(&self, _path: &str) -> Result<ServerConfig, ConfigError> {
        Ok(self.config.clone())
    }
}

#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_with_mock_loader() {
        let mock_config = ServerConfig::default();
        let loader = MockConfigLoader {
            config: mock_config.clone(),
        };
        
        // 실제 파일 없이 테스트 가능
        let loaded = loader.load("nonexistent.json").unwrap();
        assert_eq!(loaded.server.port, mock_config.server.port);
    }
}
```

이 패턴을 사용하면 파일 시스템, 데이터베이스, 네트워크 호출 등 외부 의존성을 가진 코드도 독립적으로 테스트할 수 있다.

## 실전 테스트 작성 흐름

실제로 새 기능을 추가할 때 테스트를 작성하는 흐름은 다음과 같다.

```rust
// 1단계: 먼저 테스트를 작성 (TDD 방식)
#[test]
fn test_message_with_timestamp() {
    let msg = TimestampedMessage::new(Message::Ping);
    assert!(msg.timestamp > 0);
    assert_eq!(msg.message, Message::Ping);
}

// 2단계: 컴파일 에러를 확인 (아직 TimestampedMessage가 없음)

// 3단계: 최소한의 구현
pub struct TimestampedMessage {
    pub message: Message,
    pub timestamp: u64,
}

impl TimestampedMessage {
    pub fn new(message: Message) -> Self {
        Self {
            message,
            timestamp: std::time::SystemTime::now()
                .duration_since(std::time::UNIX_EPOCH)
                .unwrap()
                .as_secs(),
        }
    }
}

// 4단계: 테스트 실행 및 통과 확인
// cargo test test_message_with_timestamp

// 5단계: 더 많은 테스트 추가
#[test]
fn test_timestamp_increases() {
    let msg1 = TimestampedMessage::new(Message::Ping);
    std::thread::sleep(std::time::Duration::from_millis(10));
    let msg2 = TimestampedMessage::new(Message::Pong);
    
    assert!(msg2.timestamp >= msg1.timestamp);
}
```

이런 방식을 Test-Driven Development (TDD)라고 한다. 먼저 테스트를 작성하고, 테스트를 통과하는 코드를 작성하며, 리팩토링한다. 테스트가 있으면 리팩토링이 안전해진다.

## 테스트 작성의 실전 팁

**작은 테스트 많이 쓰기**: 하나의 큰 테스트보다 여러 개의 작은 테스트가 낫다. 어느 부분이 실패했는지 정확히 알 수 있다.

**테스트 이름 명확하게**: `test_1`, `test_2` 대신 `test_parse_ping_case_insensitive` 같이 무엇을 테스트하는지 이름에 담는다.

**Given-When-Then 패턴**: 테스트를 세 부분으로 구성한다. Given(준비), When(실행), Then(검증).

```rust
#[test]
fn test_config_override_with_env() {
    // Given: 기본 설정과 환경 변수 설정
    std::env::set_var("SERVER_PORT", "9999");
    
    // When: 설정 로드
    let config = ServerConfig::from_file_with_env("config.json").unwrap();
    
    // Then: 환경 변수로 오버라이드되었는지 확인
    assert_eq!(config.server.port, 9999);
    
    // 정리
    std::env::remove_var("SERVER_PORT");
}
```

**테스트 독립성 유지**: 각 테스트는 다른 테스트에 의존하지 않고 독립적으로 실행 가능해야 한다. 테스트 순서가 바뀌어도 결과가 같아야 한다.

**실패 메시지 명확하게**: 커스텀 메시지를 추가해서 실패 원인을 쉽게 파악한다.

```rust
assert_eq!(
    config.server.port, 
    8080,
    "환경 변수가 없으면 기본 포트를 사용해야 함"
);
```

## 마무리

오늘은 Rust의 테스트 시스템을 전반적으로 살펴봤다. 단위 테스트부터 통합 테스트, 비동기 테스트, 그리고 소유권을 고려한 테스트 작성까지 다뤘다. 테스트는 단순히 버그를 찾는 도구가 아니라, 코드 설계를 개선하고, 리팩토링을 안전하게 만들며, 문서화 역할도 한다.

Rust의 테스트는 언어와 깊이 통합되어 있다. `#[test]` 애트리뷰트를 추가하기만 하면 되고, `cargo test` 한 줄로 모든 테스트를 실행할 수 있다. 컴파일 타임에 많은 버그를 잡아주는 Rust의 타입 시스템과 결합하면, 매우 견고한 코드를 작성할 수 있다.

테스트를 작성하는 습관을 들이면, 처음에는 시간이 더 걸리는 것처럼 느껴지지만, 장기적으로는 디버깅 시간을 크게 줄여주고 자신감 있게 코드를 변경할 수 있게 된다. 특히 소유권과 빌림 같은 Rust의 고유한 개념들도 테스트를 통해 더 명확하게 이해할 수 있다.

내일은 에러 처리 라이브러리인 `anyhow`와 `thiserror`를 사용해서 에러 처리를 더욱 개선하는 방법을 배운다. 오늘 작성한 테스트들이 에러 처리 개선 과정에서 회귀(regression)를 방지하는 안전망 역할을 할 것이다.
