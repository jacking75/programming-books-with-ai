# Day 23: 비동기에서의 소유권과 빌림

어제 tokio로 TCP 서버를 재작성하면서 비동기 프로그래밍의 효율성을 체험했다. 하지만 비동기 코드에서 소유권과 빌림은 동기 코드보다 까다롭다. 특히 `.await` 너머로 참조를 전달하거나, 여러 태스크가 데이터를 공유할 때 문제가 생긴다.

오늘은 비동기 환경에서 소유권과 빌림이 어떻게 작동하는지, 그리고 실전에서 마주치는 문제들을 어떻게 해결하는지 배운다. Day 16-17에서 배운 `Arc`와 `Mutex`가 비동기에서 어떻게 달라지는지도 알아본다.

## 문제 상황: .await 너머의 참조

먼저 동기 코드에서 잘 작동하던 패턴이 비동기에서 왜 문제가 되는지 보자:

```rust
async fn process_data(data: &str) {
    println!("처리 시작: {}", data);
    tokio::time::sleep(tokio::time::Duration::from_secs(1)).await;
    println!("처리 완료: {}", data);
}

async fn example() {
    let message = String::from("중요한 데이터");
    process_data(&message).await;  // ✅ 이건 괜찮다
}
```

이 코드는 문제없다. `message`의 참조를 전달하고, `.await`가 완료될 때까지 `message`가 살아있다. 하지만 `tokio::spawn`을 사용하면 이야기가 달라진다:

```rust
async fn broken_example() {
    let message = String::from("중요한 데이터");
    
    // ❌ 컴파일 에러!
    tokio::spawn(async {
        process_data(&message).await;
    });
}
```

컴파일러가 에러를 낸다:

```
error[E0597]: `message` does not live long enough
```

왜 그럴까? `tokio::spawn`으로 생성된 태스크는 언제 끝날지 모른다. `broken_example` 함수가 반환된 후에도 계속 실행될 수 있다. 하지만 `message`는 함수가 끝나면 소멸한다. 태스크가 이미 사라진 데이터를 참조하게 될 위험이 있다.

```
Timeline:
─────────────────────────────────────────────────→ 시간
broken_example() 시작
  │
  ├─ message 생성 (스택)
  │
  ├─ tokio::spawn() 호출
  │   └─ 새 태스크 생성
  │
broken_example() 종료
  │
  └─ message 소멸 💥
      
태스크는 계속 실행 중... ❌ message를 참조하려 함
```

## 해결책 1: 소유권 이동 (move)

가장 간단한 해결책은 데이터의 소유권을 태스크로 이동하는 것이다:

```rust
async fn working_example() {
    let message = String::from("중요한 데이터");
    
    // ✅ move로 소유권 이동
    tokio::spawn(async move {
        tokio::time::sleep(tokio::time::Duration::from_secs(1)).await;
        println!("처리 완료: {}", message);
    });
    
    // 여기서 message를 사용하려고 하면 에러
    // println!("{}", message);  // 에러: value borrowed after move
}
```

`async move`는 클로저가 캡처한 모든 변수의 소유권을 가져간다. 이제 태스크가 `message`를 소유하므로, 태스크가 끝날 때까지 안전하게 살아있다.

```
Timeline:
─────────────────────────────────────────────────→ 시간
working_example() 시작
  │
  ├─ message 생성
  │
  ├─ tokio::spawn(async move { ... })
  │   └─ message 소유권이 태스크로 이동 →
  │
working_example() 종료
  
                    태스크가 message를 소유
                    ├─ sleep(1초)
                    └─ println! 후 message 소멸 ✅
```

## 해결책 2: Arc로 공유 소유권

여러 태스크가 같은 데이터를 읽어야 한다면 `Arc`(Atomic Reference Counting)를 사용한다:

```rust
use std::sync::Arc;

async fn shared_data_example() {
    let message = Arc::new(String::from("공유 데이터"));
    
    // 여러 태스크가 같은 데이터를 공유
    for i in 0..3 {
        let message_clone = message.clone();  // Arc의 복사본 (참조 카운트 증가)
        
        tokio::spawn(async move {
            tokio::time::sleep(tokio::time::Duration::from_millis(100 * i)).await;
            println!("태스크 {}: {}", i, message_clone);
        });
    }
    
    // 원본도 여전히 사용 가능
    println!("메인에서도 사용: {}", message);
}
```

`Arc::clone()`은 데이터를 복사하지 않는다. 단지 참조 카운트를 증가시킬 뿐이다. 마지막 `Arc`가 소멸될 때 데이터가 해제된다.

```
Arc 참조 카운팅:
─────────────────────────────────────────────────→ 시간

message = Arc::new("데이터")         [카운트: 1]
  │
  ├─ clone() → 태스크 0              [카운트: 2]
  ├─ clone() → 태스크 1              [카운트: 3]
  ├─ clone() → 태스크 2              [카운트: 4]
  │
메인 함수 종료 (message 소멸)        [카운트: 3]
  │
태스크 0 완료                        [카운트: 2]
태스크 1 완료                        [카운트: 1]
태스크 2 완료                        [카운트: 0] → 데이터 해제
```

## 실전 예제: 연결된 클라이언트 수 추적

비동기 TCP 서버에서 현재 연결된 클라이언트 수를 추적하는 기능을 만든다. 이를 위해서는 여러 태스크가 공유 카운터를 읽고 쓸 수 있어야 한다:

```rust
use tokio::net::TcpListener;
use std::sync::Arc;
use std::sync::atomic::{AtomicUsize, Ordering};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("서버 시작\n");
    
    // 원자적 카운터 (Atomic): 여러 태스크가 안전하게 접근 가능
    let client_count = Arc::new(AtomicUsize::new(0));
    
    loop {
        let (socket, addr) = listener.accept().await?;
        
        // 카운터 증가
        let count = client_count.fetch_add(1, Ordering::SeqCst) + 1;
        println!("[연결] {} (현재 {}개)", addr, count);
        
        let client_count_clone = client_count.clone();
        
        tokio::spawn(async move {
            // 클라이언트 처리...
            tokio::time::sleep(tokio::time::Duration::from_secs(5)).await;
            
            // 처리 완료 후 카운터 감소
            let count = client_count_clone.fetch_sub(1, Ordering::SeqCst) - 1;
            println!("[종료] {} (현재 {}개)", addr, count);
        });
    }
}
```

**`AtomicUsize`**: 원자적(atomic) 정수다. 여러 스레드/태스크가 동시에 접근해도 안전하다. `Mutex`보다 가볍고 빠르다.

**`Ordering::SeqCst`**: 가장 강한 메모리 순서 보장이다. 단순한 카운터에는 이것으로 충분하다.

원자적 타입은 간단한 값(정수, 불린)에만 사용할 수 있다. 복잡한 데이터 구조는 `Mutex`가 필요하다.

## std::sync::Mutex의 함정

Day 17에서 배운 `std::sync::Mutex`를 비동기에서 사용하면 문제가 생긴다:

```rust
use std::sync::{Arc, Mutex};

async fn problematic_mutex() {
    let counter = Arc::new(Mutex::new(0));
    let counter_clone = counter.clone();
    
    tokio::spawn(async move {
        let mut num = counter_clone.lock().unwrap();
        *num += 1;
        
        // ❌ 문제: .await를 넘어 MutexGuard를 보유
        tokio::time::sleep(tokio::time::Duration::from_secs(1)).await;
        
        println!("값: {}", *num);
    });
}
```

이 코드는 컴파일되지 않는다:

```
error: future cannot be sent between threads safely
```

`std::sync::Mutex`의 `MutexGuard`는 `.await` 너머로 보낼 수 없다. 왜냐하면 `.await` 중에 태스크가 다른 스레드로 이동할 수 있기 때문이다. `MutexGuard`는 스레드 로컬 정보를 포함하므로, 스레드 간 이동이 불가능하다.

```
Timeline:
─────────────────────────────────────────────────→ 시간
태스크 시작 (스레드 1)
  │
  ├─ lock() → MutexGuard 획득 (스레드 1에 묶임)
  │
  ├─ .await 시작
  │   └─ 런타임이 태스크를 스레드 2로 이동 💥
  │      MutexGuard는 여전히 스레드 1 정보를 가짐
  │
  └─ .await 완료 → MutexGuard 사용 시도 ❌
```

## 해결책: tokio::sync::Mutex

tokio는 비동기 친화적인 `Mutex`를 제공한다:

```rust
use std::sync::Arc;
use tokio::sync::Mutex;

async fn correct_async_mutex() {
    let counter = Arc::new(Mutex::new(0));
    let counter_clone = counter.clone();
    
    tokio::spawn(async move {
        let mut num = counter_clone.lock().await;  // .await 필요!
        *num += 1;
        
        // ✅ 괜찮다: tokio::sync::Mutex는 .await 너머로 안전
        tokio::time::sleep(tokio::time::Duration::from_secs(1)).await;
        
        println!("값: {}", *num);
    });
}
```

**`tokio::sync::Mutex::lock().await`**: 비동기 락이다. 락을 얻을 수 없으면 다른 태스크에게 CPU를 양보한다. 블로킹하지 않는다.

차이점을 정리하면:

```
std::sync::Mutex:
- 동기적 락 (.await 불필요)
- 빠르다 (OS 기본 기능 사용)
- .await 너머로 보낼 수 없음
- 짧은 임계 영역에 적합

tokio::sync::Mutex:
- 비동기 락 (.await 필요)
- 조금 느리다 (비동기 오버헤드)
- .await 너머로 안전
- 긴 임계 영역이나 .await가 필요한 경우
```

## 락 범위 최소화하기

`Mutex`를 사용할 때는 락을 보유하는 시간을 최소화해야 한다. 락을 오래 보유하면 다른 태스크들이 기다린다:

```rust
// ❌ 나쁜 예: 락을 오래 보유
async fn bad_locking(data: Arc<Mutex<Vec<String>>>) {
    let mut list = data.lock().await;
    
    // 락을 보유한 채로 비동기 작업 수행
    for item in list.iter() {
        process_item(item).await;  // 이 동안 다른 태스크들이 대기
    }
}

// ✅ 좋은 예: 필요한 데이터만 복사하고 락 해제
async fn good_locking(data: Arc<Mutex<Vec<String>>>) {
    // 락을 잡고 빠르게 복사
    let items = {
        let list = data.lock().await;
        list.clone()  // 또는 필요한 부분만 복사
    };  // 여기서 락 자동 해제
    
    // 락 없이 처리
    for item in items.iter() {
        process_item(item).await;
    }
}

async fn process_item(item: &str) {
    tokio::time::sleep(tokio::time::Duration::from_millis(100)).await;
    println!("처리: {}", item);
}
```

중괄호 `{}`로 락의 범위를 명시적으로 제한했다. `list`가 스코프를 벗어나면 `MutexGuard`가 자동으로 drop되어 락이 해제된다.

## 실전 예제: 공유 메시지 큐

여러 클라이언트가 메시지를 큐에 추가하고, 워커 태스크가 이를 처리하는 시스템을 만든다:

```rust
use tokio::sync::Mutex;
use std::sync::Arc;
use std::collections::VecDeque;

type MessageQueue = Arc<Mutex<VecDeque<String>>>;

#[tokio::main]
async fn main() {
    let queue: MessageQueue = Arc::new(Mutex::new(VecDeque::new()));
    
    // 프로듀서 태스크들 (메시지 생성)
    for i in 0..3 {
        let queue_clone = queue.clone();
        tokio::spawn(async move {
            for j in 0..5 {
                let message = format!("프로듀서 {} - 메시지 {}", i, j);
                
                {
                    let mut q = queue_clone.lock().await;
                    q.push_back(message.clone());
                    println!("📥 추가: {}", message);
                }  // 락 해제
                
                tokio::time::sleep(tokio::time::Duration::from_millis(100)).await;
            }
        });
    }
    
    // 컨슈머 태스크 (메시지 처리)
    let queue_clone = queue.clone();
    tokio::spawn(async move {
        loop {
            let message = {
                let mut q = queue_clone.lock().await;
                q.pop_front()
            };  // 락 해제
            
            match message {
                Some(msg) => {
                    println!("📤 처리: {}", msg);
                    tokio::time::sleep(tokio::time::Duration::from_millis(200)).await;
                }
                None => {
                    // 큐가 비었으면 잠시 대기
                    tokio::time::sleep(tokio::time::Duration::from_millis(50)).await;
                }
            }
        }
    });
    
    // 충분히 실행되도록 대기
    tokio::time::sleep(tokio::time::Duration::from_secs(3)).await;
}
```

이 패턴은 실전에서 자주 사용된다. 여러 생산자가 작업을 큐에 넣고, 소비자가 처리한다. 락의 범위를 최소화해서 성능을 유지했다.

## Send와 Sync trait

비동기 코드에서 자주 마주치는 에러 중 하나가 "cannot be sent between threads safely"다. 이를 이해하려면 `Send`와 `Sync` trait을 알아야 한다.

**`Send` trait**: 타입이 스레드 간에 이동될 수 있음을 의미한다. 대부분의 타입이 `Send`를 구현한다.

**`Sync` trait**: 타입의 참조가 스레드 간에 공유될 수 있음을 의미한다. `&T`가 `Send`면 `T`는 `Sync`다.

```rust
// Send를 구현하는 타입 (대부분)
let s = String::from("hello");  // String은 Send
tokio::spawn(async move {
    println!("{}", s);  // ✅ 이동 가능
});

// Send를 구현하지 않는 타입 (Rc, 원시 포인터 등)
use std::rc::Rc;
let rc = Rc::new(5);  // Rc는 Send가 아님
// tokio::spawn(async move {
//     println!("{}", rc);  // ❌ 에러: Rc<i32> cannot be sent between threads
// });

// Arc는 Send를 구현
use std::sync::Arc;
let arc = Arc::new(5);  // Arc는 Send
tokio::spawn(async move {
    println!("{}", arc);  // ✅ 가능
});
```

tokio의 규칙은 간단하다: `tokio::spawn`에 전달하는 Future는 `Send`여야 한다. 왜냐하면 런타임이 태스크를 다른 스레드로 옮길 수 있기 때문이다.

```
타입별 Send/Sync 정리:

타입                Send    Sync    비고
─────────────────────────────────────────────
String              ✅      ✅      안전
Vec<T>              ✅      ✅      (T가 Send/Sync면)
Rc<T>               ❌      ❌      단일 스레드 전용
Arc<T>              ✅      ✅      스레드 안전
Cell<T>             ✅      ❌      내부 가변성 (단일 스레드)
Mutex<T>            ✅      ✅      스레드 안전 가변성
RefCell<T>          ✅      ❌      단일 스레드 내부 가변성
```

## 실전: 비동기 채팅 서버 상태 관리

지금까지 배운 내용을 종합해서 채팅 서버의 공유 상태를 관리한다. 연결된 모든 클라이언트 목록을 유지하고, 메시지를 브로드캐스트한다:

```rust
use tokio::net::{TcpListener, TcpStream};
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use tokio::sync::{Mutex, broadcast};
use std::sync::Arc;
use std::collections::HashMap;

type ClientId = u64;
type Clients = Arc<Mutex<HashMap<ClientId, ClientInfo>>>;

struct ClientInfo {
    addr: std::net::SocketAddr,
    tx: broadcast::Sender<String>,
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("🚀 채팅 서버 시작\n");
    
    let clients: Clients = Arc::new(Mutex::new(HashMap::new()));
    let (broadcast_tx, _) = broadcast::channel(100);
    
    let mut next_id = 0u64;
    
    loop {
        let (socket, addr) = listener.accept().await?;
        let client_id = next_id;
        next_id += 1;
        
        println!("[클라이언트 #{}] 연결: {}", client_id, addr);
        
        // 클라이언트 등록
        {
            let mut client_map = clients.lock().await;
            client_map.insert(client_id, ClientInfo {
                addr,
                tx: broadcast_tx.clone(),
            });
            println!("   현재 접속자: {}명", client_map.len());
        }  // 락 해제
        
        let clients_clone = clients.clone();
        let broadcast_rx = broadcast_tx.subscribe();
        let broadcast_tx_clone = broadcast_tx.clone();
        
        tokio::spawn(async move {
            handle_client(socket, client_id, addr, clients_clone, 
                         broadcast_rx, broadcast_tx_clone).await;
        });
    }
}

async fn handle_client(
    socket: TcpStream,
    id: ClientId,
    addr: std::net::SocketAddr,
    clients: Clients,
    mut broadcast_rx: broadcast::Receiver<String>,
    broadcast_tx: broadcast::Sender<String>,
) {
    let (reader, mut writer) = socket.into_split();
    let mut reader = BufReader::new(reader);
    let mut line = String::new();
    
    // 환영 메시지
    let welcome = format!("환영합니다! 당신은 클라이언트 #{} 입니다.\n", id);
    let _ = writer.write_all(welcome.as_bytes()).await;
    
    loop {
        tokio::select! {
            // 클라이언트로부터 메시지 읽기
            result = reader.read_line(&mut line) => {
                match result {
                    Ok(0) => break,  // 연결 종료
                    Ok(_) => {
                        let message = format!("[클라이언트 #{}] {}", id, line.trim());
                        println!("{}", message);
                        
                        // 모든 클라이언트에게 브로드캐스트
                        let _ = broadcast_tx.send(message);
                        
                        line.clear();
                    }
                    Err(_) => break,
                }
            }
            
            // 다른 클라이언트의 메시지 받기
            result = broadcast_rx.recv() => {
                match result {
                    Ok(message) => {
                        let msg = format!("{}\n", message);
                        if writer.write_all(msg.as_bytes()).await.is_err() {
                            break;
                        }
                    }
                    Err(_) => break,
                }
            }
        }
    }
    
    // 클라이언트 제거
    {
        let mut client_map = clients.lock().await;
        client_map.remove(&id);
        println!("[클라이언트 #{}] 연결 종료 (남은 접속자: {}명)", 
                 id, client_map.len());
    }
}
```

이 서버는 여러 중요한 패턴을 보여준다:

**`Arc<Mutex<HashMap<...>>>`**: 여러 태스크가 클라이언트 목록을 안전하게 공유한다.

**락 범위 최소화**: 클라이언트를 추가/제거할 때만 락을 잡는다. 메시지 처리 중에는 락을 보유하지 않는다.

**`broadcast` 채널**: 한 클라이언트의 메시지를 모든 클라이언트에게 전달한다.

**`tokio::select!`**: 여러 비동기 작업을 동시에 대기한다. 둘 중 하나가 완료되면 처리한다.

## 참조와 .await의 관계

비동기 함수에서 참조를 사용할 때는 라이프타임에 특히 주의해야 한다:

```rust
// ✅ 괜찮다: 참조가 .await 너머로 전달되지 않음
async fn good_reference(data: &str) {
    println!("시작: {}", data);
    // data를 사용
    println!("끝: {}", data);
}

// ❌ 문제: 참조가 .await 너머로 보유됨
async fn problematic_reference(data: &str) {
    println!("시작: {}", data);
    tokio::time::sleep(tokio::time::Duration::from_secs(1)).await;
    println!("끝: {}", data);  // 이 참조가 유효할까?
}

// tokio::spawn과 함께 사용하려면...
async fn spawning_example() {
    let data = String::from("테스트");
    
    // ❌ 에러: data의 라이프타임이 충분하지 않음
    // tokio::spawn(problematic_reference(&data));
    
    // ✅ 해결책 1: 소유권 이동
    let data_owned = data.clone();
    tokio::spawn(async move {
        problematic_reference(&data_owned).await;
    });
    
    // ✅ 해결책 2: Arc 사용
    let data_arc = Arc::new(data);
    let data_clone = data_arc.clone();
    tokio::spawn(async move {
        let s: &str = &data_clone;
        problematic_reference(s).await;
    });
}
```

일반적인 규칙:

1. 함수 내에서만 사용되는 참조는 문제없다
2. `tokio::spawn`에 전달되는 참조는 `'static` 라이프타임이 필요하다
3. `Arc`나 소유권 이동으로 해결한다

## 성능 고려사항

비동기에서 소유권과 빌림을 다룰 때 성능 트레이드오프가 있다:

```rust
// 방법 1: 매번 clone (간단하지만 비용 발생)
async fn approach_clone(data: Arc<Vec<String>>) {
    for _ in 0..1000 {
        let data_copy = data.clone();  // Arc clone (저렴)
        tokio::spawn(async move {
            process(&data_copy).await;
        });
    }
}

// 방법 2: Arc 재사용 (효율적)
async fn approach_reuse(data: Arc<Vec<String>>) {
    let mut handles = vec![];
    
    for _ in 0..1000 {
        let data_ref = data.clone();
        let handle = tokio::spawn(async move {
            process(&data_ref).await;
        });
        handles.push(handle);
    }
    
    for handle in handles {
        handle.await.unwrap();
    }
}

async fn process(data: &[String]) {
    // 데이터 처리...
}
```

**`Arc::clone()`의 비용**: 매우 저렴하다. 원자적 증가만 수행한다. 데이터를 복사하지 않는다.

**실제 복사 비용**: `Vec::clone()`, `String::clone()` 등은 비용이 크다. 피할 수 있으면 피한다.

**Mutex 경합**: 많은 태스크가 동시에 락을 잡으려 하면 병목이 된다. 락 범위를 최소화하거나, 락이 필요 없는 설계를 고려한다.

## 락 없는 패턴: 채널 활용

때로는 `Mutex` 대신 채널을 사용하는 것이 더 나을 수 있다:

```rust
use tokio::sync::mpsc;

// Mutex 기반 (여러 태스크가 직접 접근)
async fn mutex_approach() {
    let counter = Arc::new(Mutex::new(0));
    
    for _ in 0..10 {
        let counter_clone = counter.clone();
        tokio::spawn(async move {
            let mut num = counter_clone.lock().await;
            *num += 1;
        });
    }
}

// 채널 기반 (단일 태스크가 상태 관리)
async fn channel_approach() {
    let (tx, mut rx) = mpsc::channel::<i32>(100);
    
    // 상태 관리 태스크
    tokio::spawn(async move {
        let mut counter = 0;
        while let Some(increment) = rx.recv().await {
            counter += increment;
            println!("현재 카운터: {}", counter);
        }
    });
    
    // 워커 태스크들
    for i in 0..10 {
        let tx_clone = tx.clone();
        tokio::spawn(async move {
            tx_clone.send(1).await.unwrap();
        });
    }
}
```

채널 기반 접근은 "액터 모델"이라고 불린다. 상태를 한 곳에서만 관리하므로, 락 경합이 없고 데드락 위험도 없다.

## 디버깅: 컴파일 에러 읽기

비동기 소유권 에러는 메시지가 길고 복잡하다. 핵심만 읽는 법을 배운다:

```
error: future cannot be sent between threads safely
   --> src/main.rs:10:18
    |
10  |     tokio::spawn(async move {
    |                  ^^^^^^^^^^^ future created by async block is not `Send`
    |
    = help: the trait `Send` is not implemented for `Rc<String>`
```

이 메시지가 말하는 것:
1. "future cannot be sent" → Future가 `Send` trait을 구현하지 않음
2. "not implemented for `Rc<String>`" → 문제의 원인은 `Rc`

**해결책**: `Rc`를 `Arc`로 바꾼다.

```
error[E0597]: `data` does not live long enough
   --> src/main.rs:15:25
    |
14  |     let data = String::from("test");
    |         ---- binding `data` declared here
15  |     tokio::spawn(async {
    |                  ----- value captured here...
16  |         println!("{}", data);
    |                        ^^^^ borrowed value does not live long enough
```

이 메시지가 말하는 것:
1. "`data` does not live long enough" → 라이프타임 부족
2. "value captured here" → async 블록이 캡처함

**해결책**: `async move`로 소유권을 이동한다.

## 핵심 정리

비동기에서 소유권과 빌림의 핵심 원칙들을 정리한다:

**`.await` 너머의 참조**: `tokio::spawn`에 전달되는 Future는 `'static` 라이프타임이 필요하다. 참조 대신 소유권을 이동하거나 `Arc`를 사용한다.

**`Arc<T>`**: 여러 태스크가 읽기 전용 데이터를 공유할 때 사용한다. 복사가 저렴하고 스레드 안전하다.

**`tokio::sync::Mutex`**: 여러 태스크가 가변 데이터를 공유할 때 사용한다. `std::sync::Mutex`와 달리 `.await` 너머로 안전하다.

**락 범위 최소화**: 중괄호로 락의 범위를 제한한다. 락을 보유한 채로 `.await`를 오래 하지 않는다.

**`Send`와 `Sync`**: `tokio::spawn`은 `Send` Future만 받는다. `Rc`, `RefCell` 등은 사용할 수 없다.

**채널 활용**: 복잡한 공유 상태는 채널로 단일 태스크에 위임하는 것을 고려한다.

비동기에서 소유권과 빌림은 동기 코드보다 까다롭지만, 컴파일러가 모든 실수를 잡아준다. 에러 메시지를 잘 읽고, `Arc`와 `Mutex`를 적절히 사용하면 안전하고 효율적인 비동기 코드를 작성할 수 있다. 내일은 `tokio::select!`로 여러 작업을 동시에 대기하고, 타임아웃을 구현하는 방법을 배운다.
