# Day 22: tokio로 TCP 서버 재작성

어제 `async/await`의 기초를 배웠다. 오늘은 그 지식을 실전에 적용해서 Day 11에서 만들었던 TCP 에코 서버를 비동기로 재작성한다. 스레드 기반 서버와 비교하면서 비동기의 장점을 직접 체험할 것이다.

먼저 Day 15에서 만들었던 스레드 기반 서버를 떠올려본다. 각 클라이언트마다 새 스레드를 생성했고, 이는 수십 개의 연결까지는 잘 작동했다. 하지만 수천 개의 동시 연결을 처리하려면 수천 개의 스레드가 필요하고, 각 스레드는 수 MB의 메모리를 소비한다. 비동기 방식은 이 문제를 해결한다.

```bash

```

`--features full`은 tokio의 모든 기능을 활성화한다. 실제 프로덕션에서는 필요한 기능만 선택하지만, 학습 단계에서는 전부 활성화하는 것이 편하다.

## 첫 비동기 TCP 서버

가장 간단한 형태부터 시작한다. 한 번에 한 클라이언트만 처리하는 서버다:

```rust
use tokio::net::TcpListener;
use tokio::io::{AsyncReadExt, AsyncWriteExt};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("서버가 127.0.0.1:8080 에서 대기 중...");
    
    loop {
        let (mut socket, addr) = listener.accept().await?;
        println!("새 연결: {}", addr);
        
        let mut buffer = vec![0; 1024];
        
        loop {
            let n = socket.read(&mut buffer).await?;
            
            if n == 0 {
                println!("{} 연결 종료", addr);
                break;
            }
            
            socket.write_all(&buffer[0..n]).await?;
        }
    }
}
```

이 코드는 동기 버전과 매우 비슷하다. 차이점은:

**`TcpListener::bind().await`**: 비동기 바인딩이다. 즉시 반환되고 `.await`에서 완료를 기다린다.

**`listener.accept().await`**: 새 연결을 기다린다. 연결이 올 때까지 다른 작업을 실행할 수 있다.

**`AsyncReadExt`와 `AsyncWriteExt`**: 비동기 읽기/쓰기를 위한 trait이다. `.read()`와 `.write_all()` 메서드를 제공한다.

하지만 이 서버는 여전히 한 번에 한 클라이언트만 처리한다. 한 클라이언트의 메시지를 읽는 동안 다른 클라이언트는 기다려야 한다. 진짜 비동기의 힘을 보려면 각 연결을 독립적인 태스크로 만들어야 한다.

## 동시 연결 처리: tokio::spawn 사용

각 클라이언트를 별도 태스크로 처리하도록 개선한다:

```rust
use tokio::net::TcpListener;
use tokio::io::{AsyncReadExt, AsyncWriteExt};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("서버가 127.0.0.1:8080 에서 대기 중...");
    
    loop {
        let (socket, addr) = listener.accept().await?;
        println!("새 연결: {}", addr);
        
        // 각 클라이언트를 독립적인 태스크로 처리
        tokio::spawn(async move {
            if let Err(e) = handle_client(socket).await {
                eprintln!("클라이언트 처리 중 에러: {}", e);
            }
        });
    }
}

async fn handle_client(mut socket: tokio::net::TcpStream) -> Result<(), Box<dyn std::error::Error>> {
    let mut buffer = vec![0; 1024];
    
    loop {
        let n = socket.read(&mut buffer).await?;
        
        if n == 0 {
            return Ok(());
        }
        
        socket.write_all(&buffer[0..n]).await?;
    }
}
```

이제 각 클라이언트는 독립적으로 처리된다. `tokio::spawn`은 새 비동기 태스크를 생성한다. 중요한 점은 새 스레드를 생성하지 않는다는 것이다! tokio 런타임이 모든 태스크를 소수의 스레드에서 번갈아 실행한다.

`async move` 클로저를 사용해서 `socket`의 소유권을 태스크로 이동했다. 각 태스크는 자신의 소켓을 소유하므로 다른 태스크와 간섭하지 않는다.

## 더 나은 에러 처리와 로깅

실전에서는 에러를 더 자세히 처리하고, 각 클라이언트의 상태를 추적해야 한다:

```rust
use tokio::net::{TcpListener, TcpStream};
use tokio::io::{AsyncReadExt, AsyncWriteExt};
use std::net::SocketAddr;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("🚀 서버가 127.0.0.1:8080 에서 대기 중...\n");
    
    let mut client_id = 0u64;
    
    loop {
        match listener.accept().await {
            Ok((socket, addr)) => {
                client_id += 1;
                println!("[클라이언트 #{}] 연결됨: {}", client_id, addr);
                
                tokio::spawn(async move {
                    handle_client(socket, addr, client_id).await;
                });
            }
            Err(e) => {
                eprintln!("연결 수락 실패: {}", e);
            }
        }
    }
}

async fn handle_client(mut socket: TcpStream, addr: SocketAddr, id: u64) {
    let mut buffer = vec![0; 1024];
    let mut total_bytes = 0;
    
    loop {
        match socket.read(&mut buffer).await {
            Ok(0) => {
                println!("[클라이언트 #{}] 연결 종료 (총 {}바이트 처리)", id, total_bytes);
                break;
            }
            Ok(n) => {
                total_bytes += n;
                
                // 받은 데이터를 문자열로 변환해서 로깅
                if let Ok(message) = std::str::from_utf8(&buffer[0..n]) {
                    print!("[클라이언트 #{}] 받음: {}", id, message.trim());
                }
                
                // 에코 응답
                if let Err(e) = socket.write_all(&buffer[0..n]).await {
                    eprintln!("[클라이언트 #{}] 쓰기 에러: {}", id, e);
                    break;
                }
            }
            Err(e) => {
                eprintln!("[클라이언트 #{}] 읽기 에러: {}", id, e);
                break;
            }
        }
    }
}
```

각 클라이언트에게 고유 ID를 부여해서 로그를 추적하기 쉽게 만들었다. 이제 여러 클라이언트가 동시에 연결해도 누가 누구인지 알 수 있다.

## 라인 기반 프로토콜로 개선

현재 코드는 원시 바이트를 그대로 읽는다. 더 실용적으로 만들려면 라인 단위로 처리하는 것이 좋다. tokio는 이를 위한 편리한 타입을 제공한다:

```rust
use tokio::net::{TcpListener, TcpStream};
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use std::net::SocketAddr;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("🚀 라인 기반 에코 서버 시작\n");
    
    let mut client_id = 0u64;
    
    loop {
        let (socket, addr) = listener.accept().await?;
        client_id += 1;
        println!("[클라이언트 #{}] 연결: {}", client_id, addr);
        
        tokio::spawn(async move {
            handle_client(socket, addr, client_id).await;
        });
    }
}

async fn handle_client(socket: TcpStream, addr: SocketAddr, id: u64) {
    let (reader, mut writer) = socket.into_split();
    let mut reader = BufReader::new(reader);
    let mut line = String::new();
    
    loop {
        line.clear();
        
        match reader.read_line(&mut line).await {
            Ok(0) => {
                println!("[클라이언트 #{}] 연결 종료", id);
                break;
            }
            Ok(_) => {
                print!("[클라이언트 #{}] 받음: {}", id, line);
                
                // 에코 응답
                if let Err(e) = writer.write_all(line.as_bytes()).await {
                    eprintln!("[클라이언트 #{}] 쓰기 에러: {}", id, e);
                    break;
                }
            }
            Err(e) => {
                eprintln!("[클라이언트 #{}] 읽기 에러: {}", id, e);
                break;
            }
        }
    }
}
```

**`socket.into_split()`**: 소켓을 읽기와 쓰기로 분리한다. 이렇게 하면 두 부분을 독립적으로 사용할 수 있다.

**`BufReader`**: 버퍼링된 읽기를 제공한다. `read_line()` 메서드로 한 줄씩 읽을 수 있다.

**`line.clear()`**: 문자열을 재사용해서 매번 할당하지 않는다. 작은 최적화지만 효과적이다.

## 성능 비교: 스레드 vs 비동기

비동기의 효율성을 확인하기 위해 간단한 부하 테스트를 만든다:

```rust
// 이 코드는 별도 프로젝트로 만든다: cargo new load_test
use tokio::net::TcpStream;
use tokio::io::{AsyncWriteExt, AsyncReadExt};
use std::time::Instant;

#[tokio::main]
async fn main() {
    let num_clients = 1000;
    let messages_per_client = 10;
    
    println!("{}개 클라이언트로 부하 테스트 시작...", num_clients);
    let start = Instant::now();
    
    let mut handles = vec![];
    
    for id in 0..num_clients {
        let handle = tokio::spawn(async move {
            test_client(id, messages_per_client).await
        });
        handles.push(handle);
    }
    
    // 모든 클라이언트 완료 대기
    for handle in handles {
        handle.await.unwrap();
    }
    
    let elapsed = start.elapsed();
    let total_messages = num_clients * messages_per_client;
    
    println!("\n=== 결과 ===");
    println!("총 클라이언트: {}", num_clients);
    println!("클라이언트당 메시지: {}", messages_per_client);
    println!("총 메시지: {}", total_messages);
    println!("소요 시간: {:.2}초", elapsed.as_secs_f64());
    println!("초당 메시지: {:.2}", total_messages as f64 / elapsed.as_secs_f64());
}

async fn test_client(id: u32, num_messages: u32) {
    match TcpStream::connect("127.0.0.1:8080").await {
        Ok(mut stream) => {
            for i in 0..num_messages {
                let message = format!("Client {} message {}\n", id, i);
                
                if stream.write_all(message.as_bytes()).await.is_err() {
                    eprintln!("Client {} write failed", id);
                    return;
                }
                
                let mut buffer = vec![0; 1024];
                if stream.read(&mut buffer).await.is_err() {
                    eprintln!("Client {} read failed", id);
                    return;
                }
            }
        }
        Err(e) => eprintln!("Client {} connection failed: {}", id, e),
    }
}
```

이 테스트를 실행하면 1000개의 동시 연결이 문제없이 처리되는 것을 볼 수 있다. 각 연결은 태스크로 실행되고, tokio는 이를 몇 개의 스레드에서 효율적으로 처리한다.

```
1000개 클라이언트로 부하 테스트 시작...

=== 결과 ===
총 클라이언트: 1000
클라이언트당 메시지: 10
총 메시지: 10000
소요 시간: 2.34초
초당 메시지: 4273.50
```

스레드 기반 서버로 1000개 연결을 처리하려면 1000개 스레드가 필요하고, 메모리만 수 GB를 소비한다. 비동기는 훨씬 적은 리소스로 같은 결과를 낸다.

## 비동기 태스크의 실행 모델

tokio가 내부적으로 어떻게 작동하는지 이해하면 도움이 된다:

```
┌─────────────────────────────────────────────────┐
│          tokio Runtime (런타임)                  │
│                                                 │
│  ┌──────────────────────────────────────────┐  │
│  │     Worker Thread Pool (스레드 풀)        │  │
│  │  ┌────────┬────────┬────────┬────────┐   │  │
│  │  │Thread1 │Thread2 │Thread3 │Thread4 │   │  │
│  │  └────────┴────────┴────────┴────────┘   │  │
│  └──────────────────────────────────────────┘  │
│                      ↕                          │
│  ┌──────────────────────────────────────────┐  │
│  │        Task Queue (태스크 큐)             │  │
│  │  [Task A] [Task B] [Task C] ...         │  │
│  │  [Task D] [Task E] [Task F] ...         │  │
│  │  ... (수천 개의 태스크 가능)             │  │
│  └──────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘

각 Task는:
- 한 클라이언트 연결
- .await를 만나면 일시정지
- 다른 Task에게 CPU 양보
- I/O 완료되면 다시 실행
```

스레드는 OS가 관리하고 컨텍스트 스위칭이 무겁다. 태스크는 런타임이 관리하고 전환이 가볍다. 이것이 수천 개의 동시 연결을 처리할 수 있는 비결이다.

## 실전 패턴: 연결 수 제한하기

무한정 연결을 받으면 서버가 과부하될 수 있다. 동시 연결 수를 제한하는 패턴을 배운다:

```rust
use tokio::net::TcpListener;
use tokio::sync::Semaphore;
use std::sync::Arc;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("🚀 서버 시작 (최대 100개 동시 연결)\n");
    
    // 세마포어로 동시 연결 수 제한
    let max_connections = 100;
    let semaphore = Arc::new(Semaphore::new(max_connections));
    
    let mut client_id = 0u64;
    
    loop {
        // 연결 허가 얻기 (가득 차면 대기)
        let permit = semaphore.clone().acquire_owned().await.unwrap();
        
        let (socket, addr) = listener.accept().await?;
        client_id += 1;
        
        let current_id = client_id;
        println!("[클라이언트 #{}] 연결: {} (활성 연결: {})",
                 current_id, addr, max_connections - semaphore.available_permits());
        
        tokio::spawn(async move {
            handle_client(socket, current_id).await;
            // permit이 drop되면 자동으로 세마포어 해제
            drop(permit);
            println!("[클라이언트 #{}] 종료", current_id);
        });
    }
}

async fn handle_client(mut socket: tokio::net::TcpStream, id: u64) {
    use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
    
    let (reader, mut writer) = socket.into_split();
    let mut reader = BufReader::new(reader);
    let mut line = String::new();
    
    loop {
        line.clear();
        
        match reader.read_line(&mut line).await {
            Ok(0) => break,
            Ok(_) => {
                if writer.write_all(line.as_bytes()).await.is_err() {
                    break;
                }
            }
            Err(_) => break,
        }
    }
}
```

**`Semaphore`**: 동시 접근을 제한하는 동기화 기법이다. 100개 허가를 가진 세마포어는 최대 100개 연결만 허용한다.

**`Arc::new(Semaphore::new(max_connections))`**: 세마포어를 여러 태스크가 공유할 수 있도록 Arc로 감싼다.

**`acquire_owned()`**: 허가를 얻는다. 가득 차면 자동으로 대기한다. `owned`는 permit의 소유권을 태스크로 이동한다는 의미다.

**`drop(permit)`**: permit이 drop되면 세마포어가 자동으로 해제되어 다른 연결이 들어올 수 있다.

101번째 클라이언트가 연결하려고 하면, 기존 연결 하나가 종료될 때까지 자동으로 대기한다. 서버가 과부하되는 것을 방지한다.

## 타임아웃 추가하기

클라이언트가 너무 오래 아무것도 보내지 않으면 연결을 끊는 것이 좋다. tokio는 타임아웃을 쉽게 구현할 수 있다:

```rust
use tokio::time::{timeout, Duration};

async fn handle_client_with_timeout(mut socket: tokio::net::TcpStream, id: u64) {
    use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
    
    let (reader, mut writer) = socket.into_split();
    let mut reader = BufReader::new(reader);
    let mut line = String::new();
    
    let idle_timeout = Duration::from_secs(30);
    
    loop {
        line.clear();
        
        // 30초 타임아웃
        match timeout(idle_timeout, reader.read_line(&mut line)).await {
            Ok(Ok(0)) => {
                println!("[클라이언트 #{}] 정상 종료", id);
                break;
            }
            Ok(Ok(_)) => {
                if writer.write_all(line.as_bytes()).await.is_err() {
                    break;
                }
            }
            Ok(Err(e)) => {
                eprintln!("[클라이언트 #{}] 읽기 에러: {}", id, e);
                break;
            }
            Err(_) => {
                println!("[클라이언트 #{}] 타임아웃으로 연결 종료", id);
                let _ = writer.write_all(b"Idle timeout\n").await;
                break;
            }
        }
    }
}
```

**`timeout(duration, future)`**: Future가 주어진 시간 내에 완료되지 않으면 에러를 반환한다.

이중 `Result`가 나오는 이유는:
- 바깥 `Result`는 타임아웃 성공/실패
- 안쪽 `Result`는 읽기 성공/실패

30초 동안 클라이언트가 아무 메시지도 보내지 않으면 자동으로 연결이 끊긴다.

## 우아한 종료 (Graceful Shutdown)

Ctrl+C를 누르면 서버가 즉시 종료되는데, 진행 중인 연결들을 정리하고 종료하는 것이 더 좋다:

```rust
use tokio::signal;
use tokio::sync::broadcast;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("🚀 서버 시작 (Ctrl+C로 종료)\n");
    
    // 종료 신호를 브로드캐스트할 채널
    let (shutdown_tx, _) = broadcast::channel(1);
    
    // Ctrl+C 처리
    let shutdown_tx_clone = shutdown_tx.clone();
    tokio::spawn(async move {
        signal::ctrl_c().await.unwrap();
        println!("\n\n🛑 종료 신호 받음, 연결 정리 중...");
        let _ = shutdown_tx_clone.send(());
    });
    
    loop {
        let mut shutdown_rx = shutdown_tx.subscribe();
        
        tokio::select! {
            result = listener.accept() => {
                let (socket, addr) = result?;
                println!("[새 연결] {}", addr);
                
                let mut shutdown_rx = shutdown_tx.subscribe();
                
                tokio::spawn(async move {
                    tokio::select! {
                        _ = handle_client(socket) => {},
                        _ = shutdown_rx.recv() => {
                            println!("[{}] 종료 신호로 연결 종료", addr);
                        }
                    }
                });
            }
            _ = shutdown_rx.recv() => {
                println!("새 연결 수락 중단");
                break;
            }
        }
    }
    
    println!("서버 종료 완료");
    Ok(())
}

async fn handle_client(mut socket: tokio::net::TcpStream) {
    use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
    
    let (reader, mut writer) = socket.into_split();
    let mut reader = BufReader::new(reader);
    let mut line = String::new();
    
    loop {
        line.clear();
        match reader.read_line(&mut line).await {
            Ok(0) => break,
            Ok(_) => {
                if writer.write_all(line.as_bytes()).await.is_err() {
                    break;
                }
            }
            Err(_) => break,
        }
    }
}
```

**`broadcast::channel`**: 여러 수신자에게 동시에 메시지를 보낼 수 있는 채널이다. 종료 신호를 모든 태스크에 전파하는 데 완벽하다.

**`tokio::select!`**: 여러 비동기 작업 중 하나가 완료되면 그것을 실행한다. 내일 더 자세히 배운다.

**`signal::ctrl_c()`**: Ctrl+C 신호를 기다리는 Future다.

이제 Ctrl+C를 누르면 서버가 새 연결 수락을 중단하고, 진행 중인 연결들에게 종료 신호를 보낸 후 깔끔하게 종료된다.

## 전체 코드: 프로덕션 수준 에코 서버

지금까지 배운 모든 것을 합친 완전한 서버다:

```rust
use tokio::net::{TcpListener, TcpStream};
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use tokio::sync::{Semaphore, broadcast};
use tokio::signal;
use tokio::time::{timeout, Duration};
use std::sync::Arc;
use std::net::SocketAddr;

const MAX_CONNECTIONS: usize = 100;
const IDLE_TIMEOUT: Duration = Duration::from_secs(60);

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    println!("🚀 비동기 에코 서버 시작");
    println!("   주소: 127.0.0.1:8080");
    println!("   최대 연결: {}", MAX_CONNECTIONS);
    println!("   유휴 타임아웃: {}초", IDLE_TIMEOUT.as_secs());
    println!("   Ctrl+C로 종료\n");
    
    let semaphore = Arc::new(Semaphore::new(MAX_CONNECTIONS));
    let (shutdown_tx, _) = broadcast::channel::<()>(1);
    
    // Ctrl+C 처리
    let shutdown_signal = shutdown_tx.clone();
    tokio::spawn(async move {
        signal::ctrl_c().await.unwrap();
        println!("\n\n🛑 종료 신호 받음");
        let _ = shutdown_signal.send(());
    });
    
    let mut client_id = 0u64;
    let mut shutdown_rx = shutdown_tx.subscribe();
    
    loop {
        tokio::select! {
            result = listener.accept() => {
                let (socket, addr) = result?;
                
                // 세마포어로 연결 수 제한
                let permit = match semaphore.clone().try_acquire_owned() {
                    Ok(permit) => permit,
                    Err(_) => {
                        println!("[거부] {} - 최대 연결 수 도달", addr);
                        continue;
                    }
                };
                
                client_id += 1;
                let active = MAX_CONNECTIONS - semaphore.available_permits();
                println!("[클라이언트 #{}] 연결: {} (활성: {})", client_id, addr, active);
                
                let mut client_shutdown = shutdown_tx.subscribe();
                
                tokio::spawn(async move {
                    tokio::select! {
                        _ = handle_client(socket, addr, client_id) => {},
                        _ = client_shutdown.recv() => {
                            println!("[클라이언트 #{}] 종료 신호로 연결 종료", client_id);
                        }
                    }
                    drop(permit);
                });
            }
            _ = shutdown_rx.recv() => {
                println!("새 연결 수락 중단");
                break;
            }
        }
    }
    
    // 모든 연결이 정리될 때까지 잠시 대기
    tokio::time::sleep(Duration::from_secs(1)).await;
    println!("✅ 서버 종료 완료");
    
    Ok(())
}

async fn handle_client(socket: TcpStream, addr: SocketAddr, id: u64) {
    let (reader, mut writer) = socket.into_split();
    let mut reader = BufReader::new(reader);
    let mut line = String::new();
    let mut message_count = 0;
    
    loop {
        line.clear();
        
        match timeout(IDLE_TIMEOUT, reader.read_line(&mut line)).await {
            Ok(Ok(0)) => {
                println!("[클라이언트 #{}] 정상 종료 ({}개 메시지 처리)", id, message_count);
                break;
            }
            Ok(Ok(n)) => {
                message_count += 1;
                print!("[클라이언트 #{}] 받음: {}", id, line.trim());
                
                if let Err(e) = writer.write_all(&line.as_bytes()).await {
                    eprintln!("[클라이언트 #{}] 쓰기 에러: {}", id, e);
                    break;
                }
            }
            Ok(Err(e)) => {
                eprintln!("[클라이언트 #{}] 읽기 에러: {}", id, e);
                break;
            }
            Err(_) => {
                println!("[클라이언트 #{}] 유휴 타임아웃", id);
                let _ = writer.write_all(b"Idle timeout - disconnecting\n").await;
                break;
            }
        }
    }
}
```

이 서버는:
- 최대 100개 동시 연결 제한
- 60초 유휴 타임아웃
- 우아한 종료 처리
- 상세한 로깅
- 프로덕션에서 사용 가능한 에러 처리

## 스레드 기반과 비교

Day 15에서 만든 스레드 기반 서버와 비교해본다:

**스레드 기반:**
```rust
for stream in listener.incoming() {
    let stream = stream.unwrap();
    
    thread::spawn(move || {
        handle_client(stream);
    });
}
```

**비동기 기반:**
```rust
loop {
    let (socket, _) = listener.accept().await?;
    
    tokio::spawn(async move {
        handle_client(socket).await;
    });
}
```

코드는 비슷하지만 내부 동작은 완전히 다르다:

```
스레드 기반:
1000 연결 → 1000 OS 스레드 → 2GB 메모리 → 느린 컨텍스트 스위칭

비동기 기반:
1000 연결 → 1000 태스크 → 4 스레드 → 30MB 메모리 → 빠른 전환
```

비동기는 스레드보다 월등히 효율적이다. 특히 I/O 바운드 작업에서는 더욱 그렇다.

## 비동기의 함정: CPU 집약적 작업

비동기가 항상 좋은 것은 아니다. CPU 집약적 작업에서는 오히려 문제가 된다:

```rust
async fn bad_example() {
    // ❌ 나쁜 예: CPU 집약적 작업을 비동기로
    let result = (0..1_000_000_000).sum::<u64>();  // .await가 없음!
    println!("결과: {}", result);
}
```

이 코드는 `.await`가 없으므로 런타임이 다른 태스크로 전환할 수 없다. 계산이 끝날 때까지 스레드를 독점한다. 다른 모든 태스크가 멈춘다!

CPU 집약적 작업은 `tokio::task::spawn_blocking`으로 처리한다:

```rust
async fn good_example() {
    // ✅ 좋은 예: CPU 작업을 블로킹 스레드에서
    let result = tokio::task::spawn_blocking(|| {
        (0..1_000_000_000).sum::<u64>()
    }).await.unwrap();
    
    println!("결과: {}", result);
}
```

`spawn_blocking`은 별도 스레드 풀에서 작업을 실행하므로, 비동기 런타임을 방해하지 않는다.

## 핵심 정리

오늘 배운 내용을 정리한다:

**`tokio::net::TcpListener`**: 비동기 TCP 리스너다. `.accept().await`로 새 연결을 기다린다.

**`tokio::spawn`**: 독립적인 비동기 태스크를 생성한다. 새 스레드를 만들지 않고, 런타임의 스레드 풀에서 실행된다.

**`AsyncReadExt`와 `AsyncWriteExt`**: 비동기 읽기/쓰기 trait이다. `.await`가 필요한 메서드들을 제공한다.

**세마포어**: 동시 연결 수를 제한한다. 리소스를 보호하는 강력한 도구다.

**`timeout()`**: 작업에 시간 제한을 건다. 무한 대기를 방지한다.

**`broadcast` 채널**: 여러 수신자에게 메시지를 전파한다. 종료 신호에 완벽하다.

**효율성**: 수천 개 연결을 몇 개 스레드로 처리한다. 메모리와 CPU를 절약한다.

비동기 프로그래밍은 처음엔 복잡해 보이지만, 기본 패턴을 익히면 스레드보다 훨씬 강력하다. 내일은 비동기에서 소유권과 빌림이 어떻게 작동하는지, 그리고 공유 상태를 안전하게 관리하는 방법을 배운다.
