# 6장. 라이프타임과 복잡한 빌림
라이프타임은 Rust를 배울 때 가장 어려운 부분 중 하나다. C++에서는 댕글링 포인터 문제를 런타임에 발견하거나 아예 발견하지 못하지만, Rust는 컴파일 타임에 이를 방지한다. 이 장에서는 라이프타임 표기가 필요한 복잡한 상황들과 그 해결 방법을 실전 예제를 통해 살펴본다.

## 6.1 구조체에 참조를 저장할 때

### 문제 상황: 참조를 필드로 가지는 구조체
C++에서는 클래스 멤버로 참조나 포인터를 자유롭게 저장할 수 있다. Rust에서도 가능하지만 라이프타임을 명시해야 한다.

```rust
// 컴파일 실패!
struct Parser {
    source: &str,  // 에러: 라이프타임 명시 필요
    position: usize,
}
```

컴파일러는 `source` 참조가 얼마나 오래 유효한지 알 수 없다. 라이프타임 매개변수를 추가해야 한다.

```rust
struct Parser<'a> {
    source: &'a str,
    position: usize,
}

impl<'a> Parser<'a> {
    fn new(source: &'a str) -> Self {
        Parser {
            source,
            position: 0,
        }
    }
    
    fn current_char(&self) -> Option<char> {
        self.source.chars().nth(self.position)
    }
    
    fn advance(&mut self) {
        self.position += 1;
    }
    
    fn remaining(&self) -> &'a str {
        &self.source[self.position..]
    }
}

fn main() {
    let text = String::from("Hello, Rust!");
    let mut parser = Parser::new(&text);
    
    while let Some(ch) = parser.current_char() {
        println!("{}", ch);
        parser.advance();
        if parser.position >= 5 {
            break;
        }
    }
}
```

`'a`는 라이프타임 매개변수다. "`Parser`는 적어도 `'a`만큼 살아있는 `str` 참조를 가진다"는 의미다.

### 실패하는 케이스: 라이프타임 위반

```rust
fn create_parser() -> Parser {
    let text = String::from("temporary");
    Parser::new(&text)  // 에러: text가 함수 끝에서 drop됨
}
```

`text`는 함수가 끝나면 사라지는데, 반환되는 `Parser`가 그것을 참조하므로 댕글링 참조가 생긴다. Rust는 이를 컴파일 타임에 방지한다.

### 해결책 1: 소유권 이동
참조 대신 소유권을 가지도록 변경한다.

```rust
struct Parser {
    source: String,  // 소유
    position: usize,
}

impl Parser {
    fn new(source: String) -> Self {
        Parser {
            source,
            position: 0,
        }
    }
    
    fn from_str(source: &str) -> Self {
        Parser {
            source: source.to_string(),
            position: 0,
        }
    }
    
    fn remaining(&self) -> &str {
        &self.source[self.position..]
    }
}

fn create_parser() -> Parser {
    Parser::new(String::from("This works!"))
}
```

이제 `Parser`가 데이터를 소유하므로 라이프타임 문제가 없다. 메모리 복사 비용이 들지만 안전하다.

### 해결책 2: 참조가 정말 필요하다면
참조를 반드시 사용해야 한다면 호출자가 데이터를 소유하게 한다.

```rust
struct Parser<'a> {
    source: &'a str,
    position: usize,
}

fn process_text(text: &str) {
    let parser = Parser::new(text);
    // parser는 text의 라이프타임 동안만 유효
    // text는 이 함수 전체에서 살아있으므로 OK
}

struct Document {
    content: String,
}

impl Document {
    fn parse(&self) -> Parser {
        Parser::new(&self.content)
    }
}
```

`Parser`의 라이프타임은 `Document`의 라이프타임과 연결된다.

### 복잡한 예제: 토크나이저
실전적인 토크나이저 구현을 보자.

```rust
#[derive(Debug, PartialEq)]
enum Token<'a> {
    Word(&'a str),
    Number(i32),
    Symbol(char),
}

struct Tokenizer<'a> {
    input: &'a str,
    position: usize,
}

impl<'a> Tokenizer<'a> {
    fn new(input: &'a str) -> Self {
        Tokenizer { input, position: 0 }
    }
    
    fn current(&self) -> Option<char> {
        self.input.chars().nth(self.position)
    }
    
    fn peek_ahead(&self, offset: usize) -> Option<char> {
        self.input.chars().nth(self.position + offset)
    }
    
    fn advance(&mut self) {
        self.position += 1;
    }
    
    fn skip_whitespace(&mut self) {
        while let Some(ch) = self.current() {
            if !ch.is_whitespace() {
                break;
            }
            self.advance();
        }
    }
    
    fn read_word(&mut self) -> &'a str {
        let start = self.position;
        
        while let Some(ch) = self.current() {
            if !ch.is_alphabetic() {
                break;
            }
            self.advance();
        }
        
        &self.input[start..self.position]
    }
    
    fn read_number(&mut self) -> i32 {
        let start = self.position;
        
        while let Some(ch) = self.current() {
            if !ch.is_numeric() {
                break;
            }
            self.advance();
        }
        
        self.input[start..self.position].parse().unwrap_or(0)
    }
    
    fn next_token(&mut self) -> Option<Token<'a>> {
        self.skip_whitespace();
        
        let ch = self.current()?;
        
        if ch.is_alphabetic() {
            let word = self.read_word();
            Some(Token::Word(word))
        } else if ch.is_numeric() {
            let num = self.read_number();
            Some(Token::Number(num))
        } else {
            self.advance();
            Some(Token::Symbol(ch))
        }
    }
    
    fn tokenize(&mut self) -> Vec<Token<'a>> {
        let mut tokens = Vec::new();
        
        while let Some(token) = self.next_token() {
            tokens.push(token);
        }
        
        tokens
    }
}

fn main() {
    let source = "let x = 42 + y;";
    let mut tokenizer = Tokenizer::new(source);
    let tokens = tokenizer.tokenize();
    
    for token in tokens {
        println!("{:?}", token);
    }
}
```

`Token::Word`가 원본 문자열의 슬라이스를 가리키므로, 추가 할당 없이 효율적이다. 라이프타임 `'a`가 이 관계를 보장한다.

### 여러 참조를 가진 구조체
구조체가 여러 참조를 가질 때는 각각 다른 라이프타임을 가질 수 있다.

```rust
struct Context<'a, 'b> {
    config: &'a str,
    data: &'b [u8],
}

impl<'a, 'b> Context<'a, 'b> {
    fn new(config: &'a str, data: &'b [u8]) -> Self {
        Context { config, data }
    }
    
    fn get_config(&self) -> &'a str {
        self.config
    }
    
    fn get_data(&self) -> &'b [u8] {
        self.data
    }
}

fn main() {
    let config = String::from("debug=true");
    let data = vec![1, 2, 3, 4, 5];
    
    let ctx = Context::new(&config, &data);
    
    println!("Config: {}", ctx.get_config());
    println!("Data: {:?}", ctx.get_data());
}
```

`config`와 `data`는 서로 다른 라이프타임을 가질 수 있다. 이는 각각 독립적으로 관리될 수 있음을 의미한다.

### 실전 패턴: 빌더와 참조
빌더 패턴에서 참조를 다루는 방법을 보자.

```rust
struct Query<'a> {
    table: &'a str,
    conditions: Vec<(&'a str, &'a str)>,
    limit: Option<usize>,
}

struct QueryBuilder<'a> {
    table: Option<&'a str>,
    conditions: Vec<(&'a str, &'a str)>,
    limit: Option<usize>,
}

impl<'a> QueryBuilder<'a> {
    fn new() -> Self {
        QueryBuilder {
            table: None,
            conditions: Vec::new(),
            limit: None,
        }
    }
    
    fn table(mut self, table: &'a str) -> Self {
        self.table = Some(table);
        self
    }
    
    fn where_clause(mut self, field: &'a str, value: &'a str) -> Self {
        self.conditions.push((field, value));
        self
    }
    
    fn limit(mut self, limit: usize) -> Self {
        self.limit = Some(limit);
        self
    }
    
    fn build(self) -> Option<Query<'a>> {
        Some(Query {
            table: self.table?,
            conditions: self.conditions,
            limit: self.limit,
        })
    }
}

fn main() {
    let table_name = "users";
    let field = "status";
    let value = "active";
    
    let query = QueryBuilder::new()
        .table(table_name)
        .where_clause(field, value)
        .limit(10)
        .build()
        .unwrap();
    
    println!("Table: {}", query.table);
    println!("Conditions: {:?}", query.conditions);
}
```

빌더의 모든 참조가 같은 라이프타임 `'a`를 공유한다. 이는 모든 입력이 같은 스코프에서 와야 함을 의미한다.
  

## 6.2 여러 라이프타임이 얽힌 상황

### 문제 상황: 두 참조를 비교하여 반환
두 개의 참조 중 하나를 선택하여 반환하는 함수는 라이프타임이 복잡해진다.

```rust
// 컴파일 실패!
fn longest(x: &str, y: &str) -> &str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}
```

컴파일러는 반환되는 참조가 `x`에서 왔는지 `y`에서 왔는지 알 수 없다. 따라서 라이프타임을 명시해야 한다.

```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}

fn main() {
    let string1 = String::from("long string is long");
    let result;
    
    {
        let string2 = String::from("xyz");
        result = longest(&string1, &string2);
        println!("Longest: {}", result);  // OK: result는 이 블록 안에서만 사용
    }
    
    // println!("{}", result);  // 에러: string2가 이미 drop됨
}
```

반환값의 라이프타임은 두 입력 중 짧은 쪽에 맞춰진다. `string2`가 먼저 drop되므로 `result`는 그 블록 밖에서 사용할 수 없다.

### 실전 예제: 구성 병합
여러 설정 소스를 병합하는 예제다.

```rust
struct Config<'a> {
    values: Vec<(&'a str, &'a str)>,
}

impl<'a> Config<'a> {
    fn new() -> Self {
        Config { values: Vec::new() }
    }
    
    fn add(&mut self, key: &'a str, value: &'a str) {
        self.values.push((key, value));
    }
    
    fn get(&self, key: &str) -> Option<&'a str> {
        self.values.iter()
            .find(|(k, _)| *k == key)
            .map(|(_, v)| *v)
    }
    
    fn merge(&mut self, other: &Config<'a>) {
        for &(key, value) in &other.values {
            // 기존 값이 없으면 추가
            if self.get(key).is_none() {
                self.add(key, value);
            }
        }
    }
}

fn main() {
    let mut defaults = Config::new();
    defaults.add("host", "localhost");
    defaults.add("port", "8080");
    
    let mut user_config = Config::new();
    user_config.add("port", "3000");
    user_config.add("debug", "true");
    
    user_config.merge(&defaults);
    
    println!("port: {:?}", user_config.get("port"));  // Some("3000")
    println!("host: {:?}", user_config.get("host"));  // Some("localhost")
    println!("debug: {:?}", user_config.get("debug"));  // Some("true")
}
```

모든 설정 키와 값이 같은 라이프타임을 공유한다. 병합할 때 참조가 복사되므로 원본 데이터가 살아있어야 한다.

### 다른 라이프타임을 가진 경우
때로는 입력과 출력의 라이프타임이 달라야 한다.

```rust
struct Container<'a> {
    data: &'a str,
}

impl<'a> Container<'a> {
    fn get_prefix<'b>(&'b self, len: usize) -> &'a str
    where
        'a: 'b,  // 'a는 'b보다 오래 살아야 함
    {
        let end = len.min(self.data.len());
        &self.data[..end]
    }
}

fn main() {
    let text = String::from("Hello, World!");
    let container = Container { data: &text };
    
    let prefix = container.get_prefix(5);
    println!("Prefix: {}", prefix);
}
```

`'a: 'b`는 라이프타임 바운드로, "`'a`는 적어도 `'b`만큼 살아야 한다"는 의미다. 반환값은 `self`의 라이프타임이 아니라 원본 데이터의 라이프타임을 따른다.

### 복잡한 예제: 캐시 시스템
참조를 캐싱하는 시스템을 만들어보자.

```rust
use std::collections::HashMap;

struct Cache<'a> {
    storage: HashMap<String, &'a str>,
}

impl<'a> Cache<'a> {
    fn new() -> Self {
        Cache {
            storage: HashMap::new(),
        }
    }
    
    fn insert(&mut self, key: String, value: &'a str) {
        self.storage.insert(key, value);
    }
    
    fn get(&self, key: &str) -> Option<&'a str> {
        self.storage.get(key).copied()
    }
    
    fn get_or_default(&self, key: &str, default: &'a str) -> &'a str {
        self.storage.get(key).copied().unwrap_or(default)
    }
}

struct DataStore {
    permanent_data: Vec<String>,
}

impl DataStore {
    fn new() -> Self {
        DataStore {
            permanent_data: vec![
                "value1".to_string(),
                "value2".to_string(),
                "value3".to_string(),
            ],
        }
    }
    
    fn build_cache(&self) -> Cache {
        let mut cache = Cache::new();
        
        for (idx, data) in self.permanent_data.iter().enumerate() {
            cache.insert(format!("key{}", idx), data);
        }
        
        cache
    }
}

fn main() {
    let store = DataStore::new();
    let cache = store.build_cache();
    
    if let Some(value) = cache.get("key0") {
        println!("Found: {}", value);
    }
    
    let default = "default_value";
    println!("With default: {}", cache.get_or_default("key10", default));
}
```

`Cache`의 라이프타임은 `DataStore`의 `permanent_data`에 묶인다. `Cache`는 `DataStore`보다 오래 살 수 없다.

### 라이프타임 서브타이핑
더 긴 라이프타임은 더 짧은 라이프타임으로 강제될 수 있다.

```rust
fn choose_first<'a, 'b>(x: &'a str, _y: &'b str) -> &'a str {
    x
}

fn main() {
    let string1 = String::from("first");
    let result;
    
    {
        let string2 = String::from("second");
        result = choose_first(&string1, &string2);
    }
    
    // string2는 drop되었지만 result는 여전히 유효
    // result는 string1만 참조하므로
    println!("{}", result);
}
```

함수 시그니처에서 반환값이 `'a`만 사용하므로, `'b`와는 무관하다.
   

## 6.3 메서드 반환값의 라이프타임

### 문제 상황: 메서드가 참조를 반환할 때
메서드가 필드의 참조를 반환하면 라이프타임이 자동으로 추론된다.

```rust
struct Document {
    title: String,
    content: String,
}

impl Document {
    fn get_title(&self) -> &str {
        &self.title
    }
    
    fn get_content(&self) -> &str {
        &self.content
    }
}
```

이것은 라이프타임 엘리전 덕분에 작동한다. 실제로는 이렇게 해석된다:

```rust
impl Document {
    fn get_title<'a>(&'a self) -> &'a str {
        &self.title
    }
}
```

### 복잡한 케이스: 여러 참조 결합
메서드가 여러 필드를 조합하여 반환하면 어떻게 될까?

```rust
struct Article {
    title: String,
    author: String,
    content: String,
}

impl Article {
    // 컴파일 실패!
    fn get_header(&self) -> (&str, &str) {
        (&self.title, &self.author)
    }
    
    // 제대로 명시
    fn get_header_explicit<'a>(&'a self) -> (&'a str, &'a str) {
        (&self.title, &self.author)
    }
    
    fn select_field(&self, use_title: bool) -> &str {
        if use_title {
            &self.title
        } else {
            &self.author
        }
    }
}

fn main() {
    let article = Article {
        title: "Rust Lifetimes".to_string(),
        author: "Rustacean".to_string(),
        content: "...".to_string(),
    };
    
    let (title, author) = article.get_header_explicit();
    println!("{} by {}", title, author);
    
    let field = article.select_field(true);
    println!("{}", field);
}
```

엘리전 규칙이 적용되므로 보통은 명시하지 않아도 된다.

### 실전 예제: 파서 상태 관리
파서가 내부 상태를 참조로 노출하는 경우다.

```rust
struct Parser<'a> {
    input: &'a str,
    position: usize,
    errors: Vec<String>,
}

impl<'a> Parser<'a> {
    fn new(input: &'a str) -> Self {
        Parser {
            input,
            position: 0,
            errors: Vec::new(),
        }
    }
    
    fn current_slice(&self) -> &'a str {
        &self.input[self.position..]
    }
    
    fn peek_window(&self, size: usize) -> &'a str {
        let end = (self.position + size).min(self.input.len());
        &self.input[self.position..end]
    }
    
    fn consume(&mut self, n: usize) -> &'a str {
        let start = self.position;
        let end = (self.position + n).min(self.input.len());
        self.position = end;
        &self.input[start..end]
    }
    
    fn get_errors(&self) -> &[String] {
        &self.errors
    }
    
    fn add_error(&mut self, msg: String) {
        self.errors.push(msg);
    }
}

fn main() {
    let source = "fn main() { }";
    let mut parser = Parser::new(source);
    
    let token = parser.consume(2);
    println!("Consumed: '{}'", token);
    
    let next = parser.peek_window(4);
    println!("Next 4 chars: '{}'", next);
    
    parser.add_error("Unexpected token".to_string());
    
    if !parser.get_errors().is_empty() {
        println!("Errors: {:?}", parser.get_errors());
    }
}
```

`current_slice`, `peek_window`, `consume`은 모두 원본 입력의 슬라이스를 반환한다. 이들의 라이프타임은 `'a`로, `Parser`가 살아있는 한 유효하다.

### 정적 라이프타임 반환
때로는 함수가 `'static` 라이프타임을 가진 참조를 반환한다.

```rust
struct Constants;

impl Constants {
    fn get_version() -> &'static str {
        "1.0.0"
    }
    
    fn get_default_config() -> &'static str {
        "default.conf"
    }
    
    fn get_error_messages() -> &'static [&'static str] {
        &[
            "File not found",
            "Permission denied",
            "Invalid input",
        ]
    }
}

fn main() {
    let version = Constants::get_version();
    println!("Version: {}", version);
    
    // 'static이므로 프로그램 전체에서 유효
    let msg = {
        let msgs = Constants::get_error_messages();
        msgs[0]
    };
    println!("Error: {}", msg);
}
```

문자열 리터럴과 정적 데이터는 `'static` 라이프타임을 가진다. 프로그램 전체 기간 동안 유효하다.

### 복잡한 예제: 뷰 시스템
데이터의 다양한 뷰를 제공하는 시스템이다.

```rust
struct Database {
    records: Vec<String>,
}

struct RecordView<'a> {
    record: &'a str,
    start: usize,
    end: usize,
}

impl<'a> RecordView<'a> {
    fn new(record: &'a str, start: usize, end: usize) -> Self {
        RecordView { record, start, end }
    }
    
    fn as_str(&self) -> &'a str {
        let end = self.end.min(self.record.len());
        &self.record[self.start..end]
    }
    
    fn subview(&self, offset: usize, len: usize) -> RecordView<'a> {
        let new_start = (self.start + offset).min(self.end);
        let new_end = (new_start + len).min(self.end);
        RecordView::new(self.record, new_start, new_end)
    }
}

impl Database {
    fn new() -> Self {
        Database {
            records: vec![
                "Record 1: Important data".to_string(),
                "Record 2: More data".to_string(),
                "Record 3: Even more data".to_string(),
            ],
        }
    }
    
    fn get_view(&self, index: usize) -> Option<RecordView> {
        self.records.get(index)
            .map(|r| RecordView::new(r, 0, r.len()))
    }
    
    fn get_partial_view(&self, index: usize, start: usize, len: usize) -> Option<RecordView> {
        self.records.get(index)
            .map(|r| RecordView::new(r, start, start + len))
    }
}

fn main() {
    let db = Database::new();
    
    if let Some(view) = db.get_view(0) {
        println!("Full view: {}", view.as_str());
        
        let subview = view.subview(0, 8);
        println!("Subview: {}", subview.as_str());
    }
    
    if let Some(partial) = db.get_partial_view(1, 7, 10) {
        println!("Partial: {}", partial.as_str());
    }
}
```

`RecordView`는 원본 레코드를 참조하고, 서브뷰는 같은 원본을 참조한다. 모든 뷰의 라이프타임이 `Database`에 묶인다.
  

## 6.4 라이프타임 엘리전 규칙 활용

### 엘리전 규칙이란
컴파일러는 세 가지 규칙으로 라이프타임을 자동 추론한다.

1. 각 참조 매개변수는 고유한 라이프타임을 얻는다
2. 입력 라이프타임이 정확히 하나면, 모든 출력에 할당된다
3. 메서드에서 `&self`나 `&mut self`가 있으면, 그 라이프타임이 모든 출력에 할당된다

```rust
// 규칙 1 적용
fn example1(x: &str, y: &str) {
    // 실제로는: fn example1<'a, 'b>(x: &'a str, y: &'b str)
}

// 규칙 2 적용
fn example2(x: &str) -> &str {
    // 실제로는: fn example2<'a>(x: &'a str) -> &'a str
    x
}

// 규칙 3 적용
struct Data {
    value: String,
}

impl Data {
    fn get_value(&self) -> &str {
        // 실제로는: fn get_value<'a>(&'a self) -> &'a str
        &self.value
    }
}
```

이 규칙들 덕분에 대부분의 경우 라이프타임을 명시하지 않아도 된다.

### 엘리전이 적용되지 않는 경우
규칙으로 해결되지 않으면 명시해야 한다.

```rust
// 에러: 어느 입력의 라이프타임을 반환값에 사용할지 모호
fn first_word(x: &str, y: &str) -> &str {
    x.split_whitespace().next().unwrap_or("")
}

// 명시적으로 지정
fn first_word_fixed<'a>(x: &'a str, _y: &str) -> &'a str {
    x.split_whitespace().next().unwrap_or("")
}
```

여러 입력이 있고 어느 것이 반환될지 컴파일러가 알 수 없으면 에러가 발생한다.

### 실전 패턴: 변환 체인
여러 단계의 변환을 거치는 파이프라인이다.

```rust
struct TextProcessor;

impl TextProcessor {
    fn trim(input: &str) -> &str {
        input.trim()
    }
    
    fn to_lowercase_owned(input: &str) -> String {
        input.to_lowercase()
    }
    
    fn extract_first_line(input: &str) -> &str {
        input.lines().next().unwrap_or("")
    }
    
    fn process(input: &str) -> String {
        let trimmed = Self::trim(input);
        let first_line = Self::extract_first_line(trimmed);
        Self::to_lowercase_owned(first_line)
    }
}

fn main() {
    let text = "  HELLO WORLD\n  Second line";
    let result = TextProcessor::process(text);
    println!("Result: '{}'", result);
}
```

각 메서드는 엘리전 규칙 2를 따른다. 입력이 하나이므로 자동으로 그 라이프타임이 출력에 할당된다.

### 스트럭처 메서드에서의 엘리전

```rust
struct DataProcessor {
    prefix: String,
}

impl DataProcessor {
    fn add_prefix(&self, input: &str) -> String {
        format!("{}{}", self.prefix, input)
    }
    
    fn get_prefix(&self) -> &str {
        &self.prefix
    }
    
    fn process_with_suffix<'a>(&self, input: &'a str, suffix: &str) -> String {
        format!("{}{}{}", self.prefix, input, suffix)
    }
}

fn main() {
    let processor = DataProcessor {
        prefix: "[LOG] ".to_string(),
    };
    
    let result = processor.add_prefix("Message");
    println!("{}", result);
    
    let prefix_ref = processor.get_prefix();
    println!("Prefix: {}", prefix_ref);
}
```

`get_prefix`는 규칙 3에 의해 `self`의 라이프타임을 반환값에 할당한다. `process_with_suffix`는 `String`을 반환하므로 라이프타임이 무관하다.

### 복잡한 예제: 문서 분석기
실전적인 텍스트 분석 도구다.

```rust
struct Document {
    content: String,
}

struct Analysis<'a> {
    document: &'a Document,
    word_count: usize,
    line_count: usize,
}

impl Document {
    fn new(content: String) -> Self {
        Document { content }
    }
    
    fn get_content(&self) -> &str {
        &self.content
    }
    
    fn get_line(&self, index: usize) -> Option<&str> {
        self.content.lines().nth(index)
    }
    
    fn analyze(&self) -> Analysis {
        Analysis {
            document: self,
            word_count: self.content.split_whitespace().count(),
            line_count: self.content.lines().count(),
        }
    }
}

impl<'a> Analysis<'a> {
    fn get_longest_line(&self) -> Option<&'a str> {
        self.document.content
            .lines()
            .max_by_key(|line| line.len())
    }
    
    fn get_words(&self) -> impl Iterator<Item = &'a str> {
        self.document.content.split_whitespace()
    }
    
    fn print_summary(&self) {
        println!("Word count: {}", self.word_count);
        println!("Line count: {}", self.line_count);
        
        if let Some(longest) = self.get_longest_line() {
            println!("Longest line: '{}'", longest);
        }
    }
}

fn main() {
    let doc = Document::new(
        "First line\n\
         This is a longer second line\n\
         Third"
            .to_string(),
    );
    
    let analysis = doc.analyze();
    analysis.print_summary();
    
    println!("\nWords:");
    for word in analysis.get_words() {
        println!("  {}", word);
    }
}
```

`Analysis`는 `Document`를 참조하고, 그 메서드들은 문서 내용의 슬라이스를 반환한다. 엘리전 규칙 덕분에 라이프타임 표기가 최소화된다.

### 고급 패턴: 라이프타임 바운드
제네릭과 라이프타임을 결합하는 경우다.

```rust
use std::fmt::Display;

struct Wrapper<'a, T>
where
    T: Display,
{
    value: &'a T,
}

impl<'a, T> Wrapper<'a, T>
where
    T: Display,
{
    fn new(value: &'a T) -> Self {
        Wrapper { value }
    }
    
    fn display(&self) {
        println!("Wrapped: {}", self.value);
    }
    
    fn get(&self) -> &'a T {
        self.value
    }
}

fn print_wrapper<'a, T>(wrapper: &Wrapper<'a, T>)
where
    T: Display,
{
    wrapper.display();
}

fn main() {
    let number = 42;
    let wrapper = Wrapper::new(&number);
    
    print_wrapper(&wrapper);
    
    let value_ref = wrapper.get();
    println!("Value: {}", value_ref);
}
```

`T: Display`는 트레이트 바운드고, `'a`는 라이프타임 매개변수다. 둘을 함께 사용하여 안전하면서도 유연한 코드를 만든다.
 
### 실전 조언

1. 대부분의 경우 엘리전 규칙이 적용되므로 라이프타임을 명시하지 않아도 된다
2. 컴파일러가 에러를 낸다면 그때 추가한다
3. 참조를 반환하는 함수는 입력의 라이프타임을 따르는지 확인한다
4. 구조체에 참조를 저장하면 그 구조체 자체가 라이프타임을 가진다
5. 가능하면 소유권을 이동하거나 복제하여 라이프타임 문제를 회피한다

라이프타임은 Rust의 강력한 안전성을 제공하지만, 처음엔 어렵게 느껴진다. 하지만 패턴을 익히고 엘리전 규칙을 활용하면 대부분의 경우 자연스럽게 해결된다. 라이프타임이 복잡해지면 설계를 다시 검토할 신호일 수 있다.
