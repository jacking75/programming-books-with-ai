## 2.1 self, &self, &mut self의 선택 기준
C++에서는 멤버 함수가 객체를 수정하는지 여부를 `const` 키워드로 표시한다. Rust에서는 `self`, `&self`, `&mut self`라는 세 가지 리시버 타입으로 더 명확하게 구분한다.

**기본 규칙:**
- `self`: 소유권을 가져온다 (값 이동)
- `&self`: 불변 참조로 빌린다
- `&mut self`: 가변 참조로 빌린다

```rust
struct Counter {
    count: i32,
}

impl Counter {
    // 새 인스턴스 생성 - self 없음 (연관 함수)
    fn new() -> Self {
        Counter { count: 0 }
    }
    
    // 읽기 전용 - &self
    fn get(&self) -> i32 {
        self.count
    }
    
    // 수정 - &mut self
    fn increment(&mut self) {
        self.count += 1;
    }
    
    // 소비 - self
    fn consume(self) -> i32 {
        println!("Counter is being destroyed");
        self.count
    }
}

fn main() {
    let mut counter = Counter::new();
    
    println!("Count: {}", counter.get());  // &self
    counter.increment();                    // &mut self
    println!("Count: {}", counter.get());  // &self
    
    let final_count = counter.consume();   // self - 소유권 이동
    // counter.increment();  // 컴파일 에러! counter는 이동되었다
}
```

**C++과의 비교:**

```cpp
class Counter {
private:
    int count;
    
public:
    Counter() : count(0) {}
    
    int get() const { return count; }        // &self에 해당
    void increment() { count++; }            // &mut self에 해당
    int consume() && { return count; }       // self에 해당 (rvalue ref)
};
```

**선택 기준:**

1. **`&self`를 사용하는 경우:**
   - 데이터를 읽기만 한다
   - 여러 곳에서 동시에 호출될 수 있어야 한다
   - 객체를 소비하지 않는다

2. **`&mut self`를 사용하는 경우:**
   - 내부 상태를 변경한다
   - 배타적 접근이 필요하다

3. **`self`를 사용하는 경우:**
   - 객체를 변환하거나 소비한다
   - 빌더 패턴의 마지막 단계
   - 리소스를 명시적으로 해제한다

**실전 예제: 잘못된 선택의 결과**

```rust
struct Database {
    connection_string: String,
    connected: bool,
}

impl Database {
    // 잘못된 예: connect가 self를 소비한다
    fn connect_wrong(self) -> bool {
        println!("Connecting to {}", self.connection_string);
        true
    }
    
    // 올바른 예: &mut self로 수정한다
    fn connect(&mut self) -> bool {
        println!("Connecting to {}", self.connection_string);
        self.connected = true;
        true
    }
    
    fn query(&self, sql: &str) -> Vec<String> {
        if !self.connected {
            panic!("Not connected!");
        }
        println!("Executing: {}", sql);
        vec![]
    }
}

fn main() {
    let mut db = Database {
        connection_string: String::from("localhost:5432"),
        connected: false,
    };
    
    // 잘못된 버전 사용 시:
    // db.connect_wrong();  // db가 소비된다
    // db.query("SELECT *");  // 컴파일 에러!
    
    // 올바른 버전:
    db.connect();
    db.query("SELECT * FROM users");  // OK
}
```
  

## 2.2 메서드 체이닝과 가변 빌림 충돌
메서드 체이닝은 유창한 인터페이스를 만드는 일반적인 패턴이다. 하지만 Rust에서는 빌림 규칙 때문에 주의가 필요하다.

**문제 상황:**

```rust
struct Builder {
    name: String,
    age: u32,
}

impl Builder {
    fn new() -> Self {
        Builder {
            name: String::new(),
            age: 0,
        }
    }
    
    // 이렇게 하면 체이닝이 안 된다
    fn set_name_wrong(&mut self, name: String) {
        self.name = name;
    }
    
    fn set_age_wrong(&mut self, age: u32) {
        self.age = age;
    }
}

fn main() {
    let mut builder = Builder::new();
    
    // 체이닝 불가
    builder.set_name_wrong(String::from("Alice"));
    builder.set_age_wrong(30);
}
```

**해결책 1: &mut self를 반환한다**

```rust
impl Builder {
    fn set_name(&mut self, name: String) -> &mut Self {
        self.name = name;
        self
    }
    
    fn set_age(&mut self, age: u32) -> &mut Self {
        self.age = age;
        self
    }
    
    fn build(&self) -> String {
        format!("{} is {} years old", self.name, self.age)
    }
}

fn main() {
    let mut builder = Builder::new();
    
    // 체이닝 가능!
    let result = builder
        .set_name(String::from("Alice"))
        .set_age(30)
        .build();
    
    println!("{}", result);
}
```

**해결책 2: self를 소비하고 반환한다 (빌더 패턴)**

```rust
struct PersonBuilder {
    name: Option<String>,
    age: Option<u32>,
}

impl PersonBuilder {
    fn new() -> Self {
        PersonBuilder {
            name: None,
            age: None,
        }
    }
    
    // self를 소비하고 반환한다
    fn name(mut self, name: String) -> Self {
        self.name = Some(name);
        self
    }
    
    fn age(mut self, age: u32) -> Self {
        self.age = Some(age);
        self
    }
    
    // 최종적으로 Person을 만든다
    fn build(self) -> Person {
        Person {
            name: self.name.expect("name is required"),
            age: self.age.expect("age is required"),
        }
    }
}

struct Person {
    name: String,
    age: u32,
}

fn main() {
    let person = PersonBuilder::new()
        .name(String::from("Bob"))
        .age(25)
        .build();
    
    println!("{} is {}", person.name, person.age);
}
```

이 패턴의 장점은 각 단계에서 빌더가 소비되므로, 잘못된 순서로 호출하거나 중간 상태를 저장하는 실수를 방지할 수 있다는 것이다.

**실전 예제: HTTP 클라이언트 빌더**

```rust
struct HttpRequest {
    url: String,
    method: String,
    headers: Vec<(String, String)>,
    body: Option<String>,
}

impl HttpRequest {
    fn new(url: String) -> Self {
        HttpRequest {
            url,
            method: String::from("GET"),
            headers: Vec::new(),
            body: None,
        }
    }
    
    fn method(mut self, method: String) -> Self {
        self.method = method;
        self
    }
    
    fn header(mut self, key: String, value: String) -> Self {
        self.headers.push((key, value));
        self
    }
    
    fn body(mut self, body: String) -> Self {
        self.body = Some(body);
        self
    }
    
    fn send(self) -> String {
        println!("Sending {} request to {}", self.method, self.url);
        for (key, value) in &self.headers {
            println!("  {}: {}", key, value);
        }
        if let Some(body) = &self.body {
            println!("  Body: {}", body);
        }
        String::from("200 OK")
    }
}

fn main() {
    let response = HttpRequest::new(String::from("https://api.example.com"))
        .method(String::from("POST"))
        .header(String::from("Content-Type"), String::from("application/json"))
        .header(String::from("Authorization"), String::from("Bearer token123"))
        .body(String::from(r#"{"key": "value"}"#))
        .send();
    
    println!("Response: {}", response);
}
```
  

## 2.3 여러 메서드를 동시에 호출할 때의 문제
C++에서는 같은 객체의 여러 메서드를 자유롭게 호출할 수 있다. Rust에서는 빌림 규칙 때문에 제약이 있다.

**문제 예제 1: 가변 참조의 중복**

```rust
struct GameCharacter {
    health: i32,
    mana: i32,
    position: (f32, f32),
}

impl GameCharacter {
    fn take_damage(&mut self, amount: i32) {
        self.health -= amount;
        println!("Health: {}", self.health);
    }
    
    fn use_mana(&mut self, amount: i32) {
        self.mana -= amount;
        println!("Mana: {}", self.mana);
    }
    
    fn is_alive(&self) -> bool {
        self.health > 0
    }
    
    // 문제가 되는 메서드
    fn attack_wrong(&mut self, target: &mut GameCharacter) {
        if self.is_alive() && target.is_alive() {
            self.use_mana(10);     // self에 대한 가변 빌림
            target.take_damage(20); // target에 대한 가변 빌림
        }
    }
}

fn main() {
    let mut player = GameCharacter {
        health: 100,
        mana: 50,
        position: (0.0, 0.0),
    };
    
    let mut enemy = GameCharacter {
        health: 80,
        mana: 30,
        position: (10.0, 10.0),
    };
    
    player.attack_wrong(&mut enemy);  // OK
    
    // 하지만 자기 자신을 공격하려 하면?
    // player.attack_wrong(&mut player);  // 컴파일 에러!
    // cannot borrow `player` as mutable more than once at a time
}
```

**해결책: 인터페이스를 재설계한다**

```rust
impl GameCharacter {
    fn attack(&mut self, damage: i32, mana_cost: i32) -> Option<i32> {
        if self.health <= 0 || self.mana < mana_cost {
            return None;
        }
        
        self.mana -= mana_cost;
        println!("Used {} mana, {} remaining", mana_cost, self.mana);
        Some(damage)
    }
    
    fn receive_damage(&mut self, amount: i32) {
        self.health -= amount;
        println!("Took {} damage, {} health remaining", amount, self.health);
    }
}

fn combat(attacker: &mut GameCharacter, defender: &mut GameCharacter) {
    if let Some(damage) = attacker.attack(20, 10) {
        defender.receive_damage(damage);
    }
}

fn main() {
    let mut player = GameCharacter {
        health: 100,
        mana: 50,
        position: (0.0, 0.0),
    };
    
    let mut enemy = GameCharacter {
        health: 80,
        mana: 30,
        position: (10.0, 10.0),
    };
    
    combat(&mut player, &mut enemy);
}
```

**문제 예제 2: 게터 체이닝 문제**

```rust
struct User {
    name: String,
    email: String,
    is_admin: bool,
}

impl User {
    fn get_name(&self) -> &String {
        &self.name
    }
    
    fn get_email(&self) -> &String {
        &self.email
    }
    
    fn set_email(&mut self, email: String) {
        self.email = email;
    }
    
    // 이메일을 변경하면서 이름을 확인하려 한다
    fn update_email_wrong(&mut self) {
        let name = self.get_name();  // 불변 빌림
        let new_email = format!("{}@newdomain.com", name);
        self.set_email(new_email);   // 컴파일 에러! 가변 빌림 시도
        // cannot borrow `*self` as mutable because it is also borrowed as immutable
    }
}
```

**해결책: 필요한 데이터를 먼저 복사한다**

```rust
impl User {
    fn update_email(&mut self) {
        // 불변 빌림을 먼저 끝낸다
        let new_email = format!("{}@newdomain.com", self.name);
        // 이제 가변 빌림이 가능하다
        self.set_email(new_email);
    }
    
    // 또는 직접 접근한다
    fn update_email_direct(&mut self) {
        self.email = format!("{}@newdomain.com", self.name);
    }
}

fn main() {
    let mut user = User {
        name: String::from("alice"),
        email: String::from("alice@example.com"),
        is_admin: false,
    };
    
    user.update_email();
    println!("New email: {}", user.email);
}
```

**문제 예제 3: 메서드 내에서 다른 메서드 호출**

```rust
struct Document {
    title: String,
    content: String,
    word_count: usize,
}

impl Document {
    fn get_title(&self) -> &str {
        &self.title
    }
    
    fn get_content(&self) -> &str {
        &self.content
    }
    
    fn update_word_count(&mut self) {
        self.word_count = self.content.split_whitespace().count();
    }
    
    // 문제가 되는 메서드
    fn set_content_wrong(&mut self, content: String) {
        self.content = content;
        self.update_word_count();  // 내부적으로는 문제없다
    }
    
    // 하지만 이런 경우는 문제가 된다
    fn set_content_with_title_wrong(&mut self) {
        let title = self.get_title();  // 불변 빌림
        self.content = format!("Title: {}\nContent...", title);
        self.update_word_count();  // 컴파일 에러!
        // cannot borrow `*self` as mutable because it is also borrowed as immutable
    }
}
```

**해결책: 빌림을 명시적으로 종료한다**

```rust
impl Document {
    fn set_content_with_title(&mut self) {
        // 방법 1: title을 복사한다
        let title = self.title.clone();
        self.content = format!("Title: {}\nContent...", title);
        self.update_word_count();
    }
    
    fn set_content_with_title_v2(&mut self) {
        // 방법 2: 임시 변수로 빌림을 끝낸다
        let new_content = {
            let title = &self.title;
            format!("Title: {}\nContent...", title)
        };  // title 빌림이 여기서 끝난다
        
        self.content = new_content;
        self.update_word_count();
    }
}
```
  

## 2.4 내부 가변성이 필요한 순간
때로는 불변 참조를 통해 객체의 내부 상태를 변경해야 할 때가 있다. 이는 캐싱, 지연 초기화, 참조 카운팅 등에서 흔히 나타난다.

**문제 상황: 캐싱**

```rust
struct ExpensiveCalculator {
    input: i32,
    cached_result: Option<i32>,
}

impl ExpensiveCalculator {
    fn new(input: i32) -> Self {
        ExpensiveCalculator {
            input,
            cached_result: None,
        }
    }
    
    // 문제: &self로는 캐시를 업데이트할 수 없다
    fn calculate_wrong(&self) -> i32 {
        if let Some(result) = self.cached_result {
            return result;
        }
        
        // 무거운 계산
        let result = self.input * self.input;
        // self.cached_result = Some(result);  // 컴파일 에러!
        // cannot assign to `self.cached_result` which is behind a `&` reference
        
        result
    }
}
```

**해결책 1: &mut self로 변경 (항상 가능하지는 않다)**

```rust
impl ExpensiveCalculator {
    fn calculate_mut(&mut self) -> i32 {
        if let Some(result) = self.cached_result {
            return result;
        }
        
        let result = self.input * self.input;
        self.cached_result = Some(result);
        result
    }
}

fn main() {
    let mut calc = ExpensiveCalculator::new(5);
    println!("{}", calc.calculate_mut());  // 계산
    println!("{}", calc.calculate_mut());  // 캐시에서 가져옴
}
```

하지만 만약 여러 곳에서 동시에 `calculate`를 호출해야 한다면? `&mut self`는 배타적이므로 불가능하다.

**해결책 2: Cell 사용 (Copy 타입만 가능)**

```rust
use std::cell::Cell;

struct ExpensiveCalculator {
    input: i32,
    cached_result: Cell<Option<i32>>,
}

impl ExpensiveCalculator {
    fn new(input: i32) -> Self {
        ExpensiveCalculator {
            input,
            cached_result: Cell::new(None),
        }
    }
    
    fn calculate(&self) -> i32 {
        if let Some(result) = self.cached_result.get() {
            return result;
        }
        
        let result = self.input * self.input;
        self.cached_result.set(Some(result));
        result
    }
}

fn main() {
    let calc = ExpensiveCalculator::new(5);
    println!("{}", calc.calculate());  // 계산
    println!("{}", calc.calculate());  // 캐시에서 가져옴
}
```

`Cell`은 `Copy` 타입(여기서는 `Option<i32>`)에 대해서만 동작한다.

**해결책 3: RefCell 사용 (모든 타입 가능, 런타임 검사)**

```rust
use std::cell::RefCell;

struct DocumentCache {
    file_path: String,
    cached_content: RefCell<Option<String>>,
}

impl DocumentCache {
    fn new(file_path: String) -> Self {
        DocumentCache {
            file_path,
            cached_content: RefCell::new(None),
        }
    }
    
    fn get_content(&self) -> String {
        // 캐시가 있는지 확인
        if let Some(content) = self.cached_content.borrow().as_ref() {
            return content.clone();
        }
        
        // 파일에서 읽는다 (시뮬레이션)
        println!("Reading file: {}", self.file_path);
        let content = format!("Content of {}", self.file_path);
        
        // 캐시에 저장
        *self.cached_content.borrow_mut() = Some(content.clone());
        
        content
    }
    
    fn invalidate_cache(&self) {
        *self.cached_content.borrow_mut() = None;
    }
}

fn main() {
    let cache = DocumentCache::new(String::from("data.txt"));
    
    println!("{}", cache.get_content());  // 파일 읽음
    println!("{}", cache.get_content());  // 캐시에서 가져옴
    
    cache.invalidate_cache();
    println!("{}", cache.get_content());  // 다시 파일 읽음
}
```

**RefCell의 런타임 패닉 주의:**

```rust
use std::cell::RefCell;

struct Container {
    value: RefCell<i32>,
}

fn main() {
    let container = Container {
        value: RefCell::new(42),
    };
    
    let borrow1 = container.value.borrow_mut();  // 가변 빌림
    // let borrow2 = container.value.borrow_mut();  // 런타임 패닉!
    // thread 'main' panicked at 'already borrowed: BorrowMutError'
    
    drop(borrow1);  // 명시적으로 빌림을 해제
    let borrow2 = container.value.borrow_mut();  // OK
}
```

**실전 예제: 지연 초기화**

```rust
use std::cell::OnceCell;

struct LazyDatabase {
    config: String,
    connection: OnceCell<DatabaseConnection>,
}

struct DatabaseConnection {
    handle: String,
}

impl LazyDatabase {
    fn new(config: String) -> Self {
        LazyDatabase {
            config,
            connection: OnceCell::new(),
        }
    }
    
    fn get_connection(&self) -> &DatabaseConnection {
        self.connection.get_or_init(|| {
            println!("Initializing database connection...");
            DatabaseConnection {
                handle: format!("Connection to {}", self.config),
            }
        })
    }
    
    fn query(&self, sql: &str) -> Vec<String> {
        let conn = self.get_connection();
        println!("Querying {} with: {}", conn.handle, sql);
        vec![]
    }
}

fn main() {
    let db = LazyDatabase::new(String::from("localhost:5432"));
    
    // 첫 호출에서 연결 초기화
    db.query("SELECT * FROM users");
    
    // 이후 호출에서는 기존 연결 사용
    db.query("SELECT * FROM orders");
}
```

**내부 가변성을 선택하는 기준:**

1. **Cell**: 
   - Copy 타입만 가능
   - 오버헤드 없음
   - 참조를 반환할 수 없음

2. **RefCell**:
   - 모든 타입 가능
   - 런타임 빌림 검사
   - 참조를 반환할 수 있음
   - 싱글스레드만 안전

3. **Mutex/RwLock**:
   - 멀티스레드 환경
   - 성능 오버헤드 있음
   - 7장에서 상세히 다룸

4. **OnceCell**:
   - 한 번만 초기화
   - 지연 초기화 패턴
   - 오버헤드 최소

**2장 요약**

이 장에서는 구조체와 메서드에서 발생하는 실전 빌림 문제들을 다뤘다:

- **self 타입 선택**: `self`, `&self`, `&mut self`의 의미와 선택 기준을 이해했다
- **메서드 체이닝**: `&mut self` 반환 또는 `self` 소비 패턴으로 체이닝을 구현하는 방법을 배웠다
- **동시 메서드 호출**: 빌림 규칙으로 인한 제약과 인터페이스 재설계를 통한 해결책을 살펴봤다
- **내부 가변성**: `Cell`, `RefCell`, `OnceCell`을 사용해 불변 참조를 통한 상태 변경 방법을 익혔다

다음 장에서는 멤버 변수에 직접 접근할 때 발생하는 더 복잡한 빌림 문제들과 분할 빌림 패턴을 다룬다.
