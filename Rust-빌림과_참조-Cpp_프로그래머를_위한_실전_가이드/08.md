# 8장. 실전 리팩토링 사례
이 장에서는 C++에서 자연스럽게 작성했던 코드들을 Rust로 옮길 때 마주치는 전형적인 빌림 문제와 해결책을 다룬다. 실제로 겪게 되는 구체적인 상황들을 통해 Rust다운 설계 방식을 익혀본다.

## 8.1 C++ 스타일 클래스를 Rust로 변환하기
C++에서는 클래스가 자유롭게 멤버를 수정하고 다른 객체를 참조할 수 있다. Rust로 같은 패턴을 구현하려 하면 빌림 검사기와 충돌한다.

### 문제 상황: C++ 스타일의 Player 클래스
먼저 C++로 작성된 전형적인 게임 캐릭터 클래스를 보자.

```cpp
// C++ 버전
class Player {
private:
    std::string name;
    int health;
    int max_health;
    Inventory* inventory;
    
public:
    Player(const std::string& name, int max_health) 
        : name(name), health(max_health), max_health(max_health) {
        inventory = new Inventory();
    }
    
    void take_damage(int damage) {
        health -= damage;
        if (health < 0) health = 0;
        std::cout << name << " took " << damage << " damage. Health: " 
                  << health << "/" << max_health << std::endl;
    }
    
    void heal(int amount) {
        health += amount;
        if (health > max_health) health = max_health;
    }
    
    void use_potion() {
        if (inventory->has_item("health_potion")) {
            inventory->remove_item("health_potion");
            heal(50);
            std::cout << name << " used a health potion!" << std::endl;
        }
    }
    
    bool is_alive() const {
        return health > 0;
    }
};
```

이를 그대로 Rust로 옮기려 하면 문제가 생긴다.

```rust
// 순진한 첫 번째 시도 - 컴파일 실패!
struct Player {
    name: String,
    health: i32,
    max_health: i32,
    inventory: Inventory,
}

impl Player {
    fn new(name: String, max_health: i32) -> Self {
        Player {
            name,
            health: max_health,
            max_health,
            inventory: Inventory::new(),
        }
    }
    
    fn take_damage(&mut self, damage: i32) {
        self.health -= damage;
        if self.health < 0 {
            self.health = 0;
        }
        println!("{} took {} damage. Health: {}/{}", 
                 self.name, damage, self.health, self.max_health);
    }
    
    fn heal(&mut self, amount: i32) {
        self.health += amount;
        if self.health > self.max_health {
            self.health = self.max_health;
        }
    }
    
    // 컴파일 에러!
    fn use_potion(&mut self) {
        if self.inventory.has_item("health_potion") {
            self.inventory.remove_item("health_potion");
            self.heal(50); // 에러: cannot borrow `*self` as mutable more than once
            println!("{} used a health potion!", self.name);
        }
    }
}
```

`use_potion` 메서드에서 문제가 발생한다. `self.inventory`를 통해 이미 `self`를 가변으로 빌렸는데, 다시 `self.heal()`을 호출하려 하기 때문이다.

### 해결책 1: 메서드 분리와 명시적 호출
첫 번째 해결책은 힐링 로직을 직접 구현하는 것이다.

```rust
struct Inventory {
    items: Vec<String>,
}

impl Inventory {
    fn new() -> Self {
        Inventory { items: Vec::new() }
    }
    
    fn add_item(&mut self, item: &str) {
        self.items.push(item.to_string());
    }
    
    fn has_item(&self, item: &str) -> bool {
        self.items.iter().any(|i| i == item)
    }
    
    fn remove_item(&mut self, item: &str) -> bool {
        if let Some(pos) = self.items.iter().position(|i| i == item) {
            self.items.remove(pos);
            true
        } else {
            false
        }
    }
}

struct Player {
    name: String,
    health: i32,
    max_health: i32,
    inventory: Inventory,
}

impl Player {
    fn new(name: String, max_health: i32) -> Self {
        Player {
            name,
            health: max_health,
            max_health,
            inventory: Inventory::new(),
        }
    }
    
    fn take_damage(&mut self, damage: i32) {
        self.health -= damage;
        if self.health < 0 {
            self.health = 0;
        }
        println!("{} took {} damage. Health: {}/{}", 
                 self.name, damage, self.health, self.max_health);
    }
    
    fn heal(&mut self, amount: i32) {
        self.health += amount;
        if self.health > self.max_health {
            self.health = self.max_health;
        }
    }
    
    fn use_potion(&mut self) {
        if self.inventory.remove_item("health_potion") {
            // heal() 메서드를 호출하지 않고 로직을 직접 구현
            self.health += 50;
            if self.health > self.max_health {
                self.health = self.max_health;
            }
            println!("{} used a health potion!", self.name);
        }
    }
    
    fn is_alive(&self) -> bool {
        self.health > 0
    }
}

fn main() {
    let mut player = Player::new("Hero".to_string(), 100);
    player.inventory.add_item("health_potion");
    
    player.take_damage(60);
    player.use_potion();
    println!("Player alive: {}", player.is_alive());
}
```

이 방식은 작동하지만 코드 중복이 발생한다. 더 나은 방법을 찾아보자.

### 해결책 2: 헬퍼 함수로 추출
힐링 로직을 독립적인 함수로 분리한다.

```rust
fn apply_healing(health: &mut i32, max_health: i32, amount: i32) {
    *health += amount;
    if *health > max_health {
        *health = max_health;
    }
}

impl Player {
    fn heal(&mut self, amount: i32) {
        apply_healing(&mut self.health, self.max_health, amount);
    }
    
    fn use_potion(&mut self) {
        if self.inventory.remove_item("health_potion") {
            apply_healing(&mut self.health, self.max_health, 50);
            println!("{} used a health potion!", self.name);
        }
    }
}
```

### 해결책 3: 분할 빌림 활용
Rust는 구조체의 서로 다른 필드를 동시에 가변 빌림할 수 있다는 점을 활용한다.

```rust
impl Player {
    fn use_potion(&mut self) {
        // 분할 빌림: inventory와 나머지 필드들을 분리
        let Player { ref mut inventory, ref mut health, max_health, ref name, .. } = *self;
        
        if inventory.remove_item("health_potion") {
            *health += 50;
            if *health > *max_health {
                *health = *max_health;
            }
            println!("{} used a health potion!", name);
        }
    }
}
```

### 해결책 4: Rust다운 설계로 재구성
가장 좋은 방법은 처음부터 Rust의 철학에 맞게 설계하는 것이다.

```rust
struct Player {
    name: String,
    health: i32,
    max_health: i32,
    inventory: Inventory,
}

impl Player {
    fn new(name: String, max_health: i32) -> Self {
        Player {
            name,
            health: max_health,
            max_health,
            inventory: Inventory::new(),
        }
    }
    
    fn take_damage(&mut self, damage: i32) {
        self.health = (self.health - damage).max(0);
        println!("{} took {} damage. Health: {}/{}", 
                 self.name, damage, self.health, self.max_health);
    }
    
    fn heal(&mut self, amount: i32) {
        self.health = (self.health + amount).min(self.max_health);
    }
    
    // 아이템 사용을 별도 메서드로 분리
    fn use_item(&mut self, item_name: &str) -> bool {
        if !self.inventory.has_item(item_name) {
            return false;
        }
        
        self.inventory.remove_item(item_name);
        
        // 아이템 효과를 매칭으로 처리
        match item_name {
            "health_potion" => {
                self.heal(50);
                println!("{} used a health potion!", self.name);
                true
            }
            "super_potion" => {
                self.heal(100);
                println!("{} used a super potion!", self.name);
                true
            }
            _ => false,
        }
    }
    
    fn is_alive(&self) -> bool {
        self.health > 0
    }
}

fn main() {
    let mut player = Player::new("Hero".to_string(), 100);
    player.inventory.add_item("health_potion");
    player.inventory.add_item("super_potion");
    
    player.take_damage(80);
    player.use_item("health_potion");
    player.take_damage(40);
    player.use_item("super_potion");
    
    println!("Final health: {}/{}", player.health, player.max_health);
}
```
  

## 8.2 게터/세터 패턴 재고하기
C++에서는 캡슐화를 위해 게터/세터를 많이 사용한다. Rust에서는 이 패턴을 다시 생각해볼 필요가 있다.

### C++ 스타일의 게터/세터

```rust
// C++를 직역한 스타일 - 권장하지 않음
struct BankAccount {
    balance: f64,
    owner: String,
}

impl BankAccount {
    fn new(owner: String) -> Self {
        BankAccount {
            balance: 0.0,
            owner,
        }
    }
    
    // 게터들
    fn get_balance(&self) -> f64 {
        self.balance
    }
    
    fn get_owner(&self) -> &str {
        &self.owner
    }
    
    // 세터들
    fn set_balance(&mut self, balance: f64) {
        self.balance = balance;
    }
    
    fn set_owner(&mut self, owner: String) {
        self.owner = owner;
    }
}
```

이 패턴의 문제점은 캡슐화가 제대로 이루어지지 않는다는 것이다. `set_balance`를 통해 잔액을 음수로 만들 수 있다.

### Rust다운 접근: 동작 기반 API

```rust
struct BankAccount {
    balance: f64,
    owner: String,
    transaction_history: Vec<Transaction>,
}

#[derive(Debug, Clone)]
struct Transaction {
    amount: f64,
    description: String,
    timestamp: std::time::SystemTime,
}

#[derive(Debug)]
enum TransactionError {
    InsufficientFunds,
    InvalidAmount,
}

impl BankAccount {
    fn new(owner: String) -> Self {
        BankAccount {
            balance: 0.0,
            owner,
            transaction_history: Vec::new(),
        }
    }
    
    // 동작 기반 메서드
    fn deposit(&mut self, amount: f64) -> Result<(), TransactionError> {
        if amount <= 0.0 {
            return Err(TransactionError::InvalidAmount);
        }
        
        self.balance += amount;
        self.transaction_history.push(Transaction {
            amount,
            description: format!("Deposit: ${:.2}", amount),
            timestamp: std::time::SystemTime::now(),
        });
        
        Ok(())
    }
    
    fn withdraw(&mut self, amount: f64) -> Result<(), TransactionError> {
        if amount <= 0.0 {
            return Err(TransactionError::InvalidAmount);
        }
        
        if amount > self.balance {
            return Err(TransactionError::InsufficientFunds);
        }
        
        self.balance -= amount;
        self.transaction_history.push(Transaction {
            amount: -amount,
            description: format!("Withdrawal: ${:.2}", amount),
            timestamp: std::time::SystemTime::now(),
        });
        
        Ok(())
    }
    
    fn transfer(&mut self, to: &mut BankAccount, amount: f64) -> Result<(), TransactionError> {
        self.withdraw(amount)?;
        to.deposit(amount)?;
        Ok(())
    }
    
    // 읽기 전용 접근은 공개 메서드로
    fn balance(&self) -> f64 {
        self.balance
    }
    
    fn owner(&self) -> &str {
        &self.owner
    }
    
    fn transaction_history(&self) -> &[Transaction] {
        &self.transaction_history
    }
}

fn main() {
    let mut account1 = BankAccount::new("Alice".to_string());
    let mut account2 = BankAccount::new("Bob".to_string());
    
    account1.deposit(1000.0).unwrap();
    account1.withdraw(300.0).unwrap();
    
    match account1.transfer(&mut account2, 200.0) {
        Ok(_) => println!("Transfer successful"),
        Err(e) => println!("Transfer failed: {:?}", e),
    }
    
    println!("Alice's balance: ${:.2}", account1.balance());
    println!("Bob's balance: ${:.2}", account2.balance());
}
```

### 필드 접근이 필요한 경우
때로는 필드에 직접 접근하는 것이 더 나을 수 있다. Rust는 필드를 `pub`으로 선언하여 외부에 노출할 수 있다.

```rust
// 간단한 데이터 구조체는 필드를 공개
#[derive(Debug, Clone)]
pub struct Point {
    pub x: f64,
    pub y: f64,
}

impl Point {
    pub fn new(x: f64, y: f64) -> Self {
        Point { x, y }
    }
    
    pub fn distance_to(&self, other: &Point) -> f64 {
        let dx = self.x - other.x;
        let dy = self.y - other.y;
        (dx * dx + dy * dy).sqrt()
    }
}

fn main() {
    let mut p1 = Point::new(0.0, 0.0);
    let p2 = Point::new(3.0, 4.0);
    
    // 필드에 직접 접근
    p1.x = 1.0;
    p1.y = 1.0;
    
    println!("Distance: {}", p1.distance_to(&p2));
}
```

### 불변성을 활용한 설계
가장 Rust다운 방법은 불변성을 기본으로 하고, 변경이 필요할 때만 가변 참조를 사용하는 것이다.

```rust
#[derive(Debug, Clone)]
struct Config {
    pub host: String,
    pub port: u16,
    pub timeout_ms: u64,
    pub max_connections: usize,
}

impl Config {
    fn new() -> Self {
        Config {
            host: "localhost".to_string(),
            port: 8080,
            timeout_ms: 5000,
            max_connections: 100,
        }
    }
    
    // 빌더 스타일로 새 인스턴스 생성
    fn with_host(mut self, host: String) -> Self {
        self.host = host;
        self
    }
    
    fn with_port(mut self, port: u16) -> Self {
        self.port = port;
        self
    }
    
    fn with_timeout(mut self, timeout_ms: u64) -> Self {
        self.timeout_ms = timeout_ms;
        self
    }
}

fn main() {
    let config = Config::new()
        .with_host("0.0.0.0".to_string())
        .with_port(9090)
        .with_timeout(10000);
    
    println!("Config: {:?}", config);
}
```
  

## 8.3 빌더 패턴과 빌림
빌더 패턴은 Rust에서 매우 유용하지만, 빌림과 관련된 미묘한 문제들이 있다.

### 기본 빌더 패턴

```rust
struct HttpRequest {
    method: String,
    url: String,
    headers: Vec<(String, String)>,
    body: Option<String>,
}

struct HttpRequestBuilder {
    method: String,
    url: String,
    headers: Vec<(String, String)>,
    body: Option<String>,
}

impl HttpRequestBuilder {
    fn new(url: String) -> Self {
        HttpRequestBuilder {
            method: "GET".to_string(),
            url,
            headers: Vec::new(),
            body: None,
        }
    }
    
    fn method(mut self, method: &str) -> Self {
        self.method = method.to_string();
        self
    }
    
    fn header(mut self, key: &str, value: &str) -> Self {
        self.headers.push((key.to_string(), value.to_string()));
        self
    }
    
    fn body(mut self, body: String) -> Self {
        self.body = Some(body);
        self
    }
    
    fn build(self) -> HttpRequest {
        HttpRequest {
            method: self.method,
            url: self.url,
            headers: self.headers,
            body: self.body,
        }
    }
}

fn main() {
    let request = HttpRequestBuilder::new("https://api.example.com/users".to_string())
        .method("POST")
        .header("Content-Type", "application/json")
        .header("Authorization", "Bearer token123")
        .body(r#"{"name": "John"}"#.to_string())
        .build();
    
    println!("Method: {}", request.method);
    println!("URL: {}", request.url);
}
```

빌더 메서드들이 `self`를 소유하므로, 한 번 사용하면 이동된다.

### 해결책 1: 참조 기반 빌더

```rust
struct HttpRequestBuilder {
    method: String,
    url: String,
    headers: Vec<(String, String)>,
    body: Option<String>,
}

impl HttpRequestBuilder {
    fn new(url: String) -> Self {
        HttpRequestBuilder {
            method: "GET".to_string(),
            url,
            headers: Vec::new(),
            body: None,
        }
    }
    
    // &mut self를 받아서 재사용 가능하게
    fn method(&mut self, method: &str) -> &mut Self {
        self.method = method.to_string();
        self
    }
    
    fn header(&mut self, key: &str, value: &str) -> &mut Self {
        self.headers.push((key.to_string(), value.to_string()));
        self
    }
    
    fn body(&mut self, body: String) -> &mut Self {
        self.body = Some(body);
        self
    }
    
    fn build(&self) -> HttpRequest {
        HttpRequest {
            method: self.method.clone(),
            url: self.url.clone(),
            headers: self.headers.clone(),
            body: self.body.clone(),
        }
    }
    
    fn reset(&mut self) {
        self.method = "GET".to_string();
        self.headers.clear();
        self.body = None;
    }
}

fn main() {
    let mut builder = HttpRequestBuilder::new("https://api.example.com".to_string());
    
    builder.header("Authorization", "Bearer token123");
    
    let request1 = builder
        .method("GET")
        .build();
    
    builder.reset();
    
    let request2 = builder
        .method("POST")
        .body(r#"{"data": "value"}"#.to_string())
        .build();
    
    println!("Request 1 method: {}", request1.method);
    println!("Request 2 method: {}", request2.method);
}
```

### 해결책 2: 타입 상태 패턴으로 컴파일 타임 안전성 확보

```rust
// 타입 상태로 빌더의 상태를 표현
struct NoMethod;
struct HasMethod;
struct NoBody;
struct HasBody;

struct TypedRequestBuilder<M, B> {
    url: String,
    method: Option<String>,
    headers: Vec<(String, String)>,
    body: Option<String>,
    _method_state: std::marker::PhantomData<M>,
    _body_state: std::marker::PhantomData<B>,
}

impl TypedRequestBuilder<NoMethod, NoBody> {
    fn new(url: String) -> Self {
        TypedRequestBuilder {
            url,
            method: None,
            headers: Vec::new(),
            body: None,
            _method_state: std::marker::PhantomData,
            _body_state: std::marker::PhantomData,
        }
    }
}

impl<B> TypedRequestBuilder<NoMethod, B> {
    fn method(self, method: &str) -> TypedRequestBuilder<HasMethod, B> {
        TypedRequestBuilder {
            url: self.url,
            method: Some(method.to_string()),
            headers: self.headers,
            body: self.body,
            _method_state: std::marker::PhantomData,
            _body_state: std::marker::PhantomData,
        }
    }
}

impl<M, B> TypedRequestBuilder<M, B> {
    fn header(mut self, key: &str, value: &str) -> Self {
        self.headers.push((key.to_string(), value.to_string()));
        self
    }
}

impl<M> TypedRequestBuilder<M, NoBody> {
    fn body(self, body: String) -> TypedRequestBuilder<M, HasBody> {
        TypedRequestBuilder {
            url: self.url,
            method: self.method,
            headers: self.headers,
            body: Some(body),
            _method_state: std::marker::PhantomData,
            _body_state: std::marker::PhantomData,
        }
    }
}

impl<B> TypedRequestBuilder<HasMethod, B> {
    fn build(self) -> HttpRequest {
        HttpRequest {
            method: self.method.unwrap(),
            url: self.url,
            headers: self.headers,
            body: self.body,
        }
    }
}

fn main() {
    // 컴파일 타임에 메서드 호출 순서 강제
    let request = TypedRequestBuilder::new("https://api.example.com".to_string())
        .header("Authorization", "Bearer token")
        .method("POST")  // method()를 호출하지 않으면 build() 불가능
        .body(r#"{"key": "value"}"#.to_string())
        .build();
    
    println!("Built request with method: {}", request.method);
}
```

## 8.4 옵저버 패턴 구현하기

옵저버 패턴은 빌림 문제가 가장 심각하게 나타나는 패턴 중 하나다. Subject가 Observer를 참조하고, Observer가 Subject를 참조하는 순환 구조가 만들어지기 때문이다.

### 잘못된 시도: 직접 참조

```rust
// 컴파일되지 않는 코드
trait Observer {
    fn update(&mut self, value: i32);
}

struct Subject {
    value: i32,
    observers: Vec<Box<dyn Observer>>,
}

impl Subject {
    fn attach(&mut self, observer: Box<dyn Observer>) {
        self.observers.push(observer);
    }
    
    fn set_value(&mut self, value: i32) {
        self.value = value;
        self.notify();
    }
    
    fn notify(&mut self) {
        for observer in &mut self.observers {
            observer.update(self.value);
        }
    }
}
```

이 코드는 동작하지만, Observer가 Subject를 역참조하려고 하면 순환 참조 문제가 발생한다.

### 해결책 1: 콜백 함수 사용

```rust
type Callback = Box<dyn FnMut(i32)>;

struct Subject {
    value: i32,
    observers: Vec<Callback>,
}

impl Subject {
    fn new(initial_value: i32) -> Self {
        Subject {
            value: initial_value,
            observers: Vec::new(),
        }
    }
    
    fn subscribe<F>(&mut self, callback: F) 
    where
        F: FnMut(i32) + 'static,
    {
        self.observers.push(Box::new(callback));
    }
    
    fn set_value(&mut self, value: i32) {
        if self.value != value {
            self.value = value;
            self.notify();
        }
    }
    
    fn notify(&mut self) {
        for callback in &mut self.observers {
            callback(self.value);
        }
    }
    
    fn value(&self) -> i32 {
        self.value
    }
}

fn main() {
    let mut subject = Subject::new(0);
    
    // 간단한 로거 옵저버
    subject.subscribe(|value| {
        println!("Logger: Value changed to {}", value);
    });
    
    // 통계 수집 옵저버
    let mut max_value = 0;
    subject.subscribe(move |value| {
        if value > max_value {
            max_value = value;
            println!("Stats: New maximum value: {}", max_value);
        }
    });
    
    subject.set_value(5);
    subject.set_value(10);
    subject.set_value(3);
    subject.set_value(15);
}
```

### 해결책 2: 이벤트 채널 사용

```rust
use std::sync::mpsc::{channel, Sender, Receiver};

#[derive(Debug, Clone)]
enum Event {
    ValueChanged(i32),
    Reset,
}

struct EventSubject {
    value: i32,
    sender: Sender<Event>,
}

impl EventSubject {
    fn new(initial_value: i32) -> (Self, Receiver<Event>) {
        let (sender, receiver) = channel();
        let subject = EventSubject {
            value: initial_value,
            sender,
        };
        (subject, receiver)
    }
    
    fn set_value(&mut self, value: i32) {
        if self.value != value {
            self.value = value;
            let _ = self.sender.send(Event::ValueChanged(value));
        }
    }
    
    fn reset(&mut self) {
        self.value = 0;
        let _ = self.sender.send(Event::Reset);
    }
}

struct Observer {
    name: String,
    max_seen: i32,
}

impl Observer {
    fn new(name: String) -> Self {
        Observer {
            name,
            max_seen: 0,
        }
    }
    
    fn handle_event(&mut self, event: &Event) {
        match event {
            Event::ValueChanged(value) => {
                println!("{}: Received value {}", self.name, value);
                if *value > self.max_seen {
                    self.max_seen = *value;
                    println!("{}: New max: {}", self.name, self.max_seen);
                }
            }
            Event::Reset => {
                println!("{}: Reset received", self.name);
                self.max_seen = 0;
            }
        }
    }
}

fn main() {
    let (mut subject, receiver) = EventSubject::new(0);
    
    let mut observer1 = Observer::new("Observer1".to_string());
    let mut observer2 = Observer::new("Observer2".to_string());
    
    // 별도 스레드에서 이벤트 처리
    std::thread::spawn(move || {
        while let Ok(event) = receiver.recv() {
            observer1.handle_event(&event);
            observer2.handle_event(&event);
        }
    });
    
    subject.set_value(5);
    std::thread::sleep(std::time::Duration::from_millis(100));
    subject.set_value(10);
    std::thread::sleep(std::time::Duration::from_millis(100));
    subject.reset();
    std::thread::sleep(std::time::Duration::from_millis(100));
}
```

### 해결책 3: Weak 참조를 활용한 등록 시스템

```rust
use std::rc::{Rc, Weak};
use std::cell::RefCell;

trait Observer {
    fn on_change(&mut self, new_value: i32);
}

struct Subject {
    value: i32,
    observers: Vec<Weak<RefCell<dyn Observer>>>,
}

impl Subject {
    fn new(initial_value: i32) -> Self {
        Subject {
            value: initial_value,
            observers: Vec::new(),
        }
    }
    
    fn attach(&mut self, observer: Weak<RefCell<dyn Observer>>) {
        self.observers.push(observer);
    }
    
    fn set_value(&mut self, value: i32) {
        if self.value != value {
            self.value = value;
            self.notify();
        }
    }
    
    fn notify(&mut self) {
        // 유효한 옵저버들에게만 알림
        self.observers.retain(|weak| {
            if let Some(observer) = weak.upgrade() {
                observer.borrow_mut().on_change(self.value);
                true
            } else {
                false // 옵저버가 삭제되었으면 목록에서 제거
            }
        });
    }
}

struct DisplayObserver {
    name: String,
    count: i32,
}

impl DisplayObserver {
    fn new(name: String) -> Rc<RefCell<Self>> {
        Rc::new(RefCell::new(DisplayObserver {
            name,
            count: 0,
        }))
    }
}

impl Observer for DisplayObserver {
    fn on_change(&mut self, new_value: i32) {
        self.count += 1;
        println!("{}: Value is now {} (update #{})", 
                 self.name, new_value, self.count);
    }
}

fn main() {
    let mut subject = Subject::new(0);
    
    let observer1 = DisplayObserver::new("Display1".to_string());
    let observer2 = DisplayObserver::new("Display2".to_string());
    
    subject.attach(Rc::downgrade(&observer1));
    subject.attach(Rc::downgrade(&observer2));
    
    subject.set_value(10);
    subject.set_value(20);
    
    // observer1을 drop
    drop(observer1);
    
    subject.set_value(30); // observer2만 알림 받음
}
```
  

## 8.5 상태 기계 패턴과 소유권 이동
상태 기계는 Rust의 타입 시스템과 소유권을 활용하면 매우 안전하게 구현할 수 있다.

### C++ 스타일: 열거형과 조건문

```rust
// 전통적인 방식 - 런타임 검사
enum ConnectionState {
    Disconnected,
    Connecting,
    Connected,
    Error(String),
}

struct Connection {
    state: ConnectionState,
    address: String,
}

impl Connection {
    fn new(address: String) -> Self {
        Connection {
            state: ConnectionState::Disconnected,
            address,
        }
    }
    
    fn connect(&mut self) -> Result<(), String> {
        match self.state {
            ConnectionState::Disconnected => {
                println!("Connecting to {}...", self.address);
                self.state = ConnectionState::Connecting;
                Ok(())
            }
            _ => Err("Cannot connect: not in disconnected state".to_string()),
        }
    }
    
    fn finalize_connection(&mut self) -> Result<(), String> {
        match self.state {
            ConnectionState::Connecting => {
                println!("Connection established");
                self.state = ConnectionState::Connected;
                Ok(())
            }
            _ => Err("Cannot finalize: not in connecting state".to_string()),
        }
    }
    
    fn send_data(&self, data: &str) -> Result<(), String> {
        match self.state {
            ConnectionState::Connected => {
                println!("Sending: {}", data);
                Ok(())
            }
            _ => Err("Cannot send: not connected".to_string()),
        }
    }
}
```

이 방식의 문제점은 런타임에 상태를 검사해야 하고, 잘못된 상태 전이를 컴파일 타임에 잡을 수 없다는 것이다.

### Rust다운 방식: 타입 상태 패턴

```rust
// 각 상태를 별도 타입으로
struct Disconnected;
struct Connecting;
struct Connected;

struct Connection<State> {
    address: String,
    state: std::marker::PhantomData<State>,
}

// Disconnected 상태의 메서드
impl Connection<Disconnected> {
    fn new(address: String) -> Self {
        Connection {
            address,
            state: std::marker::PhantomData,
        }
    }
    
    fn connect(self) -> Connection<Connecting> {
        println!("Connecting to {}...", self.address);
        Connection {
            address: self.address,
            state: std::marker::PhantomData,
        }
    }
}

// Connecting 상태의 메서드
impl Connection<Connecting> {
    fn finalize(self) -> Connection<Connected> {
        println!("Connection established to {}", self.address);
        Connection {
            address: self.address,
            state: std::marker::PhantomData,
        }
    }
    
    fn fail(self, reason: String) -> Connection<Disconnected> {
        println!("Connection failed: {}", reason);
        Connection {
            address: self.address,
            state: std::marker::PhantomData,
        }
    }
}

// Connected 상태의 메서드
impl Connection<Connected> {
    fn send(&self, data: &str) {
        println!("Sending to {}: {}", self.address, data);
    }
    
    fn disconnect(self) -> Connection<Disconnected> {
        println!("Disconnecting from {}", self.address);
        Connection {
            address: self.address,
            state: std::marker::PhantomData,
        }
    }
}

fn main() {
    let conn = Connection::new("192.168.1.1".to_string());
    
    // 컴파일 타임에 강제되는 상태 전이
    let conn = conn.connect();
    // conn.send("data"); // 컴파일 에러! Connecting 상태에서는 send 불가
    
    let conn = conn.finalize();
    conn.send("Hello, world!"); // OK
    
    let conn = conn.disconnect();
    // conn.send("data"); // 컴파일 에러! Disconnected 상태에서는 send 불가
    
    // 다시 연결
    let conn = conn.connect().finalize();
    conn.send("Reconnected!");
}
```

### 실전 예제: TCP 연결 상태 기계
더 복잡한 예제로 TCP 연결 상태를 구현해본다.

```rust
use std::marker::PhantomData;

// 상태 타입들
struct Closed;
struct Listen;
struct SynSent;
struct SynReceived;
struct Established;
struct CloseWait;

struct TcpConnection<State> {
    local_addr: String,
    remote_addr: Option<String>,
    sequence_number: u32,
    state: PhantomData<State>,
}

impl TcpConnection<Closed> {
    fn new(local_addr: String) -> Self {
        TcpConnection {
            local_addr,
            remote_addr: None,
            sequence_number: 0,
            state: PhantomData,
        }
    }
    
    // 서버로 동작: listen
    fn listen(self) -> TcpConnection<Listen> {
        println!("[{}] Listening...", self.local_addr);
        TcpConnection {
            local_addr: self.local_addr,
            remote_addr: None,
            sequence_number: 0,
            state: PhantomData,
        }
    }
    
    // 클라이언트로 동작: connect
    fn connect(mut self, remote_addr: String) -> TcpConnection<SynSent> {
        println!("[{}] Connecting to {}...", self.local_addr, remote_addr);
        self.remote_addr = Some(remote_addr);
        self.sequence_number = 100; // 초기 시퀀스 번호
        TcpConnection {
            local_addr: self.local_addr,
            remote_addr: self.remote_addr,
            sequence_number: self.sequence_number,
            state: PhantomData,
        }
    }
}

impl TcpConnection<Listen> {
    fn accept_connection(mut self, remote_addr: String) -> TcpConnection<SynReceived> {
        println!("[{}] Accepting connection from {}", self.local_addr, remote_addr);
        self.remote_addr = Some(remote_addr);
        self.sequence_number = 200;
        TcpConnection {
            local_addr: self.local_addr,
            remote_addr: self.remote_addr,
            sequence_number: self.sequence_number,
            state: PhantomData,
        }
    }
}

impl TcpConnection<SynSent> {
    fn receive_syn_ack(self) -> TcpConnection<Established> {
        println!("[{}] Received SYN-ACK, connection established", self.local_addr);
        TcpConnection {
            local_addr: self.local_addr,
            remote_addr: self.remote_addr,
            sequence_number: self.sequence_number + 1,
            state: PhantomData,
        }
    }
}

impl TcpConnection<SynReceived> {
    fn receive_ack(self) -> TcpConnection<Established> {
        println!("[{}] Received ACK, connection established", self.local_addr);
        TcpConnection {
            local_addr: self.local_addr,
            remote_addr: self.remote_addr,
            sequence_number: self.sequence_number + 1,
            state: PhantomData,
        }
    }
}

impl TcpConnection<Established> {
    fn send(&mut self, data: &[u8]) {
        if let Some(ref remote) = self.remote_addr {
            println!("[{}] Sending {} bytes to {}", 
                     self.local_addr, data.len(), remote);
            self.sequence_number += data.len() as u32;
        }
    }
    
    fn receive(&self, data: &[u8]) {
        if let Some(ref remote) = self.remote_addr {
            println!("[{}] Received {} bytes from {}", 
                     self.local_addr, data.len(), remote);
        }
    }
    
    fn close(self) -> TcpConnection<CloseWait> {
        println!("[{}] Closing connection...", self.local_addr);
        TcpConnection {
            local_addr: self.local_addr,
            remote_addr: self.remote_addr,
            sequence_number: self.sequence_number,
            state: PhantomData,
        }
    }
}

impl TcpConnection<CloseWait> {
    fn finish(self) -> TcpConnection<Closed> {
        println!("[{}] Connection closed", self.local_addr);
        TcpConnection {
            local_addr: self.local_addr,
            remote_addr: None,
            sequence_number: 0,
            state: PhantomData,
        }
    }
}

fn client_scenario() {
    println!("=== Client Scenario ===");
    let conn = TcpConnection::new("192.168.1.100:5000".to_string());
    let conn = conn.connect("192.168.1.1:80".to_string());
    let mut conn = conn.receive_syn_ack();
    
    conn.send(b"GET / HTTP/1.1\r\n");
    conn.receive(b"HTTP/1.1 200 OK\r\n");
    
    let conn = conn.close();
    let _conn = conn.finish();
}

fn server_scenario() {
    println!("\n=== Server Scenario ===");
    let conn = TcpConnection::new("192.168.1.1:80".to_string());
    let conn = conn.listen();
    let conn = conn.accept_connection("192.168.1.100:5000".to_string());
    let mut conn = conn.receive_ack();
    
    conn.receive(b"GET / HTTP/1.1\r\n");
    conn.send(b"HTTP/1.1 200 OK\r\n");
    
    let conn = conn.close();
    let _conn = conn.finish();
}

fn main() {
    client_scenario();
    server_scenario();
}
```

### 타입 상태 패턴의 장단점

**장점:**
- 잘못된 상태 전이를 컴파일 타임에 방지한다
- 각 상태에서 가능한 작업이 타입으로 명확히 표현된다
- 런타임 검사가 필요 없어 성능이 좋다

**단점:**
- 상태가 많아지면 코드가 길어진다
- 상태를 런타임에 결정해야 하는 경우 사용하기 어렵다
- 상태 전이마다 소유권이 이동하므로 이전 상태로 되돌리기 어렵다

### 하이브리드 접근: 필요한 곳에만 타입 상태 사용

```rust
enum State {
    Idle,
    Processing { progress: f64 },
    Completed { result: String },
    Failed { error: String },
}

struct Task {
    id: u32,
    state: State,
}

impl Task {
    fn new(id: u32) -> Self {
        Task {
            id,
            state: State::Idle,
        }
    }
    
    fn start(&mut self) -> Result<(), String> {
        match self.state {
            State::Idle => {
                self.state = State::Processing { progress: 0.0 };
                Ok(())
            }
            _ => Err("Task already started".to_string()),
        }
    }
    
    fn update_progress(&mut self, progress: f64) -> Result<(), String> {
        match &mut self.state {
            State::Processing { progress: p } => {
                *p = progress;
                Ok(())
            }
            _ => Err("Task not in processing state".to_string()),
        }
    }
    
    fn complete(&mut self, result: String) -> Result<(), String> {
        match self.state {
            State::Processing { .. } => {
                self.state = State::Completed { result };
                Ok(())
            }
            _ => Err("Task not in processing state".to_string()),
        }
    }
}

fn main() {
    let mut task = Task::new(1);
    
    task.start().unwrap();
    task.update_progress(0.5).unwrap();
    task.update_progress(1.0).unwrap();
    task.complete("Success!".to_string()).unwrap();
}
```

이 장에서는 실전에서 자주 마주치는 패턴들을 Rust로 구현하면서 발생하는 빌림 문제와 해결책을 살펴보았다. 핵심은 C++의 패턴을 그대로 옮기려 하지 말고, Rust의 철학에 맞게 재설계하는 것이다. 소유권과 빌림 규칙을 제약이 아닌 설계 도구로 활용하면, 더 안전하고 명확한 코드를 작성할 수 있다.
