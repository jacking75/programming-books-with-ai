# 4장. 컬렉션과 반복자에서의 빌림
컬렉션을 다루다 보면 빌림 검사기와 자주 충돌하게 된다. C++에서는 컨테이너를 순회하면서 동시에 수정하는 것이 가능했지만, Rust에서는 이것이 컴파일 타임에 차단된다. 이 장에서는 컬렉션 사용 시 발생하는 실전 빌림 문제들과 그 해결 방법을 살펴본다.

## 4.1 Vec 순회 중 수정이 필요할 때

### 문제 상황: 순회 중 요소 추가하기
C++에서는 벡터를 순회하면서 조건에 따라 새로운 요소를 추가하는 코드를 쉽게 작성할 수 있었다. 하지만 Rust에서는 다음과 같은 코드가 컴파일되지 않는다.

```rust
// 컴파일 실패!
fn process_items(items: &mut Vec<i32>) {
    for item in items.iter() {
        if *item > 10 {
            items.push(*item * 2);  // 에러: items가 이미 불변 빌림됨
        }
    }
}
```

`items.iter()`가 `items`에 대한 불변 참조를 생성하는데, 그 상태에서 `items.push()`를 호출하려고 하면 가변 참조가 필요해서 에러가 발생한다. 불변 빌림과 가변 빌림을 동시에 할 수 없다는 Rust의 핵심 규칙이 적용되는 것이다.

### 해결책 1: 인덱스 기반 순회
가장 직관적인 해결책은 반복자 대신 인덱스를 사용하는 것이다.

```rust
fn process_items(items: &mut Vec<i32>) {
    let original_len = items.len();
    for i in 0..original_len {
        if items[i] > 10 {
            items.push(items[i] * 2);
        }
    }
}
```

이 방법은 원본 길이를 미리 저장해두고, 그 범위만큼만 순회한다. 각 인덱스 접근마다 새로운 빌림이 발생하고 즉시 해제되므로, `push` 호출 시점에는 빌림이 남아있지 않다.

### 해결책 2: 수집 후 확장
더 Rust다운 방법은 추가할 항목들을 먼저 수집한 다음 한 번에 확장하는 것이다.

```rust
fn process_items(items: &mut Vec<i32>) {
    let new_items: Vec<i32> = items.iter()
        .filter(|&&x| x > 10)
        .map(|&x| x * 2)
        .collect();
    
    items.extend(new_items);
}
```

이 방식은 두 단계로 나뉜다. 첫 번째 단계에서 불변 참조로 순회하며 새 값들을 별도 벡터에 수집하고, 두 번째 단계에서 가변 참조로 확장한다. 두 빌림이 시간적으로 겹치지 않으므로 문제가 없다.

### 해결책 3: drain으로 소유권 이동
요소들을 제거하면서 처리해야 한다면 `drain`을 사용할 수 있다.

```rust
fn process_and_filter(items: &mut Vec<i32>) {
    let processed: Vec<i32> = items.drain(..)
        .filter(|&x| x > 10)
        .map(|x| x * 2)
        .collect();
    
    *items = processed;
}
```

`drain`은 요소들의 소유권을 가져오므로 빌림 문제가 발생하지 않는다. 원본 벡터는 비워지고, 처리된 결과로 교체된다.
  

## 4.2 HashMap/BTreeMap 사용 시 빌림 문제

### 문제 상황: 맵 조회 후 수정
맵에서 값을 조회한 결과를 바탕으로 맵을 수정하려고 하면 빌림 충돌이 발생한다.

```rust
use std::collections::HashMap;

// 컴파일 실패!
fn update_based_on_lookup(map: &mut HashMap<String, i32>) {
    if let Some(&value) = map.get("key1") {
        map.insert("key2".to_string(), value * 2);  // 에러!
    }
}
```

`map.get()`이 맵에 대한 불변 참조를 반환하고, 그 참조가 `if let` 스코프 안에서 살아있는 동안 `map.insert()`로 가변 참조를 얻으려고 해서 에러가 발생한다.

### 해결책: 값을 먼저 복사
참조가 아닌 값 자체를 먼저 복사해두면 빌림이 즉시 해제된다.

```rust
fn update_based_on_lookup(map: &mut HashMap<String, i32>) {
    if let Some(&value) = map.get("key1") {
        let copied_value = value;  // i32는 Copy이므로 자동 복사
        map.insert("key2".to_string(), copied_value * 2);
    }
}

// 또는 더 간결하게
fn update_based_on_lookup_v2(map: &mut HashMap<String, i32>) {
    let value = map.get("key1").copied();  // Option<i32>로 복사
    if let Some(v) = value {
        map.insert("key2".to_string(), v * 2);
    }
}
```

`copied()` 메서드는 `Option<&T>`를 `Option<T>`로 변환한다(T가 Copy일 때). 이렇게 하면 맵에 대한 참조를 더 이상 보유하지 않으므로 가변 빌림이 가능해진다.

### 복잡한 사례: 여러 키 참조 후 수정

```rust
// 컴파일 실패!
fn complex_update(map: &mut HashMap<String, Vec<i32>>) {
    if let (Some(vec1), Some(vec2)) = (map.get("a"), map.get("b")) {
        let sum: i32 = vec1.iter().sum::<i32>() + vec2.iter().sum::<i32>();
        map.insert("c".to_string(), vec![sum]);  // 에러!
    }
}
```

이 경우 `vec1`과 `vec2` 참조가 모두 살아있는 상태에서 `insert`를 호출하므로 에러가 발생한다.

```rust
fn complex_update(map: &mut HashMap<String, Vec<i32>>) {
    let sum = {
        let sum1 = map.get("a").map(|v| v.iter().sum::<i32>()).unwrap_or(0);
        let sum2 = map.get("b").map(|v| v.iter().sum::<i32>()).unwrap_or(0);
        sum1 + sum2
    };  // 이 시점에 모든 불변 빌림 해제
    
    map.insert("c".to_string(), vec![sum]);
}
```

중괄호로 스코프를 만들어 계산을 먼저 끝내고 빌림을 해제한 후 `insert`를 호출한다.
  

## 4.3 반복자를 사용하면서 컬렉션 수정하기

### 문제 상황: entry API의 필요성
맵에서 키가 존재하면 값을 수정하고, 없으면 새로 삽입하는 로직은 매우 흔하다.

```rust
// 비효율적이지만 작동하는 버전
fn increment_count(map: &mut HashMap<String, i32>, key: &str) {
    if map.contains_key(key) {
        *map.get_mut(key).unwrap() += 1;
    } else {
        map.insert(key.to_string(), 1);
    }
}
```

이 코드는 맵을 두 번 조회한다. `contains_key`로 한 번, `get_mut`로 또 한 번. 

### 해결책: entry API 활용
`entry` API는 이런 패턴을 위해 설계되었다.

```rust
fn increment_count(map: &mut HashMap<String, i32>, key: &str) {
    *map.entry(key.to_string()).or_insert(0) += 1;
}
```

`entry`는 맵에서 해당 키 위치에 대한 "진입점"을 반환한다. 이 진입점을 통해 값이 있으면 수정하고, 없으면 삽입할 수 있다. 맵 조회가 단 한 번만 일어난다.

더 복잡한 예시를 보자.

```rust
#[derive(Debug)]
struct PlayerStats {
    score: i32,
    games_played: i32,
}

fn update_player_stats(
    stats: &mut HashMap<String, PlayerStats>,
    player: &str,
    score: i32,
) {
    stats.entry(player.to_string())
        .and_modify(|s| {
            s.score += score;
            s.games_played += 1;
        })
        .or_insert(PlayerStats {
            score,
            games_played: 1,
        });
}
```

`and_modify`는 값이 존재할 때 실행되는 클로저를 받고, `or_insert`는 값이 없을 때 삽입할 값을 받는다. 이 패턴으로 복잡한 갱신 로직도 깔끔하게 처리할 수 있다.

### retain으로 조건부 제거
컬렉션을 순회하면서 조건에 맞는 요소만 남기고 싶을 때 `retain`을 사용한다.

```rust
fn remove_low_scores(stats: &mut HashMap<String, PlayerStats>) {
    stats.retain(|_player, stat| stat.score >= 100);
}

fn keep_even_numbers(numbers: &mut Vec<i32>) {
    numbers.retain(|&x| x % 2 == 0);
}
```

`retain`은 가변 참조를 받아 in-place로 수정하므로 추가 메모리 할당이 없다. C++의 `erase-remove` 이디엄과 비슷하지만 더 간결하다.
  

## 4.4 인덱스 기반 접근 vs 반복자

### 반복자를 선호해야 하는 이유
Rust에서는 대부분의 경우 인덱스보다 반복자를 사용하는 것이 좋다. 반복자는 경계 검사를 최적화할 수 있고, 더 표현력이 뛰어나며, 병렬 처리로 쉽게 전환할 수 있다.

```rust
// 인덱스 기반 - 매 접근마다 경계 검사 발생 가능
fn sum_positive_indices(data: &Vec<i32>) -> i32 {
    let mut sum = 0;
    for i in 0..data.len() {
        if data[i] > 0 {
            sum += data[i];
        }
    }
    sum
}

// 반복자 기반 - 더 효율적이고 명확함
fn sum_positive_iterator(data: &Vec<i32>) -> i32 {
    data.iter()
        .filter(|&&x| x > 0)
        .sum()
}
```

### 인덱스가 필요한 경우
하지만 실제로 인덱스 값이 필요하거나, 여러 컬렉션을 동시에 접근해야 할 때는 인덱스가 유용하다.

```rust
fn find_matching_pairs(left: &Vec<i32>, right: &Vec<i32>) -> Vec<(usize, i32)> {
    let mut pairs = Vec::new();
    for i in 0..left.len().min(right.len()) {
        if left[i] == right[i] {
            pairs.push((i, left[i]));
        }
    }
    pairs
}

// 반복자로는 이렇게
fn find_matching_pairs_iter(left: &Vec<i32>, right: &Vec<i32>) -> Vec<(usize, i32)> {
    left.iter()
        .zip(right.iter())
        .enumerate()
        .filter(|(_, (&l, &r))| l == r)
        .map(|(i, (&val, _))| (i, val))
        .collect()
}
```

두 방식 모두 유효하다. 인덱스 기반이 더 읽기 쉽다고 느껴진다면 그것을 선택해도 괜찮다.
  

### 순회 중 인덱스로 다른 요소 접근하기
특정 요소를 처리하면서 다른 요소도 참조해야 하는 경우가 있다.

```rust
// 컴파일 실패!
fn process_with_neighbors(data: &mut Vec<i32>) {
    for item in data.iter_mut() {
        let context = data[0];  // 에러: data가 이미 가변 빌림됨
        *item += context;
    }
}
```

이 경우 `split_at_mut` 같은 메서드를 사용하거나, 인덱스 기반으로 전환해야 한다.

```rust
fn process_with_context(data: &mut Vec<i32>) {
    if data.is_empty() {
        return;
    }
    
    let context = data[0];  // 먼저 값을 복사
    for i in 1..data.len() {
        data[i] += context;
    }
}

// 또는 split_at_mut 사용
fn process_with_split(data: &mut Vec<i32>) {
    if data.len() < 2 {
        return;
    }
    
    let (first, rest) = data.split_at_mut(1);
    let context = first[0];
    for item in rest.iter_mut() {
        *item += context;
    }
}
```

`split_at_mut`은 슬라이스를 두 개의 가변 슬라이스로 나눈다. 이렇게 하면 각 부분에 대한 가변 참조를 동시에 가질 수 있다.

### 복잡한 사례: 윈도우 슬라이딩
이전 요소, 현재 요소, 다음 요소를 동시에 보면서 처리해야 하는 경우도 있다.

```rust
fn smooth_data(data: &Vec<f64>) -> Vec<f64> {
    if data.len() < 3 {
        return data.clone();
    }
    
    data.windows(3)
        .map(|window| (window[0] + window[1] + window[2]) / 3.0)
        .collect()
}

// 패딩을 포함한 버전
fn smooth_with_padding(data: &Vec<f64>) -> Vec<f64> {
    if data.is_empty() {
        return Vec::new();
    }
    
    let mut result = Vec::with_capacity(data.len());
    
    // 첫 요소
    result.push((data[0] + data[1]) / 2.0);
    
    // 중간 요소들
    for i in 1..data.len() - 1 {
        result.push((data[i - 1] + data[i] + data[i + 1]) / 3.0);
    }
    
    // 마지막 요소
    let len = data.len();
    result.push((data[len - 2] + data[len - 1]) / 2.0);
    
    result
}
```

`windows` 메서드는 겹치는 윈도우들을 반복하므로 슬라이딩 윈도우 패턴을 구현하기 쉽다.

### enumerate로 인덱스와 값 동시 얻기
인덱스가 필요하지만 반복자의 장점도 활용하고 싶다면 `enumerate`를 사용한다.

```rust
fn mark_positions(items: &Vec<String>, target: &str) -> Vec<usize> {
    items.iter()
        .enumerate()
        .filter(|(_, item)| item.as_str() == target)
        .map(|(index, _)| index)
        .collect()
}

// 인덱스 기반 동등 코드
fn mark_positions_idx(items: &Vec<String>, target: &str) -> Vec<usize> {
    let mut positions = Vec::new();
    for i in 0..items.len() {
        if items[i] == target {
            positions.push(i);
        }
    }
    positions
}
```

두 방식의 성능은 거의 동일하지만, 반복자 버전이 더 함수형 스타일이다.

### 실전 예제: 중복 제거하면서 순서 유지
해시셋을 사용하면서 원본 순서도 유지해야 하는 경우를 보자.

```rust
use std::collections::HashSet;

fn deduplicate_preserve_order(items: &Vec<i32>) -> Vec<i32> {
    let mut seen = HashSet::new();
    items.iter()
        .filter(|&x| seen.insert(*x))  // insert는 새 값이면 true 반환
        .copied()
        .collect()
}

// 원본을 수정하는 버전
fn deduplicate_in_place(items: &mut Vec<i32>) {
    let mut seen = HashSet::new();
    items.retain(|x| seen.insert(*x));
}
```

`HashSet::insert`는 값이 이미 존재하면 `false`를, 새로 삽입하면 `true`를 반환한다. 이 특성을 활용하면 중복 제거를 한 번의 순회로 처리할 수 있다.

### 성능 고려사항
반복자는 대부분 제로 비용 추상화이지만, 긴 체인은 컴파일 시간을 늘릴 수 있다. 성능이 중요한 내부 루프에서는 프로파일링 결과를 보고 결정하는 것이 좋다.

```rust
// 짧고 명확한 체인은 좋다
data.iter().filter(|&&x| x > 0).sum()

// 너무 긴 체인은 가독성을 해칠 수 있다
data.iter()
    .filter(|&&x| x > 0)
    .map(|&x| x * 2)
    .filter(|&x| x < 100)
    .map(|x| x as f64)
    .map(|x| x.sqrt())
    .collect::<Vec<_>>()

// 중간 단계로 나누는 것을 고려
let positive: Vec<_> = data.iter().filter(|&&x| x > 0).copied().collect();
let scaled: Vec<_> = positive.iter().map(|&x| x * 2).collect();
let result: Vec<_> = scaled.iter()
    .filter(|&&x| x < 100)
    .map(|&x| (x as f64).sqrt())
    .collect();
```

컬렉션과 반복자를 다룰 때는 빌림 규칙을 이해하고, 적절한 API를 선택하며, 필요할 때는 인덱스와 반복자를 유연하게 전환하는 것이 핵심이다. C++처럼 마음대로 수정할 수는 없지만, Rust의 방식에 익숙해지면 더 안전하고 표현력 있는 코드를 작성할 수 있다.
