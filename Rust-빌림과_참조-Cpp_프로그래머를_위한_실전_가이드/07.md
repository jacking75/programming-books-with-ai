# 7장. 내부 가변성 패턴
Rust의 빌림 규칙은 컴파일 타임에 메모리 안전성을 보장하지만, 때로는 이 규칙이 지나치게 제한적으로 느껴진다. 특히 C++에서 자유롭게 사용하던 패턴들을 구현할 때 벽에 부딪히게 된다. 이럴 때 필요한 것이 바로 **내부 가변성(Interior Mutability)** 패턴이다.

내부 가변성은 불변 참조(`&T`)를 통해서도 내부 데이터를 변경할 수 있게 해주는 패턴이다. 일반적인 Rust의 빌림 규칙을 런타임으로 미루는 것으로, 컴파일러가 아닌 런타임에 빌림 규칙을 검사한다.

## 7.1 Cell과 RefCell의 차이와 사용법

### Cell<T>: Copy 타입을 위한 간단한 내부 가변성
`Cell<T>`는 `Copy` 트레이트를 구현하는 타입(정수, bool, char 등)에 사용된다. 참조를 제공하지 않고 값을 복사해서 주고받는다.

```rust
use std::cell::Cell;

struct Config {
    retry_count: Cell<u32>,
    timeout_ms: Cell<u64>,
}

impl Config {
    fn new() -> Self {
        Config {
            retry_count: Cell::new(3),
            timeout_ms: Cell::new(1000),
        }
    }
    
    // &self로 받지만 내부 값을 변경할 수 있다
    fn increment_retry(&self) {
        let current = self.retry_count.get();
        self.retry_count.set(current + 1);
    }
    
    fn adjust_timeout(&self, delta: i64) {
        let current = self.timeout_ms.get() as i64;
        let new_value = (current + delta).max(0) as u64;
        self.timeout_ms.set(new_value);
    }
}

fn main() {
    let config = Config::new();
    
    // 불변 참조만 가지고 있어도 내부 값 변경 가능
    config.increment_retry();
    config.adjust_timeout(500);
    
    println!("Retry count: {}", config.retry_count.get());
    println!("Timeout: {}", config.timeout_ms.get());
}
```

C++에서 `mutable` 키워드를 사용하던 것과 유사한 패턴이다. const 메서드 안에서도 특정 멤버 변수를 변경할 수 있게 해준다.

### RefCell<T>: 모든 타입을 위한 런타임 빌림 검사
`RefCell<T>`는 모든 타입에 사용할 수 있으며, 참조를 빌려주는 방식으로 동작한다. 빌림 규칙은 런타임에 검사되며, 규칙 위반 시 패닉이 발생한다.

```rust
use std::cell::RefCell;

struct Logger {
    logs: RefCell<Vec<String>>,
    max_size: usize,
}

impl Logger {
    fn new(max_size: usize) -> Self {
        Logger {
            logs: RefCell::new(Vec::new()),
            max_size,
        }
    }
    
    // &self로 받아도 내부 Vec를 수정 가능
    fn log(&self, message: &str) {
        let mut logs = self.logs.borrow_mut();
        
        if logs.len() >= self.max_size {
            logs.remove(0); // 가장 오래된 로그 제거
        }
        
        logs.push(message.to_string());
    }
    
    fn print_logs(&self) {
        // 불변 빌림으로 읽기
        let logs = self.logs.borrow();
        for (i, log) in logs.iter().enumerate() {
            println!("[{}] {}", i, log);
        }
    }
    
    fn count(&self) -> usize {
        self.logs.borrow().len()
    }
}

fn main() {
    let logger = Logger::new(5);
    
    logger.log("Application started");
    logger.log("Processing request");
    logger.log("Request completed");
    
    logger.print_logs();
}
```

### 실전 예제: 캐시 구현
다음은 메모이제이션 캐시를 구현한 예제다. C++에서는 `mutable std::map`으로 간단히 해결되지만, Rust에서는 `RefCell`이 필요하다.

```rust
use std::cell::RefCell;
use std::collections::HashMap;

struct ExpensiveCalculator {
    cache: RefCell<HashMap<u64, u64>>,
    computation_count: RefCell<usize>,
}

impl ExpensiveCalculator {
    fn new() -> Self {
        ExpensiveCalculator {
            cache: RefCell::new(HashMap::new()),
            computation_count: RefCell::new(0),
        }
    }
    
    // &self로 받지만 캐시를 업데이트할 수 있다
    fn fibonacci(&self, n: u64) -> u64 {
        // 먼저 캐시를 확인
        {
            let cache = self.cache.borrow();
            if let Some(&result) = cache.get(&n) {
                return result;
            }
        } // borrow()로 얻은 참조를 여기서 drop
        
        // 캐시에 없으면 계산
        let result = match n {
            0 => 0,
            1 => 1,
            _ => self.fibonacci(n - 1) + self.fibonacci(n - 2),
        };
        
        // 계산 결과를 캐시에 저장
        self.cache.borrow_mut().insert(n, result);
        *self.computation_count.borrow_mut() += 1;
        
        result
    }
    
    fn stats(&self) -> (usize, usize) {
        let cache_size = self.cache.borrow().len();
        let computations = *self.computation_count.borrow();
        (cache_size, computations)
    }
}

fn main() {
    let calc = ExpensiveCalculator::new();
    
    println!("fib(10) = {}", calc.fibonacci(10));
    println!("fib(10) = {}", calc.fibonacci(10)); // 캐시에서 즉시 반환
    println!("fib(15) = {}", calc.fibonacci(15));
    
    let (cache_size, computations) = calc.stats();
    println!("Cache size: {}, Computations: {}", cache_size, computations);
}
```
  

## 7.2 Mutex와 RwLock으로 스레드 안전성 확보
`Cell`과 `RefCell`은 단일 스레드에서만 안전하다. 멀티스레드 환경에서는 `Mutex`와 `RwLock`을 사용해야 한다.

### Mutex<T>: 상호 배제 잠금
`Mutex`는 한 번에 하나의 스레드만 데이터에 접근할 수 있게 보장한다.

```rust
use std::sync::{Arc, Mutex};
use std::thread;

struct Counter {
    value: Mutex<i32>,
}

impl Counter {
    fn new() -> Self {
        Counter {
            value: Mutex::new(0),
        }
    }
    
    fn increment(&self) {
        let mut val = self.value.lock().unwrap();
        *val += 1;
    } // lock()으로 얻은 가드가 drop되면서 자동으로 unlock
    
    fn get(&self) -> i32 {
        *self.value.lock().unwrap()
    }
}

fn main() {
    let counter = Arc::new(Counter::new());
    let mut handles = vec![];
    
    for _ in 0..10 {
        let counter_clone = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            for _ in 0..1000 {
                counter_clone.increment();
            }
        });
        handles.push(handle);
    }
    
    for handle in handles {
        handle.join().unwrap();
    }
    
    println!("Final count: {}", counter.get());
}
```

### RwLock<T>: 읽기-쓰기 잠금
읽기가 빈번하고 쓰기가 드문 경우 `RwLock`을 사용하면 성능을 개선할 수 있다. 여러 읽기 잠금은 동시에 허용되지만, 쓰기 잠금은 독점적이다.

```rust
use std::sync::{Arc, RwLock};
use std::thread;
use std::collections::HashMap;

struct UserDatabase {
    users: RwLock<HashMap<u64, String>>,
}

impl UserDatabase {
    fn new() -> Self {
        UserDatabase {
            users: RwLock::new(HashMap::new()),
        }
    }
    
    fn add_user(&self, id: u64, name: String) {
        let mut users = self.users.write().unwrap();
        users.insert(id, name);
    }
    
    fn get_user(&self, id: u64) -> Option<String> {
        let users = self.users.read().unwrap();
        users.get(&id).cloned()
    }
    
    fn user_count(&self) -> usize {
        let users = self.users.read().unwrap();
        users.len()
    }
}

fn main() {
    let db = Arc::new(UserDatabase::new());
    
    // 쓰기 스레드
    let db_clone = Arc::clone(&db);
    let writer = thread::spawn(move || {
        for i in 0..100 {
            db_clone.add_user(i, format!("User{}", i));
        }
    });
    
    // 읽기 스레드들 (동시에 실행 가능)
    let mut readers = vec![];
    for _ in 0..5 {
        let db_clone = Arc::clone(&db);
        let reader = thread::spawn(move || {
            for _ in 0..20 {
                let count = db_clone.user_count();
                println!("Current user count: {}", count);
            }
        });
        readers.push(reader);
    }
    
    writer.join().unwrap();
    for reader in readers {
        reader.join().unwrap();
    }
}
```

### 실전 예제: 설정 관리자
애플리케이션 설정을 여러 스레드에서 읽고, 가끔 업데이트하는 경우의 구현이다.

```rust
use std::sync::{Arc, RwLock};
use std::thread;
use std::time::Duration;

#[derive(Clone, Debug)]
struct AppConfig {
    thread_pool_size: usize,
    request_timeout: Duration,
    max_retries: u32,
}

struct ConfigManager {
    config: RwLock<AppConfig>,
}

impl ConfigManager {
    fn new(initial_config: AppConfig) -> Self {
        ConfigManager {
            config: RwLock::new(initial_config),
        }
    }
    
    fn get_config(&self) -> AppConfig {
        self.config.read().unwrap().clone()
    }
    
    fn update_timeout(&self, new_timeout: Duration) {
        let mut config = self.config.write().unwrap();
        config.request_timeout = new_timeout;
        println!("Timeout updated to {:?}", new_timeout);
    }
    
    fn update_pool_size(&self, new_size: usize) {
        let mut config = self.config.write().unwrap();
        config.thread_pool_size = new_size;
        println!("Thread pool size updated to {}", new_size);
    }
}

fn main() {
    let initial_config = AppConfig {
        thread_pool_size: 4,
        request_timeout: Duration::from_secs(30),
        max_retries: 3,
    };
    
    let manager = Arc::new(ConfigManager::new(initial_config));
    
    // 워커 스레드들 (설정을 읽기만 함)
    let mut workers = vec![];
    for i in 0..3 {
        let manager_clone = Arc::clone(&manager);
        let worker = thread::spawn(move || {
            for _ in 0..5 {
                let config = manager_clone.get_config();
                println!("Worker {} using timeout: {:?}", i, config.request_timeout);
                thread::sleep(Duration::from_millis(100));
            }
        });
        workers.push(worker);
    }
    
    // 관리 스레드 (가끔 설정을 업데이트)
    let manager_clone = Arc::clone(&manager);
    let admin = thread::spawn(move || {
        thread::sleep(Duration::from_millis(250));
        manager_clone.update_timeout(Duration::from_secs(60));
        
        thread::sleep(Duration::from_millis(250));
        manager_clone.update_pool_size(8);
    });
    
    for worker in workers {
        worker.join().unwrap();
    }
    admin.join().unwrap();
}
```
  

## 7.3 내부 가변성의 런타임 비용
내부 가변성은 편리하지만 비용이 따른다. 각 패턴의 런타임 오버헤드를 이해하고 적절히 선택해야 한다.

### 성능 특성 비교

```rust
use std::cell::{Cell, RefCell};
use std::sync::{Mutex, RwLock};
use std::time::Instant;

fn benchmark_cell() {
    let value = Cell::new(0);
    let start = Instant::now();
    
    for _ in 0..10_000_000 {
        value.set(value.get() + 1);
    }
    
    println!("Cell: {:?}", start.elapsed());
}

fn benchmark_refcell() {
    let value = RefCell::new(0);
    let start = Instant::now();
    
    for _ in 0..10_000_000 {
        *value.borrow_mut() += 1;
    }
    
    println!("RefCell: {:?}", start.elapsed());
}

fn benchmark_mutex() {
    let value = Mutex::new(0);
    let start = Instant::now();
    
    for _ in 0..10_000_000 {
        *value.lock().unwrap() += 1;
    }
    
    println!("Mutex: {:?}", start.elapsed());
}

fn benchmark_rwlock() {
    let value = RwLock::new(0);
    let start = Instant::now();
    
    for _ in 0..10_000_000 {
        *value.write().unwrap() += 1;
    }
    
    println!("RwLock: {:?}", start.elapsed());
}

fn main() {
    benchmark_cell();    // 가장 빠름
    benchmark_refcell(); // Cell보다 약간 느림 (런타임 검사)
    benchmark_mutex();   // 훨씬 느림 (OS 수준 동기화)
    benchmark_rwlock();  // Mutex와 비슷하거나 약간 느림
}
```

실행 결과를 보면 대략적인 성능 차이를 확인할 수 있다:

- `Cell`: 거의 오버헤드 없음 (직접 접근과 동일)
- `RefCell`: 약간의 오버헤드 (빌림 카운터 검사)
- `Mutex`/`RwLock`: 상당한 오버헤드 (OS 커널 호출)

### RefCell의 런타임 패닉 예제
`RefCell`의 빌림 규칙 위반은 컴파일 타임이 아닌 런타임에 감지된다.

```rust
use std::cell::RefCell;

fn problematic_code() {
    let data = RefCell::new(vec![1, 2, 3]);
    
    let borrowed = data.borrow(); // 불변 빌림 시작
    
    // 패닉 발생! 불변 빌림이 활성화된 상태에서 가변 빌림 시도
    // let mut_borrowed = data.borrow_mut(); 
    
    println!("{:?}", borrowed);
} // borrowed가 drop되어야 가변 빌림 가능

fn correct_code() {
    let data = RefCell::new(vec![1, 2, 3]);
    
    // 스코프를 명확히 분리
    {
        let borrowed = data.borrow();
        println!("Reading: {:?}", borrowed);
    } // borrowed drop
    
    {
        let mut mut_borrowed = data.borrow_mut();
        mut_borrowed.push(4);
        println!("After modification: {:?}", mut_borrowed);
    }
}

fn main() {
    correct_code();
}
```

### 실전 예제: 빌림 충돌 회피 패턴
복잡한 메서드에서 `RefCell`을 사용할 때 빌림 충돌을 피하는 패턴이다.

```rust
use std::cell::RefCell;

struct Document {
    content: RefCell<String>,
    metadata: RefCell<Vec<String>>,
}

impl Document {
    fn new(text: &str) -> Self {
        Document {
            content: RefCell::new(text.to_string()),
            metadata: RefCell::new(Vec::new()),
        }
    }
    
    // 잘못된 패턴: 여러 RefCell을 동시에 빌림
    fn bad_process(&self) {
        let content = self.content.borrow();
        let mut metadata = self.metadata.borrow_mut();
        
        // content를 읽으면서 metadata를 업데이트
        let word_count = content.split_whitespace().count();
        metadata.push(format!("Words: {}", word_count));
        
        // content borrow가 살아있는 동안 content를 수정하려 하면 패닉
        // let mut content_mut = self.content.borrow_mut(); // 패닉!
    }
    
    // 올바른 패턴: 빌림을 최소 범위로 제한
    fn good_process(&self) {
        let word_count = {
            let content = self.content.borrow();
            content.split_whitespace().count()
        }; // content borrow 종료
        
        // 이제 안전하게 metadata 업데이트 가능
        self.metadata.borrow_mut().push(format!("Words: {}", word_count));
        
        // 필요하다면 content도 수정 가능
        self.content.borrow_mut().push_str(" [processed]");
    }
    
    // 더 나은 패턴: 데이터를 복사해서 빌림 해제
    fn better_process(&self) {
        let content_copy = self.content.borrow().clone();
        
        // 이제 content의 빌림이 해제되어 자유롭게 작업 가능
        let word_count = content_copy.split_whitespace().count();
        let char_count = content_copy.len();
        
        let mut metadata = self.metadata.borrow_mut();
        metadata.push(format!("Words: {}", word_count));
        metadata.push(format!("Chars: {}", char_count));
        
        // content도 수정 가능
        self.content.borrow_mut().push_str(" [analyzed]");
    }
}

fn main() {
    let doc = Document::new("Hello world from Rust");
    doc.good_process();
    doc.better_process();
    
    println!("Content: {}", doc.content.borrow());
    println!("Metadata: {:?}", doc.metadata.borrow());
}
```
  

## 7.4 언제 내부 가변성을 선택해야 하는가
내부 가변성은 강력하지만 남용하면 안 된다. 다음 상황에서 고려해볼 만하다.

### 1. 논리적으로 불변이지만 구현상 변경이 필요한 경우

```rust
use std::cell::RefCell;
use std::time::Instant;

struct PerformanceMonitor {
    last_check: RefCell<Instant>,
    check_count: RefCell<u64>,
}

impl PerformanceMonitor {
    fn new() -> Self {
        PerformanceMonitor {
            last_check: RefCell::new(Instant::now()),
            check_count: RefCell::new(0),
        }
    }
    
    // 논리적으로는 읽기 전용이지만, 내부적으로 통계를 업데이트
    fn check_health(&self) -> bool {
        *self.check_count.borrow_mut() += 1;
        *self.last_check.borrow_mut() = Instant::now();
        
        // 실제 헬스 체크 로직
        true
    }
    
    fn get_stats(&self) -> (u64, Instant) {
        (*self.check_count.borrow(), *self.last_check.borrow())
    }
}
```

### 2. 공유된 소유권에서 수정이 필요한 경우

```rust
use std::rc::Rc;
use std::cell::RefCell;

#[derive(Debug)]
struct Node {
    value: i32,
    children: Vec<Rc<RefCell<Node>>>,
}

impl Node {
    fn new(value: i32) -> Rc<RefCell<Self>> {
        Rc::new(RefCell::new(Node {
            value,
            children: Vec::new(),
        }))
    }
    
    fn add_child(&mut self, child: Rc<RefCell<Node>>) {
        self.children.push(child);
    }
}

fn build_tree() -> Rc<RefCell<Node>> {
    let root = Node::new(1);
    let child1 = Node::new(2);
    let child2 = Node::new(3);
    
    root.borrow_mut().add_child(Rc::clone(&child1));
    root.borrow_mut().add_child(Rc::clone(&child2));
    
    // child1에 자식 추가
    let grandchild = Node::new(4);
    child1.borrow_mut().add_child(grandchild);
    
    root
}

fn main() {
    let tree = build_tree();
    println!("Root value: {}", tree.borrow().value);
    println!("Children count: {}", tree.borrow().children.len());
}
```

### 3. 캐싱과 메모이제이션
앞서 본 `ExpensiveCalculator` 예제처럼, 순수 함수의 결과를 캐싱할 때 유용하다.

### 4. 이벤트 리스너와 콜백

```rust
use std::cell::RefCell;
use std::rc::Rc;

type Callback = Box<dyn Fn(&str)>;

struct EventEmitter {
    listeners: RefCell<Vec<Callback>>,
}

impl EventEmitter {
    fn new() -> Self {
        EventEmitter {
            listeners: RefCell::new(Vec::new()),
        }
    }
    
    fn on(&self, callback: Callback) {
        self.listeners.borrow_mut().push(callback);
    }
    
    fn emit(&self, event: &str) {
        let listeners = self.listeners.borrow();
        for listener in listeners.iter() {
            listener(event);
        }
    }
}

fn main() {
    let emitter = EventEmitter::new();
    
    emitter.on(Box::new(|event| {
        println!("Listener 1 received: {}", event);
    }));
    
    emitter.on(Box::new(|event| {
        println!("Listener 2 received: {}", event);
    }));
    
    emitter.emit("click");
    emitter.emit("hover");
}
```

### 내부 가변성을 피해야 하는 경우
일반적인 경우 다음 패턴들이 더 낫다:

```rust
// 피해야 할 패턴: 불필요한 RefCell
struct BadCounter {
    count: RefCell<i32>,
}

impl BadCounter {
    fn increment(&self) {
        *self.count.borrow_mut() += 1;
    }
}

// 권장 패턴: 명시적인 가변성
struct GoodCounter {
    count: i32,
}

impl GoodCounter {
    fn increment(&mut self) {
        self.count += 1;
    }
}

// 피해야 할 패턴: 모든 필드를 RefCell로
struct BadConfig {
    host: RefCell<String>,
    port: RefCell<u16>,
    timeout: RefCell<u64>,
}

// 권장 패턴: 구조체 전체를 RefCell로
struct GoodConfig {
    host: String,
    port: u16,
    timeout: u64,
}

// 사용처에서 RefCell로 감싸기
fn use_config() {
    let config = RefCell::new(GoodConfig {
        host: "localhost".to_string(),
        port: 8080,
        timeout: 30,
    });
    
    config.borrow_mut().port = 9090;
}
```

### 의사결정 가이드
내부 가변성 사용 여부를 결정할 때 다음 질문들을 던져보라:

1. **정말 불변 참조를 통한 수정이 필요한가?** - 대부분의 경우 `&mut self`로 충분하다.

2. **단일 스레드인가, 멀티스레드인가?** - 단일 스레드면 `RefCell`, 멀티스레드면 `Mutex`/`RwLock`을 사용한다.

3. **Copy 타입인가?** - `Copy` 타입이면 `Cell`이 가장 효율적이다.

4. **빌림 충돌 가능성은?** - 복잡한 메서드 호출 그래프에서 `RefCell`을 사용하면 런타임 패닉 위험이 있다.

5. **성능이 중요한가?** - 핫 패스에서는 내부 가변성의 오버헤드가 문제가 될 수 있다.

### 실전 예제: 게임 엔티티 시스템
마지막으로 게임 개발에서 자주 마주치는 패턴을 구현해본다. 엔티티들이 서로를 참조하고 수정해야 하는 상황이다.

```rust
use std::cell::RefCell;
use std::rc::Rc;

type EntityRef = Rc<RefCell<Entity>>;

#[derive(Debug)]
struct Entity {
    id: u32,
    health: i32,
    position: (f32, f32),
    target: Option<EntityRef>,
}

impl Entity {
    fn new(id: u32, x: f32, y: f32) -> EntityRef {
        Rc::new(RefCell::new(Entity {
            id,
            health: 100,
            position: (x, y),
            target: None,
        }))
    }
    
    fn set_target(&mut self, target: EntityRef) {
        self.target = Some(target);
    }
    
    fn attack(&mut self) {
        if let Some(target) = &self.target {
            let mut target_entity = target.borrow_mut();
            target_entity.health -= 10;
            println!(
                "Entity {} attacked Entity {}. Health: {}",
                self.id, target_entity.id, target_entity.health
            );
        }
    }
    
    fn distance_to(&self, other: &Entity) -> f32 {
        let dx = self.position.0 - other.position.0;
        let dy = self.position.1 - other.position.1;
        (dx * dx + dy * dy).sqrt()
    }
}

struct Game {
    entities: Vec<EntityRef>,
}

impl Game {
    fn new() -> Self {
        Game {
            entities: Vec::new(),
        }
    }
    
    fn spawn_entity(&mut self, x: f32, y: f32) -> EntityRef {
        let id = self.entities.len() as u32;
        let entity = Entity::new(id, x, y);
        self.entities.push(Rc::clone(&entity));
        entity
    }
    
    fn update(&self) {
        for entity in &self.entities {
            let mut entity_ref = entity.borrow_mut();
            
            // 가장 가까운 적을 찾아 공격
            if entity_ref.target.is_none() {
                let closest = self.find_closest_enemy(&*entity_ref);
                if let Some(target) = closest {
                    entity_ref.set_target(target);
                }
            }
            
            entity_ref.attack();
        }
    }
    
    fn find_closest_enemy(&self, entity: &Entity) -> Option<EntityRef> {
        let mut closest: Option<(f32, EntityRef)> = None;
        
        for other in &self.entities {
            let other_borrowed = other.borrow();
            if other_borrowed.id == entity.id {
                continue;
            }
            
            let distance = entity.distance_to(&*other_borrowed);
            
            match closest {
                None => closest = Some((distance, Rc::clone(other))),
                Some((min_dist, _)) if distance < min_dist => {
                    closest = Some((distance, Rc::clone(other)));
                }
                _ => {}
            }
        }
        
        closest.map(|(_, entity)| entity)
    }
}

fn main() {
    let mut game = Game::new();
    
    let _player = game.spawn_entity(0.0, 0.0);
    let _enemy1 = game.spawn_entity(10.0, 5.0);
    let _enemy2 = game.spawn_entity(-5.0, 8.0);
    
    println!("=== Turn 1 ===");
    game.update();
    
    println!("\n=== Turn 2 ===");
    game.update();
}
```

이 예제는 `Rc<RefCell<T>>` 패턴의 전형적인 사용 사례다. 여러 엔티티가 서로를 참조하고, 실행 중에 동적으로 관계가 변경되며, 각 엔티티가 독립적으로 수정될 수 있다.

내부 가변성은 Rust의 빌림 규칙을 우회하는 escape hatch지만, 신중하게 사용하면 복잡한 데이터 구조와 패턴을 안전하게 구현할 수 있다. 핵심은 **언제 사용할지 아는 것**과 **불필요하게 남용하지 않는 것** 사이의 균형을 찾는 것이다.
