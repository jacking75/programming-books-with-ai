# Chapter 10: 고급 쿼리 최적화

데이터가 쌓이고 쌓여서 이제 수억 건이 넘어갔다. 압축 정책도 잘 설정했고, 연속 집계도 활용하고 있다. 그런데 어느 날 운영팀에서 요청한 복잡한 분석 쿼리가 10분이 넘게 걸린다. "지난 3개월간 서버별 플레이어 행동 패턴과 아이템 사용 통계를 동시에 보고 싶어요"라는 요구사항은 간단해 보이지만, 실제로는 여러 테이블을 조인하고 수억 건의 데이터를 스캔해야 한다.

이번 챕터에서는 이런 느린 쿼리를 어떻게 분석하고 최적화하는지 배운다. 쿼리 실행 계획을 읽는 방법, 적절한 인덱스를 설계하는 전략, 파티션 프루닝을 활용하는 기법 등 실전에서 바로 사용할 수 있는 최적화 기술을 다룬다. 이 챕터를 마치면 수억 건의 데이터에서도 1초 안에 결과를 얻을 수 있는 쿼리를 작성할 수 있다.

## 10.1 EXPLAIN ANALYZE로 쿼리 분석하기

느린 쿼리를 최적화하려면 먼저 문제가 무엇인지 정확히 파악해야 한다. PostgreSQL과 TimescaleDB는 `EXPLAIN ANALYZE`라는 강력한 도구를 제공한다. 이 명령은 쿼리가 실제로 어떻게 실행되는지, 어디서 시간이 오래 걸리는지를 상세히 보여준다.

**EXPLAIN ANALYZE의 작동 원리**

```
일반 쿼리 실행:
  SQL 입력 → 파싱 → 실행 → 결과 반환

EXPLAIN ANALYZE 실행:
  SQL 입력 → 파싱 → 실행 계획 생성 → 실제 실행 + 성능 측정 → 상세 보고서 반환
```

간단한 예제부터 시작해보자. 게임 서버의 플레이어 로그인 로그를 조회하는 쿼리다.

```sql
EXPLAIN ANALYZE
SELECT 
    player_id,
    COUNT(*) as login_count,
    MAX(time) as last_login
FROM player_login_logs
WHERE time >= NOW() - INTERVAL '7 days'
GROUP BY player_id
ORDER BY login_count DESC
LIMIT 100;
```

실행 결과는 다음과 같은 형태로 나타난다.

```
Limit  (cost=15234.56..15234.81 rows=100 width=24) (actual time=345.123..345.234 rows=100 loops=1)
  ->  Sort  (cost=15234.56..15456.78 rows=88889 width=24) (actual time=345.120..345.180 rows=100 loops=1)
        Sort Key: (count(*)) DESC
        Sort Method: top-N heapsort  Memory: 32kB
        ->  HashAggregate  (cost=12345.67..13234.56 rows=88889 width=24) (actual time=320.456..338.789 rows=45678 loops=1)
              Group Key: player_id
              ->  Custom Scan (ChunkAppend) on player_login_logs  (cost=0.00..8976.54 rows=450123 width=16) (actual time=0.234..245.678 rows=450123 loops=1)
                    Chunks excluded during runtime: 45
                    ->  Index Scan using _hyper_3_15_chunk_player_login_logs_time_idx on _hyper_3_15_chunk  (cost=0.43..2345.67 rows=35000 width=16) (actual time=0.123..23.456 rows=35000 loops=1)
                          Index Cond: (time >= (now() - '7 days'::interval))
Planning Time: 2.345 ms
Execution Time: 345.567 ms
```

이 출력을 읽는 것이 처음에는 어려워 보이지만, 몇 가지 핵심만 알면 이해할 수 있다.

**EXPLAIN ANALYZE 출력 해석하기**

각 줄은 쿼리 실행의 한 단계를 나타낸다. 중요한 요소들을 살펴보자.

**1. Cost (비용 추정치)**
- `cost=15234.56..15234.81`은 PostgreSQL이 추정한 실행 비용이다
- 첫 번째 숫자는 첫 행을 반환하는 비용, 두 번째는 모든 행을 반환하는 비용이다
- 이 값 자체는 의미가 없고 상대적 비교에만 사용된다

**2. Actual Time (실제 소요 시간)**
- `actual time=345.123..345.234`는 밀리초 단위의 실제 실행 시간이다
- 첫 행 반환 시간과 전체 완료 시간을 보여준다
- 이 값이 실제 성능 최적화의 핵심 지표다

**3. Rows (행 수)**
- `rows=100`은 추정 행 수, `actual rows=100`은 실제 반환된 행 수다
- 이 둘의 차이가 크면 통계 정보가 부정확하다는 신호다

**4. Loops (반복 횟수)**
- `loops=1`은 해당 노드가 실행된 횟수다
- Nested Loop Join 같은 경우 수백~수천 번 반복될 수 있다

**5. Planning Time vs Execution Time**
- Planning Time은 쿼리 계획을 세우는 데 걸린 시간이다
- Execution Time이 실제 쿼리 실행 시간이다
- 최적화의 대상은 주로 Execution Time이다

**C#에서 EXPLAIN ANALYZE 실행하기**

SQLKata는 EXPLAIN을 직접 지원하지 않으므로 Npgsql을 사용하여 분석한다.

```csharp
using Npgsql;
using System;
using System.Threading.Tasks;
using System.Collections.Generic;

public class QueryAnalyzer
{
    private readonly string _connectionString;

    public QueryAnalyzer(string connectionString)
    {
        _connectionString = connectionString;
    }

    // 쿼리 실행 계획 분석
    public async Task<QueryPlan> ExplainAnalyzeAsync(string sql)
    {
        await using var connection = new NpgsqlConnection(_connectionString);
        await connection.OpenAsync();

        var explainSql = $"EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) {sql}";
        
        await using var cmd = new NpgsqlCommand(explainSql, connection);
        await using var reader = await cmd.ExecuteReaderAsync();

        if (await reader.ReadAsync())
        {
            var jsonResult = reader.GetString(0);
            return ParseQueryPlan(jsonResult);
        }

        return null;
    }

    // 간단한 텍스트 형식으로 EXPLAIN 실행
    public async Task<List<string>> ExplainAnalyzeTextAsync(string sql)
    {
        await using var connection = new NpgsqlConnection(_connectionString);
        await connection.OpenAsync();

        var explainSql = $"EXPLAIN (ANALYZE, BUFFERS) {sql}";
        
        await using var cmd = new NpgsqlCommand(explainSql, connection);
        await using var reader = await cmd.ExecuteReaderAsync();

        var lines = new List<string>();
        while (await reader.ReadAsync())
        {
            lines.Add(reader.GetString(0));
        }

        return lines;
    }

    // 실행 계획 요약 정보 추출
    public async Task<ExecutionSummary> GetExecutionSummaryAsync(string sql)
    {
        var lines = await ExplainAnalyzeTextAsync(sql);
        
        var summary = new ExecutionSummary
        {
            OriginalQuery = sql,
            PlanLines = lines
        };

        // 실행 시간 추출
        var lastLine = lines[^1];
        if (lastLine.Contains("Execution Time:"))
        {
            var timeStr = lastLine.Split(':')[1].Trim().Replace(" ms", "");
            summary.ExecutionTimeMs = double.Parse(timeStr);
        }

        // Planning 시간 추출
        var planningLine = lines[^2];
        if (planningLine.Contains("Planning Time:"))
        {
            var timeStr = planningLine.Split(':')[1].Trim().Replace(" ms", "");
            summary.PlanningTimeMs = double.Parse(timeStr);
        }

        // Seq Scan 감지 (전체 스캔은 성능 문제의 주범)
        summary.HasSequentialScan = lines.Any(l => l.Contains("Seq Scan"));
        
        // Index Scan 감지
        summary.HasIndexScan = lines.Any(l => l.Contains("Index Scan"));

        // Chunk 제외 정보 (TimescaleDB 특화)
        var chunkLine = lines.FirstOrDefault(l => l.Contains("Chunks excluded"));
        if (chunkLine != null)
        {
            var match = System.Text.RegularExpressions.Regex.Match(
                chunkLine, @"Chunks excluded during runtime: (\d+)");
            if (match.Success)
            {
                summary.ChunksExcluded = int.Parse(match.Groups[1].Value);
            }
        }

        return summary;
    }

    // 쿼리 비교 도구
    public async Task<ComparisonResult> CompareQueriesAsync(
        string query1, 
        string query2, 
        string description1 = "Before", 
        string description2 = "After")
    {
        Console.WriteLine($"\n=== 쿼리 성능 비교: {description1} vs {description2} ===\n");

        var summary1 = await GetExecutionSummaryAsync(query1);
        var summary2 = await GetExecutionSummaryAsync(query2);

        var result = new ComparisonResult
        {
            Before = summary1,
            After = summary2,
            BeforeDescription = description1,
            AfterDescription = description2
        };

        result.CalculateImprovement();

        return result;
    }

    private QueryPlan ParseQueryPlan(string json)
    {
        // JSON 파싱 로직 (System.Text.Json 사용)
        // 실제 구현에서는 JSON을 파싱하여 구조화된 객체로 변환
        return new QueryPlan { RawJson = json };
    }
}

public class ExecutionSummary
{
    public string OriginalQuery { get; set; }
    public List<string> PlanLines { get; set; }
    public double ExecutionTimeMs { get; set; }
    public double PlanningTimeMs { get; set; }
    public double TotalTimeMs => ExecutionTimeMs + PlanningTimeMs;
    public bool HasSequentialScan { get; set; }
    public bool HasIndexScan { get; set; }
    public int ChunksExcluded { get; set; }

    public void Print()
    {
        Console.WriteLine("=== 실행 계획 요약 ===");
        Console.WriteLine($"Planning Time: {PlanningTimeMs:F2} ms");
        Console.WriteLine($"Execution Time: {ExecutionTimeMs:F2} ms");
        Console.WriteLine($"Total Time: {TotalTimeMs:F2} ms");
        Console.WriteLine($"Index Scan 사용: {(HasIndexScan ? "✓" : "✗")}");
        Console.WriteLine($"Sequential Scan 발생: {(HasSequentialScan ? "⚠ 주의 필요" : "없음")}");
        
        if (ChunksExcluded > 0)
        {
            Console.WriteLine($"제외된 Chunk: {ChunksExcluded}개 (파티션 프루닝 적용됨)");
        }

        Console.WriteLine("\n상세 실행 계획:");
        foreach (var line in PlanLines)
        {
            Console.WriteLine($"  {line}");
        }
    }
}

public class ComparisonResult
{
    public ExecutionSummary Before { get; set; }
    public ExecutionSummary After { get; set; }
    public string BeforeDescription { get; set; }
    public string AfterDescription { get; set; }
    public double ImprovementPercent { get; set; }
    public double TimeSavedMs { get; set; }

    public void CalculateImprovement()
    {
        TimeSavedMs = Before.ExecutionTimeMs - After.ExecutionTimeMs;
        ImprovementPercent = (TimeSavedMs / Before.ExecutionTimeMs) * 100;
    }

    public void Print()
    {
        Console.WriteLine($"\n{BeforeDescription}:");
        Console.WriteLine($"  실행 시간: {Before.ExecutionTimeMs:F2} ms");
        Console.WriteLine($"  Sequential Scan: {(Before.HasSequentialScan ? "있음" : "없음")}");

        Console.WriteLine($"\n{AfterDescription}:");
        Console.WriteLine($"  실행 시간: {After.ExecutionTimeMs:F2} ms");
        Console.WriteLine($"  Sequential Scan: {(After.HasSequentialScan ? "있음" : "없음")}");

        Console.WriteLine($"\n개선 결과:");
        Console.WriteLine($"  시간 단축: {TimeSavedMs:F2} ms");
        Console.WriteLine($"  성능 향상: {ImprovementPercent:F2}%");

        if (ImprovementPercent > 50)
        {
            Console.WriteLine("  ✓ 큰 폭의 성능 개선!");
        }
        else if (ImprovementPercent > 20)
        {
            Console.WriteLine("  ✓ 의미 있는 성능 개선");
        }
        else if (ImprovementPercent < 0)
        {
            Console.WriteLine("  ⚠ 성능이 오히려 저하됨");
        }
    }
}

public class QueryPlan
{
    public string RawJson { get; set; }
}
```

**EXPLAIN ANALYZE 해석의 핵심 패턴**

실행 계획에서 성능 문제를 나타내는 몇 가지 신호가 있다.

```
🚨 위험 신호:

1. Seq Scan (Sequential Scan)
   → 전체 테이블 스캔, 인덱스가 없거나 사용되지 않음
   
2. actual time이 cost보다 훨씬 큼
   → 통계 정보가 부정확하거나 예상치 못한 문제 발생
   
3. actual rows >> estimated rows
   → 카디널리티 추정 오류, ANALYZE 실행 필요
   
4. Nested Loop with high loops
   → 중첩 루프가 수천 번 반복, JOIN 전략 재검토 필요
   
5. Sort with "external merge" or "disk"
   → 메모리 부족으로 디스크 사용, work_mem 증가 필요

✅ 좋은 신호:

1. Index Scan / Index Only Scan
   → 인덱스를 효율적으로 사용 중
   
2. Custom Scan (ChunkAppend)
   → TimescaleDB가 파티션 프루닝 적용 중
   
3. Chunks excluded during runtime: N
   → N개의 Chunk를 스캔하지 않음 (매우 효율적)
   
4. Hash Join / Merge Join
   → 대용량 JOIN에 효율적인 전략 사용
```

## 10.2 효과적인 인덱스 전략

인덱스는 쿼리 성능 최적화의 가장 강력한 도구다. 하지만 무작정 인덱스를 만들면 오히려 INSERT/UPDATE 성능이 저하되고 디스크 공간만 낭비된다. 적재적소에 필요한 인덱스만 만드는 것이 중요하다.

**인덱스의 작동 원리**

```
인덱스 없이 검색 (Sequential Scan):
┌─────────────────────────────────────┐
│ Row 1: player_id=1001, time=...    │ ← 처음부터
│ Row 2: player_id=2345, time=...    │
│ Row 3: player_id=1001, time=...    │ ← 이것도 확인
│ Row 4: player_id=5678, time=...    │
│ ...                                 │
│ Row 1000000: player_id=1001, ...   │ ← 끝까지 전부 스캔
└─────────────────────────────────────┘
전체 100만 건을 모두 읽어야 함

인덱스로 검색 (Index Scan):
┌──────────────────┐
│   인덱스 (B-Tree) │
├──────────────────┤
│ 1001 → Row 1    │ ← 바로 찾음!
│ 1001 → Row 3    │
│ 1001 → Row 999  │
│ 2345 → Row 2    │
│ 5678 → Row 4    │
└──────────────────┘
3번의 읽기로 끝!
```

**TimescaleDB의 기본 인덱스**

Hypertable을 생성하면 TimescaleDB는 자동으로 시간 컬럼에 인덱스를 만든다. 하지만 다른 조건으로 검색할 때는 추가 인덱스가 필요하다.

```sql
-- Hypertable 생성 시 자동으로 만들어지는 인덱스
CREATE TABLE server_metrics (
    time TIMESTAMPTZ NOT NULL,
    server_name TEXT NOT NULL,
    metric_name TEXT NOT NULL,
    value DOUBLE PRECISION
);

SELECT create_hypertable('server_metrics', 'time');

-- 자동 생성되는 인덱스:
-- 1. time 컬럼에 대한 B-tree 인덱스 (각 Chunk마다)
```

**게임 서버에서 자주 사용하는 인덱스 패턴**

게임 서버 로그를 조회하는 전형적인 쿼리 패턴을 보자.

```sql
-- 패턴 1: 시간 + 서버 이름으로 필터링
SELECT * FROM server_metrics
WHERE time >= NOW() - INTERVAL '1 hour'
  AND server_name = 'game-server-01';

-- 패턴 2: 시간 + 여러 조건
SELECT * FROM player_actions
WHERE time >= NOW() - INTERVAL '24 hours'
  AND player_id = 12345
  AND action_type = 'item_use';

-- 패턴 3: 집계 쿼리
SELECT 
    server_name,
    AVG(value) as avg_cpu
FROM server_metrics
WHERE time >= NOW() - INTERVAL '7 days'
  AND metric_name = 'CPU'
GROUP BY server_name;
```

이런 패턴에 맞춰 인덱스를 설계한다.

```sql
-- 복합 인덱스 1: (time, server_name)
CREATE INDEX idx_server_metrics_time_server 
ON server_metrics (time DESC, server_name);

-- 복합 인덱스 2: (time, player_id, action_type)
CREATE INDEX idx_player_actions_time_player_action
ON player_actions (time DESC, player_id, action_type);

-- 복합 인덱스 3: (metric_name, time, server_name)
-- WHERE metric_name = 'CPU'를 먼저 필터링
CREATE INDEX idx_server_metrics_metric_time_server
ON server_metrics (metric_name, time DESC, server_name);
```

**복합 인덱스 순서의 중요성**

복합 인덱스는 컬럼 순서가 매우 중요하다. 전화번호부를 생각해보자.

```
좋은 인덱스: (성, 이름, 생년월일)
"김" → "김철수" → "1990-01-01"
"김" → "김철수" → "1992-05-15"
"김" → "김영희" → "1991-03-20"

나쁜 인덱스: (생년월일, 성, 이름)
"1990-01-01" → "김" → "김철수"
"1990-01-02" → "박" → "박민수"
"1990-01-03" → "이" → "이지은"
```

"김씨 성을 가진 사람"을 찾을 때, 첫 번째 인덱스는 빠르게 찾지만 두 번째는 전체를 스캔해야 한다.

**C#에서 인덱스 관리하기**

```csharp
public class IndexManager
{
    private readonly string _connectionString;

    public IndexManager(string connectionString)
    {
        _connectionString = connectionString;
    }

    // 복합 인덱스 생성
    public async Task CreateCompositeIndexAsync(
        string tableName,
        string indexName,
        string[] columns,
        bool descending = false)
    {
        await using var connection = new NpgsqlConnection(_connectionString);
        await connection.OpenAsync();

        var columnList = string.Join(", ", columns.Select(c => 
            descending && c == "time" ? $"{c} DESC" : c));

        var sql = $@"
            CREATE INDEX IF NOT EXISTS {indexName}
            ON {tableName} ({columnList});";

        await using var cmd = new NpgsqlCommand(sql, connection);
        
        var stopwatch = System.Diagnostics.Stopwatch.StartNew();
        await cmd.ExecuteNonQueryAsync();
        stopwatch.Stop();

        Console.WriteLine($"인덱스 생성 완료: {indexName} ({stopwatch.ElapsedMilliseconds}ms)");
    }

    // 인덱스 사용 통계 조회
    public async Task<List<IndexUsageStats>> GetIndexUsageStatsAsync(string tableName)
    {
        await using var connection = new NpgsqlConnection(_connectionString);
        await connection.OpenAsync();

        var sql = $@"
            SELECT 
                schemaname,
                tablename,
                indexname,
                idx_scan as index_scans,
                idx_tup_read as tuples_read,
                idx_tup_fetch as tuples_fetched,
                pg_size_pretty(pg_relation_size(indexrelid)) as index_size
            FROM pg_stat_user_indexes
            WHERE tablename = @tableName
            ORDER BY idx_scan DESC;";

        await using var cmd = new NpgsqlCommand(sql, connection);
        cmd.Parameters.AddWithValue("tableName", tableName);

        await using var reader = await cmd.ExecuteReaderAsync();

        var stats = new List<IndexUsageStats>();
        while (await reader.ReadAsync())
        {
            stats.Add(new IndexUsageStats
            {
                SchemaName = reader.GetString(0),
                TableName = reader.GetString(1),
                IndexName = reader.GetString(2),
                IndexScans = reader.GetInt64(3),
                TuplesRead = reader.GetInt64(4),
                TuplesFetched = reader.GetInt64(5),
                IndexSize = reader.GetString(6)
            });
        }

        return stats;
    }

    // 사용되지 않는 인덱스 찾기
    public async Task<List<string>> FindUnusedIndexesAsync(int minScanThreshold = 10)
    {
        await using var connection = new NpgsqlConnection(_connectionString);
        await connection.OpenAsync();

        var sql = $@"
            SELECT 
                schemaname || '.' || indexname as full_index_name,
                pg_size_pretty(pg_relation_size(indexrelid)) as size,
                idx_scan as scans
            FROM pg_stat_user_indexes
            WHERE idx_scan < @threshold
              AND indexrelname NOT LIKE '%_pkey'
              AND indexrelname NOT LIKE '%_hyper_%'
            ORDER BY pg_relation_size(indexrelid) DESC;";

        await using var cmd = new NpgsqlCommand(sql, connection);
        cmd.Parameters.AddWithValue("threshold", minScanThreshold);

        await using var reader = await cmd.ExecuteReaderAsync();

        var unusedIndexes = new List<string>();
        while (await reader.ReadAsync())
        {
            var indexName = reader.GetString(0);
            var size = reader.GetString(1);
            var scans = reader.GetInt64(2);

            unusedIndexes.Add($"{indexName} (크기: {size}, 스캔: {scans}회)");
        }

        return unusedIndexes;
    }

    // 중복 인덱스 찾기
    public async Task<List<string>> FindDuplicateIndexesAsync()
    {
        await using var connection = new NpgsqlConnection(_connectionString);
        await connection.OpenAsync();

        var sql = @"
            SELECT 
                pg_size_pretty(SUM(pg_relation_size(idx))::BIGINT) AS size,
                (array_agg(idx))[1] AS idx1,
                (array_agg(idx))[2] AS idx2,
                (array_agg(idx))[3] AS idx3,
                (array_agg(idx))[4] AS idx4
            FROM (
                SELECT 
                    indexrelid::regclass AS idx,
                    (indrelid::text ||E'\n'|| indclass::text ||E'\n'|| 
                     indkey::text ||E'\n'|| COALESCE(indexprs::text,'')||E'\n' ||
                     COALESCE(indpred::text,'')) AS key
                FROM pg_index
            ) sub
            GROUP BY key 
            HAVING COUNT(*) > 1
            ORDER BY SUM(pg_relation_size(idx)) DESC;";

        await using var cmd = new NpgsqlCommand(sql, connection);
        await using var reader = await cmd.ExecuteReaderAsync();

        var duplicates = new List<string>();
        while (await reader.ReadAsync())
        {
            var size = reader.GetString(0);
            var indexes = new List<string>();
            
            for (int i = 1; i < 5; i++)
            {
                if (!reader.IsDBNull(i))
                {
                    indexes.Add(reader.GetString(i));
                }
            }

            duplicates.Add($"중복 그룹 (낭비 공간: {size}): {string.Join(", ", indexes)}");
        }

        return duplicates;
    }
```

## 10.3 복합 조건 최적화

실제 게임 서버 쿼리는 단순하지 않다. 여러 조건을 동시에 만족해야 하고, 때로는 OR 조건도 포함된다. 이런 복잡한 조건을 최적화하는 방법을 알아보자.

**문제가 되는 쿼리 패턴**

```csharp
// 나쁜 예: OR 조건이 많은 쿼리
var badQuery = await db.Query("player_actions")
    .Where("time", ">=", DateTime.UtcNow.AddDays(-7))
    .Where(q => q
        .Where("action_type", "item_purchase")
        .OrWhere("action_type", "item_use")
        .OrWhere("action_type", "item_trade")
    )
    .GetAsync();

// 생성되는 SQL:
// SELECT * FROM player_actions
// WHERE time >= '2024-12-24'
//   AND (action_type = 'item_purchase' 
//        OR action_type = 'item_use' 
//        OR action_type = 'item_trade');
// → OR 조건은 인덱스를 효율적으로 사용하지 못함
```

**최적화된 패턴: IN 사용**

```csharp
// 좋은 예: IN 조건 사용
var goodQuery = await db.Query("player_actions")
    .Where("time", ">=", DateTime.UtcNow.AddDays(-7))
    .WhereIn("action_type", new[] { "item_purchase", "item_use", "item_trade" })
    .GetAsync();

// 생성되는 SQL:
// SELECT * FROM player_actions
// WHERE time >= '2024-12-24'
//   AND action_type IN ('item_purchase', 'item_use', 'item_trade');
// → IN 조건은 인덱스를 효율적으로 사용 가능
```

**복잡한 조건 분해하기**

때로는 복잡한 쿼리를 여러 개의 단순한 쿼리로 나누는 것이 더 빠르다.

```csharp
public class ComplexQueryOptimizer
{
    private readonly QueryFactory _db;

    public ComplexQueryOptimizer(QueryFactory db)
    {
        _db = db;
    }

    // 비최적화 버전: 복잡한 JOIN과 서브쿼리
    public async Task<IEnumerable<dynamic>> GetPlayerSummarySlowAsync(DateTime startTime)
    {
        // 이 쿼리는 느릴 수 있음
        return await _db.Query("players as p")
            .Join("player_actions as pa", "p.player_id", "pa.player_id")
            .LeftJoin("player_purchases as pp", "p.player_id", "pp.player_id")
            .Where("pa.time", ">=", startTime)
            .Where("pp.time", ">=", startTime)
            .Select("p.player_id")
            .SelectRaw("COUNT(DISTINCT pa.action_id) as total_actions")
            .SelectRaw("COUNT(DISTINCT pp.purchase_id) as total_purchases")
            .SelectRaw("SUM(pp.amount) as total_spent")
            .GroupBy("p.player_id")
            .GetAsync();
    }

    // 최적화 버전: 쿼리 분해 및 애플리케이션 레벨 조인
    public async Task<List<PlayerSummary>> GetPlayerSummaryFastAsync(DateTime startTime)
    {
        // 1. 플레이어 목록 조회
        var players = (await _db.Query("players")
            .Select("player_id", "player_name")
            .GetAsync())
            .ToList();

        // 2. 액션 통계 조회 (인덱스 활용)
        var actions = (await _db.Query("player_actions")
            .Where("time", ">=", startTime)
            .Select("player_id")
            .SelectRaw("COUNT(*) as action_count")
            .GroupBy("player_id")
            .GetAsync())
            .ToDictionary(
                x => (int)((IDictionary<string, object>)x)["player_id"],
                x => (long)((IDictionary<string, object>)x)["action_count"]
            );

        // 3. 구매 통계 조회 (인덱스 활용)
        var purchases = (await _db.Query("player_purchases")
            .Where("time", ">=", startTime)
            .Select("player_id")
            .SelectRaw("COUNT(*) as purchase_count")
            .SelectRaw("SUM(amount) as total_amount")
            .GroupBy("player_id")
            .GetAsync())
            .ToDictionary(
                x => (int)((IDictionary<string, object>)x)["player_id"],
                x => new {
                    Count = (long)((IDictionary<string, object>)x)["purchase_count"],
                    Amount = ((IDictionary<string, object>)x)["total_amount"] as decimal? ?? 0
                }
            );

        // 4. 애플리케이션 레벨에서 결합
        var results = new List<PlayerSummary>();
        foreach (var player in players)
        {
            var dict = player as IDictionary<string, object>;
            var playerId = (int)dict["player_id"];

            results.Add(new PlayerSummary
            {
                PlayerId = playerId,
                PlayerName = dict["player_name"].ToString(),
                TotalActions = actions.GetValueOrDefault(playerId, 0),
                TotalPurchases = purchases.ContainsKey(playerId) ? purchases[playerId].Count : 0,
                TotalSpent = purchases.ContainsKey(playerId) ? purchases[playerId].Amount : 0
            });
        }

        return results;
    }
```

## 10.4 윈도우 함수 활용

윈도우 함수는 복잡한 분석 쿼리를 간결하고 효율적으로 작성할 수 있게 해준다. 특히 순위 계산, 이동 평균, 누적 합계 등에 유용하다.

**기본 윈도우 함수 예제**

```csharp
public class WindowFunctionExamples
{
    private readonly QueryFactory _db;

    public WindowFunctionExamples(QueryFactory db)
    {
        _db = db;
    }

    // 서버별 CPU 사용률 순위
    public async Task<IEnumerable<dynamic>> GetServerRankingAsync()
    {
        var sql = @"
            SELECT 
                time_bucket('1 hour', time) as hour,
                server_name,
                AVG(value) as avg_cpu,
                RANK() OVER (
                    PARTITION BY time_bucket('1 hour', time) 
                    ORDER BY AVG(value) DESC
                ) as cpu_rank
            FROM server_metrics
            WHERE time >= NOW() - INTERVAL '24 hours'
              AND metric_name = 'CPU'
            GROUP BY hour, server_name
            ORDER BY hour DESC, cpu_rank";

        return await _db.StatementAsync(sql);
    }

    // 플레이어별 연속 로그인 일수 계산
    public async Task<IEnumerable<dynamic>> GetLoginStreakAsync()
    {
        var sql = @"
            WITH daily_logins AS (
                SELECT 
                    player_id,
                    DATE(time) as login_date
                FROM player_login_logs
                WHERE time >= NOW() - INTERVAL '30 days'
                GROUP BY player_id, DATE(time)
            ),
            streak_groups AS (
                SELECT 
                    player_id,
                    login_date,
                    login_date - (ROW_NUMBER() OVER (
                        PARTITION BY player_id 
                        ORDER BY login_date
                    ))::integer * INTERVAL '1 day' as streak_group
                FROM daily_logins
            )
            SELECT 
                player_id,
                MIN(login_date) as streak_start,
                MAX(login_date) as streak_end,
                COUNT(*) as streak_days
            FROM streak_groups
            GROUP BY player_id, streak_group
            HAVING COUNT(*) >= 3
            ORDER BY streak_days DESC
            LIMIT 100";

        return await _db.StatementAsync(sql);
    }

    // 이동 평균 계산 (7일 이동 평균)
    public async Task<IEnumerable<dynamic>> GetMovingAverageAsync(string serverName)
    {
        var sql = @"
            SELECT 
                time_bucket('1 day', time) as day,
                AVG(value) as daily_avg,
                AVG(AVG(value)) OVER (
                    ORDER BY time_bucket('1 day', time)
                    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
                ) as moving_avg_7d
            FROM server_metrics
            WHERE server_name = @serverName
              AND metric_name = 'CPU'
              AND time >= NOW() - INTERVAL '30 days'
            GROUP BY day
            ORDER BY day";

        return await _db.StatementAsync(sql, new { serverName });
    }

    // 누적 합계 (당일 누적 접속자)
    public async Task<IEnumerable<dynamic>> GetCumulativeDailyLoginsAsync(DateTime date)
    {
        var sql = @"
            SELECT 
                time_bucket('1 hour', time) as hour,
                COUNT(DISTINCT player_id) as hourly_logins,
                SUM(COUNT(DISTINCT player_id)) OVER (
                    ORDER BY time_bucket('1 hour', time)
                ) as cumulative_logins
            FROM player_login_logs
            WHERE DATE(time) = @date
            GROUP BY hour
            ORDER BY hour";

        return await _db.StatementAsync(sql, new { date = date.Date });
    }

    // LAG/LEAD를 사용한 변화율 계산
    public async Task<IEnumerable<dynamic>> GetMetricChangeRateAsync(string serverName, string metricName)
    {
        var sql = @"
            WITH hourly_metrics AS (
                SELECT 
                    time_bucket('1 hour', time) as hour,
                    AVG(value) as avg_value
                FROM server_metrics
                WHERE server_name = @serverName
                  AND metric_name = @metricName
                  AND time >= NOW() - INTERVAL '24 hours'
                GROUP BY hour
            )
            SELECT 
                hour,
                avg_value as current_value,
                LAG(avg_value) OVER (ORDER BY hour) as previous_value,
                ROUND(
                    ((avg_value - LAG(avg_value) OVER (ORDER BY hour)) / 
                     LAG(avg_value) OVER (ORDER BY hour) * 100)::numeric, 
                    2
                ) as change_percent
            FROM hourly_metrics
            ORDER BY hour";

        return await _db.StatementAsync(sql, new { serverName, metricName });
    }

    // NTILE을 사용한 백분위 분석
    public async Task<IEnumerable<dynamic>> GetPlayerSpendingPercentileAsync()
    {
        var sql = @"
            WITH player_spending AS (
                SELECT 
                    player_id,
                    SUM(amount) as total_spent
                FROM player_purchases
                WHERE time >= NOW() - INTERVAL '30 days'
                GROUP BY player_id
            )
            SELECT 
                player_id,
                total_spent,
                NTILE(100) OVER (ORDER BY total_spent) as percentile,
                CASE 
                    WHEN NTILE(100) OVER (ORDER BY total_spent) >= 90 THEN 'Whale'
                    WHEN NTILE(100) OVER (ORDER BY total_spent) >= 70 THEN 'Dolphin'
                    WHEN NTILE(100) OVER (ORDER BY total_spent) >= 50 THEN 'Minnow'
                    ELSE 'Free'
                END as player_tier
            FROM player_spending
            ORDER BY total_spent DESC";

        return await _db.StatementAsync(sql);
    }
```

**윈도우 함수 성능 최적화 팁**

```
✅ 최적화 팁:

1. PARTITION BY 컬럼에 인덱스 생성
   → 파티션별로 데이터를 빠르게 그룹화

2. ORDER BY 컬럼도 인덱스에 포함
   → 정렬 작업 최적화

3. 윈도우 프레임 크기 제한
   → ROWS BETWEEN 100 PRECEDING AND CURRENT ROW
   → 전체 파티션이 아닌 필요한 범위만 계산

4. 복잡한 윈도우 함수는 CTE로 분리
   → 가독성 향상 및 중간 결과 재사용

5. 필요한 경우 Materialized View 활용
   → 자주 사용하는 윈도우 함수 결과를 미리 계산
```

## 10.5 파티션 프루닝 이해하기

TimescaleDB의 가장 강력한 최적화 기능 중 하나가 파티션 프루닝이다. 시간 범위를 기반으로 불필요한 Chunk를 자동으로 스킵하여 쿼리 속도를 극적으로 향상시킨다.

**파티션 프루닝의 작동 방식**

```
전체 데이터 (압축 포함):
┌────────────────────────────────────────────────────────┐
│ Chunk 1: 2024-10-01 ~ 2024-10-07 (압축됨, 1GB)       │
│ Chunk 2: 2024-10-08 ~ 2024-10-14 (압축됨, 1GB)       │
│ Chunk 3: 2024-10-15 ~ 2024-10-21 (압축됨, 1GB)       │
│ ...                                                    │
│ Chunk 12: 2024-12-22 ~ 2024-12-28 (활성, 500MB)      │
│ Chunk 13: 2024-12-29 ~ 현재 (활성, 200MB)            │
└────────────────────────────────────────────────────────┘

쿼리: WHERE time >= '2024-12-25'
      
파티션 프루닝 적용:
┌────────────────────────────────────────────────────────┐
│ Chunk 1-11: ✗ 스캔하지 않음 (시간 범위 불일치)        │
│ Chunk 12: ✓ 스캔 (일부 데이터 포함, 500MB)            │

결과: 13GB → 0.7GB만 스캔 (94% 감소!)
```

```csharp
               OR time >= '2024-12-01' AND time < '2024-12-07'";

        return await _db.StatementAsync(sql);

    // 파티션 프루닝 확인 도구
    public async Task<PartitionPruningReport> CheckPartitionPruningAsync(string query)
    {
        await using var connection = new NpgsqlConnection(_connectionString);
        await connection.OpenAsync();

        var explainSql = $"EXPLAIN (ANALYZE, BUFFERS) {query}";
        
        await using var cmd = new NpgsqlCommand(explainSql, connection);
        await using var reader = await cmd.ExecuteReaderAsync();

        var plan = new List<string>();
        while (await reader.ReadAsync())
        {
            plan.Add(reader.GetString(0));
        }

        var report = new PartitionPruningReport();

        // ChunkAppend 노드 찾기
        var chunkAppendLine = plan.FirstOrDefault(l => l.Contains("Custom Scan (ChunkAppend)"));
        if (chunkAppendLine != null)
        {
            report.UsesChunkAppend = true;
        }

        // 제외된 Chunk 수 찾기
        var excludedLine = plan.FirstOrDefault(l => l.Contains("Chunks excluded during runtime"));
        if (excludedLine != null)
        {
            var match = System.Text.RegularExpressions.Regex.Match(
                excludedLine, @"Chunks excluded during runtime: (\d+)");
            if (match.Success)
            {
                report.ChunksExcluded = int.Parse(match.Groups[1].Value);
            }
        }

        // 실제 스캔된 Chunk 수 계산
        var chunkScans = plan.Count(l => l.Contains("_hyper_") && l.Contains("Scan"));
        report.ChunksScanned = chunkScans;

        report.FullPlan = plan;

        return report;
    }
```

```csharp
public class SlowQueryCase1
{
    private readonly QueryFactory _db;
    private readonly QueryAnalyzer _analyzer;

    public SlowQueryCase1(QueryFactory db, QueryAnalyzer analyzer)
    {
        _db = db;
        _analyzer = analyzer;
    }

    // 문제: 실행 시간 15초
    public async Task<IEnumerable<dynamic>> OriginalSlowQueryAsync()
    {
        var sql = @"
            SELECT 
                p.player_id,
                p.player_name,
                p.level,
                COUNT(DISTINCT pa.action_id) as total_actions,
                COUNT(DISTINCT pl.login_id) as login_count,
                MAX(pl.time) as last_login,
                COALESCE(SUM(pp.amount), 0) as total_spent
            FROM players p
            LEFT JOIN player_actions pa 
                ON p.player_id = pa.player_id 
                AND pa.time >= NOW() - INTERVAL '30 days'
            LEFT JOIN player_login_logs pl 
                ON p.player_id = pl.player_id 
                AND pl.time >= NOW() - INTERVAL '30 days'
            LEFT JOIN player_purchases pp 
                ON p.player_id = pp.player_id 
                AND pp.time >= NOW() - INTERVAL '30 days'
            GROUP BY p.player_id, p.player_name, p.level
            ORDER BY total_actions DESC
            LIMIT 100";

        return await _db.StatementAsync(sql);
    }

    // 1단계: 분석
    public async Task AnalyzeSlowQueryAsync()
    {
        var query = await GetOriginalQuerySqlAsync();
        var summary = await _analyzer.GetExecutionSummaryAsync(query);
        
        Console.WriteLine("=== 원본 쿼리 분석 ===");
        summary.Print();

        // 문제점 진단:
        // 1. 3개의 테이블을 LEFT JOIN
        // 2. 각 JOIN마다 시간 조건 필터링
        // 3. 플레이어 수가 많으면 cartesian product 위험
    }

    // 2단계: 최적화 - 쿼리 분해
    public async Task<List<PlayerDashboard>> OptimizedQuery_Step1_Async()
    {
        Console.WriteLine("=== 1단계 최적화: 쿼리 분해 ===");

        var startTime = DateTime.UtcNow.AddDays(-30);

        // 각 통계를 별도로 계산
        var actionStats = (await _db.Query("player_actions")
            .Where("time", ">=", startTime)
            .Select("player_id")
            .SelectRaw("COUNT(*) as action_count")
            .GroupBy("player_id")
            .GetAsync())
            .ToDictionary(
                x => (int)((IDictionary<string, object>)x)["player_id"],
                x => (long)((IDictionary<string, object>)x)["action_count"]
            );

        var loginStats = (await _db.Query("player_login_logs")
            .Where("time", ">=", startTime)
            .Select("player_id")
            .SelectRaw("COUNT(*) as login_count")
            .SelectRaw("MAX(time) as last_login")
            .GroupBy("player_id")
            .GetAsync())
            .ToDictionary(
                x => (int)((IDictionary<string, object>)x)["player_id"],
                x => new {
                    Count = (long)((IDictionary<string, object>)x)["login_count"],
                    LastLogin = (DateTime)((IDictionary<string, object>)x)["last_login"]
                }
            );

        var purchaseStats = (await _db.Query("player_purchases")
            .Where("time", ">=", startTime)
            .Select("player_id")
            .SelectRaw("SUM(amount) as total_spent")
            .GroupBy("player_id")
            .GetAsync())
            .ToDictionary(
                x => (int)((IDictionary<string, object>)x)["player_id"],
                x => (decimal)((IDictionary<string, object>)x)["total_spent"]
            );

        // 활동이 있는 플레이어만 조회
        var activePlayerIds = actionStats.Keys
            .Union(loginStats.Keys)
            .Union(purchaseStats.Keys)
            .ToList();

        var players = (await _db.Query("players")
            .WhereIn("player_id", activePlayerIds)
            .Select("player_id", "player_name", "level")
            .GetAsync())
            .ToList();

        // 조합
        var results = players.Select(p =>
        {
            var dict = p as IDictionary<string, object>;
            var playerId = (int)dict["player_id"];

            return new PlayerDashboard
            {
                PlayerId = playerId,
                PlayerName = dict["player_name"].ToString(),
                Level = (int)dict["level"],
                TotalActions = actionStats.GetValueOrDefault(playerId, 0),
                LoginCount = loginStats.ContainsKey(playerId) ? loginStats[playerId].Count : 0,
                LastLogin = loginStats.ContainsKey(playerId) ? loginStats[playerId].LastLogin : default,
                TotalSpent = purchaseStats.GetValueOrDefault(playerId, 0)
            };
        })
        .OrderByDescending(x => x.TotalActions)
        .Take(100)
        .ToList();

        return results;
    }

    }

    // 4단계: Materialized View 활용
    public async Task OptimizedQuery_Step3_MaterializedViewAsync()
    {
        Console.WriteLine("=== 3단계 최적화: Materialized View 생성 ===");

        await using var connection = new NpgsqlConnection(_connectionString);
        await connection.OpenAsync();

        var sql = @"
            CREATE MATERIALIZED VIEW IF NOT EXISTS player_activity_summary AS
            SELECT 
                p.player_id,
                p.player_name,
                p.level,
                COALESCE(pa.action_count, 0) as total_actions,
                COALESCE(pl.login_count, 0) as login_count,
                pl.last_login,
                COALESCE(pp.total_spent, 0) as total_spent,
                NOW() as last_updated
            FROM players p
            LEFT JOIN (
                SELECT 
                    player_id,
                    COUNT(*) as action_count
                FROM player_actions
                WHERE time >= NOW() - INTERVAL '30 days'
                GROUP BY player_id
            ) pa ON p.player_id = pa.player_id
            LEFT JOIN (
                SELECT 
                    player_id,
                    COUNT(*) as login_count,
                    MAX(time) as last_login
                FROM player_login_logs
                WHERE time >= NOW() - INTERVAL '30 days'
                GROUP BY player_id
            ) pl ON p.player_id = pl.player_id
            LEFT JOIN (
                SELECT 
                    player_id,
                    SUM(amount) as total_spent
                FROM player_purchases
                WHERE time >= NOW() - INTERVAL '30 days'
                GROUP BY player_id
            ) pp ON p.player_id = pp.player_id;

            CREATE INDEX IF NOT EXISTS idx_player_activity_summary_actions
            ON player_activity_summary (total_actions DESC);
        ";

        await ExecuteSqlAsync(connection, sql);

        // 정기적으로 리프레시 (예: 매 시간)
        await ExecuteSqlAsync(connection, "REFRESH MATERIALIZED VIEW player_activity_summary;");

        Console.WriteLine("Materialized View 생성 완료!");
    }
```

**핵심 요약**

이번 챕터에서는 복잡한 쿼리를 최적화하는 실전 기법을 배웠다. EXPLAIN ANALYZE로 쿼리 실행 계획을 분석하고 병목 지점을 찾는 방법, 효과적인 인덱스 전략을 세워 검색 속도를 극적으로 향상시키는 방법, 복합 조건을 최적화하고 윈도우 함수를 활용하여 복잡한 분석 쿼리를 간결하게 작성하는 방법을 다뤘다. TimescaleDB의 파티션 프루닝을 최대한 활용하면 수억 건의 데이터에서도 필요한 Chunk만 스캔하여 빠른 응답을 얻을 수 있다. 실제 느린 쿼리 사례를 단계별로 최적화하면서 15초 걸리던 쿼리를 10ms 이하로 줄이는 과정을 경험했다. 이러한 기법들을 적용하면 라이브 게임 서버에서 실시간 대시보드와 복잡한 분석 쿼리를 동시에 빠르게 처리할 수 있다.

다음 챕터에서는 데이터를 안전하게 보호하는 백업과 복구 전략을 배운다. 장애가 발생해도 데이터를 잃지 않고 빠르게 복구하는 방법을 살펴본다.
