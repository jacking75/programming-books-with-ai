# Chapter 17: 성능 튜닝 실전 케이스 스터디

이론과 실전은 다르다. 아무리 완벽한 설계와 최적화 기법을 알고 있어도 실제 운영 환경에서는 예상치 못한 성능 문제가 발생한다. 이번 장에서는 실제 라이브 게임 서버에서 발생했던 성능 문제들과 그 해결 과정을 상세히 다룬다. 각 사례는 실제 상황을 기반으로 하며, 문제 발견부터 원인 분석, 해결책 적용, 그리고 사후 모니터링까지 전체 과정을 단계별로 설명한다.

성능 문제는 대부분 갑자기 발생한다. 평소에는 문제없이 잘 돌아가던 시스템이 특정 이벤트나 업데이트 이후 갑자기 느려지거나 멈추는 경우가 많다. 이럴 때 당황하지 않고 체계적으로 문제를 진단하고 해결하는 능력이 필요하다. 이 장에서 소개하는 사례들을 통해 여러분도 실전에서 마주칠 성능 문제에 효과적으로 대응할 수 있게 될 것이다.

## 17.1 사례 1: 갑자기 느려진 대시보드

**상황 설명**

월요일 오전 9시, 운영팀으로부터 급한 연락이 왔다. "대시보드가 너무 느려서 사용할 수 없어요! 데이터가 전혀 로드되지 않습니다." 주말 동안 특별한 배포나 변경 사항은 없었는데 갑자기 실시간 모니터링 대시보드의 응답 속도가 극도로 느려진 것이다. 평소 1-2초 안에 로드되던 대시보드가 30초 이상 걸리거나 타임아웃이 발생했다.

**문제 진단 과정**

먼저 대시보드가 실행하는 쿼리들을 확인한다. TimescaleDB의 `pg_stat_statements` 확장을 사용하면 실행된 모든 쿼리의 통계를 볼 수 있다.

```sql
-- pg_stat_statements 확장 활성화 (최초 1회)
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 가장 느린 쿼리 Top 10 조회
SELECT 
    calls,
    ROUND(total_exec_time::numeric, 2) as total_time_ms,
    ROUND(mean_exec_time::numeric, 2) as avg_time_ms,
    ROUND((100 * total_exec_time / SUM(total_exec_time) OVER ())::numeric, 2) as percentage,
    query
FROM pg_stat_statements
WHERE query NOT LIKE '%pg_stat_statements%'
ORDER BY mean_exec_time DESC
LIMIT 10;
```

이 쿼리를 실행한 결과, 특정 쿼리 하나가 평균 25초가 걸리는 것을 발견했다.

```sql
-- 문제의 쿼리 (간소화 버전)
SELECT 
    time_bucket('5 minutes', event_time) as bucket,
    player_id,
    event_type,
    COUNT(*) as event_count
FROM player_events
WHERE event_time >= NOW() - INTERVAL '24 hours'
GROUP BY bucket, player_id, event_type
ORDER BY bucket DESC;
```

이 쿼리는 최근 24시간 동안의 플레이어 이벤트를 5분 단위로 집계하는데, 문제는 `player_id`와 `event_type`으로도 그룹핑한다는 점이다. 데이터가 누적되면서 최근 24시간 동안의 레코드 수가 수천만 건에 달하게 되었고, 이를 플레이어별, 이벤트 타입별로 그룹핑하는 작업이 매우 무거워진 것이다.

**해결책 1: 인덱스 최적화**

먼저 적절한 인덱스를 추가한다.

```sql
-- 기존 인덱스 확인
SELECT 
    tablename,
    indexname,
    indexdef
FROM pg_indexes
WHERE tablename = 'player_events'
ORDER BY indexname;

-- 복합 인덱스 생성
CREATE INDEX CONCURRENTLY idx_player_events_time_player_type 
ON player_events (event_time DESC, player_id, event_type)
WHERE event_time >= NOW() - INTERVAL '30 days';

-- 통계 업데이트
ANALYZE player_events;
```

`CREATE INDEX CONCURRENTLY`를 사용하면 테이블을 잠그지 않고 인덱스를 생성할 수 있다. 또한 `WHERE` 절을 추가하여 최근 30일 데이터만 인덱싱하는 부분 인덱스(Partial Index)를 만들었다. 이렇게 하면 인덱스 크기가 작아지고 유지보수 비용도 줄어든다.

**해결책 2: 연속 집계 활용**

더 근본적인 해결책은 실시간으로 집계하는 대신 연속 집계를 사용하는 것이다.

```sql
-- 5분 단위 플레이어 이벤트 집계 뷰
CREATE MATERIALIZED VIEW player_events_5min
WITH (timescaledb.continuous) AS
SELECT 
    time_bucket('5 minutes', event_time) as bucket,
    player_id,
    event_type,
    COUNT(*) as event_count,
    COUNT(DISTINCT session_id) as session_count
FROM player_events
GROUP BY bucket, player_id, event_type
WITH NO DATA;

-- 리프레시 정책 설정 (1분마다 갱신)
SELECT add_continuous_aggregate_policy('player_events_5min',
    start_offset => INTERVAL '1 day',
    end_offset => INTERVAL '1 minute',
    schedule_interval => INTERVAL '1 minute');

-- 보관 정책 (30일 후 삭제)
SELECT add_retention_policy('player_events_5min', INTERVAL '30 days');
```

**C# 성능 모니터링 코드**

문제를 조기에 발견하기 위한 모니터링 코드를 추가한다.

```csharp
// GameAnalytics.Infrastructure/Monitoring/QueryPerformanceMonitor.cs
using Npgsql;
using Dapper;
using System.Diagnostics;

namespace GameAnalytics.Infrastructure.Monitoring
{
    public class QueryPerformanceMonitor
    {
        private readonly string _connectionString;
        private readonly ILogger<QueryPerformanceMonitor> _logger;
        
        public QueryPerformanceMonitor(
            string connectionString,
            ILogger<QueryPerformanceMonitor> logger)
        {
            _connectionString = connectionString;
            _logger = logger;
        }
        
        // 느린 쿼리 감지
        public async Task<List<SlowQuery>> DetectSlowQueriesAsync(
            double thresholdMs = 1000)
        {
            using var connection = new NpgsqlConnection(_connectionString);
            
            var sql = @"
                SELECT 
                    queryid,
                    calls,
                    ROUND(total_exec_time::numeric, 2) as total_time_ms,
                    ROUND(mean_exec_time::numeric, 2) as avg_time_ms,
                    ROUND(stddev_exec_time::numeric, 2) as stddev_time_ms,
                    ROUND(max_exec_time::numeric, 2) as max_time_ms,
                    LEFT(query, 200) as query_preview
                FROM pg_stat_statements
                WHERE mean_exec_time > @Threshold
                    AND query NOT LIKE '%pg_stat_statements%'
                ORDER BY mean_exec_time DESC
                LIMIT 20";
            
            var results = await connection.QueryAsync<SlowQuery>(
                sql,
                new { Threshold = thresholdMs }
            );
            
            var slowQueries = results.ToList();
            
            if (slowQueries.Any())
            {
                _logger.LogWarning($"Detected {slowQueries.Count} slow queries exceeding {thresholdMs}ms");
                foreach (var query in slowQueries)
                {
                    _logger.LogWarning(
                        $"Query {query.QueryId}: avg={query.AvgTimeMs}ms, " +
                        $"max={query.MaxTimeMs}ms, calls={query.Calls}");
                }
            }
            
            return slowQueries;
        }
        
        // 쿼리 실행 시간 측정 래퍼
        public async Task<T> ExecuteWithTimingAsync<T>(
            Func<Task<T>> queryFunc,
            string queryName,
            double warningThresholdMs = 1000)
        {
            var stopwatch = Stopwatch.StartNew();
            
            try
            {
                var result = await queryFunc();
                stopwatch.Stop();
                
                var elapsedMs = stopwatch.Elapsed.TotalMilliseconds;
                
                if (elapsedMs > warningThresholdMs)
                {
                    _logger.LogWarning(
                        $"Query '{queryName}' took {elapsedMs:N2}ms " +
                        $"(threshold: {warningThresholdMs}ms)");
                }
                else
                {
                    _logger.LogDebug($"Query '{queryName}' completed in {elapsedMs:N2}ms");
                }
                
                return result;
            }
            catch (Exception ex)
            {
                stopwatch.Stop();
                _logger.LogError(ex, 
                    $"Query '{queryName}' failed after {stopwatch.Elapsed.TotalMilliseconds:N2}ms");
                throw;
            }
        }
        
        // 데이터베이스 통계 리셋 (개발/테스트 환경용)
        public async Task ResetStatisticsAsync()
        {
            using var connection = new NpgsqlConnection(_connectionString);
            await connection.ExecuteAsync("SELECT pg_stat_statements_reset()");
            _logger.LogInformation("Query statistics reset");
        }
    }
    
    public class SlowQuery
    {
        public long QueryId { get; set; }
        public long Calls { get; set; }
        public decimal TotalTimeMs { get; set; }
        public decimal AvgTimeMs { get; set; }
        public decimal StddevTimeMs { get; set; }
        public decimal MaxTimeMs { get; set; }
        public string QueryPreview { get; set; }
    }
}
```

```csharp
// GameAnalytics.API/Middleware/QueryTimeoutMiddleware.cs
public class QueryTimeoutMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<QueryTimeoutMiddleware> _logger;
    
    public QueryTimeoutMiddleware(
        RequestDelegate next,
        ILogger<QueryTimeoutMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }
    
    public async Task InvokeAsync(HttpContext context)
    {
        // API 요청별 타임아웃 설정
        using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(30));
        context.RequestAborted = cts.Token;
        
        var stopwatch = Stopwatch.StartNew();
        
        try
        {
            await _next(context);
            stopwatch.Stop();
            
            if (stopwatch.Elapsed.TotalSeconds > 5)
            {
                _logger.LogWarning(
                    $"Slow request: {context.Request.Path} took {stopwatch.Elapsed.TotalSeconds:N2}s");
            }
        }
        catch (OperationCanceledException)
        {
            _logger.LogError($"Request timeout: {context.Request.Path}");
            context.Response.StatusCode = 504; // Gateway Timeout
            await context.Response.WriteAsync("Request timeout");
        }
    }
}
```

## 17.2 사례 2: 디스크 공간 부족 긴급 대응

**상황 설명**

금요일 저녁 6시, 모니터링 시스템에서 치명적인 알림이 울렸다. "디스크 사용률 95%! 여유 공간 5GB 미만!" TimescaleDB가 설치된 서버의 디스크가 거의 가득 찬 것이다. 디스크가 완전히 가득 차면 데이터베이스가 멈추고 게임 서비스 전체가 중단될 수 있는 매우 위험한 상황이다.

**긴급 진단**

먼저 어떤 테이블이 가장 많은 공간을 차지하는지 확인한다.

```sql
-- 테이블별 디스크 사용량 조회
SELECT 
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS total_size,
    pg_size_pretty(pg_relation_size(schemaname||'.'||tablename)) AS table_size,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename) - 
                   pg_relation_size(schemaname||'.'||tablename)) AS indexes_size
FROM pg_tables
WHERE schemaname NOT IN ('pg_catalog', 'information_schema')
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC
LIMIT 20;
```

결과를 보니 `player_events` 테이블이 무려 450GB를 차지하고 있었다.

```
         tablename         | total_size | table_size | indexes_size 
---------------------------+------------+------------+--------------
 player_events             | 450 GB     | 320 GB     | 130 GB
 player_sessions           | 85 GB      | 60 GB      | 25 GB
 server_metrics            | 42 GB      | 30 GB      | 12 GB
```

**하이퍼테이블 청크 분석**

TimescaleDB의 하이퍼테이블은 내부적으로 청크(chunk)로 나뉘어 저장된다. 각 청크의 크기와 나이를 확인한다.

```sql
-- 하이퍼테이블 청크 정보
SELECT 
    chunk_schema,
    chunk_name,
    range_start,
    range_end,
    pg_size_pretty(pg_total_relation_size(format('%I.%I', chunk_schema, chunk_name))) as chunk_size,
    NOW() - range_end as chunk_age
FROM timescaledb_information.chunks
WHERE hypertable_name = 'player_events'
ORDER BY range_start DESC
LIMIT 20;
```

6개월 이전의 오래된 청크들이 여전히 저장되어 있었다. 문제는 데이터 보관 정책(Retention Policy)이 설정되지 않았다는 것이다.

**즉시 조치: 수동 청크 삭제**

가장 오래된 청크부터 삭제하여 즉시 공간을 확보한다.

```sql
-- 90일 이전 데이터 청크 삭제
SELECT drop_chunks('player_events', INTERVAL '90 days');

-- 삭제 결과 확인
SELECT 
    chunk_schema || '.' || chunk_name as dropped_chunk
FROM timescaledb_information.chunks
WHERE hypertable_name = 'player_events'
    AND range_end < NOW() - INTERVAL '90 days';

-- 실제로 삭제
DO $$
DECLARE
    chunk_name TEXT;
BEGIN
    FOR chunk_name IN 
        SELECT format('%I.%I', chunk_schema, chunk_name)
        FROM timescaledb_information.chunks
        WHERE hypertable_name = 'player_events'
            AND range_end < NOW() - INTERVAL '90 days'
    LOOP
        EXECUTE 'DROP TABLE IF EXISTS ' || chunk_name || ' CASCADE';
        RAISE NOTICE 'Dropped chunk: %', chunk_name;
    END LOOP;
END $$;
```

이 작업으로 약 280GB의 공간을 확보했다.

**근본 해결: 자동 보관 정책 설정**

앞으로 같은 문제가 발생하지 않도록 자동 보관 정책을 설정한다.

```sql
-- 테이블별 보관 정책 설정
SELECT add_retention_policy('player_events', INTERVAL '90 days');
SELECT add_retention_policy('player_sessions', INTERVAL '180 days');
SELECT add_retention_policy('server_metrics', INTERVAL '365 days');
SELECT add_retention_policy('error_logs', INTERVAL '30 days');

-- 설정된 정책 확인
SELECT 
    hypertable_name,
    drop_after
FROM timescaledb_information.jobs
WHERE proc_name = 'policy_retention'
ORDER BY hypertable_name;
```

**압축 활성화**

보관 정책과 함께 압축을 활성화하여 디스크 사용량을 더욱 줄인다.

```sql
-- 압축 설정 (7일 이후 데이터)
ALTER TABLE player_events SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'player_id, event_type',
    timescaledb.compress_orderby = 'event_time DESC'
);

SELECT add_compression_policy('player_events', INTERVAL '7 days');

-- 기존 데이터 수동 압축
SELECT compress_chunk(chunk_schema || '.' || chunk_name)
FROM timescaledb_information.chunks
WHERE hypertable_name = 'player_events'
    AND is_compressed = false
    AND range_end < NOW() - INTERVAL '7 days';

-- 압축 효과 확인
SELECT 
    pg_size_pretty(before_compression_total_bytes) as before_size,
    pg_size_pretty(after_compression_total_bytes) as after_size,
    ROUND((1 - after_compression_total_bytes::numeric / 
           before_compression_total_bytes) * 100, 2) as compression_ratio
FROM timescaledb_information.compressed_chunk_stats
WHERE hypertable_name = 'player_events';
```

압축을 적용한 결과 데이터 크기가 평균 75% 감소했다.

**C# 디스크 모니터링 시스템**

```csharp
// GameAnalytics.Infrastructure/Monitoring/DiskSpaceMonitor.cs
using Npgsql;
using Dapper;

namespace GameAnalytics.Infrastructure.Monitoring
{
    public class DiskSpaceMonitor
    {
        private readonly string _connectionString;
        private readonly ILogger<DiskSpaceMonitor> _logger;
        
        public DiskSpaceMonitor(
            string connectionString,
            ILogger<DiskSpaceMonitor> logger)
        {
            _connectionString = connectionString;
            _logger = logger;
        }
        
        // 전체 디스크 사용량 조회
        public async Task<DatabaseDiskUsage> GetDatabaseDiskUsageAsync()
        {
            using var connection = new NpgsqlConnection(_connectionString);
            
            var sql = @"
                SELECT 
                    pg_database_size(current_database()) as total_size,
                    (SELECT SUM(pg_total_relation_size(schemaname||'.'||tablename))
                     FROM pg_tables 
                     WHERE schemaname NOT IN ('pg_catalog', 'information_schema')) as table_size,
                    (SELECT SUM(pg_indexes_size(schemaname||'.'||tablename))
                     FROM pg_tables
                     WHERE schemaname NOT IN ('pg_catalog', 'information_schema')) as index_size";
            
            var result = await connection.QueryFirstAsync<DatabaseDiskUsage>(sql);
            return result;
        }
        
        // 테이블별 사용량
        public async Task<List<TableDiskUsage>> GetTableDiskUsageAsync()
        {
            using var connection = new NpgsqlConnection(_connectionString);
            
            var sql = @"
                SELECT 
                    schemaname,
                    tablename,
                    pg_total_relation_size(schemaname||'.'||tablename) as total_bytes,
                    pg_relation_size(schemaname||'.'||tablename) as table_bytes,
                    pg_total_relation_size(schemaname||'.'||tablename) - 
                        pg_relation_size(schemaname||'.'||tablename) as index_bytes,
                    (SELECT COUNT(*) 
                     FROM timescaledb_information.chunks 
                     WHERE hypertable_name = tablename) as chunk_count
                FROM pg_tables
                WHERE schemaname NOT IN ('pg_catalog', 'information_schema', '_timescaledb_internal')
                ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC";
            
            var results = await connection.QueryAsync<TableDiskUsage>(sql);
            return results.ToList();
        }
        
        // 청크 분석
        public async Task<List<ChunkInfo>> GetChunkInfoAsync(string hypertableName)
        {
            using var connection = new NpgsqlConnection(_connectionString);
            
            var sql = @"
                SELECT 
                    chunk_name,
                    range_start::date as range_start,
                    range_end::date as range_end,
                    is_compressed,
                    pg_total_relation_size(format('%I.%I', chunk_schema, chunk_name)) as size_bytes,
                    EXTRACT(EPOCH FROM (NOW() - range_end)) / 86400 as age_days
                FROM timescaledb_information.chunks
                WHERE hypertable_name = @HypertableName
                ORDER BY range_start DESC";
            
            var results = await connection.QueryAsync<ChunkInfo>(
                sql,
                new { HypertableName = hypertableName }
            );
            
            return results.ToList();
        }

        public async Task<DiskSpaceAlert> CheckDiskSpaceAlertAsync(
            long warningThresholdBytes = 50L * 1024 * 1024 * 1024,  // 50GB
            long criticalThresholdBytes = 20L * 1024 * 1024 * 1024) // 20GB
        {
            using var connection = new NpgsqlConnection(_connectionString);
            
            // PostgreSQL 데이터 디렉토리의 여유 공간 확인
            var sql = @"
                SELECT 
                    pg_database_size(current_database()) as used_bytes,
                    (SELECT setting::bigint * 8192 FROM pg_settings WHERE name = 'block_size') as block_size";
            
            var usage = await GetDatabaseDiskUsageAsync();
            var tables = await GetTableDiskUsageAsync();
            
            // 상위 5개 테이블
            var topTables = tables.Take(5).ToList();
            
            var alert = new DiskSpaceAlert
            {
                TotalUsedBytes = usage.TotalSize,
                TotalUsedGB = usage.TotalSize / 1024.0 / 1024.0 / 1024.0,
                LargestTables = topTables,
                Timestamp = DateTime.UtcNow
            };
            
            // 경고 수준 결정 (간단한 추정)
            if (usage.TotalSize > criticalThresholdBytes)
            {
                alert.Level = AlertLevel.Critical;
                alert.Message = $"Critical: Database size {alert.TotalUsedGB:N2}GB. Immediate action required!";
                _logger.LogCritical(alert.Message);
            }
            else if (usage.TotalSize > warningThresholdBytes)
            {
                alert.Level = AlertLevel.Warning;
                alert.Message = $"Warning: Database size {alert.TotalUsedGB:N2}GB. Consider cleanup.";
                _logger.LogWarning(alert.Message);
            }
            else
            {
                alert.Level = AlertLevel.Normal;
                alert.Message = $"Normal: Database size {alert.TotalUsedGB:N2}GB.";
            }
            
            return alert;
        }
        
        // 자동 정리 제안
        public async Task<List<CleanupSuggestion>> GetCleanupSuggestionsAsync()
        {
            var suggestions = new List<CleanupSuggestion>();
            using var connection = new NpgsqlConnection(_connectionString);
            
            // 보관 정책이 없는 하이퍼테이블
            var noRetentionSql = @"
                SELECT DISTINCT h.table_name
                FROM _timescaledb_catalog.hypertable h
                WHERE NOT EXISTS (
                    SELECT 1 
                    FROM _timescaledb_config.bgw_policy_drop_chunks p
                    WHERE p.hypertable_id = h.id
                )";
            
            var tablesWithoutRetention = await connection.QueryAsync<string>(noRetentionSql);
            
            foreach (var table in tablesWithoutRetention)
            {
                suggestions.Add(new CleanupSuggestion
                {
                    Type = "Missing Retention Policy",
                    TableName = table,
                    Description = $"Table '{table}' has no retention policy. Old data will accumulate.",
                    RecommendedAction = $"SELECT add_retention_policy('{table}', INTERVAL '90 days');"
                });
            }
            
            // 압축되지 않은 오래된 청크
            var uncompressedSql = @"
                SELECT 
                    hypertable_name,
                    COUNT(*) as uncompressed_chunks,
                    SUM(pg_total_relation_size(format('%I.%I', chunk_schema, chunk_name))) as total_bytes
                FROM timescaledb_information.chunks
                WHERE is_compressed = false
                    AND range_end < NOW() - INTERVAL '7 days'
                GROUP BY hypertable_name
                HAVING COUNT(*) > 0";
            
            var uncompressedChunks = await connection.QueryAsync<dynamic>(uncompressedSql);
            
            foreach (var chunk in uncompressedChunks)
            {
                suggestions.Add(new CleanupSuggestion
                {
                    Type = "Uncompressed Old Data",
                    TableName = chunk.hypertable_name,
                    Description = $"{chunk.uncompressed_chunks} old chunks are not compressed, " +
                                $"using {(long)chunk.total_bytes / 1024 / 1024 / 1024}GB",
                    RecommendedAction = $"Enable compression policy for '{chunk.hypertable_name}'"
                });
            }
            
            return suggestions;
        }
```

## 17.3 사례 3: 복잡한 JOIN 쿼리 최적화

**상황 설명**

게임 운영팀에서 "VIP 플레이어의 상세한 활동 리포트"를 요청했다. 이 리포트는 플레이어의 기본 정보, 결제 내역, 게임 플레이 통계, 소셜 활동 등을 모두 결합하여 보여줘야 했다. 초기 버전의 쿼리는 다음과 같았다.

```sql
-- 최초 작성된 복잡한 쿼리 (문제 버전)
SELECT 
    p.player_id,
    p.first_login_date,
    p.signup_country,
    p.current_level,
    p.lifetime_revenue,
    COUNT(DISTINCT ps.session_id) as total_sessions,
    SUM(ps.session_duration_seconds) as total_playtime,
    COUNT(DISTINCT pt.transaction_id) as purchase_count,
    SUM(pt.price) as total_spent,
    COUNT(DISTINCT pe_battle.event_id) as battle_count,
    COUNT(DISTINCT pe_social.event_id) as social_interaction_count,
    AVG(pds.total_playtime_seconds) as avg_daily_playtime
FROM players p
LEFT JOIN player_sessions ps ON p.player_id = ps.player_id
LEFT JOIN purchase_transactions pt ON p.player_id = pt.player_id
LEFT JOIN player_events pe_battle ON p.player_id = pe_battle.player_id 
    AND pe_battle.event_type = 'battle_complete'
LEFT JOIN player_events pe_social ON p.player_id = pe_social.player_id 
    AND pe_social.event_type IN ('friend_add', 'guild_join', 'chat_message')
LEFT JOIN player_daily_stats pds ON p.player_id = pds.player_id
WHERE p.lifetime_revenue >= 100
    AND ps.session_start >= NOW() - INTERVAL '30 days'
    AND pt.purchase_time >= NOW() - INTERVAL '30 days'
GROUP BY p.player_id, p.first_login_date, p.signup_country, 
         p.current_level, p.lifetime_revenue
ORDER BY p.lifetime_revenue DESC;
```

여러 개의 대용량 테이블을 한 번에 JOIN하면서 엄청난 양의 행을 스캔하고 있다. 특히 `player_events` 테이블을 event_type별로 두 번 JOIN하는 것이 매우 비효율적이다.

**해결책: 쿼리 분해 및 CTE 활용**

복잡한 JOIN을 여러 단계로 나누고 CTE(Common Table Expression)를 사용하여 중간 결과를 캐시한다.

```sql
-- 최적화된 버전
WITH vip_players AS (
    -- VIP 플레이어 목록 (작은 결과셋)
    SELECT player_id, first_login_date, signup_country, 
           current_level, lifetime_revenue
    FROM players
    WHERE lifetime_revenue >= 100
),
recent_sessions AS (
    -- 최근 30일 세션 요약
    SELECT 
        player_id,
        COUNT(DISTINCT session_id) as session_count,
        SUM(session_duration_seconds) as total_duration
    FROM player_sessions
    WHERE session_start >= NOW() - INTERVAL '30 days'
        AND player_id IN (SELECT player_id FROM vip_players)
    GROUP BY player_id
),
recent_purchases AS (
    -- 최근 30일 구매 요약
    SELECT 
        player_id,
        COUNT(*) as purchase_count,
        SUM(price) as total_spent
    FROM purchase_transactions
    WHERE purchase_time >= NOW() - INTERVAL '30 days'
        AND player_id IN (SELECT player_id FROM vip_players)
    GROUP BY player_id
),
event_summary AS (
    -- 이벤트 요약 (한 번의 스캔으로 모든 이벤트 타입 집계)
    SELECT 
        player_id,
        COUNT(*) FILTER (WHERE event_type = 'battle_complete') as battle_count,
        COUNT(*) FILTER (WHERE event_type IN ('friend_add', 'guild_join', 'chat_message')) as social_count
    FROM player_events
    WHERE event_time >= NOW() - INTERVAL '30 days'
        AND player_id IN (SELECT player_id FROM vip_players)
    GROUP BY player_id
),
daily_stats AS (
    -- 일일 통계 평균
    SELECT 
        player_id,
        AVG(total_playtime_seconds) as avg_daily_playtime
    FROM player_daily_stats
    WHERE stat_date >= CURRENT_DATE - 30
        AND player_id IN (SELECT player_id FROM vip_players)
    GROUP BY player_id
)
SELECT 
    vp.player_id,
    vp.first_login_date,
    vp.signup_country,
    vp.current_level,
    vp.lifetime_revenue,
    COALESCE(rs.session_count, 0) as total_sessions,
    COALESCE(rs.total_duration, 0) as total_playtime,
    COALESCE(rp.purchase_count, 0) as purchase_count,
    COALESCE(rp.total_spent, 0) as total_spent,
    COALESCE(es.battle_count, 0) as battle_count,
    COALESCE(es.social_count, 0) as social_interaction_count,
    COALESCE(ds.avg_daily_playtime, 0) as avg_daily_playtime
FROM vip_players vp
LEFT JOIN recent_sessions rs ON vp.player_id = rs.player_id
LEFT JOIN recent_purchases rp ON vp.player_id = rp.player_id
LEFT JOIN event_summary es ON vp.player_id = es.player_id
LEFT JOIN daily_stats ds ON vp.player_id = ds.player_id
ORDER BY vp.lifetime_revenue DESC;
```

이 최적화된 쿼리는 다음과 같은 개선사항이 있다.

**필터 우선 적용:** VIP 플레이어를 먼저 필터링하여 작은 결과셋을 만든다.

**중복 스캔 제거:** `player_events` 테이블을 한 번만 스캔하고 FILTER 절로 다양한 이벤트 타입을 집계한다.

**부분 인덱스 활용:** 서브쿼리에서 IN 절을 사용하면 옵티마이저가 세미 조인으로 변환하여 인덱스를 효율적으로 활용한다.

**성능 결과**

## 17.4 사례 4: 배치 작업 성능 개선

**상황 설명**

매일 새벽 3시에 실행되는 일일 통계 배치 작업이 있다. 이 작업은 전날의 모든 플레이어 활동을 집계하여 `player_daily_stats` 테이블에 저장한다. 초기에는 문제없었지만 플레이어 수가 증가하면서 배치 작업이 6시간 이상 걸리게 되었고, 때로는 다음 날 아침까지 완료되지 않아 실시간 대시보드에 영향을 주었다.

**원래의 배치 로직**


이 방식의 문제점이다.

**N+1 문제:** 100만 명의 플레이어가 있으면 100만 번의 쿼리를 실행한다.

**단일 스레드:** 순차적으로 처리하여 CPU 코어를 제대로 활용하지 못한다.

**트랜잭션 오버헤드:** 각 플레이어마다 개별 INSERT를 수행한다.

**해결책 1: SQL 기반 집계**

가장 효율적인 방법은 데이터베이스에서 한 번에 집계하는 것이다.

```sql
-- 일일 통계 생성 프로시저 (최적화 버전)
CREATE OR REPLACE PROCEDURE generate_daily_stats(target_date DATE)
LANGUAGE plpgsql AS $$
BEGIN
    -- 기존 데이터 삭제
    DELETE FROM player_daily_stats WHERE stat_date = target_date;
    
    -- 한 번의 쿼리로 모든 플레이어의 통계 생성
    INSERT INTO player_daily_stats (
        stat_date,
        player_id,
        first_seen_date,
        days_since_install,
        login_count,
        session_count,
        total_playtime_seconds,
        total_purchases,
        purchase_count,
        level_start,
        level_end,
        levels_gained,
        is_active,
        is_paying
    )
    SELECT 
        target_date as stat_date,
        p.player_id,
        DATE(p.first_login_date) as first_seen_date,
        target_date - DATE(p.first_login_date) as days_since_install,
        
        -- 세션 정보
        COUNT(DISTINCT ps.session_id) FILTER (WHERE DATE(ps.session_start) = target_date) as login_count,
        COUNT(DISTINCT ps.session_id) FILTER (WHERE DATE(ps.session_start) = target_date) as session_count,
        COALESCE(SUM(ps.session_duration_seconds) FILTER (WHERE DATE(ps.session_start) = target_date), 0) as total_playtime_seconds,
        
        -- 구매 정보
        COALESCE(SUM(pt.price) FILTER (WHERE DATE(pt.purchase_time) = target_date), 0) as total_purchases,
        COUNT(pt.transaction_id) FILTER (WHERE DATE(pt.purchase_time) = target_date) as purchase_count,
        
        -- 레벨 정보 (간소화, 실제로는 더 복잡할 수 있음)
        MIN(pe.event_value) FILTER (WHERE DATE(pe.event_time) = target_date AND pe.event_type = 'level_up') as level_start,
        MAX(pe.event_value) FILTER (WHERE DATE(pe.event_time) = target_date AND pe.event_type = 'level_up') as level_end,
        COUNT(*) FILTER (WHERE DATE(pe.event_time) = target_date AND pe.event_type = 'level_up') as levels_gained,
        
        -- 플래그
        CASE WHEN COUNT(ps.session_id) FILTER (WHERE DATE(ps.session_start) = target_date) > 0 THEN true ELSE false END as is_active,
        CASE WHEN SUM(pt.price) FILTER (WHERE DATE(pt.purchase_time) = target_date) > 0 THEN true ELSE false END as is_paying
        
    FROM players p
    LEFT JOIN player_sessions ps ON p.player_id = ps.player_id 
        AND ps.session_start >= target_date 
        AND ps.session_start < target_date + INTERVAL '1 day'
    LEFT JOIN purchase_transactions pt ON p.player_id = pt.player_id 
        AND pt.purchase_time >= target_date 
        AND pt.purchase_time < target_date + INTERVAL '1 day'
    LEFT JOIN player_events pe ON p.player_id = pe.player_id 
        AND pe.event_time >= target_date 
        AND pe.event_time < target_date + INTERVAL '1 day'
    WHERE p.first_login_date <= target_date + INTERVAL '1 day'
    GROUP BY p.player_id, p.first_login_date;
    
    -- 통계 업데이트
    ANALYZE player_daily_stats;
    
    RAISE NOTICE 'Daily stats generated for %', target_date;
END;
$$;

-- 실행
CALL generate_daily_stats('2025-01-15');
```

**해결책 2: 병렬 처리 (대규모 데이터의 경우)**

데이터가 너무 커서 단일 쿼리로도 느린 경우 병렬 처리를 활용한다.

```csharp
// GameAnalytics.Infrastructure/BatchJobs/DailyStatsBatchJob.cs
using System.Collections.Concurrent;
using Npgsql;
using Dapper;

namespace GameAnalytics.Infrastructure.BatchJobs
{
    public class DailyStatsBatchJob
    {
        private readonly string _connectionString;
        private readonly ILogger<DailyStatsBatchJob> _logger;
        private readonly int _maxDegreeOfParallelism;
        
        public DailyStatsBatchJob(
            string connectionString,
            ILogger<DailyStatsBatchJob> logger,
            int maxDegreeOfParallelism = 4)
        {
            _connectionString = connectionString;
            _logger = logger;
            _maxDegreeOfParallelism = maxDegreeOfParallelism;
        }
        
        // SQL 프로시저 호출 (가장 효율적)
        public async Task GenerateDailyStatsViaProcedureAsync(DateTime targetDate)
        {
            _logger.LogInformation($"Generating daily stats for {targetDate:yyyy-MM-dd} via procedure");
            
            var stopwatch = Stopwatch.StartNew();
            
            using var connection = new NpgsqlConnection(_connectionString);
            await connection.OpenAsync();
            
            await connection.ExecuteAsync(
                "CALL generate_daily_stats(@TargetDate)",
                new { TargetDate = targetDate.Date },
                commandTimeout: 3600  // 1시간 타임아웃
            );
            
            stopwatch.Stop();
            _logger.LogInformation(
                $"Daily stats generation completed in {stopwatch.Elapsed.TotalMinutes:N2} minutes");
        }
        
        // 병렬 처리 버전 (플레이어를 청크로 나누어 처리)
        public async Task GenerateDailyStatsParallelAsync(
            DateTime targetDate,
            int chunkSize = 10000)
        {
            _logger.LogInformation($"Generating daily stats for {targetDate:yyyy-MM-dd} in parallel");
            
            var stopwatch = Stopwatch.StartNew();
            
            // 전체 플레이어 수 확인
            var totalPlayers = await GetActivePlayerCountAsync(targetDate);
            _logger.LogInformation($"Processing {totalPlayers:N0} players");
            
            // 플레이어를 청크로 나눔
            var chunks = (int)Math.Ceiling(totalPlayers / (double)chunkSize);
            
            // 병렬 처리
            var options = new ParallelOptions 
            { 
                MaxDegreeOfParallelism = _maxDegreeOfParallelism 
            };
            
            var processedCount = 0;
            var lockObj = new object();
            
            await Parallel.ForAsync(0, chunks, options, async (chunkIndex, cancellationToken) =>
            {
                try
                {
                    var offset = chunkIndex * chunkSize;
                    await ProcessChunkAsync(targetDate, offset, chunkSize);
                    
                    lock (lockObj)
                    {
                        processedCount += chunkSize;
                        if (processedCount % 50000 == 0)
                        {
                            _logger.LogInformation(
                                $"Progress: {processedCount:N0} / {totalPlayers:N0} players " +
                                $"({(double)processedCount / totalPlayers:P1})");
                        }
                    }
                }
                catch (Exception ex)
                {
                    _logger.LogError(ex, $"Error processing chunk {chunkIndex}");
                }
            });
            
            stopwatch.Stop();
            _logger.LogInformation(
                $"Daily stats generation completed in {stopwatch.Elapsed.TotalMinutes:N2} minutes");
        }
        
        private async Task<long> GetActivePlayerCountAsync(DateTime targetDate)
        {
            using var connection = new NpgsqlConnection(_connectionString);
            
            var sql = @"
                SELECT COUNT(DISTINCT player_id)
                FROM player_sessions
                WHERE session_start >= @Date AND session_start < @Date + INTERVAL '1 day'";
            
            return await connection.ExecuteScalarAsync<long>(
                sql,
                new { Date = targetDate }
            );
        }
        
        private async Task ProcessChunkAsync(DateTime targetDate, int offset, int limit)
        {
            using var connection = new NpgsqlConnection(_connectionString);
            await connection.OpenAsync();
            
            var sql = @"
                INSERT INTO player_daily_stats (
                    stat_date, player_id, session_count, total_playtime_seconds,
                    total_purchases, purchase_count, is_active, is_paying
                )
                SELECT 
                    @TargetDate,
                    player_id,
                    COUNT(DISTINCT session_id),
                    SUM(session_duration_seconds),
                    0, 0, true, false
                FROM player_sessions
                WHERE session_start >= @TargetDate 
                    AND session_start < @TargetDate + INTERVAL '1 day'
                    AND player_id IN (
                        SELECT player_id FROM players 
                        ORDER BY player_id 
                        LIMIT @Limit OFFSET @Offset
                    )
                GROUP BY player_id
                ON CONFLICT (stat_date, player_id) 
                DO UPDATE SET
                    session_count = EXCLUDED.session_count,
                    total_playtime_seconds = EXCLUDED.total_playtime_seconds";
            
            await connection.ExecuteAsync(
                sql,
                new { TargetDate = targetDate, Offset = offset, Limit = limit },
                commandTimeout: 300
            );
        }
        
        // 증분 업데이트 (실시간 처리용)
        public async Task UpdateDailyStatsIncrementalAsync(
            DateTime targetDate,
            List<long> playerIds)
        {
            if (!playerIds.Any()) return;
            
            using var connection = new NpgsqlConnection(_connectionString);
            
            // COPY를 사용한 대량 업데이트가 더 효율적
            var sql = @"
                INSERT INTO player_daily_stats (
                    stat_date, player_id, session_count, total_playtime_seconds,
                    is_active
                )
                SELECT 
                    @TargetDate,
                    player_id,
                    COUNT(*),
                    SUM(session_duration_seconds),
                    true
                FROM player_sessions
                WHERE session_start >= @TargetDate 
                    AND session_start < @TargetDate + INTERVAL '1 day'
                    AND player_id = ANY(@PlayerIds)
                GROUP BY player_id
                ON CONFLICT (stat_date, player_id) 
                DO UPDATE SET
                    session_count = player_daily_stats.session_count + EXCLUDED.session_count,
                    total_playtime_seconds = player_daily_stats.total_playtime_seconds + EXCLUDED.total_playtime_seconds,
                    is_active = true";
            
            await connection.ExecuteAsync(
                sql,
                new { TargetDate = targetDate, PlayerIds = playerIds.ToArray() }
            );
        }
    }
}
```

## 17.5 성능 튜닝 체크리스트

실전에서 성능 문제에 직면했을 때 체계적으로 접근하기 위한 체크리스트다.

**1. 쿼리 분석**

```markdown
□ EXPLAIN ANALYZE로 실행 계획 확인
□ pg_stat_statements로 느린 쿼리 식별
□ 테이블 스캔(Seq Scan)이 발생하는지 확인
□ 정렬(Sort)이 디스크에 spill되는지 확인
□ JOIN 순서와 방식이 적절한지 확인
□ 서브쿼리를 CTE나 JOIN으로 변환 고려
```

**2. 인덱스 최적화**

```markdown
□ 필요한 인덱스가 모두 존재하는지 확인
□ 인덱스가 실제로 사용되는지 확인
□ 사용되지 않는 인덱스 제거
□ 복합 인덱스의 컬럼 순서 최적화
□ 부분 인덱스(Partial Index) 활용 고려
□ 인덱스 통계 업데이트 (ANALYZE)
```

**3. 하이퍼테이블 관리**

```markdown
□ 청크 크기가 적절한지 확인 (1주일~1개월)
□ 압축 정책이 설정되어 있는지 확인
□ 보관 정책으로 오래된 데이터 자동 삭제
□ 연속 집계로 자주 사용되는 쿼리 미리 계산
□ 청크별 통계 정보 최신 상태 유지
```

**4. 데이터베이스 설정**

```markdown
□ shared_buffers 적절히 설정 (시스템 RAM의 25%)
□ work_mem 충분히 할당 (복잡한 쿼리용)
□ effective_cache_size 설정 (시스템 RAM의 50-75%)
□ maintenance_work_mem 증가 (인덱스 생성/VACUUM용)
□ max_parallel_workers 설정 (병렬 쿼리용)
□ random_page_cost 조정 (SSD 사용 시 낮춤)
```

**5. 애플리케이션 레벨**

```markdown
□ 연결 풀링 활성화 및 크기 최적화
□ N+1 쿼리 문제 해결
□ 불필요한 데이터 조회 제거 (SELECT *)
□ 페이징 구현 (LIMIT/OFFSET)
□ 캐싱 전략 적용 (Redis 등)
□ 비동기 처리 활용
□ 배치 작업 병렬화
```

```csharp
// GameAnalytics.Infrastructure/Diagnostics/PerformanceChecker.cs
public class PerformanceChecker
{
    private readonly string _connectionString;
    private readonly ILogger<PerformanceChecker> _logger;
    
    public async Task<PerformanceReport> RunFullCheckAsync()
    {
        var report = new PerformanceReport();
        
        _logger.LogInformation("Starting performance check...");
        
        // 1. 느린 쿼리 체크
        report.SlowQueries = await CheckSlowQueriesAsync();
        
        // 2. 인덱스 상태 체크
        report.IndexIssues = await CheckIndexIssuesAsync();
        
        // 3. 테이블 통계 체크
        report.StaleStatistics = await CheckStaleStatisticsAsync();
        
        // 4. 디스크 사용량 체크
        report.DiskUsage = await CheckDiskUsageAsync();
        
        // 5. 연결 풀 상태 체크
        report.ConnectionPoolStats = await CheckConnectionPoolAsync();
        
        // 6. 청크 상태 체크
        report.ChunkIssues = await CheckChunkIssuesAsync();
        
        _logger.LogInformation("Performance check completed");
        
        return report;
    }
    
    private async Task<List<string>> CheckSlowQueriesAsync()
    {
        var issues = new List<string>();
        using var connection = new NpgsqlConnection(_connectionString);
        
        var sql = @"
            SELECT COUNT(*)
            FROM pg_stat_statements
            WHERE mean_exec_time > 1000";
        
        var count = await connection.ExecuteScalarAsync<int>(sql);
        
        if (count > 0)
        {
            issues.Add($"Found {count} queries with average execution time > 1s");
        }
        
        return issues;
    }
    
    private async Task<List<string>> CheckIndexIssuesAsync()
    {
        var issues = new List<string>();
        using var connection = new NpgsqlConnection(_connectionString);
        
        // 사용되지 않는 인덱스
        var sql = @"
            SELECT schemaname || '.' || tablename || '.' || indexname
            FROM pg_stat_user_indexes
            WHERE idx_scan = 0
                AND schemaname NOT IN ('pg_catalog', 'information_schema')
            LIMIT 10";
        
        var unusedIndexes = await connection.QueryAsync<string>(sql);
        
        if (unusedIndexes.Any())
        {
            issues.Add($"Found {unusedIndexes.Count()} unused indexes: " +
                      string.Join(", ", unusedIndexes.Take(5)));
        }
        
        // 테이블 스캔이 많은 테이블
        var seqScanSql = @"
            SELECT schemaname || '.' || tablename
            FROM pg_stat_user_tables
            WHERE seq_scan > 1000
                AND seq_tup_read / NULLIF(seq_scan, 0) > 10000
                AND schemaname NOT IN ('pg_catalog', 'information_schema')
            ORDER BY seq_scan DESC
            LIMIT 5";
        
        var tablesNeedingIndexes = await connection.QueryAsync<string>(seqScanSql);
        
        if (tablesNeedingIndexes.Any())
        {
            issues.Add($"Tables with frequent sequential scans: " +
                      string.Join(", ", tablesNeedingIndexes));
        }
        
        return issues;
    }
    
    private async Task<List<string>> CheckStaleStatisticsAsync()
    {
        var issues = new List<string>();
        using var connection = new NpgsqlConnection(_connectionString);
        
        var sql = @"
            SELECT schemaname || '.' || tablename
            FROM pg_stat_user_tables
            WHERE last_analyze IS NULL 
                OR last_analyze < NOW() - INTERVAL '7 days'
            LIMIT 10";
        
        var staleTables = await connection.QueryAsync<string>(sql);
        
        if (staleTables.Any())
        {
            issues.Add($"Tables with stale statistics: " +
                      string.Join(", ", staleTables));
        }
        
        return issues;
    }
    
    private async Task<List<string>> CheckDiskUsageAsync()
    {
        var issues = new List<string>();
        using var connection = new NpgsqlConnection(_connectionString);
        
        var sql = @"
            SELECT pg_database_size(current_database()) as size_bytes";
        
        var sizeBytes = await connection.ExecuteScalarAsync<long>(sql);
        var sizeGB = sizeBytes / 1024.0 / 1024.0 / 1024.0;
        
        if (sizeGB > 500)
        {
            issues.Add($"Database size is {sizeGB:N2}GB. Consider archiving old data.");
        }
        
        return issues;
    }
    
    private async Task<List<string>> CheckConnectionPoolAsync()
    {
        var issues = new List<string>();
        using var connection = new NpgsqlConnection(_connectionString);
        
        var sql = @"
            SELECT COUNT(*) 
            FROM pg_stat_activity 
            WHERE state = 'active'";
        
        var activeConnections = await connection.ExecuteScalarAsync<int>(sql);
        
        if (activeConnections > 80)
        {
            issues.Add($"High number of active connections: {activeConnections}");
        }
        
        return issues;
    }
    
    private async Task<List<string>> CheckChunkIssuesAsync()
    {
        var issues = new List<string>();
        using var connection = new NpgsqlConnection(_connectionString);
        
        // 압축되지 않은 오래된 청크
        var sql = @"
            SELECT COUNT(*)
            FROM timescaledb_information.chunks
            WHERE is_compressed = false
                AND range_end < NOW() - INTERVAL '30 days'";
        
        var uncompressedChunks = await connection.ExecuteScalarAsync<int>(sql);
        
        if (uncompressedChunks > 0)
        {
            issues.Add($"Found {uncompressedChunks} old uncompressed chunks");
        }
        
        return issues;
    }
}

public class PerformanceReport
{
    public List<string> SlowQueries { get; set; } = new();
    public List<string> IndexIssues { get; set; } = new();
    public List<string> StaleStatistics { get; set; } = new();
    public List<string> DiskUsage { get; set; } = new();
    public List<string> ConnectionPoolStats { get; set; } = new();
    public List<string> ChunkIssues { get; set; } = new();
    
    public bool HasIssues => 
        SlowQueries.Any() || 
        IndexIssues.Any() || 
        StaleStatistics.Any() || 
        DiskUsage.Any() || 
        ConnectionPoolStats.Any() ||
        ChunkIssues.Any();
    
    public string GetSummary()
    {
        var sb = new StringBuilder();
        sb.AppendLine("=== Performance Report ===");
        
        if (!HasIssues)
        {
            sb.AppendLine("✓ No issues found");
            return sb.ToString();
        }
        
        if (SlowQueries.Any())
        {
            sb.AppendLine("\n[Slow Queries]");
            SlowQueries.ForEach(i => sb.AppendLine($"  - {i}"));
        }
        
        if (IndexIssues.Any())
        {
            sb.AppendLine("\n[Index Issues]");
            IndexIssues.ForEach(i => sb.AppendLine($"  - {i}"));
        }
        
        if (StaleStatistics.Any())
        {
            sb.AppendLine("\n[Stale Statistics]");
            StaleStatistics.ForEach(i => sb.AppendLine($"  - {i}"));
        }
        
        if (DiskUsage.Any())
        {
            sb.AppendLine("\n[Disk Usage]");
            DiskUsage.ForEach(i => sb.AppendLine($"  - {i}"));
        }
        
        if (ConnectionPoolStats.Any())
        {
            sb.AppendLine("\n[Connection Pool]");
            ConnectionPoolStats.ForEach(i => sb.AppendLine($"  - {i}"));
        }
        
        if (ChunkIssues.Any())
        {
            sb.AppendLine("\n[Chunk Issues]");
            ChunkIssues.ForEach(i => sb.AppendLine($"  - {i}"));
        }
        
        return sb.ToString();
    }
}
```

## 17.6 운영 팁과 베스트 프랙티스

실전 운영에서 얻은 귀중한 팁들이다.

**1. 모니터링은 필수다**

성능 문제는 발생하기 전에 예방하는 것이 최선이다. 지속적인 모니터링을 통해 문제의 조짐을 조기에 발견한다.

```csharp
// 매일 실행되는 성능 체크
public class DailyPerformanceCheckService : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                var checker = new PerformanceChecker(_connectionString, _logger);
                var report = await checker.RunFullCheckAsync();
                
                if (report.HasIssues)
                {
                    _logger.LogWarning($"Performance issues detected:\n{report.GetSummary()}");
                    // 알림 전송
                }
                
                // 매일 오전 9시에 실행
                var now = DateTime.UtcNow;
                var next9AM = now.Date.AddDays(1).AddHours(9);
                var delay = next9AM - now;
                
                await Task.Delay(delay, stoppingToken);
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "Error in daily performance check");
                await Task.Delay(TimeSpan.FromHours(1), stoppingToken);
            }
        }
    }
}
```

**2. 정기적인 VACUUM과 ANALYZE**

PostgreSQL은 자동으로 VACUUM과 ANALYZE를 수행하지만, 대용량 테이블의 경우 수동으로 스케줄링하는 것이 좋다.

```sql
-- 주말 새벽에 실행되는 유지보수 스크립트
DO $$
BEGIN
    -- Full VACUUM (디스크 공간 회수)
    VACUUM FULL ANALYZE player_events;
    VACUUM FULL ANALYZE player_sessions;
    
    -- 인덱스 재구축
    REINDEX TABLE CONCURRENTLY player_events;
    REINDEX TABLE CONCURRENTLY player_sessions;
    
    -- 통계 업데이트
    ANALYZE player_events;
    ANALYZE player_sessions;
    ANALYZE player_daily_stats;
END $$;
```

**3. 쿼리 타임아웃 설정**

모든 쿼리에 타임아웃을 설정하여 무한 대기를 방지한다.

```csharp
// 전역 설정
services.AddScoped<QueryFactory>(sp =>
```

**4. 캐싱 전략**

자주 사용되지만 자주 변경되지 않는 데이터는 캐싱한다.

```csharp
public class CachedPlayerRepository
{
    private readonly IMemoryCache _cache;
    private readonly PlayerRepository _repository;
    
    public async Task<Player> GetPlayerAsync(long playerId)
    {
        var cacheKey = $"player:{playerId}";
        
        if (_cache.TryGetValue(cacheKey, out Player cachedPlayer))
        {
            return cachedPlayer;
        }
        
        var player = await _repository.GetPlayerAsync(playerId);
        
        _cache.Set(cacheKey, player, TimeSpan.FromMinutes(10));
        
        return player;
    }
}
```
