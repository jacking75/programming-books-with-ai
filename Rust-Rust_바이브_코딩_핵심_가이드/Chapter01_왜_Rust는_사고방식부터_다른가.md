# Chapter 1. 왜 Rust는 C++/C#과 "사고방식"부터 다른가

> "언어를 배운다는 것은 문법을 외우는 것이 아니라, 그 언어가 세계를 보는 방식을 이해하는 것이다."

---

## 1.1 세 언어의 철학적 출발점

프로그래밍 언어는 철학의 산물이다. 언어를 설계한 사람들이 "무엇을 가장 중요한 가치로 두는가"에 따라 언어 전체의 구조가 결정된다. C++, C#, Rust 세 언어는 모두 성능과 안전성이라는 두 가지 목표를 가지고 있지만, 그 균형점을 완전히 다르게 잡았다.

### C++: "프로그래머를 믿는다"

C++의 철학은 **신뢰 기반**이다. 언어는 프로그래머에게 가능한 모든 권한을 부여하고, 그 권한을 올바르게 사용하는 것은 프로그래머의 책임이다. 원시 포인터를 마음대로 역참조할 수 있고, 메모리를 직접 `new`와 `delete`로 관리할 수 있으며, 타입 캐스팅도 거의 제약이 없다.

이 철학의 결과물은 **최대한의 유연성과 제로 코스트 추상화**다. 하드웨어에 가장 가깝게 작동하고, 런타임 오버헤드 없이 고수준 추상화를 만들 수 있다. 하지만 그 대가로 버퍼 오버플로, 댕글링 포인터, 데이터 레이스, 미정의 동작(undefined behavior) 같은 버그가 "프로그래머가 조심하지 않으면 언제든 발생할 수 있는" 일상적 위험이 되었다.

20년 이상 게임 서버를 작성한 C++ 개발자라면 이런 경험을 반드시 해 봤을 것이다.

```cpp
// C++ - 이 코드는 컴파일되고 실행된다. 하지만 미정의 동작이다.
std::vector<int> v = {1, 2, 3};
int* ptr = &v[0];
v.push_back(4); // 재할당이 발생할 수 있다
*ptr = 10;      // 이제 ptr은 댕글링 포인터다 - 미정의 동작!
```

컴파일러는 아무 말도 하지 않는다. 런타임에서 가끔 터지거나, 더 나쁜 경우 조용히 잘못된 데이터를 쓰고 넘어간다.

### C#: "런타임이 책임진다"

C#은 반대 방향으로 갔다. **런타임(CLR)이 모든 안전성을 책임**지는 철학이다. 가비지 컬렉터(GC)가 메모리를 자동으로 관리하고, 배열 경계 검사를 런타임에 수행하며, 타입 캐스팅 실패는 예외로 처리된다. 개발자는 메모리 관리를 신경 쓸 필요가 없다.

이 철학의 장점은 **생산성과 안전성의 균형**이다. null 참조 예외(NullReferenceException)가 발생하더라도 프로세스 전체가 죽는 것이 아니라 예외로 처리되고, 스택 추적을 통해 버그를 추적할 수 있다. 하지만 그 대가는 런타임 비용이다.

게임 서버 개발자에게 C#의 GC는 양날의 검이었다. 대부분의 시간 동안 편리하게 작동하지만, 접속자가 많아지거나 대규모 메모리 할당이 발생하면 **GC 스파이크**가 발생한다. 수십 밀리초에서 수백 밀리초까지 서버가 멈추는 GC 스파이크는 실시간 게임 서버에서 심각한 문제다.

```csharp
// C# - 이 패턴은 GC 압박을 만든다
void ProcessPackets(List<Packet> packets)
{
    // 매 틱마다 새 리스트를 만들면 GC가 자주 돌아야 한다
    var result = packets.Select(p => new ProcessedPacket(p)).ToList();
    // ...
} // result가 스코프를 벗어나면 GC 대상이 된다
```

더 근본적인 문제는 런타임 비용이다. 가상 메서드 호출의 vtable 룩업, 박싱/언박싱, 캐스팅 비용이 모두 런타임에 발생한다. 고성능 게임 서버에서는 이런 작은 비용들이 누적되어 병목이 된다.

### Rust: "컴파일러가 책임진다"

Rust는 두 언어의 장점을 모두 취하면서 단점을 제거하려는 시도다. **컴파일러가 모든 안전성을 책임**지는 철학이다. C++처럼 런타임 오버헤드가 없으면서도, C#처럼 메모리 관련 버그로부터 안전하다. 그 비결은 컴파일 타임에 모든 안전성 검증을 완료하는 것이다.

```rust
// Rust - 위의 C++ 버그 패턴은 컴파일 자체가 안 된다
let mut v = vec![1, 2, 3];
let ptr = &v[0];    // v를 불변으로 빌림
v.push(4);          // 오류! v가 이미 빌려진 상태에서 수정 불가
println!("{}", ptr);
```

```
error[E0502]: cannot borrow `v` as mutable because it is also borrowed as immutable
 --> src/main.rs:4:5
  |
3 |     let ptr = &v[0];
  |                - immutable borrow occurs here
4 |     v.push(4);
  |     ^^^^^^^^^ mutable borrow occurs here
5 |     println!("{}", ptr);
  |                    --- immutable borrow later used here
```

컴파일러가 정확히 무엇이 문제인지, 어디가 문제인지를 짚어 준다. 그리고 이 검증은 **런타임 비용이 전혀 없다**. 컴파일 타임에 끝나기 때문이다.

## 1.2 철학 차이가 만들어 내는 실질적 결과

세 언어의 철학 차이는 단순히 학문적 논의가 아니다. 실제 게임 서버 개발에서 매일 마주치는 문제들에 직접적인 영향을 미친다.

### 메모리 안전성 문제의 발생 시점

| 문제 유형 | C++ | C# | Rust |
|-----------|-----|-----|------|
| 버퍼 오버플로 | 런타임 (또는 조용히 발생) | 런타임 예외 | 컴파일 타임 오류 |
| 댕글링 포인터 | 런타임 (조용히 발생) | 불가 (GC) | 컴파일 타임 오류 |
| 이중 해제 | 런타임 크래시 | 불가 (GC) | 컴파일 타임 오류 |
| 데이터 레이스 | 런타임 (비결정적) | 런타임 예외 가능 | 컴파일 타임 오류 |
| Null 역참조 | 런타임 크래시 | 런타임 예외 | 컴파일 타임 오류 (Option) |

Rust의 열은 대부분 "컴파일 타임 오류"다. 즉 **버그가 프로덕션 환경에 도달하기 전에 개발 단계에서 차단**된다.

### 성능 비용

C++과 Rust는 "제로 코스트 추상화"를 지향한다. 고수준 코드를 작성해도 컴파일러가 최적 기계어를 생성한다는 의미다.

C#은 GC 비용, JIT 컴파일 비용, 런타임 타입 검사 비용이 있다. 현대의 .NET은 이를 상당히 줄였지만, 완전히 제거하는 것은 런타임 기반 언어의 본질적 한계다.

Rust는 C++ 수준의 성능을 내면서 안전성을 보장한다. 실제 벤치마크에서 Rust와 C++은 거의 동일한 성능을 보이는 경우가 많다.

```rust
// Rust - 이 추상화는 런타임 비용이 없다
// Iterator는 컴파일 타임에 인라인되어 단순 루프와 동일한 코드를 생성한다
let sum: u64 = (0..1_000_000)
    .filter(|x| x % 2 == 0)
    .map(|x| x * x)
    .sum();
```

어셈블리로 컴파일하면 단순한 반복문과 동일한 코드가 나온다. 클로저 호출 오버헤드, 반복자 추상화 비용이 전혀 없다.

## 1.3 바이브 코딩 관점에서의 차이

이 책의 주제인 **바이브 코딩(AI 보조 코딩)**에서 세 언어의 철학 차이는 더욱 극명하게 드러난다.

### C++로 바이브 코딩할 때의 위험

C++은 AI가 생성한 코드가 컴파일되어 실행되더라도 버그가 잠재할 수 있다. AI는 코드를 생성할 때 컴파일 가능성에 집중하는 경향이 있는데, C++에서는 컴파일이 된다고 해서 안전하다는 보장이 없다.

```cpp
// AI가 생성할 수 있는 C++ 코드 - 컴파일되지만 위험하다
class SessionManager {
    std::unordered_map<int, Session*> sessions; // 소유권 불명확
    
    Session* getSession(int id) {
        return sessions[id]; // sessions에 id가 없으면 null 반환
    }
    
    void removeSession(int id) {
        delete sessions[id]; // 이중 해제 가능성
        sessions.erase(id);
    }
};
```

이 코드는 컴파일되지만 여러 잠재적 버그를 품고 있다. AI에게 "세션 매니저 만들어 줘"라고 했을 때 이런 코드가 나왔다면, 사람이 직접 검토하지 않으면 운영 중에 터지는 버그가 된다.

### C#으로 바이브 코딩할 때의 위험

C#에서는 런타임이 많은 것을 처리해 주므로 AI 생성 코드가 안전한 경우가 많다. 하지만 게임 서버 특성상 다음과 같은 위험이 있다.

```csharp
// AI가 생성할 수 있는 C# 코드 - 동작하지만 성능 문제
public class RoomManager
{
    private Dictionary<int, Room> rooms = new Dictionary<int, Room>();
    
    // 락 없이 멀티스레드에서 접근 - 데이터 레이스!
    public Room GetRoom(int id)
    {
        return rooms.ContainsKey(id) ? rooms[id] : null;
    }
    
    // 매 호출마다 LINQ로 전체 컬렉션 탐색 - O(n) 비용
    public IEnumerable<Room> GetActiveRooms()
    {
        return rooms.Values.Where(r => r.IsActive).ToList();
    }
}
```

컴파일도 되고 단일 스레드에서는 잘 동작한다. 하지만 멀티스레드 게임 서버에서는 데이터 레이스가 발생할 수 있고, LINQ 연산의 매 호출마다 새 리스트가 생성되어 GC 압박을 만든다.

### Rust로 바이브 코딩할 때의 특성

Rust에서 AI가 생성한 코드는 컴파일이 안 되는 경우가 처음에는 많다. 이것이 오히려 **장점**이다.

```rust
// AI가 처음 생성할 수 있는 Rust 코드 - 컴파일 오류 발생
use std::collections::HashMap;

struct SessionManager {
    sessions: HashMap<u64, Session>,
}

impl SessionManager {
    fn get_session(&self, id: u64) -> Session {
        self.sessions[&id] // 오류! Session이 Copy가 아니므로 이동 불가
    }
    
    fn process_session(&mut self, id: u64) {
        let session = self.get_session(id);
        self.remove_session(id); // 오류! self가 이미 빌려진 상태
    }
}
```

이 코드는 컴파일되지 않는다. 컴파일러가 명확한 메시지를 준다. 이제 사람은 이 메시지를 읽고 AI에게 더 정확한 지시를 줄 수 있다:

> "get_session은 Session을 반환하지 말고 &Session 참조를 반환해. 그리고 process_session에서 동시에 불변 참조와 가변 접근을 하는 것은 빌림 규칙 위반이니, session ID를 먼저 조회한 후 수정하는 방식으로 분리해."

이 과정이 바이브 코딩에서 사람이 해야 할 역할이다. **AI가 생성한 코드의 컴파일 오류를 이해하고, 도메인 의도를 담은 올바른 지시로 변환하는 것.**

## 1.4 Rust가 "거추장스럽게" 만든 것들과 그 이유

처음 Rust를 접하면 "왜 이렇게 불편하게 만들었지?"라는 생각이 드는 부분들이 있다. 이것들은 실수로 만들어진 것이 아니라, 의도적으로 프로그래머가 "위험한 패턴"을 쉽게 쓰지 못하도록 설계된 것이다.

### 이동이 기본값인 이유

C++에서는 `std::move`를 명시해야 이동 시맨틱이 적용된다. Rust에서는 **이동이 기본값**이고 복사가 명시적이다. 이 역전이 불편하게 느껴질 수 있다.

```rust
let s1 = String::from("hello");
let s2 = s1; // s1의 소유권이 s2로 이동된다
println!("{}", s1); // 컴파일 오류! s1은 더 이상 유효하지 않다
```

```
error[E0382]: borrow of moved value: `s1`
```

이것이 불편하게 느껴지는 이유는 C++ 습관 때문이다. C++에서는 객체 복사가 기본값이고, 복사가 비싸다고 판단될 때만 `std::move`로 이동했다. 하지만 이 접근법은 "의도치 않은 복사"를 만들기 쉽다. 큰 버퍼를 함수에 넘길 때 실수로 복사가 발생하면 성능 문제가 생긴다.

Rust에서 이동이 기본값인 이유는 **"이 데이터는 지금 누가 갖고 있는가"를 항상 명확히 하기 위해서**다. 데이터는 정확히 한 곳에만 있고, 그것을 다른 곳으로 보내면 원래 위치에서는 사라진다. 이것이 소유권 추적을 가능하게 하는 기반이다.

### mut 키워드가 필요한 이유

Rust에서는 변수를 수정하려면 `mut`을 명시해야 한다.

```rust
let x = 5;
x = 6; // 컴파일 오류! x는 불변이다

let mut y = 5;
y = 6; // OK
```

이것도 처음에는 번거롭게 느껴진다. 하지만 이것은 **"이 코드의 어느 부분이 이 데이터를 수정하는가"를 코드를 읽는 사람(그리고 컴파일러)이 즉시 파악**할 수 있게 해 준다.

게임 서버 코드를 리뷰할 때 "이 객체가 이 함수에서 수정되는가?"를 파악하는 것은 중요하다. Rust에서는 함수 시그니처만 봐도 알 수 있다.

```rust
fn process_player(player: &Player) { ... }        // 읽기만 한다
fn level_up_player(player: &mut Player) { ... }   // 수정한다
fn consume_player(player: Player) { ... }          // 소유권을 가져간다
```

### 클론이 명시적인 이유

C++에서는 값을 복사할 때 복사 생성자가 묵시적으로 호출된다. C#에서는 참조 타입이 자동으로 공유된다. Rust에서는 `Clone`을 명시적으로 호출해야 한다.

```rust
let v1 = vec![1, 2, 3];
let v2 = v1.clone(); // 명시적으로 복사
let v3 = v1;         // v1의 소유권이 v3로 이동, v1은 더 이상 사용 불가
```

이 설계는 "복사 비용이 얼마인지를 개발자가 항상 의식하도록" 강제한다. 대용량 버퍼나 복잡한 게임 상태 객체를 실수로 복사하는 일을 방지한다.

AI에게 지시할 때 "여기서 clone()이 꼭 필요한가?"를 항상 질문해야 한다. AI가 컴파일 오류를 피하기 위해 무분별하게 `clone()`을 추가하는 것은 성능 문제의 씨앗이 된다.

## 1.5 런타임 비용 없는 안전성이 가능한 이유

Rust의 핵심 주장은 "런타임 비용 없이 메모리 안전성을 보장한다"는 것이다. 이것이 어떻게 가능한지 이해하는 것이 중요하다.

### 컴파일 타임 검증의 작동 방식

Rust 컴파일러는 코드를 컴파일할 때 단순히 문법 검사만 하는 것이 아니라, **소유권 흐름을 추적하고 빌림 규칙을 검증**한다. 이 과정은 컴파일 단계에서 완료되므로 생성된 기계어 코드에는 이 검증의 흔적이 전혀 남지 않는다.

C#에서 배열 경계 검사가 런타임에 발생하는 것과 대조적이다:

```csharp
// C# - 이 코드는 런타임에 배열 경계를 확인한다
int[] arr = new int[5];
int i = GetIndex(); // 런타임에 결정되는 인덱스
arr[i] = 10;        // 런타임에 i < 5인지 검사한다
```

Rust에서는 컴파일 타임에 검증할 수 없는 경우 경계 검사가 발생하지만, 가능한 경우 컴파일러가 이를 제거한다. 슬라이스 반복자를 사용하면 경계 검사가 전혀 없다.

```rust
// Rust - 반복자를 사용하면 경계 검사 없음
let arr = [1, 2, 3, 4, 5];
for &item in arr.iter() {
    // 컴파일러가 경계 내에 있음을 증명 - 런타임 검사 없음
}
```

### Drop 트레이트: RAII의 완성

C++에서 RAII(Resource Acquisition Is Initialization) 패턴을 사용하면 소멸자에서 자원을 해제한다. Rust는 이 패턴을 언어 수준에서 강제한다.

```rust
// Rust에서 모든 타입은 스코프를 벗어날 때 Drop이 호출된다
{
    let file = File::open("data.bin").unwrap();
    let buffer = vec![0u8; 1024 * 1024]; // 1MB 할당
    // ... 파일 읽기 ...
} // 여기서 file이 닫히고, buffer 메모리가 해제된다
  // 이것은 런타임 GC가 아니라 컴파일러가 결정하는 명확한 지점이다
```

GC가 없으므로 메모리 해제 시점이 완전히 예측 가능하다. 게임 서버에서 "이 프레임이 끝날 때 임시 객체들이 해제되는가?"라는 질문에 Rust는 컴파일 타임에 확실한 답을 줄 수 있다.

## 1.6 게임 서버 관점에서의 실질적 가치

23년 경력의 게임 서버 개발자에게 Rust가 제공하는 실질적 가치를 정리하면 다음과 같다.

### 1) 메모리 버그 클래스 전체 제거

C++ 게임 서버에서 발생하는 버그의 상당 부분은 메모리 관련이다. 댕글링 포인터, 이중 해제, 버퍼 오버플로, 초기화되지 않은 메모리 읽기... 이 모든 것이 Rust에서는 컴파일 타임에 차단된다.

게임 서버는 24/7 운영되어야 한다. 메모리 버그 하나가 서버 다운으로 이어지면 수천, 수만 명의 플레이어에게 영향을 준다. 이 버그 클래스를 언어 수준에서 제거하는 것은 운영 안정성에 직접적인 영향을 미친다.

### 2) 데이터 레이스 컴파일 타임 차단

현대 게임 서버는 멀티스레드/멀티태스크 구조다. 세션 관리, 룸 로직, DB 처리, 네트워크 I/O가 동시에 실행된다. 이 환경에서 데이터 레이스는 재현이 매우 어려운 최악의 버그 유형이다.

Rust는 데이터 레이스를 컴파일 타임에 차단한다. "이 데이터는 동시에 두 스레드에서 접근할 수 없다"를 타입 시스템이 강제한다. 락을 잊어버리거나 락 순서가 꼬이는 종류의 버그가 아예 발생하지 않는다.

### 3) GC 없는 예측 가능한 성능

게임 서버에서 지연 시간(latency)은 핵심 품질 지표다. GC 스파이크로 인한 간헐적 지연은 실시간 게임에서 매우 나쁜 사용자 경험을 만든다.

Rust는 GC가 없다. 메모리 할당과 해제가 결정론적으로 발생하므로 지연 시간이 예측 가능하다. 또한 C++ 수준의 성능을 내므로 같은 하드웨어에서 더 많은 동시 접속을 처리할 수 있다.

### 4) 바이브 코딩에서의 안전망

이 책의 핵심 주제다. AI가 생성한 코드를 프로덕션에 투입하는 과정에서 Rust 컴파일러는 강력한 안전망 역할을 한다.

AI는 컴파일이 되는 코드를 만드는 데는 능숙하지만, 미묘한 메모리 안전성 문제나 데이터 레이스를 항상 잡아내지는 못한다. C++ 코드를 AI가 생성하면 사람이 모든 잠재적 안전성 문제를 직접 검토해야 한다. Rust에서는 컴파일러가 이 검토의 상당 부분을 담당한다.

## 1.7 바이브 코딩에서 컴파일 오류를 만났을 때의 접근법

AI가 생성한 Rust 코드가 컴파일되지 않는 상황은 매우 흔하다. 이때 어떻게 접근해야 하는지를 처음부터 체화하는 것이 중요하다.

### 잘못된 접근법: "오류 없애기"

```
사람: "이 코드 컴파일 오류 고쳐줘"
AI: [clone() 추가, unsafe 블록 추가, 타입 캐스팅 강제]
결과: 컴파일은 되지만 의도가 훼손되거나 성능 문제 발생
```

이 접근법의 문제는 컴파일러가 오류를 낸 **이유**를 이해하지 않고 **증상만 해결**한다는 것이다.

### 올바른 접근법: "오류 이해하기"

```
컴파일 오류 메시지를 읽는다
→ "왜 컴파일러가 이것을 거부했는가?" 이해한다
→ "내 도메인 의도는 무엇인가?" 명확히 한다
→ "이 의도를 Rust의 소유권/빌림 규칙에 맞게 표현하면?" 판단한다
→ 이 판단을 AI에게 구체적으로 지시한다
```

예를 들어:

```
컴파일 오류: "cannot borrow `session_map` as mutable because it is also borrowed as immutable"

이해: session_map을 동시에 읽고 쓰려고 했다. Rust는 이것을 허용하지 않는다.

도메인 의도: 세션을 조회하고, 조회 결과에 따라 세션의 상태를 바꾸고 싶다.

Rust 방식으로: 먼저 세션 ID만 조회(불변 참조), 참조를 해제, 그 후 ID로 가변 접근하여 수정.

AI에게 지시: "session_map에서 조건에 맞는 세션 ID 목록을 먼저 수집하고 (불변 참조), 
그 다음 ID 목록을 순회하며 각 세션을 수정해 (가변 접근). 
두 단계로 분리해서 빌림 충돌을 피해줘."
```

## 1.8 Rust로의 전환 로드맵

C++/C# 경력자가 Rust로 전환하는 데 얼마나 걸리는가? 현실적으로:

**1~2주**: 기본 문법, 소유권 개념 이해. 간단한 프로그램을 작성할 수 있다. 하지만 컴파일러와 계속 싸우는 단계다.

**1~2개월**: 빌림 규칙이 직관적으로 느껴지기 시작한다. 컴파일러 메시지를 보고 무엇을 해야 하는지 대부분 이해한다. 간단한 멀티스레드 프로그램을 작성할 수 있다.

**3~6개월**: 트레이트, 생태계, 비동기 프로그래밍이 자연스러워진다. 현업 수준의 코드를 작성할 수 있다.

**6개월 이상**: "Rustic한" 코드 스타일이 몸에 배인다. C++/C#으로 돌아가기 불편해진다.

이 책은 이 로드맵을 **바이브 코딩**을 활용하여 단축하는 것을 목표로 한다. AI가 코드의 상당 부분을 생성해 주므로, 문법을 완전히 익히기 전에도 실용적인 코드를 만들 수 있다. 하지만 그러기 위해서는 이 책이 다루는 개념들을 이해하고 있어야 한다.

## 1.9 이 책에서 다루는 것과 다루지 않는 것

이 책은 Rust **입문서가 아니다**. Rust의 모든 문법을 순서대로 설명하지 않는다.

이 책이 다루는 것:
- 바이브 코딩으로 게임 서버를 만들 때 반드시 이해해야 하는 핵심 개념
- AI에게 올바른 코드를 만들도록 지시하기 위한 어휘와 사고 틀
- 게임 서버 도메인에서 자주 등장하는 패턴과 그것을 Rust로 표현하는 방법
- C++/C# 경력자가 함정에 빠지기 쉬운 부분과 그 해결법

이 책이 다루지 않는 것:
- Rust 문법의 완전한 레퍼런스
- 알고리즘과 자료구조 구현
- 클라이언트 사이드 개발 (게임 클라이언트, UI 등)
- Rust 언어 이론 (타입 이론, 범주론 등)

이 책을 읽는 당신은 이미 23년 경력의 게임 서버 개발자다. 프로그래밍의 기초, 멀티스레드, 네트워크, 데이터베이스 - 이 모든 것을 이미 알고 있다. 이 책은 그 지식 위에 Rust라는 새 언어를 얹는 다리 역할을 한다.

## 1.10 정리

이 챕터에서 핵심적으로 이해해야 할 것은 다음 세 가지다.

첫째, Rust는 "컴파일러가 책임진다"는 철학으로 설계되었다. 런타임 비용 없이 컴파일 타임에 메모리 안전성과 데이터 레이스 없는 동시성을 보장한다.

둘째, 이 철학 때문에 C++/C#에서 당연하게 쓰던 패턴들이 Rust에서 "거추장스럽게" 느껴진다. 하지만 그 거추장스러움은 실제 버그를 예방하는 방어막이다.

셋째, 바이브 코딩에서 Rust 컴파일러는 강력한 안전망이다. AI가 생성한 코드의 컴파일 오류는 "빨간 줄을 없애는 과제"가 아니라, 더 나은 지시를 만들기 위한 피드백이다.

다음 챕터에서는 이 모든 것의 근간이 되는 **소유권(Ownership)** 개념을 깊이 파헤친다.
