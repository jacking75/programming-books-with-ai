# 6장. SqlKata 쿼리 빌더 소개 (실습)
5장에서 MySqlConnector를 사용하여 직접 SQL 쿼리를 작성하고 실행하는 방법을 배웠다. 하지만 복잡한 쿼리를 문자열로 작성하다 보면 오타가 발생하기 쉽고, 동적으로 쿼리를 구성하기 어렵다. 이번 장에서는 SqlKata 쿼리 빌더를 사용하여 더 안전하고 직관적인 방식으로 쿼리를 작성하는 방법을 배운다.

## 6.1 SqlKata란?
SqlKata는 C#으로 SQL 쿼리를 작성할 수 있게 해주는 쿼리 빌더 라이브러리다. Fluent API 방식으로 쿼리를 구성하며, 다양한 데이터베이스 시스템을 지원한다.

### 6.1.1 SqlKata의 장점

**타입 안정성**

문자열로 쿼리를 작성하면 컴파일 시점에 오류를 찾을 수 없다. SqlKata는 C# 메서드를 사용하므로 IDE의 IntelliSense 지원을 받을 수 있고, 컴파일 시점에 많은 오류를 발견할 수 있다.

```csharp
// 문자열 쿼리: 오타를 발견하기 어려움
var query = "SELECT * FROM usres WHERE emial = @Email"; // usres, emial 오타

// SqlKata: IDE가 메서드 이름을 자동 완성해줌
var query = new Query("users").Where("email", email); // 오타 가능성 감소
```

**동적 쿼리 구성**

조건에 따라 쿼리를 동적으로 구성해야 할 때 문자열 연결보다 훨씬 깔끔하게 작성할 수 있다.

```csharp
// 문자열 방식: 복잡하고 실수하기 쉬움
var sql = "SELECT * FROM users WHERE 1=1";
if (!string.IsNullOrEmpty(nickname))
    sql += " AND nickname LIKE @Nickname";
if (minLevel > 0)
    sql += " AND level >= @MinLevel";

// SqlKata 방식: 명확하고 간결함
var query = new Query("users");
if (!string.IsNullOrEmpty(nickname))
    query.WhereLike("nickname", $"%{nickname}%");
if (minLevel > 0)
    query.Where("level", ">=", minLevel);
```

**SQL Injection 방지**

SqlKata는 모든 값을 자동으로 파라미터화하여 SQL Injection 공격을 원천적으로 방지한다.

**크로스 플랫폼 지원**

MySQL, PostgreSQL, SQL Server, SQLite 등 다양한 데이터베이스를 지원하며, 컴파일러만 변경하면 같은 코드로 다른 데이터베이스를 사용할 수 있다.

**가독성 향상**

복잡한 JOIN이나 서브쿼리를 포함한 쿼리도 체계적으로 작성할 수 있어 코드의 가독성이 높아진다.
  

## 6.2 SqlKata 패키지 설치
SqlKata를 사용하려면 두 개의 패키지를 설치해야 한다.

### 6.2.1 패키지 종류

**SqlKata**

핵심 쿼리 빌더 라이브러리다. 쿼리를 구성하는 API를 제공하지만, 실제로 데이터베이스에 쿼리를 실행하는 기능은 포함되어 있지 않다.

**SqlKata.Execution**

쿼리를 실제로 실행하는 기능을 제공한다. QueryFactory 클래스를 통해 데이터베이스 연결과 쿼리 실행을 처리한다.

### 6.2.2 패키지 설치 방법

**방법 1: NuGet 패키지 관리자 UI**

1. 솔루션 탐색기에서 프로젝트를 마우스 오른쪽 버튼으로 클릭한다
2. "NuGet 패키지 관리"를 선택한다
3. "찾아보기" 탭에서 "SqlKata"를 검색하고 설치한다
4. "SqlKata.Execution"을 검색하고 설치한다

**방법 2: 패키지 관리자 콘솔**

```powershell
Install-Package SqlKata
Install-Package SqlKata.Execution
```

**방법 3: .NET CLI**

```bash
dotnet add package SqlKata
dotnet add package SqlKata.Execution
```

설치가 완료되면 프로젝트 파일에 다음과 같이 패키지 참조가 추가된다.

```xml
<ItemGroup>
  <PackageReference Include="MySqlConnector" Version="2.3.7" />
  <PackageReference Include="SqlKata" Version="2.4.0" />
  <PackageReference Include="SqlKata.Execution" Version="2.4.0" />
</ItemGroup>
```
  

## 6.3 QueryFactory 설정
SqlKata.Execution의 `QueryFactory`는 데이터베이스 연결과 쿼리 실행을 관리하는 핵심 클래스다.

### 6.3.1 QueryFactory 클래스 생성
`Database` 폴더에 `GameQueryFactory.cs` 파일을 생성한다.

```csharp
using MySqlConnector;
using SqlKata.Compilers;
using SqlKata.Execution;
using System.Data;

namespace GameServer.Database
{
    public class GameQueryFactory
    {
        private readonly DbConnectionFactory _connectionFactory;
        private readonly MySqlCompiler _compiler;

        public GameQueryFactory(DbConnectionFactory connectionFactory)
        {
            _connectionFactory = connectionFactory;
            _compiler = new MySqlCompiler();
        }

        public async Task<QueryFactory> CreateAsync()
        {
            var connection = await _connectionFactory.CreateConnectionAsync();
            return new QueryFactory(connection, _compiler);
        }

        public QueryFactory Create()
        {
            var connection = _connectionFactory.CreateConnection();
            return new QueryFactory(connection, _compiler);
        }
    }
}
```

이 클래스는 `DbConnectionFactory`를 사용하여 데이터베이스 연결을 생성하고, `MySqlCompiler`를 설정하여 MySQL용 SQL을 생성하도록 한다.

`MySqlCompiler`는 SqlKata의 쿼리 객체를 MySQL 문법에 맞는 SQL 문자열로 변환한다. 다른 데이터베이스를 사용한다면 `PostgresCompiler`, `SqlServerCompiler` 등으로 변경할 수 있다.

### 6.3.2 Program.cs에 서비스 등록
`Program.cs`를 열고 `GameQueryFactory`를 의존성 주입 컨테이너에 등록한다.

```csharp
using GameServer.Database;

var builder = WebApplication.CreateBuilder(args);

// Configuration에서 Connection String 가져오기
var connectionString = builder.Configuration.GetConnectionString("MySQL");

// DbConnectionFactory 등록
builder.Services.AddSingleton(new DbConnectionFactory(connectionString));

// GameQueryFactory 등록
builder.Services.AddSingleton<GameQueryFactory>();

// 컨트롤러 추가
builder.Services.AddControllers();

// Swagger 설정
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseHttpsRedirection();
app.UseAuthorization();
app.MapControllers();

app.Run();
```

`AddSingleton<GameQueryFactory>()`로 등록하면 타입 기반으로 주입받을 수 있다.

### 6.3.3 QueryFactory 사용 시 주의사항
`QueryFactory`는 내부적으로 데이터베이스 연결을 가지고 있다. 사용이 끝나면 반드시 `Dispose`를 호출하여 연결을 반환해야 한다.

```csharp
// 올바른 사용
using var queryFactory = await _gameQueryFactory.CreateAsync();
var users = await queryFactory.Query("users").GetAsync();
// using 블록 종료 시 자동으로 Dispose 호출됨

// 잘못된 사용
var queryFactory = await _gameQueryFactory.CreateAsync();
var users = await queryFactory.Query("users").GetAsync();
// Dispose를 호출하지 않으면 연결이 반환되지 않음
```
  

## 6.4 기본 쿼리 작성
SqlKata로 기본적인 CRUD 쿼리를 작성하는 방법을 배운다. `Controllers` 폴더에 `SqlKataTestController.cs` 파일을 생성한다.

### 6.4.1 Insert 쿼리

```csharp
using GameServer.Database;
using Microsoft.AspNetCore.Mvc;

namespace GameServer.Controllers
{
    [ApiController]
    [Route("[controller]")]
    public class SqlKataTestController : ControllerBase
    {
        private readonly GameQueryFactory _queryFactory;

        public SqlKataTestController(GameQueryFactory queryFactory)
        {
            _queryFactory = queryFactory;
        }

        [HttpPost("insert-user")]
        public async Task<InsertUserResponse> InsertUser([FromBody] InsertUserRequest request)
        {
            try
            {
                using var db = await _queryFactory.CreateAsync();

                var userId = await db.Query("users").InsertGetIdAsync<long>(new
                {
                    email = request.Email,
                    password_hash = request.Password, // 실제로는 해싱 필요
                    nickname = request.Nickname,
                    created_at = DateTime.UtcNow
                });

                return new InsertUserResponse
                {
                    Result = ErrorCode.None,
                    UserId = userId,
                    Message = "사용자 생성 완료"
                };
            }
            catch (Exception ex)
            {
                return new InsertUserResponse
                {
                    Result = ErrorCode.DatabaseError,
                    Message = "사용자 생성 실패",
                    Error = ex.Message
                };
            }
        }
    }

    public class InsertUserRequest
    {
        public string Email { get; set; }
        public string Password { get; set; }
        public string Nickname { get; set; }
    }

    public class InsertUserResponse
    {
        public ErrorCode Result { get; set; }
        public long UserId { get; set; }
        public string Message { get; set; } = string.Empty;
        public string Error { get; set; } = string.Empty;
    }
}
```

`Query("users")`는 users 테이블을 대상으로 하는 쿼리를 시작한다. `InsertGetIdAsync`는 데이터를 삽입하고 자동 증가된 ID를 반환한다.

익명 객체를 사용하여 삽입할 데이터를 전달한다. 속성 이름은 테이블의 컬럼 이름과 일치해야 한다.

SqlKata가 생성하는 실제 SQL은 다음과 같다.

```sql
INSERT INTO `users` (`email`, `password_hash`, `nickname`, `created_at`)
VALUES (@p0, @p1, @p2, @p3);
SELECT LAST_INSERT_ID();
```

모든 값은 자동으로 파라미터화되어 SQL Injection을 방지한다.

데이터를 조회하는 방법을 배운다.

**전체 조회**

```csharp
[HttpPost("get-all-users")]
public async Task<GetAllUsersResponse> GetAllUsers()

            .GetAsync();

        return new GetAllUsersResponse

            Result = ErrorCode.None,
            Users = users.ToList()

    catch (Exception ex)

            Error = ex.Message
        };
    }

public class GetAllUsersResponse
{
    public ErrorCode Result { get; set; }
    public List<dynamic> Users { get; set; } = new();
    public string Message { get; set; } = string.Empty;
    public string Error { get; set; } = string.Empty;
}
```

`Select` 메서드로 조회할 컬럼을 지정한다. 생략하면 모든 컬럼(`*`)을 조회한다. `GetAsync`는 모든 결과를 `IEnumerable<dynamic>` 형태로 반환한다.

**조건부 조회**

```csharp
[HttpPost("get-user-by-email")]
public async Task<GetUserByEmailResponse> GetUserByEmail([FromBody] GetUserByEmailRequest request)
{
    try
    {
        using var db = await _queryFactory.CreateAsync();

        var user = await db.Query("users")
            .Select("user_id", "email", "nickname", "created_at")
            .Where("email", request.Email)
            .FirstOrDefaultAsync();

        if (user == null)
        {
            return new GetUserByEmailResponse
            {
                Result = ErrorCode.UserNotFound,
                Message = "사용자를 찾을 수 없습니다"
            };
        }

        return new GetUserByEmailResponse
        {
            Result = ErrorCode.None,
            User = user
        };
    }
    catch (Exception ex)
    {
        return new GetUserByEmailResponse
        {
            Result = ErrorCode.DatabaseError,
            Message = "조회 실패",
            Error = ex.Message
        };
    }
}

public class GetUserByEmailRequest
{
    public string Email { get; set; }
}

public class GetUserByEmailResponse
{
    public ErrorCode Result { get; set; }
    public dynamic? User { get; set; }
    public string Message { get; set; } = string.Empty;
    public string Error { get; set; } = string.Empty;
}
```

`Where` 메서드로 조건을 추가한다. `FirstOrDefaultAsync`는 첫 번째 결과를 반환하거나, 결과가 없으면 null을 반환한다.

생성되는 SQL은 다음과 같다.

```sql
SELECT `user_id`, `email`, `nickname`, `created_at`
FROM `users`
WHERE `email` = @p0
LIMIT 1
```

**복잡한 조건**

```csharp
[HttpPost("search-users")]
public async Task<SearchUsersResponse> SearchUsers([FromBody] SearchUsersRequest request)
{
    try
    {
        using var db = await _queryFactory.CreateAsync();

        var query = db.Query("users")
            .Select("user_id", "email", "nickname", "created_at");

        // 닉네임으로 검색 (선택적)
        if (!string.IsNullOrEmpty(request.Nickname))
        {
            query.WhereLike("nickname", $"%{request.Nickname}%");
        }

        // 생성일 범위 (선택적)
        if (request.StartDate.HasValue)
        {
            query.Where("created_at", ">=", request.StartDate.Value);
        }

        if (request.EndDate.HasValue)
        {
            query.Where("created_at", "<=", request.EndDate.Value);
        }

        // 페이징
        var total = await query.CountAsync<int>();
        var users = await query
            .OrderByDesc("created_at")
            .Skip(request.Page * request.PageSize)
            .Take(request.PageSize)
            .GetAsync();

        return new SearchUsersResponse
        {
            Result = ErrorCode.None,
            Total = total,
            Page = request.Page,
            PageSize = request.PageSize,
            Users = users.ToList()
        };
    }
    catch (Exception ex)
    {
        return new SearchUsersResponse
        {
            Result = ErrorCode.DatabaseError,
            Message = "검색 실패",
            Error = ex.Message
        };
    }
}

public class SearchUsersRequest
{
    public string? Nickname { get; set; }
    public DateTime? StartDate { get; set; }
    public DateTime? EndDate { get; set; }
    public int Page { get; set; } = 0;
    public int PageSize { get; set; } = 10;
}

public class SearchUsersResponse
{
    public ErrorCode Result { get; set; }
    public int Total { get; set; }
    public int Page { get; set; }
    public int PageSize { get; set; }
    public List<dynamic> Users { get; set; } = new();
    public string Message { get; set; } = string.Empty;
    public string Error { get; set; } = string.Empty;
}
```

이 예제는 동적 쿼리 구성의 장점을 보여준다. 조건이 제공되었을 때만 해당 WHERE 절이 추가된다.

`WhereLike`는 LIKE 연산자를 사용하며, `OrderByDesc`는 내림차순 정렬을 의미한다. `Skip`과 `Take`는 페이징 처리에 사용된다.

### 6.4.4 Update 쿼리

```csharp
[HttpPost("update-nickname")]
public async Task<UpdateNicknameResponse> UpdateNickname([FromBody] UpdateNicknameRequest request)
{
    try
    {
        using var db = await _queryFactory.CreateAsync();

        var rowsAffected = await db.Query("users")
            .Where("email", request.Email)
            .UpdateAsync(new
            {
                nickname = request.Nickname
            });

        if (rowsAffected == 0)
        {
            return new UpdateNicknameResponse
            {
                Result = ErrorCode.UserNotFound,
                Message = "사용자를 찾을 수 없습니다"
            };
        }

        return new UpdateNicknameResponse
        {
            Result = ErrorCode.None,
            Message = "닉네임 변경 완료"
        };
    }
    catch (Exception ex)
    {
        return new UpdateNicknameResponse
        {
            Result = ErrorCode.DatabaseError,
            Message = "닉네임 변경 실패",
            Error = ex.Message
        };
    }
}

public class UpdateNicknameRequest
{
    public string Email { get; set; }
    public string Nickname { get; set; }
}
```

`UpdateAsync` 메서드는 익명 객체로 전달된 속성들을 업데이트한다. WHERE 절은 `Where` 메서드로 지정한다.

생성되는 SQL은 다음과 같다.

```sql
UPDATE `users`
SET `nickname` = @p0
WHERE `email` = @p1
```

`DeleteAsync`는 WHERE 절과 일치하는 모든 행을 삭제한다. WHERE 절을 지정하지 않으면 모든 행이 삭제되므로 주의해야 한다.
  

## 6.5 SqlKata의 고급 기능

### 6.5.1 여러 조건 조합

**AND 조건**

```csharp
var users = await db.Query("users")
    .Where("level", ">=", 10)
    .Where("exp", ">", 1000)
    .GetAsync();
```

여러 `Where`를 체이닝하면 AND 조건으로 결합된다.

```sql
SELECT * FROM `users`
WHERE `level` >= @p0 AND `exp` > @p1
```

**OR 조건**

```csharp
var users = await db.Query("users")
    .Where(q => q
        .Where("level", ">=", 50)
        .OrWhere("vip_level", ">", 0)
    )
    .GetAsync();
```

`OrWhere` 메서드를 사용하거나, 람다 표현식으로 서브 조건을 그룹화한다.

```sql
SELECT * FROM `users`
WHERE (`level` >= @p0 OR `vip_level` > @p1)
```

**IN 조건**

```csharp
var userIds = new[] { 1, 2, 3, 4, 5 };

var users = await db.Query("users")
    .WhereIn("user_id", userIds)
    .GetAsync();
```

```sql
SELECT * FROM `users`
WHERE `user_id` IN (@p0, @p1, @p2, @p3, @p4)
```

**NOT IN 조건**

```csharp
var bannedUserIds = new[] { 10, 20, 30 };

var users = await db.Query("users")
    .WhereNotIn("user_id", bannedUserIds)
    .GetAsync();
```

**NULL 체크**

```csharp
// NULL인 경우
var users = await db.Query("users")
    .WhereNull("deleted_at")
    .GetAsync();

// NULL이 아닌 경우
var deletedUsers = await db.Query("users")
    .WhereNotNull("deleted_at")
    .GetAsync();
```

**BETWEEN 조건**

```csharp
var users = await db.Query("users")
    .WhereBetween("level", 10, 50)
    .GetAsync();
```

```sql
SELECT * FROM `users`
WHERE `level` BETWEEN @p0 AND @p1
```

### 6.5.2 JOIN 쿼리
게임에서 사용자와 캐릭터 정보를 함께 조회하는 경우를 생각해본다.

```csharp
[HttpPost("get-users-with-characters")]
public async Task<GetUsersWithCharactersResponse> GetUsersWithCharacters()
{
    try
    {
        using var db = await _queryFactory.CreateAsync();

        var result = await db.Query("users")
            .Select("users.user_id", "users.nickname", "characters.character_id", 
                    "characters.character_name", "characters.level")
            .LeftJoin("characters", "users.user_id", "characters.user_id")
            .GetAsync();

        return new GetUsersWithCharactersResponse
        {
            Result = ErrorCode.None,
            Data = result.ToList()
        };
    }
    catch (Exception ex)
    {
        return new GetUsersWithCharactersResponse
        {
            Result = ErrorCode.DatabaseError,
            Message = "조회 실패",
            Error = ex.Message
        };
    }
}

public class GetUsersWithCharactersResponse
{
    public ErrorCode Result { get; set; }
    public List<dynamic> Data { get; set; } = new();
    public string Message { get; set; } = string.Empty;
    public string Error { get; set; } = string.Empty;
}
```

`LeftJoin`의 첫 번째 인자는 조인할 테이블, 두 번째와 세 번째 인자는 조인 조건의 양쪽 컬럼이다.

생성되는 SQL은 다음과 같다.

```sql
SELECT `users`.`user_id`, `users`.`nickname`, 
       `characters`.`character_id`, `characters`.`character_name`, `characters`.`level`
FROM `users`
LEFT JOIN `characters` ON `users`.`user_id` = `characters`.`user_id`
```

**여러 JOIN**

```csharp
var result = await db.Query("users")
    .Join("characters", "users.user_id", "characters.user_id")
    .Join("inventory", "characters.character_id", "inventory.character_id")
    .Join("items", "inventory.item_id", "items.item_id")
    .Where("users.user_id", userId)
    .GetAsync();
```

### 6.5.3 집계 함수

**COUNT**

```csharp
var totalUsers = await db.Query("users").CountAsync<int>();
var activeUsers = await db.Query("users")
    .WhereNull("deleted_at")
    .CountAsync<int>();
```

**SUM, AVG, MIN, MAX**

```csharp
// 모든 사용자의 총 레벨 합
var totalLevel = await db.Query("users").SumAsync<int>("level");

// 평균 레벨
var avgLevel = await db.Query("users").AverageAsync<double>("level");

// 최소/최대 레벨
var minLevel = await db.Query("users").MinAsync<int>("level");
var maxLevel = await db.Query("users").MaxAsync<int>("level");
```

**GROUP BY**

```csharp
[HttpPost("get-level-distribution")]
public async Task<GetLevelDistributionResponse> GetLevelDistribution()
{
    try
    {
        using var db = await _queryFactory.CreateAsync();

        var distribution = await db.Query("users")
            .Select("level")
            .SelectRaw("COUNT(*) as count")
            .GroupBy("level")
            .OrderBy("level")
            .GetAsync();

        return new GetLevelDistributionResponse
        {
            Result = ErrorCode.None,
            Distribution = distribution.ToList()
        };
    }
    catch (Exception ex)
    {
        return new GetLevelDistributionResponse
        {
            Result = ErrorCode.DatabaseError,
            Message = "조회 실패",
            Error = ex.Message
        };
    }
}

public class GetLevelDistributionResponse
{
    public ErrorCode Result { get; set; }
    public List<dynamic> Distribution { get; set; } = new();
    public string Message { get; set; } = string.Empty;
    public string Error { get; set; } = string.Empty;
}
```

`SelectRaw`는 원시 SQL 표현식을 사용할 수 있게 한다.

```sql
SELECT `level`, COUNT(*) as count
FROM `users`
GROUP BY `level`
ORDER BY `level`
```

### 6.5.4 서브쿼리

```csharp
// 평균 레벨보다 높은 사용자 조회
var avgLevelQuery = new Query("users").AsAverage("level");

var users = await db.Query("users")
    .Where("level", ">", avgLevelQuery)
    .GetAsync();
```

생성되는 SQL은 다음과 같다.

```sql
SELECT * FROM `users`
WHERE `level` > (SELECT AVG(`level`) FROM `users`)
```

### 6.5.5 Raw 쿼리
SqlKata로 표현하기 어려운 복잡한 쿼리는 Raw SQL을 사용할 수 있다.

```csharp
var users = await db.SelectAsync("SELECT * FROM users WHERE email = @email", 
    new { email = "test@example.com" });
```

하지만 가능한 한 SqlKata의 쿼리 빌더 메서드를 사용하는 것이 좋다.
  

## 6.6 쿼리 디버깅
개발 중에 SqlKata가 생성하는 실제 SQL을 확인하고 싶을 때가 있다.

### 6.6.1 쿼리 문자열 출력

```csharp
[HttpPost("debug-query")]
public DebugQueryResponse DebugQuery()
{
    var compiler = new MySqlCompiler();
    
    var query = new SqlKata.Query("users")
        .Where("level", ">=", 10)
        .Where("exp", ">", 1000)
        .OrderByDesc("level")
        .Limit(10);

    var compiled = compiler.Compile(query);

    return new DebugQueryResponse
    {
        Result = ErrorCode.None,
        Sql = compiled.Sql,
        Bindings = compiled.Bindings.ToList()
    };
}

public class DebugQueryResponse
{
    public ErrorCode Result { get; set; }
    public string Sql { get; set; } = string.Empty;
    public List<object> Bindings { get; set; } = new();
}
```

이 엔드포인트를 호출하면 다음과 같은 응답을 받는다.

```json
{
  "sql": "SELECT * FROM `users` WHERE `level` >= ? AND `exp` > ? ORDER BY `level` DESC LIMIT ?",
  "bindings": [10, 1000, 10]
}
```

`?`는 파라미터 플레이스홀더이고, `bindings` 배열에 실제 값이 담긴다.

### 6.6.2 실행 전 쿼리 로깅

개발 환경에서 모든 쿼리를 로그로 출력하는 헬퍼를 만들 수 있다.

```csharp
public static class QueryExtensions
{
    public static async Task<IEnumerable<dynamic>> GetWithLoggingAsync(
        this SqlKata.Query query, 
        QueryFactory db,
        ILogger logger)
    {
        var compiler = new MySqlCompiler();
        var compiled = compiler.Compile(query);
        logger.LogInformation("Executing Query: {Sql}", compiled.Sql);
        logger.LogInformation("Parameters: {Bindings}", 
            string.Join(", ", compiled.Bindings));
        
        return await db.GetAsync(query);
    }
}
```

## 6.7 정리

이 장에서는 SqlKata 쿼리 빌더의 기초를 배웠다. SqlKata와 SqlKata.Execution 패키지를 설치하고, QueryFactory를 설정하여 데이터베이스 연결과 쿼리 실행을 관리하는 방법을 익혔다.

기본적인 CRUD 쿼리(Insert, Select, Update, Delete)를 작성하는 방법을 실습했고, WHERE 조건, JOIN, 집계 함수, 서브쿼리 등의 고급 기능도 살펴보았다. SqlKata는 타입 안정성, 동적 쿼리 구성, SQL Injection 방지 등 많은 장점을 제공한다.

다음 장에서는 Repository 패턴을 구현하여 데이터 접근 로직을 체계적으로 구조화하고, SqlKata를 사용한 더 복잡한 쿼리와 트랜잭션 처리 방법을 배운다. Repository 패턴은 비즈니스 로직과 데이터 접근 로직을 분리하여 코드의 유지보수성과 테스트 가능성을 높여준다.

지금까지 작성한 테스트 컨트롤러는 학습 목적이므로, 실제 프로젝트에서는 적절한 레이어 분리와 에러 처리, 로깅을 구현해야 한다.
