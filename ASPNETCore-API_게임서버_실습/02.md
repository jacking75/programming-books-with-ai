# 2장. HTTP POST 기반 API 서버 기초
이 장에서는 게임 서버 개발의 기초가 되는 HTTP POST 메서드와 API 설계 방식을 학습한다. 일반적인 RESTful API와 게임 서버에서 주로 사용하는 커맨드 기반 API의 차이를 이해하고, JSON 기반의 요청/응답 구조를 설계한다. 마지막으로 ASP.NET Web API에서 컨트롤러를 생성하고 라우팅을 설정하는 방법을 실습한다.

## 2.1 HTTP POST 메서드의 이해
HTTP는 클라이언트와 서버 간의 통신을 위한 프로토콜이다. HTTP는 여러 메서드를 제공하지만, 우리는 POST 메서드만을 사용하여 게임 서버를 구축한다.

### 2.1.1 HTTP 메서드 개요
HTTP는 주로 다음과 같은 메서드를 제공한다.

- **GET**: 리소스 조회
- **POST**: 리소스 생성 또는 데이터 전송
- **PUT**: 리소스 전체 수정
- **PATCH**: 리소스 일부 수정
- **DELETE**: 리소스 삭제

각 메서드는 특정 목적을 가지고 있으며, RESTful API에서는 이러한 메서드를 의미에 맞게 사용한다.

### 2.1.2 게임 서버에서 POST만 사용하는 이유
많은 온라인 게임 서버, 특히 모바일 게임 서버는 모든 요청에 POST 메서드만을 사용한다. 이러한 접근 방식을 선택하는 이유는 다음과 같다.

**보안성**: GET 메서드는 URL에 파라미터가 노출되므로 민감한 정보(사용자 ID, 토큰 등)가 로그나 브라우저 히스토리에 남을 수 있다. POST는 요청 본문에 데이터를 담아 전송하므로 상대적으로 안전하다.

**데이터 크기 제한 없음**: GET 메서드는 URL 길이 제한으로 인해 전송할 수 있는 데이터의 양이 제한된다. POST는 요청 본문에 데이터를 담으므로 크기 제한이 훨씬 크다.

**일관성**: 모든 API가 동일한 방식(POST)으로 동작하면 클라이언트 개발자가 API를 사용하기 쉽고, 서버 개발자도 일관된 패턴으로 코드를 작성할 수 있다.

**캐싱 방지**: 게임 데이터는 실시간으로 변경되는 경우가 많아 캐싱이 부적절하다. POST 요청은 기본적으로 캐싱되지 않는다.

**복잡한 데이터 구조**: 게임 API는 중첩된 객체, 배열 등 복잡한 데이터 구조를 주고받는 경우가 많다. POST는 JSON 형식으로 복잡한 데이터를 쉽게 전송할 수 있다.

### 2.1.3 POST 메서드의 특징
POST 메서드는 다음과 같은 특징을 가진다.

**요청 본문(Body) 사용**: 데이터를 URL이 아닌 요청 본문에 담아 전송한다. 일반적으로 JSON 형식을 사용한다.

**멱등성 없음**: 동일한 요청을 여러 번 수행하면 서버 상태가 매번 변경될 수 있다. 예를 들어 아이템 구매 API를 두 번 호출하면 아이템이 두 번 구매된다.

**캐싱되지 않음**: 브라우저나 프록시 서버에서 기본적으로 캐싱하지 않는다.

**안전하지 않음(Not Safe)**: 서버의 상태를 변경할 수 있다. GET은 '안전한' 메서드로 서버 상태를 변경하지 않아야 한다.
  

## 2.2 RESTful API vs 커맨드 기반 API
웹 서비스를 설계할 때 두 가지 주요 접근 방식이 있다. RESTful API와 커맨드 기반 API(RPC 스타일)다. 게임 서버 개발에서는 커맨드 기반 API가 더 적합한 경우가 많다.

### 2.2.1 RESTful API
RESTful API는 리소스 중심으로 설계되며, HTTP 메서드의 의미에 따라 동작을 정의한다.

**설계 원칙**:
- 리소스를 URI로 표현한다
- HTTP 메서드로 행위를 표현한다
- 무상태(Stateless) 통신을 한다
- 계층화된 시스템 구조를 가진다

**예시**:
```
GET    /users/123           사용자 조회
POST   /users               사용자 생성
PUT    /users/123           사용자 전체 수정
PATCH  /users/123           사용자 일부 수정
DELETE /users/123           사용자 삭제
GET    /users/123/items     사용자의 아이템 목록 조회
```

RESTful API는 웹 서비스나 오픈 API에 적합하다. 리소스와 행위가 명확하게 분리되어 있어 이해하기 쉽고, 표준화된 방식으로 설계할 수 있다.

### 2.2.2 커맨드 기반 API (RPC 스타일)
커맨드 기반 API는 행위(Action) 중심으로 설계되며, 특정 작업을 수행하는 명령으로 API를 구성한다.

**설계 원칙**:
- 수행할 작업을 URI 또는 요청 본문에 명시한다
- 모든 요청에 동일한 HTTP 메서드를 사용한다 (주로 POST)
- 요청마다 필요한 모든 정보를 전달한다
- 응답은 작업 결과를 포함한다

**예시**:
```
POST /api/account/register       회원가입
POST /api/account/login          로그인
POST /api/character/create       캐릭터 생성
POST /api/character/levelup      캐릭터 레벨업
POST /api/item/purchase          아이템 구매
POST /api/dungeon/enter          던전 입장
POST /api/dungeon/clear          던전 클리어
```

### 2.2.3 게임 서버에서 커맨드 기반 API를 선택하는 이유
게임 서버는 다음과 같은 이유로 커맨드 기반 API가 더 적합하다.

**명확한 의도 표현**: "아이템 구매", "던전 클리어", "친구 추가"와 같은 게임 액션은 리소스보다 행위로 표현하는 것이 자연스럽다.

**복잡한 비즈니스 로직**: 게임의 많은 기능은 단순 CRUD를 넘어선 복잡한 로직을 포함한다. 예를 들어 "던전 클리어"는 여러 테이블의 데이터를 동시에 변경하고 보상을 지급하는 복합적인 작업이다.

**트랜잭션 단위**: 게임의 각 액션은 하나의 트랜잭션으로 처리되어야 하는 경우가 많다. 커맨드 기반 API는 이러한 트랜잭션 단위를 명확하게 표현한다.

**클라이언트 개발 편의성**: 게임 클라이언트 개발자는 "이 버튼을 누르면 이 API를 호출한다"는 식으로 직관적으로 이해할 수 있다.

**버전 관리 용이성**: 게임은 지속적으로 업데이트되고 새로운 기능이 추가된다. 커맨드 기반 API는 새로운 엔드포인트를 추가하는 방식으로 쉽게 확장할 수 있다.

### 2.2.4 우리가 사용할 API 설계 방식
이 교육 과정에서는 커맨드 기반 API를 사용한다. 모든 API는 POST 메서드를 사용하며, 다음과 같은 URL 구조를 따른다.

```
POST /api/{카테고리}/{액션}
```

예를 들어:
```
POST /api/account/register
POST /api/account/login
POST /api/character/create
POST /api/item/purchase
```

이러한 구조는 게임의 기능을 카테고리별로 그룹화하고, 각 기능을 명확한 액션으로 표현한다.
  

## 2.3 요청/응답 데이터 구조 설계 (JSON)
게임 서버와 클라이언트는 JSON(JavaScript Object Notation) 형식으로 데이터를 주고받는다. JSON은 가독성이 좋고, 대부분의 프로그래밍 언어에서 쉽게 파싱할 수 있어 웹 API의 표준 데이터 형식으로 자리 잡았다.

### 2.3.1 JSON 기본 구조
JSON은 키-값 쌍으로 데이터를 표현한다.

**기본 예시**:
```json
{
  "userId": "player123",
  "userName": "DragonSlayer",
  "level": 45,
  "gold": 150000,
  "isActive": true
}
```

**배열**:
```json
{
  "items": [
    {
      "itemId": 1001,
      "itemName": "검",
      "quantity": 1
    },
    {
      "itemId": 2001,
      "itemName": "물약",
      "quantity": 50
    }
  ]
}
```

**중첩 객체**:
```json
{
  "character": {
    "name": "DragonSlayer",
    "stats": {
      "hp": 1000,
      "mp": 500,
      "attack": 150,
      "defense": 80
    }
  }
}
```

### 2.3.2 표준화된 요청 구조 설계
게임 서버의 모든 API는 일관된 요청 구조를 사용하면 개발과 유지보수가 편리하다. 우리는 다음과 같은 표준 요청 구조를 사용한다.

**기본 요청 구조**:
```json
{
  "userId": "player123",
  "token": "abc123def456...",
  "data": {
    // API별 실제 데이터
  }
}
```

**요청 필드 설명**:
- **userId**: 요청을 보내는 사용자의 고유 식별자
- **token**: 인증 토큰 (로그인 후 발급)
- **data**: API별로 필요한 실제 데이터

**회원가입 요청 예시**:
```json
{
  "data": {
    "email": "player@example.com",
    "password": "SecurePass123!",
    "nickname": "DragonSlayer"
  }
}
```

회원가입과 로그인 API는 아직 인증되지 않았으므로 userId와 token이 없다.

**로그인 요청 예시**:
```json
{
  "data": {
    "email": "player@example.com",
    "password": "SecurePass123!"
  }
}
```

**캐릭터 생성 요청 예시** (로그인 후):
```json
{
  "userId": "player123",
  "token": "abc123def456...",
  "data": {
    "characterName": "DragonSlayer",
    "characterClass": "Warrior"
  }
}
```

**아이템 구매 요청 예시**:
```json
{
  "userId": "player123",
  "token": "abc123def456...",
  "data": {
    "itemId": 1001,
    "quantity": 5
  }
}
```

### 2.3.3 표준화된 응답 구조 설계
응답도 일관된 구조를 사용하여 클라이언트에서 처리하기 쉽게 만든다.

**기본 응답 구조**:
```json
{
  "result": 0,
  "message": "Success",
  "data": {
    // API별 응답 데이터
  }
}
```

**응답 필드 설명**:
- **result**: 요청 처리 결과 코드 (0: 성공, 그 외: 에러 코드)
- **message**: 결과에 대한 설명 메시지
- **data**: API별 반환 데이터 (성공 시)

**성공 응답 예시**:
```json
{
  "result": 0,
  "message": "Success",
  "data": {
    "userId": "player123",
    "token": "abc123def456...",
    "nickname": "DragonSlayer",
    "level": 1,
    "gold": 1000
  }
}
```

**실패 응답 예시**:
```json
{
  "result": 1001,
  "message": "Invalid email or password",
  "data": null
}
```

**데이터가 있는 성공 응답 예시** (캐릭터 정보 조회):
```json
{
  "result": 0,
  "message": "Success",
  "data": {
    "characterId": 456,
    "characterName": "DragonSlayer",
    "level": 45,
    "experience": 123456,
    "stats": {
      "hp": 1000,
      "mp": 500,
      "attack": 150,
      "defense": 80
    },
    "equippedItems": [
      {
        "itemId": 1001,
        "itemName": "전설의 검",
        "slot": "weapon"
      }
    ]
  }
}
```

### 2.3.4 에러 코드 체계
일관된 에러 코드 체계를 사용하면 클라이언트에서 에러를 효과적으로 처리할 수 있다.

**에러 코드 범위 정의**:
```
0           : 성공
1000-1999   : 인증 관련 에러
2000-2999   : 데이터베이스 에러
3000-3999   : 비즈니스 로직 에러
4000-4999   : 유효성 검증 에러
5000-5999   : 서버 내부 에러
```

**주요 에러 코드 예시**:
```
0     : Success (성공)
1001  : Invalid Credentials (잘못된 인증 정보)
1002  : Token Expired (토큰 만료)
1003  : Invalid Token (유효하지 않은 토큰)
1004  : User Not Found (사용자를 찾을 수 없음)
2001  : Database Connection Failed (DB 연결 실패)
2002  : Query Execution Failed (쿼리 실행 실패)
3001  : Insufficient Gold (골드 부족)
3002  : Item Not Found (아이템을 찾을 수 없음)
3003  : Already Purchased (이미 구매함)
4001  : Invalid Request Data (잘못된 요청 데이터)
4002  : Missing Required Field (필수 필드 누락)
5001  : Internal Server Error (서버 내부 에러)
```

나중에 코드에서 이러한 에러 코드를 상수로 정의하여 사용할 것이다.
 

## 2.4 컨트롤러 생성 및 라우팅 설정
이제 실제로 ASP.NET Web API에서 컨트롤러를 생성하고 라우팅을 설정하는 방법을 실습한다.

### 2.4.1 컨트롤러의 역할
컨트롤러(Controller)는 클라이언트의 HTTP 요청을 받아 처리하고 응답을 반환하는 역할을 한다. ASP.NET Web API에서 컨트롤러는 다음과 같은 작업을 수행한다.

- 요청 데이터 수신 및 파싱
- 데이터 유효성 검증
- 비즈니스 로직 호출
- 응답 데이터 생성 및 반환
- 에러 처리

### 2.4.2 첫 번째 컨트롤러 생성
게임 서버의 첫 번째 컨트롤러로 계정 관련 기능을 담당하는 `AccountController`를 생성한다.

Visual Studio에서 다음 단계를 따른다.

1. 솔루션 탐색기에서 `Controllers` 폴더를 마우스 오른쪽 버튼으로 클릭한다.
2. '추가' > '컨트롤러'를 선택한다.
3. 'API' 섹션에서 'API Controller - Empty'를 선택하고 '추가'를 클릭한다.
4. 이름을 `AccountController`로 입력하고 '추가'를 클릭한다.

생성된 기본 코드는 다음과 같다.

```csharp
using Microsoft.AspNetCore.Mvc;

namespace GameServer.Controllers
{
    [Route("api/[controller]")]
    [ApiController]
    public class AccountController : ControllerBase
    {
    }
}
```

### 2.4.3 라우팅 설정
라우팅은 클라이언트의 요청 URL을 적절한 컨트롤러와 액션 메서드로 연결하는 메커니즘이다.

**[Route] 특성 수정**:

기본 생성된 `[Route("api/[controller]")]`는 컨트롤러 이름을 URL에 포함한다. 우리는 더 명확한 URL 구조를 위해 다음과 같이 수정한다.

```csharp
[Route("api/account")]
[ApiController]
public class AccountController : ControllerBase
{
}
```

이제 이 컨트롤러의 모든 액션은 `/api/account/` 경로로 시작한다.

### 2.4.4 첫 번째 액션 메서드 작성
간단한 테스트용 액션 메서드를 작성한다.

```csharp
using Microsoft.AspNetCore.Mvc;

namespace GameServer.Controllers
{
    [Route("api/account")]
    [ApiController]
    public class AccountController : ControllerBase
    {
        [HttpPost("test")]
        public TestResponse Test()
        {
            return new TestResponse
            {
                Result = 0,
                Message = "Success",
                Data = new TestResponseData
                {
                    ServerTime = DateTime.Now,
                    Status = "Running"
                }
            };
        }
    }

    public class TestResponse
    {
        public int Result { get; set; }
        public string Message { get; set; } = string.Empty;
        public TestResponseData? Data { get; set; }
    }

    public class TestResponseData
    {
        public DateTime ServerTime { get; set; }
        public string Status { get; set; } = string.Empty;
    }
}
```

**코드 설명**:

- **[HttpPost("test")]**: 이 메서드가 POST 요청을 처리하며, URL 경로는 `/api/account/test`임을 나타낸다.
- **타입이 지정된 응답 클래스**: `IActionResult` 대신 `TestResponse`를 직접 반환해 응답 스키마가 타입으로 고정된다. 클라이언트와의 계약이 코드로 명확해진다.
- **Data 필드**: 응답 본문에 포함할 데이터는 별도 DTO(`TestResponseData`)로 분리해 가독성을 높인다.
- **성공 시 200 OK**: 프레임워크가 객체 반환 시 자동으로 200 OK + JSON 직렬화로 응답한다.

### 2.4.5 요청 데이터 받기
클라이언트로부터 데이터를 받는 방법을 실습한다. 먼저 요청 데이터를 담을 모델 클래스를 생성한다.

`Models` 폴더에 `AccountModels.cs` 파일을 생성하고 다음 코드를 작성한다.

```csharp
namespace GameServer.Models
{
    public class TestRequest
    {
        public string? Message { get; set; }
    }

    public class ApiResponse<T>
    {
        public int Result { get; set; }
        public string Message { get; set; } = string.Empty;
        public T? Data { get; set; }
    }
}
```

컨트롤러에 새로운 액션을 추가한다.

```csharp
using GameServer.Models;
using Microsoft.AspNetCore.Mvc;

namespace GameServer.Controllers
{
    [Route("api/account")]
    [ApiController]
    public class AccountController : ControllerBase
    {
        [HttpPost("test")]
        public TestResponse Test()
        {
            return new TestResponse
            {
                Result = 0,
                Message = "Success",
                Data = new TestResponseData
                {
                    ServerTime = DateTime.Now,
                    Status = "Running"
                }
            };
        }

        [HttpPost("echo")]
        public EchoResponse Echo([FromBody] TestRequest request)
        {
            if (request == null || string.IsNullOrEmpty(request.Message))
            {
                return new EchoResponse
                {
                    Result = 4001,
                    Message = "Invalid request data",
                    Data = null
                };
            }

            return new EchoResponse
            {
                Result = 0,
                Message = "Success",
                Data = new EchoResponseData
                {
                    ReceivedMessage = request.Message,
                    Timestamp = DateTime.Now
                }
            };
        }
    }

    public class EchoResponse
    {
        public int Result { get; set; }
        public string Message { get; set; } = string.Empty;
        public EchoResponseData? Data { get; set; }
    }

    public class EchoResponseData
    {
        public string ReceivedMessage { get; set; } = string.Empty;
        public DateTime Timestamp { get; set; }
    }
}
```

**코드 설명**:

- **[FromBody]**: HTTP 요청 본문에서 데이터를 파싱하여 객체로 변환한다.
- **데이터 검증**: 요청 데이터가 유효한지 확인한다.
- **타입 응답**: `IActionResult` 대신 `EchoResponse`를 반환하도록 해 요청 DTO(`TestRequest`)와 짝을 이루는 응답 DTO를 명확히 했다. 오류는 `Result` 코드로 표현하고 프레임워크가 기본 200 OK로 직렬화한다.
- **BadRequest()**: HTTP 400 상태 코드를 반환한다.
- **제네릭 응답 클래스**: `ApiResponse<T>`를 사용하여 타입 안전성을 확보한다.

## 2.5 JSON 직렬화 설정
ASP.NET Core는 기본적으로 System.Text.Json을 사용하여 JSON 직렬화를 수행한다. 게임 서버에 맞게 몇 가지 설정을 추가한다.

`Program.cs`를 다음과 같이 수정한다.

```csharp
using System.Text.Json;
using System.Text.Json.Serialization;

var builder = WebApplication.CreateBuilder(args);

// Add services to the container.
builder.Services.AddControllers()
    .AddJsonOptions(options =>
    {
        // 속성 이름을 camelCase로 변환 (기본값)
        options.JsonSerializerOptions.PropertyNamingPolicy = JsonNamingPolicy.CamelCase;
        
        // null 값을 가진 속성은 JSON에서 제외
        options.JsonSerializerOptions.DefaultIgnoreCondition = JsonIgnoreCondition.WhenWritingNull;
        
        // 숫자를 문자열로 읽기 허용
        options.JsonSerializerOptions.NumberHandling = JsonNumberHandling.AllowReadingFromString;
    });

var app = builder.Build();

app.UseAuthorization();
app.MapControllers();

app.Run();
```

**설정 설명**:

- **PropertyNamingPolicy.CamelCase**: C# 속성 이름(PascalCase)을 JSON 키 이름(camelCase)으로 자동 변환한다. 예: `Result` → `result`
- **DefaultIgnoreCondition.WhenWritingNull**: null 값을 가진 속성은 JSON에 포함하지 않는다.
- **NumberHandling.AllowReadingFromString**: 클라이언트가 숫자를 문자열로 보내도 허용한다.
