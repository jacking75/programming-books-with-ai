# 8장. 커스텀 토큰 기반 인증 설계

## 8.1 게임 서버 인증 방식 이해

게임 서버에서 인증은 매우 중요한 보안 요소다. 웹 서비스와 달리 게임 서버는 짧은 시간에 수많은 API 요청이 발생하며, 각 요청마다 사용자를 식별하고 인증해야 한다.

### 8.1.1 일반적인 인증 방식

웹 애플리케이션에서는 주로 다음과 같은 인증 방식을 사용한다.

- **세션 기반 인증**: 서버에 세션을 저장하고 클라이언트에게 세션 ID를 쿠키로 전달한다.
- **JWT(JSON Web Token)**: 자체적으로 검증 가능한 토큰을 사용하며, 토큰 내부에 사용자 정보를 포함한다.
- **OAuth 2.0**: 제3자 인증을 위한 표준 프로토콜이다.

### 8.1.2 게임 서버에 적합한 인증 방식

수집형 RPG 게임 서버에서는 다음과 같은 요구사항이 있다.

**성능 요구사항**
- 매 API 요청마다 빠른 인증 처리가 필요하다.
- Redis를 활용한 빠른 토큰 검증이 가능해야 한다.
- DB 조회를 최소화해야 한다.

**보안 요구사항**
- 토큰 탈취 시 피해를 최소화해야 한다.
- 토큰 만료 시간을 관리할 수 있어야 한다.
- 중복 로그인이나 강제 로그아웃을 지원해야 한다.

**구현 요구사항**
- 서버에서 토큰을 생성하고 검증할 수 있어야 한다.
- 토큰을 무효화할 수 있어야 한다.
- 사용자 식별이 간단해야 한다.

이러한 이유로 이 교육에서는 커스텀 토큰 기반 인증 방식을 설계한다. JWT를 사용하지 않는 이유는 게임 서버 특성상 토큰을 서버에서 직접 관리하고 무효화할 수 있어야 하기 때문이다.

## 8.2 토큰 생성 알고리즘 설계

### 8.2.1 HMAC-SHA256 이해

HMAC(Hash-based Message Authentication Code)은 해시 함수를 이용한 메시지 인증 코드다. HMAC-SHA256은 SHA256 해시 함수를 사용하는 HMAC이다.

**HMAC-SHA256의 특징**
- 비밀 키를 사용하여 메시지의 무결성과 인증을 보장한다.
- 같은 입력과 같은 키로는 항상 같은 해시 값을 생성한다.
- 해시 값으로부터 원본 데이터를 복원할 수 없다.
- 비밀 키 없이는 유효한 해시 값을 생성할 수 없다.

### 8.2.2 토큰 생성에 필요한 요소

커스텀 토큰을 생성하기 위해 다음 요소들이 필요하다.

**사용자 식별 정보**
- UserId: 사용자를 고유하게 식별하는 정수 값이다.

**시간 정보**
- Timestamp: 토큰 생성 시각을 나타낸다. Unix timestamp를 사용한다.

**보안 요소**
- Salt: 토큰의 유일성을 보장하기 위한 랜덤 값이다.
- SecretKey: 서버만 알고 있는 비밀 키다.

**해시 값**
- Hash: 위 정보들을 조합하여 HMAC-SHA256으로 생성한 값이다.

### 8.2.3 토큰 생성 과정

토큰은 다음과 같은 과정으로 생성된다.

1. 사용자가 로그인에 성공한다.
2. 서버는 랜덤한 Salt 값을 생성한다.
3. UserId, Timestamp, Salt를 조합한 문자열을 만든다.
4. 조합된 문자열과 SecretKey를 이용해 HMAC-SHA256 해시를 생성한다.
5. 최종 토큰 문자열을 생성한다.

### 8.2.4 토큰 생성 코드 예시

실제 구현은 9장에서 진행하지만, 개념 이해를 위해 간단한 예시를 살펴본다.

```csharp
public class TokenGenerator
{
    private readonly string _secretKey;

    public TokenGenerator(string secretKey)
    {
        _secretKey = secretKey;
    }

    public string GenerateToken(long userId)
    {
        // 1. Salt 생성 (16바이트 랜덤 값)
        var salt = GenerateSalt();
        
        // 2. Timestamp 생성 (Unix timestamp)
        var timestamp = DateTimeOffset.UtcNow.ToUnixTimeSeconds();
        
        // 3. 토큰 데이터 조합
        var tokenData = $"{userId}:{timestamp}:{salt}";
        
        // 4. HMAC-SHA256 해시 생성
        var hash = ComputeHmacSha256(tokenData, _secretKey);
        
        // 5. 최종 토큰 생성 (Base64 인코딩)
        var token = $"{userId}:{timestamp}:{salt}:{hash}";
        return Convert.ToBase64String(Encoding.UTF8.GetBytes(token));
    }

    private string GenerateSalt()
    {
        var saltBytes = new byte[16];
        using (var rng = RandomNumberGenerator.Create())
        {
            rng.GetBytes(saltBytes);
        }
        return Convert.ToBase64String(saltBytes);
    }

    private string ComputeHmacSha256(string data, string key)
    {
        var keyBytes = Encoding.UTF8.GetBytes(key);
        var dataBytes = Encoding.UTF8.GetBytes(data);
        
        using (var hmac = new HMACSHA256(keyBytes))
        {
            var hashBytes = hmac.ComputeHash(dataBytes);
            return Convert.ToBase64String(hashBytes);
        }
    }
}
```

이 코드는 개념 설명을 위한 예시이며, 9장에서 실제 프로젝트에 통합하는 방법을 배운다.

## 8.3 솔트(Salt) 값 생성 및 관리

### 8.3.1 Salt의 필요성

Salt는 보안을 강화하기 위한 랜덤 값이다. Salt가 필요한 이유는 다음과 같다.

**토큰 유일성 보장**
- 같은 사용자가 여러 번 로그인해도 매번 다른 토큰이 생성된다.
- 이전 토큰이 탈취되어도 새로운 로그인으로 무효화할 수 있다.

**레인보우 테이블 공격 방지**
- 미리 계산된 해시 값 테이블을 이용한 공격을 방지한다.
- Salt가 있으면 각 토큰마다 해시 값이 달라진다.

**중복 로그인 관리**
- Salt를 포함한 토큰을 사용하면 로그인 세션을 개별적으로 관리할 수 있다.

### 8.3.2 Salt 생성 방법

Salt는 암호학적으로 안전한 난수 생성기를 사용하여 생성해야 한다.

**.NET의 RandomNumberGenerator 사용**
- `System.Security.Cryptography` 네임스페이스의 `RandomNumberGenerator` 클래스를 사용한다.
- 일반적인 `Random` 클래스는 예측 가능하므로 보안 용도로 사용하지 않는다.

**적절한 Salt 길이**
- 16바이트(128비트) 이상의 길이를 권장한다.
- 이 교육에서는 16바이트를 사용한다.

### 8.3.3 Salt 저장 및 관리

Salt는 토큰의 일부로 포함되어 전달된다. 별도로 DB에 저장할 필요는 없지만, 토큰 검증 시 필요하다.

**토큰 구조에 포함**
- 토큰 문자열 내부에 Salt 값을 포함시킨다.
- 토큰 검증 시 토큰에서 Salt를 추출하여 사용한다.

**보안 고려사항**
- Salt는 공개되어도 괜찮다. 비밀 키인 SecretKey만 안전하게 보관하면 된다.
- 그러나 토큰 전체는 HTTPS를 통해 안전하게 전송되어야 한다.

## 8.4 토큰 구조 설계

### 8.4.1 토큰 구조 결정

이 교육에서 사용할 토큰은 다음과 같은 구조를 가진다.

```
Base64(UserId:Timestamp:Salt:Hash)
```

**각 필드 설명**
- `UserId`: 사용자 고유 식별자 (예: 12345)
- `Timestamp`: Unix timestamp (예: 1700000000)
- `Salt`: Base64로 인코딩된 랜덤 값 (예: aB3dE6fG9hJ1kL4mN7o=)
- `Hash`: HMAC-SHA256으로 생성된 해시 값을 Base64로 인코딩 (예: xY8zA1bC2dE3fG4hI5jK6lM7nO8pQ9rS0tU1vW2xY3z=)

**실제 토큰 예시**
```
MTIzNDU6MTcwMDAwMDAwMDphQjNkRTZmRzloSjFrTDRtTjdvPTp4WTh6QTFiQzJkRTNmRzRoSTVqSzZsTTduTzhwUTlyUzB0VTF2VzJ4WTN6PQ==
```

### 8.4.2 토큰 구조의 장점

이러한 구조는 다음과 같은 장점을 제공한다.

**자체 포함성**
- 토큰 자체에 UserId가 포함되어 있어 별도의 조회 없이 사용자를 식별할 수 있다.
- Timestamp를 통해 토큰 생성 시각을 알 수 있다.

**검증 가능성**
- Hash 값을 통해 토큰이 위조되지 않았음을 검증할 수 있다.
- 서버의 SecretKey 없이는 유효한 Hash를 생성할 수 없다.

**만료 관리**
- Timestamp를 이용해 토큰의 유효 기간을 확인할 수 있다.
- 예를 들어 생성된 지 24시간이 지난 토큰은 만료 처리할 수 있다.

### 8.4.3 토큰 검증 과정

토큰 검증은 생성의 역순으로 진행된다.

1. 클라이언트가 API 요청 시 토큰을 전달한다.
2. 서버는 Base64 디코딩을 수행한다.
3. 디코딩된 문자열을 `:` 기준으로 분리한다.
4. UserId, Timestamp, Salt를 추출한다.
5. 추출된 값들과 SecretKey로 Hash를 재계산한다.
6. 재계산된 Hash와 토큰의 Hash를 비교한다.
7. Hash가 일치하고 Timestamp가 유효하면 인증 성공이다.

### 8.4.4 토큰 검증 코드 예시

```csharp
public class TokenValidator
{
    private readonly string _secretKey;
    private readonly int _tokenExpirationHours;

    public TokenValidator(string secretKey, int tokenExpirationHours = 24)
    {
        _secretKey = secretKey;
        _tokenExpirationHours = tokenExpirationHours;
    }

    public (bool isValid, long userId) ValidateToken(string token)
    {
        try
        {
            // 1. Base64 디코딩
            var decodedBytes = Convert.FromBase64String(token);
            var decodedToken = Encoding.UTF8.GetString(decodedBytes);
            
            // 2. 토큰 분리
            var parts = decodedToken.Split(':');
            if (parts.Length != 4)
            {
                return (false, 0);
            }
            
            var userId = long.Parse(parts[0]);
            var timestamp = long.Parse(parts[1]);
            var salt = parts[2];
            var hash = parts[3];
            
            // 3. 만료 시간 확인
            var tokenCreatedAt = DateTimeOffset.FromUnixTimeSeconds(timestamp);
            var expirationTime = tokenCreatedAt.AddHours(_tokenExpirationHours);
            if (DateTimeOffset.UtcNow > expirationTime)
            {
                return (false, 0);
            }
            
            // 4. Hash 재계산
            var tokenData = $"{userId}:{timestamp}:{salt}";
            var calculatedHash = ComputeHmacSha256(tokenData, _secretKey);
            
            // 5. Hash 비교
            if (hash != calculatedHash)
            {
                return (false, 0);
            }
            
            return (true, userId);
        }
        catch
        {
            return (false, 0);
        }
    }

    private string ComputeHmacSha256(string data, string key)
    {
        var keyBytes = Encoding.UTF8.GetBytes(key);
        var dataBytes = Encoding.UTF8.GetBytes(data);
        
        using (var hmac = new HMACSHA256(keyBytes))
        {
            var hashBytes = hmac.ComputeHash(dataBytes);
            return Convert.ToBase64String(hashBytes);
        }
    }
}
```

## 8.5 토큰 저장 전략

### 8.5.1 저장 위치 결정

생성된 토큰은 다음 두 곳에 저장된다.

**MySQL 데이터베이스**
- 영구적인 저장소로 사용된다.
- 서버 재시작 후에도 토큰 정보가 유지된다.
- Redis 캐시 미스 시 조회된다.

**Redis 캐시**
- 빠른 토큰 검증을 위해 사용된다.
- 대부분의 인증 요청은 Redis에서 처리된다.
- TTL(Time To Live)을 설정하여 자동으로 만료 처리할 수 있다.

### 8.5.2 MySQL 테이블 설계

토큰 정보를 저장할 MySQL 테이블은 다음과 같이 설계한다.

```sql
CREATE TABLE user_auth_token (
    user_id BIGINT NOT NULL,
    token VARCHAR(500) NOT NULL,
    created_at BIGINT NOT NULL,
    expired_at BIGINT NOT NULL,
    PRIMARY KEY (user_id),
    INDEX idx_token (token),
    INDEX idx_expired_at (expired_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**컬럼 설명**
- `user_id`: 사용자 식별자. Primary Key로 설정하여 한 사용자당 하나의 유효한 토큰만 존재한다.
- `token`: 생성된 토큰 문자열이다.
- `created_at`: 토큰 생성 시각 (Unix timestamp)이다.
- `expired_at`: 토큰 만료 시각 (Unix timestamp)이다.

**인덱스 설정 이유**
- `idx_token`: 토큰으로 사용자를 검색할 때 사용된다.
- `idx_expired_at`: 만료된 토큰을 정리할 때 사용된다.

### 8.5.3 Redis 저장 구조

Redis에는 다음과 같은 형태로 토큰을 저장한다.

**Key 형식**
```
auth:token:{userId}
```

**Value**
- 토큰 문자열 자체를 저장한다.

**TTL 설정**
- 토큰의 만료 시간과 동일하게 설정한다.
- 예를 들어 24시간 후 만료되는 토큰이면 TTL을 24시간으로 설정한다.

**저장 예시**
```
Key: auth:token:12345
Value: MTIzNDU6MTcwMDAwMDAwMDphQjNkRTZmRzloSjFrTDRtTjdvPTp4WTh6QTFiQzJkRTNmRzRoSTVqSzZsTTduTzhwUTlyUzB0VTF2VzJ4WTN6PQ==
TTL: 86400 seconds (24 hours)
```

### 8.5.4 이중 저장의 장점

MySQL과 Redis에 동시에 저장하는 이유는 다음과 같다.

**성능 향상**
- 대부분의 인증 요청은 Redis에서 빠르게 처리된다.
- DB 부하를 크게 줄일 수 있다.

**안정성 확보**
- Redis 장애 시 MySQL에서 토큰을 조회할 수 있다.
- 서버 재시작 후 Redis 캐시를 재구성할 수 있다.

**관리 용이성**
- MySQL에 저장된 데이터로 로그인 이력을 추적할 수 있다.
- 만료된 토큰을 배치 작업으로 정리할 수 있다.

## 8.6 보안 고려사항

### 8.6.1 SecretKey 관리

SecretKey는 토큰 보안의 핵심이다. 다음 사항을 반드시 지켜야 한다.

**안전한 저장**
- 코드에 하드코딩하지 않는다.
- appsettings.json에 직접 저장하지 않는다.
- 환경 변수나 User Secrets를 사용한다.

**충분한 길이**
- 최소 32바이트(256비트) 이상의 길이를 사용한다.
- 랜덤하게 생성된 값을 사용한다.

**주기적 교체**
- 일정 기간마다 SecretKey를 교체하는 정책을 수립한다.
- 교체 시 기존 토큰의 호환성을 고려한다.

### 8.6.2 HTTPS 사용

토큰은 반드시 HTTPS를 통해 전송되어야 한다.

**HTTP의 위험성**
- 평문으로 전송되어 중간에 가로채질 수 있다.
- 토큰이 탈취되면 해당 사용자로 위장할 수 있다.

**HTTPS 적용**
- 개발 환경에서도 HTTPS를 사용하는 습관을 들인다.
- 운영 환경에서는 필수적으로 HTTPS를 적용한다.

### 8.6.3 토큰 만료 시간

적절한 토큰 만료 시간을 설정해야 한다.

**너무 긴 만료 시간**
- 토큰 탈취 시 오랜 기간 악용될 수 있다.
- 보안 위험이 증가한다.

**너무 짧은 만료 시간**
- 사용자가 자주 로그인해야 하므로 불편하다.
- 서버 부하가 증가한다.

**권장 설정**
- 모바일 게임의 경우 24시간 ~ 7일 정도가 적절하다.
- 게임의 특성과 보안 요구사항을 고려하여 결정한다.

### 8.6.4 중복 로그인 방지

한 계정에 여러 기기에서 동시 로그인하는 것을 제한할 수 있다.

**Primary Key 활용**
- MySQL 테이블의 user_id를 Primary Key로 설정한다.
- 새로운 로그인 시 기존 토큰을 덮어쓴다.

**이전 토큰 무효화**
- 새로운 토큰이 생성되면 이전 토큰은 자동으로 무효화된다.
- Redis에서도 이전 토큰을 삭제한다.

**선택적 허용**
- 게임 정책에 따라 다중 기기 로그인을 허용할 수도 있다.
- 이 경우 테이블 구조를 변경하여 여러 토큰을 저장한다.
