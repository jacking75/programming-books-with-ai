# 4장. MySQL 데이터베이스 설계
이 장에서는 수집형 RPG 게임 서버에 필요한 데이터베이스 구조를 설계한다. 사용자, 캐릭터, 아이템, 인벤토리 등 게임의 핵심 데이터를 저장할 테이블을 설계하고, MySQL을 설치하여 실제 데이터베이스를 생성한다. 데이터베이스 설계는 게임 서버의 기초이므로 신중하게 진행해야 한다.

## 4.1 수집형 RPG 게임의 데이터 구조 이해
데이터베이스 설계를 시작하기 전에 수집형 RPG 게임의 특징과 필요한 데이터를 파악한다.

### 4.1.1 수집형 RPG 게임의 특징
수집형 RPG 게임은 다음과 같은 특징을 가진다.

**캐릭터 수집**: 플레이어는 다양한 캐릭터를 수집하고 육성한다. 각 캐릭터는 고유한 스탯과 능력을 가진다.

**아이템 시스템**: 무기, 방어구, 소비 아이템 등 다양한 아이템을 획득하고 관리한다.

**던전 및 스테이지**: 여러 던전과 스테이지를 클리어하며 보상을 획득한다.

**재화 시스템**: 골드, 다이아몬드 등 여러 종류의 게임 화폐를 사용한다.

**소셜 기능**: 친구 추가, 선물 보내기 등의 소셜 기능이 있다.

**랭킹 시스템**: 플레이어 간 경쟁과 순위를 기록한다.

### 4.1.2 필요한 주요 데이터 카테고리
게임 서버에 필요한 데이터를 카테고리별로 분류한다.

**계정 데이터**: 사용자 계정 정보, 인증 토큰

**캐릭터 데이터**: 플레이어가 보유한 캐릭터, 캐릭터 스탯

**아이템 데이터**: 아이템 마스터 데이터, 인벤토리

**게임 진행 데이터**: 던전 클리어 기록, 퀘스트 진행 상황

**재화 데이터**: 골드, 다이아몬드 등의 보유량

**소셜 데이터**: 친구 관계, 우편함

**설정 데이터**: 게임 설정, 튜토리얼 진행 상황

### 4.1.3 데이터베이스 설계 원칙
게임 서버 데이터베이스를 설계할 때 다음 원칙을 따른다.

**정규화와 비정규화의 균형**: 데이터 일관성을 위해 적절히 정규화하되, 성능을 위해 필요한 경우 의도적으로 비정규화한다.

**확장 가능한 구조**: 향후 추가될 기능을 고려하여 유연한 구조로 설계한다.

**인덱스 전략**: 자주 조회되는 컬럼에 인덱스를 설정하여 성능을 최적화한다.

**데이터 타입 선택**: 적절한 데이터 타입을 선택하여 저장 공간을 효율적으로 사용한다.

**명명 규칙 통일**: 일관된 테이블명, 컬럼명 규칙을 사용한다.
  

## 4.2 사용자 테이블 설계
게임의 가장 기본이 되는 사용자(계정) 테이블을 설계한다.

### 4.2.1 사용자 테이블 요구사항
사용자 테이블은 다음 정보를 저장한다.

- 사용자 고유 식별자
- 이메일 (로그인 ID)
- 암호화된 비밀번호
- 닉네임
- 레벨 및 경험치
- 보유 재화 (골드, 다이아몬드)
- 계정 생성 일시
- 최근 로그인 일시
- 계정 상태 (정상, 정지 등)

### 4.2.2 users 테이블 스키마

```sql
CREATE TABLE users (
    user_id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '사용자 고유 ID',
    email VARCHAR(100) NOT NULL UNIQUE COMMENT '이메일 (로그인 ID)',
    hashed_password VARCHAR(255) NOT NULL COMMENT '암호화된 비밀번호',
    nickname VARCHAR(20) NOT NULL UNIQUE COMMENT '닉네임',
    level INT NOT NULL DEFAULT 1 COMMENT '사용자 레벨',
    experience BIGINT NOT NULL DEFAULT 0 COMMENT '경험치',
    gold BIGINT NOT NULL DEFAULT 10000 COMMENT '보유 골드',
    diamond INT NOT NULL DEFAULT 100 COMMENT '보유 다이아몬드',
    account_status ENUM('active', 'suspended', 'deleted') NOT NULL DEFAULT 'active' COMMENT '계정 상태',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '생성 일시',
    last_login_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '최근 로그인 일시',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시',
    INDEX idx_email (email),
    INDEX idx_nickname (nickname),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='사용자 계정 정보';
```

**스키마 설명**:

**user_id**: AUTO_INCREMENT를 사용한 자동 증가 정수 키다. 프라이머리 키로 사용한다.

**email**: 로그인에 사용되므로 UNIQUE 제약조건을 설정한다. 인덱스가 자동 생성된다.

**hashed_password**: BCrypt 해시는 최대 60자이지만 여유를 두어 255자로 설정한다.

**nickname**: 중복을 허용하지 않으므로 UNIQUE 제약조건을 설정한다.

**account_status**: ENUM 타입으로 제한된 값만 저장할 수 있다.

**created_at, updated_at**: 타임스탬프 자동 관리를 위해 DEFAULT와 ON UPDATE를 설정한다.

**인덱스**: 자주 검색되는 email, nickname에 인덱스를 추가한다.

**ENGINE=InnoDB**: 트랜잭션을 지원하는 InnoDB 엔진을 사용한다.

**CHARSET=utf8mb4**: 이모지를 포함한 모든 유니코드를 저장할 수 있다.
  

## 4.3 인증 토큰 관리 테이블 설계
로그인 후 발급되는 인증 토큰을 관리하는 테이블을 설계한다.

### 4.3.1 토큰 테이블 요구사항

- 사용자별 토큰 저장
- 토큰 생성 시간
- 토큰 만료 시간
- 디바이스 정보 (다중 디바이스 지원)
- 토큰 무효화 기능

### 4.3.2 user_tokens 테이블 스키마

```sql
CREATE TABLE user_tokens (
    token_id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '토큰 고유 ID',
    user_id BIGINT NOT NULL COMMENT '사용자 ID',
    token VARCHAR(255) NOT NULL UNIQUE COMMENT '인증 토큰',
    device_id VARCHAR(100) DEFAULT NULL COMMENT '디바이스 식별자',
    device_type ENUM('ios', 'android', 'web') DEFAULT NULL COMMENT '디바이스 타입',
    is_valid BOOLEAN NOT NULL DEFAULT TRUE COMMENT '토큰 유효 여부',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '생성 일시',
    expires_at DATETIME NOT NULL COMMENT '만료 일시',
    last_accessed_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '마지막 접속 일시',
    INDEX idx_user_id (user_id),
    INDEX idx_token (token),
    INDEX idx_expires_at (expires_at),
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='사용자 인증 토큰';
```

**스키마 설명**:

**token**: UNIQUE 제약조건으로 중복 토큰을 방지한다.

**is_valid**: 로그아웃 시 FALSE로 변경하여 토큰을 무효화한다.

**expires_at**: 토큰 만료 시간을 저장한다. 만료된 토큰은 자동으로 무효화된다.

**FOREIGN KEY**: users 테이블과 관계를 설정한다. CASCADE 옵션으로 사용자 삭제 시 토큰도 함께 삭제된다.
  

## 4.4 캐릭터 관련 테이블 설계
게임의 핵심인 캐릭터 시스템을 위한 테이블을 설계한다.

### 4.4.1 캐릭터 마스터 테이블
캐릭터의 기본 정보를 담는 마스터 데이터 테이블이다. 게임 기획 데이터이며 운영자만 수정할 수 있다.

```sql
CREATE TABLE character_master (
    character_id INT PRIMARY KEY COMMENT '캐릭터 마스터 ID',
    character_name VARCHAR(50) NOT NULL COMMENT '캐릭터 이름',
    character_class ENUM('warrior', 'mage', 'archer', 'priest') NOT NULL COMMENT '캐릭터 직업',
    grade INT NOT NULL COMMENT '등급 (1-5성)',
    base_hp INT NOT NULL COMMENT '기본 HP',
    base_attack INT NOT NULL COMMENT '기본 공격력',
    base_defense INT NOT NULL COMMENT '기본 방어력',
    growth_rate DECIMAL(3,2) NOT NULL DEFAULT 1.00 COMMENT '성장률 (1.00 = 100%)',
    description TEXT COMMENT '캐릭터 설명',
    icon_url VARCHAR(255) COMMENT '캐릭터 아이콘 URL',
    is_available BOOLEAN NOT NULL DEFAULT TRUE COMMENT '획득 가능 여부',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '생성 일시'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='캐릭터 마스터 데이터';
```

### 4.4.2 사용자 보유 캐릭터 테이블
플레이어가 실제로 보유한 캐릭터 인스턴스를 저장하는 테이블이다.

```sql
CREATE TABLE user_characters (
    user_character_id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '사용자 캐릭터 고유 ID',
    user_id BIGINT NOT NULL COMMENT '사용자 ID',
    character_id INT NOT NULL COMMENT '캐릭터 마스터 ID',
    level INT NOT NULL DEFAULT 1 COMMENT '캐릭터 레벨',
    experience BIGINT NOT NULL DEFAULT 0 COMMENT '경험치',
    enhancement_level INT NOT NULL DEFAULT 0 COMMENT '강화 레벨',
    current_hp INT NOT NULL COMMENT '현재 HP',
    current_attack INT NOT NULL COMMENT '현재 공격력',
    current_defense INT NOT NULL COMMENT '현재 방어력',
    is_main BOOLEAN NOT NULL DEFAULT FALSE COMMENT '메인 캐릭터 여부',
    acquired_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '획득 일시',
    INDEX idx_user_id (user_id),
    INDEX idx_character_id (character_id),
    INDEX idx_user_char (user_id, character_id),
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    FOREIGN KEY (character_id) REFERENCES character_master(character_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='사용자 보유 캐릭터';
```

**스키마 설명**:

**user_character_id**: 같은 캐릭터를 여러 개 보유할 수 있으므로 별도의 고유 ID가 필요하다.

**current_hp, current_attack, current_defense**: 레벨업과 강화에 따라 변하는 실제 스탯이다.

**is_main**: 대표 캐릭터를 표시한다. 애플리케이션 레벨에서 사용자당 하나만 TRUE가 되도록 제어한다.

**복합 인덱스**: user_id와 character_id를 함께 자주 조회하므로 복합 인덱스를 추가한다.
  

## 4.5 아이템 및 인벤토리 테이블 설계
아이템 시스템을 위한 테이블을 설계한다.

### 4.5.1 아이템 마스터 테이블
아이템의 기본 정보를 담는 마스터 데이터 테이블이다.

```sql
CREATE TABLE item_master (
    item_id INT PRIMARY KEY COMMENT '아이템 마스터 ID',
    item_name VARCHAR(100) NOT NULL COMMENT '아이템 이름',
    item_type ENUM('weapon', 'armor', 'accessory', 'consumable', 'material') NOT NULL COMMENT '아이템 타입',
    grade INT NOT NULL COMMENT '등급 (1-5)',
    max_stack INT NOT NULL DEFAULT 1 COMMENT '최대 중첩 수량',
    buy_price INT DEFAULT NULL COMMENT '구매 가격',
    sell_price INT DEFAULT NULL COMMENT '판매 가격',
    effect_type VARCHAR(50) DEFAULT NULL COMMENT '효과 타입',
    effect_value INT DEFAULT NULL COMMENT '효과 수치',
    description TEXT COMMENT '아이템 설명',
    icon_url VARCHAR(255) COMMENT '아이템 아이콘 URL',
    is_tradable BOOLEAN NOT NULL DEFAULT FALSE COMMENT '거래 가능 여부',
    is_available BOOLEAN NOT NULL DEFAULT TRUE COMMENT '획득 가능 여부',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '생성 일시'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='아이템 마스터 데이터';
```

### 4.5.2 사용자 인벤토리 테이블
플레이어가 보유한 아이템을 저장하는 테이블이다.

```sql
CREATE TABLE user_inventory (
    inventory_id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '인벤토리 고유 ID',
    user_id BIGINT NOT NULL COMMENT '사용자 ID',
    item_id INT NOT NULL COMMENT '아이템 마스터 ID',
    quantity INT NOT NULL DEFAULT 1 COMMENT '보유 수량',
    is_equipped BOOLEAN NOT NULL DEFAULT FALSE COMMENT '장착 여부',
    enhancement_level INT NOT NULL DEFAULT 0 COMMENT '강화 레벨 (장비 아이템)',
    acquired_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '획득 일시',
    INDEX idx_user_id (user_id),
    INDEX idx_item_id (item_id),
    INDEX idx_user_item (user_id, item_id),
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    FOREIGN KEY (item_id) REFERENCES item_master(item_id),
    UNIQUE KEY uk_user_item (user_id, item_id, enhancement_level)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='사용자 인벤토리';
```

**스키마 설명**:

**UNIQUE KEY**: 같은 사용자가 동일한 아이템(강화 레벨 포함)을 중복 보유하지 않도록 한다. 수량은 quantity로 관리한다.

**is_equipped**: 장비 아이템의 장착 상태를 표시한다.

**enhancement_level**: 장비 아이템의 강화 레벨이다. 같은 아이템도 강화 레벨이 다르면 별도의 행으로 저장된다.
  

## 4.6 던전 및 게임 진행 테이블 설계
게임 진행 상황을 기록하는 테이블을 설계한다.

### 4.6.1 던전 마스터 테이블

```sql
CREATE TABLE dungeon_master (
    dungeon_id INT PRIMARY KEY COMMENT '던전 마스터 ID',
    dungeon_name VARCHAR(100) NOT NULL COMMENT '던전 이름',
    difficulty ENUM('easy', 'normal', 'hard', 'hell') NOT NULL COMMENT '난이도',
    required_level INT NOT NULL DEFAULT 1 COMMENT '입장 필요 레벨',
    entry_cost INT NOT NULL DEFAULT 0 COMMENT '입장 비용 (골드)',
    max_attempts_per_day INT NOT NULL DEFAULT 10 COMMENT '일일 입장 제한',
    reward_gold INT NOT NULL DEFAULT 0 COMMENT '클리어 보상 골드',
    reward_experience INT NOT NULL DEFAULT 0 COMMENT '클리어 보상 경험치',
    description TEXT COMMENT '던전 설명',
    is_available BOOLEAN NOT NULL DEFAULT TRUE COMMENT '개방 여부',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '생성 일시'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='던전 마스터 데이터';
```

### 4.6.2 사용자 던전 클리어 기록 테이블

```sql
CREATE TABLE user_dungeon_records (
    record_id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '기록 고유 ID',
    user_id BIGINT NOT NULL COMMENT '사용자 ID',
    dungeon_id INT NOT NULL COMMENT '던전 마스터 ID',
    clear_count INT NOT NULL DEFAULT 0 COMMENT '총 클리어 횟수',
    best_clear_time INT DEFAULT NULL COMMENT '최고 클리어 시간 (초)',
    last_cleared_at DATETIME DEFAULT NULL COMMENT '마지막 클리어 일시',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '최초 도전 일시',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시',
    INDEX idx_user_id (user_id),
    INDEX idx_dungeon_id (dungeon_id),
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    FOREIGN KEY (dungeon_id) REFERENCES dungeon_master(dungeon_id),
    UNIQUE KEY uk_user_dungeon (user_id, dungeon_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='사용자 던전 클리어 기록';
```

### 4.6.3 일일 던전 입장 기록 테이블
일일 입장 제한을 관리하기 위한 테이블이다.

```sql
CREATE TABLE user_dungeon_daily_entries (
    entry_id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '입장 기록 ID',
    user_id BIGINT NOT NULL COMMENT '사용자 ID',
    dungeon_id INT NOT NULL COMMENT '던전 마스터 ID',
    entry_date DATE NOT NULL COMMENT '입장 날짜',
    entry_count INT NOT NULL DEFAULT 1 COMMENT '입장 횟수',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '생성 일시',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시',
    INDEX idx_user_date (user_id, entry_date),
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    FOREIGN KEY (dungeon_id) REFERENCES dungeon_master(dungeon_id),
    UNIQUE KEY uk_user_dungeon_date (user_id, dungeon_id, entry_date)
```

## 4.8 ERD 작성
설계한 테이블들의 관계를 시각화한 ERD(Entity Relationship Diagram)를 작성한다.

### 4.8.1 주요 엔티티와 관계

**users (1) --- (N) user_tokens**: 한 사용자는 여러 토큰을 가질 수 있다 (다중 디바이스)

**users (1) --- (N) user_characters**: 한 사용자는 여러 캐릭터를 보유한다

**character_master (1) --- (N) user_characters**: 마스터 데이터 하나에 여러 사용자 캐릭터가 연결된다

**users (1) --- (N) user_inventory**: 한 사용자는 여러 아이템을 보유한다

**item_master (1) --- (N) user_inventory**: 마스터 데이터 하나에 여러 사용자 인벤토리가 연결된다

**users (1) --- (N) user_dungeon_records**: 한 사용자는 여러 던전 기록을 가진다

**dungeon_master (1) --- (N) user_dungeon_records**: 한 던전은 여러 사용자의 기록을 가진다

**users (1) --- (N) user_mail**: 한 사용자는 여러 우편을 받는다

**user_mail (1) --- (N) user_mail_items**: 한 우편에 여러 아이템이 첨부될 수 있다

**users (N) --- (N) users**: 사용자 간 친구 관계 (user_friends를 통한 다대다 관계)

### 4.8.2 ERD 다이어그램
텍스트로 ERD를 표현하면 다음과 같다.

```
users
  |
  +--- user_tokens
  |
  +--- user_characters --- character_master
  |
  +--- user_inventory --- item_master
  |
  +--- user_dungeon_records --- dungeon_master
  |     |
  |     +--- user_dungeon_daily_entries
  |
  +--- user_mail (receiver)
  |     |
  |     +--- user_mail_items --- item_master
  |
  +--- user_friends (양방향)
```

실무에서는 MySQL Workbench, dbdiagram.io, ERDCloud 등의 도구를 사용하여 시각적인 ERD를 작성한다.
  

## 4.9 MySQL 설치
이제 설계한 데이터베이스를 실제로 구축하기 위해 MySQL을 설치한다.

### 4.9.1 MySQL 다운로드

1. 웹 브라우저에서 https://dev.mysql.com/downloads/mysql/ 에 접속한다.
2. 운영체제에 맞는 MySQL Community Server를 다운로드한다.
3. Windows의 경우 'MySQL Installer'를 사용하면 편리하다.

### 4.9.2 MySQL 설치 (Windows)

1. 다운로드한 MySQL Installer를 실행한다.
2. 'Developer Default' 또는 'Server only'를 선택한다.
3. 'Next'를 클릭하여 설치를 진행한다.
4. MySQL Server Configuration 단계에서 다음을 설정한다:
   - Config Type: Development Computer
   - Port: 3306 (기본값)
   - Root Password: 강력한 비밀번호 설정 (예: MyS3cur3P@ss!)
5. 설치를 완료한다.

### 4.9.3 MySQL Workbench 설치 확인
MySQL Installer를 사용했다면 MySQL Workbench도 함께 설치된다. MySQL Workbench는 데이터베이스를 관리하는 GUI 도구다.

1. MySQL Workbench를 실행한다.
2. 'Local instance MySQL'을 더블클릭한다.
3. Root 비밀번호를 입력하여 접속한다.

### 4.9.4 명령줄에서 MySQL 접속 확인

1. 명령 프롬프트 또는 PowerShell을 연다.
2. 다음 명령을 실행한다:

```bash
mysql -u root -p
```

3. 비밀번호를 입력하면 MySQL 프롬프트가 표시된다.

```
Welcome to the MySQL monitor.
mysql>
```

4. 종료하려면 `exit`를 입력한다.
  

## 4.10 데이터베이스 생성
게임 서버용 데이터베이스를 생성하고 테이블을 만든다.

### 4.10.1 데이터베이스 생성
MySQL Workbench 또는 명령줄에서 다음 SQL을 실행한다.

```sql
-- 데이터베이스 생성
CREATE DATABASE IF NOT EXISTS game_db
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;

-- 데이터베이스 선택
USE game_db;
```

**CHARACTER SET utf8mb4**: 이모지를 포함한 모든 유니코드 문자를 저장할 수 있다.

**COLLATE utf8mb4_unicode_ci**: 대소문자 구분 없이(case-insensitive) 정렬 및 비교를 수행한다.

### 4.10.2 테이블 생성 스크립트
앞서 설계한 모든 테이블을 생성하는 스크립트를 작성한다. 프로젝트 루트에 `SQL` 폴더를 생성하고 `01_create_tables.sql` 파일을 만든다.

```sql
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시',

    INDEX idx_nickname (nickname),

) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='사용자 우편함';

-- 11. 우편 첨부 아이템 테이블
CREATE TABLE user_mail_items (
    mail_item_id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '우편 아이템 ID',
    mail_id BIGINT NOT NULL COMMENT '우편 ID',
    item_id INT NOT NULL COMMENT '아이템 마스터 ID',
    quantity INT NOT NULL DEFAULT 1 COMMENT '수량',
    INDEX idx_mail_id (mail_id),
    FOREIGN KEY (mail_id) REFERENCES user_mail(mail_id) ON DELETE CASCADE,
    FOREIGN KEY (item_id) REFERENCES item_master(item_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='우편 첨부 아이템';

-- 12. 친구 관계 테이블
CREATE TABLE user_friends (
    friend_id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '친구 관계 ID',
    user_id BIGINT NOT NULL COMMENT '사용자 ID',
    friend_user_id BIGINT NOT NULL COMMENT '친구 사용자 ID',
    status ENUM('pending', 'accepted', 'blocked') NOT NULL DEFAULT 'pending' COMMENT '친구 상태',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '요청 일시',
    accepted_at DATETIME DEFAULT NULL COMMENT '승인 일시',
    INDEX idx_user_id (user_id),
    INDEX idx_friend_user_id (friend_user_id),
    INDEX idx_status (status),
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    FOREIGN KEY (friend_user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    UNIQUE KEY uk_user_friend (user_id, friend_user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='사용자 친구 관계';
```

## 4.12 데이터베이스 설계 정리
이 장에서 완성한 데이터베이스 구조는 다음과 같은 특징을 가진다.

**계층적 구조**: 마스터 데이터와 사용자 데이터를 분리하여 관리한다.

**관계 무결성**: 외래 키를 사용하여 데이터 간 관계를 보장한다.

**확장 가능성**: 새로운 기능 추가 시 테이블을 쉽게 추가할 수 있는 구조다.

**성능 최적화**: 적절한 인덱스를 설정하여 조회 성능을 향상시킨다.

**데이터 타입 최적화**: 각 컬럼에 적합한 데이터 타입을 선택하여 저장 공간을 효율적으로 사용한다.

다음 장에서는 ASP.NET Web API에서 MySQL 데이터베이스에 연결하고, SqlKata를 사용하여 쿼리를 작성하는 방법을 학습한다.
