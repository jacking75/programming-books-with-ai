# 24장. 설정 관리 및 환경 분리

## 24.1 개요

게임 서버를 개발하고 운영하는 과정에서 개발, 스테이징, 운영 환경은 각각 다른 설정을 필요로 한다. 개발 환경에서는 로컬 데이터베이스를 사용하지만, 운영 환경에서는 실제 서비스용 데이터베이스를 사용한다. 로그 레벨이나 캐시 정책도 환경에 따라 달라질 수 있다.

ASP.NET Core는 환경별로 설정을 분리하고 관리할 수 있는 강력한 기능을 제공한다. 이번 장에서는 appsettings.json을 활용한 설정 관리와 환경 분리, 민감한 정보를 안전하게 관리하는 방법을 학습한다.

설정 관리의 주요 목표는 다음과 같다.

**환경 분리**: 개발, 스테이징, 운영 환경을 명확히 구분한다.

**보안**: 비밀번호, API 키 같은 민감한 정보를 안전하게 관리한다.

**유연성**: 코드 수정 없이 설정만 변경하여 동작을 제어한다.

**유지보수성**: 설정을 중앙에서 관리하여 변경이 용이하다.

## 24.6 환경 변수를 통한 설정 오버라이드

환경 변수를 사용하여 설정을 오버라이드할 수 있다. 이 방법은 컨테이너 환경이나 클라우드 배포 시 유용하다.

환경 변수 명명 규칙은 섹션을 `__`(언더스코어 2개)로 구분한다.

```bash
# Windows
set Database__MySQL__ConnectionString=Server=prod-db.example.com;...
set Redis__ConnectionString=prod-redis.example.com:6379
set Authentication__TokenSecretKey=production-secret-key

# Linux/Mac
export Database__MySQL__ConnectionString="Server=prod-db.example.com;..."
export Redis__ConnectionString="prod-redis.example.com:6379"
export Authentication__TokenSecretKey="production-secret-key"
```

Docker Compose 예제

```yaml
# docker-compose.yml
version: '3.8'
services:
  game-api:
    image: game-api-server:latest
    environment:
      - ASPNETCORE_ENVIRONMENT=Production
      - Database__MySQL__ConnectionString=Server=mysql;Port=3306;Database=game_db;Uid=game_user;Pwd=game_password;
      - Redis__ConnectionString=redis:6379
      - Authentication__TokenSecretKey=${TOKEN_SECRET_KEY}
    ports:
      - "5000:80"
    depends_on:
      - mysql
      - redis
  
  mysql:
    image: mysql:8.0
    environment:
      - MYSQL_ROOT_PASSWORD=root_password
      - MYSQL_DATABASE=game_db
      - MYSQL_USER=game_user
      - MYSQL_PASSWORD=game_password
    volumes:
      - mysql_data:/var/lib/mysql
  
  redis:
    image: redis:7-alpine
    volumes:
      - redis_data:/data

volumes:
  mysql_data:
  redis_data:
```

## 24.7 User Secrets로 민감 정보 관리

개발 환경에서 민감한 정보를 코드 저장소에 커밋하지 않기 위해 User Secrets를 사용한다.

User Secrets 초기화

```bash
# 프로젝트 디렉터리에서 실행
dotnet user-secrets init
```

이 명령은 프로젝트 파일(.csproj)에 UserSecretsId를 추가한다.

```xml
<Project Sdk="Microsoft.NET.Sdk.Web">
  <PropertyGroup>
    <TargetFramework>net9.0</TargetFramework>
    <UserSecretsId>your-unique-guid-here</UserSecretsId>
  </PropertyGroup>
</Project>
```

User Secrets에 민감 정보 추가

```bash
# 데이터베이스 연결 문자열
dotnet user-secrets set "Database:MySQL:ConnectionString" "Server=localhost;Port=3306;Database=game_db_dev;Uid=dev_user;Pwd=dev_password123;"

# Redis 연결 문자열
dotnet user-secrets set "Redis:ConnectionString" "localhost:6379,password=redis_password123"

# 인증 시크릿 키
dotnet user-secrets set "Authentication:TokenSecretKey" "development-super-secret-key-do-not-commit"

# 외부 API 키
dotnet user-secrets set "ExternalAPI:GooglePlayStoreKey" "your-google-api-key"
dotnet user-secrets set "ExternalAPI:AppleAppStoreKey" "your-apple-api-key"
```

User Secrets 목록 확인

```bash
dotnet user-secrets list
```

특정 시크릿 제거

```bash
dotnet user-secrets remove "Authentication:TokenSecretKey"
```

모든 시크릿 제거

```bash
dotnet user-secrets clear
```

Visual Studio에서 User Secrets 관리

Visual Studio 2022에서는 프로젝트를 우클릭하고 "Manage User Secrets"를 선택하면 secrets.json 파일이 열린다.

## 24.9 점검 모드 구현

게임 서버 점검 시 사용할 점검 모드를 구현한다.

```csharp
// Middlewares/MaintenanceMiddleware.cs
using Microsoft.Extensions.Options;
using GameAPIServer.Configuration;

namespace GameAPIServer.Middlewares;

public class MaintenanceMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<MaintenanceMiddleware> _logger;
    
    public MaintenanceMiddleware(
        RequestDelegate next,
        ILogger<MaintenanceMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }
    
    public async Task InvokeAsync(
        HttpContext context,
        IOptionsMonitor<MaintenanceSettings> maintenanceSettings)
    {
        var settings = maintenanceSettings.CurrentValue;
        
        // 점검 모드 확인
        if (settings.IsUnderMaintenance)
        {
            // 특정 IP는 점검 중에도 접속 허용 (관리자용)
            var allowedIPs = new[] { "127.0.0.1", "::1" };
            var clientIP = context.Connection.RemoteIpAddress?.ToString();
            
            if (!allowedIPs.Contains(clientIP))
            {
                _logger.LogWarning("점검 중 접속 시도: IP: {IP}", clientIP);
                
                context.Response.StatusCode = 503;
                context.Response.ContentType = "application/json";
                
                await context.Response.WriteAsJsonAsync(new
                {
                    errorCode = "MAINTENANCE",
                    message = settings.MaintenanceMessage
                });
                
                return;
            }
        }
        
        await _next(context);
    }
}

// Program.cs에 등록
app.UseMiddleware<MaintenanceMiddleware>();
```

점검 모드 설정 API

```csharp
// Controllers/AdminController.cs
using Microsoft.AspNetCore.Mvc;
using Microsoft.Extensions.Options;
using GameAPIServer.Configuration;

namespace GameAPIServer.Controllers;

[ApiController]
[Route("api/[controller]")]
public class AdminController : ControllerBase
{
    private readonly IOptionsMonitor<MaintenanceSettings> _maintenanceSettings;
    private readonly ILogger<AdminController> _logger;
    
    public AdminController(
        IOptionsMonitor<MaintenanceSettings> maintenanceSettings,
        ILogger<AdminController> logger)
    {
        _maintenanceSettings = maintenanceSettings;
        _logger = logger;
    }
    
    [HttpPost("maintenance/start")]
    public StartMaintenanceResponse StartMaintenance([FromBody] StartMaintenanceRequest request)
    {
        // 실제로는 관리자 인증 필요
        
        // 설정 파일을 동적으로 수정하는 것은 권장하지 않음
        // 대신 Redis나 데이터베이스에 플래그를 저장하고 참조
        
        _logger.LogWarning("점검 모드 시작: Message: {Message}", request.Message);
        
        return new StartMaintenanceResponse
        {
            Result = ErrorCode.None,
            Message = "점검 모드가 시작되었다"
        };
    }
    
    [HttpPost("maintenance/end")]
    public EndMaintenanceResponse EndMaintenance()
    {
        _logger.LogInformation("점검 모드 종료");
        
        return new EndMaintenanceResponse
        {
            Result = ErrorCode.None,
            Message = "점검 모드가 종료되었다"
        };
    }
}
```

## 24.10 설정 변경 감지

런타임에 설정 변경을 감지하고 반영한다.

```csharp
// Services/ConfigurationMonitorService.cs
using Microsoft.Extensions.Options;
using GameAPIServer.Configuration;

namespace GameAPIServer.Services;

public class ConfigurationMonitorService : BackgroundService
{
    private readonly IOptionsMonitor<GameSettings> _gameSettings;
    private readonly IOptionsMonitor<CacheSettings> _cacheSettings;
    private readonly ILogger<ConfigurationMonitorService> _logger;
    
    public ConfigurationMonitorService(
        IOptionsMonitor<GameSettings> gameSettings,
        IOptionsMonitor<CacheSettings> cacheSettings,
        ILogger<ConfigurationMonitorService> logger)
    {
        _gameSettings = gameSettings;
        _cacheSettings = cacheSettings;
        _logger = logger;
        
        // 설정 변경 이벤트 구독
        _gameSettings.OnChange(OnGameSettingsChanged);
        _cacheSettings.OnChange(OnCacheSettingsChanged);
    }
    
    private void OnGameSettingsChanged(GameSettings settings)
    {
        _logger.LogInformation(
            "게임 설정 변경 감지: DefaultGold: {Gold}, DefaultGem: {Gem}",
            settings.DefaultPlayerGold,
            settings.DefaultPlayerGem);
    }
    
    private void OnCacheSettingsChanged(CacheSettings settings)
    {
        _logger.LogInformation(
            "캐시 설정 변경 감지: MasterDataCache: {Minutes}분",
            settings.MasterDataCacheMinutes);
    }
    
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            await Task.Delay(TimeSpan.FromMinutes(1), stoppingToken);
        }
    }
}

// Program.cs에 등록
builder.Services.AddHostedService<ConfigurationMonitorService>();
```

## 24.11 설정 검증

애플리케이션 시작 시 필수 설정 값을 검증한다.

```csharp
// Configuration/ConfigurationValidator.cs
using Microsoft.Extensions.Options;

namespace GameAPIServer.Configuration;

public class ConfigurationValidator
{
    public static void ValidateOnStartup(IServiceProvider services)
    {
        // Database 설정 검증
        var dbSettings = services.GetRequiredService<IOptions<DatabaseSettings>>().Value;
        if (string.IsNullOrEmpty(dbSettings.MySQL.ConnectionString))
        {
            throw new InvalidOperationException("MySQL 연결 문자열이 설정되지 않았다");
        }
        
        // Redis 설정 검증
        var redisSettings = services.GetRequiredService<IOptions<RedisSettings>>().Value;
        if (string.IsNullOrEmpty(redisSettings.ConnectionString))
        {
            throw new InvalidOperationException("Redis 연결 문자열이 설정되지 않았다");
        }
        
        // Authentication 설정 검증
        var authSettings = services.GetRequiredService<IOptions<AuthenticationSettings>>().Value;
        if (string.IsNullOrEmpty(authSettings.TokenSecretKey))
        {
            throw new InvalidOperationException("토큰 시크릿 키가 설정되지 않았다");
        }
        
        if (authSettings.TokenSecretKey.Length < 32)
        {
            throw new InvalidOperationException("토큰 시크릿 키는 최소 32자 이상이어야 한다");
        }
        
        // GameSettings 검증
        var gameSettings = services.GetRequiredService<IOptions<GameSettings>>().Value;
        if (gameSettings.MaxInventorySlots <= 0)
        {
            throw new InvalidOperationException("인벤토리 슬롯 수는 1 이상이어야 한다");
        }
        
        Console.WriteLine("모든 설정 검증 완료");
    }
}

// Program.cs에서 호출
var app = builder.Build();

// 설정 검증
ConfigurationValidator.ValidateOnStartup(app.Services);

app.Run();
```

## 24.13 설정 우선순위

ASP.NET Core는 다음 순서로 설정을 로드한다. 나중에 로드된 설정이 이전 설정을 덮어쓴다.

1. appsettings.json
2. appsettings.{Environment}.json
3. User Secrets (개발 환경만)
4. 환경 변수
5. 명령줄 인자

예를 들어, appsettings.json에 `Database:MySQL:ConnectionString`이 있고, 환경 변수로도 설정되어 있다면 환경 변수 값이 최종적으로 사용된다.

## 24.14 실전 배포 시나리오

**시나리오 1: 로컬 개발 환경**

개발자는 User Secrets에 데이터베이스 비밀번호를 저장하고, appsettings.Development.json으로 환경을 구성한다.

```bash
dotnet user-secrets set "Database:MySQL:ConnectionString" "Server=localhost;..."
dotnet run --environment Development
```

**시나리오 2: Docker 컨테이너 배포**

Docker Compose로 환경 변수를 주입한다.

```bash
docker-compose up -d
```

환경 변수는 docker-compose.yml이나 .env 파일에서 관리한다.

**시나리오 3: 클라우드 배포 (Azure)**

Azure App Service에서는 Application Settings에 환경 변수를 설정한다.

```
Database__MySQL__ConnectionString = Server=xxx.mysql.database.azure.com;...
Redis__ConnectionString = xxx.redis.cache.windows.net:6380,password=xxx,ssl=True
Authentication__TokenSecretKey = production-secret-key-from-key-vault
```

**시나리오 4: Kubernetes 배포**

ConfigMap과 Secret을 사용한다.

```yaml
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: game-api-config
data:
  ASPNETCORE_ENVIRONMENT: "Production"
  GameSettings__MaxInventorySlots: "100"
  Cache__MasterDataCacheMinutes: "60"

---
# secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: game-api-secrets
type: Opaque
stringData:
  database-connection: "Server=mysql;Port=3306;Database=game_db;..."
  redis-connection: "redis:6379"
  token-secret: "production-secret-key"
```

## 24.15 보안 체크리스트

설정 관리 시 다음 사항을 확인한다.

**민감 정보 보호**: 비밀번호, API 키, 토큰은 절대 코드 저장소에 커밋하지 않는다.

**환경별 분리**: 개발/스테이징/운영 환경의 설정을 명확히 분리한다.

**최소 권한 원칙**: 각 환경의 데이터베이스 계정은 필요한 권한만 부여한다.

**토큰 키 강도**: 운영 환경의 토큰 시크릿 키는 최소 32자 이상의 랜덤 문자열을 사용한다.

**설정 검증**: 애플리케이션 시작 시 필수 설정을 검증한다.

**로그 민감도**: 운영 환경에서는 로그 레벨을 Warning 이상으로 설정하여 불필요한 정보 노출을 방지한다.

## 24.16 마무리

이번 장에서는 게임 서버의 설정을 체계적으로 관리하는 방법을 학습했다. appsettings.json을 활용한 계층적 설정 구조, 환경별 설정 분리, User Secrets와 환경 변수를 통한 민감 정보 보호, 설정 검증과 모니터링까지 다루었다.

설정 관리는 단순해 보이지만 서비스 운영의 안정성과 보안에 직결되는 중요한 요소다. 개발 환경과 운영 환경을 명확히 분리하고, 민감한 정보는 안전하게 관리하며, 배포 자동화를 통해 휴먼 에러를 최소화해야 한다.

이제 기본적인 게임 서버 개발에 필요한 모든 주요 기능을 학습했다. 인증, 데이터베이스, 캐시, 게임 로직, 로깅, 설정 관리까지 실무에서 바로 활용할 수 있는 기반을 마련했다. 이를 바탕으로 실제 게임 서비스를 개발하고 운영할 수 있는 역량을 갖추게 되었다.
