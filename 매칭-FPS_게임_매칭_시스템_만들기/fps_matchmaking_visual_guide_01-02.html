매칭 시스템은 무엇을 해결할까?

좋은 매칭은 완벽한 상대를 찾는 일이 아니라, 제한 시간 안에서 공정함과 쾌적함을 최대한 맞추는 일입니다. 문서의 핵심은 세 가지 기준으로 압축할 수 있습니다.

Fair

실력 균형

양 팀의 평균 MMR 차이를 줄이고, 팀 내부의 실력 편차도 너무 커지지 않게 조정합니다.

Fast

대기 시간

30초에서 2분 사이에 게임이 시작되도록, 기다린 시간이 길수록 조건을 조금씩 완화합니다.

Stable

낮은 지연

FPS에서는 50ms 차이도 체감됩니다. 그래서 지역과 서버 거리, 핑 범위를 매칭 조건에 넣습니다.

플레이어가 게임에 들어가기까지

매칭은 버튼 하나로 보이지만 내부에서는 여러 필터와 계산이 순서대로 지나갑니다.

1. 큐 입장 플레이어의 MMR, 지역, 역할, 파티 정보를 대기열에 넣습니다.
2. 후보 검색 Redis Sorted Set처럼 빠른 저장소에서 비슷한 MMR 구간을 찾습니다.
3. 조건 확인 핑, 파티 크기, 역할 구성, 치터 의심 여부를 함께 검사합니다.
4. 팀 분배 양 팀의 평균과 편차가 비슷해지도록 플레이어를 나눕니다.
5. 세션 배정 게임 서버를 잡고, 매칭 완료 이벤트를 각 클라이언트에 보냅니다.

왜 대기 시간이 길어질수록 조건을 넓힐까?

처음에는 아주 좋은 조건만 허용합니다. 하지만 플레이어를 너무 오래 기다리게 하면 게임을 떠나므로, 시간이 지날수록 허용 범위를 단계적으로 넓힙니다.

0~30초

핑 50ms 이하, MMR 차이 작음, 같은 지역 우선.

30~60초

핑 80ms까지 허용, MMR 범위를 조금 확대.

60초 이후

핑 120ms까지 허용, 빠른 시작을 더 중시.

팀 밸런싱은 평균만 맞추면 끝일까?

아닙니다. 평균 MMR이 같아도 한 팀 안에 초보와 고수가 과하게 섞이면 체감 품질은 나빠집니다. 그래서 평균과 편차를 함께 봅니다.

장애가 나면?

분산 시스템에서는 네트워크 파티셔닝이 발생할 수 있습니다. 매칭 시스템은 보통 가용성을 우선하고, 짧은 불일치는 최종 일관성으로 수렴시킵니다.

분산 락 헬스 체크 서킷 브레이커 수평 확장 Stateless 서버 Kafka/RabbitMQ

ELO는 “예상과 실제의 차이”를 점수로 바꾼다

ELO는 상대와의 레이팅 차이로 기대 승률을 계산하고, 실제 결과가 기대와 얼마나 달랐는지에 따라 점수를 조정합니다.

E_A = 1 / (1 + 10^((R_B - R_A) / 400))
R'_A = R_A + K × (S_A - E_A)
플레이어 A 기대 승률
76%
A가 이기면 +8점, 지면 -24점

MMR은 ELO보다 더 많은 맥락을 담는다

ELO가 하나의 점수라면, 현대 MMR은 “실력 추정치”와 “확신도”를 함께 다룹니다. 신규 플레이어는 불확실성이 높아 점수가 빠르게 움직이고, 오래 플레이한 계정은 더 안정적으로 변합니다.

한 장으로 정리

좋은 매칭

MMR 차이가 작고, 핑이 낮고, 역할과 파티 구성이 자연스럽고, 대기 시간이 길지 않은 게임입니다.

좋은 서버 구조

매칭 서버는 Stateless로 늘리고, Redis로 실시간 상태를 공유하며, DB와 메시지 큐로 결과를 안정적으로 처리합니다.

좋은 레이팅

ELO처럼 이해하기 쉬운 모델에서 시작하되, FPS 팀 게임에는 불확실성·개인 기여도·역할별 지표를 함께 반영합니다.
