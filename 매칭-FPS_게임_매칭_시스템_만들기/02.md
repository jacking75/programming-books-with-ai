# 2. 스킬 레이팅 시스템
스킬 레이팅 시스템은 매칭 시스템의 핵심이다. 플레이어의 실력을 수치화하여 비슷한 수준의 상대를 찾아주는 것이 목적이다. 좋은 레이팅 시스템은 플레이어의 진짜 실력을 정확하게 반영하고, 게임 결과에 따라 적절하게 조정되며, 시간이 지나도 안정적으로 유지되어야 한다.

## 2.1 ELO 레이팅 시스템
ELO 레이팅은 체스 선수의 실력을 평가하기 위해 개발된 시스템이지만, 현재는 다양한 게임에서 널리 사용된다. 간단하면서도 효과적인 수학적 모델로, 1대1 대전 게임에 특히 적합하다.

### ELO의 기본 원리와 수식
ELO 시스템의 핵심 아이디어는 간단하다. 실력이 높은 플레이어는 높은 점수를, 낮은 플레이어는 낮은 점수를 가진다. 두 플레이어가 대결할 때, 점수 차이를 바탕으로 승리 확률을 계산하고, 실제 결과와 비교하여 점수를 조정한다. 예상보다 강한 상대를 이기면 많은 점수를 얻고, 약한 상대에게 지면 많은 점수를 잃는다.

기대 승률을 계산하는 공식은 다음과 같다. 플레이어 A의 레이팅이 R_A이고 플레이어 B의 레이팅이 R_B일 때, A가 승리할 기대 확률 E_A는 다음 공식으로 구한다.

```
E_A = 1 / (1 + 10^((R_B - R_A) / 400))
```

이 공식은 로지스틱 곡선을 따른다. 두 플레이어의 점수가 같으면 승률은 50%가 된다. 점수 차이가 400점이면 승률은 약 91%가 된다. 200점 차이는 약 76%, 100점 차이는 약 64%의 승률을 의미한다.

실제 게임이 끝난 후, 레이팅을 업데이트하는 공식은 다음과 같다.

```
R'_A = R_A + K × (S_A - E_A)
```

여기서 R'_A는 새로운 레이팅, K는 K-팩터(변동성 계수), S_A는 실제 결과(승리=1, 패배=0, 무승부=0.5)다. K-팩터는 레이팅이 얼마나 빠르게 변할지를 결정한다. 일반적으로 16에서 32 사이의 값을 사용한다.

구체적인 예를 들어보자. 레이팅 1600인 플레이어 A가 레이팅 1400인 플레이어 B와 대결한다. A의 기대 승률은 `1 / (1 + 10^((1400-1600)/400)) = 1 / (1 + 10^(-0.5)) = 0.76`이다. 약 76%의 확률로 A가 이길 것으로 예상된다.

만약 A가 예상대로 승리하고 K-팩터가 32라면, A의 새 레이팅은 `1600 + 32 × (1 - 0.76) = 1600 + 7.68 = 1607.68`이 된다. 반면 B가 승리한다면, A의 레이팅은 `1600 + 32 × (0 - 0.76) = 1600 - 24.32 = 1575.68`로 크게 하락한다.

### FPS 게임에서의 ELO 적용 방법
ELO는 원래 1대1 게임을 위해 설계되었지만, FPS의 팀 기반 게임에도 적용할 수 있다. 가장 단순한 방법은 팀의 평균 레이팅을 계산하고, 두 팀을 하나의 플레이어처럼 취급하는 것이다.

5v5 게임에서 팀 A의 플레이어들 레이팅이 1400, 1500, 1600, 1700, 1800이고, 팀 B가 1450, 1550, 1650, 1750, 1850이라면, 팀 A의 평균은 1600, 팀 B의 평균은 1650이다. 이 두 값을 사용하여 팀 A의 기대 승률을 계산한다.

```
E_A = 1 / (1 + 10^((1650-1600)/400)) = 0.465
```

팀 A가 승리하면, 각 플레이어는 `K × (1 - 0.465) = K × 0.535`만큼 레이팅이 상승한다. K-팩터를 32로 설정하면 약 17점씩 상승한다. 패배하면 약 15점씩 하락한다.

그러나 이 방법은 모든 플레이어를 동등하게 취급한다는 문제가 있다. 게임에서 압도적인 활약을 한 플레이어나 전혀 기여하지 못한 플레이어나 같은 점수 변동을 받는다. 이를 개선하기 위해 개인 성적을 반영한 가중치를 적용할 수 있다.

예를 들어, 팀 게임에서 기본 점수 변동에 개인 성적 계수를 곱하는 방식이다. 킬/데스 비율이 팀 평균보다 높으면 계수를 1.2로, 낮으면 0.8로 설정한다. MVP를 받은 플레이어는 1.5배, 게임 중 이탈한 플레이어는 2배의 페널티를 받는다.

```csharp
public class EloCalculator
{
    private const double BaseRating = 1500.0;
    private const double KFactor = 32.0;

    public double CalculateExpectedScore(double ratingA, double ratingB)
    {
        return 1.0 / (1.0 + Math.Pow(10, (ratingB - ratingA) / 400.0));
    }

    public double UpdateRating(double currentRating, double expectedScore, 
                               double actualScore, double performanceMultiplier = 1.0)
    {
        double change = KFactor * (actualScore - expectedScore) * performanceMultiplier;
        return currentRating + change;
    }

    public double CalculateTeamAverageRating(List<double> teamRatings)
    {
        return teamRatings.Average();
    }
}
```

### 장단점 분석
ELO 시스템의 가장 큰 장점은 단순함이다. 수학적으로 명확하고 구현이 쉬우며, 플레이어에게 설명하기도 쉽다. 오랜 기간 검증된 시스템이라 신뢰성도 높다. 계산 비용이 적어 대규모 환경에서도 빠르게 처리할 수 있다.

또한 제로섬(Zero-sum) 특성을 가진다. 한 플레이어가 얻는 점수만큼 다른 플레이어가 잃기 때문에, 전체 시스템의 평균 레이팅이 일정하게 유지된다. 이는 인플레이션이나 디플레이션을 방지한다.

그러나 단점도 명확하다. 첫째, ELO는 플레이어의 실력이 이미 알려져 있다고 가정한다. 새로운 플레이어의 실력을 빠르게 파악하는 데는 한계가 있다. 신규 계정이 적정 레이팅에 도달하기까지 많은 게임이 필요하다.

둘째, 불확실성을 표현하지 못한다. 100게임을 한 1500점 플레이어와 10게임만 한 1500점 플레이어의 신뢰도는 다르지만, ELO는 이를 구분하지 않는다. 실제로는 후자의 레이팅이 훨씬 불확실하다.

셋째, 팀 게임에서의 적용이 완벽하지 않다. 개인의 기여도를 정확히 평가하기 어렵고, 팀원의 실력에 자신의 레이팅이 영향받는 문제가 있다. 운 좋게 좋은 팀원을 만나면 실력 이상의 레이팅을 받을 수 있다.

넷째, K-팩터 조정이 까다롭다. 너무 크면 레이팅이 불안정하게 요동치고, 너무 작으면 실력 변화를 제때 반영하지 못한다. 신규 플레이어와 베테랑 플레이어에게 같은 K-팩터를 적용하는 것도 비효율적이다.
  

## 2.2 MMR (Matchmaking Rating) 시스템
MMR은 ELO의 한계를 극복하기 위해 발전한 레이팅 시스템들을 통칭한다. 단순히 승패만 고려하는 것이 아니라, 레이팅의 불확실성, 시간 경과에 따른 실력 변화, 게임 특성 등을 종합적으로 반영한다.

### MMR의 개념과 ELO와의 차이점
MMR은 플레이어의 실력을 나타내는 숨겨진 점수다. 많은 게임이 플레이어에게 보여주는 랭크(브론즈, 실버, 골드 등)와 내부적으로 사용하는 MMR을 분리한다. 랭크는 플레이어의 동기부여와 성취감을 위한 것이고, MMR은 정확한 매칭을 위한 것이다.

ELO가 단일 숫자로 실력을 표현한다면, 대부분의 MMR 시스템은 평균(μ)과 불확실성(σ)의 두 값을 사용한다. 예를 들어, 플레이어의 실력이 μ=1500, σ=200이라면, 실제 실력이 1300에서 1700 사이에 있을 확률이 68%(1 표준편차)라는 의미다.

게임을 많이 할수록 σ값이 감소하여 레이팅의 신뢰도가 높아진다. 새 플레이어는 σ가 크기 때문에 승패에 따라 레이팅이 크게 변동하고, 베테랑은 σ가 작아서 레이팅이 안정적이다. 이는 ELO의 고정된 K-팩터보다 훨씬 유연하다.

또한 MMR 시스템은 시간 감쇠(Time decay)를 적용할 수 있다. 오랫동안 게임을 하지 않은 플레이어는 실력이 녹슬었을 가능성이 있으므로, σ값을 증가시켜 불확실성을 반영한다. 예를 들어, 3개월 이상 게임을 하지 않으면 매주 σ를 10씩 증가시키는 방식이다.

### Glicko/Glicko-2 레이팅 시스템
Glicko는 ELO를 개선한 레이팅 시스템으로, 마크 글릭만(Mark Glickman)이 개발했다. ELO와 달리 레이팅의 신뢰도(RD, Rating Deviation)를 함께 관리한다.

Glicko 시스템에서 플레이어는 레이팅 r과 RD 값을 가진다. 초기 값은 보통 r=1500, RD=350으로 설정한다. RD가 클수록 레이팅이 불확실하다는 의미다. 게임을 할 때마다 RD는 감소하고, 시간이 지나면 다시 증가한다.

두 플레이어의 대전 결과를 반영할 때, RD가 큰 플레이어의 레이팅은 더 크게 변동한다. 이는 ELO의 K-팩터를 동적으로 조정하는 효과를 낸다. 또한 RD 값을 참고하여 비슷한 불확실성을 가진 플레이어끼리 매칭할 수 있다.

Glicko-2는 Glicko를 더욱 개선한 버전이다. 변동성(Volatility) 개념을 추가하여, 플레이어의 실력이 얼마나 일관적인지를 측정한다. 어떤 날은 잘하고 어떤 날은 못하는 플레이어는 높은 변동성을 가지며, 이를 레이팅 계산에 반영한다.

Glicko-2의 업데이트 과정은 복잡하지만, 대략적인 흐름은 다음과 같다. 먼저 레이팅 r, RD, 변동성 v를 Glicko-2 스케일로 변환한다. 대전 상대들의 정보를 바탕으로 새로운 변동성을 계산한다. 그다음 새로운 RD를 계산하고, 최종적으로 새로운 레이팅을 계산한다.

```csharp
public class Glicko2Calculator
{
    private const double Tau = 0.5; // 시스템 상수
    private const double Epsilon = 0.000001;

    public class Rating
    {
        public double R { get; set; } = 1500;
        public double RD { get; set; } = 350;
        public double Volatility { get; set; } = 0.06;
    }

    // Glicko-2 스케일로 변환
    private double ConvertRatingToGlicko2Scale(double rating)
    {
        return (rating - 1500) / 173.7178;
    }

    private double ConvertRDToGlicko2Scale(double rd)
    {
        return rd / 173.7178;
    }

    // g 함수 계산
    private double CalculateG(double phi)
    {
        return 1 / Math.Sqrt(1 + 3 * phi * phi / (Math.PI * Math.PI));
    }

    // 기대 승률 계산
    private double CalculateE(double mu, double muJ, double phiJ)
    {
        return 1 / (1 + Math.Exp(-CalculateG(phiJ) * (mu - muJ)));
    }

    // 간소화된 업데이트 (실제 구현은 더 복잡함)
    public Rating UpdateRating(Rating current, List<(Rating opponent, double score)> results)
    {
        double mu = ConvertRatingToGlicko2Scale(current.R);
        double phi = ConvertRDToGlicko2Scale(current.RD);
        double sigma = current.Volatility;

        // v와 delta 계산
        double v = 0;
        double delta = 0;
        
        foreach (var (opponent, score) in results)
        {
            double muJ = ConvertRatingToGlicko2Scale(opponent.R);
            double phiJ = ConvertRDToGlicko2Scale(opponent.RD);
            double gPhiJ = CalculateG(phiJ);
            double e = CalculateE(mu, muJ, phiJ);
            
            v += gPhiJ * gPhiJ * e * (1 - e);
            delta += gPhiJ * (score - e);
        }
        
        v = 1 / v;
        delta *= v;

        // 새로운 변동성 계산 (반복 알고리즘)
        double newSigma = CalculateNewVolatility(sigma, phi, v, delta);

        // 새로운 RD 계산
        double phiStar = Math.Sqrt(phi * phi + newSigma * newSigma);
        double newPhi = 1 / Math.Sqrt(1 / (phiStar * phiStar) + 1 / v);

        // 새로운 레이팅 계산
        double newMu = mu;
        foreach (var (opponent, score) in results)
        {
            double muJ = ConvertRatingToGlicko2Scale(opponent.R);
            double phiJ = ConvertRDToGlicko2Scale(opponent.RD);
            double gPhiJ = CalculateG(phiJ);
            double e = CalculateE(mu, muJ, phiJ);
            
            newMu += newPhi * newPhi * gPhiJ * (score - e);
        }

        return new Rating
        {
            R = newMu * 173.7178 + 1500,
            RD = newPhi * 173.7178,
            Volatility = newSigma
        };
    }

    private double CalculateNewVolatility(double sigma, double phi, double v, double delta)
    {
        // Illinois 알고리즘을 사용한 반복 계산
        // 실제 구현은 매우 복잡하므로 간소화
        return sigma; // 실제로는 복잡한 반복 계산 필요
    }
}
```

### TrueSkill 알고리즘
TrueSkill은 마이크로소프트가 개발한 레이팅 시스템으로, Xbox Live에서 사용된다. Glicko와 비슷하게 평균 μ와 표준편차 σ를 사용하지만, 베이지안 추론을 기반으로 하여 더욱 정교하다.

TrueSkill의 가장 큰 특징은 팀 게임에 최적화되어 있다는 점이다. 여러 명이 팀을 이루는 게임에서 각 플레이어의 기여도를 확률적으로 추정한다. 2v2, 5v5 같은 팀 게임뿐만 아니라, 배틀로얄처럼 다수가 경쟁하는 게임에도 적용할 수 있다.

TrueSkill은 각 플레이어의 "진짜 실력"이 정규분포를 따른다고 가정한다. 게임 결과를 관찰할 때마다, 베이지안 추론으로 이 분포를 업데이트한다. 즉, 사전 확률(게임 전 믿음)과 사후 확률(게임 후 믿음)을 계산한다.

TrueSkill의 보수적인 레이팅은 μ - 3σ로 계산된다. 이는 99.7% 확률로 플레이어의 실제 실력이 이 값보다 높다는 의미다. 매칭 시에는 이 보수적 레이팅을 사용하여, 신규 플레이어가 과대평가되어 불공정한 매칭이 일어나는 것을 방지한다.

TrueSkill의 단점은 계산이 매우 복잡하다는 것이다. 특히 플레이어가 많은 팀 게임에서는 행렬 연산과 수치 적분이 필요하여, 실시간으로 계산하기 부담스럽다. 또한 마이크로소프트의 특허로 보호받고 있어, 상업적 사용에 제약이 있을 수 있다.

### FPS 게임에 적합한 MMR 선택
FPS 게임에서 어떤 MMR 시스템을 선택할지는 게임의 특성에 달려 있다. 1v1 듀얼 게임이라면 Glicko-2가 적합하다. 레이팅 불확실성을 잘 관리하고, 계산도 비교적 간단하다. 오픈소스 구현체도 많아 도입이 쉽다.

5v5 같은 팀 기반 게임이라면 TrueSkill이나 TrueSkill의 오픈소스 대안을 고려할 수 있다. 팀 밸런싱에 최적화되어 있고, 각 플레이어의 기여도를 확률적으로 평가한다. 다만 계산 복잡도가 높으므로, 게임 서버가 아닌 별도의 매칭 서버에서 비동기로 처리하는 것이 좋다.

배틀로얄 같은 대규모 경쟁 게임이라면, 간소화된 MMR 시스템을 자체 개발하는 것도 방법이다. 순위에 따라 점수를 차등 지급하고, 킬 수와 생존 시간을 보조 지표로 활용한다. 예를 들어, 1위는 +50점, 10위는 +10점, 50위는 -20점 같은 방식이다.

대부분의 현대 FPS 게임은 하이브리드 방식을 사용한다. 기본적으로는 ELO나 Glicko-2를 사용하되, 개인 성적(K/D, 헤드샷률, 목표 달성도)을 보정 계수로 적용한다. 또한 연승 보너스, 승급전 가중치, 시즌 초기화 같은 게임플레이 요소를 추가하여 플레이어의 흥미를 유지한다.

구현 시 고려할 점은 다음과 같다. 첫째, MMR 계산은 게임 서버가 아닌 별도의 백엔드 서버에서 처리한다. 게임 결과만 전송하고, MMR 업데이트는 비동기로 수행한다. 둘째, MMR 히스토리를 저장하여, 이상 패턴을 감지하고 어뷰징을 방지한다. 셋째, 주기적으로 MMR 분포를 분석하여, 인플레이션이나 디플레이션이 발생하지 않는지 확인한다.
  

## 2.3 멀티 팩터 레이팅
단순히 승패만으로 플레이어를 평가하는 것은 한계가 있다. 같은 승리라도 압도적인 활약으로 이긴 경우와 팀에 캐리당한 경우는 다르다. 멀티 팩터 레이팅은 여러 지표를 종합하여 플레이어의 실력을 더 정확하게 평가한다.

### K/D 비율, 승률, 팀 기여도 등 복합 지표
FPS 게임에서 가장 직관적인 지표는 킬/데스(K/D) 비율이다. 적을 많이 처치하고 적게 죽을수록 실력이 높다고 볼 수 있다. 그러나 K/D만으로 실력을 판단하면 문제가 생긴다. 킬을 많이 얻기 위해 팀 플레이를 무시하고 혼자 돌격하는 플레이어가 높은 평가를 받을 수 있다.

따라서 K/D 외에 다른 지표들을 함께 고려해야 한다. 승률은 가장 중요한 지표다. 아무리 킬을 많이 해도 게임에서 지면 의미가 없다. 헤드샷률은 정확도를 나타낸다. FPS에서 헤드샷은 높은 기술력을 요구하므로, 헤드샷률이 높은 플레이어는 실력이 우수하다고 볼 수 있다.

객관적 기여도도 중요하다. 폭탄 설치/해체, 거점 점령, 인질 구출 같은 목표 달성은 승리에 직접적으로 기여한다. 어시스트 수는 팀워크를 나타낸다. 적을 처치하지는 못했지만 데미지를 주어 팀원이 마무리할 수 있도록 도운 것이다. 생존 시간도 의미가 있다. 배틀로얄 게임에서는 특히 중요하다.

대미지 딜(Damage Dealt)과 대미지 받음(Damage Taken)의 비율도 유용하다. 킬을 못하더라도 적에게 큰 타격을 주었다면 팀에 기여한 것이다. 반대로 많이 죽지 않더라도 적에게 타격을 전혀 못 주었다면 기여도가 낮다.

경제 관리도 일부 FPS 게임에서 중요하다. CS:GO 같은 게임에서는 돈을 효율적으로 사용하고, 팀원과 공유하는 것이 승리에 기여한다. 무기와 장비를 적절히 구매하고, 죽지 않아서 다음 라운드에 자산을 이어가는 것도 실력이다.

```csharp
public class MultiFactorRating
{
    public class PlayerPerformance
    {
        public int Kills { get; set; }
        public int Deaths { get; set; }
        public int Assists { get; set; }
        public int Headshots { get; set; }
        public int TotalShots { get; set; }
        public int ObjectiveScore { get; set; } // 목표 달성 점수
        public double DamageDealt { get; set; }
        public double DamageTaken { get; set; }
        public TimeSpan SurvivalTime { get; set; }
        public bool IsWinner { get; set; }
    }

    public double CalculatePerformanceScore(PlayerPerformance perf)
    {
        // K/D 비율 점수 (최대 40점)
        double kdRatio = perf.Deaths > 0 ? (double)perf.Kills / perf.Deaths : perf.Kills;
        double kdScore = Math.Min(kdRatio * 10, 40);

        // 헤드샷률 점수 (최대 20점)
        double headshotRate = perf.TotalShots > 0 ? 
            (double)perf.Headshots / perf.TotalShots : 0;
        double headshotScore = headshotRate * 100; // 20% 헤드샷률 = 20점

        // 어시스트 점수 (최대 15점)
        double assistScore = Math.Min(perf.Assists * 3, 15);

        // 목표 기여도 점수 (최대 25점)
        double objectiveScore = Math.Min(perf.ObjectiveScore / 4.0, 25);

        // 대미지 효율 점수 (최대 20점)
        double damageEfficiency = perf.DamageTaken > 0 ? 
            perf.DamageDealt / perf.DamageTaken : perf.DamageDealt / 100;
        double damageScore = Math.Min(damageEfficiency * 5, 20);

        // 승패 보너스/페널티 (±30점)
        double winBonus = perf.IsWinner ? 30 : -30;

        // 총점 계산
        double totalScore = kdScore + headshotScore + assistScore + 
                           objectiveScore + damageScore + winBonus;

        return totalScore;
    }
}
```

### 가중치 설정 전략
여러 지표를 종합할 때 가장 중요한 것은 가중치 설정이다. 어떤 지표에 얼마의 비중을 둘지 결정해야 한다. 가중치는 게임의 특성과 플레이 스타일에 따라 달라진다.

팀 데스매치 같은 킬 중심 모드에서는 K/D에 높은 가중치를 준다. 40~50% 정도가 적절하다. 반면 폭탄 설치/해체 모드에서는 목표 달성에 더 높은 가중치를 준다. 폭탄을 설치하거나 해체하면 킬보다 더 많은 점수를 받아야 한다.

역할별로 가중치를 다르게 적용할 수도 있다. 스나이퍼는 킬과 헤드샷에 높은 가중치, 서포터는 어시스트와 생존 시간에 높은 가중치를 준다. 탱커 역할은 대미지 받음과 생존 시간이 중요하고, 딜러는 대미지 딜이 중요하다.

시간에 따른 가중치 변화도 고려할 수 있다. 게임 초반의 킬보다 후반의 킬이 더 중요할 수 있다. 특히 라운드제 게임에서 마지막 라운드의 성적은 승패를 직접 가르므로 더 높은 가중치를 받을 수 있다. 클러치 상황(1 vs 다수)에서의 킬은 보너스 점수를 받는다.

가중치를 설정하는 방법은 크게 두 가지다. 첫째는 게임 디자이너의 경험과 직관으로 수동 설정하는 것이다. 베타 테스트를 거쳐 밸런스를 조정한다. 둘째는 기계학습을 사용하여 데이터 기반으로 최적화하는 것이다. 수천 개의 게임 데이터를 분석하여, 어떤 지표가 실제로 승리와 높은 상관관계를 가지는지 찾아낸다.

가중치 예시는 다음과 같다. 킬/데스 비율 35%, 승패 30%, 목표 기여도 20%, 헤드샷률 10%, 어시스트 5%. 이 비율은 게임 모드별로 조정된다. 랭크 게임에서는 승패 비중을 50%까지 높이고, 캐주얼 게임에서는 개인 성적 비중을 높인다.

정규화(Normalization)도 중요하다. 서로 다른 스케일의 지표를 합치려면 0~1 범위로 정규화해야 한다. 예를 들어, K/D 비율은 0~5 범위, 헤드샷률은 0~100% 범위를 가질 수 있다. 이를 동일한 척도로 변환한 후 가중 합산한다.

```csharp
public class WeightedRatingCalculator
{
    public class Weights
    {
        public double KillDeathRatio { get; set; } = 0.35;
        public double WinRate { get; set; } = 0.30;
        public double ObjectiveScore { get; set; } = 0.20;
        public double HeadshotRate { get; set; } = 0.10;
        public double AssistRate { get; set; } = 0.05;
    }

    private Weights weights;

    public WeightedRatingCalculator(Weights weights = null)
    {
        this.weights = weights ?? new Weights();
    }

    // 0~1 범위로 정규화
    private double Normalize(double value, double min, double max)
    {
        if (max <= min) return 0;
        return Math.Clamp((value - min) / (max - min), 0, 1);
    }

    public double CalculateFinalRating(MultiFactorRating.PlayerPerformance perf,
                                       double teamAverageKD,
                                       double teamWinRate)
    {
        // 각 지표를 정규화
        double kdRatio = perf.Deaths > 0 ? (double)perf.Kills / perf.Deaths : perf.Kills;
        double normalizedKD = Normalize(kdRatio, 0, 5); // 0~5 범위를 0~1로

        double headshotRate = perf.TotalShots > 0 ? 
            (double)perf.Headshots / perf.TotalShots : 0;
        double normalizedHeadshot = headshotRate; // 이미 0~1 범위

        double assistRate = perf.Kills + perf.Deaths > 0 ? 
            (double)perf.Assists / (perf.Kills + perf.Deaths) : 0;
        double normalizedAssist = Normalize(assistRate, 0, 2);

        double normalizedObjective = Normalize(perf.ObjectiveScore, 0, 100);

        double winScore = perf.IsWinner ? 1.0 : 0.0;

        // 가중 합산
        double finalScore = 
            normalizedKD * weights.KillDeathRatio +
            winScore * weights.WinRate +
            normalizedObjective * weights.ObjectiveScore +
            normalizedHeadshot * weights.HeadshotRate +
            normalizedAssist * weights.AssistRate;

        // 0~100 스케일로 변환
        return finalScore * 100;
    }

    // 게임 모드에 따른 가중치 조정
    public static Weights GetWeightsForGameMode(string gameMode)
    {
        return gameMode switch
        {
            "TeamDeathmatch" => new Weights 
            { 
                KillDeathRatio = 0.50, 
                WinRate = 0.25, 
                ObjectiveScore = 0.10,
                HeadshotRate = 0.10,
                AssistRate = 0.05
            },
            "BombDefusal" => new Weights 
            { 
                KillDeathRatio = 0.25, 
                WinRate = 0.30, 
                ObjectiveScore = 0.30,
                HeadshotRate = 0.10,
                AssistRate = 0.05
            },
            "Ranked" => new Weights 
            { 
                KillDeathRatio = 0.20, 
                WinRate = 0.50, 
                ObjectiveScore = 0.20,
                HeadshotRate = 0.05,
                AssistRate = 0.05
            },
            _ => new Weights() // 기본 가중치
        };
    }
}
```

멀티 팩터 레이팅의 최종 목표는 플레이어의 진짜 기여도를 측정하는 것이다. 팀에 얼마나 도움이 되었는지, 승리에 얼마나 기여했는지를 정확히 평가하여, 매칭 품질을 높이고 플레이어 만족도를 향상시킨다. 다만 너무 복잡한 공식은 플레이어가 이해하기 어려우므로, 투명성과 정확성 사이의 균형을 찾는 것이 중요하다.
