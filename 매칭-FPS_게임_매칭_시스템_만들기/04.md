# 4. 고성능 매칭 시스템 아키텍처

## 4.1 시스템 구조 설계

### 매칭 서버 vs 게임 서버 분리
10만 명 이상의 동시 유저를 처리하려면 매칭 로직과 게임 실행 로직을 완전히 분리해야 한다. 매칭 서버는 플레이어를 모으고 팀을 구성하는 역할만 담당하고, 게임 서버는 실제 게임 세션을 호스팅한다.

매칭 서버는 상태를 가지는(stateful) 서비스로 설계한다. 대기 중인 플레이어 정보를 메모리에 유지하며 지속적으로 매칭을 시도한다. 반면 게임 서버는 매칭이 완료된 후에만 생성되는 단기 생명주기를 가진다.

```csharp
public class MatchmakingServer
{
    private readonly IMatchmakingEngine _engine;
    private readonly IGameServerOrchestrator _orchestrator;
    private readonly IPlayerSessionManager _sessionManager;
    
    public async Task<MatchResult> ProcessMatchRequest(MatchRequest request)
    {
        // 1. 플레이어 세션 등록
        await _sessionManager.RegisterPlayer(request.Player);
        
        // 2. 매칭 큐에 추가
        await _engine.EnqueuePlayer(request);
        
        // 3. 매칭 완료 대기 (비동기)
        var match = await WaitForMatch(request.RequestId);
        
        // 4. 게임 서버 할당 요청
        var gameServer = await _orchestrator.AllocateGameServer(match);
        
        // 5. 플레이어들에게 게임 서버 정보 전달
        await NotifyPlayers(match, gameServer);
        
        return new MatchResult 
        { 
            MatchId = match.Id, 
            GameServerEndpoint = gameServer.Endpoint 
        };
    }
}
```

게임 서버 오케스트레이터는 매칭이 완료되면 적절한 게임 서버를 할당하거나 새로 생성한다. Kubernetes나 Docker Swarm 같은 컨테이너 오케스트레이션 플랫폼을 활용하면 동적으로 게임 서버를 확장할 수 있다.

```csharp
public class GameServerOrchestrator
{
    private readonly IKubernetesClient _k8sClient;
    private readonly ConcurrentBag<GameServerInstance> _availableServers;
    private readonly SemaphoreSlim _allocationLock = new SemaphoreSlim(1, 1);
    
    public async Task<GameServerInstance> AllocateGameServer(Match match)
    {
        await _allocationLock.WaitAsync();
        try
        {
            // 사용 가능한 서버가 있으면 재사용
            if (_availableServers.TryTake(out var server))
            {
                await InitializeServer(server, match);
                return server;
            }
            
            // 없으면 새로 생성
            return await CreateNewGameServer(match);
        }
        finally
        {
            _allocationLock.Release();
        }
    }
    
    private async Task<GameServerInstance> CreateNewGameServer(Match match)
    {
        // Kubernetes Pod 생성
        var podSpec = new V1Pod
        {
            Metadata = new V1ObjectMeta 
            { 
                Name = $"game-server-{match.Id}",
                Labels = new Dictionary<string, string>
                {
                    { "app", "fps-game-server" },
                    { "match-id", match.Id }
                }
            },
            Spec = new V1PodSpec
            {
                Containers = new List<V1Container>
                {
                    new V1Container
                    {
                        Name = "game-server",
                        Image = "game-server:latest",
                        Env = new List<V1EnvVar>
                        {
                            new V1EnvVar { Name = "MATCH_ID", Value = match.Id },
                            new V1EnvVar { Name = "MATCH_DATA", Value = SerializeMatch(match) }
                        },
                        Resources = new V1ResourceRequirements
                        {
                            Requests = new Dictionary<string, ResourceQuantity>
                            {
                                { "cpu", new ResourceQuantity("500m") },
                                { "memory", new ResourceQuantity("1Gi") }
                            }
                        }
                    }
                }
            }
        };
        
        var createdPod = await _k8sClient.CreateNamespacedPodAsync(podSpec, "game-servers");
        
        // Pod가 준비될 때까지 대기
        await WaitForPodReady(createdPod.Metadata.Name);
        
        return new GameServerInstance
        {
            PodName = createdPod.Metadata.Name,
            Endpoint = GetPodEndpoint(createdPod),
            MatchId = match.Id
        };
    }
}
```

### 마이크로서비스 아키텍처
대규모 매칭 시스템은 여러 독립적인 마이크로서비스로 구성하는 것이 효과적이다. 각 서비스는 단일 책임을 가지며 독립적으로 확장할 수 있다.

주요 서비스 구성은 다음과 같다:

1. **매칭 게이트웨이**: 클라이언트 요청을 받아 적절한 매칭 노드로 라우팅한다
2. **매칭 엔진**: 실제 매칭 알고리즘을 실행하고 팀을 구성한다
3. **MMR 서비스**: 플레이어의 MMR을 계산하고 관리한다
4. **플레이어 프로필 서비스**: 플레이어의 통계와 설정을 관리한다
5. **게임 서버 관리자**: 게임 서버의 생성, 할당, 종료를 담당한다

```csharp
public class MatchmakingGateway
{
    private readonly IServiceDiscovery _serviceDiscovery;
    private readonly ILoadBalancer _loadBalancer;
    private readonly HttpClient _httpClient;
    
    public async Task<string> SubmitMatchRequest(MatchRequestDto request)
    {
        // 1. 플레이어 정보 조회 (Player Profile Service)
        var playerProfile = await GetPlayerProfile(request.PlayerId);
        
        // 2. 최신 MMR 조회 (MMR Service)
        var mmr = await GetPlayerMMR(request.PlayerId);
        
        // 3. 적절한 매칭 노드 선택
        var matchingNode = await SelectMatchingNode(mmr, playerProfile.Region);
        
        // 4. 매칭 요청 전달
        var fullRequest = new MatchRequest
        {
            Player = new Player
            {
                Id = request.PlayerId,
                MMR = mmr,
                Region = playerProfile.Region
            },
            PingMs = request.PingMs,
            SubmitTime = DateTime.UtcNow,
            RequestId = Guid.NewGuid().ToString()
        };
        
        var response = await _httpClient.PostAsJsonAsync(
            $"{matchingNode.Endpoint}/api/matching/enqueue", 
            fullRequest);
        
        return await response.Content.ReadAsStringAsync();
    }
    
    private async Task<MatchingNodeInfo> SelectMatchingNode(
        int mmr, string region)
    {
        // MMR과 지역 기반으로 적절한 노드 선택
        var availableNodes = await _serviceDiscovery
            .GetHealthyNodes("matching-engine");
        
        // 부하 분산을 고려한 노드 선택
        return _loadBalancer.SelectNode(availableNodes, mmr, region);
    }
}
```

### 로드 밸런싱 전략
10만 명의 동시 유저를 처리하려면 여러 매칭 엔진 인스턴스를 운영해야 한다. 로드 밸런서는 요청을 적절히 분산시킨다.

단순한 라운드 로빈 방식은 피해야 한다. 매칭은 상태를 가지므로 같은 MMR 범위의 플레이어들이 같은 노드로 라우팅되어야 효율적이다. 일관된 해싱(Consistent Hashing)을 사용하면 이를 달성할 수 있다.

```csharp
public class ConsistentHashLoadBalancer
{
    private readonly SortedDictionary<ulong, MatchingNodeInfo> _ring;
    private readonly int _virtualNodesPerNode;
    
    public ConsistentHashLoadBalancer(
        List<MatchingNodeInfo> nodes, 
        int virtualNodesPerNode = 150)
    {
        _virtualNodesPerNode = virtualNodesPerNode;
        _ring = new SortedDictionary<ulong, MatchingNodeInfo>();
        
        foreach (var node in nodes)
        {
            AddNode(node);
        }
    }
    
    public void AddNode(MatchingNodeInfo node)
    {
        for (int i = 0; i < _virtualNodesPerNode; i++)
        {
            string virtualKey = $"{node.Id}:{i}";
            ulong hash = ComputeHash(virtualKey);
            _ring[hash] = node;
        }
    }
    
    public MatchingNodeInfo SelectNode(int mmr, string region)
    {
        // MMR과 지역을 조합하여 라우팅 키 생성
        string routingKey = $"{region}:{mmr / 100}";
        ulong hash = ComputeHash(routingKey);
        
        // 링에서 hash보다 큰 첫 번째 노드 찾기
        var node = _ring.FirstOrDefault(kvp => kvp.Key >= hash);
        
        if (node.Value == null)
        {
            // 링의 끝에 도달하면 처음으로 돌아감
            node = _ring.First();
        }
        
        return node.Value;
    }
    
    private ulong ComputeHash(string key)
    {
        using (var md5 = MD5.Create())
        {
            byte[] hash = md5.ComputeHash(Encoding.UTF8.GetBytes(key));
            return BitConverter.ToUInt64(hash, 0);
        }
    }
}
```

또한 각 노드의 현재 부하를 고려한 동적 라우팅도 필요하다. 노드가 과부하 상태면 다른 노드로 요청을 보낸다.

```csharp
public class AdaptiveLoadBalancer
{
    private readonly ConsistentHashLoadBalancer _consistentHash;
    private readonly INodeHealthMonitor _healthMonitor;
    
    public async Task<MatchingNodeInfo> SelectNode(int mmr, string region)
    {
        var primaryNode = _consistentHash.SelectNode(mmr, region);
        var health = await _healthMonitor.GetNodeHealth(primaryNode.Id);
        
        // 노드가 과부하 상태인지 확인
        if (health.QueueSize > 10000 || health.CpuUsage > 80)
        {
            // 대체 노드 찾기
            return await FindAlternativeNode(mmr, region, primaryNode.Id);
        }
        
        return primaryNode;
    }
    
    private async Task<MatchingNodeInfo> FindAlternativeNode(
        int mmr, string region, string excludeNodeId)
    {
        var allNodes = await _healthMonitor.GetHealthyNodes();
        
        // 부하가 낮은 노드 중에서 MMR 범위가 비슷한 노드 선택
        var candidates = allNodes
            .Where(n => n.Id != excludeNodeId)
            .OrderBy(n => n.Health.QueueSize)
            .Take(3);
        
        return candidates.First();
    }
}
```
 

## 4.2 데이터 구조 및 최적화

### 인메모리 캐싱 (Redis 활용)
매칭 성능을 위해서는 빠른 데이터 접근이 필수적이다. 플레이어 프로필, MMR, 매칭 상태 등은 Redis 같은 인메모리 데이터베이스에 캐싱한다.

Redis는 단순한 캐시 이상의 역할을 한다. Sorted Set을 사용하면 MMR 순으로 플레이어를 정렬하여 저장할 수 있고, Pub/Sub으로 매칭 이벤트를 실시간으로 전파할 수 있다.

```csharp
public class RedisMatchingQueue
{
    private readonly IConnectionMultiplexer _redis;
    private readonly IDatabase _db;
    private const string QUEUE_KEY_PREFIX = "matching:queue:";
    
    public RedisMatchingQueue(IConnectionMultiplexer redis)
    {
        _redis = redis;
        _db = redis.GetDatabase();
    }
    
    public async Task EnqueuePlayer(MatchRequest request)
    {
        string queueKey = GetQueueKey(request.Player.Region);
        
        // Sorted Set에 MMR을 점수로 사용하여 저장
        await _db.SortedSetAddAsync(
            queueKey,
            request.RequestId,
            request.Player.MMR);
        
        // 플레이어 상세 정보는 Hash로 저장
        string detailKey = $"matching:request:{request.RequestId}";
        await _db.HashSetAsync(detailKey, new HashEntry[]
        {
            new HashEntry("playerId", request.Player.Id),
            new HashEntry("mmr", request.Player.MMR),
            new HashEntry("ping", request.PingMs),
            new HashEntry("submitTime", request.SubmitTime.Ticks),
            new HashEntry("region", request.Player.Region)
        });
        
        // 5분 후 자동 만료
        await _db.KeyExpireAsync(detailKey, TimeSpan.FromMinutes(5));
    }
    
    public async Task<List<MatchRequest>> GetCandidates(
        int targetMMR, int range, string region, int count = 50)
    {
        string queueKey = GetQueueKey(region);
        
        // MMR 범위 내의 플레이어들 조회
        var requestIds = await _db.SortedSetRangeByScoreAsync(
            queueKey,
            targetMMR - range,
            targetMMR + range,
            take: count);
        
        var requests = new List<MatchRequest>();
        foreach (var requestId in requestIds)
        {
            var request = await GetRequestDetails(requestId.ToString());
            if (request != null)
                requests.Add(request);
        }
        
        return requests;
    }
    
    private async Task<MatchRequest> GetRequestDetails(string requestId)
    {
        string detailKey = $"matching:request:{requestId}";
        var hash = await _db.HashGetAllAsync(detailKey);
        
        if (hash.Length == 0)
            return null;
        
        return new MatchRequest
        {
            RequestId = requestId,
            Player = new Player
            {
                Id = hash.FirstOrDefault(h => h.Name == "playerId").Value,
                MMR = (int)hash.FirstOrDefault(h => h.Name == "mmr").Value
            },
            PingMs = (int)hash.FirstOrDefault(h => h.Name == "ping").Value,
            SubmitTime = new DateTime((long)hash.FirstOrDefault(h => h.Name == "submitTime").Value)
        };
    }
    
    private string GetQueueKey(string region)
    {
        return $"{QUEUE_KEY_PREFIX}{region}";
    }
}
```

Redis의 Lua 스크립트를 사용하면 원자적(atomic) 작업을 수행할 수 있다. 이는 동시성 문제를 해결하는 데 유용하다.

```csharp
public class AtomicMatchingOperations
{
    private readonly IDatabase _db;
    
    // 여러 플레이어를 동시에 큐에서 제거하는 Lua 스크립트
    private const string REMOVE_PLAYERS_SCRIPT = @"
        local queueKey = KEYS[1]
        local requestIds = ARGV
        local removed = 0
        
        for i, requestId in ipairs(requestIds) do
            local result = redis.call('ZREM', queueKey, requestId)
            if result == 1 then
                removed = removed + 1
                redis.call('DEL', 'matching:request:' .. requestId)
            end
        end
        
        return removed
    ";
    
    public async Task<int> RemovePlayersFromQueue(
        string region, List<string> requestIds)
    {
        string queueKey = $"matching:queue:{region}";
        
        var result = await _db.ScriptEvaluateAsync(
            REMOVE_PLAYERS_SCRIPT,
            new RedisKey[] { queueKey },
            requestIds.Select(id => (RedisValue)id).ToArray());
        
        return (int)result;
    }
}
```

### 효율적인 검색 알고리즘
10만 명의 대기열에서 적합한 플레이어를 빠르게 찾으려면 효율적인 검색 알고리즘이 필요하다. 모든 플레이어를 순회하는 O(n) 알고리즘은 사용할 수 없다.

공간 분할(Spatial Partitioning) 기법을 MMR과 핑에 적용할 수 있다. 2D 그리드에서 MMR을 X축, 핑을 Y축으로 하여 플레이어를 배치한다. 검색 시에는 타겟 주변의 셀만 확인하면 된다.

```csharp
public class SpatialMatchingIndex
{
    private const int MMR_CELL_SIZE = 50;
    private const int PING_CELL_SIZE = 20;
    
    // 각 셀은 ConcurrentBag으로 관리
    private readonly ConcurrentDictionary<(int mmrCell, int pingCell), 
        ConcurrentBag<MatchRequest>> _grid;
    
    public SpatialMatchingIndex()
    {
        _grid = new ConcurrentDictionary<(int, int), ConcurrentBag<MatchRequest>>();
    }
    
    public void AddRequest(MatchRequest request)
    {
        var cell = GetCell(request.Player.MMR, request.PingMs);
        var bag = _grid.GetOrAdd(cell, _ => new ConcurrentBag<MatchRequest>());
        bag.Add(request);
    }
    
    public List<MatchRequest> FindCandidates(
        int targetMMR, int targetPing, int mmrRange, int pingRange)
    {
        var candidates = new List<MatchRequest>();
        
        // 검색 범위의 셀들 계산
        int minMMRCell = GetMMRCell(targetMMR - mmrRange);
        int maxMMRCell = GetMMRCell(targetMMR + mmrRange);
        int minPingCell = GetPingCell(targetPing - pingRange);
        int maxPingCell = GetPingCell(targetPing + pingRange);
        
        // 해당 셀들만 검색
        for (int mmrCell = minMMRCell; mmrCell <= maxMMRCell; mmrCell++)
        {
            for (int pingCell = minPingCell; pingCell <= maxPingCell; pingCell++)
            {
                if (_grid.TryGetValue((mmrCell, pingCell), out var bag))
                {
                    foreach (var request in bag)
                    {
                        if (IsInRange(request, targetMMR, targetPing, 
                            mmrRange, pingRange))
                        {
                            candidates.Add(request);
                        }
                    }
                }
            }
        }
        
        return candidates;
    }
    
    private (int mmrCell, int pingCell) GetCell(int mmr, int ping)
    {
        return (GetMMRCell(mmr), GetPingCell(ping));
    }
    
    private int GetMMRCell(int mmr)
    {
        return mmr / MMR_CELL_SIZE;
    }
    
    private int GetPingCell(int ping)
    {
        return ping / PING_CELL_SIZE;
    }
    
    private bool IsInRange(MatchRequest request, int targetMMR, int targetPing,
        int mmrRange, int pingRange)
    {
        return Math.Abs(request.Player.MMR - targetMMR) <= mmrRange &&
               Math.Abs(request.PingMs - targetPing) <= pingRange;
    }
}
```

더 나아가 k-d 트리나 R-트리 같은 다차원 인덱싱 구조를 사용할 수도 있다. 하지만 동시성 환경에서는 트리 재조정 비용이 크므로 신중하게 선택해야 한다.

### 락(Lock) 최소화 전략
고성능 시스템에서는 락 경합(lock contention)이 가장 큰 병목이다. 락을 최소화하거나 제거하는 전략이 필수적이다.

첫 번째 전략은 락프리(lock-free) 자료구조를 사용하는 것이다. C#의 `ConcurrentQueue`, `ConcurrentDictionary`, `ConcurrentBag`은 내부적으로 Compare-And-Swap(CAS) 같은 원자적 연산을 사용하여 락 없이 동작한다.

```csharp
public class LockFreeMatchingQueue
{
    // 락프리 큐 사용
    private readonly ConcurrentQueue<MatchRequest> _queue;
    
    // 원자적 카운터
    private int _queueSize = 0;
    
    public void Enqueue(MatchRequest request)
    {
        _queue.Enqueue(request);
        Interlocked.Increment(ref _queueSize);
    }
    
    public bool TryDequeue(out MatchRequest request)
    {
        if (_queue.TryDequeue(out request))
        {
            Interlocked.Decrement(ref _queueSize);
            return true;
        }
        return false;
    }
    
    public int Count => Volatile.Read(ref _queueSize);
}
```

두 번째 전략은 낙관적 동시성 제어(Optimistic Concurrency Control)다. 충돌이 드물다고 가정하고 먼저 작업을 수행한 후, 충돌이 발생하면 재시도한다.

```csharp
public class OptimisticMatchCreator
{
    private readonly ConcurrentDictionary<string, MatchRequest> _activeRequests;
    
    public async Task<Match> TryCreateMatch(List<string> requestIds)
    {
        // 1. 플레이어들을 낙관적으로 획득
        var requests = new List<MatchRequest>();
        var acquiredIds = new List<string>();
        
        foreach (var id in requestIds)
        {
            if (_activeRequests.TryGetValue(id, out var request))
            {
                requests.Add(request);
                acquiredIds.Add(id);
            }
        }
        
        // 필요한 수만큼 획득하지 못했으면 실패
        if (requests.Count < 10)
        {
            return null;
        }
        
        // 2. 매치 생성 시도
        var match = new Match
        {
            Id = Guid.NewGuid().ToString(),
            Players = requests.Select(r => r.Player).ToList(),
            CreatedAt = DateTime.UtcNow
        };
        
        // 3. 원자적으로 플레이어들 제거 (CAS 사용)
        bool success = true;
        foreach (var id in acquiredIds)
        {
            if (!_activeRequests.TryRemove(id, out _))
            {
                success = false;
                break;
            }
        }
        
        // 4. 실패하면 null 반환 (다른 스레드가 이미 매칭했음)
        if (!success)
        {
            // 제거된 플레이어들을 다시 추가
            foreach (var request in requests)
            {
                _activeRequests.TryAdd(request.RequestId, request);
            }
            return null;
        }
        
        return match;
    }
}
```

세 번째 전략은 파티셔닝이다. 데이터를 독립적인 파티션으로 나누면 각 파티션은 자신의 락만 관리하면 된다. MMR 범위별로 파티션을 나누면 대부분의 경우 다른 파티션과 경합하지 않는다.

```csharp
public class PartitionedMatchingQueue
{
    private const int PARTITION_COUNT = 32;
    private readonly MatchingQueuePartition[] _partitions;
    
    public PartitionedMatchingQueue()
    {
        _partitions = new MatchingQueuePartition[PARTITION_COUNT];
        for (int i = 0; i < PARTITION_COUNT; i++)
        {
            _partitions[i] = new MatchingQueuePartition(i);
        }
    }
    
    public void Enqueue(MatchRequest request)
    {
        int partitionIndex = GetPartitionIndex(request.Player.MMR);
        _partitions[partitionIndex].Enqueue(request);
    }
    
    public List<MatchRequest> GetCandidates(int targetMMR, int range)
    {
        var candidates = new List<MatchRequest>();
        
        // 관련된 파티션들만 검색
        var partitions = GetRelevantPartitions(targetMMR, range);
        foreach (var partition in partitions)
        {
            candidates.AddRange(partition.GetCandidates(targetMMR, range));
        }
        
        return candidates;
    }
    
    private int GetPartitionIndex(int mmr)
    {
        // MMR을 기반으로 파티션 결정
        return (mmr / 100) % PARTITION_COUNT;
    }
    
    private IEnumerable<MatchingQueuePartition> GetRelevantPartitions(
        int targetMMR, int range)
    {
        int minMMR = targetMMR - range;
        int maxMMR = targetMMR + range;
        
        var partitionIndices = new HashSet<int>();
        for (int mmr = minMMR; mmr <= maxMMR; mmr += 100)
        {
            partitionIndices.Add(GetPartitionIndex(mmr));
        }
        
        return partitionIndices.Select(i => _partitions[i]);
    }
}

public class MatchingQueuePartition
{
    private readonly int _partitionId;
    private readonly ConcurrentDictionary<int, ConcurrentBag<MatchRequest>> _buckets;
    private readonly ReaderWriterLockSlim _lock = new ReaderWriterLockSlim();
    
    public MatchingQueuePartition(int partitionId)
    {
        _partitionId = partitionId;
        _buckets = new ConcurrentDictionary<int, ConcurrentBag<MatchRequest>>();
    }
    
    public void Enqueue(MatchRequest request)
    {
        int bucket = request.Player.MMR / 50;
        var bag = _buckets.GetOrAdd(bucket, _ => new ConcurrentBag<MatchRequest>());
        bag.Add(request);
    }
    
    public List<MatchRequest> GetCandidates(int targetMMR, int range)
    {
        // 읽기 락만 사용 (여러 스레드 동시 읽기 가능)
        _lock.EnterReadLock();
        try
        {
            var candidates = new List<MatchRequest>();
            int minBucket = (targetMMR - range) / 50;
            int maxBucket = (targetMMR + range) / 50;
            
            for (int bucket = minBucket; bucket <= maxBucket; bucket++)
            {
                if (_buckets.TryGetValue(bucket, out var bag))
                {
                    candidates.AddRange(bag);
                }
            }
            
            return candidates;
        }
        finally
        {
            _lock.ExitReadLock();
        }
    }
}
```
   

## 4.3 확장성 확보

### 수평적 확장(Horizontal Scaling)
단일 서버로는 10만 명을 처리할 수 없으므로 여러 서버로 부하를 분산해야 한다. 수평적 확장은 서버를 추가하여 처리량을 늘리는 방식이다.

매칭 시스템은 기본적으로 상태를 가지므로 무작정 서버를 추가하면 안 된다. 각 서버가 독립적으로 동작하되, 필요할 때 협력할 수 있는 구조가 필요하다.

```csharp
public class ScalableMatchmakingCluster
{
    private readonly List<MatchmakingNode> _nodes;
    private readonly IServiceRegistry _serviceRegistry;
    private readonly IMessageBus _messageBus;
    
    public async Task ScaleUp(int additionalNodes)
    {
        for (int i = 0; i < additionalNodes; i++)
        {
            var newNode = await CreateNewNode();
            _nodes.Add(newNode);
            
            // 서비스 레지스트리에 등록
            await _serviceRegistry.RegisterNode(newNode);
            
            // 기존 노드들에게 새 노드 추가 알림
            await _messageBus.PublishAsync("cluster.node.added", new
            {
                NodeId = newNode.Id,
                Endpoint = newNode.Endpoint
            });
        }
        
        // 부하 재분배
        await RebalanceLoad();
    }
    
    private async Task RebalanceLoad()
    {
        // 각 노드의 현재 부하 확인
        var nodeLoads = new Dictionary<string, int>();
        foreach (var node in _nodes)
        {
            nodeLoads[node.Id] = await node.GetQueueSize();
        }
        
        // 부하가 높은 노드에서 낮은 노드로 일부 플레이어 이동
        var overloadedNodes = nodeLoads
            .Where(kvp => kvp.Value > 5000)
            .OrderByDescending(kvp => kvp.Value);
        
        var underloadedNodes = nodeLoads
            .Where(kvp => kvp.Value < 2000)
            .OrderBy(kvp => kvp.Value);
        
        foreach (var overloaded in overloadedNodes)
        {
            var targetNode = underloadedNodes.FirstOrDefault();
            if (targetNode.Key == null) break;
            
            var sourceNode = _nodes.First(n => n.Id == overloaded.Key);
            var destNode = _nodes.First(n => n.Id == targetNode.Key);
            
            // 일부 플레이어 이동
            await MigrateRequests(sourceNode, destNode, 1000);
        }
    }
}
```

### 샤딩(Sharding) 전략
샤딩은 데이터를 여러 서버에 분산하는 기법이다. 매칭 시스템에서는 MMR 범위나 지역을 기준으로 샤딩할 수 있다.

MMR 기반 샤딩은 비슷한 실력의 플레이어들을 같은 샤드에 모으므로 매칭 효율이 높다. 하지만 최상위나 최하위 MMR 구간은 플레이어가 적어서 불균형이 발생할 수 있다.

```csharp
public class MMRBasedShardingStrategy
{
    private readonly List<ShardInfo> _shards;
    
    public MMRBasedShardingStrategy()
    {
        _shards = new List<ShardInfo>
        {
            new ShardInfo { Id = "shard-0", MinMMR = 0, MaxMMR = 1000 },
            new ShardInfo { Id = "shard-1", MinMMR = 1000, MaxMMR = 2000 },
            new ShardInfo { Id = "shard-2", MinMMR = 2000, MaxMMR = 3000 },
            new ShardInfo { Id = "shard-3", MinMMR = 3000, MaxMMR = 4000 },
            new ShardInfo { Id = "shard-4", MinMMR = 4000, MaxMMR = int.MaxValue }
        };
    }
    
    public ShardInfo GetShard(int mmr)
    {
        return _shards.FirstOrDefault(s => mmr >= s.MinMMR && mmr < s.MaxMMR);
    }
    
    public List<ShardInfo> GetRelevantShards(int mmr, int range)
    {
        int minMMR = mmr - range;
        int maxMMR = mmr + range;
        
        return _shards
            .Where(s => !(s.MaxMMR < minMMR || s.MinMMR > maxMMR))
            .ToList();
    }
}
```

지역 기반 샤딩은 네트워크 레이턴시를 최소화한다. 각 지역마다 별도의 매칭 서버를 운영하면 대부분의 매칭이 같은 지역 내에서 완료된다.

```csharp
public class RegionBasedShardingStrategy
{
    private readonly Dictionary<string, List<MatchingNodeInfo>> _regionShards;
    
    public async Task<MatchingNodeInfo> GetNodeForRegion(string region)
    {
        if (!_regionShards.TryGetValue(region, out var nodes))
        {
            // 해당 지역에 노드가 없으면 가장 가까운 지역 사용
            return await GetNearestRegionNode(region);
        }
        
        // 해당 지역 내에서 부하가 가장 낮은 노드 선택
        var healthScores = new Dictionary<MatchingNodeInfo, double>();
        foreach (var node in nodes)
        {
            var health = await GetNodeHealth(node);
            healthScores[node] = CalculateHealthScore(health);
        }
        
        return healthScores.OrderByDescending(kvp => kvp.Value).First().Key;
    }
    
    private double CalculateHealthScore(NodeHealth health)
    {
        // 큐 크기, CPU 사용률, 메모리 사용률을 종합하여 점수 계산
        double queueScore = Math.Max(0, 1.0 - (health.QueueSize / 10000.0));
        double cpuScore = Math.Max(0, 1.0 - (health.CpuUsage / 100.0));
        double memoryScore = Math.Max(0, 1.0 - (health.MemoryUsage / 100.0));
        
        return (queueScore * 0.5) + (cpuScore * 0.3) + (memoryScore * 0.2);
    }
}
```

### 비동기 처리 패턴
매칭은 본질적으로 비동기 작업이다. 플레이어는 요청을 보내고 매칭이 완료될 때까지 기다린다. 이를 효율적으로 처리하려면 비동기 패턴을 활용해야 한다.

Task 기반 비동기 패턴(TAP)을 사용하면 스레드를 효율적으로 활용할 수 있다. 대기 중인 작업은 스레드를 차단하지 않고 완료를 기다린다.

```csharp
public class AsyncMatchmakingService
{
    private readonly ConcurrentDictionary<string, TaskCompletionSource<Match>> 
        _pendingMatches;
    
    public AsyncMatchmakingService()
    {
        _pendingMatches = new ConcurrentDictionary<string, 
            TaskCompletionSource<Match>>();
        
        // 백그라운드 매칭 프로세스 시작
        _ = Task.Run(async () => await RunMatchingLoop());
    }
    
    public async Task<Match> RequestMatch(MatchRequest request)
    {
        var tcs = new TaskCompletionSource<Match>();
        _pendingMatches.TryAdd(request.RequestId, tcs);
        
        // 큐에 추가
        await EnqueueRequest(request);
        
        // 매칭 완료 대기 (타임아웃 5분)
        var matchTask = tcs.Task;
        var timeoutTask = Task.Delay(TimeSpan.FromMinutes(5));
        
        var completedTask = await Task.WhenAny(matchTask, timeoutTask);
        
        if (completedTask == timeoutTask)
        {
            _pendingMatches.TryRemove(request.RequestId, out _);
            throw new TimeoutException("Matching timeout");
        }
        
        return await matchTask;
    }
    
    private async Task RunMatchingLoop()
    {
        while (true)
        {
            try
            {
                await ProcessMatchingCycle();
                await Task.Delay(100); // 100ms마다 실행
            }
            catch (Exception ex)
            {
                // 로깅
                Console.WriteLine($"Matching loop error: {ex.Message}");
            }
        }
    }
    
    private async Task ProcessMatchingCycle()
    {
        var matches = await FindPossibleMatches();
        
        foreach (var match in matches)
        {
            // 매치가 성공하면 대기 중인 플레이어들에게 알림
            foreach (var player in match.Players)
            {
                if (_pendingMatches.TryRemove(player.RequestId, out var tcs))
                {
                    tcs.SetResult(match);
                }
            }
        }
    }
}
```

SignalR이나 WebSocket을 사용하면 클라이언트와의 실시간 양방향 통신이 가능하다. 매칭 진행 상황을 클라이언트에게 실시간으로 알릴 수 있다.

```csharp
public class MatchmakingHub : Hub
{
    private readonly IMatchmakingService _matchmaking;
    
    public async Task<string> StartMatchmaking(MatchRequestDto request)
    {
        var requestId = Guid.NewGuid().ToString();
        
        // 매칭 시작
        _ = Task.Run(async () =>
        {
            try
            {
                var match = await _matchmaking.RequestMatch(request);
                
                // 매칭 완료 알림
                await Clients.Client(Context.ConnectionId)
                    .SendAsync("MatchFound", new
                    {
                        MatchId = match.Id,
                        GameServer = match.GameServer.Endpoint,
                        Players = match.Players
                    });
            }
            catch (TimeoutException)
            {
                await Clients.Client(Context.ConnectionId)
                    .SendAsync("MatchingTimeout");
            }
        });
        
        // 주기적으로 진행 상황 업데이트
        _ = Task.Run(async () =>
        {
            while (!_matchmaking.IsCompleted(requestId))
            {
                var status = await _matchmaking.GetStatus(requestId);
                
                await Clients.Client(Context.ConnectionId)
                    .SendAsync("MatchingProgress", new
                    {
                        EstimatedWaitTime = status.EstimatedWaitTime,
                        QueuePosition = status.QueuePosition
                    });
                
                await Task.Delay(2000); // 2초마다 업데이트
            }
        });
        
        return requestId;
    }
    
    public async Task CancelMatchmaking(string requestId)
    {
        await _matchmaking.CancelRequest(requestId);
    }
}
```
