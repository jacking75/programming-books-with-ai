# 3. 매칭 알고리즘 설계

## 3.1 매칭 풀(Pool) 관리

### 대기열 구조 설계
10만 명 이상의 동시 유저를 처리하기 위해서는 효율적인 대기열 구조가 필수적이다. 단순한 선입선출(FIFO) 큐로는 MMR 기반 매칭의 복잡한 요구사항을 충족할 수 없다.

가장 기본적인 접근은 MMR 범위별로 대기열을 분할하는 것이다. 예를 들어 MMR 0-1000, 1000-2000처럼 구간을 나누면 검색 공간을 크게 줄일 수 있다. 하지만 이 방식은 경계값 근처의 플레이어들이 매칭되기 어렵다는 단점이 있다.

더 나은 방법은 다층 인덱싱 구조를 사용하는 것이다. 첫 번째 레이어에서는 MMR 범위로, 두 번째 레이어에서는 대기 시간으로, 세 번째 레이어에서는 핑(latency)으로 분류한다. 이렇게 하면 각 기준에 따라 빠르게 후보를 필터링할 수 있다.

```csharp
public class MatchmakingPool
{
    // MMR 범위별로 버킷 생성 (100 단위)
    private const int MMR_BUCKET_SIZE = 100;
    private readonly ConcurrentDictionary<int, ConcurrentQueue<MatchRequest>> _mmrBuckets;
    private readonly ConcurrentDictionary<string, MatchRequest> _activeRequests;
    
    public MatchmakingPool()
    {
        _mmrBuckets = new ConcurrentDictionary<int, ConcurrentQueue<MatchRequest>>();
        _activeRequests = new ConcurrentDictionary<string, MatchRequest>();
    }
    
    private int GetBucketIndex(int mmr)
    {
        return mmr / MMR_BUCKET_SIZE;
    }
    
    public void AddRequest(MatchRequest request)
    {
        int bucketIndex = GetBucketIndex(request.Player.MMR);
        var bucket = _mmrBuckets.GetOrAdd(bucketIndex, 
            _ => new ConcurrentQueue<MatchRequest>());
        
        bucket.Enqueue(request);
        _activeRequests.TryAdd(request.RequestId, request);
    }
}
```

### 우선순위 큐 vs 버킷 시스템
우선순위 큐는 대기 시간이 긴 플레이어를 우선적으로 매칭시킬 수 있다는 장점이 있다. 하지만 10만 명 규모에서는 힙(Heap) 연산의 오버헤드가 문제가 된다. 삽입과 삭제가 O(log n)이므로 대규모 환경에서는 병목이 발생한다.

버킷 시스템은 MMR 범위별로 플레이어를 분류하여 검색 범위를 제한한다. 각 버킷 내에서는 간단한 큐나 리스트를 사용하므로 삽입이 O(1)이다. 매칭 시에는 현재 버킷과 인접 버킷만 검색하면 되므로 효율적이다.

실전에서는 하이브리드 접근이 효과적이다. 기본적으로는 버킷 시스템을 사용하되, 각 버킷 내에서 대기 시간이 임계값을 초과한 플레이어들은 별도의 "급행" 풀에 넣는다. 이 풀의 플레이어들은 MMR 매칭 기준을 완화하여 더 빠르게 매칭된다.

```csharp
public class HybridMatchingPool
{
    private readonly Dictionary<int, LinkedList<MatchRequest>> _mmrBuckets;
    private readonly PriorityQueue<MatchRequest, DateTime> _expressQueue;
    private readonly TimeSpan _expressThreshold = TimeSpan.FromSeconds(60);
    
    public IEnumerable<MatchRequest> GetCandidates(int targetMMR, int maxRange)
    {
        // 급행 큐에서 먼저 확인
        var expressCandidates = GetExpressCandidates();
        
        // MMR 버킷에서 후보 수집
        int centerBucket = GetBucketIndex(targetMMR);
        int bucketRange = maxRange / MMR_BUCKET_SIZE + 1;
        
        var candidates = new List<MatchRequest>();
        for (int offset = 0; offset <= bucketRange; offset++)
        {
            AddBucketCandidates(candidates, centerBucket + offset, targetMMR, maxRange);
            if (offset > 0)
                AddBucketCandidates(candidates, centerBucket - offset, targetMMR, maxRange);
        }
        
        return expressCandidates.Concat(candidates);
    }
    
    private void AddBucketCandidates(List<MatchRequest> candidates, int bucketIndex, 
        int targetMMR, int maxRange)
    {
        if (!_mmrBuckets.TryGetValue(bucketIndex, out var bucket))
            return;
            
        foreach (var request in bucket)
        {
            if (Math.Abs(request.Player.MMR - targetMMR) <= maxRange)
                candidates.Add(request);
        }
    }
}
```

### 10만+ 유저 처리를 위한 분산 처리
단일 서버로 10만 명을 처리하는 것은 메모리와 CPU 측면에서 도전적이다. 분산 처리가 필수적이다.

가장 간단한 방법은 MMR 범위별로 서버를 샤딩하는 것이다. 예를 들어 서버 1은 MMR 0-2000, 서버 2는 MMR 2000-4000을 담당한다. 각 서버는 독립적으로 매칭을 처리하므로 확장이 용이하다. 하지만 경계 구간의 플레이어들이 매칭되기 어렵다는 문제가 있다.

더 나은 접근은 일관된 해싱(Consistent Hashing)을 사용하는 것이다. 플레이어의 지역, 플랫폼, MMR 범위 등을 조합한 키로 해싱하여 서버를 할당한다. 서버가 추가되거나 제거될 때 재배치되는 플레이어 수를 최소화할 수 있다.

```csharp
public class DistributedMatchmakingCoordinator
{
    private readonly List<MatchmakingNode> _nodes;
    private readonly ConsistentHash<MatchmakingNode> _hashRing;
    
    public DistributedMatchmakingCoordinator(List<MatchmakingNode> nodes)
    {
        _nodes = nodes;
        _hashRing = new ConsistentHash<MatchmakingNode>(nodes, 150); // 150개 가상 노드
    }
    
    public async Task<string> SubmitMatchRequest(MatchRequest request)
    {
        // 플레이어 속성으로 라우팅 키 생성
        string routingKey = $"{request.Player.Region}:{GetMMRTier(request.Player.MMR)}";
        var targetNode = _hashRing.GetNode(routingKey);
        
        return await targetNode.AddToQueue(request);
    }
    
    private string GetMMRTier(int mmr)
    {
        // 500 단위로 티어 구분
        return (mmr / 500).ToString();
    }
}
```

각 매칭 노드는 독립적으로 동작하지만, 특정 노드에 플레이어가 부족할 경우 인접 노드와 협력할 수 있다. 이를 위해 노드 간 통신 채널을 구현한다.

## 3.2 매칭 기준과 허용 범위

### MMR 차이 허용 범위 설정
FPS 게임에서 MMR 차이는 게임의 재미에 직접적인 영향을 미친다. 너무 큰 차이는 일방적인 게임을 만들고, 너무 좁은 범위는 매칭 시간을 늘린다.

초기 허용 범위는 ±100 MMR 정도로 시작하는 것이 일반적이다. 이는 약 5% 정도의 실력 차이에 해당한다. 5v5 팀 게임의 경우 각 팀의 평균 MMR 차이가 50 이내가 되도록 조정한다.

```csharp
public class MatchingCriteria
{
    private const int INITIAL_MMR_RANGE = 100;
    private const int MAX_MMR_RANGE = 500;
    private const int RANGE_EXPANSION_INTERVAL_SECONDS = 10;
    private const int RANGE_EXPANSION_AMOUNT = 50;
    
    public int GetAllowedMMRRange(DateTime requestTime)
    {
        var waitTime = DateTime.UtcNow - requestTime;
        int expansions = (int)(waitTime.TotalSeconds / RANGE_EXPANSION_INTERVAL_SECONDS);
        
        int range = INITIAL_MMR_RANGE + (expansions * RANGE_EXPANSION_AMOUNT);
        return Math.Min(range, MAX_MMR_RANGE);
    }
    
    public bool IsAcceptableMatch(List<Player> team1, List<Player> team2)
    {
        double team1Avg = team1.Average(p => p.MMR);
        double team2Avg = team2.Average(p => p.MMR);
        
        // 팀 평균 차이 확인
        if (Math.Abs(team1Avg - team2Avg) > 50)
            return false;
        
        // 각 팀 내 최고-최저 차이 확인
        if (GetMMRSpread(team1) > MAX_MMR_RANGE || 
            GetMMRSpread(team2) > MAX_MMR_RANGE)
            return false;
        
        return true;
    }
    
    private double GetMMRSpread(List<Player> team)
    {
        return team.Max(p => p.MMR) - team.Min(p => p.MMR);
    }
}
```

### 대기 시간에 따른 동적 범위 확장
플레이어가 오래 기다릴수록 매칭 기준을 완화해야 한다. 하지만 무제한 확장은 게임 품질을 해친다. 따라서 최대 범위를 설정하고 단계적으로 확장한다.

동적 확장은 선형 또는 지수적으로 할 수 있다. 선형 확장은 예측 가능하지만 초반에는 너무 느리고 후반에는 너무 빠를 수 있다. 지수적 확장은 초반에는 엄격하게 유지하다가 일정 시간 후 빠르게 완화된다.

```csharp
public class DynamicRangeCalculator
{
    public int CalculateMMRRange(MatchRequest request, RangeExpansionStrategy strategy)
    {
        var waitTime = DateTime.UtcNow - request.SubmitTime;
        
        return strategy switch
        {
            RangeExpansionStrategy.Linear => CalculateLinear(waitTime),
            RangeExpansionStrategy.Exponential => CalculateExponential(waitTime),
            RangeExpansionStrategy.Stepped => CalculateStepped(waitTime),
            _ => INITIAL_MMR_RANGE
        };
    }
    
    private int CalculateLinear(TimeSpan waitTime)
    {
        int range = INITIAL_MMR_RANGE + (int)(waitTime.TotalSeconds * 2);
        return Math.Min(range, MAX_MMR_RANGE);
    }
    
    private int CalculateExponential(TimeSpan waitTime)
    {
        double multiplier = Math.Pow(1.05, waitTime.TotalSeconds / 5);
        int range = (int)(INITIAL_MMR_RANGE * multiplier);
        return Math.Min(range, MAX_MMR_RANGE);
    }
    
    private int CalculateStepped(TimeSpan waitTime)
    {
        // 30초마다 단계적으로 확장
        int step = (int)(waitTime.TotalSeconds / 30);
        int[] ranges = { 100, 150, 250, 400, 500 };
        return ranges[Math.Min(step, ranges.Length - 1)];
    }
}
```

실제 운영에서는 플레이어의 MMR 티어에 따라 다른 전략을 사용한다. 최상위 티어는 플레이어 수가 적으므로 더 빠르게 범위를 확장하고, 중간 티어는 플레이어가 많으므로 엄격하게 유지한다.

### 팀 밸런싱 알고리즘
5v5 팀 게임에서는 단순히 10명을 모으는 것이 아니라 균형 잡힌 두 팀을 만들어야 한다. 가장 간단한 방법은 MMR 순으로 정렬 후 뱀 드래프트(Snake Draft) 방식으로 배정하는 것이다.

```csharp
public class TeamBalancer
{
    public (List<Player> team1, List<Player> team2) BalanceTeams(List<Player> players)
    {
        if (players.Count != 10)
            throw new ArgumentException("Exactly 10 players required");
        
        // MMR 기준 내림차순 정렬
        var sorted = players.OrderByDescending(p => p.MMR).ToList();
        
        var team1 = new List<Player>();
        var team2 = new List<Player>();
        
        // 뱀 드래프트: 1,2,2,1,1,2,2,1,1,2
        for (int i = 0; i < sorted.Count; i++)
        {
            if ((i / 2) % 2 == 0)
                team1.Add(sorted[i]);
            else
                team2.Add(sorted[i]);
        }
        
        return (team1, team2);
    }
}
```

하지만 이 방법은 역할(롤) 시스템이 있는 게임에서는 부적절하다. 예를 들어 모든 고랭크 플레이어가 같은 역할을 선호하면 한 팀에 몰릴 수 있다. 더 정교한 접근은 조합 최적화를 사용하는 것이다.

```csharp
public class AdvancedTeamBalancer
{
    public (List<Player> team1, List<Player> team2) BalanceWithRoles(
        List<Player> players, Dictionary<Player, List<Role>> rolePreferences)
    {
        var bestSplit = FindOptimalSplit(players, rolePreferences);
        return bestSplit;
    }
    
    private (List<Player>, List<Player>) FindOptimalSplit(
        List<Player> players, Dictionary<Player, List<Role>> rolePreferences)
    {
        // 가능한 모든 5:5 조합을 시도하기엔 너무 많으므로 (252가지)
        // 휴리스틱 접근 사용
        
        double bestScore = double.MaxValue;
        var bestTeam1 = new List<Player>();
        var bestTeam2 = new List<Player>();
        
        // 시드로 몇 가지 좋은 후보 생성
        var candidates = GenerateCandidateSplits(players, 50);
        
        foreach (var (team1, team2) in candidates)
        {
            double score = EvaluateSplit(team1, team2, rolePreferences);
            if (score < bestScore)
            {
                bestScore = score;
                bestTeam1 = team1;
                bestTeam2 = team2;
            }
        }
        
        return (bestTeam1, bestTeam2);
    }
    
    private double EvaluateSplit(List<Player> team1, List<Player> team2,
        Dictionary<Player, List<Role>> rolePreferences)
    {
        // 여러 요소를 고려한 점수 계산
        double mmrDiff = Math.Abs(team1.Average(p => p.MMR) - team2.Average(p => p.MMR));
        double roleBalance = EvaluateRoleBalance(team1, rolePreferences) + 
                            EvaluateRoleBalance(team2, rolePreferences);
        
        return mmrDiff * 2.0 + roleBalance * 1.0;
    }
    
    private double EvaluateRoleBalance(List<Player> team, 
        Dictionary<Player, List<Role>> rolePreferences)
    {
        // 각 역할이 잘 커버되는지 평가
        var rolesCovered = new HashSet<Role>();
        foreach (var player in team)
        {
            if (rolePreferences.TryGetValue(player, out var roles))
                rolesCovered.UnionWith(roles);
        }
        
        return 5 - rolesCovered.Count; // 5개 역할 중 몇 개가 없는지
    }
}
```
  

## 3.3 지역/핑 기반 매칭

### 단일 서버 위치에서의 핑 최적화
전세계에서 하나의 물리적 서버 위치를 사용할 때는 핑이 매칭 품질의 핵심 요소가 된다. FPS 게임에서 100ms 이상의 핑 차이는 명중률과 반응 속도에 큰 영향을 미친다.

먼저 각 플레이어가 매칭 요청 시 서버까지의 핑을 측정하여 전송해야 한다. 이 정보를 바탕으로 핑이 비슷한 플레이어끼리 매칭한다.

### 지역별 우선순위 설정
같은 지역의 플레이어끼리 먼저 매칭을 시도하고, 충분한 플레이어가 없을 때만 다른 지역으로 확장한다. 지역 간 우선순위는 서버 위치와의 물리적 거리를 고려한다.

```csharp
public class RegionBasedMatcher
{
    private readonly Dictionary<string, int> _regionPriority;
    private readonly string _serverRegion;
    
    public RegionBasedMatcher(string serverRegion)
    {
        _serverRegion = serverRegion;
        _regionPriority = new Dictionary<string, int>
        {
            { "NA-EAST", 0 },    // 서버가 동부에 있다고 가정
            { "NA-WEST", 1 },
            { "EU-WEST", 2 },
            { "EU-EAST", 2 },
            { "ASIA-EAST", 3 },
            { "OCEANIA", 4 }
        };
    }
    
    public async Task<Match> FindMatch(MatchRequest request)
    {
        // 1단계: 같은 지역에서만 검색
        var sameRegionMatch = await FindInRegion(request, request.Region);
        if (sameRegionMatch != null)
            return sameRegionMatch;
        
        // 2단계: 대기 시간이 30초 이상이면 인접 지역 포함
        if (GetWaitTime(request) > TimeSpan.FromSeconds(30))
        {
            var nearbyRegions = GetNearbyRegions(request.Region);
            foreach (var region in nearbyRegions)
            {
                var match = await FindInRegion(request, region);
                if (match != null)
                    return match;
            }
        }
        
        // 3단계: 60초 이상이면 모든 지역 포함 (핑 제한만 유지)
        if (GetWaitTime(request) > TimeSpan.FromSeconds(60))
        {
            return await FindGlobally(request);
        }
        
        return null;
    }
    
    private List<string> GetNearbyRegions(string region)
    {
        int basePriority = _regionPriority[region];
        return _regionPriority
            .Where(kvp => Math.Abs(kvp.Value - basePriority) <= 1)
            .Select(kvp => kvp.Key)
            .ToList();
    }
}
```

### 핑 임계값 관리
핑 차이가 너무 크면 한쪽이 불리해진다. 따라서 매치 내 최대 핑 차이를 제한해야 한다. 일반적으로 50ms 차이가 체감 가능한 수준이므로 이를 기준으로 설정한다.

```csharp
public class PingConstraintValidator
{
    private const int MAX_PING_DIFFERENCE = 80;
    private const int ABSOLUTE_MAX_PING = 200;
    private const int PREFERRED_MAX_PING = 100;
    
    public bool IsValidPingDistribution(List<Player> players, 
        Dictionary<Player, int> pingMap)
    {
        var pings = players.Select(p => pingMap[p]).ToList();
        
        // 절대 핑 제한
        if (pings.Any(p => p > ABSOLUTE_MAX_PING))
            return false;
        
        // 핑 차이 제한
        int minPing = pings.Min();
        int maxPing = pings.Max();
        if (maxPing - minPing > MAX_PING_DIFFERENCE)
            return false;
        
        return true;
    }
    
    public double CalculatePingQualityScore(List<Player> players,
        Dictionary<Player, int> pingMap)
    {
        var pings = players.Select(p => pingMap[p]).ToList();
        double avgPing = pings.Average();
        double pingSpread = pings.Max() - pings.Min();
        
        // 낮은 평균 핑과 작은 핑 차이가 좋은 점수
        double avgScore = Math.Max(0, (PREFERRED_MAX_PING - avgPing) / PREFERRED_MAX_PING);
        double spreadScore = Math.Max(0, (MAX_PING_DIFFERENCE - pingSpread) / MAX_PING_DIFFERENCE);
        
        return (avgScore * 0.6) + (spreadScore * 0.4);
    }
}
```

실전에서는 핑과 MMR의 우선순위를 동적으로 조정한다. 초반에는 MMR을 우선하다가 대기 시간이 길어지면 핑을 더 중요하게 고려한다.

```csharp
public class AdaptiveMatchingStrategy
{
    public List<Player> FindBestMatch(MatchRequest seed, List<MatchRequest> candidates)
    {
        var waitTime = DateTime.UtcNow - seed.SubmitTime;
        double mmrWeight = CalculateMMRWeight(waitTime);
        double pingWeight = 1.0 - mmrWeight;
        
        var scoredCandidates = candidates
            .Select(c => new {
                Candidate = c,
                Score = CalculateMatchScore(seed, c, mmrWeight, pingWeight)
            })
            .OrderBy(x => x.Score)
            .Take(9)
            .Select(x => x.Candidate.Player)
            .ToList();
        
        scoredCandidates.Insert(0, seed.Player);
        return scoredCandidates;
    }
    
    private double CalculateMMRWeight(TimeSpan waitTime)
    {
        // 0초: 0.8, 30초: 0.6, 60초: 0.4, 90초+: 0.3
        if (waitTime.TotalSeconds < 30)
            return 0.8;
        if (waitTime.TotalSeconds < 60)
            return 0.6;
        if (waitTime.TotalSeconds < 90)
            return 0.4;
        return 0.3;
    }
    
    private double CalculateMatchScore(MatchRequest seed, MatchRequest candidate,
        double mmrWeight, double pingWeight)
    {
        double mmrDiff = Math.Abs(seed.Player.MMR - candidate.Player.MMR);
        double mmrScore = mmrDiff / 500.0; // 정규화
        
        double pingDiff = Math.Abs(seed.PingMs - candidate.PingMs);
        double pingScore = pingDiff / 100.0; // 정규화
        
        return (mmrScore * mmrWeight) + (pingScore * pingWeight);
    }
}
```

이러한 매칭 알고리즘들을 조합하면 10만 명 이상의 동시 유저를 처리하면서도 빠른 매칭 시간과 높은 게임 품질을 유지할 수 있다. 핵심은 초기에는 엄격한 기준을 적용하되, 대기 시간이 길어질수록 점진적으로 완화하는 것이다. 또한 분산 처리를 통해 처리량을 확보하고, 지역과 핑 정보를 활용하여 레이턴시가 낮은 매치를 만드는 것이 중요하다.
